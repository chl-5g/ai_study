## 0. 与 A1、C4 的衔接

**本节概念**：**MCP** 把「工具有哪些、怎么调用」从各框架私货，提升为 **可复用协议**；**C4** 的 `tools` 列表是 **单次 HTTP 会话里的视图**，MCP 是 **进程级或远程级的能力发布**。

A1 里你写过 **注册表 + 装饰器** 式的 API；`ToolRegistry` 本质上就是 **name → callable** 的 dict。MCP 做的事是把这张 dict **序列化协议化**，让 Cursor、Claude Desktop、自研 Agent 宿主都能 **同一套 list/call** 消费，而不是每个产品重写一遍 Slack/Git 连接器。

**本节小结（三张卡片）**

- **一句话定义**：MCP = **工具能力的 USB-C 接口**（形态统一，内里仍各自实现）。
- **典型场景**：团队维护一个「读内网 Wiki」的 Server，桌面 IDE 与批处理任务共用。
- **和 A1 类比**：MCP Server 像 **实现某一组 ABC 抽象基类的插件包**；宿主只依赖抽象，不依赖具体站点。

## 1. Why：工具爆炸与重复集成

**本节概念**：没有协议层时，每个 Agent 框架各写一套「读文件、查库、调 Slack」适配层，导致 **无法复用、测试分裂、权限模型分裂**。

当你有三个宿主（IDE Agent、客服机器人、运维 CLI）都要读同一套内网文档，最烂的做法是复制三份 **路径白名单与审计逻辑**。MCP 把 **能力清单 + 调用约定** 标准化，使 **同一实现** 可被多个宿主消费；宿主只实现 **传输 + 鉴权 + 会话**。

**本节小结（三张卡片）**

- **一句话定义**：集成分裂的根因是 **缺一层「工具 OS」**。
- **典型场景**：换框架从 LangChain 迁到自研循环，工具层零复用。
- **和 A1 类比**：像每个项目自己实现 `importlib` —— 能跑，但生态互操作性为零。

## 2. MCP 解决什么问题

**本节概念**：MCP 在工程上解决三件事：**发现（list）**、**调用（call）**、**演进（版本化 Server）**。

- **发现**：客户端能 **列举** 某 Server 提供的 tools/resources/prompts。  
- **调用**：按协议 **传参、取回内容**（文本、资源引用等）。  
- **演进**：工具实现可独立版本化，宿主侧只做 **协议兼容**。

**本节小结（三张卡片）**

- **一句话定义**：MCP 把工具从「代码里的函数指针」变成「网络上的能力描述」。
- **典型场景**：新增工具只发版 Server，宿主无感升级（只要协议向后兼容）。
- **和 A1 类比**：`dir(module)` 列出符号，`getattr` 调用 —— list/call 是其网络化版本。

## 3. 核心概念：Server、Client、Transport

**本节概念**：**Server** 挂载能力；**Client（宿主）**嵌入产品进程发起 list/call；**Transport** 决定 **信任边界与延迟**。

- **MCP Server**：挂载具体能力（读仓库、执行查询…）。  
- **MCP Client（宿主）**：发起 list/call，嵌入在自己的产品里。  
- **Transport**  
  - **stdio**：本地子进程管道，低延迟，适合本机工具；**信任模型** ≈ 同一用户启动的子进程。  
  - **HTTP/SSE/WebSocket**：远程服务，需注意 **TLS、认证、超时、多租户**；**信任模型** ≈ 内网服务调用。

**stdio 的错觉风险**：子进程继承 **环境变量与文件描述符**，误配可能把 **宿主机密钥** 泄漏给不可信 Server —— 仍要做 **最小权限启动**。

**本节小结（三张卡片）**

- **一句话定义**：Transport 选错 = **把本地 shell 能力远程暴露** 或 **把远程 RTT 引进交互环**。
- **典型场景**：本机读 repo 用 stdio；跨机房查 CMDB 用 mTLS HTTP。
- **和 A1 类比**：stdio 像 **`subprocess.run` 管道**；HTTP 像 **`urllib.request` 跨机** —— 错误处理与超时完全不同。

## 4. 与「手写 OpenAPI 网关」的对比

**本节概念**：**OpenAPI** 描述 **业务域 REST**；**MCP** 描述 **Agent 可调工具面**；二者可并存。

| | MCP | 自建 HTTP 网关 |
|--|-----|----------------|
| 目标 | 工具/资源 **跨宿主复用** | 服务 **自有业务 API** |
| 形态 | 协议 + 多语言 SDK | REST/GraphQL 自定 |
| 适用 | 桌面/Agent 生态集成 | 对外稳定业务边界 |

二者可并存：**MCP 调本地工具**，**HTTP 调公司订单服务**。不要让 MCP Server 直接当 **公网 BFF**，除非你已经按网关标准做了 **鉴权、限流、WAF**。

**本节小结（三张卡片）**

- **一句话定义**：MCP 不是 REST 的替代品，是 **工具集成层**。
- **典型场景**：订单创建仍走内部 gRPC；只有「读文档摘要」走 MCP。
- **和 A1 类比**：像 **`pathlib` 与 `os.path` 并存** —— 场景不同，不互斥。

## 5. 安全模型

**本节概念**：工具 = **任意代码执行的前门**；安全默认 **拒绝**，再 **白名单放行**。

- 任何工具都可能 **读盘/出网** → **最小权限**、路径白名单、出站 allowlist。  
- **远程 MCP** 必须 **鉴权**（Token/mTLS），防 SSRF 与凭证窃取。  
- **日志**：参数脱敏；**多租户** 实例隔离或强制带 tenant 头。

**本节小结（三张卡片）**

- **一句话定义**：MCP 安全的中心句是 **「不信任模型，也不信任 Server 实现」**。
- **典型场景**：工具参数里带 URL，Server 内网 SSRF 扫遍 `169.254.169.254`。
- **和 A1 类比**：像 **`eval` 禁用 + ast 白名单** —— 能力面必须显式列出。

## 6. 从 ToolRegistry 到 MCP 的直觉映射

**本节概念**：`ToolRegistry` 的 **`list_openai_tools` + `call`**，在概念上对应 MCP 的 **列举工具 + 调用**；`02_stdio_tool_host.py` 演示 **进程边界** 上的 JSON 命令（教学用，非完整 MCP 帧格式）。正式开发请用官方 **`mcp` Python 包** 生成符合规范的 Server。

读 demo 时抓住三条映射：**注册表 ↔ Server 提供的清单**、**call 分发 ↔ MCP handler**、**JSON 边界 ↔ Transport 帧**。其余是工程细节（重连、心跳、并发）。

**本节小结（三张卡片）**

- **一句话定义**：教学 demo 给你 **心智模型**；生产用官方 SDK 补齐 **协议完整性**。
- **典型场景**：把现有 `dict[str, Callable]` 工具层渐进迁移为 MCP Server，宿主侧逐步换 transport。
- **和 A1 类比**：手写 JSON 行协议像 **手写 HTTP 解析** —— 学习可以，上线不要。
