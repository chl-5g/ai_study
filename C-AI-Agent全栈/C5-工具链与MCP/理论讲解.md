## 0. 与 A1、C4 的衔接

**本节概念**：**MCP** 把「工具有哪些、怎么调用」从各框架的私有实现，提升为 **可复用的开放协议**；**C4** 的 `tools` 列表是 **单次 HTTP 会话里的视图**，MCP 是 **进程级或远程级的能力发布**。

在 A1 里你写过 **注册表 + 装饰器** 式的 API。回忆一下那个模式：你用一个字典把函数名映射到函数本身，然后通过 `registry["add"](1, 2)` 来调用。`ToolRegistry` 本质上就是这样一个 **name → callable** 的字典。

MCP 做的事情，是把这张字典 **序列化、协议化**。想象一下：你在 A1 里写的那个字典只能在同一个 Python 进程里用。如果你想让 Cursor、Claude Desktop、你自研的 Agent 宿主都能使用同一套 Slack 连接器或 Git 查询工具，你就需要一个「网络版的字典」——这就是 MCP。

C4 教你的 `tool_calls` 是模型说「我想调这个函数」，宿主负责执行。MCP 解决的是「宿主怎么**发现**有哪些函数可调、怎么**调用**它们」——尤其是当这些函数不在宿主进程里，而是在另一个进程甚至另一台机器上的时候。

**本节小结（三张卡片）**

- **一句话定义**：MCP = **工具能力的 USB-C 接口**（形态统一，内里仍各自实现）。
- **典型场景**：团队维护一个「读内网 Wiki」的 Server，桌面 IDE 与批处理任务共用，不用各写一份。
- **和 A1 类比**：MCP Server 像 **实现某一组 ABC 抽象基类的插件包**；宿主只依赖抽象，不依赖具体实现。

## 1. Why：工具爆炸与重复集成

**本节概念**：没有协议层时，每个 Agent 框架各写一套工具适配层，导致 **M × N 问题**：M 个工具 × N 个框架 = M×N 份适配代码，**无法复用、测试分裂、权限模型分裂**。

### 1.1 一个具体的痛苦场景

假设你在一家 AI startup 做 Agent 产品。你的 Agent 需要接入 4 个工具：

1. **Slack** — 读取消息、发送通知
2. **GitHub** — 查看 Issue、读取代码文件
3. **PostgreSQL** — 查询业务数据
4. **本地文件系统** — 读写用户文档

同时，你的公司有 3 个产品需要用这些工具：

1. **桌面 IDE 插件**（类似 Cursor 里的 Agent）
2. **客服机器人**（Web 端）
3. **运维 CLI 工具**（终端里跑的自动化脚本）

没有统一协议时，情况是这样的：

```
                    Slack   GitHub   PostgreSQL   文件系统
                    ─────   ──────   ──────────   ────────
IDE 插件             适配1    适配2     适配3        适配4
客服机器人           适配5    适配6     适配7        适配8
运维 CLI             适配9    适配10    适配11       适配12
```

4 × 3 = 12 份适配代码。每份都要写：连接逻辑、鉴权、错误处理、参数校验、日志格式。**任何一个工具 API 变了，三个产品都要改。任何一个产品架构调了，四个工具适配层都要跟着动。**

这就是 **M × N 问题**。随着工具和产品增加，工作量 **乘法增长**。

### 1.2 MCP 如何把 M×N 变成 M+N

MCP 引入了一个中间协议层。每个工具只需要实现一个 MCP Server（遵守统一协议），每个产品只需要实现一个 MCP Client（也遵守统一协议）：

```
                         MCP 协议
                            │
IDE 插件 ──── MCP Client ───┤
客服机器人 ── MCP Client ───┤──── MCP Server (Slack)
运维 CLI ──── MCP Client ───┤──── MCP Server (GitHub)
                            │──── MCP Server (PostgreSQL)
                            │──── MCP Server (文件系统)
```

现在是 4 个 Server + 3 个 Client = **7 份代码**，而不是 12 份。更重要的是：

- **Slack 的 API 变了**？只改 Slack 的 MCP Server，三个 Client 不动。
- **新增一个工具**（比如 Jira）？写一个 MCP Server 就行，三个产品自动能用。
- **新增一个产品**？写一个 MCP Client，所有已有工具自动接入。

这就是为什么 MCP 被称为「工具的 USB-C 接口」——你不用管对面插的是硬盘还是显示器，接口统一就能通信。

### 1.3 现实中的例子

你可能已经在用 MCP 了：

- **Claude Desktop** 支持用 MCP 接入本地文件系统、数据库等工具
- **Cursor IDE** 的 Agent 通过 MCP 调用各种开发工具
- **Claude Code**（你现在用的这个）本身就支持 MCP Server

这些产品不用为每个工具写专门的集成代码——它们实现了 MCP Client，然后社区或官方提供各种 MCP Server。

**本节小结（三张卡片）**

- **一句话定义**：集成分裂的根因是 **缺一层「工具协议」**，MCP 就是这层协议。
- **典型场景**：换框架从 LangChain 迁到自研循环，工具层零复用 → 用 MCP 后工具层直接插拔。
- **和 A1 类比**：像每个项目自己实现 `importlib` —— 能跑，但生态互操作性为零。用 MCP 就是用标准 `import`。

## 2. MCP 解决什么问题

**本节概念**：MCP 在工程上解决三件事：**发现（list）**、**调用（call）**、**演进（版本化 Server）**。

### 2.1 发现：「你有什么能力？」

当一个 MCP Client（比如 Claude Desktop）连接到一个 MCP Server 时，第一件事就是问：「你提供哪些工具？」

Server 会返回一份清单，包含每个工具的名字、描述、参数格式。这就像你走进一家餐厅，先看菜单（而不是冲进厨房自己找食材）。

在代码层面，这对应一个 **`tools/list`** 请求：

```json
// Client 发出的请求
{
  "jsonrpc": "2.0",
  "id": 1,
  "method": "tools/list"
}

// Server 返回的响应
{
  "jsonrpc": "2.0",
  "id": 1,
  "result": {
    "tools": [
      {
        "name": "read_file",
        "description": "读取指定路径的文件内容（只读，不会修改文件）",
        "inputSchema": {
          "type": "object",
          "properties": {
            "path": { "type": "string", "description": "文件的绝对路径" }
          },
          "required": ["path"]
        }
      },
      {
        "name": "list_directory",
        "description": "列出目录下的所有文件和子目录",
        "inputSchema": {
          "type": "object",
          "properties": {
            "path": { "type": "string", "description": "目录的绝对路径" }
          },
          "required": ["path"]
        }
      }
    ]
  }
}
```

注意这个 `inputSchema` 就是 JSON Schema —— 和 C4 里 OpenAI `tools[].function.parameters` 的格式 **完全一致**。这不是巧合，MCP 就是为了和 LLM 的工具调用无缝对接而设计的。

### 2.2 调用：「帮我执行这个」

Client 知道有哪些工具后，就可以调用了。对应 **`tools/call`** 请求：

```json
// Client 发出的请求
{
  "jsonrpc": "2.0",
  "id": 2,
  "method": "tools/call",
  "params": {
    "name": "read_file",
    "arguments": {
      "path": "/home/user/readme.md"
    }
  }
}

// Server 返回的响应
{
  "jsonrpc": "2.0",
  "id": 2,
  "result": {
    "content": [
      {
        "type": "text",
        "text": "# My Project\nThis is a demo project..."
      }
    ]
  }
}
```

Client 拿到结果后，可以直接把 `content` 里的文本塞回给 LLM 作为 tool 消息（就是 C4 里的 `role: tool`）。

### 2.3 演进：「我升级了，你不用改」

MCP Server 可以独立版本化。比如你的 `read_file` 工具从 v1 升级到 v2，新增了一个 `encoding` 参数：

- **旧参数**仍然正常工作（向后兼容）
- **新参数**是可选的，不影响已有 Client
- Client 侧只需要重新 `tools/list` 就能看到新的 schema

这意味着：**升级工具不需要重新部署所有使用它的产品**。你只需要重启 MCP Server，所有连着它的 Client 自动获得新能力。

### 2.4 MCP 不只有工具

MCP 协议除了 `tools` 外，还定义了两类能力：

| 能力 | 说明 | 类比 |
|------|------|------|
| **Tools** | 可执行的函数（读文件、查数据库） | 餐厅的菜品 |
| **Resources** | 可读取的数据源（文件内容、数据库表） | 餐厅的菜单本身 |
| **Prompts** | 预定义的提示词模板 | 餐厅的套餐推荐 |

本章主要讲 **Tools**，因为它与 C4 的 Function Calling 直接对应，是最常用的能力。

**本节小结（三张卡片）**

- **一句话定义**：MCP 把工具从「代码里的函数指针」变成「网络上的能力描述」，支持发现、调用、演进。
- **典型场景**：新增工具只发版 Server，宿主无感升级（只要协议向后兼容）。
- **和 A1 类比**：`dir(module)` 列出符号，`getattr` 调用 —— `tools/list` 和 `tools/call` 就是其网络化版本。

## 3. 核心概念：Server、Client、Transport

**本节概念**：**Server** 挂载能力（提供工具）；**Client（宿主）** 嵌入产品进程发起 list/call；**Transport** 决定两者之间的 **通信方式、信任边界与延迟**。

### 3.1 MCP Server

MCP Server 是 **提供工具的一方**。它做两件事：

1. **注册工具**：声明自己有哪些工具、每个工具的参数格式
2. **执行工具**：收到 `tools/call` 请求后，执行对应的业务逻辑，返回结果

你可以把 MCP Server 理解为一个 **「工具服务员」**——它知道厨房能做什么菜（工具清单），也负责把点单（调用请求）传到厨房并端菜回来（返回结果）。

一个 MCP Server 通常对应 **一类工具集**。比如：
- 文件系统 Server：提供 `read_file`、`write_file`、`list_directory`
- GitHub Server：提供 `get_issue`、`list_prs`、`read_file_content`
- 数据库 Server：提供 `query`、`list_tables`

### 3.2 MCP Client（宿主）

MCP Client 是 **消费工具的一方**，通常嵌入在你的产品进程里。它做三件事：

1. **连接 Server**：启动或连接到一个 MCP Server
2. **发现工具**：调用 `tools/list` 获取工具清单
3. **代理调用**：当 LLM 决定调用某个工具时，Client 把请求转发给 Server

在 Agent 开发中，Client 通常是这样运作的：

```
用户提问 → LLM 决定调用工具 → Client 找到对应的 MCP Server
→ 发送 tools/call → 拿回结果 → 塞回给 LLM 作为 observation
```

### 3.3 Transport：stdio vs HTTP

Transport 决定 Server 和 Client 之间**怎么通信**。MCP 定义了两种主要 Transport：

#### stdio（标准输入输出）

Client 把 MCP Server 作为 **子进程** 启动，通过 **stdin/stdout 管道** 通信：

```
Client 进程
  │
  ├── 启动子进程: python my_server.py
  │     stdin  ← Client 写入 JSON-RPC 请求
  │     stdout → Client 读取 JSON-RPC 响应
  │     stderr → 日志输出（不参与协议）
  │
  └── 子进程退出 = 连接断开
```

**优点**：
- 零网络配置，不需要端口、TLS、防火墙
- 低延迟（进程间管道通信）
- 部署简单：只需要 Python 脚本 + 依赖

**缺点**：
- 只能在本机使用
- 每个 Client 实例启动一个 Server 子进程，不共享

**适用场景**：本机开发工具（读文件、执行命令）、桌面 IDE 插件。

#### HTTP（Streamable HTTP / SSE）

Server 作为独立 HTTP 服务运行，Client 通过网络请求通信：

```
Client 进程 ──── HTTP/HTTPS ────→ MCP Server（独立服务）
                                    端口 8080
                                    可能在另一台机器上
```

**优点**：
- 可远程访问、可共享、可集群部署
- 多个 Client 共用一个 Server 实例
- 可以放在内网服务、Docker 容器、K8s 里

**缺点**：
- 需要处理 TLS、认证、超时、网络异常
- 延迟比 stdio 高（网络 RTT）
- 部署复杂度高

**适用场景**：共享的内部工具服务、多团队共用的 API 网关。

#### 两种 Transport 对比

| 维度 | stdio | HTTP |
|------|-------|------|
| 部署 | 随 Client 子进程启动 | 独立服务部署 |
| 延迟 | 微秒级（管道） | 毫秒级（网络） |
| 共享 | 每个 Client 一个实例 | 多 Client 共享 |
| 安全 | 继承宿主用户权限 | 需要 TLS + Token |
| 适用 | 本机工具、开发环境 | 远程服务、生产环境 |
| 调试 | 看 stderr | 看 HTTP 日志 |

### 3.4 stdio 的隐藏风险

很多人觉得 stdio 是「本地的，所以安全」。这是 **错觉**。

子进程会继承 **父进程的环境变量**。如果你的 Client 进程里有 `AWS_SECRET_ACCESS_KEY`、`DATABASE_URL` 这些环境变量，MCP Server 子进程也能读到它们。

一个恶意或者有 bug 的 MCP Server 可以：
- 读取你的 `~/.ssh/` 密钥
- 读取环境变量里的云服务凭证
- 在你的用户权限下执行任意命令

**防御**：
- 只运行你信任的 MCP Server（官方认证或自己写的）
- 用 `env` 参数限制子进程能看到的环境变量
- 在 Docker 容器或沙箱里运行 Server

**本节小结（三张卡片）**

- **一句话定义**：Transport 选错 = **把本地 shell 能力远程暴露** 或 **把远程 RTT 引进交互环**。
- **典型场景**：本机读 repo 用 stdio 零配置；跨机房查 CMDB 用 mTLS HTTP。
- **和 A1 类比**：stdio 像 **`subprocess.run` 管道**；HTTP 像 **`urllib.request` 跨机** —— 错误处理与超时模型完全不同。

## 4. MCP 消息格式与生命周期

**本节概念**：MCP 基于 **JSON-RPC 2.0** 协议通信；一次完整的 MCP 会话经历 **初始化 → 能力协商 → 工作 → 关闭** 四个阶段。

### 4.1 JSON-RPC 2.0 基础

JSON-RPC 是一个简单的远程过程调用协议。每条消息都是一个 JSON 对象，有三种类型：

**请求（Request）**：Client 发给 Server，要求执行某个方法

```json
{
  "jsonrpc": "2.0",
  "id": 1,
  "method": "tools/list",
  "params": {}
}
```

- `id`：请求编号，Server 回复时原样带回，用于匹配请求和响应
- `method`：要调用的方法名
- `params`：方法参数（可选）

**响应（Response）**：Server 返回结果

```json
{
  "jsonrpc": "2.0",
  "id": 1,
  "result": { "tools": [...] }
}
```

**错误响应**：

```json
{
  "jsonrpc": "2.0",
  "id": 1,
  "error": {
    "code": -32601,
    "message": "Method not found"
  }
}
```

**通知（Notification）**：不需要回复的单向消息（没有 `id` 字段）

```json
{
  "jsonrpc": "2.0",
  "method": "notifications/initialized"
}
```

### 4.2 MCP 会话生命周期

一次 MCP 会话分四个阶段：

```
Client                              Server
  │                                   │
  │ ─── initialize (请求) ──────────→ │   阶段1：初始化
  │ ←── initialize (响应) ──────────  │   Server 声明支持的能力
  │ ─── notifications/initialized ──→ │   Client 确认
  │                                   │
  │ ─── tools/list (请求) ──────────→ │   阶段2：发现
  │ ←── tools/list (响应) ──────────  │   返回工具清单
  │                                   │
  │ ─── tools/call (请求) ──────────→ │   阶段3：工作（可重复）
  │ ←── tools/call (响应) ──────────  │   返回执行结果
  │ ─── tools/call (请求) ──────────→ │
  │ ←── tools/call (响应) ──────────  │
  │                                   │
  │ ─── (断开连接 / 发送关闭) ──────→ │   阶段4：关闭
```

**阶段 1：初始化（Initialize）**

Client 告诉 Server：「我是谁，我支持协议的哪个版本」。Server 回复自己支持的能力（capabilities）：

```json
// Client → Server
{
  "jsonrpc": "2.0",
  "id": 1,
  "method": "initialize",
  "params": {
    "protocolVersion": "2024-11-05",
    "capabilities": {},
    "clientInfo": { "name": "my-agent", "version": "1.0" }
  }
}

// Server → Client
{
  "jsonrpc": "2.0",
  "id": 1,
  "result": {
    "protocolVersion": "2024-11-05",
    "capabilities": { "tools": {} },
    "serverInfo": { "name": "file-server", "version": "0.1" }
  }
}
```

**阶段 2：发现** — Client 调用 `tools/list` 获取工具清单（参见 2.1 节的例子）。

**阶段 3：工作** — Client 根据 LLM 的决策，调用 `tools/call` 执行具体工具。这个阶段可以重复多次。

**阶段 4：关闭** — stdio 模式下，Client 关闭子进程即可。HTTP 模式下，可以发送关闭通知或直接断开连接。

**本节小结（三张卡片）**

- **一句话定义**：MCP 会话 = **握手（初始化）→ 看菜单（发现）→ 点菜吃饭（工作）→ 结账走人（关闭）**。
- **典型场景**：Claude Desktop 启动时对每个配置的 MCP Server 执行一次初始化 + 发现，后续用户对话中按需调用。
- **和 A1 类比**：像 **数据库连接的生命周期** —— `connect → cursor → execute → close`，跳过任何一步都是 bug。

## 5. 动手写一个 MCP Server

**本节概念**：用官方 `mcp` Python 包可以快速创建一个 MCP Server；核心就是 **注册工具函数 + 启动 transport**。

### 5.1 安装

```bash
pip install mcp
```

### 5.2 最简 Server：文件读取工具

下面这个例子创建了一个只有一个工具的 MCP Server：读取文件内容。

```python
# file_server.py
"""
一个最简单的 MCP Server —— 只提供一个 read_file 工具。
用 stdio transport 运行，适合本地开发测试。
"""
from mcp.server import Server
from mcp.server.stdio import stdio_server
from mcp.types import Tool, TextContent
import json

# 第一步：创建 Server 实例
# name 参数是这个 Server 的标识，会在初始化握手时发送给 Client
app = Server("file-reader")


# 第二步：注册工具清单
# 当 Client 调用 tools/list 时，这个函数会被调用
@app.list_tools()
async def list_tools() -> list[Tool]:
    return [
        Tool(
            name="read_file",
            description="读取指定路径的文本文件内容（只读，不会修改文件）。"
                        "路径必须是绝对路径。",
            inputSchema={
                "type": "object",
                "properties": {
                    "path": {
                        "type": "string",
                        "description": "文件的绝对路径，如 /home/user/readme.md"
                    }
                },
                "required": ["path"]
            }
        )
    ]


# 第三步：注册工具执行逻辑
# 当 Client 调用 tools/call 时，这个函数会被调用
@app.call_tool()
async def call_tool(name: str, arguments: dict) -> list[TextContent]:
    if name != "read_file":
        raise ValueError(f"未知工具: {name}")

    path = arguments.get("path")
    if not path:
        raise ValueError("缺少 path 参数")

    # 安全检查：禁止读取敏感路径
    forbidden = ["/etc/shadow", "/etc/passwd"]
    if path in forbidden:
        return [TextContent(type="text", text=f"错误：禁止读取 {path}")]

    try:
        with open(path, "r", encoding="utf-8") as f:
            content = f.read()
        return [TextContent(type="text", text=content)]
    except FileNotFoundError:
        return [TextContent(type="text", text=f"错误：文件不存在 {path}")]
    except PermissionError:
        return [TextContent(type="text", text=f"错误：没有读取权限 {path}")]


# 第四步：启动 Server
# stdio_server() 让 Server 通过 stdin/stdout 通信
async def main():
    async with stdio_server() as (read_stream, write_stream):
        await app.run(
            read_stream, write_stream,
            app.create_initialization_options()
        )


if __name__ == "__main__":
    import asyncio
    asyncio.run(main())
```

**代码逐步解析**：

1. `Server("file-reader")`：创建一个 MCP Server 实例。`"file-reader"` 是 Server 名字，Client 在初始化时会看到它。

2. `@app.list_tools()`：装饰器注册「工具清单」处理函数。每当 Client 发 `tools/list` 请求，就调用这个函数。返回值是 `Tool` 对象的列表，每个 Tool 包含名字、描述、参数 schema。

3. `@app.call_tool()`：装饰器注册「工具执行」处理函数。每当 Client 发 `tools/call` 请求，就调用这个函数。参数 `name` 是工具名，`arguments` 是参数字典。你需要自己做 **名字分发**（if/elif 或字典映射）。

4. `stdio_server()`：启动 stdio transport。Server 从 stdin 读请求，往 stdout 写响应。

### 5.4 多工具的 Server

实际项目中一个 Server 通常提供多个相关工具。用字典做分发会更清晰：

```python
# 工具函数注册表（和 A1 里的注册表模式一样）
TOOL_HANDLERS = {}

def register(name):
    """装饰器：把函数注册到工具表中"""
    def decorator(func):
        TOOL_HANDLERS[name] = func
        return func
    return decorator

@register("read_file")
async def handle_read_file(arguments: dict) -> str:
    path = arguments["path"]
    with open(path, "r") as f:
        return f.read()

@register("list_directory")
async def handle_list_directory(arguments: dict) -> str:
    import os
    path = arguments["path"]
    entries = os.listdir(path)
    return "\n".join(entries)

@register("file_info")
async def handle_file_info(arguments: dict) -> str:
    import os
    path = arguments["path"]
    stat = os.stat(path)
    return f"大小: {stat.st_size} 字节, 修改时间: {stat.st_mtime}"


# 在 call_tool 里统一分发
@app.call_tool()
async def call_tool(name: str, arguments: dict) -> list[TextContent]:
    handler = TOOL_HANDLERS.get(name)
    if handler is None:
        raise ValueError(f"未知工具: {name}")
    result = await handler(arguments)
    return [TextContent(type="text", text=result)]
```

这就是 A1 里学过的 **注册表 + 装饰器模式** 在 MCP 中的实际应用。

**本节小结（三张卡片）**

- **一句话定义**：MCP Server = **`list_tools` 告诉 Client 我有什么** + **`call_tool` 执行具体业务逻辑**。
- **典型场景**：一个文件系统 Server 提供 read/write/list/info 四个工具，Claude Desktop 直接配置即可使用。
- **和 A1 类比**：工具注册表就是 A1 的 **`dict[str, Callable]` 注册表** —— 只是从进程内升级为跨进程。

## 6. 与「手写 OpenAPI 网关」的对比

**本节概念**：**OpenAPI** 描述 **业务域 REST API**（面向外部客户或前端）；**MCP** 描述 **Agent 可调用的工具面**（面向 LLM）；二者目标不同，可以并存。

很多人会问：「我已经有 REST API 了，为什么还要 MCP？」

答案是 **它们解决不同层的问题**：

| 维度 | MCP | REST API / OpenAPI |
|------|-----|---------------------|
| **目标用户** | LLM Agent（机器） | 前端/移动端/第三方（人或机器） |
| **设计重心** | 工具发现 + 调用 + Agent 上下文 | 业务资源的 CRUD |
| **描述对象** | 「Agent 能做什么」 | 「业务系统暴露什么」 |
| **传输** | stdio / HTTP | 纯 HTTP |
| **生态** | Claude/Cursor/Agent 框架 | 前端框架/Swagger/Postman |

**实际架构中两者并存**：

```
用户请求 → Agent (MCP Client)
               │
               ├── MCP Server (文件系统) ← 本地 stdio
               ├── MCP Server (知识库) ← 内网 HTTP
               └── 直接调 REST API (订单服务) ← 传统 HTTP 调用
```

经验法则：
- **已有稳定 REST API 的服务**（订单、支付、用户）→ 直接调 REST，不用包一层 MCP
- **需要跨多个 Agent 产品复用的工具**（读文件、查 Git、搜文档）→ 做成 MCP Server
- **本地开发工具**（执行命令、读日志）→ MCP stdio 最方便

**本节小结（三张卡片）**

- **一句话定义**：MCP 不是 REST 的替代品，是 **工具集成层**，解决的是 Agent 生态问题。
- **典型场景**：订单创建仍走内部 gRPC/REST；只有「读文档摘要」走 MCP。
- **和 A1 类比**：像 **`pathlib` 与 `os.path` 并存** —— 场景不同，不互斥。

## 7. 安全模型

**本节概念**：工具 = **任意代码执行的前门**；安全的默认姿态是 **拒绝一切，再白名单放行**。

### 7.1 威胁清单

MCP Server 本质是在你的机器上执行代码，所以安全问题非常实际：

| 威胁 | 说明 | 例子 |
|------|------|------|
| **路径穿越** | 工具参数里带 `../../etc/passwd` | 读文件工具被利用读取系统敏感文件 |
| **命令注入** | 参数被拼接进 shell 命令 | `filename: "test; rm -rf /"` |
| **SSRF** | Server 发起内网请求 | 参数里带 `http://169.254.169.254` 读取云元数据 |
| **凭证泄露** | Server 读取环境变量中的密钥 | `os.environ["AWS_SECRET_KEY"]` |
| **数据泄露** | 工具返回了不该返回的数据 | 查询时没有租户隔离 |

### 7.2 防御要点

**路径穿越防御**：

```python
import os

def safe_read_file(path: str, allowed_dir: str = "/home/user/docs") -> str:
    # 解析真实路径，消除 .. 和符号链接
    real_path = os.path.realpath(path)
    # 检查是否在允许的目录下
    if not real_path.startswith(os.path.realpath(allowed_dir)):
        raise PermissionError(f"路径 {path} 不在允许范围内")
    with open(real_path, "r") as f:
        return f.read()
```

**命令注入防御**：

```python
import subprocess

# ❌ 错误：字符串拼接，容易注入
result = subprocess.run(f"ls {user_input}", shell=True)

# ✅ 正确：用列表传参，shell=False
result = subprocess.run(["ls", user_input], shell=False)
```

**环境变量隔离**（启动 MCP Server 子进程时）：

```python
import subprocess

# 只传递必要的环境变量
safe_env = {
    "PATH": "/usr/bin:/usr/local/bin",
    "HOME": "/tmp/sandbox",
    # 不传 AWS_*, DATABASE_*, 等敏感变量
}
subprocess.Popen(["python", "server.py"], env=safe_env)
```

**多租户隔离**：每次工具调用都应该带 **租户标识**，查询时强制 WHERE 条件：

```python
async def query_orders(tenant_id: str, order_id: str) -> str:
    # 强制带 tenant_id，即使模型没传
    sql = "SELECT * FROM orders WHERE tenant_id = %s AND order_id = %s"
    result = await db.execute(sql, (tenant_id, order_id))
    return json.dumps(result)
```

### 7.3 安全原则总结

1. **默认拒绝**：Server 只暴露明确注册的工具，没注册的一律拒绝
2. **最小权限**：Server 运行在最低权限用户下，只能访问必要的资源
3. **参数二次校验**：即使 JSON Schema 校验通过，函数内部仍要校验（路径、SQL 参数等）
4. **审计日志**：每次工具调用记录 **谁、何时、什么参数、结果摘要**
5. **不信任模型**：模型可能被 prompt injection 攻击，传来恶意参数

**本节小结（三张卡片）**

- **一句话定义**：MCP 安全的核心是 **「不信任模型，也不信任 Server 实现」**。
- **典型场景**：工具参数里带 URL，Server 内网 SSRF 扫遍 `169.254.169.254` 获取云凭证。
- **和 A1 类比**：像 **`eval` 禁用 + `ast.literal_eval` 白名单** —— 能力面必须显式列出。

## 8. 从 ToolRegistry 到 MCP 的直觉映射

**本节概念**：`ToolRegistry` 的 **`list_openai_tools` + `call`**，在概念上一一对应 MCP 的 **列举工具 + 调用**；`02_stdio_tool_host.py` 演示 **进程边界** 上的 JSON 命令。

读 demo 时抓住三条映射关系：

| ToolRegistry（C4/demos） | MCP |
|--------------------------|-----|
| `registry.list_openai_tools()` | `tools/list` 请求 |
| `registry.call(name, **kwargs)` | `tools/call` 请求 |
| 函数返回值 | `result.content` |
| Python dict 注册表 | MCP Server 进程 |
| 同一进程内调用 | 跨进程（stdio/HTTP）调用 |

理解这个映射后，你会发现 MCP 在概念上 **没有新东西**，只是把进程内的工具注册表 **序列化到了网络上**。复杂度增加的部分都在 **传输层**（序列化、错误处理、重连、并发）。

正式开发请用官方 **`mcp` Python 包**，不要手写 JSON-RPC 解析。教学 demo 只是帮你建立心智模型，手写协议上线会踩无数坑（消息边界、编码、错误处理）。

**本节小结（三张卡片）**

- **一句话定义**：教学 demo 给你 **心智模型**；生产用官方 SDK 补齐 **协议完整性**。
- **典型场景**：把现有 `dict[str, Callable]` 工具层渐进迁移为 MCP Server，宿主侧逐步换 transport。
- **和 A1 类比**：手写 JSON 行协议像 **手写 HTTP 解析** —— 学习可以，上线不要。

## 9. 调试 MCP Server

**本节概念**：MCP Server 的 bug 分两类：**协议层**（消息格式错、生命周期错）和 **业务层**（工具逻辑错）；排查时先定位是哪一层。

### 9.1 常见错误

| 现象 | 可能原因 | 排查方法 |
|------|----------|----------|
| Client 连不上 Server | Server 未启动、路径错误、Python 环境不对 | 手动运行 `python server.py` 看报错 |
| `tools/list` 返回空 | `@app.list_tools()` 没有正确注册 | 检查装饰器是否生效 |
| 工具调用超时 | 工具函数里有阻塞操作（同步 I/O） | 用 `asyncio` 包装阻塞操作 |
| 参数解析失败 | `inputSchema` 与实际参数不匹配 | 打印 `arguments` 看实际收到什么 |
| stderr 输出干扰 | `print()` 默认输出到 stdout | MCP 用 stdout 通信，`print` 会破坏协议 |

### 9.2 最重要的一条：不要用 print()

在 stdio 模式下，**stdout 是协议通道**。如果你在 Server 代码里写了 `print("debug info")`，这段文字会被 Client 当成 JSON-RPC 消息解析，然后报错。

**正确做法**：所有调试输出写到 **stderr**：

```python
import sys

def debug(msg: str):
    print(msg, file=sys.stderr)  # 写到 stderr，不干扰协议

# 或者用 logging
import logging
logging.basicConfig(stream=sys.stderr, level=logging.DEBUG)
logger = logging.getLogger("my-server")
logger.debug("这条日志不会干扰 MCP 协议")
```

### 9.3 用 MCP Inspector 调试

MCP 官方提供了一个 Inspector 工具，可以可视化地测试 Server：

```bash
npx @modelcontextprotocol/inspector python server.py
```

它会启动一个 Web 界面，让你手动发送 `tools/list` 和 `tools/call` 请求，查看响应。这比直接在 Claude Desktop 里测试快得多——出问题时你能看到原始的 JSON 消息。

**本节小结（三张卡片）**

- **一句话定义**：调试 MCP = **先排协议层（消息格式）再排业务层（工具逻辑）**。
- **典型场景**：Server 里写了 `print()`，stdout 输出破坏 JSON-RPC 帧，Client 报 parse error。
- **和 A1 类比**：像调试 **HTTP API** —— 先用 curl 确认接口通了，再看业务逻辑。

## 10. 常见误区与踩坑

1. **「MCP 替代了 REST API」**—— 不对。MCP 是 Agent 工具层协议，REST 是业务 API 协议，两者并存。你不会用 MCP 给前端提供登录接口。

2. **「stdio 是安全的因为是本地」**—— 不对。子进程继承环境变量和文件权限，恶意 Server 可以读取你的密钥。

3. **「工具越多越好」**—— 不对。工具描述占用上下文 token，20 个工具的描述可能就占了几千 token。而且工具越多，模型选错工具的概率越高。建议按场景 **动态加载** 工具子集。

4. **「MCP Server 要处理并发」**—— stdio 模式下通常是单 Client 串行请求，不需要并发。HTTP 模式下才需要考虑。

5. **「直接用 `print()` 调试」**—— 在 stdio 模式下这会破坏协议。用 `stderr` 或 `logging`。

6. **「把密钥写在 Server 代码里」**—— 用环境变量或配置文件，代码里不应该有任何硬编码的密钥。

## 13. 面试高频考点

**Q1：MCP 和 OpenAI 的 Function Calling 有什么区别？**

Function Calling 是 **模型侧的能力**——模型根据工具描述决定调用哪个工具、传什么参数。MCP 是 **工具侧的协议**——定义了工具如何被发现、调用、版本化。两者是互补关系：MCP Server 提供工具清单 → 清单转成 OpenAI `tools` 格式喂给模型 → 模型决策后通过 MCP Client 调用 Server。

**Q2：MCP 的 Transport 有哪些？各自的安全考虑是什么？**

两种主要 Transport：
- **stdio**：本地子进程管道，低延迟零配置，但子进程继承父进程环境变量和文件权限，需要最小权限启动
- **HTTP**（Streamable HTTP / SSE）：远程服务，需要 TLS 加密、Token 认证、超时设置、防 SSRF

**Q3：如果你要给一个 Agent 系统设计工具层，你会选择 MCP 还是直接用 REST API？**

取决于场景。如果工具需要被多个 Agent 产品复用（IDE、CLI、Web），用 MCP 避免重复适配。如果只是调用公司内部已有的 REST 微服务（订单、支付），直接调 REST 更简单。两者可以并存。

**Q4：MCP Server 的安全最佳实践有哪些？**

1. 默认拒绝，白名单注册工具
2. 参数二次校验（路径穿越、SQL 注入）
3. 最小权限运行（限制环境变量、文件访问）
4. 审计日志（谁、何时、什么参数）
5. 不信任模型输入（模型可能被 prompt injection）

**Q5：描述一下你会怎么把一个现有的 Python 工具库迁移为 MCP Server。**

1. 把现有函数整理成 `name → callable` 的注册表
2. 为每个函数写 JSON Schema（参数类型、描述、必填项）
3. 用 `mcp` Python 包创建 Server，注册 `list_tools` 和 `call_tool`
4. 先用 stdio transport 本地测试
5. 用 MCP Inspector 验证协议正确性
6. 配置到 Claude Desktop 或自研 Agent 里实测
