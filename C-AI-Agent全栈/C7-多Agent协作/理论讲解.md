## 0. 与 A1、C6 的衔接

**本节概念**：**多 Agent** 不是「多开几个 chat」，而是 **多个独立 system / 工具白名单 / 观测边界** 的编排问题；复杂度近似 **随 Agent 数线性上升**。

单 Agent（C6）已经让你处理 **循环、记忆、停止**；多 Agent 在此基础上增加 **路由与合并**。A1 里你学过 **协作式多任务与锁**；这里共享状态冲突是同一类 bug，只是错误表现变成「两个子 Agent 各说各话」。

**本节小结（三张卡片）**

- **一句话定义**：多 Agent = **分治 + 编排**，不是模型变多就一定更聪明。
- **典型场景**：「只读客服」与「写库运维」必须工具面隔离，拆 Agent 比单提示里角色切换稳。
- **和 A1 类比**：像 **多进程各跑一个 interpreter**，共享要通过 **明确 IPC（消息总线/黑板）**。

## 1. Why：单 Agent 何时不够

**本节概念**：拆 Agent 的正当理由是 **隔离（安全/上下文）** 与 **并行（吞吐）**；不正当理由是 **架构焦虑**。

- **上下文隔离**：不同子任务需要 **不同 system 提示与工具白名单**（如「只读客服」vs「写库运维」）。  
- **专长模拟**：提示词分离比单提示里「你既是律师又是医生」更稳定。  
- **并行**：多检索源、多格式生成可 **同时**跑再合并。

**代价**：延迟、费用、编排复杂度、调试成本 **近似线性上升**。若只为「好看架构图」而拆，通常不值得。

**本节小结（三张卡片）**

- **一句话定义**：拆 Agent 买的是 **边界清晰**，不是免费智力加成。
- **典型场景**：同一用户会话里，一半问题 FAQ 可查，一半要工单系统；路由到不同子 Agent。
- **和 A1 类比**：像 **拆模块避免循环依赖** —— 拆完接口要比类内部方法更稳定。

## 2. 常见拓扑

**本节概念**：拓扑决定 **错误如何传播**；Supervisor 错一次全局错，Pipeline 前级偏一点后面放大。

| 拓扑 | 描述 | 风险 |
|------|------|------|
| **Supervisor / Router** | 上层分类后分发给子 Agent | 路由错则全局错 |
| **Pipeline** | 阶段 A→B→C 固定 | 前级误差放大 |
| **Peer** | 多 Agent 讨论/投票 | 难终止、成本高 |

**本节小结（三张卡片）**

- **一句话定义**：选拓扑 = 选 **失败模式**，不是选 logo。
- **典型场景**：翻译→摘要→合规检查，天然 Pipeline；开放域问答更像 Supervisor。
- **和 A1 类比**：Pipeline 像 **Unix 管道**，Supervisor 像 **`if/elif` 分发到策略类**。

## 3. 路由：规则、小模型、Embedding

**本节概念**：**路由**是把 query 映射到 **子 Agent 或工具子集** 的函数；要快、可解释、可离线评测。

- **规则**：关键词、正则、业务标签（快、可解释）。  
- **小分类模型**：成本低于主模型，需 **标注数据**。  
- **Embedding 路由**：query 与「意图描述」算相似度（需维护描述向量）。

**混合策略**最常见：**规则兜底 + 小模型处理长尾**。全 Embedding 路由要小心 **描述漂移**：意图描述陈旧会导致系统性错分。

**本节小结（三张卡片）**

- **一句话定义**：路由是 **可测试的分类器**，不是 prompt 里一句「请判断」。
- **典型场景**：含「退款」关键词直送财务子 Agent，其余走通用。
- **和 A1 类比**：像 **`match/case` 与策略 dict 组合** —— 先 O(1) 命中高频，再复杂模型吃长尾。

## 4. 共享状态与冲突

**本节概念**：多 Agent 写同一 **共享可变状态** 时，要有 **版本、锁或事务**；否则调试会地狱级。

- **黑板模式**：共享 dict/文档，需 **版本或锁**。  
- **消息总线**：事件解耦，排障要 **完整 trace**（C8）。  
- **禁止**：多 Agent **同时写**同一业务行而无事务。

**本节小结（三张卡片）**

- **一句话定义**：共享状态要么 **不可变追加**，要么 **强一致写路径**。
- **典型场景**：两个子 Agent 同时改「用户地址」，后者覆盖前者且无通知。
- **和 A1 类比**：像 **多线程写全局 list 不加锁** —— 竞态只是换了个叙事包装。

## 5. 并行：Fan-out / Fan-in

**本节概念**：**Fan-out** 提吞吐；**Fan-in** 要做 **去重、冲突消解、优先级**；否则合并阶段比串行更慢。

- **Fan-out**：多子任务 `asyncio.gather` 或线程池（阻塞 IO）。  
- **Fan-in**：合并前做 **去重、冲突消解、优先级**。  
- **超时**：子任务分别 **deadline**，避免一个拖死全局。

**本节小结（三张卡片）**

- **一句话定义**：并行省时间的前提是 **合并逻辑有界**。
- **典型场景**：三源检索各返回 10 条，合并成 15 条要去重与 rerank。
- **和 A1 类比**：`gather` 像 **批量启动协程** —— 仍要处理 `return_exceptions` 与超时。

## 6. 调试与评测

**本节概念**：多 Agent 的 bug **一半在路由**，一半在子 Agent；指标要 **分项**，只看端到端会误判。

- **统一 trace_id** 贯穿路由与各子 Agent。  
- **黄金用例**：覆盖「路由正确 / 路由边界 / 子 Agent 失败」。  
- **指标**：端到端成功率 + **路由准确率** 分项。

**本节小结（三张卡片）**

- **一句话定义**：没有 trace 的多 Agent 排障 ≈ **分布式系统盲调**。
- **典型场景**：路由准确率 95% 但端到端 60%，说明子 Agent 工具失败未被观测。
- **和 A1 类比**：像 **分层日志**：先定位 import 失败还是运行时错。
