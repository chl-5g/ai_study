## 0. 与 A1、C6 的衔接

**本节概念**：**多 Agent** 不是「多开几个 chat」，而是 **多个独立 system / 工具白名单 / 观测边界** 的编排问题；复杂度近似 **随 Agent 数线性上升**。

### 从单 Agent 到多 Agent

C6 教你建了一个 Agent——一个带工具的循环。它能查天气、取消会议、查订单。但如果你的系统要同时处理这些场景：

- 客服咨询（只能查询，不能修改任何数据）
- 运维操作（可以重启服务、修改配置，但不能看客户数据）
- 数据分析（可以执行 SQL 查询，但只能 SELECT，不能 DELETE）

你可能会想：「一个 Agent 带所有工具不行吗？」

**确实可以，但有三个问题**：

1. **安全隔离**：客服 Agent 的工具白名单里不应该有 `restart_service`。如果用一个 Agent，所有工具都暴露给模型，它可能在回答客户问题时 accidentally 调用运维工具。
2. **上下文污染**：客服和运维的 system prompt 完全不同。混在一起，模型的角色认知会模糊——有时用客服语气说技术细节，有时用运维语气回复客户。
3. **可维护性**：一个 Agent 的 system prompt 越长、工具越多，调试就越难。哪个工具被误调了？哪段 prompt 导致了偏差？混在一起很难定位。

多 Agent 的本质就是 **分治**：把一个大问题拆成多个小问题，每个小问题交给一个专门的 Agent。

### 类比

如果单 Agent 像一个 **全栈工程师**（什么都做），多 Agent 像一个 **技术团队**（前端、后端、运维各司其职）。全栈工程师在小项目里效率最高，但大项目必须分工——因为一个人的认知负荷有上限。

同样，一个 Agent 的 system prompt + 工具列表有上限（上下文窗口）。当业务复杂到需要 20+ 工具、5+ 角色时，**拆分比堆砌更可靠**。

但要注意：A1 里学过 **「多进程间共享状态需要明确的 IPC 机制」**。多 Agent 也一样——共享状态必须通过 **明确的消息总线或黑板**，不能隐式依赖。

**本节小结（三张卡片）**

- **一句话定义**：多 Agent = **分治 + 编排**，不是模型变多就一定更聪明。
- **典型场景**：「只读客服」与「写库运维」必须工具面隔离，拆 Agent 比单提示里角色切换稳。
- **和 A1 类比**：像 **多进程各跑一个 interpreter**，共享要通过 **明确 IPC（消息总线/黑板）**。

## 1. Why：单 Agent 何时不够

**本节概念**：拆 Agent 的正当理由是 **隔离（安全/上下文）** 与 **并行（吞吐）**；不正当理由是 **架构焦虑**。

### 1.1 拆 Agent 的三个正当理由

**理由一：安全隔离（最重要的理由）**

不同子任务需要 **不同的工具白名单**。客服 Agent 只能调 `get_order_status`、`get_faq`，不能调 `delete_user`、`restart_service`。

如果用一个 Agent + 所有工具，模型有可能在错误的上下文中调用危险工具。即使你在 system prompt 里说「不要调用运维工具」，模型 **不保证** 遵守——特别是在复杂对话中。

**拆成多个 Agent 后**，每个 Agent 的工具列表物理上就不包含它不该用的工具。不是靠 prompt 约束，而是靠 **代码层面的白名单**。

**理由二：专长模拟**

一个 Agent 同时扮演「法律顾问」和「医疗助手」，效果比两个专门的 Agent 差。原因是 system prompt 越长越复杂，模型的角色一致性越差。

分开后，法律 Agent 的 system prompt 专注于法律领域（引用法条、注意司法管辖区），医疗 Agent 专注于医疗领域（注意免责声明、建议就医）。两个 prompt 各自简短清晰，角色一致性更好。

**理由三：并行加速**

用户要求「同时搜索三个数据源，综合分析」。串行跑三次搜索需要 3 × T 时间，并行跑三个 Agent（或三个工具调用）只需要 max(T₁, T₂, T₃)。

```python
import asyncio

async def parallel_search(query: str) -> list[str]:
    """三个搜索任务并行执行，总耗时 ≈ 最慢的那个"""
    results = await asyncio.gather(
        search_agent_web(query),
        search_agent_docs(query),
        search_agent_db(query),
    )
    return merge_results(results)
```

### 1.2 不拆 Agent 的理由

**代价一：延迟**。每多一个 Agent，至少多一次 LLM 调用。路由 Agent → 子 Agent → 汇总 = 至少 3 次调用。

**代价二：费用**。更多 LLM 调用 = 更多 token = 更多钱。

**代价三：编排复杂度**。Agent 间的消息传递、状态同步、错误传播，全部需要你手动处理。

**代价四：调试难度**。出错时，你需要回答：是路由错了？还是子 Agent 错了？还是合并逻辑错了？链路越长，定位越难。

**判断标准**：如果一个任务用单 Agent + 5 个工具就能搞定，就不要拆成 3 个 Agent。**为「好看的架构图」而拆多 Agent，是 Agent 领域最常见的过度工程**。

### 读到这里你应该理解

- 拆 Agent 的三个正当理由：安全隔离、专长模拟、并行加速
- 拆 Agent 的代价：延迟、费用、编排复杂度、调试难度
- 判断标准：单 Agent + 少量工具能搞定的，就不要拆

**本节小结（三张卡片）**

- **一句话定义**：拆 Agent 买的是 **边界清晰**，不是免费智力加成。
- **典型场景**：同一用户会话里，一半问题 FAQ 可查，一半要工单系统；路由到不同子 Agent。
- **和 A1 类比**：像 **拆模块避免循环依赖** —— 拆完接口要比类内部方法更稳定。

## 2. 常见拓扑

**本节概念**：拓扑决定 **消息怎么流动** 和 **错误怎么传播**。选拓扑不是选 logo，是选 **失败模式**。

### 2.1 Supervisor（监督者/路由器）

最常见的拓扑。一个「上级」Agent 负责理解用户意图，然后把任务分发给专门的「下级」Agent。

```
用户 → Supervisor → ┬→ 客服 Agent → 回复
                     ├→ 运维 Agent → 回复
                     └→ 分析 Agent → 回复
```

**实现**：

```python
class SupervisorAgent:
    """
    监督者 Agent：理解用户意图，分发给对应的子 Agent。
    
    Supervisor 自己不执行业务逻辑，只做路由决策。
    """
    
    def __init__(self):
        # 子 Agent 注册表
        self.sub_agents = {
            "customer_service": CustomerServiceAgent(),
            "ops": OpsAgent(),
            "analytics": AnalyticsAgent(),
        }
        
        # 路由工具：让 LLM 选择哪个子 Agent
        self.route_tool = {
            "type": "function",
            "function": {
                "name": "route_to_agent",
                "description": "根据用户意图选择合适的子Agent处理",
                "parameters": {
                    "type": "object",
                    "properties": {
                        "agent_name": {
                            "type": "string",
                            "enum": list(self.sub_agents.keys()),
                            "description": "目标Agent名称",
                        },
                        "task_summary": {
                            "type": "string",
                            "description": "传给子Agent的任务摘要",
                        },
                    },
                    "required": ["agent_name", "task_summary"],
                },
            },
        }
    
    def handle(self, user_message: str) -> str:
        """处理用户消息：路由到对应的子 Agent"""
        
        # Step 1: 让 Supervisor 决定路由
        messages = [
            {
                "role": "system",
                "content": (
                    "你是一个任务分发器。根据用户的问题，选择最合适的Agent来处理。\n"
                    "- customer_service: 处理订单查询、退换货、FAQ\n"
                    "- ops: 处理服务器运维、部署、监控\n"
                    "- analytics: 处理数据分析、报表、SQL查询"
                ),
            },
            {"role": "user", "content": user_message},
        ]
        
        response = call_llm(messages, tools=[self.route_tool])
        assistant_msg = response.choices[0].message
        
        if not assistant_msg.tool_calls:
            return assistant_msg.content  # Supervisor 直接回答了
        
        # Step 2: 提取路由决策
        tc = assistant_msg.tool_calls[0]
        args = json.loads(tc.function.arguments)
        agent_name = args["agent_name"]
        task_summary = args["task_summary"]
        
        # Step 3: 分发给子 Agent
        if agent_name not in self.sub_agents:
            return f"未知的 Agent: {agent_name}"
        
        sub_agent = self.sub_agents[agent_name]
        print(f"📨 路由到 {agent_name}: {task_summary}")
        
        result = sub_agent.handle(task_summary)
        return result
```

**Supervisor 的风险**：路由错了，整个请求都错了。如果 Supervisor 把一个退款问题发给了运维 Agent，运维 Agent 不知道怎么处理退款——它会乱来或者返回无意义的结果。

**缓解**：路由准确率是 Supervisor 模式的 **核心指标**，要单独评测（见第 6 节）。

### 2.2 Pipeline（流水线）

任务按固定顺序经过多个 Agent，每个 Agent 处理一个阶段。

```
用户输入 → 翻译 Agent → 摘要 Agent → 合规检查 Agent → 最终输出
```

**实现**：

```python
class PipelineOrchestrator:
    """
    流水线编排器：任务按顺序经过多个 Agent。
    
    每个 Agent 的输出是下一个 Agent 的输入。
    """
    
    def __init__(self, stages: list):
        """
        stages: 有序的 Agent 列表，每个 Agent 有 .handle(input) -> output 方法
        """
        self.stages = stages
    
    def run(self, initial_input: str) -> str:
        """执行流水线"""
        current = initial_input
        
        for i, agent in enumerate(self.stages):
            stage_name = agent.__class__.__name__
            print(f"▶ Stage {i+1}/{len(self.stages)}: {stage_name}")
            
            try:
                current = agent.handle(current)
                print(f"  ✓ 输出长度: {len(current)} 字符")
            except Exception as e:
                print(f"  ✗ Stage {stage_name} 失败: {e}")
                # Pipeline 的关键决策：某一步失败，是跳过还是终止？
                # 这里选择终止，因为后续步骤依赖前序输出
                return f"Pipeline 在 {stage_name} 阶段失败: {e}"
        
        return current


# 使用示例
pipeline = PipelineOrchestrator([
    TranslateAgent(target_lang="zh"),
    SummarizeAgent(max_words=200),
    ComplianceCheckAgent(),
])

result = pipeline.run("This is a long English document about financial regulations...")
```

**Pipeline 的风险**：**误差放大**。翻译 Agent 翻错了一个关键术语，摘要 Agent 会基于错误翻译来摘要，合规检查 Agent 检查的是错误的内容。每一步的小错误在下一步会被放大。

**缓解**：每一步的输出都应该可检查（C8 可观测性）。关键步骤可以加 **质量门禁**——如果置信度低于阈值就中止流水线。

### 2.3 Peer（对等/讨论）

多个 Agent 以平等身份讨论同一个问题，然后投票或协商出结论。

```
         ┌→ Agent A 观点 →┐
用户问题 →├→ Agent B 观点 →├→ 合并/投票 → 最终答案
         └→ Agent C 观点 →┘
```

**适用场景**：需要 **多角度分析** 的决策问题。比如评估一个商业方案，让「乐观分析师」和「风险分析师」各自分析，最后综合。

**风险**：
- 成本高：3 个 Agent = 3× LLM 调用
- 难终止：如果 Agent 之间「讨论」起来（A 反驳 B，B 反驳 A），可能永远达不成共识
- 合并困难：3 个矛盾的观点怎么综合？需要额外的裁判逻辑

**实践建议**：除非业务明确需要多角度分析，否则不推荐。大多数场景 Supervisor 就够了。

### 2.4 拓扑对比总结

| 拓扑 | 适用场景 | 核心风险 | 调试难度 |
|------|----------|----------|----------|
| **Supervisor** | 多类型任务的统一入口 | 路由错误 | 中 |
| **Pipeline** | 固定顺序的多阶段处理 | 误差放大 | 低（线性追踪） |
| **Peer** | 多角度分析、需要辩论 | 难终止、成本高 | 高 |

### 读到这里你应该理解

- 三种拓扑各有适用场景和风险
- Supervisor 最常用，核心是路由准确率
- Pipeline 最简单但要防误差放大
- Peer 最贵最复杂，慎用

**本节小结（三张卡片）**

- **一句话定义**：选拓扑 = 选 **失败模式**，不是选 logo。
- **典型场景**：翻译→摘要→合规检查，天然 Pipeline；开放域问答更像 Supervisor。
- **和 A1 类比**：Pipeline 像 **Unix 管道**，Supervisor 像 **`if/elif` 分发到策略类**。

## 3. 路由：规则、小模型、Embedding

**本节概念**：**路由**是把用户输入映射到 **目标子 Agent** 的函数。它是 Supervisor 模式的核心组件。路由的要求是：**快、准、可解释、可离线评测**。

### 3.1 规则路由（最简单、最可靠）

用关键词、正则表达式或业务标签做路由。

```python
import re


def rule_based_router(user_message: str) -> str:
    """
    基于规则的路由器。
    
    为什么规则路由仍然有价值？
    1. 速度极快（微秒级），不需要 LLM 调用
    2. 100% 可解释：能精确说出「因为包含关键词X所以路由到Y」
    3. 可离线评测：构造测试用例，跑一遍就知道准确率
    4. 不受 LLM 抽风影响
    """
    message_lower = user_message.lower()
    
    # 高优先级规则：精确匹配
    if any(kw in message_lower for kw in ["退款", "退货", "换货", "投诉"]):
        return "customer_service"
    
    if any(kw in message_lower for kw in ["重启", "部署", "服务器", "CPU", "内存告警"]):
        return "ops"
    
    if any(kw in message_lower for kw in ["报表", "统计", "分析", "SQL", "数据"]):
        return "analytics"
    
    # 兜底：发给通用 Agent
    return "general"


# 更灵活的版本：用配置驱动
ROUTING_RULES = [
    {
        "target": "customer_service",
        "keywords": ["退款", "退货", "换货", "投诉", "订单", "物流"],
        "patterns": [r"订单号?\s*[A-Z0-9-]+"],  # 正则匹配订单号格式
    },
    {
        "target": "ops",
        "keywords": ["重启", "部署", "服务器", "告警", "日志", "nginx"],
        "patterns": [r"\d+\.\d+\.\d+\.\d+"],  # IP 地址
    },
    {
        "target": "analytics",
        "keywords": ["报表", "统计", "分析", "同比", "环比", "趋势"],
        "patterns": [r"(SELECT|FROM|WHERE)", r"近\d+(天|月|年)"],
    },
]


def configurable_router(user_message: str) -> str:
    """配置驱动的路由器：规则可在运行时修改，不改代码"""
    for rule in ROUTING_RULES:
        # 关键词匹配
        if any(kw in user_message for kw in rule["keywords"]):
            return rule["target"]
        # 正则匹配
        if any(re.search(p, user_message) for p in rule.get("patterns", [])):
            return rule["target"]
    return "general"
```

**规则路由的局限**：无法处理长尾问题和语义模糊的输入。比如「这个服务怎么这么慢」——是运维问题（服务性能）还是客服问题（用户抱怨体验）？

### 3.2 小模型分类路由

用一个轻量级的分类模型来做路由。它比规则更灵活，比大模型更便宜。

```python
from sklearn.feature_extraction.text import TfidfVectorizer
from sklearn.linear_model import LogisticRegression
import pickle


class MLRouter:
    """
    基于 ML 的路由器。
    
    训练流程：
    1. 收集历史对话，人工标注每条属于哪个子 Agent
    2. 训练一个文本分类模型（TF-IDF + LR 就够了）
    3. 在线推理时用训练好的模型分类
    
    优点：处理模糊语义的能力远好于规则
    缺点：需要标注数据，需要定期重训
    """
    
    def __init__(self, model_path: str = "router_model.pkl"):
        with open(model_path, "rb") as f:
            saved = pickle.load(f)
        self.vectorizer: TfidfVectorizer = saved["vectorizer"]
        self.classifier: LogisticRegression = saved["classifier"]
        self.label_map: dict[int, str] = saved["label_map"]
    
    def route(self, user_message: str) -> tuple[str, float]:
        """
        返回 (目标Agent名, 置信度)。
        
        为什么要返回置信度？
        如果置信度低于阈值，说明模型不确定——这时应该走兜底（比如让 Supervisor LLM 做二次判断）。
        """
        features = self.vectorizer.transform([user_message])
        proba = self.classifier.predict_proba(features)[0]
        predicted_idx = proba.argmax()
        confidence = proba[predicted_idx]
        
        return self.label_map[predicted_idx], confidence


# 使用示例
router = MLRouter("router_model.pkl")
target, confidence = router.route("这个月的销售额是多少？")
# target="analytics", confidence=0.92

if confidence < 0.7:
    # 置信度低，走 LLM 路由兜底
    target = llm_route(user_message)
```

### 3.3 Embedding 路由

用向量相似度做路由。预先为每个子 Agent 写一段「意图描述」，算好向量。用户输入来了，算向量后和所有意图描述比相似度。

```python
import numpy as np


class EmbeddingRouter:
    """
    基于向量相似度的路由器。
    
    思路：每个子 Agent 有一段"意图描述"，
    用户输入和意图描述算余弦相似度，选最相似的。
    """
    
    def __init__(self):
        # 意图描述 + 预计算好的向量
        self.intents = {
            "customer_service": {
                "description": "处理客户的订单查询、退换货申请、物流追踪、售后投诉等问题",
                "vector": None,  # 初始化时计算
            },
            "ops": {
                "description": "处理服务器运维、应用部署、性能监控、告警处理、日志分析",
                "vector": None,
            },
            "analytics": {
                "description": "处理数据分析请求、生成统计报表、执行SQL查询、趋势分析",
                "vector": None,
            },
        }
        
        # 初始化：计算意图描述的向量
        for name, info in self.intents.items():
            info["vector"] = get_embedding(info["description"])
    
    def route(self, user_message: str) -> tuple[str, float]:
        """返回 (目标Agent名, 相似度分数)"""
        query_vec = get_embedding(user_message)
        
        best_name = None
        best_score = -1.0
        
        for name, info in self.intents.items():
            score = cosine_similarity(query_vec, info["vector"])
            if score > best_score:
                best_score = score
                best_name = name
        
        return best_name, best_score


def cosine_similarity(a: np.ndarray, b: np.ndarray) -> float:
    """余弦相似度：两个向量的夹角余弦值，越接近1越相似"""
    return float(np.dot(a, b) / (np.linalg.norm(a) * np.linalg.norm(b)))
```

**Embedding 路由的坑——描述漂移**：如果你的意图描述写于半年前，但业务已经变了（比如新增了「直播运营」相关功能），旧的描述不包含「直播」「弹幕」「礼物」这些关键词，路由就会系统性地错分。**意图描述必须和业务同步更新**。

### 3.4 混合路由（推荐的生产做法）

```python
def hybrid_router(user_message: str) -> str:
    """
    混合路由策略：规则优先，ML兜底。
    
    为什么这样设计？
    1. 高频明确场景（80%）：规则搞定，零延迟
    2. 长尾模糊场景（15%）：ML 模型处理
    3. 极端模糊场景（5%）：LLM 兜底
    """
    # 第一层：规则路由（微秒级）
    rule_result = rule_based_router(user_message)
    if rule_result != "general":
        return rule_result
    
    # 第二层：ML 路由（毫秒级）
    ml_result, confidence = ml_router.route(user_message)
    if confidence > 0.7:
        return ml_result
    
    # 第三层：LLM 路由（秒级，最准但最贵）
    return llm_router(user_message)
```

**核心思路**：先用最便宜的方法处理最明确的情况，再逐步升级到更贵但更准的方法。和搜索引擎的「先倒排索引，再 rerank」是同一个思路。

### 读到这里你应该理解

- 路由的三种实现：规则、ML 模型、Embedding
- 生产中用混合路由：规则优先 → ML 兜底 → LLM 最后防线
- Embedding 路由要防「描述漂移」
- 路由准确率是可以离线评测的

**本节小结（三张卡片）**

- **一句话定义**：路由是 **可测试的分类器**，不是 prompt 里一句「请判断」。
- **典型场景**：含「退款」关键词直送财务子 Agent，其余走通用。
- **和 A1 类比**：像 **`match/case` 与策略 dict 组合** —— 先 O(1) 命中高频，再复杂模型吃长尾。

## 4. 共享状态与冲突

**本节概念**：多 Agent 写同一 **共享可变状态** 时，要有 **版本、锁或事务**；否则调试会地狱级。

### 4.1 为什么会有共享状态？

当多个子 Agent 需要访问或修改同一份数据时，就产生了共享状态。典型场景：

- 客服 Agent 查到了用户地址，运维 Agent 也需要这个地址来确定最近的数据中心
- 分析 Agent 发现了异常数据，需要写到一个共享的「待处理」列表里
- 多个子 Agent 都往同一个报告文档里追加内容

### 4.2 黑板模式（Blackboard）

黑板是一个 **共享的数据结构**（通常是 dict），所有 Agent 都可以读写。

```python
from threading import Lock
from datetime import datetime


class Blackboard:
    """
    黑板：多 Agent 共享的键值存储。
    
    为什么用锁？
    多 Agent 可能并行运行（Fan-out 场景），两个 Agent 同时写同一个 key 会导致数据丢失。
    这和多线程写全局变量是同一个问题。
    """
    
    def __init__(self):
        self._data: dict[str, dict] = {}  # key → {value, updated_by, updated_at, version}
        self._lock = Lock()
    
    def write(self, key: str, value, agent_name: str) -> int:
        """
        写入一个值，返回新版本号。
        
        每次写入都记录：谁写的、什么时候写的、版本号。
        这样出了冲突可以追溯。
        """
        with self._lock:
            current = self._data.get(key, {})
            new_version = current.get("version", 0) + 1
            
            self._data[key] = {
                "value": value,
                "updated_by": agent_name,
                "updated_at": datetime.now().isoformat(),
                "version": new_version,
            }
            return new_version
    
    def read(self, key: str) -> tuple:
        """读取一个值，返回 (value, version)"""
        with self._lock:
            entry = self._data.get(key)
            if entry is None:
                return None, 0
            return entry["value"], entry["version"]
    
    def compare_and_swap(self, key: str, expected_version: int, new_value, agent_name: str) -> bool:
        """
        CAS 操作：只有版本号匹配才写入。
        
        使用场景：Agent A 读了 version=3 的数据，做了计算，
        要写回去时先检查版本还是不是 3——如果已经被 Agent B 改成 4 了，
        说明数据已经变了，A 的计算结果可能不对，拒绝写入。
        """
        with self._lock:
            current = self._data.get(key, {})
            if current.get("version", 0) != expected_version:
                return False  # 版本冲突，拒绝写入
            
            self._data[key] = {
                "value": new_value,
                "updated_by": agent_name,
                "updated_at": datetime.now().isoformat(),
                "version": expected_version + 1,
            }
            return True
```

### 4.3 消息总线模式

Agent 之间不直接共享状态，而是通过 **消息传递** 通信。每个 Agent 只维护自己的本地状态，通过发送和接收消息来协调。

```python
from collections import deque


class MessageBus:
    """
    简单的消息总线：Agent 间通过消息通信，而不是共享状态。
    
    优点：
    1. 解耦：Agent 不需要知道其他 Agent 的存在
    2. 可追踪：每条消息都有 trace_id，方便排障
    3. 可重放：消息可以持久化后重放，用于调试
    """
    
    def __init__(self):
        self._queues: dict[str, deque] = {}  # agent_name → message queue
        self._history: list[dict] = []  # 所有消息的历史记录
    
    def subscribe(self, agent_name: str):
        """注册一个 Agent 的消息队列"""
        self._queues[agent_name] = deque()
    
    def publish(self, from_agent: str, to_agent: str, content: dict, trace_id: str):
        """发送消息"""
        message = {
            "from": from_agent,
            "to": to_agent,
            "content": content,
            "trace_id": trace_id,
            "timestamp": datetime.now().isoformat(),
        }
        
        if to_agent in self._queues:
            self._queues[to_agent].append(message)
        
        self._history.append(message)  # 记录历史
    
    def receive(self, agent_name: str) -> dict | None:
        """接收一条消息（先进先出）"""
        queue = self._queues.get(agent_name, deque())
        return queue.popleft() if queue else None
```

### 4.4 哪种模式更好？

| 模式 | 适用场景 | 优点 | 缺点 |
|------|----------|------|------|
| **黑板** | Agent 间需要频繁读写同一数据 | 简单直接 | 并发冲突需要锁/CAS |
| **消息总线** | Agent 间松耦合、事件驱动 | 解耦、可追踪 | 延迟更高、代码更多 |

**禁区**：多 Agent **同时写** 同一业务行（比如同一个订单的状态），**且没有任何并发控制机制**。这在传统后端是经典 bug，在多 Agent 场景里更隐蔽——因为你可能不知道两个 Agent 在同时跑。

**本节小结（三张卡片）**

- **一句话定义**：共享状态要么 **不可变追加**，要么 **强一致写路径**。
- **典型场景**：两个子 Agent 同时改「用户地址」，后者覆盖前者且无通知。
- **和 A1 类比**：像 **多线程写全局 list 不加锁** —— 竞态只是换了个叙事包装。

## 5. 并行：Fan-out / Fan-in

**本节概念**：**Fan-out** 把一个任务拆成多个并行子任务提高吞吐；**Fan-in** 把多个子结果合并成一个最终结果。关键在于 **合并逻辑必须有界**——否则合并阶段比串行更慢。

### 5.1 Fan-out：并行分发

```python
import asyncio
from dataclasses import dataclass


@dataclass
class SubTaskResult:
    """子任务结果"""
    agent_name: str
    result: str
    success: bool
    elapsed_seconds: float


async def fan_out(
    sub_tasks: dict[str, str],  # {agent_name: task_description}
    agents: dict[str, object],   # {agent_name: agent_instance}
    timeout_per_task: float = 30.0,
) -> list[SubTaskResult]:
    """
    Fan-out: 把多个子任务并行分发给对应的 Agent。
    
    关键设计：
    1. 每个子任务有独立的 timeout——一个慢的不能拖死全局
    2. 用 return_exceptions=True 让 gather 不因一个异常全部失败
    3. 收集所有结果（包括失败的），留给 Fan-in 阶段处理
    """
    
    async def run_one(agent_name: str, task: str) -> SubTaskResult:
        """执行单个子任务"""
        import time
        start = time.time()
        try:
            result = await asyncio.wait_for(
                agents[agent_name].handle_async(task),
                timeout=timeout_per_task,
            )
            return SubTaskResult(
                agent_name=agent_name,
                result=result,
                success=True,
                elapsed_seconds=time.time() - start,
            )
        except asyncio.TimeoutError:
            return SubTaskResult(
                agent_name=agent_name,
                result=f"超时 ({timeout_per_task}s)",
                success=False,
                elapsed_seconds=timeout_per_task,
            )
        except Exception as e:
            return SubTaskResult(
                agent_name=agent_name,
                result=f"错误: {e}",
                success=False,
                elapsed_seconds=time.time() - start,
            )
    
    # 并行启动所有子任务
    tasks = [run_one(name, task) for name, task in sub_tasks.items()]
    results = await asyncio.gather(*tasks)
    
    return list(results)
```

### 5.2 Fan-in：合并结果

合并不是简单的字符串拼接。你需要处理：

1. **去重**：多个 Agent 可能返回了相同的信息
2. **冲突消解**：两个 Agent 对同一个问题给出了不同答案
3. **优先级**：某些 Agent 的结果比其他的更可信
4. **失败处理**：有的子任务超时或报错了

```python
def fan_in(results: list[SubTaskResult], merge_strategy: str = "llm") -> str:
    """
    Fan-in: 合并多个子任务的结果。
    
    merge_strategy:
    - "concat": 简单拼接（最快，适合结果不冲突的场景）
    - "llm": 用 LLM 智能合并（最准，但多一次调用）
    """
    
    # 分离成功和失败的结果
    successes = [r for r in results if r.success]
    failures = [r for r in results if not r.success]
    
    if not successes:
        return "所有子任务均失败: " + "; ".join(f"{r.agent_name}: {r.result}" for r in failures)
    
    if merge_strategy == "concat":
        # 简单拼接
        parts = []
        for r in successes:
            parts.append(f"【{r.agent_name}】\n{r.result}")
        if failures:
            parts.append(f"（{len(failures)} 个子任务失败）")
        return "\n\n".join(parts)
    
    elif merge_strategy == "llm":
        # 用 LLM 智能合并
        merge_prompt = "请将以下多个来源的信息综合成一个连贯的回答。\n"
        merge_prompt += "如果不同来源有矛盾，请指出矛盾点。\n\n"
        
        for r in successes:
            merge_prompt += f"来源 [{r.agent_name}] ({r.elapsed_seconds:.1f}s):\n{r.result}\n\n"
        
        if failures:
            merge_prompt += f"注意: 以下来源未能返回结果: {', '.join(r.agent_name for r in failures)}\n"
        
        return call_llm_simple(merge_prompt)
    
    else:
        raise ValueError(f"未知的合并策略: {merge_strategy}")
```

### 5.3 完整的 Fan-out / Fan-in 示例

```python
async def multi_source_search(user_query: str) -> str:
    """
    多源搜索：并行查询三个数据源，合并结果。
    
    场景：用户问"Python GIL 是什么"，
    同时搜索 Web、内部文档库、代码库，
    综合三个来源给出最全面的回答。
    """
    sub_tasks = {
        "web_search": f"在互联网上搜索: {user_query}",
        "doc_search": f"在内部文档库中检索: {user_query}",
        "code_search": f"在代码库中搜索相关代码: {user_query}",
    }
    
    # Fan-out: 并行搜索
    results = await fan_out(sub_tasks, agents, timeout_per_task=15.0)
    
    # 打印耗时信息
    for r in results:
        status = "✓" if r.success else "✗"
        print(f"  {status} {r.agent_name}: {r.elapsed_seconds:.1f}s")
    
    # Fan-in: 合并结果
    final = fan_in(results, merge_strategy="llm")
    return final
```

### 读到这里你应该理解

- Fan-out 的关键是每个子任务有独立 timeout
- Fan-in 的关键是处理去重、冲突和失败
- 并行只在 **合并逻辑有界** 时才省时间

**本节小结（三张卡片）**

- **一句话定义**：并行省时间的前提是 **合并逻辑有界**。
- **典型场景**：三源检索各返回 10 条，合并成 15 条要去重与 rerank。
- **和 A1 类比**：`gather` 像 **批量启动协程** —— 仍要处理 `return_exceptions` 与超时。

## 6. 调试与评测

**本节概念**：多 Agent 的 bug **一半在路由**，一半在子 Agent。指标要 **分项**，只看端到端会误判。

### 6.1 统一 trace_id

这是多 Agent 调试的第一条规则：**从用户请求进来到最终回复，所有经过的 Agent 必须用同一个 trace_id**。

```python
import uuid


def handle_request(user_message: str) -> str:
    """请求入口：生成 trace_id，贯穿所有 Agent"""
    trace_id = str(uuid.uuid4())
    
    # Supervisor 带着 trace_id
    result = supervisor.handle(user_message, trace_id=trace_id)
    
    # 所有日志都带 trace_id
    log.info("Request completed", extra={
        "trace_id": trace_id,
        "input": user_message[:100],
        "output": result[:100],
    })
    
    return result
```

没有 trace_id，你只能看到「某个请求错了」，看不到「错在路由环节还是子 Agent 环节」。

### 6.2 分项指标

端到端成功率 = 路由准确率 × 子 Agent 成功率。

如果端到端成功率是 60%，你需要知道是因为：
- 路由准确率 95% × 子 Agent 成功率 63%（子 Agent 有问题）
- 路由准确率 70% × 子 Agent 成功率 86%（路由有问题）

两种情况的优化方向完全不同。

```python
# 评测框架：分项统计
@dataclass
class EvalResult:
    trace_id: str
    user_input: str
    expected_agent: str
    actual_agent: str
    routing_correct: bool
    agent_output: str
    expected_output: str
    output_correct: bool


def evaluate_multi_agent(test_cases: list[dict]) -> dict:
    """评测多 Agent 系统，分项统计路由和子 Agent 的准确率"""
    results = []
    
    for case in test_cases:
        actual_agent = router.route(case["input"])
        routing_correct = actual_agent == case["expected_agent"]
        
        output = agents[actual_agent].handle(case["input"])
        output_correct = judge_output(output, case["expected_output"])
        
        results.append(EvalResult(
            trace_id=str(uuid.uuid4()),
            user_input=case["input"],
            expected_agent=case["expected_agent"],
            actual_agent=actual_agent,
            routing_correct=routing_correct,
            agent_output=output,
            expected_output=case["expected_output"],
            output_correct=output_correct,
        ))
    
    # 分项统计
    total = len(results)
    routing_acc = sum(1 for r in results if r.routing_correct) / total
    agent_acc = sum(1 for r in results if r.output_correct) / total
    e2e_acc = sum(1 for r in results if r.routing_correct and r.output_correct) / total
    
    return {
        "routing_accuracy": routing_acc,
        "agent_accuracy": agent_acc,
        "e2e_accuracy": e2e_acc,
        "total_cases": total,
    }
```

### 6.3 黄金测试用例

评测集要覆盖三类场景：

1. **路由正确**：明确属于某个子 Agent 的问题（「我要退款」→ customer_service）
2. **路由边界**：模糊的、可能属于多个子 Agent 的问题（「系统太慢了」→ 运维还是客服？）
3. **子 Agent 失败**：路由正确但子 Agent 执行失败的场景（工具超时、外部 API 挂了）

# 黄金测试用例示例

## 8. 常见误区与踩坑

### 误区 1：为了「好看的架构图」拆 Agent

**错误想法**：架构图上有 5 个 Agent 看起来很专业，比 1 个 Agent 高级。

**现实**：每多一个 Agent，就多一倍的调试成本、多一次 LLM 调用、多一个故障点。如果单 Agent + 3 个工具就能解决问题，拆成 3 个 Agent 就是自找麻烦。

**判断标准**：问自己「拆开后，**安全性**或**并行性**有实质提升吗？」如果只是「看起来更模块化」，不要拆。

### 误区 2：路由用 LLM，每次都调大模型

**错误做法**：每个用户请求都先调一次 GPT-4 来决定路由。

**问题**：路由的延迟和成本叠加在每次请求上。如果路由用 GPT-4 花 2 秒 + $0.01，子 Agent 又花 3 秒 + $0.05，总延迟 5 秒、总成本 $0.06。

**正确做法**：80% 的请求用规则路由（微秒级），15% 用小模型（毫秒级），只有 5% 的极端情况才用 LLM。

### 误区 3：子 Agent 之间直接调用

**错误做法**：Agent A 在自己的循环里直接 `import` Agent B 然后调用。

**问题**：形成了隐式依赖。当你修改 Agent B 的接口时，Agent A 可能莫名其妙地坏掉。而且日志里看不到 A 调了 B。

**正确做法**：所有 Agent 间通信走 **Supervisor 或消息总线**，日志里有完整的消息链路。

### 误区 4：Fan-in 不处理失败

**错误做法**：`results = await asyncio.gather(*tasks)` 然后直接合并所有结果。

**问题**：如果某个子任务失败了，`gather` 可能抛异常导致整个 Fan-in 崩溃。即使用了 `return_exceptions=True`，不检查哪些成功哪些失败就直接合并，结果里可能混入异常对象。

**正确做法**：显式分离成功和失败的结果（见第 5.2 节的 `fan_in` 实现）。

### 误区 5：不分项评测

**错误做法**：只看端到端成功率。

**问题**：端到端 60% 成功率，你不知道是路由的锅还是子 Agent 的锅，优化方向完全不确定。可能花了三天优化子 Agent 的 prompt，但真正的问题是路由把 30% 的请求发错了地方。

**正确做法**：路由准确率和子 Agent 成功率分开统计。

1. **拆分决策**：举一个「必须拆 Agent」和一个「不该拆 Agent」的真实业务场景。
2. **拓扑选择**：Supervisor、Pipeline、Peer 各适用什么场景？
3. **路由层级**：混合路由的三层分别是什么？为什么这样排序？

## 10. 面试高频问题

### Q1：什么时候应该从单 Agent 拆成多 Agent？

**答**：三个正当理由：

1. **安全隔离**：不同角色需要不同的工具白名单（只读 vs 写入），用代码而非 prompt 保证隔离
2. **专长模拟**：不同领域需要不同的 system prompt，混在一起会角色混乱
3. **并行加速**：多个独立子任务可以并行执行

不正当的理由：「架构图好看」「感觉更模块化」。每多一个 Agent 就多一倍调试成本，不要为了拆而拆。

### Q2：Supervisor 模式的最大风险是什么？怎么缓解？

**答**：最大风险是 **路由错误**——Supervisor 选错了子 Agent，整个请求就错了。缓解方法：

1. 混合路由：规则优先 → ML 兜底 → LLM 最后防线
2. 分项评测：单独评测路由准确率，区分路由错误和子 Agent 错误
3. 置信度阈值：路由不确定时走兜底 Agent 或转人工
4. 黄金测试集：覆盖明确/模糊/边界三类场景

### Q3：多 Agent 之间如何共享状态？

**答**：两种模式：

1. **黑板模式**：共享的键值存储，用锁或 CAS 防并发冲突。适合 Agent 间需要频繁读写同一数据的场景
2. **消息总线**：Agent 间通过消息通信，各自维护本地状态。适合松耦合、事件驱动的场景

**禁区**：多 Agent 同时写同一业务行且无并发控制。

### Q4：Fan-out/Fan-in 的关键注意点是什么？

**答**：

- **Fan-out**：每个子任务必须有独立的 timeout，一个慢的不能拖死全局
- **Fan-in**：必须处理去重、冲突消解、失败隔离。不能简单拼接结果，需要合并策略
- **前提**：并行只在合并逻辑有界时才省时间。如果合并阶段需要复杂推理，可能比串行更慢

### Q5：如何评测一个多 Agent 系统？

**答**：分项评测 + 统一 trace_id：

1. **路由准确率**：给定输入，路由选的 Agent 是否正确
2. **子 Agent 成功率**：给定正确路由，子 Agent 的输出是否正确
3. **端到端成功率**：从用户输入到最终输出是否正确
4. **trace_id 贯穿**：从入口到每个子 Agent 到最终输出，用同一个 trace_id 串起来

端到端成功率 = 路由准确率 × 子 Agent 成功率，定位问题要看分项指标。
