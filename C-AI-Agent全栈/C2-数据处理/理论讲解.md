## 0. 与 A1、C1 的衔接

**本节概念**：**数据处理**在这里不是「数据分析报表」，而是 **把字节序列变成模型可稳定消费的证据**；质量在进嵌入与进上下文之前就已决定。

A1 教你的 **纯函数、显式编码、异常与日志**，在数据流水线里全部用得上：读文件失败要 **记录路径与编码策略**；清洗函数应 **幂等**（同一输入跑两遍结果一致），这样才敢在索引重建时反复跑。C1 里你学会了 **Token 计费**；本章告诉你：**一行没删干净的导航栏**，可能吃掉成百上千个 token 却检索不到有效信息 —— 这是「成本与质量」的双杀。

**本节小结（三张卡片）**

- **一句话定义**：Agent 链路上的数据处理 = **降噪 + 结构化 + 可追溯元数据**。
- **典型场景**：技术博客爬下来带大量「分享按钮」文案，不清洗就进向量库，用户问 API 用法时 Top-K 全是社交噪声。
- **和 A1 类比**：整条清洗链像 **管道式的生成器组合**（`strip` → `normalize` → `chunk`），每一环只处理自己的不变式，便于单测。

若你同时接触 **企业数据治理（CDGA / DMBOK）** 认证或甲方数据团队话术，可对照 **[§14 CDGA 与 DMBOK：和本章的衔接](#14-cdga-与-dmbok和本章的衔接)**，把「治理维度」映射到本章各节。

## 1. 为什么 Agent 链路里必须有「数据处理」

**本节概念**：模型「看到」的只是 **上下文里的字符串**；脏字符串会让 **检索空间与生成空间同时劣化**。

- **RAG**：页眉、导航栏、乱码表格会进入 chunk → **向量空间被噪声占据** → Top-K 常检索到无意义片段。用户问的是业务问题，命中的是站点 chrome。  
- **Tool 返回**：数据库/API 的嵌套 JSON 若原样塞进上下文，**Token 暴涨**且模型易抓错字段 —— 和把整棵 `dict` 不打 `pprint` 深度限制就 `str()` 进日志是同类问题。  
- **微调 / 评测**：JSONL schema 不一致会导致训练脚本静默失败或指标不可比；**没有稳定主键**就无法做回归对比。  
- **可观测性（C8）**：统一字段才能做聚合与告警；字段名三天一改，仪表盘全废。

**一句话**：模型「读」的是字节序列；**质量在进模型之前就已决定**。

**本节小结（三张卡片）**

- **一句话定义**：数据处理是 **在进入嵌入/上下文之前** 的质量闸门。
- **典型场景**：同一 PDF 解析器升级导致空格差异，向量全漂移，需要 **索引版本号** 与解析器版本绑定。
- **和 A1 类比**：脏数据像 **可变默认参数** —— 一开始省事，调用链一长就到处踩坑；清洗像 **在 `__init__` 里强制拷贝/规范化输入**。

## 2. 常见输入形态与风险

**本节概念**：不同形态的 **错误模式不同**，清洗策略必须 **按形态组合**，没有万能 `clean(x)`。

| 形态 | 典型来源 | 主要风险 |
|------|----------|----------|
| Markdown | 文档站、笔记 | front matter、重复标题、内嵌 HTML |
| HTML | 爬虫 | 脚本/style、模板重复块 |
| PDF | 论文、合同 | 换行错乱、多栏顺序错 |
| JSON/JSONL | 日志、导出对话 | 字段缺失、一行超大 JSON |
| CSV/Parquet | 业务表 | 缺失值、类型推断错、泄漏敏感列 |

**落地提示**：为每种形态维护 **最小可复现样本**（一两页 PDF、几篇 MD、几行 JSONL），任何清洗函数改动都跑回归 —— 这与 A1 里「改装饰器先跑全量单测」同构。

**本节小结（三张卡片）**

- **一句话定义**：形态表解决的是 **「风险长什么样」** 而不是「用什么库」。
- **典型场景**：HTML 里 `nav` 与 `main` 都含文本，不抽主内容则 RAG 永远检索到菜单。
- **和 A1 类比**：不同形态像 **多种文件对象**（文本/二进制/流），读法不同，不能共用一个粗 `open()` 默认。

## 3. 字符、编码与规范化

**本节概念**：**Unicode 同一语义多种码位**、**零宽字符**、**BOM** 会让「肉眼一样」的字符串在程序里不相等，进而破坏去重与检索键。

- **统一 UTF-8**：读文件显式 `encoding="utf-8"`，对非法字节用 **`errors="replace"`** 并计数告警 —— 静默吞掉非法字节会在下游变成「神秘空格」。  
- **Unicode 规范化**：`unicodedata.normalize("NFKC")` 处理全角/兼容字符（注意：可能改变语义，需业务确认）。  
- **零宽字符与 BOM**：影响字符串匹配与去重，应剔除（见 `demos/03_normalize_text.py`）。  
- **空白**：行尾空格、连续空行影响分块边界与展示，可压缩为「最多双换行」。

**本节小结（三张卡片）**

- **一句话定义**：规范化解决的是 **「字符串相等」与「人类视觉相等」不一致**。
- **典型场景**：用户从 Word 粘贴全角标点进知识库，检索 query 用半角，相似度莫名变低。
- **和 A1 类比**：`normalize` 像 **`pathlib.Path.resolve()`** —— 多种写法指向同一实体，先规范化再比较。

## 4. Markdown 与技术文档

**本节概念**：Markdown 是 **半结构化文本**；front matter 是机器字段，正文是人类叙述，**嵌入前常常要物理分离**。

- **YAML front matter**：元数据有用，但**嵌入前常需剥离**，避免进入向量正文。  
- **代码块**：决定是否与正文一起嵌入；有时应对 **代码块单独建 chunk** 并打 `language` 标签 —— API 说明与示例代码的向量邻域不同。  
- **链接与图片**：纯文本流水线可保留 `[text](url)` 或替换为 `text（见 url）`，避免向量里只有 URL。  
- **演示**：`demos/01_clean_markdown.py`。

**本节小结（三张卡片）**

- **一句话定义**：Markdown 清洗 = **结构标签（元数据）与语义正文分流**。
- **典型场景**：同一文档里 20 个「上一篇/下一篇」链接，不应占满 chunk 的 token 预算。
- **和 A1 类比**：剥离 front matter 像 **`splitlines` 后先识别 shebang / 编码声明**，再执行正文逻辑。

## 5. HTML / PDF 要点（不展开具体库）

**本节概念**：HTML/PDF 的难点是 **版式还原**，不是「读出一个 str」；版式错了，语义顺序就错，RAG 越准越离谱。

- **HTML**：优先抽 **main/article** 区域；剔除 `script/style/nav`；表格考虑转 Markdown 或单独存储。  
- **PDF**：先选成熟解析库；对扫描件需 **OCR** 流水线（另一岗位域）。  
- **原则**：解析器版本升级会导致文本微差，**索引版本**要与解析版本绑定记录 —— 否则你无法解释「上周能命中这周不能」。

**本节小结（三张卡片）**

- **一句话定义**：HTML/PDF 清洗买的是 **语义顺序**，不是字符数。
- **典型场景**：双栏论文左栏读完跳右栏，chunk 边界落在栏中，检索到半句话。
- **和 A1 类比**：错误版式像 **迭代器状态被外部打乱** —— 表面上还在 `for` 循环，语义已经越界。

## 6. JSON / JSONL：对话与评测数据

**本节概念**：**JSONL** 适合「大文件 + 流式读 + 单行失败可跳过」；对话数据是 **有状态协议**，需要 **role 与 tool_call_id** 等字段的一致约定。

- **一行一条**（JSONL）适合大文件流式读。  
- **role 白名单**：`system` / `user` / `assistant` / `tool`（若用工具）。  
- **每条必有稳定主键**（如 `id`、`session_id`）便于去重与回归。  
- **演示**：`demos/02_jsonl_messages.py`。

**本节小结（三张卡片）**

- **一句话定义**：JSONL 是 **可恢复、可并行** 的日志形态；大 JSON 数组适合 **小到内存放得下** 的配置。
- **典型场景**：百万条微调样本，用 JSONL + `for line in f` 逐条校验，单行坏 JSON 不会拖死全文件。
- **和 A1 类比**：读 JSONL 像 **惰性迭代大文件**；读巨大 JSON 像 **`loads` 一次性进内存** —— 边界清楚就不会混用。

## 7. 表格数据（CSV / Parquet）

**本节概念**：表格进 LLM 的关键是 **渲染为自然语言或键值叙述**，而不是把 **裸 CSV** 当「模型看得懂的图」。

- **Pandas**：`dtype` 显式指定，避免把 ID 读成 float。  
- **缺失值**：`fillna` 策略要可解释；分类任务注意 **类不平衡**。  
- **进 LLM**：常将一行 **渲染为键值文本** 或 **模板句子**，而不是裸 CSV 贴进 prompt（除非任务需要）。

**本节小结（三张卡片）**

- **一句话定义**：表格 → 文本是 **有损渲染**，要在模板层控制字段顺序与空值展示。
- **典型场景**：把 `user_id` 与 `email` 一并渲染进 prompt 做客服问答，属于 **合规事故** 而非模型问题。
- **和 A1 类比**：`to_dict` 再 `json.dumps` 像 **`dataclass.asdict`** —— 先结构化，再选序列化策略。

## 8. 去重、抽样与质量门禁

**本节概念**：**去重**决定训练与索引是否浪费算力；**门禁**决定坏样本是 **显式隔离** 还是 **静默污染**。

- **精确去重**：hash 整段文本。  
- **近似去重**：SimHash/MinHash（大语料常用）；RAG 小规模可用 **embedding 聚类** 粗判。  
- **质量门禁**：最短长度、最大长度、非法字符比例、语言检测（可选）——不通过则 **隔离到 quarantine** 而非静默丢弃。

**本节小结（三张卡片）**

- **一句话定义**：门禁像 CI —— **失败要可见、可追踪**，不要悄悄丢样本。
- **典型场景**：爬虫把错误页「404 Not Found」也索引进库，门禁应用长度与关键词拦截。
- **和 A1 类比**：quarantine 目录像 **`pytest --lf` 前的失败缓存** —— 先留住现场再修。

## 9. PII 与脱敏

**本节概念**：**PII** 一旦进向量库或日志，删除成本远高于 **在入口正则掉**；Agent 还会 **主动复述** 上下文里的敏感字段。

- **正则 + 规则**可覆盖手机、邮箱等常见模式（见 `demos/04_pii_redact.py`）。  
- **NER 模型**提高召回，有成本。  
- **原则**：脱敏在 **进向量库与写日志之前**；密钥类 **永不下发给浏览器**。

**本节小结（三张卡片）**

- **一句话定义**：脱敏是 **数据分类 + 最小暴露** 的工程纪律，不是「模型别乱说」的 prompt 技巧。
- **典型场景**：客服对话里含身份证号，RAG 检索命中后模型原样输出 —— 事故根因在数据层。
- **和 A1 类比**：脱敏像 **`logging` 里过滤 secrets** —— 默认假设任何字符串最终都会上盘或上网。

## 10. 与 C3 分块、元数据的衔接

**本节概念**：**分块**把长文档切成可嵌入单元；**元数据**把「这段从哪来、谁能看」与向量绑在一起，检索时先过滤再算相似度。

- 分块前尽量 **按语义边界**（标题、段落）而非纯 `str[i:i+512]`。  
- 每个 chunk 建议带：**`source_id`、`path`、`heading`、`updated_at`、可选 `acl_tags`**。  
- **ACL**：检索时必须 **先过滤元数据** 再算相似度，避免跨租户泄漏。  
- 详见 [C3 — RAG 与知识库](../C3-RAG与知识库/理论讲解.md)。

**本节小结（三张卡片）**

- **一句话定义**：没有元数据的向量检索 ≈ **没有权限位的文件系统**。
- **典型场景**：同一产品文档有「对外版」与「内部版」，仅靠文本相似度无法区分，必须 `acl_tags`。
- **和 A1 类比**：chunk + metadata 像 **对象 + `__dict__` 里除数据外再挂一层标签**，查询时先过滤标签再比较内容。

## 11. 工具栈速览

**本节概念**：工具选型服从 **镜像体积、许可证、团队熟悉度**；不要为了 demo 漂亮引入生产扛不动的依赖。

| 需求 | 常见选择 |
|------|----------|
| 表格 | `pandas` / `polars` |
| HTML | `beautifulsoup4` / `lxml` |
| PDF | `pypdf` / 商用解析 |
| 大流水线 | `unstructured`（重） |
| 校验 | `pydantic` / `jsonschema` |

**原则**：能标准库则标准库；引入重型依赖前确认 **部署镜像与许可证**。

**本节小结（三张卡片）**

- **一句话定义**：工具栈解决 **「谁来做脏活」**，本章解决 **「脏活验收标准」**。
- **典型场景**：边缘节点内存 512MB，装不下 `unstructured`，只能轻量解析 + 上游预处理。
- **和 A1 类比**：选型像 **标准库优先于 PyPI** —— 依赖越少，长期维护越像 `stdlib` 一样稳。

## 14. CDGA 与 DMBOK：和本章的衔接

> **本节概念**：**CDGA**（DAMA 认证体系中的数据治理助理级）与 **DMBOK** 描述的是企业里「谁对数据负责、管哪些维度、用什么制度度量」；**C2** 描述的是「在进嵌入与进上下文之前，工程上怎么洗、怎么验收」。二者**不互相替代**，但可以用同一套**词汇与检查清单**对齐业务沟通，并把治理要求落到流水线里。

**为什么值得对照**：甲方数据团队常说「质量 / 元数据 / 分级 / 生命周期」，你回「chunk、hash、quarantine」容易各说各话；对照表的作用是 **翻译层**——你知道对方在要哪一类控制点，也知道本章哪一节能落地。

| DMBOK / 治理语境里常见的关注点 | 在本章（C2）中的落点 | 说明 |
|--------------------------------|---------------------|------|
| **数据质量**（规则、度量、改进） | [§8 去重、抽样与质量门禁](#8-去重抽样与质量门禁) | 门禁、隔离区、可追踪失败，对应「质量不能靠肉眼抽检」的工程化。 |
| **元数据**（技术元数据、操作元数据） | [§10 与 C3 分块、元数据的衔接](#10-与-c3-分块元数据的衔接) | `source_id`、解析器版本、索引版本等，对应治理里的可追溯与变更解释。 |
| **数据安全 / 隐私**（分类、最小暴露） | [§9 PII 与脱敏](#9-pii-与脱敏) | 分类分级在企业的制度里；在 Agent 链路上体现为「进向量库与日志前的脱敏」。 |

**怎么用这段对照（学习顺序）**

- **以 C2 为主干**：考试与动手仍按本章目录与 `demos/` 走，保证 **RAG/Agent 工程语言**过关。  
- **以 CDGA / DMBOK 为透镜**：遇到企业客户时，把上表当作 **需求访谈提纲**——对方谈「治理」，你映射到「哪一段流水线加什么门禁与元数据」。

**本节小结（三张卡片）**

- **一句话定义**：CDGA/DMBOK 提供 **企业维度的「管什么」**；C2 提供 **LLM 链路上的「怎么洗、怎么证伪」**。
