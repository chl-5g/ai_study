## 0. 与 A1、C2 的衔接

**本节概念**：**RAG** 把「读文档」从训练期搬到推理期：用 **检索** 选出证据片段，再让 LLM **只作为推理与表达层**；**数据处理（C2）** 决定证据是否干净，**向量与索引** 决定证据是否可命中。

如果你在 A1 里已经习惯「**先写纯函数处理数据，再拼 I/O**」，那么 RAG 索引流水线几乎同构：**Loader/Splitter/Embedder** 是一条纯函数链，出错时应该能 **单元测试每一环**；`list[str]` 进、`ndarray` 或向量库句柄出，边界用类型标清楚，排障会快很多。C2 强调的 **元数据与 ACL** 不是锦上添花：没有 `tenant_id` 过滤的向量检索，和「多线程共享可变全局 dict 不加密」是同一类事故。

**本节小结（三张卡片）**

- **一句话定义**：RAG = **可换源的体外记忆** + **可控引用的生成**。
- **典型场景**：内部 wiki 日更，模型权重不能日更；用重建索引代替重训。
- **和 A1 类比**：**向量库**像 **把文档摘要成 key，放进 dict 再按相似 key 取 value**；**重排序**像 **在 `sorted(..., key=...)` 里换一个更贵但更准的 key 函数**。

## 1. RAG 概述

### 1.1 为什么需要 RAG

**本节概念**：**知识截止**与**私域缺失**是参数记忆的硬边界；**RAG**用外部语料把「事实」从权重里挪到「上下文窗口里的证据」。

大语言模型（LLM）的知识来自训练数据，存在两个根本性问题：

1. **知识截止**：模型无法知道训练截止日期之后的信息。比如你用 2024 年训练的模型问"2026 年 4 月的新闻"，它只能编。
2. **私域知识缺失**：企业内部文档、行业数据库、个人笔记等从未出现在公开训练集中。你问模型"我们公司的报销流程是什么"，它不可能知道。

RAG（Retrieval-Augmented Generation，检索增强生成）的核心思路：**先检索，再生成**。

```
传统 LLM：用户提问 → 模型从参数记忆中"回忆" → 回答（可能编造）
RAG：    用户提问 → 从知识库检索证据 → 证据+问题一起给模型 → 基于证据回答
```

**一个类比帮你理解**：传统 LLM 像一个记忆力超强但可能记错的人，闭卷答题；RAG 像让这个人打开参考书，开卷答题。开卷不保证不出错，但至少有据可查。

优势：
- 无需重新训练模型，成本远低于 Fine-tuning
- 知识可以实时更新（替换文档即可）
- 可追溯来源（返回引用片段），降低幻觉风险（工程上仍需 **引用格式、拒答策略、评测** 闭环，见本章 [§8](#8-常见问题与解决) 与 [C1 §5.5](../C1-LLM基础与Prompt/理论讲解.md)）

### 1.2 RAG vs Fine-tuning

| 维度 | RAG | Fine-tuning |
|------|-----|-------------|
| 知识更新 | 替换文档即可，秒级生效 | 需重新训练，小时/天级 |
| 成本 | 低（向量数据库 + 嵌入 API） | 高（GPU 算力 + 数据标注） |
| 适用场景 | 事实问答、文档检索、客服 | 风格迁移、领域术语、特殊格式 |
| 幻觉控制 | 较好（有证据可查） | 一般（仍可能编造） |
| 上下文长度 | 受限于 Context Window | 无此限制 |
| 可解释性 | 高（可返回检索来源） | 低 |

**口径提示**：上表里的 **Fine-tuning** 泛指「**要跑训练、会改动权重**」的一类路径；实际 **成本、周期、显存** 从 **全量微调 → LoRA → QLoRA** 差异很大，口语里的「微调」可能指其中任一种。术语与选型分层见 [C1 §2.4](../C1-LLM基础与Prompt/理论讲解.md)。

实际项目中，RAG 和 Fine-tuning 并不互斥。常见组合：先用 **LoRA / QLoRA** 等让模型熟悉领域术语与回答风格，再用 RAG 注入最新事实。

### 1.3 RAG 架构全景图

```
用户提问
    │
    ▼
┌──────────┐    Query Embedding     ┌──────────────┐
│  Query   │ ──────────────────────>│ Embedding    │
│ 处理器    │                        │ Model        │
└──────────┘                        └──────┬───────┘
                                           │ query vector
                                           ▼
                                    ┌──────────────┐
                                    │ Vector Store │ <── 离线索引流水线
                                    │ (向量数据库)  │     (文档加载→分块→嵌入→存储)
                                    └──────┬───────┘
                                           │ Top-K 相关片段
                                           ▼
                                    ┌──────────────┐
                                    │ Prompt 组装   │
                                    │ (System +     │
                                    │  Context +    │
                                    │  Question)    │
                                    └──────┬───────┘
                                           │
                                           ▼
                                    ┌──────────────┐
                                    │   LLM 生成    │
                                    └──────┬───────┘
                                           │
                                           ▼
                                       回答 + 引用来源
```

**两条线**：
- **离线线（Indexing）**：文档加载 → 清洗（C2） → 分块 → 嵌入 → 存入向量库。跑一次，可复用。
- **在线线（Query）**：用户提问 → 嵌入 → 向量检索 → 拼 Prompt → LLM 生成。每次请求都跑。

**本节小结（三张卡片）**

- **一句话定义**：RAG = 检索 + 生成，把"闭卷考试"变成"开卷考试"。
- **典型场景**：企业知识库问答——文档每周更新，不可能每周重训模型。
- **和 A1 类比**：离线索引像 **预计算缓存**，在线检索像 **cache.get()**，生成像 **基于缓存结果的业务逻辑**。

## 2. 文档处理流水线

### 2.1 文档加载（Document Loading）

**本节概念**：加载不只是 `open().read()`，关键是 **元数据保留** 和 **格式适配**。

不同格式需要不同的加载器：

| 格式 | 常用工具/库 | 注意事项 |
|------|------------|---------|
| PDF | `pypdf`、`pdfplumber`、`unstructured` | 多栏/表格容易错位（见 C2 §5） |
| Markdown | 直接读取、`markdown-it` | 剥离 front matter（见 C2 §4） |
| HTML | `BeautifulSoup`、`trafilatura` | 去 script/style/nav（见 C2 §5） |
| DOCX | `python-docx` | 注意嵌入图片和表格 |
| CSV/Excel | `pandas` | dtype 显式指定（见 C2 §7） |
| 代码文件 | 按语言 AST 解析或直接文本读取 | 函数/类级别分块 |

**关键点**：每个文档片段都应携带来源元数据：

```python
from dataclasses import dataclass, field
from datetime import datetime


@dataclass
class Document:
    """一个文档片段，携带完整元数据。"""
    content: str                          # 正文内容
    source: str                           # 来源文件路径
    metadata: dict = field(default_factory=dict)
    
    # 推荐的 metadata 字段：
    # - page: int          # 页码（PDF）
    # - heading: str       # 所属章节标题（Markdown）
    # - file_type: str     # 文件类型
    # - updated_at: str    # 文档更新时间
    # - parser_version: str # 解析器版本（见 C2 §5.3）


def load_markdown(path: str) -> list[Document]:
    """加载 Markdown 文件，按标题拆分为多个 Document。"""
    import re
    
    with open(path, encoding="utf-8") as f:
        text = f.read()
    
    # 剥离 front matter
    text = re.sub(r"^---\s*\n.*?\n---\s*\n", "", text, flags=re.DOTALL)
    
    # 按一级/二级标题拆分
    sections = re.split(r"(^#{1,2}\s+.+$)", text, flags=re.MULTILINE)
    
    documents = []
    current_heading = "Introduction"
    current_content = ""
    
    for part in sections:
        if re.match(r"^#{1,2}\s+", part):
            if current_content.strip():
                documents.append(Document(
                    content=current_content.strip(),
                    source=path,
                    metadata={"heading": current_heading, "file_type": "markdown"},
                ))
            current_heading = part.strip().lstrip("#").strip()
            current_content = ""
        else:
            current_content += part
    
    if current_content.strip():
        documents.append(Document(
            content=current_content.strip(),
            source=path,
            metadata={"heading": current_heading, "file_type": "markdown"},
        ))
    
    return documents
```

### 2.2 文本分块（Chunking）

为什么要分块？三个原因：

1. **Context Window 有限**：整篇文档塞不进去
2. **检索粒度**：文档级别太粗，搜到一篇 50 页的手册没有意义
3. **嵌入精度**：短文本的嵌入向量更能捕捉具体语义

**分块是 RAG 效果的关键瓶颈** —— 分得太大，检索噪声多（搜到了但大部分不相关）；分得太小，丢失上下文（搜到了但读不懂）。

### 2.3 嵌入（Embedding）

将文本块转为固定维度的浮点向量（如 768 维、1536 维），使得 **语义相近的文本在向量空间中距离更近**。

```python
# 嵌入的直觉理解
text_a = "Python 是一种编程语言"
text_b = "Python 是一种脚本语言"
text_c = "今天天气真好"

# 嵌入后：
# vec_a 和 vec_b 的距离很近（语义相似）
# vec_a 和 vec_c 的距离很远（语义无关）
```

### 2.4 向量存储（Vector Store）

将嵌入向量及其对应的文本、元数据持久化存储，并提供高效的 **近似最近邻（ANN）** 搜索能力。

**完整的离线索引流水线代码**：

```python
import json
import hashlib
import numpy as np
from dataclasses import dataclass


@dataclass
class Chunk:
    """一个可嵌入的文本块。"""
    text: str
    embedding: list[float] | None = None
    metadata: dict | None = None
    chunk_id: str = ""
    
    def __post_init__(self):
        if not self.chunk_id:
            self.chunk_id = hashlib.sha256(self.text.encode()).hexdigest()[:12]


class SimpleVectorStore:
    """最小可用的内存向量库（用于理解原理，生产用 ChromaDB/Qdrant）。"""
    
    def __init__(self):
        self._chunks: list[Chunk] = []
    
    def add(self, chunks: list[Chunk]):
        """添加已嵌入的 chunk。"""
        for chunk in chunks:
            if chunk.embedding is None:
                raise ValueError(f"chunk {chunk.chunk_id} 没有 embedding")
        self._chunks.extend(chunks)
    
    def search(self, query_embedding: list[float], top_k: int = 5) -> list[tuple[Chunk, float]]:
        """余弦相似度检索，返回 (chunk, score) 列表。"""
        query = np.array(query_embedding)
        query_norm = np.linalg.norm(query)
        
        scored = []
        for chunk in self._chunks:
            doc = np.array(chunk.embedding)
            # 余弦相似度 = dot(a, b) / (||a|| * ||b||)
            similarity = np.dot(query, doc) / (query_norm * np.linalg.norm(doc) + 1e-9)
            scored.append((chunk, float(similarity)))
        
        scored.sort(key=lambda x: x[1], reverse=True)
        return scored[:top_k]
    
    def search_with_filter(
        self,
        query_embedding: list[float],
        top_k: int = 5,
        filter_fn: callable = None,
    ) -> list[tuple[Chunk, float]]:
        """带元数据过滤的检索（ACL 场景必须）。"""
        query = np.array(query_embedding)
        query_norm = np.linalg.norm(query)
        
        scored = []
        for chunk in self._chunks:
            # 先过滤，再算相似度（不是反过来！）
            if filter_fn and not filter_fn(chunk):
                continue
            
            doc = np.array(chunk.embedding)
            similarity = np.dot(query, doc) / (query_norm * np.linalg.norm(doc) + 1e-9)
            scored.append((chunk, float(similarity)))
        
        scored.sort(key=lambda x: x[1], reverse=True)
        return scored[:top_k]
```

**本节小结（三张卡片）**

- **一句话定义**：文档处理流水线 = **加载 → 清洗（C2）→ 分块 → 嵌入 → 存储**，五步串行。
- **典型场景**：100 篇 Markdown 文档 → 按标题拆成 500 个 chunk → 每个 chunk 嵌入为 768 维向量 → 存入 ChromaDB。
- **和 A1 类比**：整条流水线像 **`map` + `filter` + `reduce` 的组合**，每一步输入输出类型明确，可以单独测试。

## 3. 分块策略详解

**本节概念**：分块是 RAG 中 **最值得反复调优** 的环节。分块策略直接决定检索质量——它在向量空间中定义了"你能搜到什么粒度的东西"。

### 3.1 固定大小分块（Fixed-size Chunking）

```python
def fixed_size_chunk(text: str, chunk_size: int = 500, overlap: int = 50) -> list[str]:
    """按字符数固定分块，带重叠。"""
    chunks = []
    start = 0
    while start < len(text):
        end = start + chunk_size
        chunks.append(text[start:end])
        start = end - overlap  # 重叠 overlap 个字符
    return chunks

# 示例
text = "A" * 1200  # 1200 个字符
chunks = fixed_size_chunk(text, chunk_size=500, overlap=50)
# chunks[0]: A[0:500]
# chunks[1]: A[450:950]   ← 从 450 开始，和上一个重叠 50
# chunks[2]: A[900:1200]
```

- **优点**：实现简单、chunk 大小可预测
- **缺点**：会在句子/段落中间截断，**破坏语义完整性**

### 3.2 递归分割（Recursive Character Splitting）

按优先级尝试不同分隔符：`\n\n` → `\n` → `.` → ` ` → 字符级别。

```python
def recursive_split(
    text: str,
    chunk_size: int = 500,
    separators: list[str] | None = None,
) -> list[str]:
    """递归分割：优先在语义边界切分。"""
    if separators is None:
        separators = ["\n\n", "\n", "。", ". ", " ", ""]
    
    if len(text) <= chunk_size:
        return [text] if text.strip() else []
    
    # 找到能用的最高优先级分隔符
    for sep in separators:
        if sep == "":
            # 兜底：逐字符切
            return [text[i:i+chunk_size] for i in range(0, len(text), chunk_size)]
        
        parts = text.split(sep)
        if len(parts) > 1:
            chunks = []
            current = ""
            for part in parts:
                candidate = current + sep + part if current else part
                if len(candidate) <= chunk_size:
                    current = candidate
                else:
                    if current:
                        chunks.append(current)
                    current = part
            if current:
                chunks.append(current)
            
            # 如果有 chunk 仍然超长，递归用下一级分隔符继续切
            result = []
            remaining_seps = separators[separators.index(sep) + 1:]
            for chunk in chunks:
                if len(chunk) > chunk_size and remaining_seps:
                    result.extend(recursive_split(chunk, chunk_size, remaining_seps))
                else:
                    result.append(chunk)
            return result
    
    return [text]
```

- **优点**：保持语义边界，适用面广（LangChain 的 `RecursiveCharacterTextSplitter` 就是这个策略）
- **缺点**：块大小不完全均匀

### 3.3 按语义分块（Semantic Chunking）

```python
def semantic_chunk(
    sentences: list[str],
    embed_fn: callable,
    threshold: float = 0.5,
) -> list[list[str]]:
    """
    基于相邻句子嵌入相似度的语义分块。
    当相似度低于 threshold 时，在此处分割。
    """
    if len(sentences) <= 1:
        return [sentences]
    
    # 嵌入所有句子
    embeddings = embed_fn(sentences)
    
    chunks = []
    current_chunk = [sentences[0]]
    
    for i in range(1, len(sentences)):
        # 计算相邻句子的余弦相似度
        sim = cosine_similarity(embeddings[i-1], embeddings[i])
        
        if sim < threshold:
            # 相似度低于阈值 → 语义断裂 → 新 chunk
            chunks.append(current_chunk)
            current_chunk = [sentences[i]]
        else:
            current_chunk.append(sentences[i])
    
    if current_chunk:
        chunks.append(current_chunk)
    
    return chunks


def cosine_similarity(a, b) -> float:
    """计算两个向量的余弦相似度。"""
    import numpy as np
    a, b = np.array(a), np.array(b)
    return float(np.dot(a, b) / (np.linalg.norm(a) * np.linalg.norm(b) + 1e-9))
```

- **优点**：每个块内语义高度一致
- **缺点**：需要额外调用嵌入模型，计算成本高。适合高价值文档（合同、法规）。

### 3.4 Overlap 重叠

任何分块策略都建议加 Overlap（通常 10%–20%），确保跨块边界的信息不会丢失。

```
chunk_size=500, overlap=100

文档: [=============== 1000 字符 ===============]

chunk 1: [===== 500 =====]
chunk 2:            [===== 500 =====]
                  ↑ 重叠 100 字符 ↑

没有 overlap: 如果关键句子正好在 500 字符处被切断
有 overlap:   关键句子至少在一个 chunk 中是完整的
```

### 3.5 Markdown/代码专用分块

```python
def split_markdown_by_heading(
    text: str,
    max_chunk_size: int = 1000,
) -> list[dict]:
    """按 Markdown 标题层级分块，保持语义完整。"""
    import re
    
    sections = re.split(r"(^#{1,3}\s+.+$)", text, flags=re.MULTILINE)
    
    chunks = []
    heading = ""
    content = ""
    
    for part in sections:
        if re.match(r"^#{1,3}\s+", part):
            if content.strip():
                chunks.append({"heading": heading, "content": content.strip()})
            heading = part.strip()
            content = ""
        else:
            content += part
    
    if content.strip():
        chunks.append({"heading": heading, "content": content.strip()})
    
    # 超长 chunk 二次切分
    result = []
    for chunk in chunks:
        if len(chunk["content"]) > max_chunk_size:
            sub_chunks = recursive_split(chunk["content"], max_chunk_size)
            for i, sc in enumerate(sub_chunks):
                result.append({
                    "heading": f"{chunk['heading']} (part {i+1})",
                    "content": sc,
                })
        else:
            result.append(chunk)
    
    return result
```

**本节小结（三张卡片）**

- **一句话定义**：分块 = 在"粒度"和"完整性"之间找平衡点。
- **典型场景**：API 文档用标题分块，小说用段落分块，法律合同用条款分块——没有万能策略。
- **和 A1 类比**：分块像 **`itertools.batched()` 的语义版本** —— 不是按个数切，而是按意义切。

## 4. 嵌入模型

**本节概念**：嵌入模型把文本压缩成固定长度的浮点向量，使得 **语义相近的文本在高维空间中距离更近**。选对嵌入模型 = 检索质量的地基。

### 4.1 主流嵌入模型对比

| 模型 | 维度 | 语言 | 部署方式 | 适用场景 |
|------|------|------|---------|---------|
| `text-embedding-3-small` | 1536 | 多语言 | OpenAI API | 性价比高，通用 |
| `text-embedding-3-large` | 3072 | 多语言 | OpenAI API | 精度最高 |
| `BAAI/bge-large-zh-v1.5` | 1024 | 中文优 | 本地 | **中文场景首选** |
| `BAAI/bge-m3` | 1024 | 多语言 | 本地 | 支持稀疏+稠密混合 |
| `nomic-embed-text` | 768 | 英文 | Ollama | 轻量快速 |
| `all-MiniLM-L6-v2` | 384 | 英文 | 本地 | 最轻量 |

### 4.2 嵌入的代码实现

```python
# ── 方案一：用 Ollama 本地嵌入 ──
import requests
import json


def embed_ollama(texts: list[str], model: str = "nomic-embed-text") -> list[list[float]]:
    """使用 Ollama 本地模型生成嵌入向量。"""
    embeddings = []
    for text in texts:
        resp = requests.post(
            "http://localhost:11434/api/embeddings",
            json={"model": model, "prompt": text},
        )
        resp.raise_for_status()
        embeddings.append(resp.json()["embedding"])
    return embeddings


# ── 方案二：用 sentence-transformers（更灵活） ──
from sentence_transformers import SentenceTransformer

model = SentenceTransformer("BAAI/bge-large-zh-v1.5")

def embed_local(texts: list[str]) -> list[list[float]]:
    """使用本地模型批量生成嵌入。"""
    return model.encode(texts, normalize_embeddings=True).tolist()


# ── 方案三：用 OpenAI 兼容 API ──
import openai

client = openai.OpenAI(base_url="http://localhost:11434/v1", api_key="ollama")

def embed_openai(texts: list[str], model: str = "nomic-embed-text") -> list[list[float]]:
    """通过 OpenAI 兼容 API 生成嵌入。"""
    response = client.embeddings.create(model=model, input=texts)
    return [item.embedding for item in response.data]
```

### 4.3 嵌入的关键原则

```python
# ❌ 错误：Query 用模型 A 嵌入，文档用模型 B 嵌入
query_vec = embed_with_model_A("什么是 CORS？")
doc_vecs = embed_with_model_B(documents)
# 结果：向量空间不同，相似度计算无意义！

# ✅ 正确：Query 和 Document 必须用同一个嵌入模型
query_vec = embed_with_bge("什么是 CORS？")
doc_vecs = embed_with_bge(documents)
```

**关键原则**：
1. **Query 和 Document 必须使用同一个嵌入模型**——不同模型的向量空间不兼容
2. **维度必须一致**——这是数学约束
3. **更换嵌入模型 = 需要重建整个向量索引**——和更换 PDF 解析器一样（C2 §5.3）
4. **embedding 和 chat 是两种不同的模型**——不要用 chat 模型做嵌入

**本节小结（三张卡片）**

- **一句话定义**：嵌入模型 = 把文本映射到向量空间的"编译器"，查询和文档必须用同一个。
- **典型场景**：中文知识库用 bge-large-zh，换成英文模型后检索质量暴跌——语言匹配很重要。
- **和 A1 类比**：嵌入模型像 **序列化协议**（如 JSON vs Protobuf），序列化和反序列化必须用同一套协议。

## 5. 向量数据库对比与选型

**本节概念**：向量数据库 = 高维向量的存储 + 近似最近邻搜索（ANN）。不同选型的差异在 **部署复杂度、规模上限、过滤能力**。

| 数据库 | 类型 | 部署方式 | 适用场景 | 关键特点 |
|--------|------|----------|----------|---------|
| **FAISS** | 库 | 嵌入应用 | 原型/小规模 | Meta 开源，纯内存，速度极快 |
| **ChromaDB** | 轻量DB | 嵌入/单机 | 原型/中小规模 | Python 原生，开箱即用 |
| **Qdrant** | 专用DB | Docker/Cloud | 生产环境 | Rust 编写，过滤+多租户 |
| **Milvus** | 专用DB | K8s/Cloud | 大规模生产 | 分布式，亿级向量 |
| **pgvector** | PG扩展 | PostgreSQL | 已有 PG 的团队 | SQL 查询向量 |
| **Pinecone** | 托管 | Cloud | 不想运维 | 全托管，按用量计费 |

**选型决策树**：

```
你的向量数 < 10万？
├── 是 → ChromaDB（嵌入式，无需额外服务）
└── 否 → 你已有 PostgreSQL？
          ├── 是 → pgvector（复用现有基础设施）
          └── 否 → 需要分布式？

                    └── 否 → Qdrant（Docker 单机）
```

## 6. 检索策略

**本节概念**：基础的 Top-K 相似度搜索只是起点。**MMR 多样化、混合检索、重排序** 是生产级 RAG 的三板斧。

### 6.1 相似度搜索（Similarity Search）

最基础的检索方式：计算 Query 向量与所有文档向量的距离，返回 Top-K 最近邻。

常用距离度量：

| 度量 | 公式 | 特点 |
|------|------|------|
| **余弦相似度** | `dot(a,b) / (||a|| * ||b||)` | 最常用，衡量方向一致性 |
| **欧式距离（L2）** | `sqrt(sum((a-b)^2))` | 衡量绝对距离 |
| **内积（Dot Product）** | `sum(a * b)` | 归一化后等价于余弦 |

### 6.2 MMR（最大边际相关性）

**问题**：Top-K 结果可能高度重复（都在说同一件事）。

```python
def mmr_search(
    query_embedding: list[float],
    candidate_embeddings: list[list[float]],
    candidate_texts: list[str],
    top_k: int = 5,
    lambda_mult: float = 0.7,
) -> list[int]:
    """
    MMR 检索：在相关性和多样性之间平衡。
    lambda_mult 越大越偏向相关性，越小越偏向多样性。
    """
    import numpy as np
    
    query = np.array(query_embedding)
    candidates = np.array(candidate_embeddings)
    
    # 计算所有候选与 query 的相似度
    query_sims = np.dot(candidates, query) / (
        np.linalg.norm(candidates, axis=1) * np.linalg.norm(query) + 1e-9
    )
    
    selected = []
    remaining = list(range(len(candidates)))
    
    for _ in range(min(top_k, len(candidates))):
        best_score = -float("inf")
        best_idx = -1
        
        for idx in remaining:
            # 与 query 的相关性
            relevance = query_sims[idx]
            
            # 与已选结果的最大相似度（多样性惩罚）
            if selected:
                selected_vecs = candidates[selected]
                max_sim_to_selected = max(
                    np.dot(candidates[idx], sv) / (
                        np.linalg.norm(candidates[idx]) * np.linalg.norm(sv) + 1e-9
                    )
                    for sv in selected_vecs
                )
            else:
                max_sim_to_selected = 0.0
            
            # MMR 公式
            score = lambda_mult * relevance - (1 - lambda_mult) * max_sim_to_selected
            
            if score > best_score:
                best_score = score
                best_idx = idx
        
        selected.append(best_idx)
        remaining.remove(best_idx)
    
    return selected
```

### 6.3 混合检索（Hybrid Search）

**问题**：纯向量检索在精确关键词匹配上表现不佳。比如搜"RTX-4090"这种产品型号，语义嵌入可能不如关键词匹配准。

**解决方案**：同时使用向量检索和关键词检索，融合结果。

```python
def hybrid_search(
    query: str,
    query_embedding: list[float],
    vector_store,                  # 向量检索
    keyword_index,                 # 关键词检索（BM25）
    top_k: int = 5,
    alpha: float = 0.7,           # 向量权重
) -> list[dict]:
    """
    混合检索：向量 + 关键词，RRF 融合。
    alpha 控制向量检索的权重（0.7 = 向量为主，关键词辅助）。
    """
    # 两路并行检索
    vector_results = vector_store.search(query_embedding, top_k=top_k * 2)
    keyword_results = keyword_index.search(query, top_k=top_k * 2)
    
    # RRF（Reciprocal Rank Fusion）融合
    k = 60  # RRF 常数
    scores: dict[str, float] = {}
    
    for rank, (chunk, _) in enumerate(vector_results):
        scores[chunk.chunk_id] = scores.get(chunk.chunk_id, 0) + alpha / (k + rank + 1)
    
    for rank, (chunk_id, _) in enumerate(keyword_results):
        scores[chunk_id] = scores.get(chunk_id, 0) + (1 - alpha) / (k + rank + 1)
    
    # 按融合分数排序
    sorted_ids = sorted(scores, key=scores.get, reverse=True)[:top_k]
    return sorted_ids
```

### 6.4 重排序（Reranker）

两阶段检索是生产级 RAG 的标配：

```
阶段 1（召回）：向量检索 → Top-50 候选（快但粗）
阶段 2（精排）：Cross-Encoder 逐对打分 → Top-5 结果（慢但准）
```

```python
# 为什么需要两阶段？
# Bi-Encoder（嵌入模型）：query 和 doc 分别嵌入，速度快，但精度有限
# Cross-Encoder（Reranker）：query 和 doc 拼接后一起进模型，精度高，但计算成本大

# Bi-Encoder: O(1) 查询（预计算+ANN索引）
# Cross-Encoder: O(N) 查询（N 个候选逐一打分）
# 所以：Bi-Encoder 筛选大量候选 → Cross-Encoder 精排少量候选
```

常用 Reranker：
- `BAAI/bge-reranker-v2-m3`（开源，支持中文）
- Cohere Rerank API（托管服务）
- `cross-encoder/ms-marco-MiniLM-L-6-v2`（轻量英文）

**本节小结（三张卡片）**

- **一句话定义**：检索策略 = **召回广度 × 精排准度 × 多样性平衡**。
- **典型场景**：用户搜"RTX-4090价格"，向量检索返回"显卡性能对比"，关键词检索命中"RTX-4090 售价 12999"——混合检索互补。
- **和 A1 类比**：两阶段检索像 **`filter` + `sorted`** —— 先粗筛减小候选集，再精排。

## 7. RAG 评测

**本节概念**：不评测的 RAG = 不测试的代码。你不知道改了分块策略后效果是变好了还是变差了。

### 7.1 核心指标

| 指标 | 公式 | 衡量什么 |
|------|------|---------|
| **Hit Rate（命中率）** | 正确答案在 Top-K 中出现的比例 | 检索覆盖能力 |
| **MRR（Mean Reciprocal Rank）** | 1/正确答案首次出现的排名，取均值 | 排名质量 |
| **NDCG** | 归一化折损累计增益 | 多级相关性排名 |
| **Latency** | 端到端响应时间 | 用户体验 |
| **Token 成本** | 检索片段占用的 Token 数 | 运营成本 |

### 7.2 构建评测数据集

```python
# 评测数据集的最小结构
test_cases = [
    {
        "query": "FastAPI 怎么配置 CORS？",
        "expected_chunks": ["fastapi-cors-001", "fastapi-cors-002"],  # 应该命中的 chunk
        "expected_answer_contains": ["CORSMiddleware", "allow_origins"],  # 答案应包含的关键词
    },
    {
        "query": "Docker Compose 和 Kubernetes 的区别",
        "expected_chunks": ["devops-compare-001"],
        "expected_answer_contains": ["编排", "集群"],
    },
]


def evaluate_retrieval(
    test_cases: list[dict],
    retriever,
    top_k: int = 5,
) -> dict:
    """评估检索质量。"""
    hits = 0
    mrr_sum = 0.0
    
    for case in test_cases:
        results = retriever.search(case["query"], top_k=top_k)
        result_ids = [r.chunk_id for r in results]
        
        # Hit Rate
        hit = any(eid in result_ids for eid in case["expected_chunks"])
        if hit:
            hits += 1
        
        # MRR
        for rank, rid in enumerate(result_ids, 1):
            if rid in case["expected_chunks"]:
                mrr_sum += 1.0 / rank
                break
    
    n = len(test_cases)
    return {
        "hit_rate": hits / n,
        "mrr": mrr_sum / n,
        "total_cases": n,
    }
```

### 7.3 RAGAS 框架

RAGAS（Retrieval Augmented Generation Assessment）是专门评测 RAG 系统的开源框架，提供四个核心维度：

| 维度 | 评估什么 | 不需要什么 |
|------|---------|-----------|
| **Faithfulness（忠实度）** | 回答是否基于检索到的上下文 | 不需要标准答案 |
| **Answer Relevance（答案相关性）** | 回答与问题的匹配程度 | 不需要标准答案 |
| **Context Precision（上下文精确度）** | 检索到的上下文中有多少是有用的 | 需要标准答案 |
| **Context Recall（上下文召回率）** | 回答所需的信息是否都被检索到了 | 需要标准答案 |

```python
from ragas import evaluate
from ragas.metrics import faithfulness, answer_relevancy, context_precision, context_recall

result = evaluate(
    dataset,
    metrics=[faithfulness, answer_relevancy, context_precision, context_recall],
)
print(result)
# {'faithfulness': 0.85, 'answer_relevancy': 0.92, ...}
```

**本节小结（三张卡片）**

- **一句话定义**：RAG 评测 = **检索质量（Hit Rate/MRR）+ 生成质量（Faithfulness/Relevance）**。
- **典型场景**：改了 chunk_size 从 512 到 256，Hit Rate 从 0.72 提升到 0.85——这就是数据驱动迭代。
- **和 A1 类比**：评测数据集像 **单元测试用例**，每次改动都跑一遍，确保没有退步。

## 8. 常见问题与解决

### 8.1 幻觉（Hallucination）

**本节概念**：RAG 降低的是「**无依据硬编**」的概率，不是数学上消灭幻觉；若检索片段噪声大、指令不硬、或模型习惯「填满空」，仍会编。

**缓解方案（按优先级）**：

1. **接地与拒答**：System prompt 中明确 **仅允许根据所给 CONTEXT 断言**；CONTEXT 不足以回答时，必须输出 **「无法从材料中得出」** 并说明缺什么。

```python
SYSTEM_PROMPT = """
你是一个知识库问答助手。请严格基于以下 CONTEXT 回答用户问题。

规则：
1. 只使用 CONTEXT 中的信息回答
2. 每个观点必须标注来源 [chunk_id]
3. 如果 CONTEXT 不足以回答，回复"根据现有资料无法回答此问题"
4. 不要编造任何信息

CONTEXT:
{context}
"""
```

2. **可引用输出**：要求每条结论带 **chunk_id / 标题 / 页码** 等可追溯字段
3. **检索质量**：噪声片段会「带偏」生成——先用 Reranker 过滤
4. **评测闭环**：用 Faithfulness / Context Recall 做回归
5. **后处理**：对关键字段（数字、日期）用规则或二次校验

### 8.2 检索噪声

**问题**：检索到了不相关的片段，干扰生成质量。

```python
# 解决方案：相似度阈值过滤
def search_with_threshold(
    store,
    query_embedding: list[float],
    top_k: int = 5,
    min_score: float = 0.3,  # 低于此分数的结果丢弃
) -> list[tuple]:
    """带阈值的检索——质量不够的结果不要。"""
    results = store.search(query_embedding, top_k=top_k)
    filtered = [(chunk, score) for chunk, score in results if score >= min_score]
    
    if not filtered:
        # 没有足够好的结果，返回空而不是凑数
        return []
    return filtered
```

### 8.3 上下文窗口溢出与注意力稀释

**问题 A：溢出** — 检索结果 + 历史 + 指令 **超过** Context Window，请求直接失败。

**问题 B：注意力稀释** — 未溢出但 **窗内 token 过多、噪声大**，模型对关键证据的利用下降。

**缓解方案**：

```python
def build_context(
    chunks: list[str],
    max_context_tokens: int = 3000,
    tokenizer = None,
) -> str:
    """构建不超过 token 上限的上下文。"""
    context_parts = []
    current_tokens = 0
    
    for i, chunk in enumerate(chunks):
        chunk_tokens = len(tokenizer.encode(chunk)) if tokenizer else len(chunk) // 2
        
        if current_tokens + chunk_tokens > max_context_tokens:
            break  # 够了，不要再加
        
        context_parts.append(f"[{i+1}] {chunk}")
        current_tokens += chunk_tokens
    
    return "\n\n".join(context_parts)
```

**核心原则**：RAG 的正确姿势是 **「少而精的证据窗」**，不是「把能搜到的都塞进去」。

### 8.4 多轮对话中的 RAG

**问题**：用户的后续问题依赖前文（如 "它的价格是多少？"），直接检索效果差。

```python
def rewrite_query(
    client,
    chat_history: list[dict],
    current_question: str,
) -> str:
    """用 LLM 将多轮对话压缩为独立查询。"""
    response = client.chat.completions.create(
        model="qwen3:8b",
        messages=[
            {"role": "system", "content": (
                "将用户的后续问题改写为独立的搜索查询，"
                "结合对话历史补全指代和省略。"
                "只输出改写后的查询，不要解释。"
            )},
            *chat_history[-4:],  # 最近几轮对话
            {"role": "user", "content": f"改写此问题：{current_question}"},
        ],
        temperature=0,
    )
    return response.choices[0].message.content.strip()

# 示例：
# 对话历史: "介绍一下 MacBook Pro" → "它有什么配置？"
# 改写后: "MacBook Pro 的配置有哪些？"  ← 可以独立检索了
```

**本节小结（三张卡片）**

- **一句话定义**：RAG 防幻觉 = **证据可追溯 + 检索干净 + 拒答写死**；防稀释 = **少 token、好排序、关键指令不靠中间**。
- **典型场景**：Top-10 片段全进 prompt，用户只问一句「是否支持 X」，模型综合了 9 段无关政策 —— 应先精排再进窗。
- **和 A1 类比**：CONTEXT 拼装像 **`join` 前列表要先 `filter`**，否则字符串再长也只是噪声拼接。

## 9. 高级 RAG 技术

### 9.1 HyDE（Hypothetical Document Embeddings）

**问题**：用户的短查询（如 "Python GIL"）嵌入质量不如长文档——短文本的向量表征力弱。

**思路**：先让 LLM 生成一个"假想文档"（回答该问题的理想段落），再用假想文档的嵌入去检索。

```python
def hyde_search(
    client,
    query: str,
    embed_fn: callable,
    vector_store,
    top_k: int = 5,
) -> list:
    """HyDE：用假想文档的嵌入做检索。"""
    # 第 1 步：让 LLM 生成假想回答
    response = client.chat.completions.create(
        model="qwen3:8b",
        messages=[
            {"role": "system", "content": "请为以下问题写一段详细的技术说明（约 200 字）。"},
            {"role": "user", "content": query},
        ],
    )
    hypothetical_doc = response.choices[0].message.content
    
    # 第 2 步：用假想文档的嵌入做检索（而不是用短查询的嵌入）
    hypo_embedding = embed_fn([hypothetical_doc])[0]
    return vector_store.search(hypo_embedding, top_k=top_k)
```

优点：检索质量显著提升，尤其适合短查询。
缺点：多一次 LLM 调用，增加延迟和成本。

### 9.2 Self-RAG

让 LLM 自己决定是否需要检索，以及对检索结果进行自我评估：

1. **Retrieve Token**：LLM 判断当前问题是否需要检索（有些问题不需要）
2. **ISREL Token**：评估检索到的片段是否相关
3. **ISSUP Token**：评估生成的回答是否被检索结果支持
4. **ISUSE Token**：评估回答是否有用

通过这些"反思 Token"，模型可以自适应地决定检索策略。

### 9.3 Graph RAG

将文档中的实体和关系抽取为知识图谱，检索时不仅找相似片段，还能沿图谱进行关系推理。

```
文档 → 实体抽取 → 关系抽取 → 知识图谱
查询 → 实体识别 → 子图检索 → 图上下文 + 向量上下文 → LLM 生成
```

适用场景：
- 多跳推理（"A 的老板的公司在哪个城市？"）
- 实体关系密集的领域（法律、医疗、金融）

Microsoft 的 GraphRAG 是代表性开源实现，使用 LLM 构建社区摘要层级。

**本节小结（三张卡片）**

- **一句话定义**：高级 RAG = **在基础检索之上叠加推理能力**（假想文档、自评估、图推理）。
- **典型场景**：法律问答需要跨文档推理"合同 A 引用的法条 B 是否已修订"→ Graph RAG。
- **和 A1 类比**：HyDE 像 **模板方法模式** —— 先生成一个"理想输出"做模板，再用模板去匹配真实数据。

## 10. 实战建议

**本节概念**：RAG 不是"搭完就能用"的系统，而是需要 **持续调优** 的工程。以下是按优先级排列的实战建议。

### 10.1 启动清单

```
第 1 步：跑通最简版本
  ChromaDB + 固定分块(512) + Ollama 嵌入 → 验证端到端可行性

第 2 步：建立评测数据集
  准备 20-50 个 (query, expected_chunks, expected_answer) 测试用例
  每次改动都跑评测，用数据说话

第 3 步：优化分块策略
  按文档类型选择分块器：Markdown 用标题分块，PDF 用段落分块
  chunk_size 在 256-1024 之间搜索最优值

第 4 步：加混合检索
  BM25 + 向量检索 + RRF 融合，几乎总能提升效果

第 5 步：加 Reranker
  性价比最高的精度提升手段，Top-50 召回 → Reranker 精排 → Top-5

第 6 步：监控生产指标
  延迟、Token 成本、用户满意度、Faithfulness
```

### 10.2 成本控制

```python
# 嵌入和 LLM 调用都有成本，需要关注
# 
# 嵌入成本计算：
# 1000 篇文档 × 平均 5 个 chunk × 平均 300 token/chunk = 150 万 token
# text-embedding-3-small: $0.02/1M tokens → $0.03
# text-embedding-3-large: $0.13/1M tokens → $0.20
# 
# LLM 成本计算（每次查询）：
# System prompt: ~200 token
# Context (5 chunks × 300 token): ~1500 token
# 用户问题: ~50 token
# 总输入: ~1750 token
# 输出: ~300 token
# 
# 优化方向：
# 1. 缓存频繁查询的嵌入结果
# 2. 控制 Top-K 和 chunk_size 降低输入 token
# 3. 嵌入用便宜的小模型，LLM 用强模型
```

### 10.3 完整 RAG Pipeline 代码

```python
"""
端到端 RAG Pipeline 示例（使用 ChromaDB + Ollama）。
这是把前面所有章节串起来的完整代码。
"""
import json
import chromadb
import requests


class RAGPipeline:
    """最小可用的 RAG 系统。"""
    
    def __init__(
        self,
        collection_name: str = "knowledge_base",
        ollama_base: str = "http://localhost:11434",
        embed_model: str = "nomic-embed-text",
        chat_model: str = "qwen3:8b",
    ):
        self.client = chromadb.PersistentClient(path="./rag_data")
        self.collection = self.client.get_or_create_collection(collection_name)
        self.ollama_base = ollama_base
        self.embed_model = embed_model
        self.chat_model = chat_model
    
    def _embed(self, text: str) -> list[float]:
        """调用 Ollama 生成嵌入。"""
        resp = requests.post(
            f"{self.ollama_base}/api/embeddings",
            json={"model": self.embed_model, "prompt": text},
        )
        resp.raise_for_status()
        return resp.json()["embedding"]
    
    def _chat(self, messages: list[dict]) -> str:
        """调用 Ollama 生成回答。"""
        resp = requests.post(
            f"{self.ollama_base}/api/chat",
            json={"model": self.chat_model, "messages": messages, "stream": False},
        )
        resp.raise_for_status()
        return resp.json()["message"]["content"]
    
    # ── 离线：索引文档 ──
    def index_documents(self, documents: list[dict]):
        """
        索引文档到向量库。
        documents: [{"id": "doc1", "content": "...", "metadata": {...}}, ...]
        """
        for doc in documents:
            embedding = self._embed(doc["content"])
            self.collection.add(
                ids=[doc["id"]],
                embeddings=[embedding],
                documents=[doc["content"]],
                metadatas=[doc.get("metadata", {})],
            )
    
    # ── 在线：查询 ──
    def query(self, question: str, top_k: int = 3) -> dict:
        """端到端查询：检索 → 拼 Prompt → 生成。"""
        
        # 1. 检索
        query_embedding = self._embed(question)
        results = self.collection.query(
            query_embeddings=[query_embedding],
            n_results=top_k,
        )
        
        # 2. 拼装 Context
        context_parts = []
        sources = []
        for i, (doc, meta) in enumerate(
            zip(results["documents"][0], results["metadatas"][0])
        ):
            context_parts.append(f"[{i+1}] {doc}")
            sources.append(meta)
        
        context = "\n\n".join(context_parts)
        
        # 3. 生成
        messages = [
            {"role": "system", "content": (
                "你是一个知识库问答助手。请严格基于以下 CONTEXT 回答用户问题。\n"
                "规则：1) 只使用 CONTEXT 中的信息 2) 标注来源编号 [n] "
                "3) 信息不足时说'无法回答'\n\n"
                f"CONTEXT:\n{context}"
            )},
            {"role": "user", "content": question},
        ]
        
        answer = self._chat(messages)
        
        return {
            "answer": answer,
            "sources": sources,
            "chunks_used": len(context_parts),
        }
```

**与 A1 的节奏对齐（收束全章）**：RAG 调参本质是 **「数据管道 + 检索器 + 生成器」三段各自可测**；A1 的习惯是把每段写成 **可单测的纯函数**，再用集成测试串起来。

**本节小结（三张卡片）**

- **一句话定义**：C3 的主线是可观测、可迭代的 **检索增强闭环**。
- **典型场景**：改 chunk_size 只动索引，不重训模型，用 Hit Rate 证明收益。
- **和 A1 类比**：索引流水线像 **构建中间表示（IR）**，检索像 **`dict.get` 的模糊版**，生成像 **解释执行 IR**。

## 13. 面试高频 Q&A

### Q1：RAG 和 Fine-tuning 怎么选？什么时候需要结合使用？

**A**：核心判断标准是 **知识的更新频率和类型**：

| 场景 | 选择 | 原因 |
|------|------|------|
| 知识频繁更新（每周/每天） | RAG | 换文档即可，不用重训 |
| 需要特定回答风格/格式 | Fine-tuning | 风格在权重里，RAG 加 prompt 效果有限 |
| 知识更新 + 特定风格 | RAG + Fine-tuning | 先 LoRA 调风格，再 RAG 注入事实 |
| 训练数据少于 100 条 | RAG | Fine-tuning 数据量不够 |

结合使用的典型案例：医疗问答。先用 LoRA 让模型学会医学术语和专业回答格式，再用 RAG 注入最新药品说明书和诊疗指南。

### Q2：你的 RAG 系统检索效果不好，怎么排查？

**A**：分层排查，从数据到模型：

```
第 1 层：数据质量
  → 查看 chunk 内容，是否有噪声（导航栏、页脚）？
  → 检查 C2 的清洗流水线是否到位

第 2 层：分块策略
  → chunk 是否完整？关键信息是否被切断？
  → 用几个典型 query 手动检查 Top-5 的 chunk

第 3 层：嵌入模型
  → query 和 doc 是否用同一个模型？
  → 嵌入模型的语言是否匹配？（中文文档用英文模型？）

第 4 层：检索策略
  → Top-K 是否太小？相似度阈值是否太高？
  → 是否需要混合检索？（关键词匹配场景）

第 5 层：Prompt 拼装
  → Context 是否太长导致注意力稀释？
  → System prompt 是否有明确的引用指令？
```

### Q3：向量数据库怎么选型？生产环境用什么？

**A**：看三个维度——数据规模、团队技术栈、运维能力。

- **< 10 万向量 + 小团队**：ChromaDB（嵌入式，零运维）
- **10-100 万向量 + 已有 PostgreSQL**：pgvector（复用现有 DB，SQL 生态兼容）
- **10-100 万向量 + 需要专业功能**：Qdrant（Docker 单机，支持过滤/多租户）
- **> 100 万向量**：Milvus（分布式）或 Pinecone（全托管）

面试加分点：能说清楚 pgvector 的优势——不需要额外引入向量数据库，和业务数据在同一个 PostgreSQL 里，事务一致性有保障，运维成本最低。

### Q4：分块大小怎么定？有没有公式？

**A**：没有公式，但有方法论：

1. **经验起点**：256-512 tokens
2. **搜索法**：用评测数据集，在 128/256/512/1024 四个值上比较 Hit Rate
3. **按文档类型调整**：
   - API 文档：256（每个函数/参数独立成 chunk）
   - 教程/文章：512（保持段落完整）
   - 法律合同：1024（条款通常较长）
4. **Overlap**：chunk_size 的 15-20%

关键认知：**分块是需要 A/B 测试的超参数，不是一次定死的配置**。

### Q5：如何实现多租户 RAG？不同客户的数据要隔离。

**A**：通过 **元数据过滤** 实现逻辑隔离：

```python
# 索引时打 tenant 标签
collection.add(
    ids=["doc1"],
    documents=["..."],
    metadatas=[{"tenant_id": "company_a"}],
)

# 检索时必须带 tenant 过滤
results = collection.query(
    query_embeddings=[query_vec],
    n_results=5,
    where={"tenant_id": "company_a"},  # 只搜本租户的数据
)
```

**关键点**：过滤必须在相似度计算之前（`pre-filter`），而不是之后（`post-filter`）。Post-filter 会导致 Top-K 里大量结果被过滤掉，实际返回远少于 K 条。

物理隔离方案：每个租户一个独立的 collection / namespace，更安全但管理复杂度高。

### Q6：RAG 的 Prompt 怎么设计？有什么最佳实践？

**A**：三段式结构：

```
1. System prompt（角色 + 规则 + 拒答指令）
2. Context（检索到的 chunk，带编号）
3. User question（用户原始问题）
```

最佳实践：
- **拒答指令必须写死**："如果 CONTEXT 不足以回答，请说'无法回答'"
- **要求标注来源**：每个观点后标 `[1]` `[2]` 引用 chunk 编号
- **Context 放中间**：关键指令和问题放在开头和结尾（attention 在两端更强）
- **控制 Context 长度**：3-5 个 chunk 通常够，不要塞 10+
