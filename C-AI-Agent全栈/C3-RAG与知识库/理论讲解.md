## 0. 与 A1、C2 的衔接

**本节概念**：**RAG** 把「读文档」从训练期搬到推理期：用 **检索** 选出证据片段，再让 LLM **只作为推理与表达层**；**数据处理（C2）** 决定证据是否干净，**向量与索引** 决定证据是否可命中。

如果你在 A1 里已经习惯「**先写纯函数处理数据，再拼 I/O**」，那么 RAG 索引流水线几乎同构：**Loader/Splitter/Embedder** 是一条纯函数链，出错时应该能 **单元测试每一环**；`list[str]` 进、`ndarray` 或向量库句柄出，边界用类型标清楚，排障会快很多。C2 强调的 **元数据与 ACL** 不是锦上添花：没有 `tenant_id` 过滤的向量检索，和「多线程共享可变全局 dict 不加密」是同一类事故。

**本节小结（三张卡片）**

- **一句话定义**：RAG = **可换源的体外记忆** + **可控引用的生成**。
- **典型场景**：内部 wiki 日更，模型权重不能日更；用重建索引代替重训。
- **和 A1 类比**：**向量库**像 **把文档摘要成 key，放进 dict 再按相似 key 取 value**；**重排序**像 **在 `sorted(..., key=...)` 里换一个更贵但更准的 key 函数**。

## 1. RAG 概述

### 1.1 为什么需要 RAG

**本节概念**：**知识截止**与**私域缺失**是参数记忆的硬边界；**RAG**用外部语料把「事实」从权重里挪到「上下文窗口里的证据」。

大语言模型（LLM）的知识来自训练数据，存在两个根本性问题：

1. **知识截止**：模型无法知道训练截止日期之后的信息。
2. **私域知识缺失**：企业内部文档、行业数据库、个人笔记等从未出现在公开训练集中。

RAG（Retrieval-Augmented Generation，检索增强生成）的核心思路：**先检索，再生成**。把用户问题送到外部知识库中检索相关片段，将检索结果拼入 Prompt，让 LLM 基于"真实证据"来回答，而非纯粹依赖参数记忆。

优势：
- 无需重新训练模型，成本远低于 Fine-tuning
- 知识可以实时更新（替换文档即可）
- 可追溯来源（返回引用片段），降低幻觉风险（工程上仍需 **引用格式、拒答策略、评测** 闭环，见本章 [§8](#8-常见问题与解决) 与 [C1 §5.5](../C1-LLM基础与Prompt/理论讲解.md)）

### 1.2 RAG vs Fine-tuning

| 维度 | RAG | Fine-tuning |
|------|-----|-------------|
| 知识更新 | 替换文档即可，秒级生效 | 需重新训练，小时/天级 |
| 成本 | 低（向量数据库 + 嵌入 API） | 高（GPU 算力 + 数据标注） |
| 适用场景 | 事实问答、文档检索、客服 | 风格迁移、领域术语、特殊格式 |
| 幻觉控制 | 较好（有证据可查） | 一般（仍可能编造） |
| 上下文长度 | 受限于 Context Window | 无此限制 |
| 可解释性 | 高（可返回检索来源） | 低 |

**口径提示**：上表里的 **Fine-tuning** 泛指「**要跑训练、会改动权重**」的一类路径；实际 **成本、周期、显存** 从 **全量微调 → LoRA → QLoRA** 差异很大，口语里的「微调」可能指其中任一种。术语与选型分层见 [C1 §2.4](../C1-LLM基础与Prompt/理论讲解.md)。

实际项目中，RAG 和 Fine-tuning 并不互斥。常见组合：先用 **LoRA / QLoRA** 等让模型熟悉领域术语与回答风格，再用 RAG 注入最新事实（全量微调仅在数据与算力允许时考虑）。

### 1.3 RAG 架构（文字描述）

```
用户提问
    │
    ▼
┌──────────┐    Query Embedding     ┌──────────────┐
│  Query   │ ──────────────────────▶│ Embedding    │
│ 处理器    │                        │ Model        │
└──────────┘                        └──────┬───────┘
                                           │ query vector
                                           ▼
                                    ┌──────────────┐
                                    │ Vector Store │ ◀── 离线索引流水线
                                    │ (向量数据库)  │     (文档加载→分块→嵌入→存储)
                                    └──────┬───────┘
                                           │ Top-K 相关片段
                                           ▼
                                    ┌──────────────┐
                                    │ Prompt 组装   │
                                    │ (System +     │
                                    │  Context +    │
                                    │  Question)    │
                                    └──────┬───────┘
                                           │
                                           ▼
                                    ┌──────────────┐
                                    │   LLM 生成    │
                                    └──────┬───────┘
                                           │
                                           ▼
                                       回答 + 引用来源
```

离线索引流水线（Indexing Pipeline）：

```
原始文档 (PDF/Markdown/HTML/TXT)
    │
    ▼
Document Loader（文档加载）
    │
    ▼
Text Splitter（文本分块）
    │
    ▼
Embedding Model（向量化）
    │
    ▼
Vector Store（向量存储）
```

## 2. 文档处理流水线

### 2.1 文档加载（Document Loading）

不同格式需要不同的加载器：

| 格式 | 常用工具/库 |
|------|------------|
| PDF | PyMuPDF (`fitz`)、`pdfplumber`、`unstructured` |
| Markdown | 直接读取、`markdown-it` 解析结构 |
| HTML | `BeautifulSoup`、`trafilatura`（正文提取） |
| DOCX | `python-docx` |
| CSV/Excel | `pandas` |
| 代码文件 | 按语言 AST 解析或直接文本读取 |

加载阶段的关键：**元数据保留**。每个文档片段都应携带来源文件名、页码、章节标题等元数据，便于后续引用追溯。

### 2.2 文本分块（Chunking）

为什么要分块？
- LLM 的 Context Window 有限
- 整篇文档塞入向量数据库后，检索粒度太粗
- 小块文本的嵌入向量更精确

分块是 RAG 效果的关键瓶颈 —— 分得太大，检索噪声多；分得太小，丢失上下文。

### 2.3 嵌入（Embedding）

将文本块转为固定维度的浮点向量（如 768 维、1536 维），使得语义相近的文本在向量空间中距离更近。

### 2.4 向量存储（Vector Store）

将嵌入向量及其对应的文本、元数据持久化存储，并提供高效的近似最近邻（ANN）搜索能力。

---

## 3. 分块策略详解

### 3.1 固定大小分块（Fixed-size Chunking）

```
参数：chunk_size=500, chunk_overlap=50
```

按字符数或 Token 数将文本切成等长片段，相邻片段有 Overlap（重叠）防止语义被截断。

- 优点：实现简单、可预测
- 缺点：会在句子/段落中间截断，破坏语义完整性

### 3.2 递归分割（Recursive Character Splitting）

按优先级尝试不同分隔符：`\n\n` → `\n` → `.` → ` ` → 字符级别。优先在段落边界分割，其次句子边界，最后才逐字符切割。

LangChain 的 `RecursiveCharacterTextSplitter` 就是这个策略。

- 优点：保持语义边界，适用面广
- 缺点：块大小不完全均匀

### 3.3 按语义分块（Semantic Chunking）

计算相邻句子的嵌入相似度，当相似度低于阈值时在此处分割。

- 优点：每个块内语义高度一致
- 缺点：需要额外调用嵌入模型，计算成本高

### 3.4 Overlap 重叠

任何分块策略都建议加 Overlap（通常 10%–20%），确保跨块边界的信息不会丢失。例如 chunk_size=500、overlap=100，则第二个块从第一个块的第 400 个字符开始。

### 3.5 Markdown/代码专用分块

- **Markdown**：按标题层级（`#`、`##`、`###`）分割，每个章节为一个块
- **代码**：按函数/类定义分割，保持完整的代码单元
- **表格**：整行保留，不在行中间截断

## 4. 嵌入模型

### 4.1 OpenAI text-embedding

| 模型 | 维度 | 价格 | 特点 |
|------|------|------|------|
| `text-embedding-3-small` | 1536 | $0.02/1M tokens | 性价比高 |
| `text-embedding-3-large` | 3072 | $0.13/1M tokens | 精度最高 |
| `text-embedding-ada-002` | 1536 | $0.10/1M tokens | 旧版，仍广泛使用 |

`text-embedding-3` 系列支持 `dimensions` 参数，可以降维以节省存储（如 3072→256），精度损失有限。

### 4.2 本地嵌入模型

| 模型 | 维度 | 语言 | 说明 |
|------|------|------|------|
| `BAAI/bge-large-zh-v1.5` | 1024 | 中文优 | 中文场景首选 |
| `BAAI/bge-m3` | 1024 | 多语言 | 支持稀疏+稠密混合检索 |
| `sentence-transformers/all-MiniLM-L6-v2` | 384 | 英文 | 轻量快速 |
| `nomic-embed-text` | 768 | 英文 | Ollama 可直接运行 |

使用方式：

```python
from sentence_transformers import SentenceTransformer

model = SentenceTransformer("BAAI/bge-large-zh-v1.5")
embeddings = model.encode(["你好世界", "Hello World"])
```

### 4.3 嵌入维度与质量

- 维度越高，表达能力越强，但存储和计算成本也越高
- 384 维足以应对简单场景；768–1024 维是生产常用区间；3072 维用于对精度极致追求的场景
- **关键原则**：Query 和 Document 必须使用同一个嵌入模型，维度必须一致

## 6. 检索策略

### 6.1 相似度搜索（Similarity Search）

最基础的检索方式：计算 Query 向量与所有文档向量的距离，返回 Top-K 最近邻。

常用距离度量：
- **余弦相似度（Cosine Similarity）**：衡量方向一致性，最常用
- **欧式距离（L2 Distance）**：衡量绝对距离
- **内积（Inner Product / Dot Product）**：归一化后等价于余弦相似度

### 6.2 MMR（Maximal Marginal Relevance，最大边际相关性）

问题：Top-K 结果可能高度重复（都在说同一件事）。

MMR 在选择下一个结果时，同时考虑：
- 与 Query 的相关性（越高越好）
- 与已选结果的差异性（越不同越好）

```
MMR = argmax [λ · sim(doc, query) - (1-λ) · max(sim(doc, selected_docs))]
```

λ 越大越偏向相关性，λ 越小越偏向多样性。

### 6.3 混合检索（Hybrid Search）

单纯的向量检索在精确关键词匹配上表现不佳（如产品型号 "RTX-4090"）。混合检索同时使用：

1. **向量检索**（语义层面）：捕捉语义相似性
2. **关键词检索**（词汇层面）：BM25、TF-IDF 等传统信息检索方法

融合策略：
- **RRF（Reciprocal Rank Fusion）**：`score = Σ 1/(k + rank_i)`，简单有效
- **加权求和**：`final = α × vector_score + (1-α) × keyword_score`

### 6.4 重排序（Reranker）

两阶段检索：
1. **召回**：向量检索返回 Top-50 候选
2. **精排**：用 Cross-Encoder 对每个 (query, doc) 对精确打分，重新排序后取 Top-5

常用 Reranker：
- `BAAI/bge-reranker-v2-m3`（开源）
- Cohere Rerank API（托管）
- `cross-encoder/ms-marco-MiniLM-L-6-v2`（轻量）

Reranker 的精度远高于 Bi-Encoder（嵌入模型），但计算成本也高，因此只对召回的小批量候选做精排。

## 7. RAG 评测

### 7.1 核心指标

| 指标 | 公式 | 含义 |
|------|------|------|
| **Hit Rate（命中率）** | 正确答案在 Top-K 中出现的比例 | 检索覆盖能力 |
| **MRR（Mean Reciprocal Rank）** | 1/正确答案首次出现的排名，取均值 | 排名质量 |
| **NDCG** | 归一化折损累计增益 | 多级相关性排名质量 |
| **Latency** | 端到端响应时间 | 用户体验 |
| **Token 成本** | 检索片段占用的 Token 数 | 运营成本 |

### 7.2 RAGAS 框架

RAGAS（Retrieval Augmented Generation Assessment）是专门评测 RAG 系统的开源框架，提供四个核心维度：

1. **Faithfulness（忠实度）**：回答是否基于检索到的上下文，而非自行编造
2. **Answer Relevance（答案相关性）**：回答与问题的匹配程度
3. **Context Precision（上下文精确度）**：检索到的上下文中有多少是真正有用的
4. **Context Recall（上下文召回率）**：回答所需的信息是否都被检索到了

```python
from ragas import evaluate
from ragas.metrics import faithfulness, answer_relevancy, context_precision, context_recall

result = evaluate(dataset, metrics=[faithfulness, answer_relevancy, context_precision, context_recall])
```

## 8. 常见问题与解决

### 8.1 幻觉（Hallucination）

**问题**：LLM **忽略检索结果**、**把多条证据捏合成错误因果**，或在**上下文未覆盖**时仍给出确定结论。

**本节概念**：RAG 降低的是「**无依据硬编**」的概率，不是数学上消灭幻觉；若检索片段噪声大、指令不硬、或模型习惯「填满空」，仍会编。

**缓解方案（按优先级 roughly 从高到低）**：

1. **接地与拒答**：System + 用户任务中明确 **仅允许根据所给 CONTEXT 断言**；CONTEXT 不足以回答时，必须输出 **「无法从材料中得出」** 并说明缺什么（与 [C1 §5.5](../C1-LLM基础与Prompt/理论讲解.md) 一致）。  
2. **可引用输出**：要求每条结论带 **chunk_id / 标题 / 页码** 等可追溯字段，便于人审与自动对账。  
3. **检索质量**：噪声片段会「带偏」生成，见 §8.2；必要时 **重排、阈值、HyDE/Self-RAG**（§9）。  
4. **评测闭环**：用 §7 的 **Faithfulness / Context Recall** 等做回归，避免靠体感验收。  
5. **后处理**：对关键字段（数字、日期、条款编号）用 **规则或二次校验**（与 [C4 §6](../C4-Function-Calling与结构化输出/理论讲解.md) 的结构化校验同构）。

### 8.2 检索噪声

**问题**：检索到了不相关的片段，干扰生成质量。

**缓解方案**：
- 设置相似度阈值，过滤低分结果
- 使用 Reranker 精排
- 优化分块策略，确保每个块语义单一

### 8.3 上下文窗口溢出与注意力稀释

**问题 A：溢出** — 检索结果 + 历史 + 指令 **超过** Context Window，请求直接失败或被截断。

**问题 B：注意力稀释** — 未溢出但 **窗内 token 过多、噪声大** 时，模型对关键证据与约束的**有效利用下降**（成本仍按 token 计）。溢出与稀释常一起出现：**先挤爆预算，再谈质量**。

**本节概念**：RAG 的正确姿势是 **「少而精的证据窗」**，不是「把能搜到的都塞进去」。这与 [C1 §5.5](../C1-LLM基础与Prompt/理论讲解.md) 的「控窗、控噪声」同一逻辑。

**缓解方案**：

- **控量**：限制 **Top-K**、每片段 **最大字符**、总 CONTEXT **硬上限**；超长单 chunk 先 **二次摘要** 再进窗。  
- **精排与过滤**：相似度 **阈值**、**Reranker**（§6）把最相关段留在前面，减少噪声占位。  
- **拼装顺序**：**指令与最终问题**靠近 **开头或结尾**；大段材料中间夹核心问句，往往不如 **结论前置 + 证据附录** 稳。  
- **Map-Reduce / 分块回答再合并**：每段独立抽取要点，最后一轮 **只合并已有要点**，避免单次 mega-prompt。  
- **对话历史**：多轮时先 **Query 改写**（§8.4），不要把全文历史与全文检索结果 **无裁剪堆叠**；长会话用 **摘要态**（见 [C6 §3](../C6-单Agent开发/理论讲解.md)）。

### 8.4 多轮对话中的 RAG

**问题**：用户的后续问题依赖前文（如 "它的价格是多少？"），直接检索效果差。

**缓解方案**：
- **Query 改写**：用 LLM 将多轮对话压缩为独立查询（"MacBook Pro 2024 的价格是多少？"）
- **对话历史拼接**：将最近几轮对话与当前问题拼接后再做检索

### 8.5 与「注意力稀释」相关的评测提示

上线前建议除 Hit Rate / MRR 外，增加 **长 CONTEXT 抽检**：同一问题下，故意把 **关键证据放在材料前、中、后** 或混入 **无关长文**，观察答案是否仍稳定依赖证据（与 needle-in-haystack 类测试同构）。指标上关注 §7 的 **Faithfulness、Context Precision** 是否同步下降；若下降明显，优先 **减窗、加重排、改拼装**，而不是一味换更大模型。

**本节小结（三张卡片）**

- **一句话定义**：RAG 防幻觉 = **证据可追溯 + 检索干净 + 拒答写死**；防稀释 = **少 token、好排序、关键指令不靠「埋中间」**。  
- **典型场景**：Top-10 片段全进 prompt，用户只问一句「是否支持 X」，模型综合了 9 段无关政策 —— 应先精排再进窗。  
- **和 A1 类比**：CONTEXT 拼装像 **`join` 前列表要先 `filter`**，否则字符串再长也只是噪声拼接。

## 9. 高级 RAG 技术

### 9.1 HyDE（Hypothetical Document Embeddings）

思路：用户的短查询（如 "Python GIL"）嵌入质量不如长文档。HyDE 先让 LLM 生成一个"假想文档"（回答该问题的理想段落），再用假想文档的嵌入去检索。

```
用户查询 → LLM 生成假想回答 → 嵌入假想回答 → 向量检索 → 获取真实文档
```

优点：检索质量显著提升，尤其适合短查询。
缺点：多一次 LLM 调用，增加延迟和成本。

### 9.2 Self-RAG

让 LLM 自己决定是否需要检索，以及对检索结果进行自我评估：

1. **Retrieve Token**：LLM 判断当前问题是否需要检索（有些问题不需要）
2. **ISREL Token**：评估检索到的片段是否相关
3. **ISSUP Token**：评估生成的回答是否被检索结果支持
4. **ISUSE Token**：评估回答是否有用

通过这些"反思 Token"，模型可以自适应地决定检索策略。

### 9.3 Graph RAG

将文档中的实体和关系抽取为知识图谱，检索时不仅找相似片段，还能沿图谱进行关系推理。

```
文档 → 实体抽取 → 关系抽取 → 知识图谱
查询 → 实体识别 → 子图检索 → 图上下文 + 向量上下文 → LLM 生成
```

适用场景：
- 多跳推理（"A 的老板的公司在哪个城市？"）
- 实体关系密集的领域（法律、医疗、金融）

Microsoft 的 GraphRAG 是代表性开源实现，使用 LLM 构建社区摘要层级，支持全局性问题的回答。

## 10. 实战建议

1. **先跑通最简版本**：ChromaDB + 固定分块 + OpenAI Embedding，验证端到端可行性
2. **分块策略是调优重点**：不同文档类型用不同分块器，chunk_size 通常在 256–1024 Token 之间
3. **嵌入模型选对语言**：中文场景用 BGE，英文用 OpenAI 或 all-MiniLM
4. **混合检索 > 纯向量检索**：加上 BM25 几乎总能提升效果
5. **Reranker 是性价比最高的精度提升手段**
6. **评测驱动迭代**：建立测试集（问题 + 标准答案 + 标准检索片段），用 Hit Rate/MRR 量化每次改动的效果
7. **注意成本**：嵌入和 LLM 调用都有成本，生产环境要缓存频繁查询的结果

## 与 A1 的节奏对齐（收束全章）

**本节概念**：RAG 调参本质是 **「数据管道 + 检索器 + 生成器」三段各自可测**；A1 的习惯是把每段写成 **可单测的纯函数**，再用集成测试串起来 —— 这与第 10 节「先跑通最简版本」完全一致。

建议你像读 A1 一样：**每读完一大节，默写本节小结三条**（定义 / 场景 / 类比），然后对照 `demos/` 里脚本是否覆盖。高级主题（HyDE、Self-RAG、Graph RAG）先当 **扩展阅读**，主线仍是 **分块—嵌入—检索—拼装 prompt—生成—评测**。

**本节小结（三张卡片）**

- **一句话定义**：C3 的主线是可观测、可迭代的 **检索增强闭环**。
- **典型场景**：改 chunk_size 只动索引，不重训模型，用 Hit Rate 证明收益。
- **和 A1 类比**：索引流水线像 **构建中间表示（IR）**，检索像 **`dict.get` 的模糊版**，生成像 **解释执行 IR**。
