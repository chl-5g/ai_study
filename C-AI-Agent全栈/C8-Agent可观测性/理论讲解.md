## 0. 与 A1、C6、C7 的衔接

**本节概念**：**可观测性**回答「系统内部发生了什么」；对 Agent 而言，内部包括 **检索、工具、模型多轮、路由**。没有统一 **trace_id**，你只能看到「最终答错了」，看不到 **错在哪一跳**。

A1 教你 **`logging` 模块级配置、`extra` 字段、异常链**；C8 把这些升格为：**跨异步任务传播的上下文（contextvars）**、**嵌套耗时（span）**、**聚合指标（metrics）**。单 Agent（C6）是一串 span；多 Agent（C7）是 **树或 DAG**，更需要 **父子 span** 与 **links**。

**为什么可观测性是 Agent 的生死线？**

传统 Web 应用的排障路径很短：看 HTTP 状态码、看数据库慢查询日志，基本就能定位问题。但 Agent 的请求路径是 **非确定性的**：

1. **同一输入，不同路径**：Agent 可能选择不同的工具、不同的检索策略、甚至不同的模型
2. **多跳依赖**：一次请求可能经历 检索 → LLM → 工具调用 → LLM → 再检索 → LLM，任何一跳都可能出错
3. **外部依赖不可控**：模型 API 的延迟、向量库的冷启动、第三方工具的超时——都不是你能控制的
4. **非确定性输出**：同样的 prompt + 同样的检索结果，模型可能给出不同的回答

没有可观测性，排障就是 **盲人摸象**。

**本节小结（三张卡片）**

- **一句话定义**：可观测性 = **用外部输出推断内部状态** 的能力，Agent 只是负载更丰富的服务。
- **典型场景**：用户截图说「引用错了」，你要判断是 rerank 还是模型忽略证据。
- **和 A1 类比**：`trace_id` 像 **给每次请求挂 uuid**，所有 log 行带上它 —— 只是 Agent 里还要穿过 tool 子进程。

## 1. Why：没有观测 = 黑盒

**本节概念**：用户反馈「答错了」时，工程上要回答：**检索命中了吗？调了哪个工具？模型原始 reasoning？耗时在哪一步？** 没有 trace 只能猜，猜的成本高于 **一开始打 span**。

### 1.1 Agent 排障的典型困境

假设你的 RAG Agent 上线后，用户反馈「回答不准确」。没有可观测性时，你只能做以下猜测：

| 可能的原因 | 排查方式（无观测） | 排查方式（有观测） |
|---|---|---|
| 检索没命中相关文档 | 手动复现查询，看检索结果 | 直接看 `retrieve` span 的 `doc_ids` 和 `scores` |
| 检索命中了但 rerank 把相关文档排到后面 | 不知道有没有 rerank | 看 `rerank` span 的输入输出排序变化 |
| 模型忽略了检索到的证据 | 猜 prompt 有问题 | 看 `llm` span 的 `prompt_hash` + `completion_tokens` |
| 模型幻觉，编造了不存在的引用 | 完全无法区分 | 对比 `retrieve.doc_ids` 与 `response.cited_sources` |
| 工具调用超时，Agent 降级回答 | 不知道调了什么工具 | 看 `tool` span 的 `status` 和 `duration_ms` |

**核心洞察**：Agent 排障是 **分布式追踪问题**，不是「看一行栈」问题。每一跳都可能是故障点，你需要 **全链路可见**。

### 1.2 观测驱动的迭代

可观测性不仅用于排障，还直接指导 **产品迭代方向**：

```
                    观测数据
                       │
          ┌────────────┼────────────┐
          ▼            ▼            ▼
     检索空率高     工具失败多    延迟 P95 高
          │            │            │
          ▼            ▼            ▼
    优化分块策略   修复工具逻辑   换模型/加缓存
    调嵌入模型     加重试/降级   优化 prompt 长度
```

- **检索空率**（检索到 0 个相关文档的比例）高 → 说明知识库覆盖不足或嵌入模型不匹配，应该 **优化索引**
- **工具失败率** 高 → 说明工具本身不稳定或参数解析有问题，应该 **修工具**
- **首 token 延迟** 高 → 可能是模型 API 慢或 prompt 太长，应该 **优化 prompt 或换模型**

没有这些指标，你只能凭感觉猜「应该改什么」，改了也不知道有没有效果。

### 1.3 与传统 Web 服务的对比

| 维度 | 传统 CRUD 服务 | Agent 服务 |
|---|---|---|
| 请求路径 | 固定（路由 → 业务逻辑 → DB） | **动态**（可能有 0-N 次工具调用、0-M 次 LLM 调用） |
| 确定性 | 同输入同输出 | 同输入 **可能** 不同输出 |
| 外部依赖 | 通常只有 DB | DB + 模型 API + 向量库 + N 个工具 |
| 延迟构成 | 主要是 DB 查询 | LLM 生成（几秒级）+ 检索 + 工具 |
| 错误类型 | HTTP 5xx、DB 异常 | 幻觉、解析失败、工具超时、检索空... |
| 成本维度 | CPU + 内存 | **Token 费用**（可能是最大成本项） |

**结论**：Agent 需要 **比传统服务更细粒度的观测**，因为它的内部状态更复杂、路径更多变、排障更困难。

**本节小结（三张卡片）**

- **一句话定义**：Agent 排障默认是 **分布式追踪问题**，不是「看一行栈」问题。
- **典型场景**：延迟 P95 突然升高，拆 span 发现是向量库冷启动而非 LLM。
- **和 A1 类比**：像 **`cProfile` 看热点** —— 先知道时间花在哪，再优化代码路径。

## 2. 可观测性三大支柱概览

**本节概念**：可观测性由 **Logs（日志）、Metrics（指标）、Traces（追踪）** 三大支柱构成，各有分工，互相补充。

### 2.1 三大支柱的定位

```
┌─────────────────────────────────────────────────────────────┐
│                    可观测性三大支柱                           │
│                                                             │
│  Logs（日志）          Metrics（指标）      Traces（追踪）    │
│  ────────────         ────────────        ────────────      │
│  回答：                回答：               回答：            │
│  「这一跳              「整体               「这次请求         │
│   发生了什么」          健康吗？」            的因果链          │
│                                            是什么？」         │
│  粒度：单条事件         粒度：聚合统计        粒度：单次请求     │
│  用途：排障叙事         用途：告警/容量        用途：链路分析     │
│  存储：ELK/Loki        存储：Prometheus      存储：Jaeger/Tempo│
│  成本：高（全量文本）   成本：低（数值聚合）   成本：中（结构化）  │
└─────────────────────────────────────────────────────────────┘
```

### 2.2 三者的关系

它们不是互相替代的，而是 **不同视角看同一个系统**：

- **Metrics** 告诉你 **有问题**（错误率飙升了）
- **Logs** 告诉你 **什么问题**（某个工具返回了 500）
- **Traces** 告诉你 **问题在哪**（是第 3 跳的 HTTP 工具调用超时）

**实际排障流程**：

```
Metrics 报警：工具失败率从 2% 升到 15%
    │
    ▼
Traces 定位：失败集中在 tool.weather_api，平均耗时 30s（超时）
    │
    ▼
Logs 详情：weather_api 返回 "rate limit exceeded"，retry 3 次后放弃
    │
    ▼
行动：联系天气 API 提供商提高限额 / 加缓存 / 换备用 API
```

### 2.3 OpenTelemetry 统一标准

**OpenTelemetry（OTel）** 是 CNCF 的可观测性标准，统一了 Logs、Metrics、Traces 的采集和导出：

- **一套 SDK**：Python、Go、Java 等多语言
- **一套协议**：OTLP（OpenTelemetry Protocol）
- **一套 Collector**：统一采集、处理、导出到不同后端

```python
# OpenTelemetry 的核心抽象
from opentelemetry import trace, metrics
from opentelemetry.sdk.trace import TracerProvider
from opentelemetry.sdk.metrics import MeterProvider

# Tracer：创建 span
tracer = trace.get_tracer("agent-service")

# Meter：创建 metrics
meter = metrics.get_meter("agent-service")

# 使用 tracer 创建 span
with tracer.start_as_current_span("llm_call") as span:
    span.set_attribute("model", "deepseek-chat")
    span.set_attribute("prompt_tokens", 150)
    # ... 调用 LLM
```

> **注**：本章的 demos 使用 **纯标准库实现**（contextvars + time），不依赖 OpenTelemetry SDK，目的是理解底层原理。生产环境建议直接用 OTel。

**本节小结（三张卡片）**

- **一句话定义**：Logs/Metrics/Traces 是三个互补视角，不是三选一。
- **典型场景**：Metrics 报警 → Traces 定位 → Logs 详情，三步排障。
- **和 A1 类比**：Logs 像 `print` 调试，Metrics 像 `timeit` 统计，Traces 像 `cProfile` 调用链。

## 3. Logs：结构化与关联 ID

**本节概念**：**结构化日志**（JSON 一行一条）便于检索与关联；**trace_id** 把分散服务串成一条故事线。

### 3.1 非结构化 vs 结构化日志

```python
# ❌ 非结构化日志 —— 人能读，机器难解析
logger.info(f"User {user_id} asked: {query[:50]}... Retrieved {len(docs)} docs in {elapsed:.2f}s")

# ✅ 结构化日志 —— 机器可解析，可检索，可聚合
logger.info(
    "retrieval_complete",
    extra={
        "trace_id": ctx_trace_id.get(),
        "span_id": span.span_id,
        "user_id_hash": hashlib.sha256(user_id.encode()).hexdigest()[:12],
        "query_length": len(query),
        "doc_count": len(docs),
        "top_score": docs[0].score if docs else None,
        "elapsed_ms": round(elapsed * 1000, 1),
        "model": "text-embedding-3-small",
    }
)
```

### 3.2 结构化日志的字段规范

一条 Agent 日志应包含以下字段：

```json
{
    "timestamp": "2026-04-18T05:30:00.123Z",
    "level": "INFO",
    "logger": "agent.retrieve",
    "message": "retrieval_complete",
    "trace_id": "abc123def456",
    "span_id": "span_789",
    "parent_span_id": "span_456",
    "service": "agent-api",
    "environment": "production",
    
    "user_id_hash": "a1b2c3d4e5f6",
    "query_length": 42,
    "doc_count": 5,
    "top_score": 0.87,
    "elapsed_ms": 123.4,
    "model": "text-embedding-3-small"
}
```

**字段分类**：
- **标准字段**：timestamp、level、logger、message —— 每条日志都有
- **关联字段**：trace_id、span_id、parent_span_id —— 串联链路
- **环境字段**：service、environment —— 区分来源
- **业务字段**：用户、查询、结果 —— **因场景而异**

### 3.3 用 contextvars 在 asyncio 中传递 trace_id

**问题**：在异步代码中，多个请求交替执行，线程局部变量（`threading.local`）无法隔离不同请求的上下文。

**解决**：Python 3.7+ 的 `contextvars` 模块天然支持 asyncio 的任务隔离。

```python
import contextvars
import uuid
import logging
import json
from datetime import datetime, timezone

# 定义上下文变量
ctx_trace_id = contextvars.ContextVar("trace_id", default="no-trace")
ctx_span_id = contextvars.ContextVar("span_id", default="no-span")


class JSONFormatter(logging.Formatter):
    """将日志格式化为 JSON 一行一条"""
    
    def format(self, record: logging.LogRecord) -> str:
        log_dict = {
            "timestamp": datetime.now(timezone.utc).isoformat(),
            "level": record.levelname,
            "logger": record.name,
            "message": record.getMessage(),
            "trace_id": ctx_trace_id.get(),
            "span_id": ctx_span_id.get(),
        }
        # 合并 extra 字段
        if hasattr(record, "__dict__"):
            for key in record.__dict__:
                if key not in logging.LogRecord(
                    "", 0, "", 0, "", (), None
                ).__dict__ and key not in log_dict:
                    log_dict[key] = record.__dict__[key]
        return json.dumps(log_dict, ensure_ascii=False, default=str)


def setup_json_logging():
    """配置全局 JSON 日志"""
    handler = logging.StreamHandler()
    handler.setFormatter(JSONFormatter())
    root = logging.getLogger()
    root.handlers = [handler]
    root.setLevel(logging.INFO)


# 使用方式
import asyncio

logger = logging.getLogger("agent")

async def handle_request(query: str):
    """每个请求设置独立的 trace_id"""
    # 设置 trace_id（contextvars 自动隔离不同的 asyncio Task）
    ctx_trace_id.set(uuid.uuid4().hex[:12])
    
    logger.info("request_start", extra={"query_length": len(query)})
    
    # 即使这里 await 让出控制权，trace_id 仍然正确
    await retrieve(query)
    await generate()
    
    logger.info("request_complete")

async def retrieve(query: str):
    ctx_span_id.set(uuid.uuid4().hex[:8])
    logger.info("retrieve_start")
    await asyncio.sleep(0.1)  # 模拟检索
    logger.info("retrieve_done", extra={"doc_count": 5})

async def generate():
    ctx_span_id.set(uuid.uuid4().hex[:8])
    logger.info("generate_start")
    await asyncio.sleep(0.2)  # 模拟生成
    logger.info("generate_done", extra={"tokens": 150})
```

**关键点**：
- `contextvars.ContextVar` 在 asyncio 中 **自动隔离不同 Task**
- 每个 `await` 点让出控制权后，恢复时 context 仍然正确
- 与 `threading.local` 的区别：`threading.local` 按线程隔离，asyncio 在单线程上跑多个 Task，需要 `contextvars`

### 3.4 日志中的隐私保护

**绝对禁止**写入日志的内容：

| 类型 | 错误示例 | 正确做法 |
|---|---|---|
| 完整 prompt | `logger.info(f"prompt: {prompt}")` | 记录 `prompt_hash` + `prompt_length` |
| 用户 PII | `logger.info(f"user: {email}")` | 记录 `user_id_hash`（单向哈希前12位） |
| API 密钥 | `logger.info(f"key: {api_key}")` | 绝不记录，密钥只在环境变量中 |
| 模型完整输出 | `logger.info(f"response: {response}")` | 记录 `response_length` + `response_hash` |

**原则**：日志的目的是 **定位问题**，不是 **回放内容**。回放用专门的 debug/replay 系统，带权限控制和留存期限。

**本节小结（三张卡片）**

- **一句话定义**：Logs 解决 **「这一跳发生了什么叙述」**。
- **典型场景**：线上 grep 某 `trace_id`，拉出完整工具链与错误栈摘要。
- **和 A1 类比**：`logging.Formatter` 换成 **JSON dict** —— 仍是 `Logger` 哲学，只是下游换成 ELK/Loki。

## 4. Metrics：SLI / SLO 与告警

**本节概念**：**Metrics** 是聚合视角：P50/P95、错误率、业务 KPI；用于 **告警与容量**，不用于还原单次请求细节（那是 Traces）。

### 4.1 Agent 场景的核心指标

#### 延迟类

| 指标 | 含义 | 为什么重要 |
|---|---|---|
| `ttfb_ms` | **首 token 延迟**（Time to First Byte） | 用户体感最直接——超过 3 秒就觉得卡 |
| `e2e_latency_ms` | **端到端延迟** | 从收到请求到完整响应 |
| `retrieve_latency_ms` | 检索延迟 | 区分慢在检索还是生成 |
| `llm_latency_ms` | LLM 调用延迟 | 区分慢在模型还是其他 |
| `tool_latency_ms` | 工具调用延迟 | 标签按工具名分组 |

#### 错误类

| 指标 | 含义 | 为什么重要 |
|---|---|---|
| `http_5xx_rate` | HTTP 5xx 比例 | 基础服务健康 |
| `tool_failure_rate` | 工具调用失败率 | 按工具名分组，定位不稳定的工具 |
| `parse_failure_rate` | 模型输出解析失败率 | 说明 prompt 或模型格式遵从有问题 |
| `retrieve_empty_rate` | 检索空率（Top-K 为空） | 说明知识库覆盖不足 |

#### 业务类

| 指标 | 含义 | 为什么重要 |
|---|---|---|
| `tokens_per_request` | 每请求 token 消耗 | 成本监控 |
| `tool_calls_per_request` | 每请求工具调用次数 | 检测 Agent 是否进入循环 |
| `session_cost_usd` | 每会话费用 | 计费与成本控制 |
| `cache_hit_rate` | 缓存命中率 | 衡量缓存策略有效性 |

### 4.2 用 Python 实现简单的 Metrics 收集

```python
import time
from collections import defaultdict
from dataclasses import dataclass, field
from typing import Dict, List
import threading


@dataclass
class MetricsCollector:
    """简易指标收集器——生产环境用 Prometheus client 替代"""
    
    _counters: Dict[str, int] = field(default_factory=lambda: defaultdict(int))
    _histograms: Dict[str, List[float]] = field(
        default_factory=lambda: defaultdict(list)
    )
    _lock: threading.Lock = field(default_factory=threading.Lock)
    
    def increment(self, name: str, labels: Dict[str, str] = None, value: int = 1):
        """计数器 +1（如请求数、错误数）"""
        key = self._make_key(name, labels)
        with self._lock:
            self._counters[key] += value
    
    def observe(self, name: str, value: float, labels: Dict[str, str] = None):
        """直方图记录一个值（如延迟、token 数）"""
        key = self._make_key(name, labels)
        with self._lock:
            self._histograms[key].append(value)
    
    def get_percentile(self, name: str, p: float, labels: Dict[str, str] = None) -> float:
        """计算百分位数"""
        key = self._make_key(name, labels)
        with self._lock:
            values = sorted(self._histograms.get(key, []))
        if not values:
            return 0.0
        idx = int(len(values) * p / 100)
        return values[min(idx, len(values) - 1)]
    
    def _make_key(self, name: str, labels: Dict[str, str] = None) -> str:
        if not labels:
            return name
        label_str = ",".join(f"{k}={v}" for k, v in sorted(labels.items()))
        return f"{name}{{{label_str}}}"


# 全局实例
metrics = MetricsCollector()


# 使用方式：在 Agent 各环节埋点
async def handle_agent_request(query: str):
    start = time.monotonic()
    metrics.increment("agent_requests_total")
    
    try:
        # 检索
        t0 = time.monotonic()
        docs = await retrieve(query)
        metrics.observe("retrieve_latency_ms", (time.monotonic() - t0) * 1000)
        
        if not docs:
            metrics.increment("retrieve_empty_total")
        
        # LLM 调用
        t0 = time.monotonic()
        response = await call_llm(query, docs)
        metrics.observe("llm_latency_ms", (time.monotonic() - t0) * 1000)
        metrics.observe("prompt_tokens", response.usage.prompt_tokens)
        metrics.observe("completion_tokens", response.usage.completion_tokens)
        
    except Exception:
        metrics.increment("agent_errors_total")
        raise
    finally:
        metrics.observe("e2e_latency_ms", (time.monotonic() - start) * 1000)
```

### 4.3 SLI 与 SLO

**SLI（Service Level Indicator）**：可测量的服务质量指标，如延迟 P95、错误率。

**SLO（Service Level Objective）**：对 SLI 的目标值，如「P95 延迟 < 5 秒」「错误率 < 1%」。

**SLA（Service Level Agreement）**：对外承诺，违反有后果（如赔偿）。

```
SLI ──定义──> SLO ──承诺──> SLA
  │              │              │
 「是什么」    「要多好」      「违约怎么办」
```

**Agent 服务的典型 SLO**：

| SLI | SLO | 说明 |
|---|---|---|
| 端到端延迟 P95 | < 8 秒 | Agent 多跳，比普通 API 宽松 |
| 首 token 延迟 P95 | < 3 秒 | 用户体感关键 |
| HTTP 5xx 错误率 | < 0.5% | 基础可用性 |
| 检索空率 | < 5% | 知识库覆盖度 |
| 工具调用成功率 | > 95% | 工具可靠性 |

## 5. Traces：Span 划分与因果链

**本节概念**：**Trace** 是一次端到端请求的相关 span 集合；**Span** 是一个有起止时间的命名工作单元，可嵌套。

### 5.1 Span 的核心属性

```python
@dataclass
class Span:
    """一个 Span 的核心属性"""
    trace_id: str          # 所属 trace 的全局 ID
    span_id: str           # 本 span 的 ID
    parent_span_id: str    # 父 span 的 ID（根 span 为 None）
    operation_name: str    # 操作名称，如 "llm_call", "retrieve"
    start_time: float      # 起始时间（monotonic）
    end_time: float        # 结束时间
    status: str            # OK / ERROR
    attributes: dict       # 自定义属性
    events: list           # 时间点事件（如 "retry_attempt"）
```

### 5.2 Agent 请求的典型 Span 树

一次 RAG Agent 请求的 span 结构：

```
[Trace: abc123]

http.request (e2e: 4200ms)
├── auth.jwt_verify (5ms)
├── agent.route (10ms)               # Supervisor 路由决策
├── retrieve (320ms)                  # 检索阶段
│   ├── embed_query (80ms)           # 查询向量化
│   ├── vector_search (150ms)        # 向量检索
│   └── rerank (90ms)               # 重排序
├── llm.call_1 (2800ms)             # 第一次 LLM 调用
│   ├── prompt_build (5ms)          # 构造 prompt
│   └── model_inference (2795ms)    # 模型推理
├── tool.weather_api (500ms)         # 工具调用
│   ├── http_request (480ms)        # 实际 HTTP 请求
│   └── response_parse (20ms)       # 解析工具返回
├── llm.call_2 (550ms)              # 第二次 LLM（结合工具结果）
└── response.serialize (15ms)        # 序列化响应
```

### 5.3 Span 划分的经验法则

**划分粒度的黄金法则**：**一个 span 应该代表一个可独立排障的工作单元**。

| 操作 | 是否独立 span？ | 理由 |
|---|---|---|
| 一次 LLM 调用 | **是** | 延迟和 token 用量需要独立可见 |
| 一次工具调用 | **是** | 工具可能超时/失败，需要独立监控 |
| 一次向量检索 | **是** | 检索质量和延迟需要独立可见 |
| Rerank | **是** | 可能是延迟瓶颈，需要独立测量 |
| Prompt 构造 | 可选 | 通常很快（<10ms），但复杂 prompt 构造可能值得单独 span |
| JSON 解析 | 通常不是 | 太细粒度，除非解析失败率高 |
| 重试 | **不是独立 span** | 在父 span 的 attributes 中记录 `retry_count` |

**反面案例**：
- ❌ 每行代码一个 span → 噪声太大，trace 无法阅读
- ❌ 整个请求只有一个 span → 和没有 trace 一样
- ❌ 不区分检索和 LLM → 无法判断慢在哪

### 5.4 用 Python 实现 Span 管理器

```python
import time
import uuid
import contextvars
from dataclasses import dataclass, field
from typing import Optional, Dict, Any, List

# 上下文变量：当前活跃的 span
_current_span: contextvars.ContextVar[Optional["SpanContext"]] = (
    contextvars.ContextVar("_current_span", default=None)
)
_trace_id: contextvars.ContextVar[str] = (
    contextvars.ContextVar("_trace_id", default="")
)


@dataclass
class SpanContext:
    """Span 的运行时上下文"""
    trace_id: str
    span_id: str
    parent_span_id: Optional[str]
    operation: str
    start_time: float
    attributes: Dict[str, Any] = field(default_factory=dict)
    events: List[Dict[str, Any]] = field(default_factory=list)
    children: List["SpanContext"] = field(default_factory=list)
    end_time: Optional[float] = None
    status: str = "OK"

    @property
    def duration_ms(self) -> float:
        if self.end_time is None:
            return 0.0
        return (self.end_time - self.start_time) * 1000

    def set_attribute(self, key: str, value: Any):
        self.attributes[key] = value

    def add_event(self, name: str, attributes: Dict[str, Any] = None):
        self.events.append({
            "name": name,
            "timestamp": time.monotonic(),
            "attributes": attributes or {},
        })

    def set_error(self, error: Exception):
        self.status = "ERROR"
        self.attributes["error.type"] = type(error).__name__
        self.attributes["error.message"] = str(error)


class SpanManager:
    """管理 span 的创建、嵌套和输出"""
    
    def __init__(self):
        self.completed_spans: List[SpanContext] = []

    def start_trace(self) -> str:
        """开始一个新的 trace"""
        trace_id = uuid.uuid4().hex[:12]
        _trace_id.set(trace_id)
        return trace_id

    def start_span(self, operation: str) -> SpanContext:
        """开始一个新的 span（自动嵌套到当前 span 下）"""
        parent = _current_span.get()
        trace_id = _trace_id.get() or uuid.uuid4().hex[:12]
        
        span = SpanContext(
            trace_id=trace_id,
            span_id=uuid.uuid4().hex[:8],
            parent_span_id=parent.span_id if parent else None,
            operation=operation,
            start_time=time.monotonic(),
        )
        
        if parent:
            parent.children.append(span)
        
        _current_span.set(span)
        return span

    def end_span(self, span: SpanContext):
        """结束一个 span"""
        span.end_time = time.monotonic()
        self.completed_spans.append(span)
        
        # 恢复父 span 为当前 span
        # （简化实现，生产中需要用 contextvars token 回退）

    def print_trace(self, span: SpanContext, indent: int = 0):
        """可视化打印 trace 树"""
        prefix = "  " * indent + ("├── " if indent > 0 else "")
        status_icon = "OK" if span.status == "OK" else "ERR"
        print(
            f"{prefix}{span.operation} "
            f"[{status_icon}] "
            f"{span.duration_ms:.1f}ms "
            f"{span.attributes}"
        )
        for child in span.children:
            self.print_trace(child, indent + 1)


# 使用 context manager 简化
from contextlib import contextmanager

span_manager = SpanManager()

@contextmanager
def traced(operation: str, **attrs):
    """用 with 语句自动管理 span 生命周期
    
    用法：
        with traced("llm_call", model="deepseek") as span:
            result = await call_llm(...)
            span.set_attribute("tokens", result.usage.total_tokens)
    """
    span = span_manager.start_span(operation)
    for k, v in attrs.items():
        span.set_attribute(k, v)
    try:
        yield span
    except Exception as e:
        span.set_error(e)
        raise
    finally:
        span_manager.end_span(span)
```

### 5.5 多 Agent 的 Span 拓扑

单 Agent（C6）的 span 是一条 **链**；多 Agent（C7）的 span 是 **树或 DAG**：

```
# 单 Agent：链式
http → retrieve → llm → tool → llm → response

# Supervisor 多 Agent：树形
http → supervisor.route
       ├── agent_rag → retrieve → llm
       ├── agent_code → tool.exec → llm
       └── supervisor.merge → llm → response

# Pipeline 多 Agent：链式但多 Agent
http → agent_extract → agent_validate → agent_format → response

# Fan-out 并行：DAG
http → fan_out
       ├── agent_a → llm (并行)
       ├── agent_b → tool (并行)
       └── agent_c → retrieve (并行)
       fan_in → merge → response
```

**关键概念**：
- **父子关系（Parent-Child）**：表达因果嵌套，如 `supervisor → agent_rag`
- **链接（Links）**：表达非因果的关联，如并行执行的多个 agent 互相引用

**本节小结（三张卡片）**

- **一句话定义**：Traces 把 **时间线 + 因果嵌套** 可视化。
- **典型场景**：对比金丝雀与旧版，同一查询下 `llm` span 的 token 与 latency 差异。
- **和 A1 类比**：像 **嵌套 `with timer(label):`** —— 只是升级为跨进程标准。

## 7. 完整实现：从请求入口到 Span 输出

**本节概念**：把前面的 Logs、Metrics、Traces 串联起来，展示一次 Agent 请求的 **完整观测链路**。

```python
import asyncio
import time
import uuid
import hashlib
import logging
import json
import contextvars
from dataclasses import dataclass, field
from typing import List, Optional, Dict, Any

# ========== 上下文 ==========
_trace_id = contextvars.ContextVar("trace_id", default="")
_request_start = contextvars.ContextVar("request_start", default=0.0)

logger = logging.getLogger("agent")


# ========== Span 实现 ==========
@dataclass
class SimpleSpan:
    name: str
    trace_id: str
    span_id: str = field(default_factory=lambda: uuid.uuid4().hex[:8])
    parent_id: Optional[str] = None
    start: float = field(default_factory=time.monotonic)
    end: Optional[float] = None
    attrs: Dict[str, Any] = field(default_factory=dict)
    status: str = "OK"
    children: List["SimpleSpan"] = field(default_factory=list)

    @property
    def duration_ms(self) -> float:
        return ((self.end or time.monotonic()) - self.start) * 1000

    def finish(self):
        self.end = time.monotonic()
        # 输出结构化日志
        logger.info(json.dumps({
            "type": "span",
            "trace_id": self.trace_id,
            "span_id": self.span_id,
            "parent_id": self.parent_id,
            "name": self.name,
            "duration_ms": round(self.duration_ms, 2),
            "status": self.status,
            "attributes": self.attrs,
        }, ensure_ascii=False))


# ========== 模拟 Agent 全链路 ==========
class ObservableAgent:
    """一个带完整观测的 Agent 示例"""
    
    def __init__(self):
        self.metrics = {
            "requests": 0,
            "errors": 0,
            "retrieve_empty": 0,
            "latencies": [],
        }
    
    async def handle(self, query: str) -> str:
        # 开始 trace
        trace_id = uuid.uuid4().hex[:12]
        _trace_id.set(trace_id)
        
        root = SimpleSpan(name="http.request", trace_id=trace_id)
        self.metrics["requests"] += 1
        
        try:
            # Step 1: 检索
            retrieve_span = SimpleSpan(
                name="retrieve",
                trace_id=trace_id,
                parent_id=root.span_id,
            )
            docs = await self._retrieve(query, retrieve_span)
            retrieve_span.finish()
            root.children.append(retrieve_span)
            
            if not docs:
                self.metrics["retrieve_empty"] += 1
            
            # Step 2: 构造 prompt
            prompt = self._build_prompt(query, docs)
            
            # Step 3: LLM 调用
            llm_span = SimpleSpan(
                name="llm.call",
                trace_id=trace_id,
                parent_id=root.span_id,
            )
            answer = await self._call_llm(prompt, llm_span)
            llm_span.finish()
            root.children.append(llm_span)
            
            return answer
            
        except Exception as e:
            root.status = "ERROR"
            root.attrs["error"] = str(e)
            self.metrics["errors"] += 1
            raise
        finally:
            root.finish()
            self.metrics["latencies"].append(root.duration_ms)
    
    async def _retrieve(self, query: str, span: SimpleSpan) -> list:
        """模拟检索"""
        span.attrs["retrieve.query_length"] = len(query)
        span.attrs["retrieve.top_k"] = 5
        
        await asyncio.sleep(0.1)  # 模拟向量检索
        
        docs = [{"id": "doc1", "score": 0.92}, {"id": "doc2", "score": 0.85}]
        span.attrs["retrieve.doc_count"] = len(docs)
        span.attrs["retrieve.top_score"] = docs[0]["score"]
        span.attrs["retrieve.empty"] = False
        return docs
    
    def _build_prompt(self, query: str, docs: list) -> str:
        """构造 prompt（不需要 span，通常 <1ms）"""
        context = "\n".join(d["id"] for d in docs)
        return f"Based on: {context}\nAnswer: {query}"
    
    async def _call_llm(self, prompt: str, span: SimpleSpan) -> str:
        """模拟 LLM 调用"""
        span.attrs["llm.model"] = "deepseek-chat"
        span.attrs["llm.prompt_hash"] = hashlib.md5(
            prompt.encode()
        ).hexdigest()[:8]
        span.attrs["llm.prompt_length"] = len(prompt)
        
        await asyncio.sleep(0.5)  # 模拟模型推理
        
        span.attrs["llm.completion_tokens"] = 150
        span.attrs["llm.prompt_tokens"] = 80
        span.attrs["llm.finish_reason"] = "stop"
        return "This is the answer based on the documents."


# ========== 运行示例 ==========
async def main():
    agent = ObservableAgent()
    answer = await agent.handle("什么是 RAG？")
    print(f"\n回答: {answer}")
    print(f"\n指标: {agent.metrics}")

# asyncio.run(main())
```

## 8. 隐私与采样策略

**本节概念**：**全量采集**在 Agent 场景下极易 **泄漏 prompt 与 PII**；**采样**降成本，但会丢罕见 bug —— 需要 **动态采样与错误全采** 折中。

### 8.1 采样策略

| 策略 | 做法 | 优点 | 缺点 |
|---|---|---|---|
| **全量采集** | 100% trace 全记录 | 不丢任何数据 | 成本极高，隐私风险大 |
| **头部采样** | 请求入口决定是否采集（如 1%） | 成本低 | 罕见错误可能被漏掉 |
| **尾部采样** | 请求完成后决定是否保留（根据结果） | 错误全量保留 | 需要缓冲全部 span，内存压力大 |
| **动态采样** | 正常 1%，错误 100%，特定用户 100% | 平衡成本与可观测性 | 实现复杂 |

### 8.2 推荐的分级采样策略

```python
import random

class SamplingDecision:
    """采样决策器"""
    
    def __init__(
        self,
        default_rate: float = 0.01,      # 默认 1% 采样
        error_rate: float = 1.0,          # 错误 100% 采集
        slow_threshold_ms: float = 5000,  # 慢请求阈值
        slow_rate: float = 1.0,           # 慢请求 100% 采集
        debug_users: set = None,          # 调试用户列表
    ):
        self.default_rate = default_rate
        self.error_rate = error_rate
        self.slow_threshold_ms = slow_threshold_ms
        self.slow_rate = slow_rate
        self.debug_users = debug_users or set()
    
    def should_sample_head(self, user_id: str = None) -> bool:
        """请求入口决定是否采样（头部采样）"""
        # 调试用户始终采集
        if user_id and user_id in self.debug_users:
            return True
        # 按概率采样
        return random.random() < self.default_rate
    
    def should_keep_tail(
        self,
        status: str,
        duration_ms: float,
        user_id: str = None,
    ) -> bool:
        """请求完成后决定是否保留（尾部采样）"""
        # 错误全量保留
        if status == "ERROR":
            return True
        # 慢请求全量保留
        if duration_ms > self.slow_threshold_ms:
            return True
        # 调试用户全量保留
        if user_id and user_id in self.debug_users:
            return True
        # 其他走头部采样的决定
        return False  # 已在头部决定
```

### 8.3 环境分级

| 环境 | 日志级别 | 采样率 | Prompt 记录 |
|---|---|---|---|
| **开发** | DEBUG | 100% | 可记录全文（仅开发数据） |
| **预发** | INFO | 100% | 记录 hash + 长度 |
| **生产** | INFO | 1% + 错误全量 | 只记录 hash + 长度 |
| **故障排查** | DEBUG（临时） | 提升到 50% 或指定用户 100% | 临时开启，限时 30 分钟 |

### 8.4 数据脱敏实践

```python
import hashlib
import re

def sanitize_for_logging(data: dict) -> dict:
    """对日志数据进行脱敏"""
    sanitized = {}
    for key, value in data.items():
        if key in ("prompt", "completion", "query", "response"):
            # 文本内容只记录哈希和长度
            sanitized[f"{key}_hash"] = hashlib.sha256(
                str(value).encode()
            ).hexdigest()[:12]
            sanitized[f"{key}_length"] = len(str(value))
        elif key in ("user_id", "email", "phone"):
            # PII 单向哈希
            sanitized[f"{key}_hash"] = hashlib.sha256(
                str(value).encode()
            ).hexdigest()[:12]
        elif key in ("api_key", "token", "password", "secret"):
            # 敏感凭证完全不记录
            sanitized[key] = "***REDACTED***"
        else:
            sanitized[key] = value
    return sanitized
```

**本节小结（三张卡片）**

- **一句话定义**：采样策略是 **成本—可观测性—合规** 的三方博弈。
- **典型场景**：大促流量下仍保留 **5xx 与 tool 失败** 全链路。
- **和 A1 类比**：像 **日志级别**：生产默认 INFO，故障时临时开 DEBUG 但限时长。

## 9. 告警设计与 SLO 实践

**本节概念**：告警不是「指标超了就报」，需要 **减少误报、分级响应、自动化处理**。

### 9.1 告警的层级

```
┌─────────────────────────────────────────────────────┐
│ P0: 服务不可用                                       │
│   - HTTP 5xx 率 > 10% 持续 1 分钟                    │
│   - 响应：立即处理，所有 oncall 收到通知              │
├─────────────────────────────────────────────────────┤
│ P1: 服务降级                                         │
│   - 端到端延迟 P95 > 10 秒 持续 5 分钟               │
│   - 工具失败率 > 20% 持续 5 分钟                      │
│   - 响应：15 分钟内响应                               │
├─────────────────────────────────────────────────────┤
│ P2: 业务异常                                         │
│   - 检索空率 > 10% 持续 15 分钟                       │
│   - Token 用量突增 50%                                │
│   - 响应：工作时间内处理                              │
├─────────────────────────────────────────────────────┤
│ P3: 观察                                             │
│   - 缓存命中率下降                                    │
│   - 某工具延迟趋势上升                                │
│   - 响应：下次周会讨论                                │
└─────────────────────────────────────────────────────┘
```

### 9.2 告警设计原则

**1. 基于 SLO 而非绝对阈值**

```python
# ❌ 绝对阈值——容易误报
if latency_p95 > 5000:
    alert("延迟过高")

# ✅ 基于 SLO 的烧毁率（Burn Rate）
# "过去 1 小时消耗了 5% 的月度错误预算"
error_budget_monthly = 0.001  # 99.9% 可用性
errors_last_hour = get_error_rate(window="1h")
burn_rate = errors_last_hour / (error_budget_monthly / 720)  # 720 = 30天 × 24小时

if burn_rate > 14:  # 14x = 1小时用完2天的预算
    alert_p0("SLO burn rate critical")
elif burn_rate > 6:  # 6x = 1小时用完半天的预算
    alert_p1("SLO burn rate elevated")
```

**2. 持续时间窗口——避免瞬间抖动**

```python
# ❌ 瞬间触发
if error_rate > 0.05:
    alert()

# ✅ 持续 N 分钟才触发
if error_rate_5min_avg > 0.05:  # 5分钟滑动平均
    alert()
```

**3. 告警应包含的信息**

```
[P1] Agent 工具失败率异常
时间: 2026-04-18 05:30 UTC
指标: tool_failure_rate = 18.5%（阈值 10%，持续 5 分钟）
影响范围: tool=weather_api, 其他工具正常
可能原因: 天气 API 限流 (429) 或服务端故障
相关 trace: https://jaeger.internal/trace/abc123
Runbook: https://wiki.internal/runbook/tool-failure
```

### 9.3 Agent 场景特有的告警

| 告警 | 条件 | 可能原因 |
|---|---|---|
| Agent 循环过多 | `avg(tool_calls_per_request) > 8` 持续 10 分钟 | Prompt 问题导致 Agent 不知道何时停止 |
| Token 成本突增 | `sum(tokens) > 2x 历史同期` | Prompt 变长 / 检索注入过多文档 / Agent 循环 |
| 检索质量下降 | `retrieve_empty_rate > 10%` 且 `retrieve_top_score_avg < 0.5` | 嵌入模型漂移 / 知识库过时 |
| 模型输出解析失败 | `parse_failure_rate > 5%` | 模型版本更新 / Prompt 格式变化 |

## 10. 与测试、发布的衔接

**本节概念**：**可观测性不是上线后补丁**，要与 **契约测试、特性开关、回放** 同一套工程节奏。

### 10.1 观测驱动的发布流程

```
开发 → 测试 → 金丝雀 → 全量 → 监控
  │       │       │        │       │
  │       │       │        │       └── 持续观测 Metrics/Traces
  │       │       │        └── 确认 SLO 无退化后全量
  │       │       └── 5% 流量，对比新旧版 Span 指标
  │       └── 单元测试 + 契约测试（工具/路由/输出格式）
  └── 开发时已埋好 Span 和 Metrics
```

### 10.2 金丝雀对比

金丝雀发布时，新旧版本 **同时服务**，用 **trace 维度** 对比：

```python
# 金丝雀对比维度
canary_comparison = {
    "latency": {
        "baseline_p95": 4200,  # 旧版 P95
        "canary_p95": 4500,    # 新版 P95

    },

        "baseline": 3.2,
        "canary": 2.8,         # 新版检索更好

    },
    "token_usage_avg": {
        "baseline": 850,
        "canary": 920,         # Token 用量上升 8%
        "verdict": "WATCH",    # 需要关注成本
    },
}
```

### 10.3 回放与回归测试

```python
# 用录制的 trace 做回归测试
class TraceReplayTest:
    """从生产 trace 提取测试用例"""
    
    def __init__(self, trace_data: dict):
        self.query = trace_data["query_hash"]  # 脱敏
        self.expected_tools = trace_data["tools_called"]
        self.expected_doc_ids = trace_data["retrieve.doc_ids"]
    
    def verify(self, actual_span: dict):
        """验证新版本的行为是否与录制时一致"""
        assert actual_span["retrieve.doc_count"] > 0, "检索不应为空"
        assert actual_span["llm.finish_reason"] == "stop", "应正常完成"
        # 注意：不验证具体回答内容（非确定性），
        # 只验证链路结构和关键指标
```

**本节小结（三张卡片）**

- **一句话定义**：测试固定 **行为**，观测验证 **线上行为是否漂移**。
- **典型场景**：新旧 prompt 金丝雀，对比 `faithfulness` 与工具失败率。
- **和 A1 类比**：像 **pytest fixture 录响应** —— 只是把对象从 HTTP mock 扩到 tool mock。

## 11. 常见误区

### 误区 1：只打 HTTP 状态码，不打 Agent 语义字段

```python
# ❌ 只有 HTTP 层面的信息
logger.info("Request completed", extra={"status": 200, "duration": 4.2})
# 问题：HTTP 200 但回答可能是幻觉，你完全不知道

# ✅ 补上 Agent 语义字段
logger.info("Request completed", extra={
    "status": 200,
    "duration_ms": 4200,
    "retrieve.doc_count": 5,
    "retrieve.top_score": 0.87,
    "llm.tokens": 230,
    "tool_calls": 1,
    "agent.loop_iterations": 2,
})
```

### 误区 2：把完整 Prompt 写入日志

Prompt 可能包含用户 PII、商业机密、系统指令（prompt injection 风险）。日志是 **低权限系统**，任何有日志访问权的人都能看到。

**正确做法**：只记录 `prompt_hash` + `prompt_length`，需要回放时走 **独立的 debug 系统**（有审计、有权限、有留存期限）。

### 误区 3：告警太多导致「告警疲劳」

如果每天收到 50+ 告警，团队会开始忽略告警。

**解决**：
- P0/P1 告警要 **极少且准确**（每周不超过 2-3 个）
- P2/P3 走 **日报或周报**，不实时通知
- 定期 **清理无效告警**（连续 30 天未触发的告警考虑删除）

### 误区 4：只关注单次延迟，忽略分布

```python
# ❌ "平均延迟 2 秒，没问题"
# 问题：可能 P99 是 30 秒，10% 的用户体验极差

# ✅ 关注百分位数
print(f"P50: {p50}ms, P95: {p95}ms, P99: {p99}ms")
# 如果 P50=1000, P95=5000, P99=20000
# 说明有长尾问题，需要排查慢请求的共性
```

### 误区 5：开发和生产用不同的观测代码

```python
# ❌ 开发时不打 span，上线后再加
if os.getenv("ENV") == "production":
    with traced("llm_call"):
        result = await call_llm(...)
else:
    result = await call_llm(...)

# ✅ 始终打 span，通过采样率控制输出量
with traced("llm_call") as span:
    result = await call_llm(...)
# 开发：全量输出到 console
# 生产：1% 采样 + 错误全量，输出到 Jaeger
```

## 14. 面试常见问答（5 题）

**Q1：什么是可观测性？它和监控有什么区别？**

A：**监控**是预定义的指标和告警——你事先知道要监控什么（CPU、内存、错误率），超阈值就报警。**可观测性**是从系统的外部输出（Logs、Metrics、Traces）推断系统内部任意状态的能力——你不需要事先知道会出什么问题。Agent 场景需要可观测性而不仅是监控，因为 Agent 的行为路径是非确定性的，你无法预定义所有可能的异常场景。

**Q2：Agent 服务的 Traces 和传统微服务的 Traces 有什么不同？**

A：传统微服务的 trace 是 **固定拓扑**（A→B→C），每次请求走同样的路径。Agent 的 trace 是 **动态拓扑**——同一个 Agent 可能走 0-N 次工具调用、0-M 次 LLM 调用，路径由模型推理决定。这意味着 Agent 的 span 树结构不可预测，需要更细粒度的 span 划分（每次 LLM 调用、每次工具调用独立 span）和 **Agent 特有的语义字段**（`agent.loop_iteration`、`tool.name`、`retrieve.top_score`）。

**Q3：生产环境如何平衡可观测性的成本和价值？**

A：用 **分级采样策略**：正常请求采样 1-5%（降低存储和计算成本），错误请求 100% 采集（确保不漏 bug），慢请求全量采集（P95 以上）。同时区分 Metrics（低成本聚合，全量采集）和 Traces（高成本详情，采样采集）。另外，日志记录用 **结构化 JSON + 字段脱敏**，避免记录完整 prompt/response 导致的存储膨胀和隐私风险。

**Q4：如果用户反馈「AI 回答引用了错误的来源」，你怎么用可观测性定位问题？**

A：分三步定位：
1. **Traces**：找到该请求的 trace_id，查看 span 树。看 `retrieve` span 的 `doc_ids` 和 `scores`——检索到的文档是否包含正确来源？
2. **如果检索正确但引用错误**：问题在 LLM 生成阶段。看 `llm` span 的 `prompt_hash`，用相同 prompt 回放验证是否为模型幻觉。
3. **如果检索就错了**：检查 `retrieve` span 的 `query_length` 和 `top_score`。分数太低说明嵌入模型或分块策略有问题；检索为空说明知识库没有相关内容。

**Q5：如何设计 Agent 服务的 SLO？**

A：Agent 服务的 SLO 要覆盖 **可用性、延迟、质量** 三个维度：
- **可用性**：HTTP 5xx 率 < 0.5%（月度），这是基础
- **延迟**：首 token 延迟 P95 < 3 秒，端到端延迟 P95 < 10 秒（Agent 比普通 API 宽松，因为多跳）
- **质量**：检索空率 < 5%（衡量知识库覆盖度），工具调用成功率 > 95%

SLO 的设定需要与产品/业务方共同制定，不能纯技术拍脑袋。设定后用 **错误预算**（error budget）管理：如果本月的错误预算快烧完了，就暂停新功能发布，集中修 bug。
