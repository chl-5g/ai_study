## 0. 与 A1、C6、C7 的衔接

**本节概念**：**可观测性**回答「系统内部发生了什么」；对 Agent 而言，内部包括 **检索、工具、模型多轮、路由**。没有统一 **trace_id**，你只能看到「最终答错了」，看不到 **错在哪一跳**。

A1 教你 **`logging` 模块级配置、`extra` 字段、异常链**；C8 把这些升格为：**跨异步任务传播的上下文（contextvars）**、**嵌套耗时（span）**、**聚合指标（metrics）**。单 Agent（C6）是一串 span；多 Agent（C7）是 **树或 DAG**，更需要 **父子 span** 与 **links**。

**本节小结（三张卡片）**

- **一句话定义**：可观测性 = **用外部输出推断内部状态** 的能力，Agent 只是负载更丰富的服务。
- **典型场景**：用户截图说「引用错了」，你要判断是 rerank 还是模型忽略证据。
- **和 A1 类比**：`trace_id` 像 **给每次请求挂 uuid**，所有 log 行带上它 —— 只是 Agent 里还要穿过 tool 子进程。

## 1. Why：没有观测 = 黑盒

**本节概念**：用户反馈「答错了」时，工程上要回答：**检索命中了吗？调了哪个工具？模型原始 reasoning？耗时在哪一步？** 没有 trace 只能猜，猜的成本高于 **一开始打 span**。

与传统 CRUD 服务不同，Agent 路径 **分支多、外部依赖多、同一输入可能非确定**。因此观测对象从「HTTP 状态码」扩展到 **tool 成功率、检索空率、首 token 延迟、解析失败率**。这些指标直接指导 **prompt 迭代还是索引迭代**。

**本节小结（三张卡片）**

- **一句话定义**：Agent 排障默认是 **分布式追踪问题**，不是「看一行栈」问题。
- **典型场景**：延迟 P95 突然升高，拆 span 发现是向量库冷启动而非 LLM。
- **和 A1 类比**：像 **`cProfile` 看热点** —— 先知道时间花在哪，再优化代码路径。

## 2. Logs：结构化与关联 ID

**本节概念**：**结构化日志**（JSON 一行一条）便于检索与关联；**trace_id** 把分散服务串成一条故事线。

- **JSON 一行一条**，字段含 `timestamp`、`level`、`trace_id`、`span_id`、`message`。  
- **`contextvars`** 在 asyncio 下传递 `trace_id`（见 `demos/01_trace_contextvars.py`）。  
- **禁止**：完整 prompt、PII、密钥入日志——用 **哈希或截断**。

**落地建议**：日志里保留 **模型名、token 用量、工具名、耗时**；prompt 只保留 **hash + 长度** 用于对齐回放环境。

**本节小结（三张卡片）**

- **一句话定义**：Logs 解决 **「这一跳发生了什么叙述」**。
- **典型场景**：线上 grep 某 `order_id` 哈希，拉出完整工具链与错误栈摘要。
- **和 A1 类比**：`logging.Formatter` 换成 **JSON dict** —— 仍是 `Logger` 哲学，只是下游换成 ELK。

## 3. Metrics：SLI 与告警

**本节概念**：**Metrics** 是聚合视角：P50/P95、错误率、业务 KPI；用于 **告警与容量**，不用于还原单次请求细节（那是 Traces）。

- **延迟**：P50/P95 **首 token**、**端到端**。  
- **错误率**：HTTP 5xx、工具失败、解析失败。  
- **业务**：检索空率、工具调用次数分布、**每会话 Token/费用**。  
- **告警**：错误率突增、延迟 SLO 破位、队列积压。

**本节小结（三张卡片）**

- **一句话定义**：Metrics 回答 **「整体健康吗」**，不回答 **「这一次为什么错」**。
- **典型场景**：检索空率从 2% 飙到 20%，先于用户投诉触发告警。
- **和 A1 类比**：像 **`timeit` 批量统计** —— 单次数不准，分布才有意义。

---

## 4. Traces：Span 划分

**本节概念**：**Trace** 是一次端到端请求的相关 span 集合；**Span** 是一个有起止时间的命名工作单元，可嵌套。

典型 Span：`http.request` → `agent.route` → `retrieve` → `llm` → `tool.http` → `llm` → `response.serialize`。  
**父子关系**表达嵌套；**并行检索**用多个子 span + `links`。

**划分粒度经验**：**一次 LLM 调用一个 span**（含重试则在 attributes 标 `retry_count`）；**每个工具一次一个 span**；**rerank 单独 span**，否则检索延迟全糊在 `retrieve` 里。

**本节小结（三张卡片）**

- **一句话定义**：Traces 把 **时间线 + 因果嵌套** 可视化。
- **典型场景**：对比金丝雀与旧版，同一查询下 `llm` span 的 token 与 latency 差异。
- **和 A1 类比**：像 **嵌套 `with timer(label):`** —— 只是升级为跨进程标准。

## 5. Agent 特有字段

**本节概念**：通用 APM 字段不够；Agent 要补 **检索与工具语义**，否则 span 只有「HTTP 200」。

- **检索**：`query`、`top_k`、命中 `doc_id`、分数、**是否 rerank**。  
- **工具**：`name`、**脱敏后参数**、耗时、重试次数。  
- **模型**：`model_id`、**prompt_tokens** / **completion_tokens**。

**本节小结（三张卡片）**

- **一句话定义**：Agent 观测 = **通用链路字段 + 领域语义字段**。
- **典型场景**：同一工具超时率上升，对比参数摘要发现某租户 payload 暴涨。
- **和 A1 类比**：像 **`logging` 的 `extra` 带业务 id** —— 只是字段集合更标准化。

## 6. 隐私与采样

**本节概念**：**全量采集**在 Agent 场景下极易 **泄漏 prompt 与 PII**；**采样**降成本，但会丢罕见 bug —— 需要 **动态采样与错误全采** 折中。

- **头部采样**（仅记录 1% trace）降低成本，但 **错误全量** 捕获。  
- **分级**：开发全量、生产采样 + **动态提升采样率**（某用户报错时）。  
- **合规**：日志留存周期、跨境、脱敏规则要与法务对齐（此处只点工程开关）。

**本节小结（三张卡片）**

- **一句话定义**：采样策略是 **成本—可观测性—合规** 的三方博弈。
- **典型场景**：大促流量下仍保留 **5xx 与 tool 失败** 全链路。
- **和 A1 类比**：像 **日志级别**：生产默认 INFO，故障时临时开 DEBUG 但限时长。

## 7. 与测试、发布的衔接

**本节概念**：**可观测性不是上线后补丁**，要与 **契约测试、特性开关、回放** 同一套工程节奏。

- **契约测试**：工具与路由（A4-测试与CI）。  
- **金丝雀 / 特性开关**：新 prompt 与旧版对比 **同 trace 维度**。  
- **回放**：用固定 seed + **录制的 tool mock** 做回归（注意脱敏）。

**本节小结（三张卡片）**

- **一句话定义**：测试固定 **行为**，观测验证 **线上行为是否漂移**。
- **典型场景**：新旧 prompt 金丝雀，对比 `faithfulness` 与工具失败率。
- **和 A1 类比**：像 **pytest fixture 录响应** —— 只是把对象从 HTTP mock 扩到 tool mock。
