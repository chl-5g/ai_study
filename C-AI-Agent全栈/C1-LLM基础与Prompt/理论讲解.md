## 0. 与 A1、B2 的衔接（先看清地图再调参）

**本节概念**：**大模型应用**在工程上仍是「发请求、收 JSON、管字符串与字节、处理超时与重试」；Transformer 是背景知识，**契约与成本**才是日常主战场。

很多人把 C1 读成「迷你深度学习课」，结果在 **429、流式半包、tool 消息格式** 上栽跟头。更稳的路径是：先用 A1 的习惯把 **调用封装成函数**（类型写清、日志打全），再用 B2 的视角看 **每一跳 HTTP**（DNS、TLS、代理、压缩），最后才把注意力放在 **temperature 与 prompt 模板** 上 —— 否则你会在「模型很聪明」与「代码很脆」之间反复撕裂。

**本节小结（三张卡片）**

- **一句话定义**：C1 交付的是与模型对话的**共同语言**（token、消息、采样、模板），不是把 Attention 手推一遍。
- **典型场景**：同一套 `httpx`/`requests` 客户端，换 `base_url` 与 `api_key` 即可从 Ollama 切到云厂商，差异集中在 **鉴权头与 usage 字段**。
- **和 A1 类比**：**Prompt 模板**像把重复字符串收进 **带默认参数的函数**；**流式输出**像 **`yield` 逐块吐结果**，客户端用 async 或阻塞读都要遵守「消费完或取消」的协议。

## 1. LLM 基础概念

### 1.1 Transformer 架构概述

**本节概念**：**Transformer** 是用自注意力在整段序列上并行建关联的神经网络结构；当前主流 **LLM** 多采用 **Decoder-Only** 栈，自回归地预测下一个 token。

Transformer 是 2017 年 Google 在论文 "Attention Is All You Need" 中提出的架构，是当前所有主流大语言模型的基础。

**核心组件：**

```
输入 → Tokenizer → Embedding → [Transformer Blocks × N] → Output Head → 输出

每个 Transformer Block 包含：
├── Multi-Head Self-Attention（多头自注意力）
├── Layer Normalization
├── Feed-Forward Network（前馈网络）
└── Residual Connection（残差连接）
```

**关键机制 — Self-Attention（自注意力）：**

- 每个 Token 都会"关注"序列中的所有其他 Token
- 通过 Query(Q)、Key(K)、Value(V) 三个矩阵计算注意力权重
- 公式：`Attention(Q, K, V) = softmax(QK^T / √d_k) V`
- Multi-Head 机制让模型从多个"视角"理解文本

**架构变体：**

| 架构类型 | 代表模型 | 特点 |
|---------|---------|------|
| Encoder-Only | BERT | 双向注意力，适合理解任务（分类、NER） |
| Decoder-Only | GPT、Claude、LLaMA | 单向/因果注意力，自回归生成 |
| Encoder-Decoder | T5、BART | 编码+解码，适合翻译/摘要 |

当前主流 LLM 几乎都采用 **Decoder-Only** 架构。

### 1.2 Token 与 Tokenizer

**本节概念**：**Token** 是词表与压缩算法切分后的最小处理单元；**Tokenizer** 把字节/字符序列映射为 token id 序列，直接决定 **账单与上下文窗口占用**。

**Token 不等于字/词。** Token 是模型处理文本的最小单位。

```
英文示例：
"Hello world" → ["Hello", " world"]  （2 tokens）
"unbelievable" → ["un", "believ", "able"]  （3 tokens）

中文示例：
"你好世界" → ["你好", "世界"]  （2 tokens，取决于 tokenizer）
"人工智能" → ["人工", "智能"]  （2 tokens）
```

**主流 Tokenizer 算法：**

| 算法 | 说明 | 使用者 |
|------|------|--------|
| BPE (Byte Pair Encoding) | 从字符级别逐步合并高频对 | GPT 系列 |
| WordPiece | 类似 BPE，基于似然合并 | BERT |
| SentencePiece | 直接在原始文本上训练，支持多语言 | LLaMA、Qwen |
| tiktoken | OpenAI 的 BPE 实现，速度快 | GPT-3.5/4 |

**为什么 Token 很重要：**
1. **计费单位** — API 按 Token 数收费
2. **上下文限制** — 上下文窗口以 Token 为单位；窗口能开大 **≠** 窗内每一段信息都能被稳定「同等对待」，过长时会出现 **信噪比下降**（常与注意力稀释一起讨论），工程上仍要 **控窗、摘要、检索**，见 [§5.5](#55-幻觉与注意力稀释prompt-与采样侧) 与 [C6 §3](../C6-单Agent开发/理论讲解.md)。
3. **中文通常比英文消耗更多 Token**（同样的语义）

### 1.3 采样参数：Temperature / Top-p / Top-k

**本节概念**：每步生成时模型给出 **下一个 token 的概率分布**；**采样**是在该分布上取值；Temperature、top-p、top-k 是在 **改变分布形状或截断支撑集**，不是改模型权重。

LLM 生成下一个 Token 时，会对所有候选 Token 计算概率分布，采样参数控制如何从这个分布中选择。

**Temperature（温度）：**

```
温度 = 0.0  → 贪心解码，始终选概率最高的 Token（确定性输出）
温度 = 0.7  → 适度随机，平衡创造性与连贯性（常用默认值）
温度 = 1.5  → 高随机性，更有创意但可能不连贯
```

数学上：`P(token_i) = exp(logit_i / T) / Σ exp(logit_j / T)`

**Top-p（Nucleus Sampling）：**

- 从概率最高的 Token 开始，累加概率直到总和达到 p
- 只在这些 Token 中采样
- `top_p=0.9` 意味着只考虑累积概率前 90% 的 Token

**Top-k：**

- 只考虑概率最高的 k 个 Token
- `top_k=50` 意味着只在前 50 个候选中采样

**实用建议：**

| 场景 | Temperature | Top-p | 说明 |
|------|------------|-------|------|
| 代码生成 | 0.0-0.2 | 0.9 | 需要确定性和准确性 |
| 文案写作 | 0.7-0.9 | 0.95 | 需要创造力 |
| 数据提取 | 0.0 | 1.0 | 需要严格确定性 |
| 头脑风暴 | 1.0-1.2 | 0.95 | 需要多样性 |

> 注意：Temperature 和 Top-p 通常**不建议同时调整**，选择一个调节即可。

### 1.4 上下文窗口（Context Window）

上下文窗口是模型在一次对话中能"看到"的最大 Token 数量。

**主流模型上下文窗口（2025 年）：**

| 模型 | 上下文窗口 | 说明 |
|------|-----------|------|
| GPT-4o | 128K | OpenAI 主力模型 |
| Claude Opus/Sonnet | 200K | 超长上下文理解能力强 |
| DeepSeek-V3 | 128K | 国产开源标杆 |
| Qwen2.5 | 128K | 阿里通义系列 |
| Llama 3.1 | 128K | Meta 开源 |

**上下文管理策略：**

1. **滑动窗口** — 丢弃最早的消息，保留最近的
2. **摘要压缩** — 将历史对话总结为简短摘要
3. **RAG（检索增强生成）** — 只检索相关内容放入上下文
4. **分层缓存** — 核心系统提示 + 动态上下文

## 2. 主流大模型对比与选型

### 2.1 模型对比（2025 年主力模型）

| 维度 | GPT-4o | Claude Opus 4 | DeepSeek-V3 | Qwen2.5-72B | Llama 3.1-70B |
|------|--------|-------------|-------------|-------------|--------------|
| 厂商 | OpenAI | Anthropic | DeepSeek | 阿里巴巴 | Meta |
| 开源 | 否 | 否 | 是 | 是 | 是 |
| 中文能力 | 优 | 优 | 优 | 优（最佳之一） | 良 |
| 代码能力 | 优 | 优 | 优 | 优 | 优 |
| 推理能力 | 优 | 极优 | 优 | 良 | 良 |
| 上下文 | 128K | 200K | 128K | 128K | 128K |
| 本地部署 | 不可 | 不可 | 可（需大显存） | 可 | 可 |
| API 价格 | 较高 | 较高 | 低 | 低 | 免费（自托管） |

### 2.2 选型建议

```
需要最强推理/长文本 → Claude Opus/Sonnet
需要通用能力 + 生态 → GPT-4o
预算有限/国内部署 → DeepSeek-V3 / Qwen2.5
中文场景优先 → Qwen2.5 / DeepSeek-V3
本地私有部署 → Llama / Qwen / DeepSeek（搭配 Ollama）
轻量级任务/边缘 → Qwen2.5-7B / Llama-8B（本地小模型）
```

### 2.3 小模型的价值

不是所有场景都需要大模型：

- **分类/NER** → 7B 模型即可胜任
- **嵌入/相似度** → 专用 Embedding 模型（如 `bge-m3`）
- **本地隐私场景** → Ollama + 7B 模型
- **高并发低延迟** → 小模型 + 量化（GGUF/GPTQ/AWQ）

### 2.4 训练与适配：从头预训练、全量微调与 LoRA / QLoRA

**本节概念**：「训练大模型」在工程和简历里 **口径不一**；讨论算力与交付前，应先分清 **权重最初从哪来**、**这次训练动多少参数**。

**三层常见说法（从「最重」到「最省」）**

| 层次 | 通常叫法 | 参数怎么来 / 怎么变 | 典型主体 |
|------|----------|---------------------|----------|
| **从头预训练** | train from scratch、基座预训练 | 多为 **随机初始化** 起，经海量语料自监督训练，**整网权重由该次训练塑造成型**（口语里「所有参数都是训出来的」多指这层） | 厂商与研究机构；数据与算力天文级 |
| **全量微调** | full fine-tuning | **加载已有预训练权重**，在你的数据上 **仍更新几乎全部参数** | 有卡团队、要强改行为或格式的场景 |
| **参数高效微调（PEFT）** | LoRA、QLoRA、Adapter 等 | **主干大量冻结**，只训 **少量附加参数**（如低秩矩阵） | 一线交付最常见；多租户可「一底座多套适配器」 |

**与口语「微调」的关系**：**「微调大模型」常是统称**——凡在预训练权重之上再用监督/偏好数据训一段，都可叫微调。沟通时应追问：对方指的是 **全量微调** 还是 **LoRA/QLoRA**。  
**「全量全部训练」** 若指 **整模从随机到可用**，对应 **从头预训练**；若指 **在已有底座上整网权重一起更新**，对应 **全量微调**——二者 **起点与成本** 完全不同。

**LoRA 与 QLoRA（速览）**

| | **LoRA** | **QLoRA** |
|--|----------|-----------|
| **核心** | 在低秩子空间近似 **ΔW ≈ B·A**（秩 **r** 很小），**原权重 W 冻结** | 同上；底座 **W 常以 4-bit 等形式驻显存**，省显存大头 |
| **显存** | 显著低于全量微调 | 通常 **更低**，单卡可尝试更大底座或更大 batch |
| **取舍** | 多数领域/格式适配够用；极强行为改写有时不如全量 | 性价比常高；量化路径上可能有 **轻微数值损失**，需用评测验证 |

**和 RAG 的分工**：**RAG** 解决 **可引用、可更新的外部事实**；**微调（含 LoRA）** 更偏 **术语、口吻、输出形态与任务习惯**；二者常组合使用，见 [C3 §1.2](../C3-RAG与知识库/理论讲解.md)。

**本节小结（三张卡片）**

- **一句话定义**：先分清 **从头预训练**、**全量微调**、**PEFT（LoRA/QLoRA）** 各自动了谁、从哪份权重出发。  
- **典型场景**：单卡 24G 要对 70B 做 SFT，全量往往不现实，**QLoRA** 更常见。  
- **和 A1 类比**：全量微调像 **整库 fork 改核心逻辑**；LoRA 像 **插件或适配层**——动面小、版本多、回滚快。

## 3. Ollama 本地部署

### 3.1 安装

```bash
# macOS / Linux
curl -fsSL https://ollama.com/install.sh | sh

# macOS（Homebrew）
brew install ollama

# 验证安装
ollama --version
```

### 3.2 模型下载与管理

```bash
# 下载模型
ollama pull qwen2.5:7b          # 阿里通义 7B
ollama pull deepseek-v2:16b     # DeepSeek 16B
ollama pull llama3.1:8b         # Llama 3.1 8B
ollama pull codellama:7b        # 代码专用

# 查看已下载模型
ollama list

# 删除模型
ollama rm qwen2.5:7b

# 交互式对话
ollama run qwen2.5:7b
```

### 3.3 Ollama HTTP API

Ollama 默认在 `http://localhost:11434` 提供 REST API。

**生成接口（/api/generate）：**

```bash
curl http://localhost:11434/api/generate -d '{
  "model": "qwen2.5:7b",
  "prompt": "什么是 Transformer？",
  "stream": false,
  "options": {
    "temperature": 0.7,
    "top_p": 0.9,
    "num_predict": 512
  }
}'
```

**对话接口（/api/chat）：**

```bash
curl http://localhost:11434/api/chat -d '{
  "model": "qwen2.5:7b",
  "messages": [
    {"role": "system", "content": "你是一个 Python 专家"},
    {"role": "user", "content": "解释列表推导式"}
  ],
  "stream": false
}'
```

**流式输出：** 将 `stream` 设为 `true`（默认），响应以 SSE 方式逐行返回 JSON。

### 3.4 Ollama 参数配置

通过 Modelfile 自定义模型参数：

```dockerfile
FROM qwen2.5:7b

# 系统提示
SYSTEM "你是一位专业的 Python 开发助手。"

# 参数配置
PARAMETER temperature 0.7
PARAMETER top_p 0.9
PARAMETER num_predict 1024
PARAMETER stop "<|endoftext|>"
```

```bash
ollama create my-assistant -f Modelfile
ollama run my-assistant
```

## 4. OpenAI 兼容 API

### 4.1 统一接口规范

OpenAI 的 Chat Completions API 已成为事实上的行业标准，几乎所有 LLM 提供商都支持该接口格式：

| 提供商 | 兼容 OpenAI API | Endpoint |
|--------|----------------|----------|
| OpenAI | 原生 | `https://api.openai.com/v1` |
| Claude（via proxy） | 通过适配层 | 各代理服务不同 |
| DeepSeek | 原生兼容 | `https://api.deepseek.com/v1` |
| Qwen（通义） | 原生兼容 | `https://dashscope.aliyuncs.com/compatible-mode/v1` |
| Ollama | 原生兼容 | `http://localhost:11434/v1` |
| vLLM | 原生兼容 | 自定义端口 |

### 4.2 请求格式

```python
import requests

url = "https://api.openai.com/v1/chat/completions"
headers = {
    "Authorization": "Bearer sk-xxx",
    "Content-Type": "application/json"
}
payload = {
    "model": "gpt-4o",
    "messages": [
        {"role": "system", "content": "你是一个有用的助手"},
        {"role": "user", "content": "你好"}
    ],
    "temperature": 0.7,
    "max_tokens": 1024,
    "stream": False
}

response = requests.post(url, headers=headers, json=payload)
data = response.json()
```

### 4.3 响应格式

```json
{
  "id": "chatcmpl-xxx",
  "object": "chat.completion",
  "created": 1234567890,
  "model": "gpt-4o",
  "choices": [
    {
      "index": 0,
      "message": {
        "role": "assistant",
        "content": "你好！有什么可以帮助你的？"
      },
      "finish_reason": "stop"
    }
  ],
  "usage": {
    "prompt_tokens": 20,
    "completion_tokens": 15,
    "total_tokens": 35
  }
}
```

### 4.4 流式输出（SSE）

流式输出使用 Server-Sent Events 协议，设置 `"stream": true` 后，响应以 `data:` 前缀的行逐步返回：

```
data: {"id":"chatcmpl-xxx","choices":[{"delta":{"role":"assistant"},...}]}
data: {"id":"chatcmpl-xxx","choices":[{"delta":{"content":"你"},...}]}
data: {"id":"chatcmpl-xxx","choices":[{"delta":{"content":"好"},...}]}
data: {"id":"chatcmpl-xxx","choices":[{"delta":{"content":"！"},...}]}
data: [DONE]
```

- 每个 chunk 的 `choices[0].delta` 包含增量内容
- 第一个 chunk 通常只包含 `role`
- 最后一个 `data: [DONE]` 表示结束

## 5. Prompt Engineering 系统化

### 5.1 基础技巧

#### 角色设定（Role Prompting）

给模型一个明确的角色身份，可以显著提升特定领域的输出质量。

```
你是一位拥有 10 年经验的 Python 高级工程师。
你擅长代码审查，会关注：
1. 代码可读性
2. 性能问题
3. 安全漏洞
4. 最佳实践

请审查以下代码：
{code}
```

#### Few-shot（少样本学习）

通过提供示例，让模型理解期望的输入/输出格式。

```
将以下句子分类为"正面"或"负面"：

示例：
句子：这个产品非常好用 → 正面
句子：服务态度太差了 → 负面
句子：物流速度很快 → 正面

现在分类：
句子：{input} →
```

**Zero-shot vs Few-shot vs Many-shot：**
- Zero-shot：不给例子，直接问
- One-shot：给 1 个例子
- Few-shot：给 2-5 个例子
- Many-shot：给大量例子（利用长上下文窗口）

#### Chain-of-Thought (CoT)

引导模型逐步推理，而非直接给出答案。

```
请逐步思考以下问题：

问题：一个商店有 5 箱苹果，每箱 24 个。卖掉了总数的 3/8，还剩多少个？

让我们一步一步来：
1. 首先计算总数...
2. 然后计算卖掉的数量...
3. 最后计算剩余数量...
```

**Zero-shot CoT** — 只需加一句 "Let's think step by step"：

```
Q: 一个商店有 5 箱苹果，每箱 24 个。卖掉了总数的 3/8，还剩多少个？
A: Let's think step by step.
```

### 5.2 高级技巧

#### ReAct（Reasoning + Acting）

将推理和行动交替进行，适合需要工具调用的场景。

```
回答以下问题。你可以使用以下工具：
- search(query): 搜索信息
- calculate(expression): 计算数学表达式

格式：
Thought: 我需要思考什么
Action: 使用什么工具
Observation: 工具返回了什么
... (重复直到得到答案)
Answer: 最终答案

问题：北京和上海的人口之和是多少？
```

#### Tree-of-Thought (ToT)

探索多条推理路径，选择最优解。

```
问题：{problem}

请按照以下步骤解题：

步骤 1：生成 3 种不同的解题思路
```

### 5.4 常见 Prompt 反模式与优化

| 反模式 | 问题 | 优化方式 |
|--------|------|---------|
| 指令模糊 | "帮我写点代码" | 明确语言、功能、输入输出 |
| 信息过载 | 塞入过多无关上下文 | 只提供必要信息 |
| 格式缺失 | 不指定输出格式 | 明确 JSON/Markdown/列表 |

| 缺乏示例 | 对复杂任务不给例子 | 添加 Few-shot 示例 |

### 5.5 幻觉与注意力稀释：Prompt 与采样侧

**本节概念**：**幻觉**本质是模型在**证据不足或未被有效读入**时仍做「补全」；**注意力稀释**指上下文过长或噪声过多时，**有效信号在整窗内的信噪比下降**（还常伴随中间段相对难被稳定利用等现象）。二者相关但不等同：**稀释**会让该读到的证据读不稳，从而**间接加重幻觉**。

**降低幻觉（本章可直接落地的部分）**

| 手段 | 做法要点 |
|------|----------|
| **接地** | 事实类任务尽量 **RAG / 工具** 提供可引用片段，见 [C3 — RAG 与知识库](../C3-RAG与知识库/理论讲解.md) §8；勿把「百科」指望在权重记忆里。 |
| **可验证输出** | 结构化字段 + 程序校验、claim–evidence 对照，见 [C4 — Function Calling](../C4-Function-Calling与结构化输出/理论讲解.md) §6。 |
| **Prompt** | 要求 **有依据才断言**；缺证据时输出 **明确「未知」+ 缺什么**；关键约束在 **system 与任务段** 写清，避免埋在长材料正中仅靠一次提及。 |
| **采样** | 抽取、分类、强事实场景可适当 **降低 temperature / 收紧 top_p**；创意任务再调高（采样不是防幻觉万能药）。 |
| **流程** | 草稿 + **自检轮**（「上文是否支持结论？」）；高风险答案走 **人审或规则闸门**。 |

**缓解注意力稀释（本章视角）**

- **少而精**：只把与当前问题相关的材料放进窗；无关长文 = 噪声。  
- **关键信息摆放**：**任务说明、硬约束、最终问题**可在 **开头与结尾各出现一次**（中间大段只放证据）；避免「唯一真值」只出现一次且埋在超长上下文中段。  
- **会话态**：多轮历史会挤占证据位，需 **摘要 / 滑动窗口 / 外部记忆**，见 [C6 — 单 Agent](../C6-单Agent开发/理论讲解.md) §3。

**本节小结（三张卡片）**

- **一句话定义**：防幻觉 = **证据 + 约束 + 可校验**；防稀释 = **控长度、控噪声、关键指令短窗重复**。  
- **典型场景**：合同 RAG 把整份 PDF 原样塞进 prompt，模型既贵又容易忽略免责条款 —— 应先检索再组装短上下文。  
- **和 A1 类比**：长上下文像 **把全局变量全 import 进一个函数**；应只传 **调用所需的最小参数集**。

## 6. LangChain 入门

### 6.1 核心概念

**LangChain 是什么**：面向 LLM 应用的**编排层（orchestration）**——把「提示模板 → 模型调用 → 解析 / 后处理 → 检索 / 工具 / 记忆」等步骤，用统一接口串成可复用的流水线。它**不替代**模型本身的能力，而是帮你**少写胶水代码、少对接一次一家厂商的差异**（在理想情况下）。

**和「直接调官方 SDK」的分界**：官方 SDK 解决「这一次 HTTP/流式/鉴权怎么发」；LangChain 更偏向「这条业务链路里有哪些可替换的积木、怎么组合、怎么观测」。链路越简单，框架收益越小；链路越长（RAG、多步工具、多模型切换），抽象带来的收益与成本会一起放大。

**生态分包（读文档时先建立地图）**：实际项目里常见的是**多包组合**，而不是单一 `langchain` 大包一把梭。

| 包 / 方向 | 大致职责 |
|-----------|----------|
| `langchain-core` | 核心抽象：`Runnable`、消息结构、部分模板与解析器基类；**与具体厂商弱绑定**。 |
| `langchain-community` | 社区维护的集成：各家 ChatModel、向量库、Loader、Tool 等，**版本与质量参差**，升级时要盯 changelog。 |
| `langchain`（高层） | 更贴近「开箱即用」的链路与 Agent 封装，迭代快，示例多。 |
| **LangGraph**（同生态、偏独立心智模型） | 用**有向图 / 状态机**描述多步 Agent、分支与循环，适合复杂控制流；可与 LCEL 混用，但学习曲线在「图」而不是「管道」。 |
| **LangSmith**（可选） | 调用链 trace、数据集与评测等**观测与实验**能力；不用也能跑，用上排障通常更直观（注意成本与数据出境合规）。 |

**关键抽象：`Runnable` 与 LCEL**：组件实现统一的 `invoke` / `stream` / `batch`（以及异步变体）等入口，便于在中间层加日志、重试、缓存。LCEL（LangChain Expression Language）用 `|` 把多个 `Runnable` 连成**声明式管道**（见 §6.5），心智上等价于「函数组合」，优点是**组合方式一眼能读完**；缺点是调试时要习惯**栈更深、错误先打在框架层**。

**典型数据流（概念上的一条线）**：用户输入 →（可选）**Memory** 取出历史摘要或窗口 → **PromptTemplate** 拼出 messages → **ChatModel** 调用 → **OutputParser** 或结构化约束 → 若走 RAG 则在模型前后插入 **Retriever / 文档 Loader**；若走 Agent 则模型输出驱动 **Tool** 调用并进入下一轮。你脑中可以把它画成一条主链，RAG / Agent / Memory 是挂在链上的「侧枝或回路」。

```
LangChain 核心组件（与 §6.2–§6.5 代码示例对应）：

├── LLM / ChatModel          # 模型封装（§6.3）
├── PromptTemplate           # 提示词模板（§6.2）
├── OutputParser             # 输出解析器（§6.4）
├── Chain / LCEL             # 链：可组合 Runnable（§6.5）
├── Memory                   # 对话记忆（长会话需配合摘要/窗口，见前文 context 讨论）
├── Retriever                # 检索器（RAG 材料层）
└── Agent / Tools            # 智能体与工具调用（控制流复杂时可转向 LangGraph）
```

### 6.5 Chain（LCEL 语法）

LangChain 的新表达式语言（LCEL）使用 `|` 管道操作符组合组件：

```python
from langchain_core.prompts import ChatPromptTemplate
from langchain_community.chat_models import ChatOllama
from langchain_core.output_parsers import StrOutputParser

prompt = ChatPromptTemplate.from_template("用一句话解释{concept}")
model = ChatOllama(model="qwen2.5:7b")
parser = StrOutputParser()

# LCEL 链
chain = prompt | model | parser
result = chain.invoke({"concept": "量子计算"})
```

### 6.6 选型：优点、缺点与常见替代

**适合用 LangChain 的典型情况**

- **快速验证想法**：RAG、简单 Agent、多数据源拼接，需要大量「能跑的示例」缩短从 0 到 Demo 的时间。
- **多后端切换**：同一套抽象下换模型、换向量库、换 Embedding，对早期选型或 A/B 对比友好。
- **团队已熟悉其概念**：内部有规范与模板时，抽象成本可被摊薄。

**优点（概念层面）**

- **统一编排接口**：`Runnable` + LCEL 让「模板 | 模型 | 解析」一类流水线可读、可接流式与异步。
- **集成面广**：社区包覆盖常见模型、向量库、文件解析与工具，减少自己写适配器的次数。
- **与观测/实验工具衔接顺**：若采用 LangSmith 等，对多步链路的 trace 往往比从零接日志更省事。

**缺点（工程层面，需提前有心理预期）**

- **抽象层偏厚**：偏离默认路径时，容易变成「跟框架搏斗」；栈更深，**排障成本**高于薄封装 + 官方 SDK。
- **版本与示例漂移**：拆包与 API 迭代快，网上旧示例可能直接跑不通，升级要读 changelog。
- **「文档能跑」≠「生产能稳」**：超时、重试、并发、工具失败回退、流式边界等，往往仍需你在链外补齐。
- **简单链路可能过度工程**：单次调用 + 少量后处理时，引入整套框架的收益常抵不上依赖与心智负担。
- **类型与模块边界参差**：不同子包之间接口风格不完全一致，IDE 体验与可维护性因项目而异。
- **观测能力常与商业产品叙事绑定**：自建 trace 也可行，但社区默认路径可能让你感觉「全家桶」压力。

**常见替代（不是「更高级」，而是「更可控」）**

- **官方 SDK + 自研薄编排**：适合长期维护、强定制、团队更在意可控性与排障速度。
- **只做 RAG 时**：向量库官方客户端 + 自己的 chunk / rerank / 引用拼装，有时比堆集成更透明。
- **复杂 Agent 控制流**：除 LangGraph 外，也可用手写状态机、工作流引擎（Temporal 等）或更轻的任务图库，按团队栈选型。

**一句话收束**：LangChain 更像**可插拔积木 + 流水线 DSL**；学它的价值在于理解**「编排层」在系统里站哪一层**，而不是认定必须用这一家框架。第 4 天跑 `demos/05_langchain_basics.py` 时，建议对照本节：**哪些步骤用框架省了代码，哪些步骤你仍会想在框架外自己管。**

### 6.7 延伸：理想的编排框架长什么样（自由度与灵活性）

社区里常听到「LangChain 该重构」的声音，背后往往不是否定**编排层**本身，而是希望框架别再用「厚抽象 + 隐式约定」换走**可控性**。下面几条可作为**产品/架构层面的评价标准**（也适用于你自研薄编排时对照）。

- **更薄的稳定内核**：只承诺极少数长期稳定的契约——消息结构、单次调用的生命周期、取消/超时、观测钩子（span 边界、输入输出脱敏策略由调用方注入）。其余能力一律**可选插件**，避免默认全家桶。
- **一等公民的逃生口（escape hatch）**：任意一步都能无痛降级为「官方 SDK 一次调用」或「普通函数」，且类型与数据流在 IDE 里可追溯；框架是**适配器**，不是**中心宇宙**。
- **控制流与数据流解耦**：线性 `A | B | C` 适合简单流水线；一旦出现多分支、循环、人工审核、补偿事务，应自然落到**状态机 / 有向图 / 外部工作流引擎**，而不是把复杂逻辑硬塞进链式语法。
- **更少隐式魔法**：组合规则可读、可静态分析更好；错误栈应能尽快指向**业务代码**，而不是只在框架内部打转。
- **清晰的版本契约**：`core` 层 semver 纪律与迁移指南；`community` 集成允许快迭代，但与 core 的边界、弃用周期写清楚，降低「升一个小版本全红」的摩擦。
- **横切能力外置**：重试、限流、缓存、成本统计、审计日志更适合作为**你自己的中间件层**或 sidecar，而不是每个 Runnable 各有一套不完全兼容的实现。

**与学习的衔接**：即使未来换框架，你在 §6.1–§6.6 里建立的**分层心智**（材料 / 任务 / 编排 / 观测）仍然成立；LangChain 只是其中一种实现路径。

## 7. Token 计费与成本优化

### 7.1 主流模型定价（2025 年参考）

| 模型 | 输入价格 ($/1M tokens) | 输出价格 ($/1M tokens) |
|------|----------------------|----------------------|
| GPT-4o | $2.50 | $10.00 |
| GPT-4o-mini | $0.15 | $0.60 |
| Claude Sonnet 4 | $3.00 | $15.00 |
| Claude Haiku 3.5 | $0.80 | $4.00 |
| DeepSeek-V3 | $0.27 | $1.10 |
| Qwen-Plus | $0.80 | $2.00 |

> 价格随时调整，以各厂商官网为准。

### 7.2 Token 估算经验

- **英文**：1 Token ≈ 4 个字符 ≈ 0.75 个单词
- **中文**：1 Token ≈ 1.5-2 个汉字（取决于 tokenizer）
- **代码**：Token 消耗通常比自然语言多 20-50%

### 7.3 成本优化策略

1. **选择合适的模型** — 简单任务用小模型（GPT-4o-mini / Haiku）
2. **精简 Prompt** — 去掉冗余描述，用结构化格式
3. **缓存** — 相同 prompt 缓存结果，避免重复调用
4. **Prompt Caching** — Claude/GPT 都支持 prompt 前缀缓存，重复 system prompt 可降低成本
5. **流式输出** — 不影响 Token 消耗，但改善用户体验
6. **批量 API** — 非实时任务用 Batch API（通常有 50% 折扣）
7. **本地模型** — 高频低复杂度任务用 Ollama 本地模型，边际成本为零

### 7.4 成本监控

```python
# 从 API 响应中提取 usage 信息
usage = response["usage"]
input_tokens = usage["prompt_tokens"]
output_tokens = usage["completion_tokens"]

# 计算成本（以 GPT-4o 为例）
cost = (input_tokens / 1_000_000) * 2.50 + (output_tokens / 1_000_000) * 10.00
print(f"本次调用成本: ${cost:.6f}")
```

## 面试高频 Q&A

### Q1：Token 是什么？为什么不直接用字符？

**A**：Token 是 LLM 处理文本的基本单位，由 tokenizer 通过 BPE（Byte Pair Encoding）等算法分词得到。不用字符的原因：

1. **效率**：常见单词（如 "the"）编码为 1 个 token，比逐字符处理高效
2. **语义密度**：token 比单个字符携带更多语义信息
3. **词表大小可控**：纯字符词表太小（256），逐词词表太大（百万级），BPE 取平衡（3-10 万）

中文每个字通常 1-2 个 token；英文常见词 1 个 token，罕见词可能拆成多个。API 按 token 计费，所以中文同样内容通常比英文贵。

### Q2：temperature 和 top_p 有什么区别？生产中怎么设？

**A**：

| 参数 | 机制 | 效果 |
|------|------|------|
| **temperature** | 缩放 logits 分布 | 越高越随机，越低越确定 |
| **top_p** | 只从累计概率 ≤ p 的 token 中采样 | 动态裁剪候选集大小 |

生产建议：
- **事实性任务**（数据提取、代码生成、问答）：`temperature=0` 或 `0.1`
- **创意任务**（文案、对话）：`temperature=0.7-0.9`
- 通常只调一个，另一个保持默认（`top_p=1.0`）
- **不要** 同时把 temperature 和 top_p 都设得很极端

### Q3：Prompt Engineering 有哪些核心技巧？

**A**：按优先级排列：

1. **角色设定（System Prompt）**：明确模型的身份、能力边界、输出格式
2. **Few-shot 示例**：给 2-3 个输入→输出示例，比长篇描述更有效
3. **思维链（CoT）**：加"请一步步分析"让模型展示推理过程，复杂问题准确率显著提升
4. **输出格式约束**：明确要求 JSON / Markdown / 表格等格式
5. **负面约束**：明确说"不要做什么"（不要编造、不要回答超出范围的问题）
6. **分步拆解**：复杂任务拆成多轮，每轮只做一件事

面试加分点：能说出 **prompt 是可版本化的文本资产**——应该有版本号、有 A/B 测试、有评测数据集验证效果。

### Q4：OpenAI 兼容 API 是什么？为什么重要？

**A**：OpenAI 定义了 `/v1/chat/completions` 等 API 接口的请求/响应格式，现在几乎所有 LLM 厂商（Ollama、vLLM、LiteLLM、各大云厂商）都实现了兼容接口。

重要性：
- **代码可移植**：同一套代码换 `base_url` 就能切换模型提供方
- **工具生态通用**：LangChain、OpenAI SDK 等工具库可直接对接
- **降低锁定风险**：不被某个厂商绑定

```python
# 同一套代码，只换 base_url 和 api_key
# Ollama
client = OpenAI(base_url="http://localhost:11434/v1", api_key="ollama")
# 云端 API
client = OpenAI(base_url="https://api.openai.com/v1", api_key="sk-...")
# 其他兼容服务
client = OpenAI(base_url="https://api.deepseek.com/v1", api_key="ds-...")
```

### Q5：流式响应（Streaming）在生产中怎么用？

**A**：流式响应让用户看到"逐字出现"的效果，而不是等 10 秒后一次性看到完整回答。

工程要点：
- **前端**：用 SSE（Server-Sent Events）或 WebSocket 消费流
- **后端**：`stream=True` 后，响应是 generator，需要逐 chunk 拼接
- **注意事项**：
  - `tool_calls` 的 `arguments` 在流式模式下是分片到达的，需要拼完整再 `json.loads`
  - 客户端断连时要及时取消上游请求（避免浪费 token）
  - 用 `usage` 字段做成本统计时，流式模式下通常在最后一个 chunk 才返回

### Q6：模型的 Context Window 是什么？对应用有什么影响？

**A**：Context Window 是模型一次能处理的最大 token 数（输入 + 输出合计）。

| 模型 | Context Window |
|------|---------------|
| GPT-4o | 128K |
| Claude Opus 4 | 200K |
| Qwen3 | 32K-128K |
| Llama 3.1 | 128K |

对应用的影响：
- **RAG**：决定了一次能塞多少 chunk 进 prompt。窗口大不等于该塞满——注意力稀释（见 C3 §8.3）
- **多轮对话**：历史消息过长时需要截断或摘要
- **成本**：输入 token 越多越贵。128K 窗口塞满一次请求可能花几美元
- **本地模型**：窗口越大显存占用越高。A5000 24GB 跑 128K 窗口会很吃力

### Q7：LangChain 在实际项目中怎么定位？

**A**：LangChain 是一个编排框架，适合快速原型，但生产中需要谨慎评估：

**适合用的场景**：
- 快速搭建 RAG 原型（内置 loader/splitter/retriever）
- 需要对接多种向量库/LLM 的场景

**不适合的场景**：
- 对延迟敏感的生产环境（LangChain 有较多抽象层开销）
- 简单的 API 调用（直接用 OpenAI SDK 更轻量）
- 需要深度定制的场景（LangChain 的抽象可能成为障碍）

面试中的正确态度：LangChain 是工具之一，不是必须的。理解底层原理（messages 格式、embedding 流程、向量检索）比会用框架更重要。框架可以换，原理不会变。
