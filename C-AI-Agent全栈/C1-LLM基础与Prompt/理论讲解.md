## 1. LLM 基础概念

### 1.1 Transformer 架构概述

Transformer 是 2017 年 Google 在论文 "Attention Is All You Need" 中提出的架构，是当前所有主流大语言模型的基础。

**核心组件：**

```
输入 → Tokenizer → Embedding → [Transformer Blocks × N] → Output Head → 输出

每个 Transformer Block 包含：
├── Multi-Head Self-Attention（多头自注意力）
├── Layer Normalization
├── Feed-Forward Network（前馈网络）
└── Residual Connection（残差连接）
```

**关键机制 — Self-Attention（自注意力）：**

- 每个 Token 都会"关注"序列中的所有其他 Token
- 通过 Query(Q)、Key(K)、Value(V) 三个矩阵计算注意力权重
- 公式：`Attention(Q, K, V) = softmax(QK^T / √d_k) V`
- Multi-Head 机制让模型从多个"视角"理解文本

**架构变体：**

| 架构类型 | 代表模型 | 特点 |
|---------|---------|------|
| Encoder-Only | BERT | 双向注意力，适合理解任务（分类、NER） |
| Decoder-Only | GPT、Claude、LLaMA | 单向/因果注意力，自回归生成 |
| Encoder-Decoder | T5、BART | 编码+解码，适合翻译/摘要 |

当前主流 LLM 几乎都采用 **Decoder-Only** 架构。

### 1.2 Token 与 Tokenizer

**Token 不等于字/词。** Token 是模型处理文本的最小单位。

```
英文示例：
"Hello world" → ["Hello", " world"]  （2 tokens）
"unbelievable" → ["un", "believ", "able"]  （3 tokens）

中文示例：
"你好世界" → ["你好", "世界"]  （2 tokens，取决于 tokenizer）
"人工智能" → ["人工", "智能"]  （2 tokens）
```

**主流 Tokenizer 算法：**

| 算法 | 说明 | 使用者 |
|------|------|--------|
| BPE (Byte Pair Encoding) | 从字符级别逐步合并高频对 | GPT 系列 |
| WordPiece | 类似 BPE，基于似然合并 | BERT |
| SentencePiece | 直接在原始文本上训练，支持多语言 | LLaMA、Qwen |
| tiktoken | OpenAI 的 BPE 实现，速度快 | GPT-3.5/4 |

**为什么 Token 很重要：**
1. **计费单位** — API 按 Token 数收费
2. **上下文限制** — 上下文窗口以 Token 为单位
3. **中文通常比英文消耗更多 Token**（同样的语义）

### 1.3 采样参数：Temperature / Top-p / Top-k

LLM 生成下一个 Token 时，会对所有候选 Token 计算概率分布，采样参数控制如何从这个分布中选择。

**Temperature（温度）：**

```
温度 = 0.0  → 贪心解码，始终选概率最高的 Token（确定性输出）
温度 = 0.7  → 适度随机，平衡创造性与连贯性（常用默认值）
温度 = 1.5  → 高随机性，更有创意但可能不连贯
```

数学上：`P(token_i) = exp(logit_i / T) / Σ exp(logit_j / T)`

**Top-p（Nucleus Sampling）：**

- 从概率最高的 Token 开始，累加概率直到总和达到 p
- 只在这些 Token 中采样
- `top_p=0.9` 意味着只考虑累积概率前 90% 的 Token

**Top-k：**

- 只考虑概率最高的 k 个 Token
- `top_k=50` 意味着只在前 50 个候选中采样

**实用建议：**

| 场景 | Temperature | Top-p | 说明 |
|------|------------|-------|------|
| 代码生成 | 0.0-0.2 | 0.9 | 需要确定性和准确性 |
| 文案写作 | 0.7-0.9 | 0.95 | 需要创造力 |
| 数据提取 | 0.0 | 1.0 | 需要严格确定性 |
| 头脑风暴 | 1.0-1.2 | 0.95 | 需要多样性 |

> 注意：Temperature 和 Top-p 通常**不建议同时调整**，选择一个调节即可。

### 1.4 上下文窗口（Context Window）

上下文窗口是模型在一次对话中能"看到"的最大 Token 数量。

**主流模型上下文窗口（2025 年）：**

| 模型 | 上下文窗口 | 说明 |
|------|-----------|------|
| GPT-4o | 128K | OpenAI 主力模型 |
| Claude Opus/Sonnet | 200K | 超长上下文理解能力强 |
| DeepSeek-V3 | 128K | 国产开源标杆 |
| Qwen2.5 | 128K | 阿里通义系列 |
| Llama 3.1 | 128K | Meta 开源 |

**上下文管理策略：**

1. **滑动窗口** — 丢弃最早的消息，保留最近的
2. **摘要压缩** — 将历史对话总结为简短摘要
3. **RAG（检索增强生成）** — 只检索相关内容放入上下文
4. **分层缓存** — 核心系统提示 + 动态上下文

## 2. 主流大模型对比与选型

### 2.1 模型对比（2025 年主力模型）

| 维度 | GPT-4o | Claude Opus 4 | DeepSeek-V3 | Qwen2.5-72B | Llama 3.1-70B |
|------|--------|-------------|-------------|-------------|--------------|
| 厂商 | OpenAI | Anthropic | DeepSeek | 阿里巴巴 | Meta |
| 开源 | 否 | 否 | 是 | 是 | 是 |
| 中文能力 | 优 | 优 | 优 | 优（最佳之一） | 良 |
| 代码能力 | 优 | 优 | 优 | 优 | 优 |
| 推理能力 | 优 | 极优 | 优 | 良 | 良 |
| 上下文 | 128K | 200K | 128K | 128K | 128K |
| 本地部署 | 不可 | 不可 | 可（需大显存） | 可 | 可 |
| API 价格 | 较高 | 较高 | 低 | 低 | 免费（自托管） |

### 2.2 选型建议

```
需要最强推理/长文本 → Claude Opus/Sonnet
需要通用能力 + 生态 → GPT-4o
预算有限/国内部署 → DeepSeek-V3 / Qwen2.5
中文场景优先 → Qwen2.5 / DeepSeek-V3
本地私有部署 → Llama / Qwen / DeepSeek（搭配 Ollama）
轻量级任务/边缘 → Qwen2.5-7B / Llama-8B（本地小模型）
```

### 2.3 小模型的价值

不是所有场景都需要大模型：

- **分类/NER** → 7B 模型即可胜任
- **嵌入/相似度** → 专用 Embedding 模型（如 `bge-m3`）
- **本地隐私场景** → Ollama + 7B 模型
- **高并发低延迟** → 小模型 + 量化（GGUF/GPTQ/AWQ）

## 3. Ollama 本地部署

### 3.1 安装

```bash
# macOS / Linux
curl -fsSL https://ollama.com/install.sh | sh

# macOS（Homebrew）
brew install ollama

# 验证安装
ollama --version
```

### 3.2 模型下载与管理

```bash
# 下载模型
ollama pull qwen2.5:7b          # 阿里通义 7B
ollama pull deepseek-v2:16b     # DeepSeek 16B
ollama pull llama3.1:8b         # Llama 3.1 8B
ollama pull codellama:7b        # 代码专用

# 查看已下载模型
ollama list

# 删除模型
ollama rm qwen2.5:7b

# 交互式对话
ollama run qwen2.5:7b
```

### 3.3 Ollama HTTP API

Ollama 默认在 `http://localhost:11434` 提供 REST API。

**生成接口（/api/generate）：**

```bash
curl http://localhost:11434/api/generate -d '{
  "model": "qwen2.5:7b",
  "prompt": "什么是 Transformer？",
  "stream": false,
  "options": {
    "temperature": 0.7,
    "top_p": 0.9,
    "num_predict": 512
  }
}'
```

**对话接口（/api/chat）：**

```bash
curl http://localhost:11434/api/chat -d '{
  "model": "qwen2.5:7b",
  "messages": [
    {"role": "system", "content": "你是一个 Python 专家"},
    {"role": "user", "content": "解释列表推导式"}
  ],
  "stream": false
}'
```

**流式输出：** 将 `stream` 设为 `true`（默认），响应以 SSE 方式逐行返回 JSON。

### 3.4 Ollama 参数配置

通过 Modelfile 自定义模型参数：

```dockerfile

# 系统提示
```


## 4. OpenAI 兼容 API

### 4.1 统一接口规范

OpenAI 的 Chat Completions API 已成为事实上的行业标准，几乎所有 LLM 提供商都支持该接口格式：

| 提供商 | 兼容 OpenAI API | Endpoint |
|--------|----------------|----------|
| OpenAI | 原生 | `https://api.openai.com/v1` |
| Claude（via proxy） | 通过适配层 | 各代理服务不同 |
| DeepSeek | 原生兼容 | `https://api.deepseek.com/v1` |
| Qwen（通义） | 原生兼容 | `https://dashscope.aliyuncs.com/compatible-mode/v1` |
| Ollama | 原生兼容 | `http://localhost:11434/v1` |
| vLLM | 原生兼容 | 自定义端口 |

### 4.2 请求格式

```python
import requests

url = "https://api.openai.com/v1/chat/completions"
headers = {
    "Authorization": "Bearer sk-xxx",
    "Content-Type": "application/json"
}
payload = {
    "model": "gpt-4o",
    "messages": [
        {"role": "system", "content": "你是一个有用的助手"},
        {"role": "user", "content": "你好"}
    ],
    "temperature": 0.7,
    "max_tokens": 1024,
    "stream": False
}

response = requests.post(url, headers=headers, json=payload)
data = response.json()
```

### 4.3 响应格式

```json
{
  "id": "chatcmpl-xxx",
  "object": "chat.completion",
  "created": 1234567890,
  "model": "gpt-4o",
  "choices": [
    {
      "index": 0,
      "message": {
        "role": "assistant",
        "content": "你好！有什么可以帮助你的？"
      },
      "finish_reason": "stop"
    }
  ],
  "usage": {
    "prompt_tokens": 20,
    "completion_tokens": 15,
    "total_tokens": 35
  }
}
```

### 4.4 流式输出（SSE）

流式输出使用 Server-Sent Events 协议，设置 `"stream": true` 后，响应以 `data:` 前缀的行逐步返回：

```
data: {"id":"chatcmpl-xxx","choices":[{"delta":{"role":"assistant"},...}]}
data: {"id":"chatcmpl-xxx","choices":[{"delta":{"content":"你"},...}]}
data: {"id":"chatcmpl-xxx","choices":[{"delta":{"content":"好"},...}]}
data: {"id":"chatcmpl-xxx","choices":[{"delta":{"content":"！"},...}]}
data: [DONE]
```

- 每个 chunk 的 `choices[0].delta` 包含增量内容
- 第一个 chunk 通常只包含 `role`
- 最后一个 `data: [DONE]` 表示结束

## 5. Prompt Engineering 系统化

### 5.1 基础技巧

#### 角色设定（Role Prompting）

给模型一个明确的角色身份，可以显著提升特定领域的输出质量。

```
你是一位拥有 10 年经验的 Python 高级工程师。
你擅长代码审查，会关注：
1. 代码可读性
2. 性能问题
3. 安全漏洞
4. 最佳实践

请审查以下代码：
{code}
```

#### Few-shot（少样本学习）

通过提供示例，让模型理解期望的输入/输出格式。

```
将以下句子分类为"正面"或"负面"：

示例：
句子：这个产品非常好用 → 正面
句子：服务态度太差了 → 负面
句子：物流速度很快 → 正面

现在分类：
句子：{input} →
```

**Zero-shot vs Few-shot vs Many-shot：**
- Zero-shot：不给例子，直接问
- One-shot：给 1 个例子
- Few-shot：给 2-5 个例子
- Many-shot：给大量例子（利用长上下文窗口）

#### Chain-of-Thought (CoT)

引导模型逐步推理，而非直接给出答案。

```
请逐步思考以下问题：

问题：一个商店有 5 箱苹果，每箱 24 个。卖掉了总数的 3/8，还剩多少个？

让我们一步一步来：
1. 首先计算总数...
2. 然后计算卖掉的数量...
3. 最后计算剩余数量...
```

**Zero-shot CoT** — 只需加一句 "Let's think step by step"：

```
Q: 一个商店有 5 箱苹果，每箱 24 个。卖掉了总数的 3/8，还剩多少个？
A: Let's think step by step.
```

### 5.2 高级技巧

#### ReAct（Reasoning + Acting）

将推理和行动交替进行，适合需要工具调用的场景。

```
回答以下问题。你可以使用以下工具：
- search(query): 搜索信息
- calculate(expression): 计算数学表达式

格式：
Thought: 我需要思考什么
Action: 使用什么工具
Observation: 工具返回了什么
... (重复直到得到答案)
Answer: 最终答案

问题：北京和上海的人口之和是多少？
```

#### Tree-of-Thought (ToT)

探索多条推理路径，选择最优解。

```
问题：{problem}

请按照以下步骤解题：

步骤 1：生成 3 种不同的解题思路
思路 A：...
思路 B：...
思路 C：...

步骤 2：评估每种思路的可行性（1-10 分）
思路 A 评分：... 理由：...
思路 B 评分：... 理由：...
思路 C 评分：... 理由：...

步骤 3：选择最优思路，深入求解
选择思路：...
详细解答：...
```

#### Self-Consistency

多次采样（高 Temperature），取多数投票结果，提升推理准确性。

```python
# 伪代码
answers = []
for i in range(5):
    answer = llm.generate(prompt, temperature=0.7)
    answers.append(answer)

# 取出现次数最多的答案
final_answer = majority_vote(answers)
```

### 5.3 Prompt 模板设计原则

**CRAFT 框架：**

| 要素 | 说明 | 示例 |
|------|------|------|
| **C**ontext（上下文） | 背景信息 | "你在审查一个 Python Web 项目" |
| **R**ole（角色） | 模型角色 | "作为高级后端工程师" |
| **A**ction（行动） | 具体任务 | "审查以下代码的安全性" |
| **F**ormat（格式） | 输出格式 | "以 JSON 格式返回结果" |
| **T**one（语气） | 风格要求 | "简洁、专业" |

**模板变量：**

```python
template = """
【角色】
你是{role}。

【任务】
{task}

【输入】
{input}

【输出要求】
- 格式：{format}
- 语言：{language}
"""
```

### 5.4 常见 Prompt 反模式与优化

| 反模式 | 问题 | 优化方式 |
|--------|------|---------|
| 指令模糊 | "帮我写点代码" | 明确语言、功能、输入输出 |
| 信息过载 | 塞入过多无关上下文 | 只提供必要信息 |
| 格式缺失 | 不指定输出格式 | 明确 JSON/Markdown/列表 |
| 否定指令 | "不要做 X" | 改为"请做 Y"（正向指令） |
| 多任务混合 | 一个 prompt 做多件事 | 拆分为多个 prompt |
| 缺乏示例 | 对复杂任务不给例子 | 添加 Few-shot 示例 |

**优化前 vs 优化后：**

```
❌ 差的 prompt：
"总结这段文字"

✅ 好的 prompt：
"请用 3 个要点总结以下文章的核心观点，每个要点不超过 30 字。
输出格式：
1. ...
2. ...
3. ...

文章内容：
{article}"
```

## 6. LangChain 入门

### 6.1 核心概念

LangChain 是构建 LLM 应用的框架，提供了标准化的抽象层。

```
LangChain 核心组件：

├── LLM / ChatModel          # 模型封装
├── PromptTemplate            # 提示词模板
├── OutputParser              # 输出解析器
├── Chain                     # 链（串联多个步骤）
├── Memory                    # 对话记忆
├── Retriever                 # 检索器（用于 RAG）
└── Agent                     # 智能体（工具调用）
```

### 6.2 PromptTemplate

```python
from langchain.prompts import PromptTemplate, ChatPromptTemplate

# 简单模板
prompt = PromptTemplate.from_template(
    "请将以下文本翻译成{language}：\n{text}"
)
result = prompt.format(language="英文", text="你好世界")

# Chat 模板
chat_prompt = ChatPromptTemplate.from_messages([
    ("system", "你是一位{role}"),
    ("human", "{question}")
])
```

### 6.3 LLM 与 ChatModel

```python
from langchain_community.llms import Ollama
from langchain_community.chat_models import ChatOllama

# LLM（文本补全）
llm = Ollama(model="qwen2.5:7b", base_url="http://localhost:11434")

# ChatModel（对话）
chat = ChatOllama(model="qwen2.5:7b")
```

### 6.4 OutputParser

```python
from langchain.output_parsers import PydanticOutputParser
from pydantic import BaseModel, Field

class ReviewResult(BaseModel):
    score: int = Field(description="代码评分 1-10")
    issues: list[str] = Field(description="发现的问题列表")
    suggestion: str = Field(description="改进建议")

parser = PydanticOutputParser(pydantic_object=ReviewResult)

# parser.get_format_instructions() 返回格式说明，拼入 prompt
```

### 6.5 Chain（LCEL 语法）

LangChain 的新表达式语言（LCEL）使用 `|` 管道操作符组合组件：

```python
from langchain_core.prompts import ChatPromptTemplate
from langchain_community.chat_models import ChatOllama
from langchain_core.output_parsers import StrOutputParser

prompt = ChatPromptTemplate.from_template("用一句话解释{concept}")
model = ChatOllama(model="qwen2.5:7b")
parser = StrOutputParser()

# LCEL 链
chain = prompt | model | parser
result = chain.invoke({"concept": "量子计算"})
```

## 7. Token 计费与成本优化

### 7.1 主流模型定价（2025 年参考）

| 模型 | 输入价格 ($/1M tokens) | 输出价格 ($/1M tokens) |
|------|----------------------|----------------------|
| GPT-4o | $2.50 | $10.00 |
| GPT-4o-mini | $0.15 | $0.60 |
| Claude Sonnet 4 | $3.00 | $15.00 |
| Claude Haiku 3.5 | $0.80 | $4.00 |
| DeepSeek-V3 | $0.27 | $1.10 |
| Qwen-Plus | $0.80 | $2.00 |

> 价格随时调整，以各厂商官网为准。

### 7.2 Token 估算经验

- **英文**：1 Token ≈ 4 个字符 ≈ 0.75 个单词
- **中文**：1 Token ≈ 1.5-2 个汉字（取决于 tokenizer）
- **代码**：Token 消耗通常比自然语言多 20-50%

### 7.3 成本优化策略

1. **选择合适的模型** — 简单任务用小模型（GPT-4o-mini / Haiku）
2. **精简 Prompt** — 去掉冗余描述，用结构化格式
3. **缓存** — 相同 prompt 缓存结果，避免重复调用
4. **Prompt Caching** — Claude/GPT 都支持 prompt 前缀缓存，重复 system prompt 可降低成本
5. **流式输出** — 不影响 Token 消耗，但改善用户体验
6. **批量 API** — 非实时任务用 Batch API（通常有 50% 折扣）
7. **本地模型** — 高频低复杂度任务用 Ollama 本地模型，边际成本为零

### 7.4 成本监控

```python
# 从 API 响应中提取 usage 信息
usage = response["usage"]
input_tokens = usage["prompt_tokens"]
output_tokens = usage["completion_tokens"]

# 计算成本（以 GPT-4o 为例）
cost = (input_tokens / 1_000_000) * 2.50 + (output_tokens / 1_000_000) * 10.00
print(f"本次调用成本: ${cost:.6f}")
```
