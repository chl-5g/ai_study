## 0. 与 A1、C4 的衔接

**本节概念**：**单 Agent** 在实现层通常就是一个 **`while` 循环** 套 **LLM 调用**；循环体里分支处理 **无工具 / 有工具 / 用户中断**；这与 A1 里「**显式状态机优于深层递归**」一致。

### 从 C4 到 C6：从「一轮工具调用」到「多轮自主循环」

回忆 C4，你已经学会了一件事：LLM 不直接执行工具，而是通过 `tool_calls` 向宿主「提议」要调用什么函数，宿主执行后用 `role: tool` 消息把结果写回。那是 **一轮** 工具调用。

现在想象一个更复杂的场景：用户说「帮我查一下上海明天的天气，如果下雨就帮我取消明天的户外会议」。这句话背后需要：

1. 先调用天气 API（工具调用 1）
2. 看天气结果，判断是否下雨（推理）
3. 如果下雨，调用日历 API 取消会议（工具调用 2）
4. 生成最终回复告诉用户处理结果

**一轮调用搞不定**——你需要一个 **循环**，每一轮里模型推理一次，决定要不要调工具，调完再看结果，继续推理，直到任务完成。

这个循环就是 **Agent**。

### 类比理解

如果 C4 的一轮工具调用像 **纯函数**（输入确定，输出确定，一次完成），那 Agent 就像 **带状态的交互式程序**——你不知道它要跑几步，但你知道它 **必须在某个条件下停下来**。

就像你在终端里写了一个 `while True` 循环，如果不写 `break` 条件，它就永远跑下去。Agent 也一样：**循环何时退出**、**历史如何裁剪**、**记忆放哪一层**——这三个问题就是 C6 要回答的核心。

**本节小结（三张卡片）**

- **一句话定义**：单 Agent = **带工具的多轮状态机**。
- **典型场景**：「查三次库还不满意就转人工」—— 这是产品策略，不是 prompt 里一句话能保障的。
- **和 A1 类比**：循环像 **`for` + `try/finally`** —— 无论如何要关连接、打 trace、归还预算。

## 1. Agent 与「一次 ChatCompletion」

**本节概念**：**单次补全**假设「一问一答」；**Agent** 假设「一目标多步」，中间可多次调用外部世界。

### 1.1 对比表

| | 单次补全 | Agent |
|--|----------|--------|
| 轮次 | 通常 1 次往返 | **多轮**内部循环 |
| 工具 | 可选 0～1 轮 | **可能多轮** tool |
| 控制 | 用户发一句 | **程序驱动** + 用户目标 |
| 状态 | 无状态（每次独立） | **有状态**（消息历史累积） |
| 停止 | API 返回即结束 | **需要显式停止条件** |

### 1.2 为什么不能「多调几次 API」凑合？

你可能想：我在外面套个 `for` 循环，每次调一次 `chat.completions`，不就行了？

问题出在这几个地方：

1. **上下文爆炸**：每轮循环，messages 列表都在增长。10 轮循环后，可能已经有几万 token 了。不管理历史，要么超出上下文窗口，要么花费暴涨。
2. **死循环**：模型可能反复调同一个工具，或者永远觉得「还没完成」，循环永不退出。
3. **一致性**：早期轮次的关键信息可能被后续消息淹没，模型「忘了」用户的约束。

Agent 框架要解决的就是这三个问题：**历史管理**、**停止条件**、**记忆分层**。

### 1.3 Agent 的最小定义

用代码说话，一个 Agent 的最小骨架是：

```python
def run_agent(user_goal: str, tools: list[dict], max_steps: int = 10) -> str:
    """最小 Agent 骨架：循环 + 工具调用 + 停止条件"""
    messages = [
        {"role": "system", "content": "你是一个有用的助手。"},
        {"role": "user", "content": user_goal},
    ]
    
    for step in range(max_steps):
        # 1. 调用 LLM
        response = call_llm(messages, tools=tools)
        assistant_msg = response.choices[0].message
        messages.append(assistant_msg)  # 把 assistant 消息追加到历史
        
        # 2. 判断：有没有工具调用？
        if not assistant_msg.tool_calls:
            # 没有工具调用 → 任务完成，返回最终回复
            return assistant_msg.content
        
        # 3. 执行每个工具调用
        for tool_call in assistant_msg.tool_calls:
            result = execute_tool(tool_call)  # 宿主执行
            messages.append({
                "role": "tool",
                "tool_call_id": tool_call.id,
                "content": result,
            })
    
    # 超过最大步数，强制返回
    return "抱歉，我已经尝试了多次但未能完成任务。"
```

**这段代码上方说的**：接下来演示 Agent 的最小结构——一个带 `max_steps` 限制的 `for` 循环。

**这段代码下方逐行解释**：

- **第 3-6 行**：初始化 `messages` 列表，这是 Agent 的 **全部状态**。system 消息设定角色，user 消息传入目标。
- **第 8 行**：`for step in range(max_steps)` 是 **硬上限**——不管发生什么，最多跑 `max_steps` 轮。
- **第 10-11 行**：调 LLM，把 assistant 回复 **追加** 到 messages。注意：必须追加，因为如果有 tool_calls，后面的 tool 消息需要对齐 `tool_call_id`。
- **第 14-15 行**：如果 assistant 没有发起工具调用，说明它认为任务已经完成，直接返回。
- **第 18-22 行**：有工具调用就逐个执行，把结果以 `role: tool` 的格式追加。这和 C4 学的一模一样。
- **第 25 行**：超过最大步数还没完成，返回一个兜底回复。这比让循环无限跑下去安全得多。

### 读到这里你应该理解

- Agent 的核心结构就是一个 **循环 + 分支判断**
- 循环的退出有两种：**模型不再调用工具**（正常完成）和 **超过最大步数**（兜底保护）
- `messages` 列表是 Agent 的 **唯一状态**，所有的历史都在里面

**本节小结（三张卡片）**

- **一句话定义**：Agent 把 **对话 API** 当成 **规划与执行的引擎**，而不是一次性文本生成器。
- **典型场景**：用户只说「帮我订明天最早一班高铁」，背后需要多工具、多校验。
- **和 A1 类比**：单次调用像 **纯函数**；Agent 像 **带 I/O 副作用的交互程序**。

## 2. ReAct 循环（思想与实现）

**本节概念**：**ReAct**（Reason + Act）是 2022 年 Yao et al. 提出的范式。核心思想是让模型 **交替执行「推理」和「行动」**，而不是一口气想完再做、或不想就做。在工程上常落地为：**模型推理 → 若有 tool_calls 则执行 → 把观察写回 messages → 再推理**，直到无工具或触发停止。

### 2.1 为什么不能只推理或只行动？

**只推理不行动（纯 CoT）**：模型可以写出漂亮的推理链，但如果不能查数据库、调 API，推理再正确也是「纸上谈兵」。它可能说「根据我的知识，上海明天会下雨」——但这个「知识」可能已经过时了。

**只行动不推理（盲目调工具）**：模型拿到工具就直接调，不先想一想「我应该先查什么」。结果可能是：先取消了会议，再去查天气——顺序反了。

ReAct 的核心是 **交替**：

```
思考: 用户要我查天气，如果下雨就取消会议。我应该先查天气。
行动: 调用 get_weather(city="上海", date="2026-04-17")
观察: {"weather": "rain", "temperature": 18, "humidity": 85}
思考: 天气结果显示下雨，我需要取消明天的户外会议。
行动: 调用 cancel_meeting(meeting_id="outdoor-001")
观察: {"status": "cancelled", "message": "已取消"}
思考: 天气查了，会议也取消了，我可以给用户最终回复了。
最终回答: 上海明天预报有雨（18°C），已帮您取消了明天的户外会议（outdoor-001）。
```

### 2.2 ReAct 在 OpenAI API 中的实现

OpenAI 的 `tool_calls` 机制 **天然就是 ReAct 的工程实现**。你不需要手动解析「思考/行动/观察」——API 帮你做了：

- **思考**：模型内部的推理（有时体现在 `content` 中，有时不可见）
- **行动**：`tool_calls` 字段
- **观察**：你写回的 `role: tool` 消息

完整实现：

```python
import json
import hashlib
from typing import Any


# ── 工具注册表 ──────────────────────────────────────────
# 把所有可用工具放在一个 dict 里，key 是工具名，value 是可调用函数。
# 这样做的好处：
# 1. 白名单天然生效——不在 dict 里的工具名会被拒绝
# 2. 易于测试——可以用 mock 函数替换真实实现
TOOL_REGISTRY: dict[str, callable] = {}

def register_tool(name: str, description: str, parameters: dict):
    """装饰器：注册工具到全局注册表，同时生成 OpenAI 格式的 tool schema"""
    def decorator(func):
        TOOL_REGISTRY[name] = {
            "function": func,
            "schema": {
                "type": "function",
                "function": {
                    "name": name,
                    "description": description,
                    "parameters": parameters,
                },
            },
        }
        return func
    return decorator


# ── 注册两个示例工具 ──────────────────────────────────────
@register_tool(
    name="get_weather",
    description="查询指定城市的天气预报",
    parameters={
        "type": "object",
        "properties": {
            "city": {"type": "string", "description": "城市名称"},
            "date": {"type": "string", "description": "日期，格式 YYYY-MM-DD"},
        },
        "required": ["city", "date"],
    },
)
def get_weather(city: str, date: str) -> dict:
    """实际的天气查询函数（这里用 mock 数据演示）"""
    return {"city": city, "date": date, "weather": "rain", "temperature": 18}


@register_tool(
    name="cancel_meeting",
    description="取消指定 ID 的会议",
    parameters={
        "type": "object",
        "properties": {
            "meeting_id": {"type": "string", "description": "会议 ID"},
        },
        "required": ["meeting_id"],
    },
)
def cancel_meeting(meeting_id: str) -> dict:
    """实际的会议取消函数（mock）"""
    return {"status": "cancelled", "meeting_id": meeting_id}


# ── 工具执行器 ──────────────────────────────────────────
def execute_tool_call(tool_call) -> str:
    """
    安全执行一次工具调用。
    
    为什么要单独抽成函数？
    1. 白名单检查集中在一处
    2. 异常处理统一——工具执行失败时返回错误信息而非让程序崩溃
    3. 审计日志集中打印
    """
    name = tool_call.function.name
    
    # 白名单检查
    if name not in TOOL_REGISTRY:
        return json.dumps({"error": f"未知工具: {name}"}, ensure_ascii=False)
    
    # 解析参数
    try:
        args = json.loads(tool_call.function.arguments)
    except json.JSONDecodeError as e:
        return json.dumps({"error": f"参数解析失败: {e}"}, ensure_ascii=False)
    
    # 执行
    try:
        result = TOOL_REGISTRY[name]["function"](**args)
        return json.dumps(result, ensure_ascii=False)
    except Exception as e:
        return json.dumps({"error": f"执行失败: {e}"}, ensure_ascii=False)


# ── ReAct 主循环 ──────────────────────────────────────────
def react_loop(
    user_goal: str,
    system_prompt: str = "你是一个有用的助手。根据用户目标，使用可用工具完成任务。",
    max_steps: int = 10,
    max_repeated: int = 3,
) -> str:
    """
    完整的 ReAct 循环实现。
    
    参数:
        user_goal: 用户的目标/问题
        system_prompt: 系统提示词
        max_steps: 最大循环步数（硬上限）
        max_repeated: 同一工具+参数连续重复多少次就强制中止
    
    返回:
        Agent 的最终回复文本
    """
    # 初始化消息历史
    messages = [
        {"role": "system", "content": system_prompt},
        {"role": "user", "content": user_goal},
    ]
    
    # 收集所有工具的 schema，传给 LLM
    tools = [info["schema"] for info in TOOL_REGISTRY.values()]
    
    # 重复检测器：记录最近的 (工具名, 参数hash)
    recent_calls: list[str] = []
    
    for step in range(max_steps):
        print(f"\n{'='*50}")
        print(f"Step {step + 1}/{max_steps}")
        print(f"{'='*50}")
        
        # ── 1. 调用 LLM ──
        response = call_llm(messages, tools=tools)
        assistant_msg = response.choices[0].message
        
        # 必须把 assistant 消息追加到历史
        # 即使有 tool_calls，也要先追加 assistant，再追加 tool 结果
        messages.append(assistant_msg)
        
        # ── 2. 没有 tool_calls → 任务完成 ──
        if not assistant_msg.tool_calls:
            print(f"✓ Agent 完成，最终回复: {assistant_msg.content[:100]}...")
            return assistant_msg.content
        
        # ── 3. 有 tool_calls → 逐个执行 ──
        for tc in assistant_msg.tool_calls:
            # 重复检测
            call_key = f"{tc.function.name}:{hashlib.md5(tc.function.arguments.encode()).hexdigest()}"
            recent_calls.append(call_key)
            
            # 检查最近 N 次是否完全相同
            if len(recent_calls) >= max_repeated:
                last_n = recent_calls[-max_repeated:]
                if len(set(last_n)) == 1:
                    print(f"⚠ 检测到重复调用 {tc.function.name}，强制中止")
                    return "检测到重复操作，已中止。请检查任务描述是否清晰。"
            
            # 执行工具
            print(f"→ 调用工具: {tc.function.name}({tc.function.arguments})")
            result = execute_tool_call(tc)
            print(f"← 工具返回: {result[:200]}")
            
            # 把工具结果写回 messages
            messages.append({
                "role": "tool",
                "tool_call_id": tc.id,
                "content": result,
            })
    
    # 超过最大步数
    print(f"⚠ 达到最大步数 {max_steps}，强制结束")
    return "已达到最大操作步数，任务未完成。请简化需求或联系人工。"
```

**代码逐段解释**：

**工具注册表（第 9-30 行）**：用装饰器模式注册工具。每个工具同时保存 **执行函数** 和 **OpenAI 格式的 schema**。这样做比手动维护两个列表更安全——添加新工具时不会忘记写 schema。

**工具执行器（第 67-87 行）**：集中处理三类错误：工具不存在（白名单）、参数 JSON 解析失败、执行异常。注意所有错误都 **返回 JSON 字符串** 而非抛异常——因为错误信息要写回 messages 让模型知道出了什么问题，模型可能会据此调整策略。

**ReAct 主循环（第 91-159 行）**：
- `recent_calls` 是 **重复检测器**：如果模型连续 3 次调同一个工具、传同样的参数，说明它卡住了，强制中止。
- 每轮循环的流程是固定的：调 LLM → 追加 assistant → 判断有无 tool_calls → 执行 → 追加 tool → 下一轮。
- 两个退出点：模型不再调用工具（正常完成）、超过 `max_steps`（兜底保护）。

### 2.3 观察的质量决定闭环的质量

ReAct 循环里最容易被忽视的环节是 **观察（Observation）**——也就是工具执行后写回给模型的结果。

**好的观察**：

```json
{"order_id": "ORD-20260416-001", "status": "shipped", "tracking_no": "SF1234567890", "eta": "2026-04-18"}
```

机器生成、结构化、可验证。模型拿到后能精确推理下一步。

**坏的观察**：

```
查询成功了，订单已经发货了，大概后天到。
```

含糊的自然语言。「大概后天」到底是哪天？「查询成功了」这个信息有什么用？模型拿到这种观察，推理质量会直线下降。

**原则**：观察必须是 **机器生成的客观字符串**（JSON 格式的查询结果、错误码+错误消息、状态枚举值）。不要把含糊的自然语言当观察——那等于在闭环里注入噪声。

### 读到这里你应该理解

- ReAct = **推理与行动交替**，不是一口气想完再做
- 在 OpenAI API 中，`tool_calls` 就是「行动」，`role: tool` 就是「观察」
- 观察必须是结构化的客观数据，不是含糊的自然语言
- 重复检测是防死循环的第一道防线

**本节小结（三张卡片）**

- **一句话定义**：ReAct 闭环的灵魂是 **可验证的观察**，不是华丽的 CoT 话术。
- **典型场景**：SQL 执行失败，应把 **错误码 + 截断 SQL** 写回，而不是「查询失败请重试」糊弄模型。
- **和 A1 类比**：观察像 **单元测试的 assert 信息** —— 越具体，下一轮修复越快。

## 3. 状态：消息历史与摘要

**本节概念**：**messages** 列表是 Agent 的 **显式状态**。每调一次 LLM，历史就增长一轮。不裁剪会 **上下文爆炸**；乱裁剪会 **丢失关键 tool 结果**。

### 3.1 上下文爆炸是什么？

想象你在微信群里接力聊天。每个人发的消息都会被所有人看到。一开始大家各说一句，10 条消息很快就看完了。但如果群里有个人不停地发长文，几十轮下来整个聊天记录变成了好几万字——后来的人根本没耐心从头看，只看最近几条。

Agent 也是一样的。`messages` 列表线性增长，每一轮增加 **至少两条消息**（assistant + tool），长工具返回值可能一条就几千 token。10 轮循环下来：

```
初始: system(200 tokens) + user(100 tokens) = 300 tokens
第1轮: + assistant(150) + tool(500) = 950
第2轮: + assistant(200) + tool(800) = 1950
...
第10轮: 可能已经 8000-15000 tokens
```

如果用的是 4K 上下文的模型，第 5 轮就可能溢出。即使用 128K 上下文的模型，token 数 = 钱，不管理历史 = 烧钱。

### 3.2 裁剪策略

**策略一：滑动窗口——保留最近 K 轮**

```python
def trim_messages(messages: list[dict], max_pairs: int = 5) -> list[dict]:
    """
    保留 system 消息 + 最近 max_pairs 轮对话。
    
    一轮 = 一个 assistant 消息 + 它关联的所有 tool 消息。
    system 和最初的 user 消息永远保留（它们定义了任务）。
    """
    # 永远保留的部分
    system_msgs = [m for m in messages if m.get("role") == "system"]
    first_user = next((m for m in messages if m.get("role") == "user"), None)
    preserved = system_msgs + ([first_user] if first_user else [])
    
    # 剩余消息
    rest = [m for m in messages if m not in preserved]
    
    # 从后往前数 max_pairs 轮
    # 简单做法：保留最后 max_pairs * 3 条消息（每轮约 assistant + 1-2 tool）
    keep_count = max_pairs * 3
    trimmed = rest[-keep_count:] if len(rest) > keep_count else rest
    
    return preserved + trimmed
```

**这段代码在做什么**：把 messages 分成「永远保留」和「可裁剪」两部分。system 消息和第一条 user 消息（任务定义）永远不能丢。其他消息只保留最近 K 轮。

**优点**：实现简单，开销为零。  
**缺点**：早期轮次的信息完全丢失。如果用户在第 1 轮说了「不要日文输出」，到第 8 轮就被裁掉了，模型可能开始输出日文。

**策略二：摘要压缩——把早期历史压成一段总结**

```python
def summarize_old_messages(
    messages: list[dict],
    keep_recent: int = 3,
) -> list[dict]:
    """
    把早期消息用 LLM 压缩成一段摘要，插在 system 消息后面。
    
    流程:
    1. 保留 system + 最近 keep_recent 轮
    2. 把中间被裁掉的消息交给 LLM 生成摘要
    3. 摘要作为一条 system 消息插入
    """
    system_msgs = [m for m in messages if m.get("role") == "system"]
    non_system = [m for m in messages if m.get("role") != "system"]
    
    if len(non_system) <= keep_recent * 3:
        return messages  # 不够长，不需要摘要
    
    # 分割：要摘要的 vs 保留的
    cutoff = len(non_system) - keep_recent * 3
    to_summarize = non_system[:cutoff]
    to_keep = non_system[cutoff:]
    
    # 用 LLM 生成摘要
    summary_prompt = (
        "请用 3-5 个要点总结以下对话历史，保留所有关键事实、"
        "用户约束和工具执行结果。不要遗漏任何重要数字或 ID。\n\n"
    )
    for msg in to_summarize:
        role = msg.get("role", "unknown")
        content = msg.get("content", "")
        if content:
            summary_prompt += f"[{role}]: {content[:500]}\n"
    
    summary = call_llm_simple(summary_prompt)  # 用小模型生成摘要
    
    # 构建新的消息列表
    summary_msg = {
        "role": "system",
        "content": f"[历史摘要] {summary}",
    }
    
    return system_msgs + [summary_msg] + to_keep
```

**优点**：保留了早期的关键信息（约束、ID、结论）。  
**缺点**：摘要本身需要一次 LLM 调用（额外成本+延迟），而且是 **有损压缩**——摘要可能遗漏细节。

### 3.3 关键 tool 结果的保留

裁剪时有一个特别容易踩的坑：**tool 消息和 assistant 消息是配对的**。

```
messages = [
    ...,
    {"role": "assistant", "content": null, "tool_calls": [{"id": "call_abc", ...}]},  # ← 这个
    {"role": "tool", "tool_call_id": "call_abc", "content": "..."},                     # ← 和这个是一对
]
```

如果你裁掉了 assistant 但保留了 tool，或者反过来，API 会报错。**必须成对裁剪或成对保留**。

更好的做法：对「关键」的工具结果打标记（比如用户确认过的订单号、最终状态），裁剪时永远保留这些标记过的消息对。

### 读到这里你应该理解

- `messages` 是 Agent 的全部状态，会线性增长
- 裁剪有两种基本策略：滑动窗口（简单但丢信息）和摘要压缩（保留信息但有成本）
- tool 消息和 assistant 消息必须成对操作，不能单独裁剪

**本节小结（三张卡片）**

- **一句话定义**：状态管理 = **在预算内保留最大信息量**。
- **典型场景**：长会话里早期用户给了约束「不要日文输出」，摘要必须保留该约束。
- **和 A1 类比**：像 **`deque(maxlen=...)`** —— 超过 maxlen 要定义丢哪头、是否先归档。

## 4. 记忆：短期、摘要、长期

**本节概念**：Agent 的记忆不止是 `messages` 列表。就像人有工作记忆（正在想的事）、短期记忆（今天发生的事）和长期记忆（一直记得的事）一样，Agent 的记忆也分层。

### 4.1 三层记忆模型

| 层级 | 存储位置 | 生命周期 | 容量 | 读取方式 |
|------|----------|----------|------|----------|
| **短期记忆** | `messages` 列表（上下文窗口） | 当前会话 | 受限于 context window | 直接在上下文中 |
| **中期记忆** | 会话摘要 | 当前会话 | 几百 token 的文本 | 插入 system 消息 |
| **长期记忆** | 外部存储（向量库/数据库） | 跨会话持久化 | 理论无限 | 通过 RAG 检索或工具查询 |

### 4.2 短期记忆：messages 列表

这就是第 3 节讲的 `messages`。它的特点是：

- **全量可见**：上下文窗口里的每条消息，模型都能「看到」
- **容量有限**：受限于模型的上下文窗口长度
- **会话结束即消失**：对话关了，messages 就没了

适合存放：**当前任务的即时信息**，比如正在处理的订单号、刚查到的天气数据、用户刚说的约束条件。

### 4.3 中期记忆：会话摘要

第 3 节的 `summarize_old_messages` 就是中期记忆的实现。它把早期历史压缩成摘要，插回 system 消息里。

适合存放：**本次会话的关键事实和约束**，比如「用户偏好中文回答」「已确认订单号 ORD-001 需要退款」。

### 4.4 长期记忆：外部存储

长期记忆跨会话存在，需要外部存储。两种常见实现：

**方式一：向量库（RAG 检索）**

```python
def store_user_fact(user_id: str, fact: str, vector_db):
    """把用户事实存入向量库，下次对话时可以检索出来"""
    embedding = get_embedding(fact)
    vector_db.insert(
        id=f"{user_id}_{hash(fact)}",
        vector=embedding,
        metadata={"user_id": user_id, "fact": fact, "created_at": "2026-04-16"},
    )


def recall_user_facts(user_id: str, query: str, vector_db, top_k: int = 5) -> list[str]:
    """根据当前对话内容，从向量库检索相关的用户事实"""
    query_embedding = get_embedding(query)
    results = vector_db.search(
        vector=query_embedding,
        filter={"user_id": user_id},
        top_k=top_k,
    )
    return [r.metadata["fact"] for r in results]
```

这和 C3（RAG）学的检索逻辑完全一样，只是检索的不是文档，而是 **用户事实**。

**方式二：关系数据库（结构化查询）**

```python
def store_user_preference(user_id: str, key: str, value: str, db):
    """存储用户偏好到关系数据库"""
    db.execute(
        "INSERT INTO user_preferences (user_id, key, value, updated_at) "
        "VALUES (?, ?, ?, datetime('now')) "
        "ON CONFLICT(user_id, key) DO UPDATE SET value=?, updated_at=datetime('now')",
        (user_id, key, value, value),
    )


def get_user_preferences(user_id: str, db) -> dict[str, str]:
    """查询用户的所有偏好"""
    rows = db.execute(
        "SELECT key, value FROM user_preferences WHERE user_id = ?",
        (user_id,),
    ).fetchall()
    return {row[0]: row[1] for row in rows}
```

适合 **结构明确** 的信息（用户偏好、订单状态、会员等级），不需要语义检索。

### 4.5 在 Agent 循环中集成长期记忆

```python
def react_loop_with_memory(user_id: str, user_goal: str, vector_db, pref_db):
    """在 ReAct 循环开始前，先拉取长期记忆，拼入 system prompt"""
    
    # 1. 检索相关的用户事实
    facts = recall_user_facts(user_id, user_goal, vector_db)
    facts_text = "\n".join(f"- {f}" for f in facts) if facts else "（无历史记录）"
    
    # 2. 查询用户偏好
    prefs = get_user_preferences(user_id, pref_db)
    prefs_text = "\n".join(f"- {k}: {v}" for k, v in prefs.items()) if prefs else "（无偏好设置）"
    
    # 3. 把长期记忆拼入 system prompt
    system_prompt = f"""你是一个有用的助手。

## 用户历史信息
{facts_text}

## 用户偏好
{prefs_text}

请基于以上信息和可用工具完成用户的请求。"""
    
    # 4. 正常走 ReAct 循环
    return react_loop(user_goal, system_prompt=system_prompt)
```

**核心思路**：长期记忆在 **循环开始前** 拉取，注入到 system prompt 里。Agent 循环内部不需要关心记忆从哪来——它只看 messages 列表。

### 4.6 遗忘权与合规

记忆分层之后，有一个必须考虑的问题：**当用户要求删除数据时，三层记忆必须一致清除**。

- **短期**：清空当前 messages（会话结束自动清除）
- **中期**：删除摘要
- **长期**：从向量库和数据库中删除该用户的所有记录

如果你只删了数据库但没删向量库，下次检索还是能检索到——这就是数据泄漏。

### 读到这里你应该理解

- 记忆分三层：短期（messages）、中期（摘要）、长期（外部存储）
- 短期记忆容量有限但读取最快，长期记忆容量无限但需要检索
- 长期记忆的两种实现：向量库（语义检索）和关系数据库（结构化查询）
- 用户要求删除时，三层记忆必须一致清除

**本节小结（三张卡片）**

- **一句话定义**：记忆分层解决 **「热数据在上下文，冷数据在外存」**。
- **典型场景**：用户偏好「素食」存长期，当餐点菜上下文放短期。
- **和 A1 类比**：像 **L1/L2/L3 缓存** —— 命中策略与一致性都要设计，不是默认 magically 正确。

## 5. 停止条件与防死循环

**本节概念**：**不写停止条件的 Agent 循环**就像不写 `break` 的 `while True`——在图灵完备意义上可能永远不停。工程上必须 **硬上限 + 软启发 + 用户取消** 三道防线。

### 5.1 为什么模型会死循环？

你可能觉得「模型应该知道什么时候该停」。但实际上，模型经常陷入以下循环：

**场景一：同一工具反复调用**
```
Step 1: 调用 search(query="Python GIL")  → 结果: 3篇文章
Step 2: 调用 search(query="Python GIL")  → 结果: 同样3篇文章
Step 3: 调用 search(query="Python GIL")  → 结果: 同样3篇文章
...（模型认为结果不够好，一直重试）
```

**场景二：两个工具互相调用**
```
Step 1: 调用 get_order(id="001")  → 结果: 状态=待支付
Step 2: 调用 check_payment(order="001")  → 结果: 未支付
Step 3: 调用 get_order(id="001")  → 结果: 状态=待支付
Step 4: 调用 check_payment(order="001")  → 结果: 未支付
...（模型在等一个永远不会发生的状态变化）
```

**场景三：任务无法完成但模型不放弃**
```
Step 1-10: 各种工具调用...
模型: "我再试试另一种方法..."
Step 11-20: 更多工具调用...
（模型不愿意说"我做不到"）
```

### 5.2 三道防线

**第一道：硬上限（Hard Limit）**

绝对不能突破的上限，无论发生什么都会停。

```python
# 步数上限
MAX_STEPS = 15

# 时间上限
MAX_WALL_TIME_SECONDS = 120

# 成本上限
MAX_COST_USD = 0.50
```

这些参数是 **产品决策**，不是调试参数。比如一个客服 Agent，业务方可能要求「最多 5 轮交互就要给出结果或转人工」——这是 SLA（服务级别协议）的一部分。

**第二道：软启发（Soft Heuristics）**

在触发硬上限之前，用启发式规则检测异常模式，提前中止。

```python
import time
from collections import Counter


class StopConditionChecker:
    """停止条件检查器：集中管理所有停止判断逻辑"""
    
    def __init__(
        self,
        max_steps: int = 15,
        max_wall_time: float = 120.0,
        max_cost_usd: float = 0.50,
        max_repeated_calls: int = 3,
        max_empty_responses: int = 2,
    ):
        self.max_steps = max_steps
        self.max_wall_time = max_wall_time
        self.max_cost_usd = max_cost_usd
        self.max_repeated_calls = max_repeated_calls
        self.max_empty_responses = max_empty_responses
        
        self.start_time = time.time()
        self.total_cost = 0.0
        self.call_history: list[str] = []  # "tool_name:args_hash"
        self.empty_count = 0
    
    def check(self, step: int, tool_calls=None, response_content=None) -> str | None:
        """
        检查所有停止条件。
        
        返回 None 表示继续；返回字符串表示应该停止（字符串是原因）。
        """
        # 硬上限：步数
        if step >= self.max_steps:
            return f"达到最大步数 {self.max_steps}"
        
        # 硬上限：时间
        elapsed = time.time() - self.start_time
        if elapsed > self.max_wall_time:
            return f"超时 ({elapsed:.1f}s > {self.max_wall_time}s)"
        
        # 硬上限：成本
        if self.total_cost > self.max_cost_usd:
            return f"成本超限 (${self.total_cost:.3f} > ${self.max_cost_usd})"
        
        # 软启发：重复工具调用
        if tool_calls:
            for tc in tool_calls:
                key = f"{tc.function.name}:{hashlib.md5(tc.function.arguments.encode()).hexdigest()}"
                self.call_history.append(key)
                
                # 检查最近 N 次是否完全相同
                if len(self.call_history) >= self.max_repeated_calls:
                    recent = self.call_history[-self.max_repeated_calls:]
                    if len(set(recent)) == 1:
                        return f"重复调用 {tc.function.name} {self.max_repeated_calls} 次"
        
        # 软启发：空回复
        if response_content is not None and not response_content.strip():
            self.empty_count += 1
            if self.empty_count >= self.max_empty_responses:
                return f"连续 {self.empty_count} 次空回复"
        else:
            self.empty_count = 0  # 重置计数
        
        return None  # 所有检查通过，继续
    
    def add_cost(self, prompt_tokens: int, completion_tokens: int, model: str = "gpt-4"):
        """累加本轮的 token 成本"""
        # 简化的价格表（实际应从配置读取）
        prices = {
            "gpt-4": {"prompt": 0.03 / 1000, "completion": 0.06 / 1000},
            "gpt-3.5-turbo": {"prompt": 0.0005 / 1000, "completion": 0.0015 / 1000},
        }
        price = prices.get(model, prices["gpt-4"])
        self.total_cost += prompt_tokens * price["prompt"] + completion_tokens * price["completion"]
```

**第三道：用户取消（User Cancellation）**

流式请求下，用户应该随时能中断 Agent。实现方式取决于框架：

### 5.3 停止后的善后

停止不是简单地 `return`。生产级 Agent 在停止时还需要：

1. **打日志**：记录停止原因、已完成的步骤、总 token 消耗（C8 可观测性）
2. **通知用户**：告诉用户为什么停了，已经做到哪一步了
3. **保存中间状态**：如果任务可以续做，把 messages 存起来

```python
def on_agent_stop(reason: str, messages: list, step: int, checker: StopConditionChecker):
    """Agent 停止时的善后处理"""
    log.warning(
        "Agent stopped",
        extra={
            "reason": reason,
            "steps_completed": step,
            "total_cost_usd": checker.total_cost,
            "elapsed_seconds": time.time() - checker.start_time,
            "tool_calls_count": len(checker.call_history),
        },
    )
    # 可选：保存 messages 到数据库，供后续恢复
```

### 读到这里你应该理解

- Agent 死循环有三种常见模式：同一工具重复调用、两工具互调、不愿放弃
- 三道防线：硬上限（绝对天花板）、软启发（异常模式检测）、用户取消
- 停止条件是产品 SLA 的一部分，不是调试参数
- 停止后需要善后：打日志、通知用户、保存状态

**本节小结（三张卡片）**

- **一句话定义**：停止条件是 **产品 SLA 的一部分**，不是调试参数。
- **典型场景**：模型陷入「查空表 →再查→再查」；应用重复检测打断。
- **和 A1 类比**：像 **`signal`/`contextlib.timeout` 的边界** —— 资源与时间都不可无限占用。

## 6. 规划与执行的适度拆分

**本节概念**：**Planner–Executor** 拆分是把「想」和「做」分成两个独立步骤。Planner 先产出一个步骤计划，Executor 逐步执行。代价是 **更多失败面** 与 **编排复杂度**。

### 6.1 什么时候需要 Plan-and-Execute？

大多数情况下，**单循环 ReAct 就够了**。模型在每一轮自己决定下一步做什么，隐式地在「规划」。

但有些场景需要显式规划：

- **多步骤有依赖**：步骤 3 的输入依赖步骤 1 和步骤 2 的输出，需要提前规划好顺序
- **需要人工审批**：某些步骤（比如删除数据、发送邮件）需要用户确认，先列出计划让用户审核
- **任务很复杂**：10+ 步的任务，单循环 ReAct 容易在中途迷失方向

### 6.2 Plan-and-Execute 的实现

```python
def plan_and_execute(user_goal: str, tools: list[dict], max_steps: int = 10) -> str:
    """
    Plan-and-Execute 模式：先规划，再逐步执行。
    
    与 ReAct 的区别：
    - ReAct: 每一步都是「想+做」，边走边看
    - Plan-and-Execute: 先「想完所有步骤」，再逐步「做」
    """
    
    # ── Phase 1: 规划 ──
    plan_prompt = f"""用户目标: {user_goal}

可用工具: {json.dumps([t['function']['name'] + ': ' + t['function']['description'] for t in tools], ensure_ascii=False)}

请为这个目标制定执行计划。以 JSON 数组格式输出，每个元素包含:
- step: 步骤编号
- action: 要执行的动作描述
- tool: 要使用的工具名（如果需要）
- depends_on: 依赖哪些步骤的结果（步骤编号列表）

只输出 JSON，不要其他内容。"""
    
    plan_response = call_llm_simple(plan_prompt)
    
    try:
        plan = json.loads(plan_response)
    except json.JSONDecodeError:
        return "规划失败：无法解析计划。"
    
    print(f"📋 生成计划: {len(plan)} 个步骤")
    for step in plan:
        print(f"  Step {step['step']}: {step['action']}")
    
    # ── Phase 2: 逐步执行 ──
    step_results = {}  # step_number → result
    
    for step_info in plan:
        step_num = step_info["step"]
        print(f"\n▶ 执行 Step {step_num}: {step_info['action']}")
        
        # 构建执行上下文：包含依赖步骤的结果
        context_parts = []
        for dep in step_info.get("depends_on", []):
            if dep in step_results:
                context_parts.append(f"Step {dep} 结果: {step_results[dep]}")
        
        context = "\n".join(context_parts) if context_parts else "（无前置依赖）"
        
        # 用单步 ReAct 执行这一步
        exec_prompt = f"""执行以下步骤:
{step_info['action']}

前置步骤结果:
{context}

请使用工具完成这一步。"""
        
        result = react_loop(exec_prompt, max_steps=3)  # 单步限制更紧
        step_results[step_num] = result
        print(f"  ✓ Step {step_num} 完成")
    
    # ── Phase 3: 汇总 ──
    summary_prompt = f"""用户目标: {user_goal}

执行结果:
{json.dumps(step_results, ensure_ascii=False, indent=2)}

请根据所有步骤的执行结果，给用户一个完整的回答。"""
    
    return call_llm_simple(summary_prompt)
```

### 6.3 Plan-and-Execute 的问题

先别急着用这个模式。它有几个实际问题：

1. **计划可能过时**：执行到第 3 步时发现第 1 步的结果和预期不同，整个计划可能需要重做
2. **更多 LLM 调用**：规划 1 次 + 每步执行 1+ 次 + 汇总 1 次，比纯 ReAct 多很多调用
3. **错误放大**：如果规划阶段就搞错了步骤顺序，后续所有执行都是浪费

**实践建议**：先用单循环 ReAct 跑通业务闭环。只在 **明确需要人工审批** 或 **任务步骤超过 7-8 步** 时才考虑 Plan-and-Execute。过早拆分是 **分布式微服务病的 Agent 版**——架构图好看，但调试地狱。

### 6.4 动态重规划

如果确实需要 Plan-and-Execute，建议加入 **重规划机制**：当某一步的执行结果与预期严重不符时，回到 Planner 重新生成后续计划。

```python
# 伪代码
for step_info in plan:
    result = execute_step(step_info)
    
    # 检查结果是否与预期相符
    if result_diverges_from_plan(result, step_info):
        # 重新规划剩余步骤
        remaining_plan = replan(user_goal, completed_steps, result)
        plan = completed_steps + remaining_plan
```

这让 Plan-and-Execute 兼具了 ReAct 的灵活性，但复杂度也更高。

**本节小结（三张卡片）**

- **一句话定义**：拆分是为了 **降低单提示的认知负荷**，不是为了架构图好看。
- **典型场景**：合规审核链：Planner 列出「查政策→比对合同→生成结论」，每步可回放。
- **和 A1 类比**：像 **把一个 500 行函数拆成私有方法** —— 拆完每块仍要单测。

## 7. 与 C3、C5 的集成点

**本节概念**：RAG 与 MCP 在循环里的位置不同：**RAG 常在 tool 前预处理**，或作为 **一个 `retrieve` 工具**；MCP 是 **工具来源之一**，循环调度仍在你手里。

### 7.1 RAG 在 Agent 中的两种用法

**用法一：作为预处理（循环外）**

在 Agent 循环开始前，先用 RAG 检索相关文档片段，拼入 system prompt。

```python
# 循环开始前
relevant_docs = rag_retrieve(user_query, top_k=3)
system_prompt = f"参考资料:\n{relevant_docs}\n\n请基于以上资料回答。"

# 然后正常进入 ReAct 循环
react_loop(user_goal, system_prompt=system_prompt)
```

优点：简单，不增加循环复杂度。  
缺点：如果检索结果不对，Agent 无法自主重新检索。

**用法二：作为工具（循环内）**

把 `retrieve` 注册为一个工具，让模型自己决定何时检索、用什么 query 检索。

```python
@register_tool(
    name="search_knowledge_base",
    description="在知识库中搜索相关文档。当你需要查找事实信息时使用。",
    parameters={
        "type": "object",
        "properties": {
            "query": {"type": "string", "description": "搜索关键词或问题"},
        },
        "required": ["query"],
    },
)
def search_knowledge_base(query: str) -> str:
    results = rag_retrieve(query, top_k=3)
    return json.dumps(results, ensure_ascii=False)
```

优点：模型可以根据第一次检索结果调整 query 重新检索（自适应）。  
缺点：增加循环轮次，成本更高。

### 7.2 MCP 工具在 Agent 中的使用

C5 学过 MCP（Model Context Protocol）是工具的标准化协议。在 Agent 中，MCP Server 提供的工具和本地工具没有区别——都注册到工具注册表中，都通过 `tool_calls` 调用。

区别在于 **执行层**：本地工具直接调函数，MCP 工具需要通过 MCP Client 发 JSON-RPC 请求到 MCP Server。

```python
# MCP 工具的执行器（和本地工具接口一致）
async def execute_mcp_tool(tool_name: str, args: dict) -> str:
    """通过 MCP Client 调用远程工具"""
    result = await mcp_client.call_tool(tool_name, args)
    return json.dumps(result, ensure_ascii=False)
```

**关键认知**：MCP 是 **工具的来源**，Agent 循环是 **工具的调度者**。MCP 负责「这个工具怎么被调用」，Agent 负责「什么时候调用哪个工具」。

## 9. 常见误区与踩坑

### 误区 1：「模型会自己停下来的」

**错误想法**：模型足够聪明，应该知道什么时候该停。

**现实**：模型的停止行为取决于 system prompt 和温度设置，**不确定性很大**。同一个 prompt，温度 0.0 可能 3 步就停，温度 0.7 可能 15 步还在探索。永远不要依赖模型「自觉」停止——**必须写硬上限**。

### 误区 2：把完整 prompt 放进 tool 结果里

**错误做法**：

```python
# ❌ 把整个 API 响应（含 prompt 和配置信息）原样返回
tool_result = json.dumps(full_api_response)  # 可能几千 token
```

**正确做法**：

```python
# ✅ 只提取模型需要的字段
tool_result = json.dumps({
    "status": response["status"],
    "order_id": response["data"]["order_id"],
    "total": response["data"]["total"],
})
```

工具返回的 token 数直接影响成本和上下文容量。**只返回模型推理需要的字段**。

### 误区 3：裁剪 messages 时拆散 tool 配对

如第 3 节所说，assistant（带 tool_calls）和对应的 tool 消息 **必须成对**。如果你写了一个简单的 `messages = messages[-10:]`，很可能把一对拆开，导致 API 报错。

### 误区 4：所有任务都上 Plan-and-Execute

简单任务（查个天气、查个订单）用 Plan-and-Execute 就是大炮打蚊子。多一轮 LLM 调用 = 多花钱 + 多延迟。**默认用 ReAct，明确需要时才用 Plan-and-Execute**。

### 误区 5：长期记忆不设过期

用户 2024 年的偏好（「我住在北京」）到 2026 年可能已经不准了。长期记忆需要 **时间戳**，过时的记忆要有机制降权或清除。

### 误区 6：忽视工具执行的错误处理

```python
# ❌ 工具出错直接崩溃
result = tool_function(**args)  # 如果抛异常，整个 Agent 就死了

# ✅ 捕获异常，把错误信息返回给模型
try:
    result = tool_function(**args)
except Exception as e:
    result = {"error": str(e)}  # 模型看到错误可能会换个策略
```

工具执行失败是正常情况（网络超时、权限不足、参数错误）。**错误信息应该写回 messages**，让模型有机会调整策略，而不是让整个 Agent 崩溃。

## 11. 面试高频问题

### Q1：什么是 Agent？和普通的 LLM 调用有什么区别？

**答**：Agent 是一个 **多轮自主循环**，以 LLM 为推理引擎，在循环中交替进行推理和行动（调用工具），直到任务完成或触发停止条件。与普通 LLM 调用的区别：

1. **多轮** vs 单次往返
2. **有状态**（messages 累积）vs 无状态
3. **需要显式停止条件** vs API 返回即结束
4. **可以调用多个工具** vs 最多一轮工具调用

### Q2：ReAct 模式是什么？为什么不能只推理不行动？

**答**：ReAct（Reason + Act）让模型交替执行推理和行动。只推理不行动（纯 CoT）的问题是无法访问真实数据，推理基于可能过时的参数知识。只行动不推理会导致盲目调用工具，顺序混乱。ReAct 的核心价值在于每次行动后有 **可验证的观察**，让推理基于真实数据。

### Q3：Agent 循环中如何防止死循环？

**答**：三道防线：

1. **硬上限**：max_steps、max_wall_time、max_cost_usd，绝对不可突破
2. **软启发**：重复调用检测（同 tool+args 连续出现 N 次）、空回复检测、循环依赖检测
3. **用户取消**：流式请求下支持 abort，异步场景用 cancel event

### Q4：消息历史太长怎么处理？

**答**：两种策略：

1. **滑动窗口**：保留 system + 最近 K 轮，简单但丢信息
2. **摘要压缩**：用小模型把早期历史压缩成 bullet memory 插回 system，保留关键事实但有成本

注意 tool 消息和 assistant 消息必须成对操作，不能拆散。

### Q5：什么时候用 Plan-and-Execute，什么时候用 ReAct？

**答**：**默认用 ReAct**。只在以下场景考虑 Plan-and-Execute：

- 任务步骤多（7-8 步以上），单循环容易迷失方向
- 需要人工审批中间步骤
- 步骤间有复杂依赖关系

Plan-and-Execute 的代价是更多 LLM 调用、计划可能过时、需要重规划机制。过早引入是「分布式微服务病」的 Agent 版。
