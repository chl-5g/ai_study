## 1. Agent 与「一次 ChatCompletion」

| | 单次补全 | Agent |
|--|----------|--------|
| 轮次 | 通常 1 次往返 | **多轮**内部循环 |
| 工具 | 可选 0～1 轮 | **可能多轮** tool |
| 控制 | 用户发一句 | **程序驱动** + 用户目标 |

---

## 2. ReAct 循环（思想与实现）

典型伪代码：

```text
messages = [system, user_goal]
for step in range(MAX_STEPS):
    resp = llm.chat(messages, tools=TOOLS)
    if resp 无 tool_calls:
        return resp.content
    执行每个 tool → 追加 assistant + tool 消息到 messages
```

**观察**必须是 **机器生成的客观字符串**（查询结果、错误栈摘要），减少模型编造空间。

---

## 3. 状态：消息历史与摘要

- **messages** 线性增长 → **上下文爆炸**。  
- **裁剪**：保留 system + 最近 k 轮 + 关键 tool 结果。  
- **摘要**：用次小模型或规则把早期对话压成 **bullet memory**（有损）。

---

## 4. 记忆：短期、摘要、长期

- **短期**：当前会话 buffer。  
- **中期**：会话级摘要。  
- **长期**：向量库/DB 存 **用户事实**，按查询 **RAG 拉回**（见 C3）。  
注意 **遗忘权与合规**（C2/C8）。

---

## 5. 停止条件与防死循环

- **硬上限**：`MAX_STEPS`、`max_wall_time`、`max_cost_usd`。  
- **软启发**：同一 `(tool_name, args_json)` **连续出现** 则中止或换策略。  
- **用户取消**：流式请求要能 **abort**。

---

## 6. 规划与执行的适度拆分

- **简单任务**：单循环足够。  
- **复杂任务**：Planner 产出步骤列表，Executor 逐步执行——但增加 **失败面**，需 **每步可观测**（C8）。

---

## 7. 与 C3、C5 的集成点
