## 0. 与 A1、C4 的衔接

**本节概念**：**单 Agent** 在实现层通常就是一个 **`while` 循环** 套 **LLM 调用**；循环体里分支处理 **无工具 / 有工具 / 用户中断**；这与 A1 里「**显式状态机优于深层递归**」一致。

C4 已经教会 **`tool_calls` 与 `tool` 消息** 的拼法；C6 回答的是：**循环何时退出**、**历史如何裁剪**、**记忆放哪一层**。如果你只会调一次 `chat.completions`，你写的是 **函数**；加上受控循环，你写的是 **有界停机程序** —— 停机条件必须写进代码，不能交给模型「自觉」。

**本节小结（三张卡片）**

- **一句话定义**：单 Agent = **带工具的多轮状态机**。
- **典型场景**：「查三次库还不满意就转人工」—— 这是产品策略，不是 prompt 里一句话能保障的。
- **和 A1 类比**：循环像 **`for` + `try/finally`** —— 无论如何要关连接、打 trace、归还预算。

## 1. Agent 与「一次 ChatCompletion」

**本节概念**：**单次补全**假设「一问一答」；**Agent** 假设「一目标多步」，中间可多次调用外部世界。

| | 单次补全 | Agent |
|--|----------|--------|
| 轮次 | 通常 1 次往返 | **多轮**内部循环 |
| 工具 | 可选 0～1 轮 | **可能多轮** tool |
| 控制 | 用户发一句 | **程序驱动** + 用户目标 |

**本节小结（三张卡片）**

- **一句话定义**：Agent 把 **对话 API** 当成 **规划与执行的引擎**，而不是一次性文本生成器。
- **典型场景**：用户只说「帮我订明天最早一班高铁」，背后需要多工具、多校验。
- **和 A1 类比**：单次调用像 **纯函数**；Agent 像 **带 I/O 副作用的交互程序**。

## 2. ReAct 循环（思想与实现）

**本节概念**：**ReAct**（Reason + Act）在工程上常落地为：**模型推理 → 若有 tool_calls 则执行 → 把观察写回 messages → 再推理**，直到无工具或触发停止。

典型伪代码：

```text
messages = [system, user_goal]
for step in range(MAX_STEPS):
    resp = llm.chat(messages, tools=TOOLS)
    if resp 无 tool_calls:
        return resp.content
    执行每个 tool → 追加 assistant + tool 消息到 messages
```

**观察**必须是 **机器生成的客观字符串**（查询结果、错误栈摘要），减少模型编造空间。若把「我以为数据库返回了…」这种含糊自然语言当 observation，闭环质量会崩。

**本节小结（三张卡片）**

- **一句话定义**：ReAct 闭环的灵魂是 **可验证的观察**，不是华丽的 CoT 话术。
- **典型场景**：SQL 执行失败，应把 **错误码 + 截断 SQL** 写回，而不是「查询失败请重试」糊弄模型。
- **和 A1 类比**：观察像 **单元测试的 assert 信息** —— 越具体，下一轮修复越快。

## 3. 状态：消息历史与摘要

**本节概念**：**messages** 是 Agent 的 **显式状态**；不裁剪会 **上下文爆炸**；乱裁剪会 **丢失关键 tool 结果**。

- **messages** 线性增长 → **上下文爆炸**。  
- **裁剪**：保留 system + 最近 k 轮 + 关键 tool 结果 —— 「关键」要靠 **启发式或结构化标记**（例如用户点过「记住这个订单号」）。  
- **摘要**：用次小模型或规则把早期对话压成 **bullet memory**（有损），摘要本身也要 **版本化**（C8 里可观测）。

**本节小结（三张卡片）**

- **一句话定义**：状态管理 = **在预算内保留最大信息量**。
- **典型场景**：长会话里早期用户给了约束「不要日文输出」，摘要必须保留该约束。
- **和 A1 类比**：像 **`deque(maxlen=...)`** —— 超过 maxlen 要定义丢哪头、是否先归档。

## 4. 记忆：短期、摘要、长期

**本节概念**：**短期**是会话 buffer；**中期**是会话摘要；**长期**是外存（向量库/关系库），通过 **RAG 或工具查询** 拉回。

- **短期**：当前会话 buffer。  
- **中期**：会话级摘要。  
- **长期**：向量库/DB 存 **用户事实**，按查询 **RAG 拉回**（见 C3）。  
注意 **遗忘权与合规**（C2/C8）：用户要求删除时，三层记忆要 **一致清除**。

**本节小结（三张卡片）**

- **一句话定义**：记忆分层解决 **「热数据在上下文，冷数据在外存」**。
- **典型场景**：用户偏好「素食」存长期，当餐点菜上下文放短期。
- **和 A1 类比**：像 **L1/L2/L3 缓存** —— 命中策略与一致性都要设计，不是默认 magically 正确。

## 5. 停止条件与防死循环

**本节概念**：**不写停止条件的 Agent 循环**在图灵完备意义上可能不停；工程上必须 **硬上限 + 软启发 + 用户取消**。

- **硬上限**：`MAX_STEPS`、`max_wall_time`、`max_cost_usd`。  
- **软启发**：同一 `(tool_name, args_json)` **连续出现** 则中止或换策略。  
- **用户取消**：流式请求要能 **abort**（客户端断开或 cooperative cancel）。

**本节小结（三张卡片）**

- **一句话定义**：停止条件是 **产品 SLA 的一部分**，不是调试参数。
- **典型场景**：模型陷入「查空表 →再查→再查」；应用重复检测打断。
- **和 A1 类比**：像 **`signal`/`contextlib.timeout` 的边界** —— 资源与时间都不可无限占用。

## 6. 规划与执行的适度拆分

**本节概念**：**Planner–Executor** 拆分换来 **可读计划** 与 **独立测试**，代价是 **更多失败面** 与 **编排复杂度**。

- **简单任务**：单循环足够。  
- **复杂任务**：Planner 产出步骤列表，Executor 逐步执行——但增加 **失败面**，需 **每步可观测**（C8）。

**实践建议**：先单循环跑通业务闭环，再只在 **明确需要并行/审批** 的节点引入 Planner；过早拆分是 **分布式微服务病** 的 Agent 版。

**本节小结（三张卡片）**

- **一句话定义**：拆分是为了 **降低单提示的认知负荷**，不是为了架构图好看。
- **典型场景**：合规审核链：Planner 列出「查政策→比对合同→生成结论」，每步可回放。
- **和 A1 类比**：像 **把一个 500 行函数拆成私有方法** —— 拆完每块仍要单测。

## 7. 与 C3、C5 的集成点

**本节概念**：RAG 与 MCP 在循环里的位置不同：**RAG 常在 tool 前预处理**，或作为 **一个 `retrieve` 工具**；MCP 是 **工具来源之一**，循环调度仍在你手里。

- **C3**：检索可作为 **一步工具** 或进入循环前的 **预处理**。  
- **C5**：工具实现来自 MCP Server；循环逻辑仍在 **你的宿主进程**。
