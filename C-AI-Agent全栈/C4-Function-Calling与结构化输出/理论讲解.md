## 1. Why：模型不仅要说话，还要触发动作

业务需要：**查订单、调内部 API、写数据库、触发工作流**。自然语言无法直接被程序安全执行。约定 **函数名 + JSON 参数**（或等价结构）使 **宿主代码** 可控执行。

---

## 2. OpenAI Chat Completions 里的 tools 长什么样

每条 tool 通常为：

```json
{
  "type": "function",
  "function": {
    "name": "get_invoice",
    "description": "按订单号查询电子发票链接（只读）",
    "parameters": {
      "type": "object",
      "properties": {
        "order_id": { "type": "string", "description": "订单号" }
      },
      "required": ["order_id"]
    }
  }
}
```

请求体里 `tools: [...]`，`tool_choice: "auto"` 或强制某函数。

---

## 3. 一轮完整工具调用时序

1. Client → API：`messages` + `tools`。  
2. API → Client：`assistant` 消息带 **`tool_calls`**（含 `id`、`function.name`、`function.arguments` 字符串）。  
3. Client：**校验 name 在白名单**，`json.loads(arguments)`，**执行业务函数**。  
4. Client → API：追加 `assistant` 原消息，再追加若干 **`role: tool`**，`tool_call_id` 对齐，`content` 为字符串结果。  
5. API → Client：模型基于观察生成 **最终自然语言**（或再次 tool_calls）。

**执行权永远在服务端**；模型不能直连生产网。

---

## 4. 工具 Schema 怎么写才好用

- **`description`**：写清 **只读/写入**、**副作用**、**失败时返回什么**。  
- **`required`**：宁多勿少；可选参数给 `default` 或在描述里说明默认行为。  
- **`enum`**：限制取值，减少胡编。  
- **体量**：工具过多过长会挤占上下文，需 **动态筛选** 或 **分层工具**。

---

## 5. 服务端执行与安全

- **白名单**：仅允许注册过的 `name`。  
- **参数校验**：用 JSON Schema 或 Pydantic **二次校验**（防注入、类型漂移）。  
- **权限**：结合登录态 / API Key / 租户 ID 决定能否调用某工具。  
- **限流与审计**：每次调用记 **谁、何时、参数摘要、耗时、错误**。

---

## 6. 结构化输出：JSON Mode 与 Pydantic

| 手段 | 适用 |
|------|------|
| **response_format: json_object** | 强制 JSON 语法（字段仍需你在 prompt 里约定） |
| **Pydantic `model_validate_json`** | 校验 + 类型 + 默认值；失败可 **重试或降级** |
| **Function Calling** | 「动作」而不仅是「数据形状」 |

生产上：**parse 失败 → 收紧 prompt 或降温度 → 有限次重试**。

---

## 7. 常见坑与排错

- **模型不调工具**：system 指令不够硬、或模型/服务端 **不支持 tools**。  
- **`arguments` 非合法 JSON**：需捕获异常并反馈给模型或走降级。  
- **死循环**：同参重复调用 → **max_iterations** + **去重键**（name+args hash）。  
- **把密钥写进 tool 示例**：schema 与 prompt 中一律用占位符。

---

## 8. 与 C5、C6 的关系

- **C5**：把工具 **以标准协议（如 MCP）** 暴露，宿主仍负责鉴权与执行映射。  
- **C6**：ReAct 循环 = 多轮「可能含 tool_calls 的 assistant」与 **tool** 消息交替，直到终止条件。
