## 0. 与 A1、C1 的衔接

**本节概念**：**Function Calling** 不是「远程执行 Python」，而是 **模型输出结构化意图**（函数名 + 参数 JSON），由 **宿主进程** 像调用普通函数一样执行；这与 A1 里强调的 **「显式优于隐式」** 完全一致 —— 执行路径必须在代码里可见。

你在 C1 已经熟悉 **messages 列表** 与 **流式响应**；C4 在此基础上增加 **`tool_calls` 与 `role: tool` 消息**，列表变长、顺序变严格。用 A1 的思维：**把一轮 tool 往返当成一个小状态机**，比把 API 当「魔法黑盒」好调试得多。

**举个例子感受差异**：

在 C1 里，messages 列表一般长这样：

```python
messages = [
    {"role": "system", "content": "你是一个客服助手"},
    {"role": "user", "content": "我的订单到哪了？"},
    {"role": "assistant", "content": "请提供您的订单号..."},
]
```

在 C4 之后，消息列表会变成这样（注意多了两种新角色）：

```python
messages = [
    {"role": "system", "content": "你是一个客服助手"},
    {"role": "user", "content": "帮我查一下订单 ORD-12345"},
    # ↓ assistant 不再直接回答，而是"提议"调用工具
    {"role": "assistant", "content": None, "tool_calls": [
        {"id": "call_abc123", "type": "function",
         "function": {"name": "get_order_status", "arguments": '{"order_id": "ORD-12345"}'}}
    ]},
    # ↓ 宿主执行完毕，把结果以 role: tool 写回
    {"role": "tool", "tool_call_id": "call_abc123",
     "content": '{"status": "已发货", "tracking": "SF1234567890"}'},
    # ↓ 模型基于工具返回值，生成最终回复
    {"role": "assistant", "content": "您的订单 ORD-12345 已发货，快递单号 SF1234567890。"},
]
```

**关键认知**：从 C1 到 C4，messages 列表从「人机对话」升级为「人-机-工具三方协作协议」。assistant 不再只产出自然语言，它可以产出 **结构化的工具调用请求**，而宿主负责执行并把结果写回。

**本节小结（三张卡片）**

- **一句话定义**：Function Calling = **模型提议，宿主裁决与执行**。
- **典型场景**：订单查询只读 API；模型再聪明也不能直连生产库 connection string。
- **和 A1 类比**：`tool_calls` 像 **`getattr(registry, name)(**kwargs)`** —— 名字必须落在白名单映射上。

## 1. Why：模型不仅要说话，还要触发动作

**本节概念**：自然语言 **不可被机器安全执行**；业务需要 **查订单、调内部 API、写数据库、触发工作流**，必须落到 **确定性的函数名与参数**。

### 1.1 没有 Function Calling 之前的世界

在 Function Calling 出现之前（2023年6月以前），开发者要让 LLM 调用外部工具，只能用 **提示词 hack**：

```python
# 古早方案：在 prompt 里告诉模型"用特定格式输出"
system_prompt = """
当用户询问天气时，请用以下格式回答：
ACTION: get_weather
PARAMS: {"city": "城市名"}

不要直接回答天气，等我把结果给你。
"""

# 然后在代码里用正则解析模型输出
import re
match = re.search(r"ACTION: (\w+)\nPARAMS: (.+)", response)
if match:
    action = match.group(1)
    params = json.loads(match.group(2))  # 经常崩，因为模型输出的不是合法 JSON
```

**这个方案有三个致命问题**：

1. **格式不稳定**：模型可能输出 `ACTION：get_weather`（中文冒号）、`Action: get_weather`（大小写变化）、或者干脆不按格式来。你得写一堆容错正则，越写越脏。

2. **安全性为零**：模型可以输出 `ACTION: drop_database`，你的正则不会拦。必须在解析之后再加白名单检查，但很多人忘了。

3. **参数校验全靠运气**：模型输出的 JSON 可能少字段、多字段、类型错（字符串 `"123"` 而不是数字 `123`）。每种边界都得手写处理逻辑。

### 1.2 Function Calling 解决了什么

Function Calling 本质上是把上面的 hack **内化到协议层**：

| 以前（Prompt Hack） | 现在（Function Calling） |
|-----|------|
| 模型输出自由文本，你解析 | 模型输出结构化 JSON，API 保证格式 |
| 工具列表藏在 system prompt 里 | 工具列表是请求体的一级字段 `tools` |
| 没有调用 ID 概念 | 每次调用有唯一 `tool_call_id` |
| 正则解析，经常崩 | `json.loads(arguments)` 基本可靠 |
| 安全完全靠应用层 | 安全仍靠应用层，但契约更清晰 |

**但要注意**：Function Calling **不是远程执行**。模型不能直接调函数——它只是 **输出了一段 JSON 说"我想调这个函数，参数是这些"**。真正的执行权在你的代码里。这个区分非常重要，很多初学者搞混了。

### 1.3 一个类比帮你彻底理解

把模型想象成一个 **只能打电话的指挥官**，它坐在指挥部里，面前只有一部电话和一本通讯录。

- **通讯录** = `tools` 列表（只能打里面有的号码）
- **打电话** = 产出 `tool_calls`（"我要打给 get_weather，告诉他城市是北京"）
- **接线员** = 你的宿主代码（检查号码是否在册，拨通，把对方说的话转述回来）
- **对方回复** = `role: tool` 消息（"北京今天 25°C 晴"）

指挥官 **永远不能自己跑出去**。他只能通过电话（结构化请求）和外界交互。这就是 Function Calling 的设计哲学：**能力受限但协议明确**。

**本节小结（三张卡片）**

- **一句话定义**：工具调用的本质是 **受限指令集** 上的规划，不是通用脚本引擎。
- **典型场景**：客服机器人只能 `get_order_status`，不能 `drop_table`。
- **和 A1 类比**：像只暴露 **少量公开方法** 的类，把危险操作藏在 **私有方法** 里，外部永远调不到。

## 2. OpenAI Chat Completions 里的 tools 长什么样

**本节概念**：每条 tool 在协议层是 **JSON Schema 子集** + **自然语言描述**；描述会进上下文，影响模型是否选对工具。

### 2.1 单个工具的完整结构

```json
{
  "type": "function",
  "function": {
    "name": "get_invoice",
    "description": "按订单号查询电子发票链接（只读，不会修改任何数据）。若订单未开票则返回 {\"error\": \"未开票\"}",
    "parameters": {
      "type": "object",
      "properties": {
        "order_id": {
          "type": "string",
          "description": "订单号，格式为 ORD-数字，如 ORD-12345"
        }
      },
      "required": ["order_id"]
    }
  }
}
```

逐字段解读：

| 字段 | 作用 | 给谁看 |
|------|------|--------|
| `type: "function"` | 协议保留，未来可能有 `"code_interpreter"` 等 | API 解析 |
| `function.name` | 函数标识，宿主用它查 dispatch 表 | 模型 + 宿主 |
| `function.description` | 自然语言说明，**直接影响模型选择准确率** | 模型 |
| `function.parameters` | JSON Schema，约束参数形状 | 模型 + 宿主 |
| `properties[].description` | 每个参数的说明 | 模型 |
| `required` | 必填字段列表 | 模型 + 宿主 |

### 2.2 请求体完整示例

```python
import openai

client = openai.OpenAI(
    base_url="http://localhost:11434/v1",  # Ollama
    api_key="ollama",
)

tools = [
    {
        "type": "function",
        "function": {
            "name": "get_weather",
            "description": "查询指定城市的当前天气。返回温度(°C)和天气状况。",
            "parameters": {
                "type": "object",
                "properties": {
                    "city": {
                        "type": "string",
                        "description": "城市名称，如'北京'、'上海'"
                    },
                    "unit": {
                        "type": "string",
                        "enum": ["celsius", "fahrenheit"],
                        "description": "温度单位，默认 celsius"
                    }
                },
                "required": ["city"]
            }
        }
    }
]

response = client.chat.completions.create(
    model="qwen3:8b",
    messages=[{"role": "user", "content": "北京今天天气怎么样？"}],
    tools=tools,
    tool_choice="auto",  # 模型自己决定是否调工具
)
```

### 2.3 tool_choice 的三种姿态

| `tool_choice` 值 | 含义 | 适用场景 |
|---|---|---|
| `"auto"` | 模型自行决定是否调工具 | 通用对话，兼顾回答和工具调用 |
| `"required"` | 必须调用工具（但选哪个由模型决定） | 确定需要执行动作的场景 |
| `{"type":"function","function":{"name":"get_weather"}}` | 强制调用指定工具 | 已知意图，只需模型填参数 |
| `"none"` | 禁止调工具，纯文本回答 | 总结阶段，不需要再查数据 |

**实践提示**：`description` 与 `properties[].description` 是 **给模型读的文档**；写得太短，模型会乱选；写得像内部黑话，模型会胡填。保持 **用户视角 + 边界条件**（只读/写入、失败时返回什么）。

### 2.4 description 写得好 vs 写得烂

```python
# ❌ 烂 description：模型不知道什么时候该调它
{
    "name": "query",
    "description": "查询数据"  # 查什么数据？哪个库？只读还是写入？
}

# ✅ 好 description：模型能精准判断
{
    "name": "query_order_status",
    "description": "根据订单号查询物流状态（只读）。"
                   "返回 JSON 包含 status（枚举：待发货/已发货/已签收/已退款）和 tracking_number。"
                   "若订单号不存在，返回 {\"error\": \"订单不存在\"}。"
}
```

**经验法则**：好的 description 回答三个问题——(1) 什么时候该用它？(2) 它返回什么？(3) 它可能失败吗？

**本节小结（三张卡片）**

- **一句话定义**：tools 列表是 **宿主提前注册的「可调 API 目录」**。
- **典型场景**：同一 `name` 在开发/生产指向不同 base URL，由宿主映射，不进 schema。
- **和 A1 类比**：parameters schema 像 **TypedDict / dataclass 的字段约束** —— 先约束形状，再谈业务逻辑。

## 3. 一轮完整工具调用时序

**本节概念**：一轮往返里，**assistant 消息可能带 `tool_calls`**；宿主执行后必须用 **`role: tool`** 把 **observation** 写回，且 **`tool_call_id` 对齐**，模型才能把「行动结果」纳入后续推理。

### 3.1 时序图（文字版）

```
用户（User）          宿主代码（Host）           LLM API
   │                     │                        │
   │  "北京天气怎么样"    │                        │
   │────────────────────>│                        │
   │                     │  messages + tools       │
   │                     │───────────────────────>│
   │                     │                        │
   │                     │  assistant: tool_calls  │
   │                     │  [{id:"call_1",         │
   │                     │    name:"get_weather",  │
   │                     │    args:'{"city":"北京"}'}]
   │                     │<───────────────────────│
   │                     │                        │
   │                     │  ① 校验 name 在白名单   │
   │                     │  ② json.loads(args)     │
   │                     │  ③ 执行 get_weather()   │
   │                     │  ④ 拿到结果             │
   │                     │                        │
   │                     │  messages 追加:          │
   │                     │  - assistant 原消息      │
   │                     │  - {role:tool,           │
   │                     │     tool_call_id:"call_1",
   │                     │     content: 结果JSON}   │
   │                     │───────────────────────>│
   │                     │                        │
   │                     │  assistant: "北京25°C晴" │
   │                     │<───────────────────────│
   │  "北京25°C晴"       │                        │
   │<────────────────────│                        │
```

### 3.2 用代码实现完整一轮

```python
import json
import openai

client = openai.OpenAI(base_url="http://localhost:11434/v1", api_key="ollama")

# ─── 第 0 步：注册工具和 dispatch 表 ───
def get_weather(city: str, unit: str = "celsius") -> dict:
    """模拟天气查询。生产环境中这里调真实 API。"""
    fake_db = {"北京": {"temp": 25, "condition": "晴"},
               "上海": {"temp": 28, "condition": "多云"}}
    if city not in fake_db:
        return {"error": f"未找到城市: {city}"}
    return fake_db[city]

# dispatch 表：名字 → 函数。这就是"白名单"
TOOL_REGISTRY = {
    "get_weather": get_weather,
}

tools_schema = [
    {
        "type": "function",
        "function": {
            "name": "get_weather",
            "description": "查询指定城市的当前天气（只读）。返回 temp(int) 和 condition(str)。",
            "parameters": {
                "type": "object",
                "properties": {
                    "city": {"type": "string", "description": "城市名称"},
                    "unit": {"type": "string", "enum": ["celsius", "fahrenheit"]}
                },
                "required": ["city"]
            }
        }
    }
]

# ─── 第 1 步：发送用户消息 ───
messages = [
    {"role": "system", "content": "你是一个天气助手，用中文回答。"},
    {"role": "user", "content": "北京和上海今天天气怎么样？"},
]

response = client.chat.completions.create(
    model="qwen3:8b",
    messages=messages,
    tools=tools_schema,
    tool_choice="auto",
)

assistant_msg = response.choices[0].message

# ─── 第 2 步：检查是否有 tool_calls ───
if assistant_msg.tool_calls:
    # 必须把 assistant 原消息原样追加到 messages
    messages.append(assistant_msg.model_dump())
    
    for tool_call in assistant_msg.tool_calls:
        func_name = tool_call.function.name
        call_id = tool_call.id
        
        # ─── 第 3 步：白名单校验 ───
        if func_name not in TOOL_REGISTRY:
            result = json.dumps({"error": f"未知工具: {func_name}"})
        else:
            # ─── 第 4 步：解析参数并执行 ───
            try:
                args = json.loads(tool_call.function.arguments)
                result = json.dumps(TOOL_REGISTRY[func_name](**args), ensure_ascii=False)
            except (json.JSONDecodeError, TypeError) as e:
                result = json.dumps({"error": f"参数错误: {e}"})
        
        # ─── 第 5 步：写回 tool 消息，tool_call_id 必须对齐 ───
        messages.append({
            "role": "tool",
            "tool_call_id": call_id,  # 🔑 这个 ID 必须和上面的对应
            "content": result,
        })
    
    # ─── 第 6 步：再次请求，模型基于工具结果生成最终回答 ───
    final_response = client.chat.completions.create(
        model="qwen3:8b",
        messages=messages,
    )
    print(final_response.choices[0].message.content)
else:
    # 模型直接回答了，没有调用工具
    print(assistant_msg.content)
```

### 3.3 为什么 tool_call_id 必须原样带回？

这个问题面试高频。答案有两层：

1. **消歧义**：一轮中模型可能并行调用多个工具（如同时查北京和上海天气），API 用 `tool_call_id` 把每个调用和它的结果一一匹配。如果 ID 对不上，模型不知道哪个结果对应哪个请求。

2. **状态一致性**：messages 列表是 **有序状态机**。`tool_call_id` 就像数据库外键，保证 `assistant.tool_calls[i]` 和 `tool[j]` 之间的引用完整性。丢了 ID 就像 SQL 里外键为 NULL——关系断裂。

### 3.4 messages 列表的严格顺序

```
user → assistant(tool_calls) → tool → tool → ... → assistant(最终回复)
                                 ↑ 多个 tool 消息对应多个 tool_call
```

**不能做的事**：
- ❌ 把 `tool` 消息放在 `assistant(tool_calls)` 前面
- ❌ 漏掉 `assistant(tool_calls)` 原消息，直接追加 `tool`
- ❌ `tool_call_id` 和 `assistant.tool_calls` 里的 `id` 不匹配

以上任何一个都会导致 API 报错（400 Bad Request）或模型行为异常。

**本节小结（三张卡片）**

- **一句话定义**：时序本质是 **「提议 → 执行 → 观察写回」** 的闭环。
- **典型场景**：多工具并行调用时，多个 `tool_call_id` 与多条 `tool` 消息一一对应。
- **和 A1 类比**：像 **递归前先压栈** —— 不把 `assistant` 原消息带着，`tool` 就成了无根之木。

## 4. 工具 Schema 怎么写才好用

**本节概念**：好 schema **减少歧义**、**减少非法参数**、**减少上下文体积**；坏 schema 让模型在几十个工具里 **瞎蒙**。

### 4.1 Schema 设计的五条原则

**原则一：名字要动宾结构**

```python
# ❌ 模糊
"name": "order"         # 查订单？改订单？删订单？
"name": "data"           # 什么数据？

# ✅ 清晰
"name": "get_order_status"
"name": "cancel_order"
"name": "search_products"
```

**原则二：description 写边界条件**

```python
# ❌ 只写了正常路径
"description": "查询用户信息"

# ✅ 写清楚异常路径和副作用
"description": "根据用户 ID 查询用户基本信息（只读）。"
               "返回 name/email/phone。"
               "若用户不存在返回 {\"error\": \"user_not_found\"}。"
               "手机号会脱敏为 138****1234 格式。"
```

**原则三：用 enum 收窄取值**

```python
# ❌ 模型可能填任意值
"status": {"type": "string", "description": "订单状态"}

# ✅ 限定范围
"status": {
    "type": "string",
    "enum": ["pending", "shipped", "delivered", "refunded"],
    "description": "订单状态筛选条件"
}
```

**原则四：required 宁多勿少**

可选参数越多，模型越容易忘填或乱填。如果某个参数 95% 的场景都要传，直接放 `required`。

**原则五：控制工具总量**

```python
# 工具列表会占用上下文 token。经验数据：
# - 每个工具定义 ≈ 100-300 tokens
# - 30 个工具 ≈ 3000-9000 tokens
# - 模型在 10+ 工具时选择准确率开始下降

# 解决方案：动态挂载（按会话阶段只加载相关工具子集）
def get_tools_for_stage(stage: str) -> list[dict]:
    """根据对话阶段返回相关工具子集。"""
    tool_groups = {
        "greeting": ["get_user_info"],
        "order_inquiry": ["get_order_status", "get_invoice", "get_tracking"],
        "complaint": ["create_ticket", "escalate_to_human"],
    }
    return [t for t in ALL_TOOLS if t["function"]["name"] in tool_groups.get(stage, [])]
```

### 4.2 复杂参数的 Schema 写法

当参数是嵌套对象或数组时：

```json
{
  "name": "create_order",
  "description": "创建新订单（写入操作，会扣库存）。",
  "parameters": {
    "type": "object",
    "properties": {
      "customer_id": {"type": "string"},
      "items": {
        "type": "array",
        "items": {
          "type": "object",
          "properties": {
            "product_id": {"type": "string"},
            "quantity": {"type": "integer", "minimum": 1, "maximum": 99}
          },
          "required": ["product_id", "quantity"]
        },
        "minItems": 1,
        "description": "订单商品列表，至少一个商品"
      },
      "delivery_date": {
        "type": "string",
        "pattern": "^\\d{4}-\\d{2}-\\d{2}$",
        "description": "期望送达日期，格式 YYYY-MM-DD"
      }
    },
    "required": ["customer_id", "items"]
  }
}
```

**注意**：JSON Schema 的 `minimum`、`maximum`、`pattern` 等约束，模型 **不一定严格遵守**（取决于模型能力）。所以宿主侧的 Pydantic 校验是必须的，不能只靠 schema。

**本节小结（三张卡片）**

- **一句话定义**：Schema 是 **人机共读契约**，偏向模型可读性设计。
- **典型场景**：日期字段要求 `YYYY-MM-DD`，用 pattern 或描述写死，避免 `2025/4/15` 与 `15-04-2025` 混用。
- **和 A1 类比**：`enum` 像 **Literal 类型** —— 把「魔法字符串」收窄成安全集合。

## 5. 服务端执行与安全

**本节概念**：**任何** `function.name` 最终都会映射到宿主代码；宿主必须假设 **模型是恶意的合谋者** —— 会尝试越权参数、SQL 片段、路径穿越。

### 5.1 安全四层模型

```
┌─────────────────────────────────────┐
│  第 1 层：白名单（name 必须在册）      │  ← 挡未知工具
├─────────────────────────────────────┤
│  第 2 层：参数校验（Pydantic/Schema）  │  ← 挡类型漂移、注入
├─────────────────────────────────────┤
│  第 3 层：权限控制（who can call what）│  ← 挡越权调用
├─────────────────────────────────────┤
│  第 4 层：审计与限流                   │  ← 事后追溯 + 防滥用
└─────────────────────────────────────┘
```

### 5.2 用代码实现安全执行器

```python
import json
import time
import logging
from typing import Any, Callable
from pydantic import BaseModel, ValidationError

logger = logging.getLogger(__name__)


class ToolExecutor:
    """安全的工具执行器，实现四层安全模型。"""
    
    def __init__(self):
        self._registry: dict[str, Callable] = {}
        self._validators: dict[str, type[BaseModel]] = {}
        self._permissions: dict[str, set[str]] = {}  # tool_name → allowed_roles
        self._call_counts: dict[str, int] = {}
        self._rate_limit = 100  # 每分钟最多调用次数
    
    def register(
        self,
        name: str,
        func: Callable,
        validator: type[BaseModel] | None = None,
        allowed_roles: set[str] | None = None,
    ):
        """注册工具到白名单。"""
        self._registry[name] = func
        if validator:
            self._validators[name] = validator
        if allowed_roles:
            self._permissions[name] = allowed_roles
    
    def execute(
        self,
        name: str,
        arguments_json: str,
        user_role: str = "user",
        request_id: str = "",
    ) -> str:
        """安全执行工具调用，返回 JSON 字符串结果。"""
        start = time.monotonic()
        
        # ── 第 1 层：白名单 ──
        if name not in self._registry:
            logger.warning("blocked unknown tool: %s [req=%s]", name, request_id)
            return json.dumps({"error": f"未知工具: {name}"})
        
        # ── 第 2 层：参数校验 ──
        try:
            args = json.loads(arguments_json)
        except json.JSONDecodeError as e:
            logger.error("bad JSON from model: %s [req=%s]", e, request_id)
            return json.dumps({"error": "参数 JSON 格式错误"})
        
        if name in self._validators:
            try:
                validated = self._validators[name].model_validate(args)
                args = validated.model_dump()
            except ValidationError as e:
                logger.warning("validation failed: %s [req=%s]", e, request_id)
                return json.dumps({"error": f"参数校验失败: {e.error_count()} 个错误"})
        
        # ── 第 3 层：权限控制 ──
        if name in self._permissions:
            if user_role not in self._permissions[name]:
                logger.warning(
                    "permission denied: %s for role %s [req=%s]",
                    name, user_role, request_id,
                )
                return json.dumps({"error": "权限不足"})
        
        # ── 第 4 层：审计与限流 ──
        self._call_counts[name] = self._call_counts.get(name, 0) + 1
        
        # ── 执行 ──
        try:
            result = self._registry[name](**args)
            elapsed = time.monotonic() - start
            logger.info(
                "tool_call name=%s elapsed=%.3fs [req=%s]",
                name, elapsed, request_id,
            )
            return json.dumps(result, ensure_ascii=False, default=str)
        except Exception as e:
            logger.exception("tool execution error: %s [req=%s]", name, request_id)
            return json.dumps({"error": f"执行失败: {type(e).__name__}"})
```

### 5.3 常见攻击向量与防御

| 攻击 | 模型可能输出的参数 | 防御 |
|------|-------------------|------|
| **路径穿越** | `{"path": "../../etc/passwd"}` | 函数内 `os.path.realpath()` + 检查前缀 |
| **SQL 注入** | `{"query": "'; DROP TABLE users;--"}` | 参数化查询，永远不拼接 SQL |
| **命令注入** | `{"cmd": "ls; rm -rf /"}` | 不暴露任何执行 shell 的工具 |
| **SSRF** | `{"url": "http://169.254.169.254/..."}` | URL 白名单 + 禁内网 IP 段 |
| **参数膨胀** | 传入 10MB 的 JSON 字符串 | 参数长度限制 |

**核心原则**：永远不信任模型输出的任何字段值。把模型当作「聪明但可能被注入的外部用户」。

**本节小结（三张卡片）**

- **一句话定义**：安全模型 = **身份 + 授权 + 审计 + 限流** 四层，缺一不可。
- **典型场景**：工具参数里带 `../../etc/passwd`，白名单挡不住路径穿越，要在函数内再校验 realpath。
- **和 A1 类比**：二次校验像 **`@wraps` 外层再包一层权限装饰器** —— 业务函数专注逻辑，横切关注点统一处理。

## 6. 结构化输出：JSON Mode 与 Pydantic

**本节概念**：**JSON Mode** 保证语法是对象；**Pydantic** 保证语义与类型；**Function Calling** 解决「动作」而不仅是「数据形状」。三者不是互斥的，而是解决不同层面的问题。

### 6.1 三种结构化手段对比

| 手段 | 保证什么 | 不保证什么 | 适用场景 |
|------|---------|-----------|---------|
| **response_format: json_object** | 输出是合法 JSON | 字段名/类型/必填 | 简单提取，字段少 |
| **response_format: json_schema** | 输出严格符合 schema | 字段值的业务合理性 | OpenAI 最新能力，字段多且严格 |
| **Pydantic model_validate_json** | 类型+约束+默认值 | 模型是否输出了 JSON | 后端校验层，任何场景 |
| **Function Calling** | 函数名+参数 JSON | 函数是否存在、参数是否合理 | 需要执行动作 |

### 6.2 JSON Mode 实战

```python
# JSON Mode：最简单的结构化输出
response = client.chat.completions.create(
    model="qwen3:8b",
    messages=[
        {"role": "system", "content": (
            "从用户描述中提取结构化信息，以 JSON 格式输出。"
            "字段：name(str), age(int), skills(list[str])"
        )},
        {"role": "user", "content": "我叫张三，今年28岁，会Python和Java"},
    ],
    response_format={"type": "json_object"},  # 保证输出是 JSON
)

# 模型输出：{"name": "张三", "age": 28, "skills": ["Python", "Java"]}
# 但！字段名可能是 "姓名" 而不是 "name"，没有强制约束
data = json.loads(response.choices[0].message.content)
```

**JSON Mode 的坑**：
- 只保证语法是 JSON，不保证字段名、类型、数量
- 必须在 system/user message 里提到 "JSON"，否则部分模型会忽略 response_format
- 模型可能输出 `{"name": "张三", "年龄": "二十八"}` —— 字段名和类型都跑偏

### 6.3 Pydantic 校验层

Pydantic 是 Python 生态中最强的数据校验库。它解决的是 JSON Mode 解决不了的问题：**字段必须叫什么名字、必须是什么类型、缺了怎么办**。

```python
from pydantic import BaseModel, Field, field_validator


class UserProfile(BaseModel):
    """用户信息提取结果。"""
    name: str = Field(description="用户姓名")
    age: int = Field(ge=0, le=150, description="年龄")
    skills: list[str] = Field(default_factory=list, description="技能列表")
    email: str | None = Field(default=None, description="邮箱，可选")
    
    @field_validator("name")
    @classmethod
    def name_not_empty(cls, v: str) -> str:
        if not v.strip():
            raise ValueError("姓名不能为空")
        return v.strip()
    
    @field_validator("skills")
    @classmethod
    def deduplicate_skills(cls, v: list[str]) -> list[str]:
        return list(dict.fromkeys(v))  # 保序去重


# 使用：
raw_json = response.choices[0].message.content  # 模型输出的 JSON 字符串

try:
    profile = UserProfile.model_validate_json(raw_json)
    print(f"解析成功: {profile.name}, {profile.age}岁")
except ValidationError as e:
    print(f"校验失败: {e.error_count()} 个错误")
    for err in e.errors():
        print(f"  - {err['loc']}: {err['msg']}")
```

### 6.4 Pydantic 生成 JSON Schema

Pydantic 模型可以直接导出 JSON Schema，用在 tools 的 parameters 字段里。这避免了手写 schema 和代码不同步的问题：

```python
# 从 Pydantic 模型自动生成 JSON Schema
class WeatherQuery(BaseModel):
    city: str = Field(description="城市名称")
    unit: str = Field(
        default="celsius",
        description="温度单位",
        json_schema_extra={"enum": ["celsius", "fahrenheit"]},
    )

# 自动导出
schema = WeatherQuery.model_json_schema()
print(json.dumps(schema, indent=2, ensure_ascii=False))
# 输出：
# {
#   "type": "object",
#   "properties": {
#     "city": {"type": "string", "description": "城市名称"},
#     "unit": {"type": "string", "description": "温度单位",
#              "enum": ["celsius", "fahrenheit"], "default": "celsius"}
#   },
#   "required": ["city"]
# }

# 直接用在 tools 里
tool_def = {
    "type": "function",
    "function": {
        "name": "get_weather",
        "description": "查询天气",
        "parameters": WeatherQuery.model_json_schema(),
    }
}
```

**这个模式的好处**：schema 和校验代码是同一个 Pydantic 模型生成的，不会出现「schema 改了但校验没改」的不一致。

### 6.5 解析失败的三步处理

生产环境中模型输出不是 100% 可靠的 JSON，需要有优雅的降级策略：

```python
import json
from pydantic import BaseModel, ValidationError

MAX_RETRIES = 2

def parse_with_retry(
    client,
    model: str,
    messages: list[dict],
    output_model: type[BaseModel],
    max_retries: int = MAX_RETRIES,
) -> BaseModel | None:
    """解析结构化输出，失败时带错误信息重试。"""
    
    for attempt in range(max_retries + 1):
        response = client.chat.completions.create(
            model=model,
            messages=messages,
            response_format={"type": "json_object"},
            temperature=0.0 if attempt > 0 else 0.3,  # 重试时降温度
        )
        
        raw = response.choices[0].message.content
        
        # 第 1 步：尝试解析 JSON
        try:
            data = json.loads(raw)
        except json.JSONDecodeError as e:
            if attempt < max_retries:
                # 把错误信息追加到 messages，让模型修正
                messages.append({"role": "assistant", "content": raw})
                messages.append({"role": "user", "content": (
                    f"你的输出不是合法 JSON，错误：{e}。"
                    "请重新输出，确保是合法的 JSON 格式。"
                )})
                continue
            return None
        
        # 第 2 步：Pydantic 校验
        try:
            return output_model.model_validate(data)
        except ValidationError as e:
            if attempt < max_retries:
                error_detail = "; ".join(
                    f"{'.'.join(str(x) for x in err['loc'])}: {err['msg']}"
                    for err in e.errors()
                )
                messages.append({"role": "assistant", "content": raw})
                messages.append({"role": "user", "content": (
                    f"JSON 格式正确但字段校验失败：{error_detail}。"
                    "请修正后重新输出。"
                )})
                continue
            return None
    
    return None
```

**三步处理总结**：
1. **json.loads 失败** → 降温度 + 把错误信息反馈给模型重试
2. **Pydantic 校验失败** → 把具体哪个字段错了告诉模型重试  
3. **重试耗尽** → 降级（返回 None / 走人工 / 用默认值）

### 6.6 与幻觉控制的关系（衔接 C1 / C3）

结构化输出解决的是「**形状合法**」；**事实是否成立**仍要靠 **RAG 证据、工具回包、业务规则** 校验。

举个例子：模型可能输出完美合法的 JSON：

```json
{"invoice_amount": 99999.99, "invoice_date": "2026-04-18"}
```

Pydantic 校验通过——类型对、格式对。但金额可能是编造的！所以：
- **Pydantic 管形状**：字段名对不对、类型对不对、格式对不对
- **业务规则管事实**：金额是否与数据库一致、日期是否在合理范围内

把关键字段（金额、日期、枚举）放进 Pydantic 后，再用 **代码或规则引擎** 做二次断言，才能把「流畅但错误」挡在上线前。

**本节小结（三张卡片）**

- **一句话定义**：JSON Mode 管括号，Pydantic 管字段含义。
- **典型场景**：抽取发票字段，`amount` 有时是字符串 `"1,234.56"`，Pydantic 前置规范化。
- **和 A1 类比**：`model_validate_json` 像 **`ast.literal_eval` + 类型检查** 的工业加强版。

## 7. 多工具并行调用

**本节概念**：模型可以在一次响应中同时调用多个工具（`tool_calls` 数组长度 > 1）。宿主需要并行执行并把所有结果写回。

### 7.1 什么时候会触发并行调用

当用户的请求天然涉及多个独立查询时：

```
用户：北京和上海今天天气怎么样？
```

模型可能返回：

```python
assistant_msg.tool_calls = [
    ToolCall(id="call_1", function=Function(name="get_weather", arguments='{"city":"北京"}')),
    ToolCall(id="call_2", function=Function(name="get_weather", arguments='{"city":"上海"}')),
]
```

### 7.2 并行执行的实现

```python
import asyncio
import json


async def execute_tools_parallel(
    tool_calls: list,
    registry: dict[str, Callable],
) -> list[dict]:
    """并行执行多个工具调用。"""
    
    async def run_one(tool_call) -> dict:
        name = tool_call.function.name
        call_id = tool_call.id
        
        if name not in registry:
            content = json.dumps({"error": f"未知工具: {name}"})
        else:
            try:
                args = json.loads(tool_call.function.arguments)
                # 如果底层函数是同步的，用 to_thread 避免阻塞事件循环
                result = await asyncio.to_thread(registry[name], **args)
                content = json.dumps(result, ensure_ascii=False, default=str)
            except Exception as e:
                content = json.dumps({"error": str(e)})
        
        return {"role": "tool", "tool_call_id": call_id, "content": content}
    
    # 并发执行所有工具调用
    return await asyncio.gather(*(run_one(tc) for tc in tool_calls))
```

### 7.3 写回顺序

并行执行后，`tool` 消息的写回顺序 **不需要和 tool_calls 顺序一致**（因为有 `tool_call_id` 做匹配），但建议保持一致以便调试。

```python
# 执行完毕后，按顺序追加到 messages
messages.append(assistant_msg.model_dump())  # assistant 原消息
for tool_result in tool_results:  # 所有 tool 消息
    messages.append(tool_result)
```

**本节小结（三张卡片）**

- **一句话定义**：并行调用 = 一个 assistant 消息里多个 `tool_calls`，每个都要有对应的 `tool` 回包。
- **典型场景**：同时查天气、查航班、查酒店 → 三个并行 tool_calls → 三个 tool 消息写回。
- **和 A1 类比**：像 `asyncio.gather()` —— 独立任务并发跑，全部完成后汇总。

## 8. 常见坑与排错

**本节概念**：排错时区分 **「模型没提议」**、**「提议了但 JSON 坏了」**、**「执行失败」**、**「写回格式错了」** 四层，每层对策不同。

### 8.1 四层排错法

```
问题现象              可能原因                        排查方法
─────────────────────────────────────────────────────────────
模型不调工具         ① description 太模糊              检查 description 是否清晰
                    ② 模型不支持 tools                 查模型文档/版本
                    ③ tool_choice="none"              检查请求参数
                    ④ system prompt 暗示直接回答       审查 system prompt

arguments 非法JSON   ① 模型能力不足                    换更强的模型
                    ② 参数 schema 太复杂              简化嵌套层级
                    ③ 流式拼接没拼完                   检查流式处理逻辑

执行失败             ① 参数值非法（类型对但值错）       加 Pydantic validator
                    ② 下游服务超时                     加 timeout + 重试
                    ③ 权限不足                        检查鉴权链路

写回后模型行为异常    ① tool_call_id 不匹配             检查 ID 是否原样复制
                    ② 漏掉 assistant 原消息            检查 messages 追加逻辑
                    ③ tool content 太长               截断 + 摘要
```

### 8.2 典型 Bug 案例

**案例一：死循环**

```python
# 模型反复调同一个工具，相同参数
# 原因：工具返回的结果模型"看不懂"，所以它再试一次

# 解决方案：
class LoopDetector:
    """检测工具调用死循环。"""
    
    def __init__(self, max_repeats: int = 3):
        self._history: list[str] = []
        self._max_repeats = max_repeats
    
    def check(self, name: str, arguments: str) -> bool:
        """返回 True 表示检测到循环。"""
        key = f"{name}:{arguments}"
        self._history.append(key)
        
        # 检查最近 N 次是否都是同一个调用
        if len(self._history) >= self._max_repeats:
            recent = self._history[-self._max_repeats:]
            if len(set(recent)) == 1:
                return True
        return False
```

**案例二：流式响应中的 tool_calls 拼接**

```python
# 流式模式下，tool_calls 的 arguments 是分块到达的
# 不能每到一个 chunk 就 json.loads，需要拼完整再解析

tool_calls_buffer: dict[int, dict] = {}  # index → {id, name, args_chunks}

for chunk in stream:
    delta = chunk.choices[0].delta
    if delta.tool_calls:
        for tc in delta.tool_calls:
            idx = tc.index
            if idx not in tool_calls_buffer:
                tool_calls_buffer[idx] = {
                    "id": tc.id,
                    "name": tc.function.name if tc.function and tc.function.name else "",
                    "args": "",
                }
            if tc.function and tc.function.arguments:
                tool_calls_buffer[idx]["args"] += tc.function.arguments

# 流结束后，tool_calls_buffer 里有完整的工具调用
for idx, tc in tool_calls_buffer.items():
    args = json.loads(tc["args"])  # 现在才能安全解析
    # ...执行工具
```

**案例三：把密钥写进 schema 示例**

```python
# ❌ 绝对不要这样写
"description": "调用支付接口，api_key 用 sk-live-abc123..."

# ✅ 正确做法
"description": "调用支付接口，api_key 在服务端配置，无需传入"
# 密钥在宿主代码的环境变量里，永远不进 schema/prompt
```

### 8.3 Ollama 特定问题

如果你用 Ollama 本地模型练习 Function Calling，注意以下问题：

| 问题 | 原因 | 解决方案 |
|------|------|---------|
| 永远不返回 `tool_calls` | 模型不支持 tools | 换 qwen3:8b / llama3.1 等支持 tools 的模型 |
| `arguments` 是空字符串 | Ollama 版本太旧 | 升级到 0.5+ |
| 中文参数乱码 | 模型 tokenizer 问题 | `ensure_ascii=False` + 检查模型中文能力 |
| 调用了但函数名拼错 | 模型能力不足 | 简化函数名（避免太长或含特殊字符） |

**本节小结（三张卡片）**

- **一句话定义**：工具排错 = **契约层 + 执行层 + 消息层** 分层二分。
- **典型场景**：Ollama 某旧版对 `tool_calls` 支持不完整，表现为永远无 `tool_calls` 字段 —— 先查服务端版本矩阵。
- **和 A1 类比**：像调试 **多层装饰器** —— 先确定哪一层把异常吃了。

## 9. 与 C5、C6 的关系

**本节概念**：C4 定义 **单进程内的工具循环**；C5 把工具 **跨进程/跨产品复用**；C6 把循环 **产品化**（停止条件、记忆、观测）。

### 9.1 三章的递进关系

```
C4: Function Calling                C5: 工具链与 MCP              C6: 单 Agent 开发
┌─────────────────────┐      ┌─────────────────────┐      ┌─────────────────────┐
│ 模型 ←→ 单个宿主     │      │ 模型 ←→ 多个工具服务  │      │ 模型 + 工具 + 记忆    │
│ 工具=宿主内的函数     │  →   │ 工具=独立进程/远程API │  →   │ + 停止条件 + 观测     │
│ 手写 dispatch 表     │      │ 标准协议(MCP)自动发现 │      │ = 完整 Agent         │
│ 单轮/少轮            │      │ 工具跨项目复用       │      │ 多轮自主循环         │
└─────────────────────┘      └─────────────────────┘      └─────────────────────┘
```

- **C4 → C5**：C4 里你手动维护 `TOOL_REGISTRY` 字典。C5 用 MCP 协议让工具 **自描述** + **自注册**，宿主不需要提前知道有哪些工具。

- **C4 → C6**：C4 里通常只有 1-2 轮工具调用就结束了。C6 的 Agent 会 **自主循环**——调用工具、观察结果、决定下一步——直到达到目标或触发停止条件。C4 的 `tool_call → execute → write_back` 循环是 C6 Agent 的原子操作。

### 9.2 一句话总结

> C4 教你 **砖怎么烧**（单次工具调用协议），C5 教你 **砖从哪来**（工具发现与复用），C6 教你 **怎么砌墙**（把砖组装成完整的 Agent）。

## 12. 面试高频 Q&A

### Q1：Function Calling 和 RAG 有什么区别？

**A**：它们解决的问题层面不同：

| | Function Calling | RAG |
|---|---|---|
| **解决什么** | 让模型触发动作（查询、写入、调用 API） | 让模型基于外部知识回答问题 |
| **数据流向** | 模型 → 宿主执行 → 结果回模型 | 知识库 → 检索 → 拼入 prompt → 模型 |
| **执行权** | 宿主 | 无执行，只是拼上下文 |
| **典型场景** | 下单、查物流、发邮件 | 回答产品手册问题、法规查询 |

在实际 Agent 中二者经常组合：先用 RAG 检索相关知识，再用 Function Calling 执行具体动作。

### Q2：模型返回的 arguments 不是合法 JSON 怎么办？

**A**：分三步走：

1. **第一道防线**：用 `json.loads()` 捕获 `JSONDecodeError`，记录原始输出到日志
2. **重试策略**：把错误信息追加到 messages 告诉模型"你的输出不是合法 JSON"，降低 temperature 重试
3. **降级策略**：重试 2-3 次仍然失败 → 降级到纯文本回复或走人工处理

代码实现见 §6.5 的 `parse_with_retry` 函数。

### Q3：工具太多（50+）时模型选不准怎么办？

**A**：工具膨胀是生产级 Agent 的常见问题。解决方案按优先级：

1. **动态挂载**：根据对话阶段/用户意图，只挂载相关的工具子集（5-10个）。比如客服场景，先判断用户是"查询"还是"投诉"，再挂载对应工具组。

2. **两级路由**：先用一个轻量分类器（或 LLM）判断意图分类，再把请求路由到专门的子 Agent，每个子 Agent 只有 3-5 个工具。

3. **工具描述优化**：精简 description，去掉冗余信息，减少 token 消耗和模型困惑。

4. **合并同类工具**：如果有 `get_order_by_id`、`get_order_by_phone`、`get_order_by_email`，考虑合并成一个 `get_order(query_type, query_value)`。

### Q4：Function Calling 和 Agent 的关系是什么？

**A**：Function Calling 是 Agent 的 **原子能力**。

- **Function Calling** = 模型的一次工具调用请求（`tool_calls`）+ 宿主执行 + 结果写回。这是一个单步操作。
- **Agent** = 多轮 Function Calling 的自主循环。Agent 的核心循环是：观察 → 思考 → 行动（调工具）→ 观察结果 → 思考 → 行动...直到任务完成。

没有 Function Calling，Agent 就只能"说话"不能"做事"。Function Calling 给了 Agent 与外部世界交互的能力。

### Q5：如何测试 Function Calling 的正确性？

**A**：分层测试：

```python
# 1. 单元测试：测工具函数本身（不涉及模型）
def test_get_weather():
    result = get_weather("北京")
    assert "temp" in result
    assert "condition" in result

# 2. 集成测试：测 dispatch + 校验逻辑（用 mock 模型输出）
def test_tool_executor():
    executor = ToolExecutor()
    executor.register("get_weather", get_weather, WeatherQuery)
    
    # 模拟模型输出
    result = executor.execute("get_weather", '{"city": "北京"}')
    assert "error" not in result
    
    # 测白名单
    result = executor.execute("drop_table", '{}')
    assert "未知工具" in result
    
    # 测参数校验
    result = executor.execute("get_weather", '{"city": 123}')
    assert "校验失败" in result

# 3. E2E 测试：真实调用模型（成本高，少量关键路径）
def test_e2e_weather_query():
    messages = [{"role": "user", "content": "北京天气"}]
    response = client.chat.completions.create(
        model="qwen3:8b", messages=messages, tools=tools
    )
    assert response.choices[0].message.tool_calls is not None
    assert response.choices[0].message.tool_calls[0].function.name == "get_weather"
```

### Q6：Pydantic v1 和 v2 在 LLM 应用中的主要区别？

**A**：
- **v2 性能提升 5-50 倍**（Rust 核心），高并发 Agent 场景差异明显
- **v2 的 `model_validate_json()`** 直接解析 JSON 字符串，跳过中间 dict，比 v1 的 `parse_raw()` 更快
- **v2 的 `model_json_schema()`** 输出更标准的 JSON Schema，与 OpenAI tools 格式兼容性更好
- **v2 淘汰了 `schema()` 和 `parse_raw()`**，迁移时注意方法名变化
- LangChain / OpenAI SDK 等主流框架已全面迁移到 v2

### Q7：实际项目中 tool_choice 怎么选？

**A**：根据对话阶段动态调整：

```python
def decide_tool_choice(conversation_stage: str, user_intent: str) -> str | dict:
    """根据对话阶段决定 tool_choice。"""
    if conversation_stage == "greeting":
        return "none"  # 打招呼阶段不需要工具
    
    if conversation_stage == "data_collection":
        return "auto"  # 模型自己判断是否需要查询
    
    if user_intent == "explicit_action":
        return "required"  # 用户明确要求执行动作
    
    if conversation_stage == "summary":
        return "none"  # 总结阶段不再调工具
    
    return "auto"  # 默认
```
