## 1. 为什么要写测试

### 1.1 一个真实的故事

想象你正在做一个"电商退款接口"。需求很简单：用户点击退款按钮，后端扣减已发货商品数量、把钱退到原支付账户、发一条短信通知。

第一天你写完了，手动点了一下，能跑通，上线。

第二天产品说："退款要发邮件而不是短信。" 你改了一行，上线。

第三天又来个需求："如果退款金额超过 1000 元，要先走审批。" 你加了个 if 判断。

第四天运营过来问你："为什么那个昨天支付的用户的退款没到账？"

你去查日志：**报错发生在发短信的地方，因为某个手机号格式异常，抛出了异常，整个函数还没走到"退钱"那一步就被中断了。但订单状态已经提前被改成了"已退款"。**

你盯着屏幕想：我怎么不知道这会出问题？

答案是：因为你没有测试。

**没有测试的代码，意味着每一次修改都是赌博**。你赌改动的部分不会影响其他部分，你赌你没忘记某个分支，你赌线上数据和你在本地想象的一样。赌赢了叫"迭代快"，赌输了叫"线上事故"。

### 1.2 手动测试为什么不够

你可能会说："那我改完多测几遍不就行了吗？"

问题在于：

1. **手动测试的成本随代码规模爆炸式增长**。你的后端有 20 个接口，每个接口平均 5 种情况（正常、参数错、无权限、并发、数据不存在），总共 100 种组合。每次修改一行代码，你愿意手动跑 100 遍吗？
2. **人会遗忘分支**。写这行代码时你可能考虑到了"金额为负"的情况，但一周后改的时候你还记得吗？
3. **人会犯懒**。跑 3 遍都正常，第 4 遍你就不跑了。偏偏就是第 4 遍会出问题。
4. **手动测试无法自动回归**。所谓"回归测试"是指：改一个地方，把所有以前跑过的用例全再跑一遍，确认没把别的地方改坏。手动回归的成本几乎等于从头测一遍。

### 1.3 用装修毛坯房类比

写代码没测试，像**装修毛坯房没验收**。

- 水电走完，不通电不打水压——验不了漏。
- 瓷砖贴完，不做空鼓检测——三个月后开裂。
- 墙漆刷完，不等干透就压东西——留指印。

验收不是拖慢工期，验收是**拦截 80% 返工的唯一手段**。代码里的测试就是"每道工序完成后的小验收"。

### 1.4 测试到底给你带来什么

一个合格的测试套件让你获得以下四件事：

1. **信心（Confidence）**。改完代码敢直接合入主干。没测试的代码你连删一行都怕。
2. **文档（Documentation）**。测试本身就是"这个函数该怎么用"的最准确文档——因为它是会被执行验证的。
3. **设计反馈（Design feedback）**。写不出测试的代码大概率是烂设计。函数依赖 20 个全局变量？难测。类之间耦合到拆不开？难测。测试是照妖镜。
4. **重构安全网（Refactoring safety net）**。有测试你才敢说"这块代码我要重写一遍"。没测试，你只能在烂代码上继续糊屎。

> **读到这里你应该理解**：测试不是"质量保证部门"的事，它是**写代码的人自己的生产力工具**。写测试不是在给产品经理交差，是在给未来的自己留一张安全网。

### 1.5 常见误区

- **"我这个项目小，写测试是浪费时间"**。小项目往往是被"慢慢变大"的项目，等你发现它变大时已经写不动测试了。正确的做法是从第一天就写，哪怕只写一两个。
- **"测试覆盖率要 100%"**。覆盖率 80% 和 100% 的成本差距能翻倍，但收益很小。真正值得测的是核心业务逻辑和边界条件。
- **"测试代码不是生产代码，不讲究质量"**。错。烂的测试代码会把你拖进"改一行业务代码要改十行测试"的噩梦。测试代码的可维护性和业务代码同等重要。

## 2. 测试金字塔

### 2.1 三层分类

软件测试通常按"**粒度**"和"**速度**"分为三层，画出来像个金字塔：

```
           ┌─────────────┐
           │   E2E/UI    │   少量，慢，贵
           │  端到端测试  │
           └─────────────┘
         ┌─────────────────┐
         │   Integration   │   中等数量，中等速度
         │   集成测试        │
         └─────────────────┘
       ┌─────────────────────┐
       │      Unit Tests     │   大量，快，便宜
       │      单元测试         │
       └─────────────────────┘
```

### 2.2 单元测试（Unit Test）

**定义**：测一个"最小可测单元"——通常是一个函数或一个类的一个方法——**不依赖任何外部资源**（数据库、网络、文件系统）。

**特点**：
- 一个测试跑起来应该在几毫秒以内。
- 一次可以跑几千个。
- 失败时定位问题非常精确："就是这个函数的这个分支错了"。

**例子**：测试一个 `calculate_price(base_price, discount_rate)` 函数，给不同输入，断言返回值对不对。这个函数内部不查数据库、不调 API，纯计算，非常适合单测。

**在 Agent 后端里**：测 "token 计数函数"、"prompt 模板渲染函数"、"消息格式转换函数"——这些都是纯函数，写单测成本最低、收益最高。

### 2.3 集成测试（Integration Test）

**定义**：测**多个单元组合在一起**是否正常工作，通常会涉及**真实的外部资源**，比如真的连一个测试数据库、真的启一个 HTTP server。

**特点**：
- 一个测试可能要几十毫秒到几秒。
- 数量比单测少一个数量级。
- 失败时问题可能在被测代码，也可能在环境（数据库坏了、网络抖了）。

**例子**：
- 测 `UserRepository.create()` + `OrderRepository.create()` 能否在同一事务里正常工作。
- 测 FastAPI 的一个 endpoint，发 HTTP 请求进去，看数据库真的写进记录没有。

**在 Agent 后端里**：测"调用 LLM + 解析响应 + 写入对话历史"这一条链路，通常会 mock 掉真实的 LLM 调用但保留数据库操作。

### 2.4 E2E 测试（End-to-End）

**定义**：模拟**真实用户行为**，从前端点一下按钮开始，一直到数据库里数据改变、邮件发出，整个流程都测。

**特点**：
- 一个测试几秒甚至几十秒。
- 极不稳定（flaky），网络抖一下就失败。
- 维护成本高，前端改个按钮 id 测试就挂。
- 但它是唯一能验证"用户真的能完成一个完整工作流"的测试。

**例子**：用 Playwright/Selenium 启个浏览器，打开网站，登录，点"下单"，等订单出现在"我的订单"里。

### 2.5 为什么是金字塔而不是倒三角

初学者常见的错误认知是："我最关心用户体验，所以我应该多写 E2E 测试。" 这就是所谓的"**冰淇淋反模式**"。

错的原因：

1. **速度**。E2E 一个 30 秒，单测一个 5 毫秒，速度差 6000 倍。CI 全挂上 E2E，跑一次半小时起步，开发体验灾难。
2. **稳定性**。E2E 依赖的环节太多，任意一环抖动都会失败。你会花大量时间"修复不是 bug 的失败"。
3. **定位问题难**。E2E 失败只告诉你"某个流程挂了"，不告诉你"哪个函数错了"。
4. **覆盖能力差**。一个 E2E 只能覆盖一条路径，一个单测一个用例也只覆盖一条路径，但单测的用例可以写 100 个，E2E 不行。

**合理比例**大致是：**单测 70% / 集成 20% / E2E 10%**。

### 2.6 类比：汽车制造

想象你是汽车厂质检：

- **单测**就像"每个零件单独测"：这颗螺丝能不能承受 500 牛顿的扭力？这个灯泡亮度够不够？快、便宜、可以全量测。
- **集成测试**就像"装好发动机试运转"：汽缸、活塞、点火系统组合起来能不能正常工作？中等成本。
- **E2E**就像"整车路试"：司机开上跑道，从打火到停车全流程走一遍。最贵、最慢、最像真实场景，但绝不能作为唯一质检手段。

> **读到这里你应该理解**：测试是分层的，不同层解决不同问题。写业务代码要刻意去想"这段逻辑该用哪一层测"，而不是一股脑全用单测或全用 E2E。

### 2.7 常见误区

- **"我都写单测就好了"**。纯单测覆盖不了模块之间接口的问题。单测全过不代表系统跑得起来。
- **"集成测试和 E2E 是一回事吧？"** 不是。集成测试一般在进程内跑（不启动浏览器），E2E 才模拟完整用户环境。
- **"单元测试的'单元'就是一个文件/一个类"**。粒度其实是"一个行为"而不是"一个类"。测一个类的时候经常会用到它依赖的其他类（用真对象，不 mock），这也还是单测。

## 3. unittest vs pytest：为什么 pytest 赢了

### 3.1 Python 标准库自带的 unittest

Python 的标准库里有一个 `unittest` 模块，学名"xUnit 风格"。来自 Java 的 JUnit 那一脉，特征是"一切都是类"。

一个 unittest 风格的测试长这样：

```python
import unittest

class TestCalculator(unittest.TestCase):
    def setUp(self):
        # 每个测试方法运行前执行
        self.calc = Calculator()

    def tearDown(self):
        # 每个测试方法运行后执行
        self.calc.close()

    def test_add(self):
        self.assertEqual(self.calc.add(1, 2), 3)

    def test_divide_by_zero(self):
        with self.assertRaises(ZeroDivisionError):
            self.calc.divide(10, 0)

if __name__ == "__main__":
    unittest.main()
```

看着挺正式，但问题很大。

### 3.2 unittest 的痛点

**痛点一：必须继承类，必须写在方法里。**
一个简单的加法测试也要写一个类。对于一个 10 行代码的小脚本，测试要先写 5 行模板代码。

**痛点二：断言 API 又多又难记。**
你不能写 `assert x == y`，要写 `self.assertEqual(x, y)`。还有 `assertTrue`、`assertIn`、`assertAlmostEqual`、`assertIsInstance`、`assertRaisesRegex`……一共几十个。
每次都要问 IDE "assert 开头有哪些方法来着"。

**痛点三：setUp/tearDown 不够灵活。**
如果两个测试需要不同的准备逻辑，你要么拆成两个类，要么在 setUp 里用 if 判断方法名——都很丑。

**痛点四：失败时的错误信息贫瘠。**
`self.assertEqual(a, b)` 失败时只告诉你 "a != b"，不告诉你差在哪里，尤其是比较列表/字典时根本看不清差异。

### 3.3 pytest 的四大杀手级特性

`pytest` 是第三方库，但已经是 Python 测试的**事实标准**（几乎所有主流开源项目都用它）。它解决了 unittest 的所有痛点：

**特性一：原生 assert，失败信息自动展开。**
你只要写 `assert x == y`，pytest 会在背后做"**assertion rewriting**"（断言重写），自动打印出 `x` 和 `y` 的详细值。比如：

```
>       assert user.name == "Alice"
E       AssertionError: assert 'alice' == 'Alice'
E         - Alice
E         + alice
```

列表/字典对比时还会高亮差异。对比 unittest 的 `"a != b"` 直接起飞。

**特性二：函数就是测试，不用写类。**
```python
def test_add():
    assert add(1, 2) == 3
```
三行搞定。

**特性三：Fixture 系统。**
后面 5 节专门讲。简单说 fixture 是一个比 setUp/tearDown 强十倍的"测试数据准备机制"，支持作用域、依赖注入、yield 清理。这是 pytest 最核心的创新。

**特性四：丰富的插件生态。**
- `pytest-asyncio` 测异步
- `pytest-cov` 测覆盖率
- `pytest-mock` 方便 mock
- `pytest-xdist` 并行运行
- `pytest-httpx` 测 HTTP 客户端
- `pytest-vcr` 录制回放 HTTP 响应

你想要的几乎都有。

### 3.4 兼容性

**好消息**：pytest **向下兼容 unittest**。你用 unittest 写的测试，pytest 能直接运行。所以迁移成本几乎为零——旧代码不动，新代码用 pytest 风格写。

### 3.5 结论

**从今天起只学 pytest**。unittest 只要看得懂就行（因为老项目会遇到），不要再用它写新代码。本章后面所有内容都围绕 pytest。

> **读到这里你应该理解**：pytest 不是"又一个测试框架"，它是 Python 社区在使用 unittest 多年后总结出"什么才叫好用"并重新设计的产物。它的核心创新是 fixture 系统和 assertion rewriting。

## 4. pytest 基础

### 4.1 安装

```bash
pip install pytest
```

就这一行。pytest 自带命令行工具 `pytest`，也可以用 `python -m pytest` 调用。

### 4.2 第一个测试

创建一个文件 `test_demo.py`：

```python
def add(a, b):
    return a + b

def test_add():
    assert add(1, 2) == 3

def test_add_negative():
    assert add(-1, -2) == -3
```

在这个文件所在目录运行：

```bash
pytest
```

你会看到类似这样的输出：

```
test_demo.py ..                                          [100%]

================= 2 passed in 0.01s ==================
```

两个点代表两个测试全过。`[100%]` 是进度条。

### 4.3 测试发现规则

pytest 怎么知道哪些是测试、哪些不是？规则很简单：

1. **文件名**：`test_*.py` 或 `*_test.py`。
2. **函数名**：以 `test_` 开头的函数。
3. **类名**：以 `Test` 开头的类（类里的测试方法还是要以 `test_` 开头）。而且**类不能有 `__init__` 方法**，否则 pytest 拒收。
4. **搜索目录**：默认从运行 `pytest` 命令的当前目录开始递归。

你也可以在 `pytest.ini` 或 `pyproject.toml` 里改这些规则，但不建议——遵守默认约定最省心。

### 4.4 assert 的魔法

在普通 Python 代码里，`assert x == y` 失败时抛出的 `AssertionError` 不会告诉你 x 和 y 的值。但在 pytest 里会。

原因是 pytest 做了**assertion rewriting**：它在收集测试文件时重写 AST，把每个 `assert` 语句替换成一段"失败时打印详细信息"的代码。听起来像黑魔法，但你完全不用关心——你只管写 `assert`，pytest 帮你变 magic。

能打印详细信息的包括：

```python
# 数字/字符串比较
assert x == y

# 列表/字典/集合比较（自动高亮差异）
assert [1, 2, 3] == [1, 2, 4]

# 包含关系
assert "hello" in response.text

# 属性访问
assert user.name == "Alice"

# 复杂表达式
assert sum(items) > len(items) * 10
```

### 4.5 常用命令行参数

```bash
# 只跑一个文件
pytest test_demo.py

# 只跑一个测试函数
pytest test_demo.py::test_add

# 只跑一个类里的一个方法
pytest test_demo.py::TestUser::test_create

# 按名字模糊匹配（-k 是 keyword）
pytest -k "add and not negative"

# 详细输出（-v 是 verbose）
pytest -v

# 特别详细（-vv，对断言差异尤其有用）
pytest -vv

# 第一个失败就停（-x 是 exit on first failure）
pytest -x

# 允许 print 输出（-s 默认会捕获 stdout，加 -s 才能看到 print）
pytest -s

# 显示最慢的 10 个测试（--durations）
pytest --durations=10

# 只跑上次失败的（--lf 是 last failed）
pytest --lf

# 先跑上次失败的，再跑剩下的（--ff 是 failed first）
pytest --ff

# 组合：跑上次失败的，详细输出，第一个失败就停
pytest --lf -vv -x
```

我日常最常用的是 `pytest -vv --lf -x`——开发时改完代码先跑上次失败的，详细输出，挂了就停。

### 4.6 测试函数的"三段式"结构

一个好的测试函数通常分三段，叫 **AAA 模式**：

```python
def test_calculate_discount():
    # Arrange：准备数据
    cart = Cart()
    cart.add_item("apple", price=10, quantity=3)
    cart.add_item("bread", price=5, quantity=2)

    # Act：执行被测行为
    total = cart.calculate_total(discount_rate=0.1)

    # Assert：断言结果
    assert total == 36  # (30 + 10) * 0.9 = 36
```

或者叫 **Given-When-Then**：
- Given：给定什么前置条件
- When：当发生什么动作
- Then：应该得到什么结果

这不是强制语法，是写测试的好习惯。代码结构清晰，别人读起来也容易。

### 4.7 测试"异常"

有时候你要测的不是函数的返回值，而是"它该在某个情况下抛异常"。pytest 提供 `pytest.raises` 做这件事：

```python
import pytest

def divide(a, b):
    if b == 0:
        raise ValueError("除数不能为 0")
    return a / b

def test_divide_by_zero():
    with pytest.raises(ValueError) as exc_info:
        divide(10, 0)
    assert "除数不能为 0" in str(exc_info.value)
```

`with pytest.raises(ValueError)` 的意思是："我期待这个 with 块内部抛出一个 ValueError"。如果真的抛了，测试通过；如果没抛（或者抛的是别的异常），测试失败。

`exc_info` 是异常信息对象，可以用来检查异常消息内容。

### 4.8 跳过和期望失败

有时候你要临时跳过某个测试（比如等 bug 修）：

```python
import pytest

@pytest.mark.skip(reason="等 API 修复")
def test_broken_api():
    ...

# 条件跳过
@pytest.mark.skipif(sys.platform == "win32", reason="仅 Linux")
def test_linux_only():
    ...

# 期望失败（xfail 的意思是"这个本来就应该失败，失败才正常"）
@pytest.mark.xfail(reason="新需求，还没实现")
def test_future_feature():
    ...
```

### 4.9 常见误区

- **测试函数写了返回值**。测试函数不应该 `return` 任何东西，pytest 会警告。测试的唯一效果是"断言通过或失败"。
- **在测试函数里打印调试信息但看不到**。pytest 默认吞掉 stdout，加 `-s` 才能看到 print。
- **多个 assert 混在一个测试里**。一个测试应该只验证一个"事情"。多个 assert 里第一个失败了，后面的根本不会执行，你会看不到全貌。

> **读到这里你应该理解**：pytest 的基础就是"写函数 + 写 assert"。比你想象得简单。复杂的东西（fixture、mock、参数化）是在这个基础上叠加的。

## 5. Fixture 深度解析

### 5.1 问题来源

假设你要测 10 个 API 端点，每个测试都要：
1. 创建一个数据库连接
2. 清空测试表
3. 插入一批假数据
4. 执行测试
5. 关掉连接

如果每个测试函数都写一遍这 5 步，你会疯掉。代码重复、修改成本高、容易漏。

解决方案是"**把准备逻辑提取出来**"——这个"提取出来的准备逻辑"就叫 **fixture**。

### 5.2 最简单的 fixture

```python
import pytest

@pytest.fixture
def sample_user():
    return {"id": 1, "name": "Alice", "age": 30}

def test_user_name(sample_user):
    assert sample_user["name"] == "Alice"

def test_user_age(sample_user):
    assert sample_user["age"] == 30
```

解释：
- `@pytest.fixture` 装饰器把一个函数标记为 fixture。
- fixture 函数的返回值就是"准备好的数据"。
- 测试函数只要**把 fixture 名字写在参数里**，pytest 会自动调用 fixture 并把返回值传进来。这个机制叫**依赖注入（Dependency Injection）**。

"依赖注入"听起来高级，其实就是："我说我需要一个 sample_user，你帮我准备好递进来"。和餐厅你点"一份牛排"服务员给你端上来一样。

### 5.3 yield：在测试后做清理

有些 fixture 需要在测试结束后做清理工作——关闭连接、删除临时文件、回滚事务。pytest 用 `yield` 语法表达这件事：

```python
import pytest

@pytest.fixture
def db_connection():
    # setup 阶段（yield 之前）
    conn = connect_to_db()
    print("打开连接")

    yield conn  # 把 conn 交给测试函数

    # teardown 阶段（yield 之后）
    conn.close()
    print("关闭连接")

def test_query(db_connection):
    result = db_connection.execute("SELECT 1")
    assert result == 1
```

执行顺序：
1. pytest 遇到 `test_query(db_connection)`，发现需要 `db_connection`。
2. pytest 调用 `db_connection()` 函数，走到 `yield conn`，把 `conn` 传给测试。
3. `test_query` 执行，用 `conn` 查询。
4. `test_query` 结束后（无论成功失败），pytest 回到 fixture 的 `yield` 之后，执行 `conn.close()`。

**关键理解**：`yield` 不是 return，`yield` 之后的代码不是马上执行，而是等测试跑完再执行。

**即使测试失败或抛异常，yield 之后的清理代码也会执行。** 这是 yield 相比普通 return 的最大好处。

### 5.4 作用域（Scope）

默认情况下，每个测试函数都会重新调用一次 fixture。如果 fixture 很重（比如启动一个 Docker 容器），这会很慢。

pytest 提供了 5 种作用域，让 fixture 可以在多个测试之间共享：

| scope | 含义 | 什么时候重建 |
|-------|------|-------------|
| `function` | 默认 | 每个测试函数一次 |
| `class` | 类级 | 每个测试类一次 |
| `module` | 模块级 | 每个测试文件一次 |
| `package` | 包级 | 每个测试包一次 |
| `session` | 会话级 | 整个 pytest 运行一次 |

用法：

```python
@pytest.fixture(scope="session")
def expensive_resource():
    print("启动 Docker 容器")
    container = start_container()
    yield container
    print("停止 Docker 容器")
    container.stop()
```

现在整个测试会话里不管有多少个测试用到 `expensive_resource`，容器只启动一次，所有测试结束时才停。

### 5.5 作用域的选择原则

- **尽量用小作用域（function）**。作用域越大，测试之间的"数据污染"风险越大——上一个测试改了 fixture 里的数据，下一个测试看到的就不是"干净"状态。
- **只有当创建/销毁成本很高时才用大作用域**。典型场景：启动数据库容器、加载 ML 模型、建立长连接。
- **改过就污染**。如果你用 session 作用域的数据库，千万不要在测试里真的 INSERT/UPDATE，否则会影响其他测试。要么用事务回滚，要么每个测试前清表。

### 5.6 fixture 之间可以相互依赖

fixture 也可以"用"其他 fixture，只要在参数里写出来：

```python
@pytest.fixture(scope="session")
def db_engine():
    return create_engine("sqlite:///:memory:")

@pytest.fixture(scope="function")
def db_session(db_engine):
    session = Session(bind=db_engine)
    yield session
    session.rollback()  # 每个测试结束回滚
    session.close()

@pytest.fixture
def sample_user(db_session):
    user = User(name="Alice")
    db_session.add(user)
    db_session.flush()
    return user

def test_user_email(sample_user):
    assert sample_user.name == "Alice"
```

这里的依赖链是：`test_user_email → sample_user → db_session → db_engine`。pytest 会自动按依赖顺序实例化所有 fixture。

**注意作用域搭配规则**：**大作用域不能依赖小作用域**。`db_engine`（session）不能依赖 `db_session`（function），否则就没意义了。反过来可以：function 的 `db_session` 可以用 session 的 `db_engine`。

### 5.7 autouse：自动应用

有时候某个 fixture 你希望"所有测试都自动用到"，不用每个测试都写参数。用 `autouse=True`：

```python
@pytest.fixture(autouse=True)
def reset_cache():
    global_cache.clear()
    yield
    global_cache.clear()
```

这个 fixture 会在每个测试运行前后自动执行，清空全局缓存。

**谨慎使用**。autouse 是隐式的，读代码时容易漏，过度用会让测试行为难以追踪。一般只用于"**每个测试都必须做的全局清理**"。

### 5.8 参数化 fixture

fixture 也可以用 `params` 参数一次生成多个：

```python
@pytest.fixture(params=["sqlite", "postgres", "mysql"])
def db_type(request):
    return request.param

def test_db(db_type):
    print(f"测试数据库类型: {db_type}")
```

这个测试会跑 3 遍，每次 `db_type` 是不同的值。`request.param` 是 pytest 注入的一个特殊对象，可以访问当前参数。

### 5.9 依赖注入模型详解

pytest 的 fixture 系统本质上是一个**依赖注入容器**。理解这点后很多东西就不是魔法了：

- 你声明"我需要 X"（写在参数里）。
- pytest 查找名为 X 的 fixture（本文件 → conftest.py → 插件）。
- 找到后调用它（如果它还需要其他 fixture，递归解析）。
- 把返回值注入测试函数。
- 测试结束后按反向顺序执行所有清理（teardown）。

这个思路和 FastAPI 的 `Depends()` 几乎一模一样。学过 FastAPI 的同学会感到亲切。

> **读到这里你应该理解**：fixture = 被依赖注入的"测试资源"。它不是某种特殊的"测试前置"，而是一种资源管理模式。`yield` 让它可以做清理，`scope` 让它可以共享，`autouse` 让它可以自动触发。

### 5.10 常见误区

- **在 fixture 里写 assert**。fixture 是准备数据的，不是做断言的。断言放测试函数里。
- **忘了 yield 清理**。比如开了 DB 连接却没关，连接数会越来越多。
- **作用域和依赖搞混**。session 作用域的 fixture 想用 function 作用域的 fixture——报错。
- **autouse 滥用**。除非确实全局必需，否则用显式依赖更清楚。

## 6. conftest.py 与 Fixture 共享

### 6.1 问题

你在 `test_user.py` 里定义了一个 `sample_user` fixture。现在 `test_order.py` 也要用。怎么办？复制一份？当然不行——违反 DRY 原则。

pytest 的解决方案是：`conftest.py`。

### 6.2 conftest.py 是什么

`conftest.py` 是一个"**魔法文件**"。只要它存在于某个目录，pytest 会自动加载它，**这个目录及其所有子目录下的测试都能直接使用里面定义的 fixture，不需要 import**。

目录结构示例：

```
tests/
├── conftest.py          # 全局 fixture
├── test_user.py
├── test_order.py
└── integration/
    ├── conftest.py      # 集成测试专用 fixture
    └── test_checkout.py
```

`tests/conftest.py` 里定义的 fixture 所有测试都能用；`tests/integration/conftest.py` 里定义的只有 `integration/` 下的测试能用。

**不要 import conftest.py**。pytest 自动加载，你手动 import 反而出错。

### 6.3 典型的 conftest.py

```python
# tests/conftest.py
import pytest
from sqlalchemy import create_engine
from sqlalchemy.orm import Session

from myapp.models import Base, User, Order

@pytest.fixture(scope="session")
def engine():
    """整个测试会话共享一个 in-memory 数据库"""
    eng = create_engine("sqlite:///:memory:")
    Base.metadata.create_all(eng)
    yield eng
    Base.metadata.drop_all(eng)

@pytest.fixture
def db(engine):
    """每个测试一个新的 session，用事务回滚保证隔离"""
    conn = engine.connect()
    trans = conn.begin()
    session = Session(bind=conn)
    yield session
    session.close()
    trans.rollback()
    conn.close()

@pytest.fixture
def sample_user(db):
    user = User(name="Alice", email="alice@test.com")
    db.add(user)
    db.flush()
    return user
```

现在 `tests/` 下任意一个测试文件，只要在参数里写 `sample_user`，就能直接用。

### 6.4 多层 conftest.py 的作用范围

pytest 会从测试文件所在目录往上**一直向上搜索 conftest.py**，直到项目根。所有找到的 conftest.py 里的 fixture 都可用。内层 conftest.py 可以"覆盖"同名 fixture。

**这意味着**：
- 项目根放全局 fixture（数据库、配置）
- 子模块放模块特定 fixture（mock 特定服务）
- 不会相互污染，作用域清晰

### 6.5 conftest.py 还能放什么

除了 fixture，conftest.py 还能放：

- **Hooks**（pytest 钩子函数）：比如 `pytest_collection_modifyitems` 用来修改测试收集行为。
- **plugin 注册**。
- **全局配置**：比如设置 fixture finalizer 顺序、自动 mark。

初学者只需要知道 fixture 共享这个用法就够了，其他进阶功能用到再查。

> **读到这里你应该理解**：conftest.py 不是"Python 模块"，而是"pytest 配置 + fixture 共享"的约定文件。放进去的东西由 pytest 自动发现，不需要 import。

### 6.6 常见误区

- **在测试文件里 `import conftest`**。错，会导致 fixture 被注册两次或加载异常。
- **多个 conftest.py 里定义同名 fixture 不知道哪个生效**。内层覆盖外层，但这会让代码很难读，应尽量避免同名。
- **把业务代码放 conftest.py**。conftest.py 只该放测试基础设施，不该放被测代码。

## 7. Parametrize 参数化测试

### 7.1 问题

你要测一个 `is_valid_email` 函数。要测的输入有：

- 正常邮箱 `alice@test.com` → True
- 缺 @ `alicetest.com` → False
- 缺域名 `alice@` → False
- 空字符串 `""` → False
- `None` → False

最笨的写法：

```python
def test_valid():
    assert is_valid_email("alice@test.com") is True

def test_missing_at():
    assert is_valid_email("alicetest.com") is False

def test_missing_domain():
    assert is_valid_email("alice@") is False

def test_empty():
    assert is_valid_email("") is False

def test_none():
    assert is_valid_email(None) is False
```

5 个几乎一样的函数。改测试逻辑要改 5 个地方。

### 7.2 pytest.mark.parametrize

pytest 提供 `@pytest.mark.parametrize` 装饰器把多组数据驱动到一个测试函数：

```python
import pytest

@pytest.mark.parametrize("email, expected", [
    ("alice@test.com", True),
    ("alicetest.com", False),
    ("alice@", False),
    ("", False),
    (None, False),
])
def test_is_valid_email(email, expected):
    assert is_valid_email(email) == expected
```

pytest 会把这一个测试函数**展开成 5 个独立的测试**。跑起来你会看到：

```
test_is_valid_email[alice@test.com-True] PASSED
test_is_valid_email[alicetest.com-False] PASSED
test_is_valid_email[alice@-False] PASSED
test_is_valid_email[-False] PASSED
test_is_valid_email[None-False] PASSED
```

每个用例失败时会精确显示是哪组参数失败，定位问题非常容易。

### 7.3 语法细节

`parametrize` 第一个参数是"参数名字符串"，可以多个参数用逗号分隔。

```python
# 单参数
@pytest.mark.parametrize("x", [1, 2, 3])
def test_x(x): ...

# 多参数（用逗号或列表都行）
@pytest.mark.parametrize("a, b, expected", [
    (1, 2, 3),
    (10, 20, 30),
])
def test_add(a, b, expected):
    assert add(a, b) == expected
```

### 7.4 ids：给每个用例起名字

默认用例 id 是 pytest 自动从参数值拼出来的，有时候不好看（比如参数是个大字典）。用 `ids` 自定义：

```python
@pytest.mark.parametrize("user, expected", [
    ({"name": "Alice", "age": 30}, True),
    ({"name": "", "age": 30}, False),
    ({"name": "Bob", "age": -1}, False),
], ids=["正常用户", "空名字", "负数年龄"])
def test_user_valid(user, expected):
    assert validate_user(user) == expected
```

现在测试名变成：

```
test_user_valid[正常用户]
test_user_valid[空名字]
test_user_valid[负数年龄]
```

### 7.5 多个 parametrize 叠加（笛卡尔积）

如果你叠两层 parametrize，pytest 会**自动生成所有组合**：

```python
@pytest.mark.parametrize("x", [1, 2, 3])
@pytest.mark.parametrize("y", ["a", "b"])
def test_combo(x, y):
    print(x, y)
```

这会跑 3 × 2 = 6 个测试：`(1,a) (1,b) (2,a) (2,b) (3,a) (3,b)`。

### 7.6 indirect：把参数传给 fixture

有时候你想让参数化**作用在 fixture 上**而不是测试函数上。用 `indirect=True`：

```python
@pytest.fixture
def user(request):
    return {"name": request.param, "age": 30}

@pytest.mark.parametrize("user", ["Alice", "Bob"], indirect=True)
def test_user_name(user):
    assert user["name"] in ["Alice", "Bob"]
```

这里 `"Alice"` 和 `"Bob"` 被传给 `user` fixture 而不是测试函数。fixture 内部通过 `request.param` 拿到。

**什么时候用 indirect**：当参数需要经过 fixture 做一些复杂处理（比如建数据库记录）才能给测试用时。

### 7.7 和 fixture 的选择

- **简单数据对比**：用 parametrize 直接写死在装饰器里。
- **需要建对象 / 涉及资源**：用 fixture。
- **fixture 本身需要多个参数版本**：用参数化 fixture 或 indirect。

> **读到这里你应该理解**：parametrize 让你用**一份代码覆盖多组输入**。它本质是"写一次测试逻辑，跑 N 次不同数据"。这是 pytest 相比 unittest 最显著的生产力提升之一。

### 7.8 常见误区

- **用 for 循环代替 parametrize**。在一个测试函数里写 `for x in [1,2,3]: assert f(x) == x*2`——错。for 循环里第一个失败就整个测试失败，看不到后面的。parametrize 才是正确做法。
- **参数里放可变对象**。如 `[{"key": "value"}]`，多个用例共享同一个 dict 实例，一个修改会影响其他。写成工厂函数或 `copy.deepcopy`。
- **参数太多，一次跑几千个**。参数化爆炸要靠 mark 分组或 `-k` 过滤。

## 8. Mark 标记与选择性运行

### 8.1 问题

你有 300 个测试。其中 20 个是"慢测试"（调用真实 API、跑数据库迁移），跑一次要 5 分钟。你不希望每次开发都跑这 20 个，只想在提交前跑一次。

解决方案：**给测试打标签（mark），按标签选择性运行**。

### 8.2 定义和使用 mark

```python
import pytest

@pytest.mark.slow
def test_full_import():
    # 慢测试，跑 2 分钟
    ...

@pytest.mark.slow
@pytest.mark.db
def test_migration():
    # 既慢又依赖数据库
    ...

def test_add():
    # 没 mark 的普通测试
    assert 1 + 1 == 2
```

`@pytest.mark.XXX` 会给测试打上 `XXX` 标签。标签名你自己定，常用的有 `slow`、`integration`、`db`、`network`、`gpu`。

### 8.3 按 mark 选择运行

```bash
# 只跑打了 slow 标签的
pytest -m slow

# 跳过慢测试（常见做法）
pytest -m "not slow"

# 同时打了 slow 和 db 标签的
pytest -m "slow and db"

# slow 或 integration
pytest -m "slow or integration"
```

`-m` 后面是一个简单的布尔表达式，支持 `and not or`。

### 8.4 注册自定义 mark

pytest 从 5.0 开始对未注册的 mark 会发警告。在 `pyproject.toml`（推荐）或 `pytest.ini` 里注册：

```toml
# pyproject.toml
[tool.pytest.ini_options]
markers = [
    "slow: 慢测试，CI 里单独跑",
    "db: 需要真实数据库",
    "network: 需要真实网络",
    "llm: 会调用 LLM，花 token",
]
```

```ini
# pytest.ini
[pytest]
markers =
    slow: 慢测试
    db: 需要数据库
```

注册后警告消失，也方便新人看有哪些 mark。

### 8.5 内置 mark

pytest 自带一些 mark：

- `@pytest.mark.skip(reason=...)`：无条件跳过
- `@pytest.mark.skipif(condition, reason=...)`：条件跳过
- `@pytest.mark.xfail(reason=...)`：期望失败（失败=预期，通过=意外）
- `@pytest.mark.parametrize`：前面讲过，参数化

### 8.6 典型 CI 策略

```yaml
# CI 里分两步跑测试
- name: 快测试（每次 push 都跑）
  run: pytest -m "not slow"

- name: 慢测试（只在 main 分支跑）
  if: github.ref == 'refs/heads/main'
  run: pytest -m slow
```

开发分支只跑快测试保证 CI 快；合入主干前跑一次全量。

> **读到这里你应该理解**：mark 是"标签系统"，帮你**把测试分类**。分类后可以选择性运行，大大降低日常开发时的反馈延迟。

### 8.7 常见误区

- **忘了注册 mark 导致一堆警告**。按第 8.4 节注册一下。
- **mark 当成条件判断**。mark 本身不改变测试行为，只是给它打标签。真正的"跳过"需要用 `skip`/`skipif`。
- **把 mark 名写错导致筛选不到**。pytest 不会报错"找不到 mark"，只是不跑任何测试。跑完看到"0 collected"就要怀疑 mark 名拼错。

## 9. Mock 与 Monkeypatch

### 9.1 为什么需要 mock

你在测一个"发送验证码"的函数：

```python
def send_verification_code(phone: str) -> str:
    code = generate_random_code()
    sms_client.send(phone, f"您的验证码是 {code}")
    return code
```

这个函数真实情况下会调短信 API 花钱。你总不能每跑一次测试就发一条真短信吧？而且测试环境也不一定有网。

你需要的是：**在测试的时候，让 `sms_client.send` 什么都不做，只是"假装"发了**。这就是 mock 要解决的问题。

**Mock 的本质**：用"假的替身"替换掉"真的依赖"，让测试变得可控、快、不花钱。

### 9.2 三种 mock 方式

Python 里 mock 主要有三种方式：

1. `unittest.mock`（标准库）—— 最基础、最灵活
2. `monkeypatch` fixture（pytest 内置）—— 简单场景最方便
3. `pytest-mock` 的 `mocker` fixture —— 前两者的混合版

下面一个个讲。

### 9.3 unittest.mock 基础

**Mock 对象**：一个"万能假对象"，你对它做任何操作都不报错，而且会记录所有调用。

```python
from unittest.mock import Mock

m = Mock()
m.foo(1, 2, 3)         # 调用任意方法都不报错
m.bar.baz("hello")     # 任意层级属性都能访问

# 记录了所有调用
print(m.foo.call_count)        # 1
print(m.foo.call_args)         # call(1, 2, 3)
print(m.bar.baz.called)        # True
```

**设置返回值**：

```python
m = Mock()
m.get_user.return_value = {"name": "Alice"}
print(m.get_user(1))  # {'name': 'Alice'}
print(m.get_user(999))  # {'name': 'Alice'}（返回值和参数无关）
```

**设置多次调用返回不同值**：

```python
m.get_user.side_effect = [
    {"name": "Alice"},
    {"name": "Bob"},
    {"name": "Charlie"},
]
print(m.get_user(1))  # Alice
print(m.get_user(1))  # Bob
print(m.get_user(1))  # Charlie
```

**让 mock 抛异常**：

```python
m.get_user.side_effect = ValueError("用户不存在")
m.get_user(1)  # 抛 ValueError
```

### 9.4 patch：替换真实对象

光造假对象没用，还要"**把假对象塞到被测代码里**"。这就是 `patch`。

```python
# myapp/user.py
from myapp import sms_client

def send_verification_code(phone):
    code = "123456"
    sms_client.send(phone, code)
    return code
```

```python
# test_user.py
from unittest.mock import patch
from myapp.user import send_verification_code

def test_send_verification_code():
    with patch("myapp.user.sms_client") as mock_sms:
        code = send_verification_code("13800000000")
        mock_sms.send.assert_called_once_with("13800000000", "123456")
        assert code == "123456"
```

解释：
- `patch("myapp.user.sms_client")`：**在 `myapp.user` 这个模块里**，把 `sms_client` 暂时替换成一个 Mock 对象。
- `with` 块内用的是 mock；`with` 块退出后自动恢复。
- `mock_sms.send.assert_called_once_with(...)` 断言 send 方法被调用过且参数正确。

**最容易踩坑的点：patch 的路径**。这是 Python mock 初学者最大的困惑，必须讲清楚。

### 9.5 patch 路径的黄金法则

**Patch where it's used, not where it's defined.**
**在"使用它"的地方打补丁，而不是"定义它"的地方。**

举例：

```python
# mymodule/api.py
import requests

def fetch_user(uid):
    return requests.get(f"/users/{uid}").json()
```

测试时你要 mock 掉 `requests.get`。**错误的写法**：

```python
with patch("requests.get") as mock:  # ❌
    ...
```

**正确的写法**：

```python
with patch("mymodule.api.requests.get") as mock:  # ✅
    ...
```

**为什么**：`from module import X` 会在当前模块创建一个**对 X 的本地引用**。你在原模块替换 X 并不影响本地已经复制过的引用。必须替换本地引用才有效。

这是 Python mock 最常见的坑。**规则一句话**：**patch 的路径应该是"被测代码里写的那个名字的完整路径"**。

再举一个例子：

```python
# mymodule/service.py
from mymodule.db import get_connection

def list_users():
    conn = get_connection()
    return conn.query("SELECT * FROM users")
```

测试要 mock `get_connection`：

```python
with patch("mymodule.service.get_connection") as mock_conn:  # ✅
    mock_conn.return_value.query.return_value = [{"id": 1}]
    assert list_users() == [{"id": 1}]
```

注意是 `mymodule.service.get_connection`，不是 `mymodule.db.get_connection`——因为是在 `service.py` 里"用"的它。

### 9.6 patch 的三种用法

**1. 装饰器方式**（应用到整个测试函数）：

```python
from unittest.mock import patch

@patch("myapp.user.sms_client")
def test_send(mock_sms):
    send_verification_code("13800000000")
    mock_sms.send.assert_called_once()
```

注意装饰器方式里，mock 对象会**作为参数注入**测试函数。

**2. 上下文管理器方式**（只 patch 一段代码）：

```python
def test_send():
    with patch("myapp.user.sms_client") as mock_sms:
        send_verification_code("13800000000")
        mock_sms.send.assert_called_once()
```

**3. patch.object**（patch 一个对象的属性而不是模块路径）：

```python
import myapp.user

def test_send():
    with patch.object(myapp.user, "sms_client") as mock_sms:
        send_verification_code("13800000000")
        mock_sms.send.assert_called_once()
```

三种方式效果一样。日常用前两种就够。

### 9.7 MagicMock 和普通 Mock 的区别

`Mock` 是基础版，`MagicMock` 多了对 Python"魔术方法"的支持（`__len__`、`__iter__`、`__getitem__` 等）。`patch()` 默认创建的就是 MagicMock，所以大部分情况你不用手动区分。

```python
from unittest.mock import MagicMock

m = MagicMock()
len(m)        # 返回 0，不报错
list(m)       # 返回 []，不报错
m[0]          # 返回新的 MagicMock，不报错
```

### 9.8 monkeypatch fixture

pytest 内置了一个 `monkeypatch` fixture，用法比 patch 更简单：

```python
def test_send(monkeypatch):
    mock_send = Mock()
    monkeypatch.setattr("myapp.user.sms_client.send", mock_send)

    send_verification_code("13800000000")
    mock_send.assert_called_once()
```

`monkeypatch.setattr(target, value)` 的意思是：**把 target 暂时替换成 value，测试结束后自动还原**。

**monkeypatch 常见方法**：

```python
monkeypatch.setattr("module.func", new_func)     # 替换属性
monkeypatch.setenv("API_KEY", "fake-key")         # 设置环境变量
monkeypatch.delenv("PROD_MODE", raising=False)    # 删环境变量
monkeypatch.setitem(my_dict, "key", "value")      # 修改字典
monkeypatch.chdir("/tmp")                         # 改工作目录
```

**monkeypatch vs patch**：
- 都能替换，都自动还原。
- monkeypatch 是 pytest 风格（fixture），patch 是 unittest 风格（装饰器/上下文）。
- monkeypatch 更适合"临时改个属性/环境变量"的简单场景。
- patch 更适合"替换成 Mock 对象做调用断言"的复杂场景。
- 两者可以混用。

### 9.9 pytest-mock 的 mocker fixture

`pytest-mock` 是第三方包，提供一个 `mocker` fixture，它是 `unittest.mock` 的 pytest 包装版：

```bash
pip install pytest-mock
```

```python
def test_send(mocker):
    mock_sms = mocker.patch("myapp.user.sms_client")
    send_verification_code("13800000000")
    mock_sms.send.assert_called_once()
```

对比原生 `patch`：不用写 `with`、不用写装饰器，测试结束自动恢复。这是目前最推荐的 mock 方式。

### 9.10 什么时候 mock、什么时候不 mock

**应该 mock**：
- 外部 API（HTTP 调用、第三方 SDK）
- 数据库（除非你做集成测试）
- 文件系统（除非测文件操作本身）
- 时间函数（`datetime.now()` 在测"过期"时会抖）
- 随机数（`random`、`uuid`）

**不应该 mock**：
- 你自己的纯函数（直接调就行，mock 失去意义）
- 数据类（Pydantic 模型、dataclass）
- 标准库的 list、dict、str（`[1,2,3]` 不如直接造一个真的）

**过度 mock 的症状**：
- 测试代码比业务代码还长。
- 每次改业务代码都要改一堆 mock。
- 测试全过但上线就挂（因为 mock 的行为和真实行为不一致）。

> **读到这里你应该理解**：mock 是"测试的隔离术"，把难控的东西（网络、时间、外部服务）替换成可控的假对象。核心三点：**mock 的路径**（where it's used）、**return_value vs side_effect**、**断言调用（assert_called_with）**。

### 9.11 常见误区

- **patch 路径搞错**（前面讲过的头号坑）。
- **`return_value` 和 `side_effect` 混淆**。return_value 是固定返回，side_effect 是每次调用返回不同值（或抛异常）。
- **断言方法名错**：`assert_called_once()`、`assert_called_with(args)`、`assert_called_once_with(args)`——记住有下划线。更坑的是，**写错方法名不会报错**（Mock 对象的任意属性都合法），测试会"假通过"。解决方法是用 `spec` 或 `autospec`。
- **mock 了之后没断言**。只 mock 不断言等于没测。至少要断言"这个依赖被正确调用了"。

## 10. 测试异步代码

### 10.1 async 函数不能直接测

看一个简单的 async 函数：

```python
async def fetch_data():
    return {"status": "ok"}
```

你以为这样就能测：

```python
def test_fetch_data():  # ❌
    result = fetch_data()
    assert result == {"status": "ok"}
```

实际上 `fetch_data()` 返回的不是 `{"status": "ok"}`，而是一个 **coroutine 对象**。你需要 `await` 它才能拿到值，但 `await` 只能在 async 函数里用。

### 10.2 pytest-asyncio

这就是 `pytest-asyncio` 插件要解决的问题。安装：

```bash
pip install pytest-asyncio
```

然后在测试函数前加 `@pytest.mark.asyncio`：

```python
import pytest

@pytest.mark.asyncio
async def test_fetch_data():
    result = await fetch_data()
    assert result == {"status": "ok"}
```

这样 pytest 会自动启动一个 event loop 去跑这个 async 测试函数。

### 10.3 自动模式（推荐）

每个 async 测试都加 `@pytest.mark.asyncio` 很啰嗦。在 `pyproject.toml` 里设置**自动模式**：

```toml
[tool.pytest.ini_options]
asyncio_mode = "auto"
```

或者 `pytest.ini`：

```ini
[pytest]
asyncio_mode = auto
```

之后所有 `async def test_*` 都会被当作 async 测试运行，不用加 mark。

### 10.4 async fixture

fixture 也可以是 async 的：

```python
import pytest

@pytest.fixture
async def async_client():
    client = AsyncClient()
    await client.connect()
    yield client
    await client.close()

async def test_with_client(async_client):
    result = await async_client.get("/users/1")
    assert result.status == 200
```

自动模式下直接能用；非自动模式需要 `@pytest.fixture` 后面加 `@pytest.mark.asyncio`（或用 `@pytest_asyncio.fixture`）。

### 10.5 event_loop fixture

pytest-asyncio 提供一个 `event_loop` fixture。默认每个测试一个新 loop（function 作用域）。如果你希望整个 session 共享一个 loop（比如测试里用了全局异步资源），可以：

```python
# conftest.py
import asyncio
import pytest

@pytest.fixture(scope="session")
def event_loop():
    loop = asyncio.new_event_loop()
    yield loop
    loop.close()
```

**注意**：共享 event_loop 有很多坑。新版本 pytest-asyncio 推荐用 `asyncio_default_fixture_loop_scope` 配置代替自定义 event_loop fixture。查官方文档确认你版本的用法。

### 10.6 测试异步代码的核心原则

- async 测试函数要 `await` 所有异步调用。
- async fixture 要和 async 测试配合。
- 不要混用 sync 和 async 资源——容易死锁。
- 用 `anyio` 的库和 `asyncio` 的库区分清楚（FastAPI 底层是 `anyio`，有些 fixture 需要 `anyio_backend`）。

> **读到这里你应该理解**：异步测试的复杂点主要在"event loop 谁来管"。pytest-asyncio 把 event loop 藏在 fixture 里自动管理。你只需要记住：**async 函数要 await，并且用 pytest-asyncio 跑**。

### 10.7 常见误区

- **忘了 `await`**。拿到 coroutine 对象做断言，永远失败或"通过"（实际没执行）。
- **同步 fixture 返回异步资源**。比如普通 fixture 里创建 `asyncpg` 连接——要用 async fixture。
- **多个 session 作用域 async fixture 用不同 loop**。会出现"attached to different event loop"错误。通常要统一 loop scope。

## 11. 测试 FastAPI

### 11.1 TestClient：不启动 server 就能测

FastAPI 基于 Starlette，自带一个 `TestClient`。它的神奇之处是：**不需要真的启动一个 HTTP server**，就可以对你的 app 发"虚拟 HTTP 请求"。

```python
from fastapi import FastAPI
from fastapi.testclient import TestClient

app = FastAPI()

@app.get("/hello")
def hello():
    return {"msg": "hello"}

client = TestClient(app)

def test_hello():
    response = client.get("/hello")
    assert response.status_code == 200
    assert response.json() == {"msg": "hello"}
```

**原理**：TestClient 把你的 `app`（一个 ASGI 应用）直接当作函数调用，请求/响应都在进程内传递。速度极快，不占端口。

### 11.2 发各种请求

```python
# GET 带 query
client.get("/users", params={"page": 2})

# POST 带 JSON body
client.post("/users", json={"name": "Alice"})

# POST 带表单
client.post("/login", data={"username": "alice", "password": "xxx"})

# 带 header
client.get("/me", headers={"Authorization": "Bearer xxx"})

# 上传文件
with open("avatar.png", "rb") as f:
    client.post("/upload", files={"file": ("avatar.png", f, "image/png")})
```

### 11.3 检查响应

```python
response = client.get("/users/1")

assert response.status_code == 200
assert response.headers["content-type"] == "application/json"

data = response.json()
assert data["id"] == 1
assert "name" in data
```

### 11.4 依赖覆盖（Dependency Override）

FastAPI 的核心机制是 `Depends()`。测试时你经常需要**替换某个依赖**——比如让 `get_current_user` 返回一个假用户而不是真的去查 JWT。

```python
# myapp/deps.py
def get_current_user(token: str = Header(...)):
    # 真实逻辑：解析 JWT，查数据库
    return verify_token(token)

# myapp/main.py
@app.get("/me")
def me(user: User = Depends(get_current_user)):
    return user
```

测试里覆盖：

```python
from myapp.main import app
from myapp.deps import get_current_user

def fake_user():
    return {"id": 1, "name": "TestUser"}

app.dependency_overrides[get_current_user] = fake_user

def test_me():
    response = client.get("/me")
    assert response.json()["name"] == "TestUser"

# 测完清理（否则会影响其他测试）
app.dependency_overrides.clear()
```

更好的做法是用 fixture 包一层：

```python
@pytest.fixture
def client_with_user():
    app.dependency_overrides[get_current_user] = lambda: {"id": 1, "name": "TestUser"}
    yield TestClient(app)
    app.dependency_overrides.clear()

def test_me(client_with_user):
    response = client_with_user.get("/me")
    assert response.status_code == 200
```

### 11.5 结合数据库 fixture

完整例子：测试一个"创建订单"接口。

```python
# conftest.py
@pytest.fixture
def db():
    engine = create_engine("sqlite:///:memory:")
    Base.metadata.create_all(engine)
    session = Session(bind=engine)
    yield session
    session.close()

@pytest.fixture
def client(db):
    def override_get_db():
        yield db
    app.dependency_overrides[get_db] = override_get_db
    yield TestClient(app)
    app.dependency_overrides.clear()

# test_orders.py
def test_create_order(client, db):
    # 先建个用户
    user = User(name="Alice")
    db.add(user)
    db.commit()

    # 创建订单
    response = client.post("/orders", json={"user_id": user.id, "amount": 100})

    assert response.status_code == 201
    assert response.json()["amount"] == 100

    # 验证数据库
    order = db.query(Order).first()
    assert order.user_id == user.id
```

### 11.6 测试异步 endpoint：AsyncClient

`TestClient` 是同步的。如果你的 FastAPI 应用里有需要真实异步行为的测试（比如测 WebSocket 或 SSE 流式响应），用 `httpx.AsyncClient`：

```python
import pytest
from httpx import AsyncClient, ASGITransport

@pytest.mark.asyncio
async def test_async():
    transport = ASGITransport(app=app)
    async with AsyncClient(transport=transport, base_url="http://test") as ac:
        response = await ac.get("/hello")
    assert response.status_code == 200
```

### 11.7 测试 WebSocket

```python
def test_websocket():
    with client.websocket_connect("/ws") as websocket:
        websocket.send_json({"msg": "hi"})
        data = websocket.receive_json()
        assert data == {"msg": "hi back"}
```

> **读到这里你应该理解**：测 FastAPI 的核心是 `TestClient` + `dependency_overrides`。TestClient 让你不启 server 就能测；dependency_overrides 让你替换任意依赖（数据库、认证、外部服务）。

### 11.8 常见误区

- **TestClient 在模块顶层创建了但 app.dependency_overrides 没清理**，测试之间相互污染。
- **测试异步 endpoint 用了同步 TestClient 不过**（新版 FastAPI 的 TestClient 能自动桥接，但老版本不行）。
- **没 mock 外部调用**。测试里真的调用了 OpenAI API，跑一次测试花几块钱。

## 12. 测试数据库操作

### 12.1 核心难题：测试之间不能相互污染

假设 `test_create_user` 往数据库插了一个用户，`test_list_users` 期望数据库里只有它自己建的用户。如果这俩跑在一起就完蛋——`list_users` 会看到上一个测试留下的数据。

有三种解决策略。

### 12.2 策略一：每个测试前清表

```python
@pytest.fixture(autouse=True)
def clean_db(db):
    db.query(User).delete()
    db.query(Order).delete()
    db.commit()
    yield
```

**优点**：简单直接。
**缺点**：表多的时候清表代码一大堆；DELETE 还会被外键约束卡；速度慢。

### 12.3 策略二：事务回滚（推荐）

每个测试开一个事务，测完**不 commit 而是 rollback**：

```python
@pytest.fixture
def db(engine):
    connection = engine.connect()
    trans = connection.begin()
    session = Session(bind=connection)

    yield session

    session.close()
    trans.rollback()
    connection.close()
```

**原理**：事务未 commit 的所有改动都会被 rollback 时丢弃，数据库恢复原状。速度极快（不需要真的删数据），隔离性好。

**坑**：如果被测代码内部自己 `session.commit()` 了，事务就结束了，回滚无效。需要用 SQLAlchemy 的 **SAVEPOINT（嵌套事务）** 技巧：

```python
@pytest.fixture
def db(engine):
    connection = engine.connect()
    trans = connection.begin()
    session = Session(bind=connection, join_transaction_mode="create_savepoint")

    yield session

    session.close()
    trans.rollback()
    connection.close()
```

这样被测代码 `commit()` 实际只是"释放 savepoint"，外层事务还能回滚。

### 12.4 策略三：每个测试一个新数据库

```python
@pytest.fixture
def db():
    import uuid
    db_name = f"test_{uuid.uuid4().hex}"
    create_database(db_name)
    engine = create_engine(f"postgresql://.../{db_name}")
    yield engine
    drop_database(db_name)
```

**优点**：完全隔离，不怕任何污染。
**缺点**：慢得令人发指。只适合集成测试里的关键场景。

### 12.5 测试数据库用 SQLite 还是真 PostgreSQL

**SQLite 方案**：
- 优点：零配置，in-memory 极快。
- 缺点：SQL 方言不一致。PG 的 JSONB、数组类型、窗口函数、CTE 在 SQLite 里语法或不支持或行为不同。你本地测试全过，上线挂了。

**真 PostgreSQL 方案**：
- 优点：和生产完全一致，不会遇到方言问题。
- 缺点：需要启动 PG 实例（通常 Docker 跑起来），CI 配置多一步。

**结论**：对 Agent 后端这种重度使用 PG 特性的项目，**一定要用真 PostgreSQL 测试**。SQLite 只适合"纯 CRUD、没用高级特性"的小项目。

CI 里用 Docker 起一个 PG：

```yaml
services:
  postgres:
    image: postgres:16
    env:
      POSTGRES_PASSWORD: test
    ports:
      - 5432:5432
```

### 12.6 工厂模式创建测试数据

手写 `User(name="Alice", email="alice@test.com", age=30, ...)` 很啰嗦。用 **factory_boy** 或 **factory-boy 的替代品 polyfactory**：

```python
import factory
from myapp.models import User

class UserFactory(factory.Factory):
    class Meta:
        model = User

    name = factory.Faker("name")
    email = factory.Faker("email")
    age = factory.Faker("random_int", min=18, max=80)
```

测试里：

```python
def test_user(db):
    user = UserFactory(name="Alice")
    db.add(user)
    db.flush()
    assert user.name == "Alice"
```

不关心的字段会自动生成假数据。维护成本大大降低。

> **读到这里你应该理解**：数据库测试的核心是"**如何保证测试之间隔离**"。事务回滚是最常用的方案，SQLite vs PG 的选择取决于你项目是否用了 PG 专属特性。

### 12.7 常见误区

- **测试里不清数据**。跑第二次全挂，因为有重复的主键/唯一约束。
- **用 SQLite 但生产用 PG**。某些 SQL 在 SQLite 通过，在 PG 报错。
- **数据库 fixture 作用域太大 + 没清理**。session 作用域 + 测试里真 commit → 所有测试共享脏数据。
- **外键约束导致清表失败**。删表要按依赖反向顺序，或者用 `CASCADE`。

## 13. 覆盖率 Coverage

### 13.1 什么是覆盖率

**代码覆盖率（Code Coverage）**：跑测试时，你的业务代码有多大比例被"执行到"了。100% 覆盖率=每一行都被某个测试执行过至少一次。

覆盖率是衡量"测试是否全面"的一个**粗略**指标。

### 13.2 覆盖率的类型

| 类型 | 含义 |
|------|------|
| 行覆盖（Line Coverage） | 每一行是否被执行 |
| 分支覆盖（Branch Coverage） | 每个 if 的 True/False 分支是否都走过 |
| 语句覆盖（Statement Coverage） | 每个语句是否被执行（和行覆盖类似） |
| 条件覆盖（Condition Coverage） | 每个布尔子表达式是否都取过 True/False |

**行覆盖**是最常用的，但有盲区。看例子：

```python
def login(user, password):
    if user and password:
        return do_login(user, password)
    return None
```

如果你的测试只传 `("alice", "123")`，行覆盖 100%（每行都执行了），但**分支覆盖只有 50%**（`if False` 的分支从没走过）。

**分支覆盖更严格，更能发现问题**。推荐开启。

### 13.3 coverage.py 安装和使用

```bash
pip install coverage

# 跑测试同时统计覆盖率
coverage run -m pytest

# 看报告
coverage report

# 生成 HTML 报告
coverage html
# 打开 htmlcov/index.html
```

### 13.4 pytest-cov：更方便的集成

```bash
pip install pytest-cov
```

```bash
# 跑测试 + 覆盖率一起
pytest --cov=myapp

# 同时生成终端和 HTML 报告
pytest --cov=myapp --cov-report=term --cov-report=html

# 开启分支覆盖
pytest --cov=myapp --cov-branch

# 设置最低覆盖率，低于就失败
pytest --cov=myapp --cov-fail-under=80
```

### 13.5 .coveragerc 配置

项目根放一个 `.coveragerc`（或在 `pyproject.toml` 里写 `[tool.coverage]`）：

```ini
[run]
source = myapp
branch = True
omit =
    */migrations/*
    */tests/*
    */__init__.py

[report]
exclude_lines =
    pragma: no cover
    raise NotImplementedError
    if __name__ == .__main__.:
    if TYPE_CHECKING:
show_missing = True
skip_covered = False
fail_under = 80
```

解释：
- `source`：统计哪些目录的覆盖率。
- `branch`：开启分支覆盖。
- `omit`：排除某些文件（迁移、测试代码本身、初始化文件）。
- `exclude_lines`：排除某些行（比如 `if __name__ == "__main__":` 这种测试里不会跑的代码）。
- `fail_under`：覆盖率低于 80% 就 fail。

### 13.6 排除单行

在某一行后面加注释 `# pragma: no cover`：

```python
def debug_only():  # pragma: no cover
    print("debug")
```

这一行/这个函数不会计入覆盖率。

### 13.7 覆盖率陷阱

**陷阱一：覆盖率高不等于测试好**。一个测试函数只要"跑过"被测代码就能提高覆盖率，哪怕完全没断言。你可以写：

```python
def test_fake():
    login("a", "b")  # 100% 覆盖，0% 断言
```

覆盖率 100% 但完全没验证行为。所以覆盖率只是**必要条件**不是**充分条件**。

**陷阱二：追求 100% 的边际成本极高**。最后 10%-20% 往往是：
- 极端异常分支（几乎不会发生但为了健壮性留的）
- 外部 IO 错误处理（要 mock 一堆才能测到）
- 配置驱动的代码分支

投入产出比不划算。**80% 是实践中的合理目标**，关键代码路径追求 95%+，边缘代码 50% 也行。

**陷阱三：覆盖率没覆盖的不一定是坏代码**。比如日志打印、错误兜底，没测到不代表有 bug。

> **读到这里你应该理解**：覆盖率是"测试全面度"的一个可量化指标，但不等于测试质量。用它发现"完全没测的代码"很有效，但不要当作唯一目标。

### 13.8 常见误区

- **把 100% 覆盖率当目标**。精力成本爆炸。
- **只看总体覆盖率，不看哪些文件没测**。用 HTML 报告按文件看。
- **测试代码本身算进覆盖率**。在 `.coveragerc` 里 omit 掉 `tests/`。
- **没开分支覆盖率**。单开行覆盖容易漏分支。

## 14. CI / CD 概念

### 14.1 CI：Continuous Integration（持续集成）

**定义**：开发者频繁地把代码合并到主干（一天多次），每次合并都自动触发构建和测试，**尽早发现集成问题**。

**没 CI 的时代**：开发者各自改代码，一个月合一次，合的时候冲突一大堆、bug 一大堆，叫"合并地狱（integration hell）"。

**有 CI 后**：每次 push 都有一套自动化流水线跑：
1. 拉代码
2. 安装依赖
3. 静态检查（lint、type check）
4. 跑测试
5. 出报告

任何一步失败，开发者马上收到通知。主干永远保持"可用"状态。

**核心口号**：**主干永远是绿的（Green）。**

### 14.2 CD：Delivery 还是 Deployment？

"CD" 有两个含义，容易混淆：

**Continuous Delivery（持续交付）**：CI 通过后，自动把构建产物（Docker 镜像、二进制包）**准备好**，随时可以部署。但**最后上线由人点按钮**。适合需要合规、审核的环境。

**Continuous Deployment（持续部署）**：再进一步，CI 通过后**自动部署到生产**。适合 SaaS、内部工具这种高速迭代的场景。

一句话区分：
- Delivery = "随时能部署"（人决定何时部）
- Deployment = "一路部到底"（机器自己部）

### 14.3 CI/CD 的完整流水线

一个典型的流水线长这样：

```
开发者 push
  ↓
GitHub 触发 workflow
  ↓
Job 1: Lint & Type Check（ruff, mypy）
  ↓
Job 2: Unit Tests
  ↓
Job 3: Integration Tests（起 Docker 容器 + 数据库）
  ↓
Job 4: Build Docker Image
  ↓
Job 5: Push to Registry
  ↓
Job 6: Deploy to Staging（自动）
  ↓
Job 7: E2E Tests on Staging
  ↓
Job 8: Deploy to Production（手动按钮 or 自动）
```

每一步是一个 job，前面失败后面就不执行。

### 14.4 为什么对开发者重要

- **早期发现 bug**：push 后 5 分钟就知道挂了，不用等测试同事手动找。
- **主干永远可用**：任何时候 checkout 主干都是能跑的版本。
- **降低合并恐惧**：小步合并，冲突小；不再需要"feature branch 三个月不合并"的灾难。
- **团队协作**：CI 失败的 PR 不让合，强制每个人对质量负责。

> **读到这里你应该理解**：CI 是"自动化质量关卡"，CD 是"自动化发布流水线"。它们不是工具，是开发流程。没 CI 的团队交付速度是跑不起来的。

## 15. GitHub Actions 详解

### 15.1 为什么选 GitHub Actions

市面上 CI 工具很多：Jenkins、GitLab CI、CircleCI、Travis、Buildkite……但如果你的代码在 GitHub 上，**GitHub Actions（GHA）** 是零配置的默认选择：

- 内置在 GitHub，不需要另外注册账号。
- Public 仓库免费；Private 仓库每月 2000 分钟免费额度。
- 生态丰富，社区 action 海量。
- 语法统一为 YAML。

### 15.2 核心概念

| 概念 | 含义 |
|------|------|
| Workflow | 一个 YAML 文件定义的自动化流程。放在 `.github/workflows/*.yml` |
| Event | 触发 workflow 的事件（push、pull_request、定时、手动） |
| Job | 一组在同一台机器上运行的 step |
| Step | job 里的一步，可以是 shell 命令或一个 action |
| Action | 预制好的可复用步骤（比如 checkout 代码、安装 Python） |
| Runner | 执行 workflow 的机器，有 GitHub 托管的和自建的 |

### 15.3 最小示例

在项目根建 `.github/workflows/ci.yml`：

```yaml
name: CI

on: [push, pull_request]

jobs:
  test:
    runs-on: ubuntu-latest
    steps:
      - uses: actions/checkout@v4

      - uses: actions/setup-python@v5
        with:
          python-version: "3.11"

      - run: pip install -r requirements.txt

      - run: pytest
```

解释每一行：

- `name: CI` — workflow 的显示名。
- `on: [push, pull_request]` — 在 push 或 PR 时触发。
- `jobs: test:` — 定义一个叫 `test` 的 job。
- `runs-on: ubuntu-latest` — 在最新版 Ubuntu 虚拟机上跑。
- `steps:` — 步骤列表。
  - `uses: actions/checkout@v4` — 使用官方的 checkout action，把代码拉下来。
  - `uses: actions/setup-python@v5` — 装 Python 3.11。
  - `run: pip install ...` — 执行 shell 命令。
  - `run: pytest` — 跑测试。

push 到 GitHub 后，仓库的 Actions 标签页就能看到跑的情况。

### 15.4 Triggers 详解

```yaml
# 只在 push 到 main 分支时
on:
  push:
    branches: [main]

# PR 时
on:
  pull_request:
    branches: [main]
    types: [opened, synchronize, reopened]

# 定时（cron 语法）
on:
  schedule:
    - cron: "0 2 * * *"   # 每天 UTC 02:00

# 手动触发
on:
  workflow_dispatch:
    inputs:
      version:
        description: "版本号"
        required: true

# 打 tag 时
on:
  push:
    tags:
      - "v*"

# 多个触发
on: [push, pull_request, workflow_dispatch]
```

### 15.5 Jobs、依赖、并行

多个 job 默认**并行运行**。想让 job 依赖另一个用 `needs`：

```yaml
jobs:
  lint:
    runs-on: ubuntu-latest
    steps: [...]

  test:
    runs-on: ubuntu-latest
    needs: lint   # lint 通过后才跑 test
    steps: [...]

  deploy:
    runs-on: ubuntu-latest
    needs: [lint, test]   # 等两个都通过
    steps: [...]
```

### 15.6 Matrix 构建：一次测多个版本

想同时测 Python 3.10/3.11/3.12？用 matrix：

```yaml
jobs:
  test:
    strategy:
      matrix:
        python-version: ["3.10", "3.11", "3.12"]
        os: [ubuntu-latest, macos-latest, windows-latest]
    runs-on: ${{ matrix.os }}
    steps:
      - uses: actions/checkout@v4
      - uses: actions/setup-python@v5
        with:
          python-version: ${{ matrix.python-version }}
      - run: pip install -r requirements.txt
      - run: pytest
```

这会跑 3 × 3 = 9 个 job。任何一个失败 PR 就不让合。

### 15.7 Secrets：存敏感信息

API key、数据库密码不能写进代码。GitHub 仓库设置里的 **Settings → Secrets → Actions** 可以添加 secret，workflow 里用 `${{ secrets.XXX }}` 引用：

```yaml
- run: pytest
  env:
    OPENAI_API_KEY: ${{ secrets.OPENAI_API_KEY }}
    DATABASE_URL: ${{ secrets.DATABASE_URL }}
```

**注意**：
- secrets 只在 GitHub 内部可见，日志里会被自动 mask 成 `***`。
- **PR from fork 默认拿不到 secrets**（安全考虑），跑不了需要真密钥的测试。
- 不要在 `echo` 之类的命令里打印 secret——会被日志记录。

### 15.8 Services：起配套服务

要用真 PostgreSQL 测试？用 services 声明一个：

```yaml
jobs:
  test:
    runs-on: ubuntu-latest

    services:
      postgres:
        image: postgres:16
        env:
          POSTGRES_PASSWORD: test
          POSTGRES_DB: testdb
        ports:
          - 5432:5432
        options: >-
          --health-cmd pg_isready
          --health-interval 10s
          --health-timeout 5s
          --health-retries 5

    steps:
      - uses: actions/checkout@v4
      - uses: actions/setup-python@v5
        with:
          python-version: "3.11"
      - run: pip install -r requirements.txt
      - run: pytest
        env:
          DATABASE_URL: postgresql://postgres:test@localhost:5432/testdb
```

GHA 会在 job 开始前起一个 PG 容器，`localhost:5432` 就能连。

### 15.9 缓存依赖

每次 workflow 从零装依赖很慢。用 `actions/cache` 缓存：

```yaml
- uses: actions/setup-python@v5
  with:
    python-version: "3.11"
    cache: "pip"              # 自动缓存 pip
    cache-dependency-path: requirements.txt
```

或手动缓存：

```yaml
- uses: actions/cache@v4
  with:
    path: ~/.cache/pip
    key: ${{ runner.os }}-pip-${{ hashFiles('requirements.txt') }}
```

### 15.10 工作流编排实战：测试 + 覆盖率 + 部署

```yaml
name: CI/CD

on:
  push:
    branches: [main]
  pull_request:

jobs:
  lint:
    runs-on: ubuntu-latest
    steps:
      - uses: actions/checkout@v4
      - uses: actions/setup-python@v5
        with:
          python-version: "3.11"
          cache: pip
      - run: pip install ruff mypy
      - run: ruff check .
      - run: mypy myapp

  test:
    runs-on: ubuntu-latest
    needs: lint
    services:
      postgres:
        image: postgres:16
        env:
          POSTGRES_PASSWORD: test
        ports: [5432:5432]
        options: --health-cmd pg_isready --health-interval 10s
    steps:
      - uses: actions/checkout@v4
      - uses: actions/setup-python@v5
        with:
          python-version: "3.11"
          cache: pip
      - run: pip install -r requirements.txt
      - run: pytest --cov=myapp --cov-report=xml --cov-fail-under=80
        env:
          DATABASE_URL: postgresql://postgres:test@localhost/postgres
      - uses: codecov/codecov-action@v4
        with:
          file: ./coverage.xml

  build:
    runs-on: ubuntu-latest
    needs: test
    if: github.ref == 'refs/heads/main'
    steps:
      - uses: actions/checkout@v4
      - uses: docker/build-push-action@v5
        with:
          push: true
          tags: myapp:${{ github.sha }}
        env:
          DOCKER_USERNAME: ${{ secrets.DOCKER_USERNAME }}
          DOCKER_PASSWORD: ${{ secrets.DOCKER_PASSWORD }}
```

三阶段：lint → test → build。test 里起了真 PG，跑完发覆盖率到 Codecov。build 只在 main 分支跑。

> **读到这里你应该理解**：GHA 是"YAML 写的自动化脚本"。核心是 **trigger + job + step + action**。你日常 80% 只用到 checkout、setup-python、跑命令这几件事。

### 15.11 常见误区

- **写完 workflow 不 push 分支看结果**。必须 push 才能触发。
- **secret 直接 echo 出来**。会被日志 mask 但仍不推荐。
- **matrix 跑太多组合**。8 个 Python 版本 × 4 个 OS = 32 个 job，免费额度很快烧完。
- **没设置 timeout**。一个 flaky 测试挂住 job 能跑到平台上限（6 小时）。加 `timeout-minutes`。

## 16. 本地 CI：pre-commit

### 16.1 问题

CI 是"push 之后"才跑的。如果你每次 push 后发现"lint 没过"、"单测挂了"再改再 push，反馈循环很慢。

解决方案：**在本地提交代码之前**就跑一遍检查。这就是 **git hooks**，而 `pre-commit` 是管理 git hooks 的标准工具。

### 16.2 pre-commit 简介

`pre-commit` 是一个 Python 写的工具（但能管任意语言的 hook）。安装：

```bash
pip install pre-commit
```

在项目根建 `.pre-commit-config.yaml`：

```yaml
repos:
  - repo: https://github.com/astral-sh/ruff-pre-commit
    rev: v0.5.0
    hooks:
      - id: ruff
        args: [--fix]
      - id: ruff-format

  - repo: https://github.com/pre-commit/pre-commit-hooks
    rev: v4.6.0
    hooks:
      - id: trailing-whitespace
      - id: end-of-file-fixer
      - id: check-yaml
      - id: check-added-large-files

  - repo: local
    hooks:
      - id: pytest-fast
        name: pytest (fast tests only)
        entry: pytest -m "not slow" -x
        language: system
        pass_filenames: false
```

然后安装 hook：

```bash
pre-commit install
```

现在每次 `git commit` 都会自动跑这些检查。任何一步失败就不让你 commit。

### 16.3 常用 hook

- `ruff` — Python lint + format
- `black` / `ruff-format` — 代码格式化
- `mypy` — 类型检查
- `isort` — import 排序
- `check-yaml` / `check-json` — 语法检查
- `check-added-large-files` — 禁止提交大文件
- `detect-private-key` — 防止提交密钥
- 自定义：跑自己的 pytest

### 16.4 pre-commit vs CI

- **pre-commit**：本地快速反馈，拦截常见问题。可以跳过（`git commit --no-verify`）。
- **CI**：最终关卡，不能跳过，必须通过才能合入。

两者互补。pre-commit 抓笔误、格式问题、明显的 lint；CI 抓完整测试、覆盖率、集成问题。

### 16.5 首次运行

```bash
# 对所有文件跑一遍（首次配置建议）
pre-commit run --all-files

# 只对 staged 文件跑（git commit 时自动执行的）
pre-commit run
```

> **读到这里你应该理解**：pre-commit 是"提交前最后一道检查"。和 CI 互补：pre-commit 让你本地快速发现问题，CI 强制所有人都通过。

## 17. 常见坑与反模式

### 17.1 Flaky Tests（抖动测试）

**现象**：同一个测试，同一份代码，**有时过有时不过**。这是最让人崩溃的测试类型。

**常见原因**：

1. **依赖时间**。比如测试里 `assert datetime.now().day == 10`，跨零点就挂。
2. **依赖随机**。没固定随机种子，偶尔会生成"恰好冲突"的数据。
3. **依赖网络**。真调外部 API，网抖就挂。
4. **异步顺序**。多线程/协程测试没控好顺序，靠 `sleep(0.1)` 碰运气。
5. **共享全局状态**。测试之间靠 class var 或模块级变量通信。
6. **并发数据库**。多个测试同时跑碰撞。

**解决方案**：
- 时间：`freezegun` 冻结时间，或把 "now" 注入函数成为参数。
- 随机：`random.seed(42)`。
- 网络：全部 mock，或用 `vcrpy` 录制回放。
- 异步：用 event 或 asyncio.wait，不要 sleep。
- 全局：每个测试重置状态。
- 并发：事务回滚 + function 作用域。

**警示**：**flaky 测试比没测试还糟**。它让整个测试套件失去信任感——"反正这个测试就是有时挂，重跑一下"——人一旦开始重跑测试，就不再相信测试结果了。

### 17.2 测试间污染

**现象**：单独跑某个测试通过，连着跑就失败。

**诊断**：用 `pytest --lf`（只跑失败的）+ 单独跑验证。如果单独跑都过但一起跑会挂，就是污染。

**常见原因**：
- 全局变量没清。
- monkeypatch 范围超出预期。
- 数据库没回滚。
- 文件系统留下文件。
- 单例/缓存没清。

**解决**：
- 每个测试尽量"完全重置"相关状态。
- 用小作用域 fixture。
- 跑测试时加 `pytest -p no:randomly`（排除顺序敏感）或反过来用 `pytest-randomly` 故意打乱顺序来暴露污染。

### 17.3 测试太慢

**现象**：跑一次全量测试 10 分钟。开发者放弃跑测试。

**常见原因**：
- 真调外部 API。
- 数据库 fixture session 作用域没用好，每个测试都真重建。
- E2E 测试太多。
- 同步 sleep 等条件。

**解决**：
- Mock 外部调用。
- 大作用域 fixture + 事务回滚。
- 金字塔纠偏：砍 E2E 加单测。
- 并行：`pytest-xdist` 的 `-n auto`。
- 慢测试打 `@pytest.mark.slow`，CI 分阶段跑。

### 17.4 过度 Mock

**现象**：测试代码 80% 是 mock 设置，每次改业务代码要改一堆 mock。

**反问**：**你到底在测什么？** 如果所有依赖都 mock 了，测试其实只在验证"这段代码调用顺序和你写的一样"——这叫**实现测试（testing implementation）**，不是**行为测试（testing behavior）**。

**解决思路**：
- 单元测试聚焦"纯函数"。业务逻辑抽出来，避免 mock。
- 集成测试就老老实实用真组件（真 DB、真 HTTP client 对 mock server）。
- 一个测试里 mock 超过 3 个东西，要警惕。

### 17.5 测试私有方法

**现象**：测 `_calculate_internal()` 这种带下划线的私有方法。

**问题**：私有方法是实现细节。你改实现（比如拆成两个私有方法）就要改测试，**重构成本爆炸**。

**正确做法**：**只测公有接口**。如果公有接口覆盖不到某个私有逻辑，说明这个逻辑没人用——要么删掉，要么把它提成公有。

**例外**：算法类私有函数（排序、哈希），纯粹到可以提成独立函数单独测——那就提出来。

### 17.6 一个测试做太多事

**反模式**：

```python
def test_user_flow():
    # 注册
    user = register("alice", "123")
    assert user.id
    # 登录
    token = login("alice", "123")
    assert token
    # 查用户
    me = get_user(token)
    assert me.name == "alice"
    # 改密码
    change_password(token, "123", "456")
    # 登出
    logout(token)
```

一个测试 5 件事。任意一步挂，后面都跑不了，你只知道"用户流程挂了"，不知道哪里挂。

**正确做法**：**一个测试一件事**。五件事拆五个测试，共享 fixture 提供用户和 token。

### 17.7 测试依赖顺序

**反模式**：`test_a` 创建用户，`test_b` 依赖 `test_a` 创建的用户。

**问题**：测试顺序一换就挂。单独跑 `test_b` 也挂。pytest 不保证顺序（而且推荐随机顺序）。

**解决**：每个测试都应该**自给自足**——自己建所需数据，自己清理。

### 17.8 忽略警告和失败

有时候测试抛出 `DeprecationWarning` 或 `ResourceWarning`，习惯性忽略。这些警告常常预示着：
- 库升级后要改代码。
- 资源泄漏（连接、文件句柄）。
- 时序问题。

**建议**：CI 里用 `pytest -W error` 把警告当错误。新的警告一出现就强制处理。

> **读到这里你应该理解**：测试写得好不好，体现在"**改代码时测试帮你还是坑你**"。帮你的测试捕获 bug 又快又准，坑你的测试让你每次改东西都要改一堆测试。避免这些反模式就是在避免"坑你的测试"。

## 18. AI Agent 后端的测试策略

这是本章最后一节，也是对"**测试在你未来工作里怎么用**"的回扣。技术面试里这部分内容出现频率高，也最能体现工程成熟度。

### 18.1 Agent 后端有哪些"难测"的东西

和普通 CRUD 后端相比，Agent 后端有几个独特挑战：

1. **LLM 调用**——不可重复、收费、有延迟、有 rate limit。
2. **RAG 检索**——依赖向量数据库和 embedding 模型。
3. **工具调用（Tool Use）**——Agent 会调外部 API。
4. **多轮对话**——上下文累积，测一个"对话流程"比测一个"接口"复杂得多。
5. **流式响应（SSE）**——一次返回一片段，传统 TestClient 不好测。
6. **非确定性**——同一个 prompt 可能产生不同 response，断言"response == 某字符串"不现实。

### 18.2 测试层次分解

还是金字塔，但每层具体内容变了：

**单元层：**
- 测 prompt 模板渲染——输入 context，输出 prompt 字符串。纯函数，好测。
- 测 token 计数——用 tiktoken 算 token 数，纯函数。
- 测消息格式转换——把 DB 里的消息转成 OpenAI chat format。
- 测工具 schema 生成——从 Python 函数签名生成 JSON schema。

**集成层：**
- 测"调用 LLM + 存对话历史"——mock LLM，真的写 DB。
- 测"RAG 检索 + 拼 prompt"——真的查 pgvector，真的拼字符串。
- 测"单个工具执行"——调具体工具函数（真执行或 mock 外部 API）。
- 测"tool call 解析"——LLM 返回 tool_calls 字段，你的代码能否正确路由到对应工具。

**E2E 层：**
- 一个完整对话流程：用户发消息 → LLM 响应 → 工具调用 → 最终回答。用 VCR 录制一次真实 LLM 响应，回放跑。
- Streaming SSE 端到端：模拟客户端连接，收到预期事件。

### 18.3 怎么测 LLM 调用

这是 Agent 测试的核心难点。三种方案：

**方案一：Mock OpenAI client**

最简单、最常用。用 `unittest.mock.patch` 替换 OpenAI client：

```python
@pytest.fixture
def mock_openai(monkeypatch):
    mock_client = MagicMock()
    mock_client.chat.completions.create.return_value = MagicMock(
        choices=[
            MagicMock(
                message=MagicMock(
                    content="Mocked response",
                    tool_calls=None
                )
            )
        ]
    )
    monkeypatch.setattr("myapp.llm.openai_client", mock_client)
    return mock_client

def test_chat(mock_openai):
    response = chat_service.ask("hi")
    assert response == "Mocked response"
    mock_openai.chat.completions.create.assert_called_once()
```

**优点**：快、免费、可预测。
**缺点**：你的 mock 和真实 API 行为可能不一致，一旦 OpenAI 改 API 格式你就会"上线挂"。

**方案二：VCR Cassette（录制回放）**

用 `vcrpy` 或 `pytest-vcr`：**第一次运行真的调用 API**，把 HTTP 请求和响应录到一个文件（叫 cassette）。以后运行**从文件回放**，不再真调。

```python
import vcr

@vcr.use_cassette("cassettes/test_chat.yaml")
def test_chat():
    response = chat_service.ask("What is 2+2?")
    assert "4" in response
```

**优点**：既有真实行为，又不花钱、不慢。
**缺点**：cassette 要 commit 到仓库，prompt 改了要重录。断言不能太严格（LLM 有时候同 prompt 输出都不一样，但对同一个 HTTP call，cassette 里是固定的）。

**方案三：Language Model 专用 mock 库**

比如 `langsmith`、`promptfoo`、或社区的 `openai-mock`。它们提供更结构化的 mock：你可以"按 prompt 内容匹配不同返回"。

**推荐策略**：
- 单元/集成测试：**mock**，快而稳。
- E2E 测试：**VCR 录播**，少量关键流程。
- 极关键场景：**真实调用 + 语义断言**（用另一个 LLM 判断响应好坏）。

### 18.4 怎么测 RAG

RAG 测试的核心问题：**embedding 是高维浮点数，断言不精确**。

典型策略：

```python
def test_rag_retrieval(db, knowledge_factory):
    # 建一批知识
    knowledge_factory(content="Python 是一门解释型语言")
    knowledge_factory(content="Java 是一门编译型语言")
    knowledge_factory(content="今天北京天气晴朗")

    # 检索
    results = rag.search("Python 有哪些特点？", top_k=2)

    # 断言：最相关的是 Python 那条
    assert results[0].content.startswith("Python")
    # 断言：不相关的不在 top2
    assert not any("天气" in r.content for r in results)
```

注意这里**断言的是"顺序/包含关系"**，不是"精确相似度值"。相似度数值会随 embedding 模型版本变化，断言具体数不可靠。

如果 embedding 模型调用贵，可以 mock 掉——让 embedding 函数返回固定的模式（比如根据 content 的 hash 生成），这样测的是"检索逻辑"而不是"embedding 质量"。

### 18.5 怎么测 Agent 多轮对话

多轮对话本质是一个状态机。测试要覆盖：

1. **上下文累积**：第二轮能看到第一轮的结果。
2. **工具调用闭环**：模型说要调工具 → 系统调工具 → 结果回传给模型 → 模型给最终回答。
3. **错误恢复**：工具调用失败时 Agent 能重试或放弃。

典型测试：

```python
def test_multi_turn_tool_call(mock_llm, client, db):
    # 第一轮：用户问天气
    mock_llm.set_response({
        "tool_calls": [{"name": "get_weather", "args": {"city": "Beijing"}}]
    })
    r1 = client.post("/chat", json={"msg": "北京天气？"})

    # Agent 应该调用了 get_weather
    assert "get_weather" in r1.json()["trace"]

    # 第二轮：工具返回结果后 Agent 给最终回答
    mock_llm.set_response({"content": "北京今天晴"})
    r2 = client.post("/chat", json={"continue": True})
    assert "晴" in r2.json()["content"]
```

### 18.6 怎么测流式响应（SSE）

FastAPI TestClient 的 `client.stream()` 或 `httpx.AsyncClient` 可以消费 SSE：

```python
@pytest.mark.asyncio
async def test_stream():
    async with AsyncClient(transport=ASGITransport(app=app), base_url="http://t") as ac:
        async with ac.stream("POST", "/chat/stream", json={"msg": "hi"}) as r:
            events = []
            async for line in r.aiter_lines():
                if line.startswith("data: "):
                    events.append(json.loads(line[6:]))

    assert events[0]["type"] == "start"
    assert events[-1]["type"] == "end"
    assert any("content" in e for e in events)
```

### 18.7 测试矩阵建议

给你一个实战项目的测试分布建议（假设 3000 行业务代码）：

| 层次 | 数量 | 跑一次时间 |
|------|------|-----------|
| 单元测试（纯函数） | 150-200 个 | 2-5 秒 |
| 集成测试（真 DB + mock LLM） | 30-50 个 | 10-30 秒 |
| LLM VCR 回放测试 | 5-10 个 | 5 秒 |
| E2E 测试（真 LLM） | 2-3 个，只在夜间跑 | 1-2 分钟 |

**关键原则**：日常开发只跑前三类（1 分钟内），夜间 CI 跑全量。

> **读到这里你应该理解**：Agent 后端测试的核心难点是"LLM 不可重复"。用 mock/VCR 把它变得可重复是工程上的必然选择。真实调用的测试只留给关键场景，且不放到日常 CI。

### 18.8 回扣面试话术

在技术面试里聊到测试时，可以这么说：

> "我理解 Agent 后端测试分三层：
> 单元测 prompt 模板、token 计数这种纯函数；
> 集成测 LLM 调用链路，用 monkeypatch 替换 OpenAI client；
> 关键场景用 VCR 录一次真实响应作为回归；
> 真实 LLM 调用只放夜间 CI，因为它慢、贵、不可重复。
> 这套分层让我们日常开发的反馈循环控制在 30 秒内，同时不放过核心流程的真实行为。"

这个回答体现的是"**对问题本质的理解 + 工程权衡能力**"，远比背概念有用。

## 19. 面试考点

给 5-8 道测试相关面试题与高分答案思路。

### Q1：pytest 的 fixture 是什么？和 unittest 的 setUp/tearDown 有什么区别？

**答**：
Fixture 是 pytest 的测试资源管理机制。测试函数在参数里"声明需要什么资源"（如 `def test_x(db):`），pytest 通过名字匹配自动调用同名 fixture 并把返回值注入——本质是依赖注入。

相比 setUp/tearDown：
1. **粒度更细**：每个 fixture 独立定义，可以任意组合；setUp 是"整个类共享一套前置"。
2. **支持 scope**：fixture 可以是 function/class/module/session 级共享，避免重复创建昂贵资源。
3. **支持 yield 清理**：yield 前是 setup，后是 teardown，且失败也保证执行。
4. **支持依赖**：fixture 可以依赖其他 fixture，pytest 自动解析依赖链。
5. **autouse**：可以声明全局自动应用。

### Q2：Mock 的时候 patch 路径应该怎么写？

**答**：
**"Patch where it's used, not where it's defined"**——要在"使用"它的模块里 patch，而不是"定义"它的模块。

原因：`from module import X` 会在当前模块创建对 X 的本地引用。如果你 patch 原模块的 X，当前模块的本地引用还指向原对象，patch 不生效。

例子：`mymod/api.py` 里 `from requests import get`，测试要写 `patch("mymod.api.get")` 而不是 `patch("requests.get")`。

这是 Python mock 最常见的坑。理解这点后 90% 的 mock 失败都能自己诊断。

### Q3：测试数据库怎么保证测试之间不相互污染？

**答**：
三种策略：
1. **每个测试前清表**：简单但慢，外键多的时候麻烦。
2. **事务回滚**（推荐）：每个测试开一个事务，测完 rollback。速度极快，隔离性好。被测代码内部如果 commit，要用 SAVEPOINT（嵌套事务）技巧。
3. **每个测试新建数据库**：最彻底但最慢，只适合少量关键场景。

生产实践中大部分用事务回滚。对 PG 专属特性（JSONB、数组、CTE）一定要用真 PostgreSQL 不要用 SQLite，CI 用 GitHub Actions services 起 PG 容器。

### Q4：怎么测试 AI Agent 里的 LLM 调用？

**答**：
核心难点：LLM 不可重复、收费、慢。不能每个测试都真调。

分层策略：
- **单元/集成测试 mock**：用 `unittest.mock.patch` 替换 OpenAI client，固定返回值，快速反馈。
- **关键回归用 VCR**：`vcrpy` 第一次录制真实 HTTP 响应，后续回放。兼顾真实性和可重复性。
- **真实调用只在夜间 CI**：少量关键流程，用宽松的语义断言（比如"回答里包含关键词"而不是"回答等于某字符串"）。

同时要注意："Mock" 测的是"调用逻辑"而不是"LLM 质量"。LLM 质量要靠 eval 框架（promptfoo / langsmith）单独评估，不走单元测试这条路。

### Q5：代码覆盖率 100% 是好事吗？

**答**：
不一定。覆盖率是衡量测试全面度的**必要不充分条件**：
- **必要**：覆盖率低说明有代码完全没测，肯定有盲区。
- **不充分**：覆盖率 100% 可以是"跑过代码但没做真断言"，也可以覆盖行但没覆盖分支。

实践建议：
- 行覆盖 + 分支覆盖都开。
- 目标 80% 左右最划算，核心业务 95%+，边缘代码不强求。
- 用 HTML 报告看**哪些文件低**，而不是只盯着总数字。
- 覆盖率进入 CI 做红线（比如低于 70% 失败），但不追求数字好看。

### Q6：Flaky Test 怎么处理？

**答**：
**绝对不能容忍 flaky test**。flaky test 比没测还坏，因为它摧毁团队对测试的信任——"反正这个测试有时挂，重跑一下"。人一旦开始手动重跑，测试就失去意义了。

处理步骤：
1. **隔离**：给 flaky 的测试打 `@pytest.mark.flaky`，先从主干 CI 移出。
2. **诊断根因**：常见来源——时间、随机、网络、异步顺序、全局状态、数据库并发。
3. **根治**：冻结时间（freezegun）、固定随机种子、mock 网络、异步改用 event 等待、事务回滚隔离。
4. **回归**：修好后移除 mark，合回主干。

最差的做法是"加 retry 装饰器让它自动重跑"——这是鸵鸟策略，等于掩盖 bug。

### Q7：GitHub Actions workflow 的基本结构是什么？你会怎么给一个 FastAPI + PG 项目写 CI？

**答**：
核心概念：**workflow → jobs → steps**，用 YAML 写，放在 `.github/workflows/*.yml`。

触发：`on: [push, pull_request]`。

典型 FastAPI + PG 项目结构：
1. **lint job**：ruff + mypy 检查。
2. **test job**：`needs: lint`，用 `services.postgres` 起 PG 容器，setup-python 装依赖（开启 pip cache），跑 `pytest --cov --cov-fail-under=80`，上传覆盖率到 Codecov。
3. **build job**：只在 main 分支，`needs: test`，构建 Docker 镜像推到 registry。

关键实践：用 `actions/cache` 或 `setup-python` 的 `cache: pip` 加速依赖安装；敏感信息用 `secrets.*`；`timeout-minutes` 防止挂死；按 Python 版本用 `matrix` 跑兼容性。

### Q8：你怎么平衡"写代码速度"和"写测试成本"？

**答**：
不是"要不要写"，而是"**写哪些层的**"。我的原则是：
- **纯函数必写单测**：边际成本低，回归效果好。
- **业务流程写集成测试**：覆盖核心路径，mock 外部依赖。
- **E2E 只写关键流程**：少量就够，不追求覆盖。
- **覆盖率目标 70-80%**：超出这个区间性价比递减。
- **先测边界条件**：空输入、极大值、异常路径——bug 高发区。

对初期项目我会确保 CI 能跑、核心 API 有一个"happy path" 集成测试，这样后期不会无可救药。最怕的是"等有时间再补测试"——那一天永远不会到。
