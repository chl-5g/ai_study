## 0. 写在前面：为什么要花时间学网络

### 0.1 一个你每天都在做但从没想过的动作

你打开浏览器，输入 `https://www.google.com`，按下回车。0.5 秒之后 Google 的首页出现了。

这 0.5 秒内发生了什么？粗略数一下：

1. 你的电脑问"Google 这个名字对应哪个 IP 地址？" —— **DNS 解析**
2. 电脑和 Google 的服务器建立了一条"线路" —— **TCP 三次握手**
3. 双方协商用什么密码说话，交换了密钥 —— **TLS 握手**
4. 你的电脑说了一句"请把首页给我" —— **HTTP 请求**
5. Google 的服务器说了一句"给你，这是 HTML" —— **HTTP 响应**
6. 你的浏览器把 HTML 画出来

这六步里，任何一步卡住、错了、被拦截，你都会看到熟悉的"网页打不开"。而作为后端工程师，你的工作就是**写那个回答请求的服务器**——它必须在 0.5 秒内完成所有这些动作。

### 0.2 你可能已经踩过的坑

下面这些问题，如果有任何一个你回答不上来，都说明网络基础需要补：

- 为什么我本地跑 `localhost:8000` 可以，别人用我的 IP 就打不开？
- 前端说"CORS 错误"，为什么后端明明返回了 200，浏览器还是报错？
- 为什么 POST 请求偶尔会出现两次，GET 却不会？
- 为什么 `https` 比 `http` 安全？自签名证书为什么浏览器会警告？
- 接口 504 Gateway Timeout，到底是谁超时了？我的代码还是 nginx？
- 调 OpenAI API 偶尔会断流，为什么需要配 `timeout` 和 `retry`？

这些问题的答案都在本章。

### 0.3 学完本章你能做什么

- **看懂报文**：用 `curl -v` 或浏览器 DevTools 看一个请求的完整过程，能指出每个字段的含义
- **调试接口**：接口出错时，能根据状态码、响应头迅速定位是前端、后端、网关还是网络问题
- **设计 API**：能写出符合 REST 规范、可维护的接口
- **写客户端**：能给 AI Agent 写一个带超时、重试、流式响应处理的 HTTP 客户端
- **过面试**：TCP 三次握手、HTTPS 加密流程、JWT 这些高频题目全部拿下

### 0.4 学习建议

- **不要死记**：握手的 seq、ack 不需要背，理解"为什么这样设计"才有用
- **一定要动手**：每个 Python 示例都自己跑一遍，结果和注释对不上就停下来想
- **用 Wireshark/curl 观察**：抽象的概念抓一次包就全懂了
- **和 Agent 联系起来**：每学一个概念就想"这个在调用 LLM API 时怎么用"

## 1. 网络分层模型：快递系统的 OSI 与 TCP/IP

### 1.1 Why：为什么要分层

想象你要寄一个包裹给福州的朋友。整个流程大致是：

1. 你写一封信，装进信封，写上"福州市鼓楼区 XX 路 123 号"
2. 你把信封拿到顺丰网点
3. 网点把你的信和其他很多信一起装进一个大包裹，贴上"福州分拨中心"的标签
4. 这个大包裹被装上卡车，卡车走高速路到福州
5. 福州分拨中心拆开大包裹，把你的信单独拿出来
6. 快递员骑电动车送到朋友家

这里有三个很关键的观察：

**观察一：每一层只关心自己的任务**
- 你只关心"地址写对没有"，不关心卡车走哪条高速
- 卡车司机只关心"到福州分拨中心"，不关心包裹里是信还是文件
- 快递员只关心"这封信该送到哪一户人家"，不关心信的内容

**观察二：每一层只和相邻层打交道**
- 你只和网点打交道，不会直接跟卡车司机说话
- 卡车司机只对接分拨中心，不直接对接收件人

**观察三:换掉其中一层不影响别的层**
- 卡车换成飞机、高铁，你写信的方式不变
- 你把信封换成快递盒，卡车照常跑

网络也是一样的。数据从你的浏览器到 Google 的服务器，中间要经过网卡、路由器、光纤、海底电缆……如果不分层，每换一个网络设备都要改所有代码，根本没法维护。所以**前人把网络通信切成若干层，每层只做一件事，每层之间接口固定**。这就是"网络分层模型"。

### 1.2 What：OSI 七层模型

OSI（Open Systems Interconnection）是国际标准化组织在 1984 年提出的参考模型，它把网络通信分成 7 层。从上到下是：

```
┌───────────────────────────────────┐
│ 7. 应用层 Application             │  你看得到的"业务": HTTP, FTP, SMTP, DNS
├───────────────────────────────────┤
│ 6. 表示层 Presentation            │  数据格式转换、加密压缩 (TLS 逻辑上在这)
├───────────────────────────────────┤
│ 5. 会话层 Session                 │  会话管理、断点续传
├───────────────────────────────────┤
│ 4. 传输层 Transport               │  端到端可靠传输: TCP, UDP
├───────────────────────────────────┤
│ 3. 网络层 Network                 │  路由和寻址: IP, ICMP
├───────────────────────────────────┤
│ 2. 数据链路层 Data Link           │  相邻节点间传输: Ethernet, Wi-Fi, ARP
├───────────────────────────────────┤
│ 1. 物理层 Physical                │  电信号、光信号、电磁波
└───────────────────────────────────┘
```

**类比映射回快递系统：**

| OSI 层 | 快递类比 | 做什么 |
|--------|----------|--------|
| 应用层 | 你写信的内容 | 业务数据本身 |
| 表示层 | 信封的格式、把中文翻译成英文 | 数据编码/加密 |
| 会话层 | "这是给同一个收件人的系列邮件" | 会话管理 |
| 传输层 | 顺丰 vs 平邮，要不要签收 | 可靠性保证 |
| 网络层 | 省内 / 跨省的路由规划 | 选择路径 |
| 数据链路层 | 同一个分拨中心内部的传送带 | 点对点传输 |
| 物理层 | 电动车、飞机、卡车 | 物理媒介 |

### 1.3 What：TCP/IP 四层模型（工程中真正使用的）

OSI 是教科书上的模型，好看但过于学术。工程上实际用的是 **TCP/IP 四层模型**（也叫五层混合模型）：

```
┌─────────────────────────────────┐
│ 应用层 Application              │  HTTP, WebSocket, gRPC, DNS, SSH
│ (OSI 5+6+7)                     │
├─────────────────────────────────┤
│ 传输层 Transport                │  TCP, UDP, QUIC
├─────────────────────────────────┤
│ 网络层 Network / Internet       │  IPv4, IPv6, ICMP
├─────────────────────────────────┤
│ 链路层 Link                     │  Ethernet, Wi-Fi, PPP
└─────────────────────────────────┘
```

TCP/IP 把 OSI 的 5、6、7 层合并成了"应用层"，因为实践中根本分不清。以 HTTPS 为例：TLS 算哪一层？从功能看像"表示层"，但它其实是运行在应用和传输之间的一个库，不纯粹属于任何一层。硬分没意义。

**本文后续讨论都用 TCP/IP 四层**，但你面试时要知道 OSI 是 7 层。

### 1.4 How：数据封装与解封装

当你发送一个 HTTP 请求时，数据从上往下一路"加头"，到网卡上变成一堆二进制：

```
你的 HTTP 请求报文
   "GET / HTTP/1.1\r\nHost: google.com\r\n\r\n"
           │
           ▼
   [应用层: HTTP]  这就是原始数据 (payload)
           │
           ▼
   [传输层: TCP]   [TCP 头 | HTTP 数据]   加 20 字节 TCP 头(源端口/目的端口/序列号...)
           │
           ▼
   [网络层: IP]    [IP 头 | TCP 头 | HTTP 数据]   加 20 字节 IP 头(源IP/目的IP)
           │
           ▼
   [链路层: 以太网] [以太网头 | IP 头 | TCP 头 | HTTP 数据 | 以太网尾]
           │
           ▼
   物理层 ~~~~~~~ 比特流 0101010101010... ~~~~~~~>
```

对方收到后，从下往上"剥头"：以太网层剥掉以太网头 → 交给 IP 层剥 IP 头 → 交给 TCP 层剥 TCP 头 → 最后交给应用层处理原始 HTTP 数据。

**每一层只看自己的头**。TCP 不关心 HTTP 内容是什么，IP 不关心 TCP 的内容是什么。这就是"每层只关心本层协议"的工程含义。

### 1.5 常见误区

**误区一：以为 HTTP 和 TCP 是并列的两个东西**
错。HTTP 运行在 TCP 之上，HTTP 的每个字节最终都会被 TCP 封装。没有 TCP（或 QUIC）就没有 HTTP。

**误区二：以为每层都有"协议头"加起来开销很大**
确实有，但开销只在发送开始和接收最后。中间的路由器只看 IP 头就够了，不会拆 TCP 头。

**误区三：以为"应用层 = 写代码的那一层，我不用懂下面"**
错。你写的 HTTP 客户端偶尔出现的"Connection reset"、"Timeout"、"SSL handshake failed"都是下面几层的错误冒上来。不懂下面，排错时就只能瞎猜。

## 2. IP、端口、MAC — 互联网的三层地址

### 2.1 Why：为什么需要三种地址

继续用快递类比。要把一封信送到朋友手里，你需要：

1. **小区地址**（"XX 小区 3 栋"）：把信送到对的小区 —— 对应 **IP 地址**（哪台机器）
2. **门牌号**（"301 室"）：小区里有很多住户，要送到对的那户 —— 对应 **端口**（机器上哪个程序）
3. **身份证号**（"这封信是给张三本人的"）：门牌号认门不认人，身份证号确认收件人 —— 对应 **MAC 地址**（网卡的物理标识）

现实中你写快递只写前两个（小区+门牌），MAC 地址是快递员（路由器）内部用的，你感知不到。下面分别讲。

### 2.2 IP 地址：网络层的门牌号

**IPv4** 是 32 位整数，写成 `a.b.c.d` 的四段点分形式，每段 0-255：

```
192.168.1.100
   │   │  │  │
 8位 8位 8位 8位   一共 32 位
```

32 位意味着总共有 `2^32 ≈ 43 亿` 个地址。听起来很多，但全球网民超过 50 亿，设备比人多，早就不够用了。所以有了 **IPv6**（128 位），地址看起来像这样：

```
2001:0db8:85a3:0000:0000:8a2e:0370:7334
```

目前国内大部分家宽默认是 IPv4 + NAT（一会讲）。

**特殊 IP：**

| IP 段 | 含义 |
|-------|------|
| `127.0.0.1` | 本机回环地址（localhost），只能自己访问自己 |
| `10.0.0.0/8` | 私有网络（A 类） |
| `172.16.0.0/12` | 私有网络（B 类） |
| `192.168.0.0/16` | 私有网络（C 类），家用路由器常用 |
| `0.0.0.0` | "所有 IP"，服务端 `bind("0.0.0.0")` 表示监听所有网卡 |
| `255.255.255.255` | 广播地址 |

**子网掩码与 CIDR：**

`192.168.1.0/24` 中的 `/24` 叫 CIDR，表示前 24 位是网络号，后 8 位是主机号。所以这个子网里有 `2^8 = 256` 个地址（去掉网络号和广播号还剩 254 个可用）。

```python
# Python 计算 CIDR
import ipaddress

net = ipaddress.ip_network("192.168.1.0/24")
print(net.num_addresses)   # 256
print(net.network_address) # 192.168.1.0
print(net.broadcast_address) # 192.168.1.255
print(list(net.hosts())[:3])  # 前 3 个可用主机
# [IPv4Address('192.168.1.1'), IPv4Address('192.168.1.2'), IPv4Address('192.168.1.3')]

ip = ipaddress.ip_address("192.168.1.100")
print(ip in net)  # True
```

上面这段代码里 `ipaddress` 是 Python 标准库（不用装），专门处理 IP 的各种运算。`ip_network("192.168.1.0/24")` 会解析出这个子网的所有属性，`ip in net` 判断一个 IP 是否属于这个子网——实际工作中判断"是否是内网 IP"就是这么写。

### 2.3 NAT：为什么你家里所有设备共享一个外网 IP

你手机、笔记本、电视都连家里的 Wi-Fi，它们的 IP 都是 `192.168.x.x` 这种私有地址。但家里只有一个外网 IP（运营商给的），它们怎么同时上网？

答案是 **NAT（Network Address Translation）**。家用路由器会在出站包上做一次翻译：

```
你的笔记本 (192.168.1.5:51234) ─┐
你的手机  (192.168.1.6:50000) ─┼─路由器─> 外网 (公网IP:6xxxx)
你的电视  (192.168.1.7:49999) ─┘
```

路由器维护一张表："记住这个外网端口 6xxxx 对应的是笔记本的 51234"，回包时再翻译回去。所以从外面看，你全家只有一个 IP。

**NAT 的副作用**：
- 从外网无法主动连接到你家内网的设备（"内网穿透"就是为了解决这个）
- P2P 连接变困难（所以有 STUN、TURN 这些技术）
- 让 IPv4 的 43 亿地址勉强够用了 20 多年

### 2.4 端口：同一台机器上的多个程序

一台服务器可能同时跑着 Web 服务、数据库、SSH、Redis……它们都在 IP = `192.168.1.100` 这一个地址上。外面的请求到了，操作系统怎么知道该把包交给哪个程序？

靠**端口号**。端口是一个 16 位整数（0-65535），每个程序监听一个或几个端口：

```
192.168.1.100
     │
     ├─ :22     ← SSH
     ├─ :80     ← HTTP
     ├─ :443    ← HTTPS
     ├─ :3306   ← MySQL
     ├─ :5432   ← PostgreSQL
     ├─ :6379   ← Redis
     └─ :8000   ← 你的 FastAPI
```

**端口分类：**
- `0-1023`：**知名端口**（well-known），需要管理员权限才能绑定。HTTP 80、HTTPS 443、SSH 22 都在这里
- `1024-49151`：**注册端口**，常用软件会约定俗成占用，比如 MySQL 3306、Redis 6379
- `49152-65535`：**动态端口**，操作系统给客户端临时分配的

**一个 TCP 连接是"四元组"**：源 IP、源端口、目的 IP、目的端口。哪怕同一个客户端同时开两个浏览器标签访问同一个服务器，也会用两个不同的源端口，所以是两个独立的连接。

```python
# 查看本机 TCP 连接的四元组（Python)
import psutil  # pip install psutil

for conn in psutil.net_connections(kind="tcp"):
    if conn.status == "ESTABLISHED":
        laddr = f"{conn.laddr.ip}:{conn.laddr.port}"
        raddr = f"{conn.raddr.ip}:{conn.raddr.port}"
        print(f"{laddr} <-> {raddr}  pid={conn.pid}")
```

上面的脚本会打印出本机所有处于 "已建立" 状态的 TCP 连接——你能看到浏览器连着各种服务器的情况。`laddr` 是本地地址（源 IP+端口），`raddr` 是远端地址（目的 IP+端口）。

### 2.5 MAC 地址：网卡的身份证

MAC（Media Access Control）地址是硬件层面的地址，每张网卡出厂时都被烧入一个全球唯一的 48 位地址，写成 `AA:BB:CC:DD:EE:FF`。

IP 地址可以变（你换个 Wi-Fi 就换了），MAC 不会变（除非你刷机或换网卡）。

**它存在的意义**：在同一个局域网（比如同一个 Wi-Fi）里，数据包最终是通过 MAC 地址送到具体设备的。路由器要问"谁的 IP 是 192.168.1.100？"，这个过程叫 **ARP（Address Resolution Protocol）**，答案是"我的 MAC 是 AA:BB:CC..."。

作为后端开发，你几乎永远不用直接处理 MAC 地址。知道有这么个东西、能分清它和 IP 的区别就够了。

### 2.6 一图串起来：数据包如何到达

```
你的浏览器                                     Google 服务器
[应用]  HTTP 请求
[传输]  源端口 51234, 目的端口 443 (HTTPS)
[网络]  源 IP 192.168.1.5, 目的 IP 142.250.xxx.xxx
[链路]  源 MAC 你的网卡, 目的 MAC 家里路由器的网卡
          │
          ▼
      家里路由器（NAT 翻译）
          │ 源 IP 改成公网 IP、源端口改成 6xxxx，记一笔
          ▼
      ISP 骨干网(一路 IP 路由)
          │
          ▼
      Google 数据中心入口
          │
          ▼
      Google 服务器（目的端口 443 → 交给 nginx）
```

### 2.7 常见误区

**误区一：IP 地址是固定给每台机器的**
错。IP 是网络层的逻辑地址，你换个 Wi-Fi、重启路由器都可能变。真正"固定的"是 MAC。

**误区二：端口和 IP 是分开的东西**
TCP 的连接是四元组 `(srcIP, srcPort, dstIP, dstPort)`。单独的端口号没有意义——必须和 IP 配对。

**误区三：127.0.0.1 和 0.0.0.0 一样**
差别很大。`127.0.0.1` 只能本机访问（loopback），`0.0.0.0` 表示"监听所有网卡"（包括内网、公网）。把 FastAPI 用 `uvicorn --host 127.0.0.1` 启动，同事的机器用你的 IP 访问就会连不上——他需要你监听 `0.0.0.0`。

**误区四：内网 IP 是"不安全"的**
内网 IP 本身没有安全/不安全的属性。它的特点是从公网无法直接访问，但不意味着自动安全——同一个 Wi-Fi 下的其他设备一样能访问你的内网服务。
