## 一、TCP/IP 协议栈

### 1.1 四层模型

```
┌──────────────────────┐
│   应用层 (Application)  │  HTTP, FTP, DNS, SMTP ...
├──────────────────────┤
│   传输层 (Transport)    │  TCP, UDP
├──────────────────────┤
│   网络层 (Network)      │  IP, ICMP, ARP
├──────────────────────┤
│   链路层 (Link)         │  Ethernet, Wi-Fi
└──────────────────────┘
```

数据从上到下**封装**（加头部），从下到上**解封装**（剥头部）。每一层只关心本层协议，对上层透明。

### 1.2 TCP 与 UDP 区别

| 特性 | TCP | UDP |
|------|-----|-----|
| 连接 | 面向连接（需先建立连接） | 无连接 |
| 可靠性 | 可靠传输（确认、重传、排序） | 不可靠（尽力而为） |
| 流量控制 | 有（滑动窗口） | 无 |
| 拥塞控制 | 有（慢启动、拥塞避免等） | 无 |
| 开销 | 头部 20 字节起 | 头部仅 8 字节 |
| 传输方式 | 字节流 | 数据报 |
| 适用场景 | 文件传输、Web、邮件 | 视频直播、DNS 查询、游戏 |

**关键要点：**
- TCP 牺牲速度换可靠性；UDP 牺牲可靠性换速度。
- HTTP/1.1、HTTP/2 基于 TCP；HTTP/3 (QUIC) 基于 UDP。

### 1.3 TCP 三次握手 (Three-Way Handshake)

```
客户端                          服务端
  │                               │
  │──── SYN (seq=x) ────────────>│   ① 客户端发起连接请求
  │                               │
  │<─── SYN+ACK (seq=y,ack=x+1)──│   ② 服务端确认并发起自己的连接
  │                               │
  │──── ACK (ack=y+1) ──────────>│   ③ 客户端确认服务端的连接
  │                               │
  │      === 连接建立 ===          │
```

**为什么是三次而不是两次？**
- 两次握手无法确认客户端的接收能力。如果客户端发了一个旧的 SYN（网络延迟），服务端会误以为是新连接，浪费资源。
- 三次握手确保双方都确认了对方的发送和接收能力。

### 1.4 TCP 四次挥手 (Four-Way Handshake)

```
客户端                          服务端
  │                               │
  │──── FIN (seq=u) ────────────>│   ① 客户端请求关闭（不再发送数据）
  │                               │
  │<─── ACK (ack=u+1) ──────────│   ② 服务端确认收到（但可能还有数据要发）
  │                               │
  │      (服务端继续发送剩余数据)      │
  │                               │
  │<─── FIN (seq=w) ────────────│   ③ 服务端数据发完，也请求关闭
  │                               │
  │──── ACK (ack=w+1) ──────────>│   ④ 客户端确认（进入 TIME_WAIT）
  │                               │
  │   === 连接关闭 ===             │
```

**为什么是四次而不是三次？**
- 因为 TCP 是全双工的。一方关闭只是表示自己不再发送，但还可以接收。所以需要双方各自关闭。

**TIME_WAIT 状态：**
- 客户端发完最后一个 ACK 后，会等待 2MSL（Maximum Segment Lifetime，通常 60 秒）。
- 目的：确保最后的 ACK 能到达服务端；防止旧连接的延迟包干扰新连接。

### 常见面试问题

1. **TCP 和 UDP 的区别？各自适用场景？**
2. **为什么握手是三次，挥手是四次？**
3. **TIME_WAIT 状态的作用？大量 TIME_WAIT 怎么处理？**
4. **TCP 如何保证可靠传输？** — 序列号、确认应答、超时重传、流量控制、拥塞控制。
5. **TCP 粘包问题是什么？怎么解决？** — 固定长度、分隔符、长度前缀。

## 二、DNS 原理

### 2.1 概念

DNS（Domain Name System）是将域名（如 `www.example.com`）解析为 IP 地址（如 `93.184.216.34`）的分布式数据库系统。

### 2.2 域名层级结构

```
                 . (根域)
               / | \
           com  org  cn  ...         (顶级域 TLD)
          / \
     google  example  ...           (二级域)
       |
      www                           (子域/主机名)
```

完整域名（FQDN）：`www.example.com.`（末尾有个点代表根域，通常省略）。

### 2.3 DNS 解析流程

```
用户浏览器
    │
    ▼
① 浏览器缓存 ──命中──> 返回IP
    │ 未命中
    ▼
② 操作系统缓存（/etc/hosts）──命中──> 返回IP
    │ 未命中
    ▼
③ 本地DNS服务器（递归解析器，通常是ISP提供）
    │
    ├──④──> 根DNS服务器（全球13组）  → 返回 ".com" 的NS地址
    │
    ├──⑤──> .com 顶级域服务器       → 返回 "example.com" 的NS地址
    │
    ├──⑥──> example.com 权威服务器  → 返回 www.example.com 的IP
    │
    ▼
⑦ 本地DNS服务器缓存结果，返回给用户
```

### 2.4 DNS 记录类型

| 类型 | 说明 | 示例 |
|------|------|------|
| A | 域名 → IPv4 地址 | `example.com → 93.184.216.34` |
| AAAA | 域名 → IPv6 地址 | `example.com → 2606:2800:...` |
| CNAME | 域名别名 | `www.example.com → example.com` |
| MX | 邮件服务器 | `example.com → mail.example.com` |
| NS | 域名服务器 | `example.com → ns1.example.com` |
| TXT | 文本记录（SPF、验证等） | `example.com → "v=spf1 ..."` |

### 关键要点
- DNS 查询默认使用 UDP（端口 53），响应超过 512 字节时用 TCP。
- TTL（Time To Live）控制缓存时间，TTL 越短更新越快但查询压力越大。
- DNS 劫持 / DNS 污染：中间人篡改 DNS 响应，解法是 DoH（DNS over HTTPS）或 DoT（DNS over TLS）。

### 常见面试问题

1. **DNS 解析的完整流程？**
2. **DNS 用 TCP 还是 UDP？** — 默认 UDP，数据量大或区域传输用 TCP。
3. **如何加速 DNS 解析？** — 浏览器预解析（`dns-prefetch`）、使用更快的公共 DNS（如 8.8.8.8）。
4. **CNAME 和 A 记录的区别？**

## 三、HTTP 协议

### 3.1 HTTP 方法语义

| 方法 | 语义 | 幂等性 | 安全性 | 请求体 |
|------|------|--------|--------|--------|
| GET | 获取资源 | 是 | 是 | 无 |
| POST | 创建资源 / 提交数据 | 否 | 否 | 有 |
| PUT | 全量替换资源 | 是 | 否 | 有 |
| PATCH | 部分更新资源 | 否 | 否 | 有 |
| DELETE | 删除资源 | 是 | 否 | 可选 |
| HEAD | 同 GET 但只返回头部 | 是 | 是 | 无 |
| OPTIONS | 查询支持的方法（CORS 预检） | 是 | 是 | 无 |

**幂等性 (Idempotent)：** 同一个请求执行多次，效果与执行一次相同。
**安全性 (Safe)：** 不会修改服务器上的资源。

### 3.2 HTTP 状态码

| 范围 | 类别 | 常见状态码 |
|------|------|-----------|
| 1xx | 信息性 | `100 Continue`（继续发送请求体） |
| 2xx | 成功 | `200 OK`, `201 Created`, `204 No Content` |
| 3xx | 重定向 | `301 永久重定向`, `302 临时重定向`, `304 Not Modified`（缓存命中） |
| 4xx | 客户端错误 | `400 Bad Request`, `401 Unauthorized`（未认证）, `403 Forbidden`（无权限）, `404 Not Found`, `405 Method Not Allowed`, `429 Too Many Requests` |
| 5xx | 服务端错误 | `500 Internal Server Error`, `502 Bad Gateway`, `503 Service Unavailable`, `504 Gateway Timeout` |

**易混淆：**
- `401` vs `403`：401 是"你是谁？（未登录）"，403 是"我知道你是谁，但你没权限"。
- `301` vs `302`：301 永久，浏览器会缓存重定向；302 临时，每次都会请求原地址。

### 3.3 HTTP 请求/响应结构

**请求：**
```
GET /api/users?page=1 HTTP/1.1        ← 请求行（方法 路径 版本）
Host: example.com                      ← 请求头
Accept: application/json
Authorization: Bearer xxx
                                       ← 空行
                                       ← 请求体（GET 通常无）
```

**响应：**
```
HTTP/1.1 200 OK                        ← 状态行（版本 状态码 原因短语）
Content-Type: application/json         ← 响应头
Content-Length: 128
Set-Cookie: session=abc123
                                       ← 空行
{"users": [...]}                       ← 响应体
```

### 3.4 Content-Type（MIME 类型）

| Content-Type | 说明 | 使用场景 |
|-------------|------|---------|
| `application/json` | JSON 数据 | REST API |
| `application/x-www-form-urlencoded` | 表单数据（键值对） | HTML 表单默认 |
| `multipart/form-data` | 多部分数据 | 文件上传 |
| `text/html` | HTML 文档 | 网页 |
| `text/plain` | 纯文本 | 日志等 |

### 3.5 Cookie 与 Session

```
                  ┌──────────────────────────┐
                  │        服务器              │
                  │                          │
                  │  Session 存储:            │
                  │  sid=abc123 → {user: 张三} │
                  └──────────┬───────────────┘
                             │
          ① POST /login      │    ② Set-Cookie: sid=abc123
          (用户名+密码)  ──────>│<────────────────────────
                             │
          ③ GET /profile     │    ④ 200 OK (欢迎，张三)
          Cookie: sid=abc123 ──>│<────────────────────────
                             │
                  ┌──────────┴───────────────┐
                  │        浏览器              │
                  │  Cookie: sid=abc123       │
                  └──────────────────────────┘
```

**Cookie 属性：**
- `HttpOnly`：JavaScript 无法读取（防 XSS）
- `Secure`：仅 HTTPS 传输
- `SameSite`：`Strict`（完全禁止跨站）/ `Lax`（允许导航跳转）/ `None`（允许跨站，需 Secure）
- `Max-Age` / `Expires`：过期时间

**Session vs Cookie：**
- Cookie 存在客户端，Session 存在服务端。
- Session ID 通过 Cookie 传递。
- Session 更安全（敏感数据不暴露给客户端），但增加服务器负担。

### 3.6 HTTPS

HTTPS = HTTP + TLS/SSL

```
客户端                              服务器
  │                                   │
  │──── ClientHello ───────────────>│   ① 发送支持的TLS版本和加密算法
  │                                   │
  │<─── ServerHello + 证书 ─────────│   ② 返回选定算法和服务器证书
  │                                   │
  │  验证证书（CA链）                    │   ③ 客户端验证证书合法性
  │                                   │
  │──── 密钥交换（Pre-Master Secret）──>│   ④ 用证书公钥加密，发送预主密钥
  │                                   │
  │     双方计算出对称密钥                │   ⑤ 后续通信使用对称加密（快）
  │                                   │
  │<═══ 加密的HTTP通信 ═══════════>│
```

**关键要点：**
- 非对称加密（RSA/ECDHE）用于密钥交换；对称加密（AES）用于数据传输。
- 证书由 CA（Certificate Authority）签发，浏览器内置了受信任的根证书。
- Let's Encrypt 提供免费证书。

### 常见面试问题

1. **GET 和 POST 的区别？** — 语义、幂等性、缓存、请求体、URL 长度限制（浏览器限制，非协议限制）。
2. **301 和 302 的区别？**
3. **Cookie 和 Session 的区别？**
4. **HTTPS 的加密流程？为什么需要 CA 证书？** — 防止中间人攻击。
5. **HTTP/1.1 vs HTTP/2 vs HTTP/3？** — 多路复用、头部压缩、Server Push；HTTP/3 基于 QUIC (UDP)。

## 四、CORS（跨域资源共享）

### 4.1 同源策略 (Same-Origin Policy)

浏览器安全策略：只允许网页访问**同源**的资源。

**同源** = 协议 + 域名 + 端口 完全相同。

| URL A | URL B | 是否同源 | 原因 |
|-------|-------|---------|------|
| `http://a.com/page` | `http://a.com/api` | 同源 | |
| `http://a.com` | `https://a.com` | 跨域 | 协议不同 |
| `http://a.com` | `http://b.com` | 跨域 | 域名不同 |
| `http://a.com` | `http://a.com:8080` | 跨域 | 端口不同 |

**注意：** 同源策略是浏览器行为，服务端之间通信没有跨域限制。

### 4.2 CORS 机制

```
浏览器                                  服务器 (api.example.com)
  │                                       │
  │  [简单请求: GET/POST, 常规头]           │
  │──── 请求 + Origin: http://app.com ──>│
  │<─── Access-Control-Allow-Origin ─────│
  │                                       │
  │  [预检请求: PUT/DELETE, 自定义头]       │
  │──── OPTIONS (预检) ──────────────────>│   ① 浏览器先发 OPTIONS
  │<─── Access-Control-Allow-* ──────────│   ② 服务器返回允许的方法和头
  │──── 实际请求 ────────────────────────>│   ③ 预检通过后发实际请求
  │<─── 响应 ────────────────────────────│
```

**简单请求条件（不触发预检）：**
- 方法：GET / HEAD / POST
- 头部：只有 Accept、Accept-Language、Content-Language、Content-Type（且只能是 `text/plain`、`multipart/form-data`、`application/x-www-form-urlencoded`）

**CORS 响应头：**
| 头部 | 说明 |
|------|------|
| `Access-Control-Allow-Origin` | 允许的源（`*` 或具体源） |
| `Access-Control-Allow-Methods` | 允许的 HTTP 方法 |
| `Access-Control-Allow-Headers` | 允许的自定义请求头 |
| `Access-Control-Allow-Credentials` | 是否允许携带 Cookie（`true`） |
| `Access-Control-Max-Age` | 预检请求缓存时间（秒） |

### 4.3 跨域解决方案

1. **CORS 头**（推荐）：服务端设置 `Access-Control-Allow-Origin`。
2. **代理**（开发常用）：前端请求同源的代理服务器，代理转发到目标 API。
3. **JSONP**（已过时）：利用 `<script>` 标签不受同源策略限制，只支持 GET。
4. **Nginx 反向代理**（生产常用）：统一域名，不同路径转发到不同服务。

### 常见面试问题

1. **什么是跨域？为什么会有跨域？**
2. **CORS 预检请求 (Preflight) 是什么？什么时候触发？**
3. **`Access-Control-Allow-Origin: *` 和具体域名的区别？** — 使用 `*` 时不能携带 Cookie。
4. **前后端分离项目如何处理跨域？**

## 五、RESTful API 设计

### 5.1 核心原则

REST（Representational State Transfer）是一种 API 设计风格：
- **资源为中心**：URL 代表资源，用名词不用动词。
- **HTTP 方法表达操作**：用 GET/POST/PUT/DELETE 表示 CRUD。
- **无状态**：每个请求包含所有必要信息，服务器不保存客户端状态。

### 5.2 URL 规范

```
✅ 好的设计：
GET    /api/v1/users              获取用户列表
GET    /api/v1/users/123          获取单个用户
POST   /api/v1/users              创建用户
PUT    /api/v1/users/123          全量更新用户
PATCH  /api/v1/users/123          部分更新用户
DELETE /api/v1/users/123          删除用户
GET    /api/v1/users/123/orders   获取用户的订单（嵌套资源）

❌ 差的设计：
GET    /api/getUser?id=123        动词出现在 URL 中
POST   /api/deleteUser            用 POST 做删除
GET    /api/Users                 大写（应该全小写）
```

**规范：**
- 资源名用**复数名词**：`/users` 不是 `/user`
- URL 全小写，用 `-` 连接：`/user-profiles` 不是 `/userProfiles`
- 避免深层嵌套（一般不超过 2 层）：`/users/123/orders` 可以，`/users/123/orders/456/items/789` 不好

### 5.3 CRUD 对应

| 操作 | HTTP 方法 | URL 示例 | 状态码 |
|------|----------|----------|--------|
| 创建 | POST | `/users` | 201 Created |
| 读取（列表） | GET | `/users` | 200 OK |
| 读取（单个） | GET | `/users/123` | 200 OK |
| 全量更新 | PUT | `/users/123` | 200 OK |
| 部分更新 | PATCH | `/users/123` | 200 OK |
| 删除 | DELETE | `/users/123` | 204 No Content |

### 5.4 统一响应格式

```json
// 成功响应
{
    "code": 200,
    "message": "success",
    "data": {
        "id": 123,
        "name": "张三"
    }
}

// 列表响应（带分页）
{
    "code": 200,
    "message": "success",
    "data": {
        "items": [...],
        "total": 100,
        "page": 1,
        "page_size": 20
    }
}

// 错误响应
{
    "code": 400,
    "message": "参数校验失败",
    "errors": [
        {"field": "email", "message": "邮箱格式不正确"}
    ]
}
```

### 5.5 分页、排序、过滤

```
分页：
GET /users?page=2&page_size=20

排序：
GET /users?sort=created_at&order=desc

过滤：
GET /users?status=active&role=admin

搜索：
GET /users?q=张三

组合：
GET /users?status=active&sort=name&order=asc&page=1&page_size=10
```

### 5.6 API 版本管理

| 方式 | 示例 | 优缺点 |
|------|------|--------|
| URL 路径（最常用） | `/api/v1/users` | 简单直观，但 URL 变了 |
| 请求头 | `Accept: application/vnd.api.v1+json` | URL 不变，但不直观 |
| 查询参数 | `/api/users?version=1` | 简单，但容易遗漏 |

### 常见面试问题

1. **什么是 RESTful？和普通 API 有什么区别？**
2. **PUT 和 PATCH 的区别？** — PUT 全量替换（未传字段会被清空）；PATCH 部分更新（只改传的字段）。
3. **RESTful API 如何做版本管理？**
4. **如何设计一个好的 RESTful API？**

## 六、认证与授权

### 6.1 认证 (Authentication) vs 授权 (Authorization)

- **认证**：你是谁？（登录）
- **授权**：你能做什么？（权限）

### 6.2 Session vs Token

```
Session 模式：
┌────────┐   登录    ┌────────┐
│ 浏览器  │ ───────> │ 服务器  │
│        │ <─────── │        │
│Cookie: │ sid=xxx  │Session │
│sid=xxx │          │存储:   │
│        │ 携带Cookie│sid→用户 │
│        │ ───────> │        │
└────────┘          └────────┘

Token 模式：
┌────────┐   登录    ┌────────┐
│ 客户端  │ ───────> │ 服务器  │
│        │ <─────── │        │
│Token:  │ token    │无状态! │
│eyJhb...│          │不存储  │
│        │ Header:  │token   │
│        │ Bearer   │        │
│        │ ───────> │ 验证签名│
└────────┘          └────────┘
```

| 特性 | Session | Token (JWT) |
|------|---------|-------------|
| 存储位置 | 服务端 | 客户端 |
| 扩展性 | 差（多服务器需共享 Session） | 好（无状态） |
| 跨域 | 依赖 Cookie，跨域麻烦 | 放 Header，无跨域问题 |
| 移动端 | Cookie 支持差 | 天然支持 |
| 注销 | 删除 Session 即可 | 需要黑名单机制 |

### 6.3 JWT (JSON Web Token)

JWT 由三部分组成，用 `.` 连接：

```
eyJhbGciOiJIUzI1NiJ9.eyJ1c2VyX2lkIjoxMjN9.签名
|______Header_______|.|______Payload_______|.|_Sig_|

Header (Base64):
{
    "alg": "HS256",    // 签名算法
    "typ": "JWT"
}

Payload (Base64):
{
    "user_id": 123,
    "username": "zhangsan",
    "role": "admin",
    "exp": 1735689600,  // 过期时间
    "iat": 1735603200   // 签发时间
}

Signature:
HMACSHA256(base64(header) + "." + base64(payload), secret_key)
```

**关键要点：**
- Payload 是 Base64 编码，**不是加密**！任何人都能解码看到内容。不要放敏感信息。
- 签名保证 Token 没被篡改（完整性），不保证机密性。
- JWT 无法主动失效（除非用黑名单）。常用方案：短有效期 + Refresh Token。

**Access Token + Refresh Token 模式：**
```
① 登录 → 返回 access_token(15min) + refresh_token(7天)
② 请求 API → 携带 access_token
③ access_token 过期 → 用 refresh_token 换新的 access_token
④ refresh_token 过期 → 重新登录
```

### 6.4 OAuth 2.0 入门

OAuth 2.0 是一个**授权框架**，允许第三方应用在用户授权下访问用户在另一个服务上的资源。

**场景：** 你的网站想获取用户的 GitHub 信息（"使用 GitHub 登录"）。

```
用户            你的应用           GitHub
 │                │                 │
 │  点击GitHub登录 │                 │
 │ ──────────────>│                 │
 │                │── 重定向到GitHub ─>│
 │                │                 │
 │  用户在GitHub授权│                 │
 │ ────────────────────────────────>│
 │                │                 │
 │                │<── 返回授权码code ─│   ① Authorization Code
 │                │                 │
 │                │── 用code换token ─>│   ② 后端请求，附带client_secret
 │                │                 │
 │                │<── access_token ──│   ③ 获得令牌
 │                │                 │
 │                │── 用token获取资料 ─>│   ④ 调用 GitHub API
 │                │<── 用户信息 ───────│
 │                │                 │
 │  登录成功       │                 │
 │ <──────────────│                 │
```

**四种授权模式：**
1. **Authorization Code**（最安全，推荐）：适合有后端的应用。
2. **Implicit**（已不推荐）：适合纯前端应用，Token 直接返回给前端。
3. **Password**（不推荐）：用户直接把密码给第三方应用。
4. **Client Credentials**：机器对机器，无用户参与。

### 6.5 密码安全

**绝对不能做的事：**
- 明文存储密码
- 用 MD5/SHA256 直接哈希（容易被彩虹表破解）

**正确做法：**
```python
# 使用 bcrypt（自带盐值，自动处理）
import bcrypt

# 注册时：哈希密码
password = "mypassword123"
hashed = bcrypt.hashpw(password.encode(), bcrypt.gensalt())
# 存入数据库: $2b$12$LJ3m5R5X8K7...（包含算法、盐值、哈希值）

# 登录时：验证密码
bcrypt.checkpw(password.encode(), hashed)  # True
```

**关键概念：**
- **盐值 (Salt)**：随机字符串，附加到密码后再哈希。即使两个用户密码相同，哈希值也不同。
- **bcrypt**：专为密码设计的慢哈希算法。"慢"是优点——暴力破解成本高。
- **PBKDF2、scrypt、Argon2**：同类算法，Argon2 是目前推荐的最佳选择。
