## 0. 写在前面

**本节阅读顺序**：**§0.1 ASGI（协议与 `scope/receive/send`）→ §0.2 Starlette（ASGI 上的 Web 骨架）→ §0.3 FastAPI（类型、文档、Depends）**；然后 **§0.4** 起进入「为什么选它」与自检。

### 0.1 ASGI 到底是什么（和 WSGI 差在哪）

**本节概念**：**ASGI**（**Asynchronous Server Gateway Interface**，异步服务器网关接口）是一份 **Python Web 的「接线标准」**：规定 **ASGI 服务器**（如 **Uvicorn**）如何把**已解析好的连接信息**交给 **`app`**，以及 **`app`** 如何**异步地**按**事件流**把响应送回去。后面要讲的 **Starlette `Application`、FastAPI 的 `app`**，都是**符合这条约定**、可被 `uvicorn main:app` 加载的 **ASGI 应用**。

为什么要有 ASGI？先认识老一辈 **WSGI**（**Web** Server Gateway Interface）：

- **WSGI 心智模型（简化）**：服务器准备好一个 **`environ`（大字典，像请求说明书）**，调用你的 **`application(environ, start_response)`**；你的代码**同步**跑完，通过 `start_response` 写上状态码和头，再 **`return [body]`** 式的把正文交回去。  
- **它擅长什么**：经典同步 Web、大量现成中间件与部署习惯。  
- **它别扭在哪**：**长连接**（如 WebSocket）、**异步 `await`**、**请求体流式读**等，用「一次函数调用返回」模型去硬拧，不如来一套**事件驱动**的接口。

**ASGI 心智模型（白话）**：把你的 `app` 想成一个 **async 入口**，每次连接/请求，服务器会（概念上）调用：

```text
await app(scope, receive, send)
```

三个参数各管一摊（你手写 ASGI 时才直接接触；用 Starlette/FastAPI 时由框架代劳，但**懂这三个词读官方文档会顺很多**）：

| 参数 | 可以怎么理解 |
|------|----------------|
| **`scope`** | **「这次任务的说明书」**：字典里写明类型（HTTP / WebSocket）、路径、方法、头字段等**元信息**——类似 WSGI 的 `environ`，但更偏向**结构化 + 可扩展**。 |
| **`receive`** | **「异步取件口」**：`await receive()` 拿到**下一条进站事件**（例如 HTTP body 的一段、WebSocket 一帧）；可能有多条，所以要循环读。 |
| **`send`** | **「异步发货口」**：`await send({...})` 把**出站事件**推给服务器（例如 HTTP 响应起始行、某段 body）；也可分多次发。 |

**和后面 Starlette / FastAPI 有啥关系**：

- **Uvicorn** 负责 **`receive` / `send`** 与 TCP/HTTP 解析；它按 ASGI 调用你的 **`app(scope, receive, send)`**。  
- **§0.2 Starlette**、**§0.3 FastAPI** 做的事，是把 `scope / receive / send` 翻译成你熟悉的 **`Request`、路由表、端点函数`**——你几乎**永不手写**裸 ASGI。  
- **没有 ASGI 服务器挂着，`app` 只是内存里的对象**，浏览器访问不到——这和「import 了 Flask 但没跑 `flask run`」是同一类事实。  
- **WebSocket、SSE、HTTP/2** 等能力在协议层都落在「**事件流 + async**」上，这也是 ASGI 被现代框架选中的原因；本章主要讲 HTTP API，你只要先建立 **scope / receive / send** 的图像即可。

**一句防晕**：先有 **ASGI 这条「电源规格」**，再有 **Starlette / FastAPI 这类「整机」**；开发时你写的是整机 API，但排障时要能想到**最底下仍是 `await app(...)` 这一调用形状**。

### 0.2 Starlette 到底是什么（FastAPI 的「底盘」）

**本节概念**：**Starlette** 是一个 **轻量级 ASGI 框架 / 工具包**（包名 `starlette`）：在 **§0.1** 的 ASGI 接线之上，它实现 **HTTP 路由、Request/Response 对象、中间件链、WebSocket、静态文件、Session 等**「通用 Web 积木」。**FastAPI 不是从零造轮子**，而是站在 Starlette 这套**已经能接 Uvicorn、能消化 ASGI 事件**的骨架上，再叠类型、OpenAPI、Depends。

可以把它记成三句话：

1. **Starlette 管「Web 传输层长什么样」**：例如 **`Request` 里怎么取路径、Query、body 流**，**`Response` / `JSONResponse` 怎么写状态码和头**，**中间件怎么包一层 `call_next`**——你读 FastAPI 文档时，很多行为其实是在读 **Starlette**（两个项目名词会混用）。  
2. **Starlette 管「URL → 可调用对象」这条总线**：内部有 **路由器**；**§0.3** 里 `@app.get` 这类装饰器在 FastAPI 中最终会**落到 Starlette 风格的路由注册**上（不必深究源码，只要知道：**没有 Starlette 这一层，FastAPI 连一次 HTTP 匹配都做不了**）。  
3. **Starlette 不管「用类型标注自动校验 JSON」**：那是 **Pydantic + FastAPI** 的边界——纯 Starlette 里你往往**手写**校验或自己接库；FastAPI 把这一步**默认化**了。

**和 FastAPI 的分工（背这张表就够入门）**：

| 层次 | 典型由谁提供 | 你写代码时常见触点 |
|------|----------------|-------------------|
| **ASGI 入口** | Starlette 的 `Router` / `Application` 组合体（FastAPI 封装后仍是 ASGI app） | 一般**不手写** `scope, receive, send` |
| **Request / Response** | Starlette | `Request`、`JSONResponse`、`StreamingResponse` 等 |
| **中间件洋葱模型** | Starlette | `@app.middleware("http")`、`add_middleware` |
| **WebSocket** | Starlette | `@app.websocket`（FastAPI 暴露同名能力） |
| **OpenAPI / 校验 / Depends** | **FastAPI**（底层用 Pydantic） | `response_model`、`Body(...)`、`Depends()` |

**可以单独用 Starlette 吗？**可以。纯 Starlette 更小、更「积木化」，适合你想**完全自己掌控**数据校验与文档时；代价是 **OpenAPI 与类型驱动参数**要你自己拼。选 FastAPI 通常是：**我要 Starlette 的 ASGI 能力 + 我要省掉大量样板校验与文档**。

**一句防晕**：你 `pip install fastapi` 时依赖树里有 **`starlette`**——这是**真实依赖**；栈回溯里出现 `starlette/...py`，说明请求已经进到**底盘**。

### 0.3 FastAPI 到底是什么（读完 §0.1、§0.2 再读这一节）

**本节概念**：**FastAPI** 是一个 **Python 第三方库**（包名 `fastapi`）：它在 **Starlette（§0.2）** 提供的 **ASGI 路由与 Request/Response** 之上，用 **类型标注 + Pydantic** 把 **参数解析、校验、OpenAPI 文档、Depends** 拼成一套写 **HTTP/JSON API** 的体验。你构造 **`app = FastAPI()`**，用 **`@app.get` / `@app.post` 等**把「**哪个 URL + 哪种方法**」登记到「**哪个 Python 函数**」；**Uvicorn**（§0.1）监听端口并按 ASGI 调用这个 `app`，`app` 内部再走路由、依赖、校验、端点，最后把返回值变成 **HTTP 响应**（常见是 JSON）。

把它拆成你能指着说的几件事：

1. **它不是一门语言**，也不是操作系统自带工具——和 `requests`、`pandas` 一样，要 **`pip install`**。  
2. **它默认不是「替你监听端口的程序」**：监听端口、读 TCP、解析 HTTP，是 **ASGI 服务器**（常用 **Uvicorn**，见 **§0.1**）；FastAPI 提供的是 **`app` 这一层应用逻辑**。脑子里分层：**Uvicorn = 门面接单，Starlette = 通用 HTTP 骨架，FastAPI = 类型与文档这一层「点菜规则」**。  
3. **它建立在 Starlette 之上**（**§0.2**）：路由、Request/Response、中间件、WebSocket 等「Web 底座」多来自 Starlette；FastAPI 在上面加了 **依赖注入、类型驱动参数、OpenAPI、与 Pydantic 深度集成**。  
4. **它适合干什么**：**JSON/HTTP API**（前端、移动端、微服务、Agent 工具）；**不太适合**当你首要需求是 Django 那种 **Admin + ORM 迁移一条龙** 的全栈默认值——不是不能，而是别硬套。

和常见名字**粗对比**（只建立地图，不引战）：

| 名字 | 你脑子里可以怎么记 |
|------|-------------------|
| **Flask** | 老牌轻量框架，生态大；经典部署常走 **WSGI**（如 Gunicorn），异步故事和版本历史有关。 |
| **Django** | 全栈「电池自带」：ORM、模板、Admin；写 REST 常叠 **DRF**。 |
| **FastAPI** | 偏 **API 优先**：类型 + 自动文档 + **ASGI/async** 友好；不管数据库时也能很小。 |

你写下的最小「形状」长这样（先认字，不要求一次全懂）：

```python
from fastapi import FastAPI

app = FastAPI()  # ASGI 应用（§0.1）；内部含 Starlette 路由栈（§0.2）

@app.get("/ping")  # 登记：GET /ping → 下面这个函数（装饰器工厂，见 A1）
def ping():
    return {"ok": True}  # FastAPI 序列化为 JSON
```

- **`app = FastAPI()`**：创建一个可被 **`uvicorn main:app`** 加载的 **ASGI 应用**；`@app...` 往其上**注册路由元数据**。  
- **`@app.get("/ping")`**：导入时把 `ping` 登记进路由表（与 A1「装饰器 / 注册表」同类机制）。  
- **`return {"ok": True}`**：框架补 **`Content-Type: application/json`** 并序列化。

### 0.4 为什么需要「又一个 Web 框架」

**本节概念**：**FastAPI** 定位是**现代 Python API 框架**：默认 **OpenAPI**、默认 **类型驱动校验**、原生 **`async`** 友好；适合 **JSON API、Agent 工具网关、内部微服务**。它不是全栈模板引擎框架（那是另一类需求）。

- **和「手写 BaseHTTPRequestHandler」比**：你少写解析、少拼状态码、少维护路由表。
- **和「只返回 dict 的轻量库」比**：你多拿**自动生成 schema、依赖注入、可测试边界**。
- **代价**：概念面比「单文件 Flask」宽——本章把**最小必会子集**讲透，其余可查官方文档。

3. `add_middleware` 的**注册顺序**与**执行顺序**为什么相反？

### 0.7 本节小结（三张卡片）

- **一句话**：**Uvicorn** 按 **ASGI** 调 **`app`**；**Starlette** 撑起 **Request/路由/中间件**；**FastAPI** 叠 **类型 + OpenAPI + Depends**；**Pydantic** 扛 **数据校验与 schema**。  
- **典型场景**：对外 JSON API、对内服务、给 Agent/前端提供稳定契约。  
- **和 A1 类比**：路由装饰器像「**注册表 + 高阶函数**」：`@app.get` 把 **URL→函数** 登记进路由器；**`app` 本身**就是可被服务器 `await` 的那类 **ASGI 应用**。

## 1. FastAPI 核心架构

### 1.1 ASGI 与 Uvicorn

**本节概念**：这一节把 **§0.1 的 ASGI 抽象**落到「**你机器上跑起来的进程分工**」：**Uvicorn** 是 **ASGI Server**（监听端口、驱动事件循环、调用 `app`）；**FastAPI 的 `app`** 是 **ASGI Application**（内部含 **Starlette** 路由栈，**§0.2**）。若 **scope / receive / send** 仍晕，回到上文 **§0.1** 再看一遍表格。

```
客户端请求 → Uvicorn(ASGI Server) → FastAPI(ASGI Application) → 路由匹配 → 依赖注入 → 端点函数 → 响应
```

- **Uvicorn**：高性能 ASGI 服务器，基于 uvloop + httptools，负责网络 I/O
- **Starlette**：FastAPI 的底层框架，提供路由、中间件、WebSocket 等基础设施
- **FastAPI**：在 Starlette 之上增加了类型标注驱动的参数解析、依赖注入、自动文档生成

**常见误区（架构层）**：

- 把 **「安装了 uvicorn」** 当成 **「生产就绪」**：生产还要考虑 **多 worker、graceful shutdown、日志、TLS 终止位置**（见 **B2/B3**）。  
- 以为 **FastAPI 自己处理线程池**：同步 `def` 路由的线程池来自 **Starlette 的 `run_in_threadpool`**，理解这点才懂「为什么同步路由不会卡死整个进程」的边界。  
- 把 **OpenAPI 文档** 暴露到公网却不做 **鉴权/隔离**：`/docs` 也是 HTTP 资源，默认任何人能访问。

### 1.2 路由系统

**本节概念**：**路由**把 **HTTP 方法 + URL 路径（+ 可选 host 等约束）** 映射到 **Python 可调用对象**；`APIRouter` 用于**按业务分包**，通过 `include_router` 挂到同一 `app` 上，避免单文件无限膨胀。

FastAPI 的路由系统支持 **装饰器注册** 和 **APIRouter 模块化**：

```python
from fastapi import FastAPI, APIRouter

app = FastAPI()

# 直接注册
@app.get("/items/{item_id}")
async def read_item(item_id: int):
    return {"item_id": item_id}

# APIRouter 模块化（推荐）
router = APIRouter(prefix="/api/v1", tags=["users"])

@router.get("/users")
async def list_users():
    return []

app.include_router(router)
```

路由匹配规则（务必记住，排错时天天用）：

- 按注册顺序匹配，**先注册先匹配**——因此更**具体**的路径要写在更**通用**的路径**之前**（或拆到不同 router 并控制 `include_router` 顺序）。  
- **固定路径**优先于**路径参数**：`/users/me` 应能匹配到，而不会被 `/users/{user_id}` 误吃（仍建议显式排序避免歧义）。  
- 路径转换器：`{item_id:int}`、`{file_path:path}`（`path` 可含 `/`）等，本质是 **把字符串段解析成 Python 类型**；解析失败通常是 **404**（「没有这条路由」），而不是 422。  
- **同一 URL 不同方法**：`@app.get` 与 `@app.post` 可共存；`405 Method Not Allowed` 出现在「有路由、方法不对」时。  
- **Host / root_path**：网关反代场景常配合 `root_path` 或 `ProxyHeadersMiddleware`（细节见 Starlette 文档与 **B2** 代理一节）。

**常见误区（路由层）**：

- 在路径参数里塞 **业务规则**（例如用正则穷举所有合法 id）——可读性差；更干净的是 **命中路由后再在校验层拒绝**。  
- 忘记 **trailing slash** 与前端约定：是否重定向、是否严格匹配，和客户端 fetch 的 URL 必须一致。

### 1.3 请求生命周期

**本节概念**：**请求生命周期**描述「**字节进来 → Python 函数跑完 → 字节出去**」之间的**固定阶段**；中间件、依赖、校验、端点各自占据其中一段。画熟这张图，你读任何 ASGI 框架日志都能对号入座。

一个请求从进入到返回的完整流程：

```
1. Uvicorn 接收 TCP 连接
2. 解析 HTTP 请求，构造 ASGI scope
3. 进入中间件链（洋葱模型，从外到内）
4. 路由匹配
5. 解析依赖（依赖注入树，深度优先）
6. 参数校验（Pydantic 验证路径参数、查询参数、请求体）
7. 执行端点函数
8. 序列化响应（Pydantic 模型 → JSON）
9. 中间件链返回（从内到外）
10. Uvicorn 发送 HTTP 响应
```

关键点：

- **中间件**是洋葱模型，请求时从外到内，响应时从内到外。  
- **依赖注入**在路由匹配后、端点执行前完成；依赖里抛出的 `HTTPException` **通常**在进端点前就变成响应。  
- **Pydantic 校验失败**（Query/Body/Path 等）会直接走 **422** 分支，**端点函数不会被执行**——因此不要在端点里假设「已经校验过」之外再去信任客户端。

### 1.4 常见误区（生命周期）

- 在依赖里做**重 CPU** 计算且还用 `async def` 依赖——会占满事件循环；重计算应放到 **线程池** 或独立 worker。  
- 以为 **422 一定是「JSON 体错了」**：Query 缺字段、类型转换失败同样会 422，读 `detail` 里的 `loc` 才能定位。  
- **依赖里吞异常**：依赖应**快速失败**或**明确映射**到 HTTP 状态；默默 `return None` 常把 bug 推迟到业务层。

### 1.5 本节小结（三张卡片）

- **一句话**：请求 = **中间件洋葱 → 路由 → 依赖树 → 校验 → 端点 → 序列化 → 洋葱返回**。  
- **典型场景**：定位「卡在中间件 / 卡在依赖 / 卡在校验 / 卡在端点」四选一。  
- **和 A1 类比**：依赖树像 **多层装饰器嵌套**，只不过由框架按声明顺序组装。

## 2. 依赖注入（Dependency Injection）

**本节概念**：**依赖注入（DI）**在这里指：端点函数的参数默认值写成 `Depends(可调用对象)`，由框架在调用端点前**解析依赖树**并把返回值**注入参数**；用于 **DB Session、当前用户、分页参数、特性开关** 等横切能力。

依赖注入是 FastAPI 最强大的特性之一：把「**如何拿到 db / user**」从业务函数签名里抽走，使 **Router 薄、Service 可测**。

### 2.1 Depends() 基础

**本节概念**：`Depends(some_callable)` 声明「**本参数的值 = 调用 `some_callable` 的返回值**」；`some_callable` 本身可以再依赖别的 `Depends`，形成**有向无环**依赖树（不要写环）。

```python
from fastapi import Depends

def common_params(skip: int = 0, limit: int = 10):
    return {"skip": skip, "limit": limit}

@app.get("/items")
async def read_items(params: dict = Depends(common_params)):
    return params
```

依赖可以是：
- **普通函数**（def）
- **异步函数**（async def）
- **可调用类**（实现 `__call__` 的类）

### 2.2 嵌套依赖

**本节概念**：**嵌套依赖**即依赖函数的参数里再写 `Depends(...)`；框架按 **深度优先** 解析，**同一请求内**同一依赖可缓存（默认），避免重复打开连接。

依赖可以递归依赖其他依赖，FastAPI 自动解析依赖树：

```python
def get_db():
    db = SessionLocal()
    try:
        yield db          # yield 之前是"注入"，之后是"清理"
    finally:
        db.close()

def get_current_user(db: Session = Depends(get_db)):
    # 依赖 get_db
    user = db.query(User).first()
    return user

def get_admin_user(user: User = Depends(get_current_user)):
    # 依赖 get_current_user，间接依赖 get_db
    if not user.is_admin:
        raise HTTPException(403, "Not admin")
    return user

@app.get("/admin/dashboard")
async def admin_dashboard(admin: User = Depends(get_admin_user)):
    return {"msg": f"Welcome {admin.name}"}
```

### 2.3 数据库 Session 注入

**本节概念**：**`yield` 依赖**把依赖函数变成「**先分配、再交给端点、最后回收**」的三段式；`yield` 的值注入参数，`yield` 之后的代码（或 `finally`）做 **释放**，语义接近 **context manager**。

`yield` 依赖是管理资源生命周期的标准模式：

```python
from contextlib import contextmanager

def get_db():
    """每个请求获取独立的数据库 session，请求结束自动关闭"""
    db = SessionLocal()
    try:
        yield db
    finally:
        db.close()
```

- `yield` 之前的代码在请求处理前执行（获取资源）
- `yield` 之后的代码在响应发送后执行（释放资源）
- 即使端点抛出异常，`finally` 也会执行

### 2.4 认证依赖

**本节概念**：认证依赖通常返回 **当前主体（用户、服务账号）** 或抛 **401/403**；`OAuth2PasswordBearer` 一类工具负责从 **`Authorization: Bearer`** 取 token 字符串，**验签与查库**仍由你实现。

```python
from fastapi.security import OAuth2PasswordBearer

oauth2_scheme = OAuth2PasswordBearer(tokenUrl="token")

async def get_current_user(token: str = Depends(oauth2_scheme)):
    """从 Authorization: Bearer <token> 中提取并验证 token"""
    user = decode_token(token)
    if not user:
        raise HTTPException(401, "Invalid token")
    return user
```

### 2.5 测试：`dependency_overrides`

**本节概念**：`app.dependency_overrides` 在**测试或本地调试**时把某个依赖函数**整体替换**为假实现（假 DB、假用户），**无需改业务路由代码**；用完记得 `clear()` 避免污染其他用例。

```python
from fastapi.testclient import TestClient

def fake_user():
    return {"id": 1, "name": "tester"}

app.dependency_overrides[get_current_user] = fake_user
client = TestClient(app)
# ... 发请求 ...
app.dependency_overrides.clear()
```

依赖注入的核心优势：

- **可测试性**：`dependency_overrides` 替换依赖，比全局打猴子补丁更干净。  
- **复用性**：同一依赖被多个端点引用，**改一处生效**。  
- **组合性**：依赖可嵌套，注意**缓存与作用域**（同一请求内同一依赖默认可缓存，避免重复打开连接——见官方 `use_cache` 说明）。

**常见误区（依赖层）**：

- 在 `yield` 依赖的 **`finally` 之前** `raise` 却不理解清理顺序——要确认资源是否总能关闭。  
- 把 **重业务逻辑** 全塞进依赖——依赖宜**薄**；复杂规则放 **Service**。  
- 忘记测试里 **clear overrides** 导致「单测通过、整包失败」的诡异顺序问题。

### 2.6 本节小结（三张卡片）

- **一句话**：`Depends` = **声明式装配线**，`yield` = **带清理的资源注入**。  
- **典型场景**：DB Session、当前用户、分页、特性开关、审计上下文。  
- **和 A1 类比**：依赖函数就是**返回配置好的可调用对象或上下文**的高阶函数组合。

## 3. 中间件

**本节概念**：**中间件**包装整个应用：在 **ASGI 层**对 `request` / `response` 做横切逻辑（日志、鉴权头、限流、安全头）；执行顺序呈 **洋葱**：请求从外向内，响应从内向外。

中间件在请求到达路由之前和响应返回之前执行；与 **依赖** 的差别是：中间件**看不到**「将要匹配哪条路由」之前的业务参数，适合做**纯 HTTP 层**的事。

### 3.1 CORS 中间件

**本节概念**：**CORS** 由浏览器强制执行；`CORSMiddleware` 负责在 **响应头** 上补 `Access-Control-*`，使浏览器允许跨域读响应。它**不替代**服务端鉴权。

```python
from fastapi.middleware.cors import CORSMiddleware

app.add_middleware(
    CORSMiddleware,
    allow_origins=["http://localhost:3000"],  # 允许的前端域名
    allow_credentials=True,
    allow_methods=["*"],
    allow_headers=["*"],
)
```

### 3.2 自定义中间件

**本节概念**：自定义中间件要么用 **`@app.middleware("http")`** 包一层 `call_next`，要么继承 **`BaseHTTPMiddleware`** 实现 `dispatch`；前者适合打日志，后者适合需要**类状态**（计数器、配置）时。

两种写法：

**装饰器写法**（简单场景）：
```python
@app.middleware("http")
async def log_requests(request: Request, call_next):
    start = time.time()
    response = await call_next(request)
    duration = time.time() - start
    print(f"{request.method} {request.url.path} - {duration:.3f}s")
    return response
```

**类写法**（复杂场景，推荐）：
```python
from starlette.middleware.base import BaseHTTPMiddleware

class RateLimitMiddleware(BaseHTTPMiddleware):
    def __init__(self, app, max_requests: int = 100):
        super().__init__(app)
        self.max_requests = max_requests
        self.requests = {}

    async def dispatch(self, request: Request, call_next):
        client_ip = request.client.host
        # 限流逻辑...
        response = await call_next(request)
        return response
```

### 3.3 中间件执行顺序

```python
app.add_middleware(A)  # 最后添加 = 最先执行
app.add_middleware(B)
app.add_middleware(C)  # 最先添加 = 最后执行

# 请求顺序：A → B → C → 路由
# 响应顺序：C → B → A → 客户端
```

注意：`add_middleware` 的添加顺序和执行顺序是**相反**的。最后添加的中间件最先接触到请求。

### 3.3 安全相关中间件（了解即可）

**本节概念**：生产环境常在更外层（Nginx / API Gateway）做 TLS 与限流；应用内可补充 **`TrustedHostMiddleware`**（校验 `Host` 头）等，减少 **Host 头注入 / DNS rebinding** 一类风险。具体配置以你部署拓扑为准。

**常见误区（中间件）**：

- 在中间件里做**重 CPU / 阻塞 IO** 且还用同步调用阻塞事件循环——应 `await` 异步 IO 或丢线程池。  
- CORS `allow_origins=["*"]` 且 **`allow_credentials=True`**——浏览器规则不允许，配置会无效或难以排错。  
- 以为中间件能**替代**业务鉴权：中间件只能看到 HTTP 层信息，**业务权限**仍在依赖/Service。

### 3.4 本节小结（三张卡片）

- **一句话**：中间件 = **HTTP 洋葱**；**后进先出**接触请求。  
- **典型场景**：Request-ID、耗时日志、CORS、可信 Host。  
- **和 A1 类比**：像给函数**外包一层 wrapper**，只是这层在 **ASGI** 入口统一生效。

## 4. Pydantic V2 深度

**本节概念**：**Pydantic** 用 Python 类型标注描述数据结构，在 **实例构造时**做校验与转换；FastAPI 把 **JSON 请求体 / 表单 / 查询串** 映射进模型，把 **响应模型** 序列化成 JSON。V2 校验核心用 Rust 重写，**错误模型与 API** 与 V1 有差异，读文档时务必确认主版本。

Pydantic V2 是 FastAPI 的数据校验核心：用 Rust 重写了校验层，常见工作负载下相对 V1 有显著性能提升（具体数字随场景波动，不必死记）。

### 4.1 模型定义与字段校验

**本节概念**：`BaseModel` 子类的**属性注解**即 schema；`Field()` 附加 **约束、默认值、文档元数据**，并进入 **OpenAPI**。

```python
from pydantic import BaseModel, Field

class User(BaseModel):
    name: str = Field(..., min_length=2, max_length=50, description="用户名")
    age: int = Field(ge=0, le=150, description="年龄")
    email: str = Field(pattern=r'^[\w.-]+@[\w.-]+\.\w+$')
    tags: list[str] = Field(default_factory=list)
```

`Field()` 参数：
- `...`（Ellipsis）：必填字段
- `default` / `default_factory`：默认值
- `min_length` / `max_length`：字符串长度
- `ge` / `gt` / `le` / `lt`：数值范围
- `pattern`：正则校验
- `description`：写入 OpenAPI schema

### 4.2 嵌套模型

**本节概念**：字段类型可以是另一个 **`BaseModel`** 或 **`list[Model]`**；校验时**递归**进入子模型，错误 `loc` 会带 **嵌套路径**（如 `body.company.address.city`）。

```python
class Address(BaseModel):
    city: str
    street: str

class Company(BaseModel):
    name: str
    address: Address  # 嵌套

class UserProfile(BaseModel):
    user: User
    company: Company | None = None  # 可选嵌套
    scores: list[float]             # 列表
```

### 4.3 自定义验证器

**本节概念**：**`field_validator`** 管**单字段**（可 `before`/`after`）；**`model_validator`** 管**跨字段**关系；抛 **`ValueError`** 会进入 Pydantic 错误链，最终常体现为 **422**。

Pydantic V2 使用 `field_validator` 和 `model_validator`：

```python
from pydantic import field_validator, model_validator

class Order(BaseModel):
    price: float
    quantity: int
    discount: float = 0.0

    @field_validator("price")
    @classmethod
    def price_must_be_positive(cls, v):
        if v <= 0:
            raise ValueError("价格必须大于0")
        return v

    @model_validator(mode="after")
    def check_discount(self):
        if self.discount > self.price * self.quantity:
            raise ValueError("折扣不能超过总价")
        return self
```

- `field_validator`：校验单个字段，`mode="before"` 在类型转换前执行，默认 `mode="after"`
- `model_validator`：校验整个模型，可以访问多个字段做交叉校验

### 4.4 模型继承

**本节概念**：用 **Base / Create / Update / Response** 拆模型，避免「创建时要密码、响应时绝不能带出密码」混在同一个类里；`from_attributes=True` 让 **ORM → Response** 有明确出口。

```python
class UserBase(BaseModel):
    name: str
    email: str

class UserCreate(UserBase):
    password: str          # 创建时需要密码

class UserResponse(UserBase):
    id: int                # 响应时返回 id
    created_at: datetime

    model_config = ConfigDict(from_attributes=True)  # 支持 ORM 模型转换
```

这种继承模式是 FastAPI 最佳实践：Base → Create/Update → Response。

### 4.5 Schema 生成（AI Function Calling）

**本节概念**：**`model_json_schema()`** 产出符合 JSON Schema 的字典，可供 **OpenAPI、客户端生成、LLM tools** 消费；`Field(description=...)` 会进 schema，等于写「给机器读的注释」。

Pydantic 模型可以直接导出 JSON Schema，非常适合作为 LLM Function Calling 的工具定义：

```python
class SearchParams(BaseModel):
    """搜索互联网"""
    query: str = Field(description="搜索关键词")
    max_results: int = Field(default=5, description="最大结果数")

schema = SearchParams.model_json_schema()
# 输出标准 JSON Schema，可直接用于 OpenAI/Anthropic Function Calling
```

这是 Pydantic 在 AI 工程中最重要的应用之一：把「**工具入参 schema**」与「**运行时校验**」绑在同一份模型上。

### 4.6 `model_config` 与 ORM 输出

**本节概念**：`model_config = ConfigDict(from_attributes=True)`（V2）允许从 **SQLAlchemy 等 ORM 实例**构造响应模型（旧名 `orm_mode`）；这样 Router 返回 ORM 对象时也能被 **`response_model`** 过滤字段，避免把密码哈希泄漏到 JSON。

### 4.7 常见误区（Pydantic）

- 把 **`dict`** 和 **`BaseModel`** 混用：需要 **`.model_dump()` / `.model_dump_json()`** 与纯 dict 互转时心里要有数。  
- 在 **`@field_validator`** 里做 IO（查库）——能跑但难测；更干净的是 **`model_validator`** + Service，或依赖注入层先查好。  
- **可变默认值**：`list` / `dict` 字段用 **`default_factory`**，不要写 `tags: list[str] = []`。

### 4.8 本节小结（三张卡片）

- **一句话**：Pydantic = **运行时契约**；FastAPI 用它接请求、出响应、吐 schema。  
- **典型场景**：请求体校验、响应裁剪、导出 JSON Schema 给 LLM 工具。  
- **和 A1 类比**：`BaseModel` 像 **带校验的 dataclass++**，错误体像 **`ValidationError` 结构化异常**。

## 5. 异步路由

**本节概念**：FastAPI 运行在 **asyncio** 事件循环上；**`async def` 路由**里应 **`await` 非阻塞 IO**；**`def` 路由**会被放到 **线程池**执行，避免阻塞循环。选错模型会导致「并发上不去 / 延迟毛刺」。

### 5.1 async def vs def

**本节概念**：是否写 `async def` 由 **函数体内是否 await 以及 IO 是否异步友好**决定；**同步阻塞**（`time.sleep`、`requests.get`、同步 DB 驱动）不要直接出现在 `async def` 里。

```python
# 异步路由 — 适用于 I/O 密集（网络请求、数据库、文件）
@app.get("/async")
async def async_route():
    data = await httpx.AsyncClient().get("https://api.example.com")
    return data.json()

# 同步路由 — FastAPI 会自动放到线程池执行，不会阻塞事件循环
@app.get("/sync")
def sync_route():
    time.sleep(1)  # 不会阻塞其他请求
    return {"msg": "done"}
```

关键规则：

- 如果函数内部用了 `await`，**必须**用 `async def`。  
- 如果函数内部有阻塞操作（如同步数据库驱动），**用 `def`**（FastAPI 自动放线程池）。  
- **千万不要**在 `async def` 里调用长时间阻塞操作——这会卡住**整个**事件循环，其它请求一起挨饿。  
- **`asyncio.create_task` 后台 fire-and-forget**：与 `BackgroundTasks`、进程外队列的职责不同，别混用导致「请求结束任务被掐掉」类问题。

### 5.2 异步数据库操作

**本节概念**：异步 SQLAlchemy（`AsyncSession` + `await session.execute(...)`）让 **DB IO 与事件循环协作**；仍要注意 **连接池大小、慢查询、事务持有时间**，否则 async 也救不了。

```python
from sqlalchemy.ext.asyncio import AsyncSession, create_async_engine

engine = create_async_engine("sqlite+aiosqlite:///db.sqlite3")

async def get_db():
    async with AsyncSession(engine) as session:
        yield session

@app.get("/users")
async def list_users(db: AsyncSession = Depends(get_db)):
    result = await db.execute(select(User))
    return result.scalars().all()
```

### 5.3 应用生命周期：`lifespan`（了解）

**本节概念**：需要在进程级 **启动/关闭** 时初始化连接池、订阅消息等，使用 **`@asynccontextmanager` + `FastAPI(lifespan=...)`**（取代旧式 `on_event`）；避免在模块 import 副作用里悄悄建全局连接，否则多 worker / 热重载下难排错。

### 5.4 本节小结（三张卡片）

- **一句话**：`async def` = **协作式多任务**；阻塞 = **要么线程池（`def`）要么挪出事件循环**。  
- **典型场景**：httpx AsyncClient、asyncpg、异步 ORM；同步遗留库用 `def` 路由包一层。  
- **和 A3 衔接**：事件循环、`run_in_executor`、与进程外队列的分工在 **A3 — 并发与异步**展开。

---

## 6. 后台任务

**本节概念**：**`BackgroundTasks`** 在 **响应发送之后** 于**同一进程**内顺序执行轻量函数；适合 **发通知、写审计**。**重任务**应交给 **Celery / arq / 云队列**（见 **B4**）。

`BackgroundTasks` 用于在响应返回后执行非紧急任务（发邮件、写日志等）：

```python
from fastapi import BackgroundTasks

def send_email(to: str, subject: str):
    """耗时操作，在后台执行"""
    time.sleep(5)  # 模拟发送
    print(f"邮件已发送给 {to}")

@app.post("/register")
async def register(bg: BackgroundTasks):
    # 立即返回响应
    bg.add_task(send_email, "user@example.com", "欢迎注册")
    return {"msg": "注册成功"}
```

特点：

- 响应先返回给客户端，后台任务在**同一请求上下文清理之后**运行（不要假设仍能安全使用已关闭的 DB 连接）。  
- 后台任务按添加顺序 **串行执行**，不是线程池并行队列。  
- 适合轻量任务；重任务应用 **Celery / arq** 等任务队列。

**常见误区**：在 BackgroundTasks 里做**长事务 / 可失败重试的业务**——客户端已断开，失败难以对齐用户体验；应 **异步任务系统 + 状态查询接口**。

### 6.1 本节小结（三张卡片）

- **一句话**：BackgroundTasks = **「先回答用户，再慢慢干活」的进程内队列**。  
- **典型场景**：写日志、发邮件、触发非关键下游。  
- **和 B4 类比**：要 **削峰 / 重试 / 可观测**，请用消息队列，而不是堆 BackgroundTasks。

## 7. 错误处理

**本节概念**：FastAPI 默认把 **`HTTPException`** 映射为 HTTP 响应；把 **`RequestValidationError`** 映射为 **422**；你也可以注册 **`@app.exception_handler`** 做统一错误体（注意**不要吞掉**需要透传的头部，如 CORS）。

### 7.1 HTTPException

**本节概念**：`HTTPException` 是 **早退控制流**：在任意依赖或端点里 `raise`，框架拦截并转成响应；`detail` 可字符串或可序列化结构（会进 JSON）。

```python
from fastapi import HTTPException

@app.get("/items/{item_id}")
async def read_item(item_id: int):
    if item_id not in db:
        raise HTTPException(
            status_code=404,
            detail="Item not found",
            headers={"X-Error": "not-found"}  # 可选的自定义 header
        )
```

### 7.2 自定义异常处理器

```python
class BusinessError(Exception):
    def __init__(self, code: str, message: str):
        self.code = code
        self.message = message

@app.exception_handler(BusinessError)
async def business_error_handler(request: Request, exc: BusinessError):
    return JSONResponse(
        status_code=400,
        content={
            "success": False,
            "error_code": exc.code,
            "message": exc.message
        }
    )
```

### 7.3 覆盖默认的 422 验证错误

**本节概念**：自定义 `RequestValidationError` 处理器时，建议**保留** `exc.errors()` 里的 `loc` / `type` 字段，方便前端定位；不要为了「好看」丢掉结构化信息。

```python
from fastapi.exceptions import RequestValidationError

@app.exception_handler(RequestValidationError)
async def validation_handler(request, exc):
    return JSONResponse(
        status_code=422,
        content={
            "success": False,
            "error_code": "VALIDATION_ERROR",
            "message": "请求参数校验失败",
            "details": exc.errors()
        }
    )
```

### 7.4 异常处理器顺序（实践）

**本节概念**：**更具体的异常**应先注册；`Exception` 总兜底放最后。不要把业务 `ValueError` 与 Pydantic 校验混在同一个大兜底里 unless 你非常清楚转换规则。

### 7.5 本节小结（三张卡片）

- **一句话**：`HTTPException` 管 **HTTP 语义**；`exception_handler` 管 **统一错误信封**。  
- **典型场景**：业务错误码、国际化 message、对齐前端表格校验。  
- **常见坑**：处理器里 **`return` 了非 Response** 或忘记设置 **status_code**。

## 8. OpenAPI / Swagger 自动文档

**本节概念**：FastAPI 根据 **路由、类型标注、`response_model`、Field(description=...)** 生成 **OpenAPI 3** JSON；**Swagger UI / ReDoc** 只是这份 schema 的浏览壳。生产环境常对 `/docs` 做 **鉴权或关闭**。

FastAPI 自动生成 OpenAPI 3.0 规范文档。

### 8.1 三类入口：`/docs`、`/redoc`、`/openapi.json`

**本节概念**：**`/openapi.json`** 是**规范本体**（JSON）；**`/docs`（Swagger UI）** 与 **`/redoc`** 是**渲染器**，方便人类调试。自动化流水线应消费 **`openapi.json`**，而不是去爬 HTML。

- **Swagger UI**：默认 `http://localhost:8000/docs`（交互式「试一试」）。  
- **ReDoc**：默认 `http://localhost:8000/redoc`（偏只读阅读）。  
- **OpenAPI JSON**：默认 `http://localhost:8000/openapi.json`。

### 8.2 应用级元数据与路由级注解

**本节概念**：`FastAPI(...)` 构造函数参数影响**全局**文档块（title、tags 分组说明等）；单个 `@app.get(..., summary=..., responses=...)` 影响**该路由**在 schema 里的条目与示例状态码。

自定义文档信息：

```python
app = FastAPI(
    title="My API",
    description="API 描述",
    version="1.0.0",
    docs_url="/docs",         # Swagger 路径，设为 None 可禁用
    redoc_url="/redoc",
    openapi_tags=[
        {"name": "users", "description": "用户管理"},
        {"name": "items", "description": "物品管理"},
    ]
)
```

路由的文档增强（与 `response_model` 一起构成「契约」的主视图）：

```python
@app.get(
    "/items/{item_id}",
    response_model=ItemResponse,     # 响应模型（自动生成 schema）
    summary="获取物品详情",
    description="根据 ID 获取物品的完整信息",
    tags=["items"],
    responses={404: {"description": "物品不存在"}}
)
```

### 8.3 本节小结（三张卡片）

- **一句话**：OpenAPI = **机器可读的 API 契约**；人类用 `/docs` 看。  
- **典型场景**：前后端并行、生成客户端 SDK、给 LLM 工具描述。  
- **安全**：公网 **勿裸开文档** 除非你有意识暴露。

## 9. 项目结构最佳实践

**本节概念**：目录结构服务于 **可测试与可替换**：把 **HTTP 层（Router）**、**业务（Service）**、**持久化（Repository/ORM）**、**契约（Schema）** 分开，依赖注入做粘合。

### 小型项目

```
project/
├── main.py              # FastAPI app 入口
├── models.py            # Pydantic 模型
├── database.py          # 数据库连接
└── requirements.txt
```

### 中大型项目（推荐）

```
project/
├── app/
│   ├── __init__.py
│   ├── main.py          # FastAPI app 创建、中间件注册、router 挂载
│   ├── config.py        # 配置（用 pydantic-settings）
│   ├── database.py      # 数据库引擎、session 工厂
│   ├── dependencies.py  # 公共依赖（get_db, get_current_user）
│   ├── exceptions.py    # 自定义异常 + 异常处理器
│   ├── middleware.py    # 自定义中间件
│   ├── models/          # SQLAlchemy ORM 模型
│   │   ├── user.py
│   │   └── item.py
│   ├── schemas/         # Pydantic 模型（请求/响应）
│   │   ├── user.py
│   │   └── item.py
│   ├── routers/         # 路由模块
│   │   ├── users.py
│   │   └── items.py
│   └── services/        # 业务逻辑层
│       ├── user_service.py
│       └── item_service.py
├── tests/
│   ├── conftest.py
│   ├── test_users.py
│   └── test_items.py
├── alembic/             # 数据库迁移
├── requirements.txt
└── Dockerfile
```

### 分层原则

```
Router（路由层）
  ↓ 调用
Service（业务逻辑层）
  ↓ 调用
Repository / ORM（数据访问层）
```

- **Router**：只做参数接收和响应返回，不写业务逻辑
- **Service**：业务逻辑，可被多个 Router 复用
- **Schema**（Pydantic 模型）分 `Create`、`Update`、`Response` 三类
- **依赖注入**放在 `dependencies.py`，被所有 Router 共享

### 9.1 本节小结（三张卡片）

- **一句话**：Router 薄、Service 厚、Schema 管契约、依赖管资源。  
- **典型场景**：同一业务被 **HTTP + CLI + 定时任务** 复用时只拷 Service。  
- **反模式**：Router 里直接堆 SQL、在 schema 里写业务规则。

## 11. 面试考点速查

1. **FastAPI 和 Starlette / Pydantic 的分工？**  
   - Starlette：HTTP、路由、中间件、Request/Response；Pydantic：数据模型与校验；FastAPI：把二者用类型标注粘起来并生成 OpenAPI。  
2. **`Depends` 与中间件的区别？**  
   - 中间件偏传输层横切；依赖偏业务资源与上下文，且能利用类型系统做树状组合。  
3. **同步路由 `def` 为什么能避免阻塞事件循环？**  
   - Starlette 在线程池执行同步函数；并发受线程池与 GIL 约束，与 async 不同维度的扩展。  
4. **`HTTPException` 与 `RequestValidationError` 的默认行为？**  
   - 前者任意状态码；后者默认映射 422 且带 `errors()` 列表。  
5. **如何隐藏或保护 `/docs`？**  
   - `docs_url=None` 或在前置网关加鉴权；勿误以为「没有链接就没人访问」。  
6. **`response_model` 的作用？**  
   - 过滤输出字段、生成响应 schema、驱动文档；减少「多返回敏感字段」事故。
