## 0. 写在前面：为什么要花时间学网络

### 0.1 一个你每天都在做但从没想过的动作

你打开浏览器，输入 `https://www.google.com`，按下回车。0.5 秒之后 Google 的首页出现了。

这 0.5 秒内发生了什么？粗略数一下：

1. 你的电脑问"Google 这个名字对应哪个 IP 地址？" —— **DNS 解析**
2. 电脑和 Google 的服务器建立了一条"线路" —— **TCP 三次握手**
3. 双方协商用什么密码说话，交换了密钥 —— **TLS 握手**
4. 你的电脑说了一句"请把首页给我" —— **HTTP 请求**
5. Google 的服务器说了一句"给你，这是 HTML" —— **HTTP 响应**
6. 你的浏览器把 HTML 画出来

这六步里，任何一步卡住、错了、被拦截，你都会看到熟悉的"网页打不开"。而作为后端工程师，你的工作就是**写那个回答请求的服务器**——它必须在 0.5 秒内完成所有这些动作。

### 0.2 你可能已经踩过的坑

下面这些问题，如果有任何一个你回答不上来，都说明网络基础需要补：

- 为什么我本地跑 `localhost:8000` 可以，别人用我的 IP 就打不开？
- 前端说"CORS 错误"，为什么后端明明返回了 200，浏览器还是报错？
- 为什么 POST 请求偶尔会出现两次，GET 却不会？
- 为什么 `https` 比 `http` 安全？自签名证书为什么浏览器会警告？
- 接口 504 Gateway Timeout，到底是谁超时了？我的代码还是 nginx？
- 调 OpenAI API 偶尔会断流，为什么需要配 `timeout` 和 `retry`？

这些问题的答案都在本章。

### 0.3 学完本章你能做什么

- **看懂报文**：用 `curl -v` 或浏览器 DevTools 看一个请求的完整过程，能指出每个字段的含义
- **调试接口**：接口出错时，能根据状态码、响应头迅速定位是前端、后端、网关还是网络问题
- **设计 API**：能写出符合 REST 规范、可维护的接口

- **过面试**：TCP 三次握手、HTTPS 加密流程、JWT 这些高频题目全部拿下

## 1. 网络分层模型：快递系统的 OSI 与 TCP/IP

### 1.1 Why：为什么要分层

想象你要寄一个包裹给南京的朋友。整个流程大致是：

1. 你写一封信，装进信封，写上"南京市鼓楼区 XX 路 123 号"
2. 你把信封拿到顺丰网点
3. 网点把你的信和其他很多信一起装进一个大包裹，贴上"南京分拨中心"的标签
4. 这个大包裹被装上卡车，卡车走高速路到南京
5. 南京分拨中心拆开大包裹，把你的信单独拿出来
6. 快递员骑电动车送到朋友家

这里有三个很关键的观察：

**观察一：每一层只关心自己的任务**
- 你只关心"地址写对没有"，不关心卡车走哪条高速
- 卡车司机只关心"到南京分拨中心"，不关心包裹里是信还是文件
- 快递员只关心"这封信该送到哪一户人家"，不关心信的内容

**观察二：每一层只和相邻层打交道**
- 你只和网点打交道，不会直接跟卡车司机说话
- 卡车司机只对接分拨中心，不直接对接收件人

**观察三:换掉其中一层不影响别的层**
- 卡车换成飞机、高铁，你写信的方式不变
- 你把信封换成快递盒，卡车照常跑

网络也是一样的。数据从你的浏览器到 Google 的服务器，中间要经过网卡、路由器、光纤、海底电缆……如果不分层，每换一个网络设备都要改所有代码，根本没法维护。所以**前人把网络通信切成若干层，每层只做一件事，每层之间接口固定**。这就是"网络分层模型"。

### 1.2 What：OSI 七层模型

OSI（Open Systems Interconnection）是国际标准化组织在 1984 年提出的参考模型，它把网络通信分成 7 层。从上到下是：

```
┌───────────────────────────────────┐
│ 7. 应用层 Application             │  你看得到的"业务": HTTP, FTP, SMTP, DNS
├───────────────────────────────────┤
│ 6. 表示层 Presentation            │  数据格式转换、加密压缩 (TLS 逻辑上在这)
├───────────────────────────────────┤
│ 5. 会话层 Session                 │  会话管理、断点续传
├───────────────────────────────────┤
│ 4. 传输层 Transport               │  端到端可靠传输: TCP, UDP
├───────────────────────────────────┤
│ 3. 网络层 Network                 │  路由和寻址: IP, ICMP
├───────────────────────────────────┤
│ 2. 数据链路层 Data Link           │  相邻节点间传输: Ethernet, Wi-Fi, ARP
├───────────────────────────────────┤
│ 1. 物理层 Physical                │  电信号、光信号、电磁波
└───────────────────────────────────┘
```

**类比映射回快递系统：**

| OSI 层 | 快递类比 | 做什么 |
|--------|----------|--------|
| 应用层 | 你写信的内容 | 业务数据本身 |
| 表示层 | 信封的格式、把中文翻译成英文 | 数据编码/加密 |
| 会话层 | "这是给同一个收件人的系列邮件" | 会话管理 |
| 传输层 | 顺丰 vs 平邮，要不要签收 | 可靠性保证 |
| 网络层 | 省内 / 跨省的路由规划 | 选择路径 |
| 数据链路层 | 同一个分拨中心内部的传送带 | 点对点传输 |
| 物理层 | 电动车、飞机、卡车 | 物理媒介 |

### 1.3 What：TCP/IP 四层模型（工程中真正使用的）

OSI 是教科书上的模型，好看但过于学术。工程上实际用的是 **TCP/IP 四层模型**（也叫五层混合模型）：

```
┌─────────────────────────────────┐
│ 应用层 Application              │  HTTP, WebSocket, gRPC, DNS, SSH
│ (OSI 5+6+7)                     │
├─────────────────────────────────┤
│ 传输层 Transport                │  TCP, UDP, QUIC
├─────────────────────────────────┤
│ 网络层 Network / Internet       │  IPv4, IPv6, ICMP
├─────────────────────────────────┤
│ 链路层 Link                     │  Ethernet, Wi-Fi, PPP
└─────────────────────────────────┘
```

TCP/IP 把 OSI 的 5、6、7 层合并成了"应用层"，因为实践中根本分不清。以 HTTPS 为例：TLS 算哪一层？从功能看像"表示层"，但它其实是运行在应用和传输之间的一个库，不纯粹属于任何一层。硬分没意义。

**本文后续讨论都用 TCP/IP 四层**，但你面试时要知道 OSI 是 7 层。

### 1.4 How：数据封装与解封装

当你发送一个 HTTP 请求时，数据从上往下一路"加头"，到网卡上变成一堆二进制：

```
你的 HTTP 请求报文
   "GET / HTTP/1.1\r\nHost: google.com\r\n\r\n"
           │
           ▼
   [应用层: HTTP]  这就是原始数据 (payload)
           │
           ▼
   [传输层: TCP]   [TCP 头 | HTTP 数据]   加 20 字节 TCP 头(源端口/目的端口/序列号...)
           │
           ▼
   [网络层: IP]    [IP 头 | TCP 头 | HTTP 数据]   加 20 字节 IP 头(源IP/目的IP)
           │
           ▼
   [链路层: 以太网] [以太网头 | IP 头 | TCP 头 | HTTP 数据 | 以太网尾]
           │
           ▼
   物理层 ~~~~~~~ 比特流 0101010101010... ~~~~~~~>
```

对方收到后，从下往上"剥头"：以太网层剥掉以太网头 → 交给 IP 层剥 IP 头 → 交给 TCP 层剥 TCP 头 → 最后交给应用层处理原始 HTTP 数据。

**每一层只看自己的头**。TCP 不关心 HTTP 内容是什么，IP 不关心 TCP 的内容是什么。这就是"每层只关心本层协议"的工程含义。

### 1.5 常见误区

**误区一：以为 HTTP 和 TCP 是并列的两个东西**
错。HTTP 运行在 TCP 之上，HTTP 的每个字节最终都会被 TCP 封装。没有 TCP（或 QUIC）就没有 HTTP。

**误区二：以为每层都有"协议头"加起来开销很大**
确实有，但开销只在发送开始和接收最后。中间的路由器只看 IP 头就够了，不会拆 TCP 头。

**误区三：以为"应用层 = 写代码的那一层，我不用懂下面"**
错。你写的 HTTP 客户端偶尔出现的"Connection reset"、"Timeout"、"SSL handshake failed"都是下面几层的错误冒上来。不懂下面，排错时就只能瞎猜。

## 2. IP、端口、MAC — 互联网的三层地址

### 2.1 Why：为什么需要三种地址

继续用快递类比。要把一封信送到朋友手里，你需要：

1. **小区地址**（"XX 小区 3 栋"）：把信送到对的小区 —— 对应 **IP 地址**（哪台机器）
2. **门牌号**（"301 室"）：小区里有很多住户，要送到对的那户 —— 对应 **端口**（机器上哪个程序）
3. **身份证号**（"这封信是给张三本人的"）：门牌号认门不认人，身份证号确认收件人 —— 对应 **MAC 地址**（网卡的物理标识）

现实中你写快递只写前两个（小区+门牌），MAC 地址是快递员（路由器）内部用的，你感知不到。下面分别讲。

### 2.2 IP 地址：网络层的门牌号

**IPv4** 是 32 位整数，写成 `a.b.c.d` 的四段点分形式，每段 0-255：

```
192.168.1.100
   │   │  │  │
 8位 8位 8位 8位   一共 32 位
```

32 位意味着总共有 `2^32 ≈ 43 亿` 个地址。听起来很多，但全球网民超过 50 亿，设备比人多，早就不够用了。所以有了 **IPv6**（128 位），地址看起来像这样：

```
2001:0db8:85a3:0000:0000:8a2e:0370:7334
```

目前国内大部分家宽默认是 IPv4 + NAT（一会讲）。

**特殊 IP：**

| IP 段 | 含义 |
|-------|------|
| `127.0.0.1` | 本机回环地址（localhost），只能自己访问自己 |
| `10.0.0.0/8` | 私有网络（A 类） |
| `172.16.0.0/12` | 私有网络（B 类） |
| `192.168.0.0/16` | 私有网络（C 类），家用路由器常用 |
| `0.0.0.0` | "所有 IP"，服务端 `bind("0.0.0.0")` 表示监听所有网卡 |
| `255.255.255.255` | 广播地址 |

**子网掩码与 CIDR：**

`192.168.1.0/24` 中的 `/24` 叫 CIDR，表示前 24 位是网络号，后 8 位是主机号。所以这个子网里有 `2^8 = 256` 个地址（去掉网络号和广播号还剩 254 个可用）。

```python
# Python 计算 CIDR
import ipaddress

net = ipaddress.ip_network("192.168.1.0/24")
print(net.num_addresses)   # 256
print(net.network_address) # 192.168.1.0
print(net.broadcast_address) # 192.168.1.255
print(list(net.hosts())[:3])  # 前 3 个可用主机
# [IPv4Address('192.168.1.1'), IPv4Address('192.168.1.2'), IPv4Address('192.168.1.3')]

ip = ipaddress.ip_address("192.168.1.100")
print(ip in net)  # True
```

上面这段代码里 `ipaddress` 是 Python 标准库（不用装），专门处理 IP 的各种运算。`ip_network("192.168.1.0/24")` 会解析出这个子网的所有属性，`ip in net` 判断一个 IP 是否属于这个子网——实际工作中判断"是否是内网 IP"就是这么写。

### 2.3 NAT：为什么你家里所有设备共享一个外网 IP

你手机、笔记本、电视都连家里的 Wi-Fi，它们的 IP 都是 `192.168.x.x` 这种私有地址。但家里只有一个外网 IP（运营商给的），它们怎么同时上网？

答案是 **NAT（Network Address Translation）**。家用路由器会在出站包上做一次翻译：

```
你的笔记本 (192.168.1.5:51234) ─┐
你的手机  (192.168.1.6:50000) ─┼─路由器─> 外网 (公网IP:6xxxx)
你的电视  (192.168.1.7:49999) ─┘
```

路由器维护一张表："记住这个外网端口 6xxxx 对应的是笔记本的 51234"，回包时再翻译回去。所以从外面看，你全家只有一个 IP。

**NAT 的副作用**：
- 从外网无法主动连接到你家内网的设备（"内网穿透"就是为了解决这个）
- P2P 连接变困难（所以有 STUN、TURN 这些技术）
- 让 IPv4 的 43 亿地址勉强够用了 20 多年

### 2.4 端口：同一台机器上的多个程序

一台服务器可能同时跑着 Web 服务、数据库、SSH、Redis……它们都在 IP = `192.168.1.100` 这一个地址上。外面的请求到了，操作系统怎么知道该把包交给哪个程序？

靠**端口号**。端口是一个 16 位整数（0-65535），每个程序监听一个或几个端口：

```
192.168.1.100
     │
     ├─ :22     ← SSH
     ├─ :80     ← HTTP
     ├─ :443    ← HTTPS
     ├─ :3306   ← MySQL
     ├─ :5432   ← PostgreSQL
     ├─ :6379   ← Redis
     └─ :8000   ← 你的 FastAPI
```

**端口分类：**
- `0-1023`：**知名端口**（well-known），需要管理员权限才能绑定。HTTP 80、HTTPS 443、SSH 22 都在这里
- `1024-49151`：**注册端口**，常用软件会约定俗成占用，比如 MySQL 3306、Redis 6379
- `49152-65535`：**动态端口**，操作系统给客户端临时分配的

**一个 TCP 连接是"四元组"**：源 IP、源端口、目的 IP、目的端口。哪怕同一个客户端同时开两个浏览器标签访问同一个服务器，也会用两个不同的源端口，所以是两个独立的连接。

```python
# 查看本机 TCP 连接的四元组（Python)
import psutil  # pip install psutil

for conn in psutil.net_connections(kind="tcp"):
    if conn.status == "ESTABLISHED":
        laddr = f"{conn.laddr.ip}:{conn.laddr.port}"
        raddr = f"{conn.raddr.ip}:{conn.raddr.port}"
        print(f"{laddr} <-> {raddr}  pid={conn.pid}")
```

上面的脚本会打印出本机所有处于 "已建立" 状态的 TCP 连接——你能看到浏览器连着各种服务器的情况。`laddr` 是本地地址（源 IP+端口），`raddr` 是远端地址（目的 IP+端口）。

### 2.5 MAC 地址：网卡的身份证

MAC（Media Access Control）地址是硬件层面的地址，每张网卡出厂时都被烧入一个全球唯一的 48 位地址，写成 `AA:BB:CC:DD:EE:FF`。

IP 地址可以变（你换个 Wi-Fi 就换了），MAC 不会变（除非你刷机或换网卡）。

**它存在的意义**：在同一个局域网（比如同一个 Wi-Fi）里，数据包最终是通过 MAC 地址送到具体设备的。路由器要问"谁的 IP 是 192.168.1.100？"，这个过程叫 **ARP（Address Resolution Protocol）**，答案是"我的 MAC 是 AA:BB:CC..."。

作为后端开发，你几乎永远不用直接处理 MAC 地址。知道有这么个东西、能分清它和 IP 的区别就够了。

### 2.6 一图串起来：数据包如何到达

```
你的浏览器                                     Google 服务器
[应用]  HTTP 请求
[传输]  源端口 51234, 目的端口 443 (HTTPS)
[网络]  源 IP 192.168.1.5, 目的 IP 142.250.xxx.xxx
[链路]  源 MAC 你的网卡, 目的 MAC 家里路由器的网卡
          │
          ▼
      家里路由器（NAT 翻译）
          │ 源 IP 改成公网 IP、源端口改成 6xxxx，记一笔
          ▼
      ISP 骨干网(一路 IP 路由)
          │
          ▼
      Google 数据中心入口
          │
          ▼
      Google 服务器（目的端口 443 → 交给 nginx）
```

### 2.7 常见误区

**误区一：IP 地址是固定给每台机器的**
错。IP 是网络层的逻辑地址，你换个 Wi-Fi、重启路由器都可能变。真正"固定的"是 MAC。

**误区二：端口和 IP 是分开的东西**
TCP 的连接是四元组 `(srcIP, srcPort, dstIP, dstPort)`。单独的端口号没有意义——必须和 IP 配对。

**误区三：127.0.0.1 和 0.0.0.0 一样**
差别很大。`127.0.0.1` 只能本机访问（loopback），`0.0.0.0` 表示"监听所有网卡"（包括内网、公网）。把 FastAPI 用 `uvicorn --host 127.0.0.1` 启动，同事的机器用你的 IP 访问就会连不上——他需要你监听 `0.0.0.0`。

**误区四：内网 IP 是"不安全"的**
内网 IP 本身没有安全/不安全的属性。它的特点是从公网无法直接访问，但不意味着自动安全——同一个 Wi-Fi 下的其他设备一样能访问你的内网服务。

## 3. TCP：可靠传输的秘密

### 3.1 Why：业务要的是"按顺序、不丢、不重复"

UDP 像寄明信片：便宜、快，但可能丢、可能乱序。网页、API、数据库连接要的是**字节流**：先发的一定先到达（在同一条连接内）、丢了会重传、重复了会识别。TCP（Transmission Control Protocol）就是干这个的。

### 3.2 What：连接、序号、确认、窗口

- **面向连接**：通信前先"握手"，双方确认对方在线、初始序号协商好。
- **可靠传输**：每个字节有序号；收方用 **ACK** 告诉发方"我收到到哪了"；超时未 ACK 就重传。
- **流量控制**：接收方告诉对方"我还能收多少"（接收窗口），防止把慢机器冲垮。
- **拥塞控制**：网络堵时主动减速（经典算法如 Reno、CUBIC），这是 TCP "公平"上网的原因。
- **字节流**：TCP **不保留消息边界**。你 `send` 两次 100 字节，对端可能一次 `recv` 到 200 字节。应用层要自己定"分包协议"（例如 HTTP 用 `Content-Length` 或 chunked）。

### 3.3 How：三次握手与四次挥手（理解即可）

**三次握手（建立连接）**：

1. 客户端 → 服务端：`SYN`（我带初始序号 J，想连你）
2. 服务端 → 客户端：`SYN-ACK`（我收到了，我的初始序号是 K）
3. 客户端 → 服务端：`ACK`（我收到了你的 K）

为什么三次而不是两次？防止**历史重复连接**误伤新连接，并确保双方收发能力都就绪。面试背不住细节没关系，能说清"同步双方初始序号 + 防止旧包"即可。

**四次挥手（关闭连接）**：

一方说"我发完了"（`FIN`），对方 `ACK`；对方也可能还有数据要发；最后双方都 `FIN`/`ACK` 完毕。你会在抓包里看到 `TIME_WAIT` 状态——本端要等足够久，确保最后的 ACK 对方收到，避免端口立刻复用时混包。`SO_REUSEADDR` 在服务端快速重启时常用（见 demo）。

### 3.4 How：Python 里 TCP 长什么样

标准库 `socket`：`AF_INET` + `SOCK_STREAM` 就是 IPv4 TCP。完整收发流程见：

- `demos/01_tcp_server_client.py`（`bind` / `listen` / `accept` / `recv` / `send` / `close`）

要点对照：

| 调用 | 含义 |
|------|------|
| `bind((host, port))` | 服务端占用端口；`127.0.0.1` 只本机，`0.0.0.0` 所有网卡 |
| `listen(backlog)` | 排队等待 accept 的连接数上限 |
| `accept()` | 阻塞直到有客户端连上，返回**新** socket 专门跟这个客户端聊 |
| `recv(n)` | 最多读 n 字节，可能少于 n；返回 `b""` 表示对端关了写方向 |
| `send(data)` | 可能只发送一部分，生产代码常要循环 send |

### 3.5 常见误区

- **误区**：TCP "保证消息一条是一条"。错，保证的是**字节顺序与可靠**，不保证与应用层"一条消息"对齐。
- **误区**：`accept` 返回的 socket 和监听 socket 是同一个。错，监听 socket 只负责接客，对话用新 socket。
- **误区**：客户端 `connect` 失败一定是对方没开服务。还可能是防火墙、安全组、Docker 端口没映射、监听在 127.0.0.1。

## 4. UDP：快但不可靠

### 4.1 Why：实时、广播、自己实现可靠性

视频会议、游戏同步、DNS 查询、QUIC（HTTP/3 底层）等场景：**延迟敏感**或**一次一问一答**，宁愿丢包重传由应用控制，也不要 TCP 的队头阻塞。UDP（User Datagram Protocol）无连接、无重传、无顺序保证，头部极小，发送路径简单。

### 4.2 What：数据报模型

每个 UDP 包是独立的 **datagram**（尽力投递，不保证到）。可能丢、可能重复、可能乱序。需要可靠时，自己在应用层做序号和 ACK（例如 QUIC 就是在 UDP 之上做的）。

### 4.3 How：Python UDP

`socket.SOCK_DGRAM`，用 `sendto` / `recvfrom`，无需 `listen`/`accept`。见 `demos/02_udp_server_client.py`。

### 4.4 常见误区

- **误区**：UDP 一定比 TCP 快。不一定：丢包严重时应用层重传可能更糟；要实测。
- **误区**：DNS 只有 UDP。大响应可能走 TCP（例如区域传输、大 TXT）。

## 5. DNS：把"人话"翻译成 IP

### 5.1 Why：人类记域名，路由认 IP

浏览器访问 `api.openai.com`，第一步是问 DNS：**这个名字对应的 IP（和可选的其它记录）是什么**。没有 DNS，你就得背 IP。

### 5.2 What：解析链路与记录类型

- **递归解析器**：通常是你路由器或运营商或 `8.8.8.8` 替你层层问，直到拿到结果。
- **权威 DNS**：域名注册商或 CDN 配置的最终答案来源。
- 常见记录：**A**（IPv4）、**AAAA**（IPv6）、**CNAME**（别名）、**TXT**（验证域名所有权等）。

**TTL**：缓存多久。TTL 短，切流量快；TTL 长，省查询、但故障切换慢。

### 5.3 How：本机怎么查

```bash
# 看解析结果与 TTL
dig +noall +answer api.openai.com
```

Python 标准库：

```python
import socket

# 同时支持 IPv4/IPv6 的解析（底层 getaddrinfo）
infos = socket.getaddrinfo("www.example.com", "https", type=socket.SOCK_STREAM)
for fam, typ, proto, canon, sockaddr in infos:
    print(fam, sockaddr)
```

`requests`/`httpx` 发请求前内部会做类似解析；**DNS 失败**时你会看到 `NameResolutionError` 一类错误，与 HTTP 状态码无关。

### 5.4 常见误区

- **误区**：改完 DNS 立刻全球生效。错，受 TTL 与各级缓存影响，可能要等很久。
- **误区**：`/etc/hosts` 优先级高于 DNS（通常），本地劫持调试时很有用。

## 6. HTTP 协议详解：请求与响应的一生

### 6.1 Why：应用层事实标准

HTTP（HyperText Transfer Protocol）是 Web 与绝大多数 API 的**应用层协议**。它约定：客户端发**请求报文**，服务端回**响应报文**；通常底下承载在 **TCP** 上（HTTPS 则是 TLS 再承载 HTTP）。

### 6.2 What：报文结构

**请求**：

```
方法 路径 HTTP版本\r\n
Header1: 值\r\n
Header2: 值\r\n
\r\n
[可选 body]
```

**响应**：

```
HTTP版本 状态码 原因短语\r\n
Header1: 值\r\n
\r\n
[可选 body]
```

行结束符在协议里是 **CRLF**（`\r\n`），不是单独 `\n`。`Host` 在 HTTP/1.1 里**必须有**，否则很多服务器直接 400。

### 6.3 How：用最小 TCP 服务器看裸 HTTP

`demos/04_http_server.py` 用标准库搭了一个极简 HTTP 服务：读一行请求行、可选读头、回固定响应。适合理解"HTTP 就是一段按格式的文本 + 可选二进制 body"。

用 curl 对照：

```bash
curl -v http://127.0.0.1:8000/
```

`-v` 会打印请求行、响应行和头，是后端必备技能。

### 6.4 常见误区

- **误区**：HTTP 报文全是 UTF-8 文本。头基本是 ASCII；body 编码由 `Content-Type` 决定，可能是二进制。
- **误区**：URL 里看见的中文就是服务器收到的样子。实际上会 **percent-encode**（UTF-8 字节转 `%XX`）。

## 7. HTTP 方法：GET、POST 和它的朋友们

### 7.1 Why：用动词表达意图（在 REST 风格下）

方法（Method）告诉服务器**想对资源做什么**。REST 里常把方法语义和资源 URL 组合使用。

### 7.2 What：常见方法与语义

| 方法 | 典型语义 | 是否幂等* | 是否有 body |
|------|----------|-----------|-------------|
| GET | 获取 | 是 | 通常无（不规范） |
| POST | 创建 / 触发动作 | 否 | 常有 |
| PUT | 全量替换 | 是 | 常有 |
| PATCH | 部分更新 | 视实现 | 常有 |
| DELETE | 删除 | 是 | 可有可无 |

\***幂等**：同样请求执行多次，**资源状态**与执行一次一致（数学意义在 HTTP 里是约定，不是编译器保证）。

**安全**：GET/HEAD/OPTIONS 不应产生副作用（仍可能被日志、统计记录，但不应改业务数据）。

### 7.3 How：`requests` 演示

`demos/03_http_requests.py` 对 httpbin 演示了 GET/POST/PUT/DELETE 与 `params`、`json=`、`headers`。注意：`data=` 与 `json=` 触发的 `Content-Type` 不同。

### 7.4 常见误区

- **误区**：GET 绝对不能带 body。RFC 不禁止，但中间代理、缓存、客户端实现可能不靠谱，**工程上别用 GET 传 body**。
- **误区**：POST 一定比 GET 安全。错，安全靠 **HTTPS**，不靠方法；GET 只是更容易被缓存、进浏览器历史。

## 8. HTTP 状态码：1xx–5xx 全家桶

### 8.1 Why：机器可读的第一句结论

状态码让客户端**在不解析 body** 的情况下决定重试、跳转、换 Token、还是直接报错。排障时先看状态码大类，再看具体码。

### 8.2 What：分类

- **1xx**：信息性（如 101 Switching Protocols，WebSocket 升级）
- **2xx**：成功（200 OK，201 Created，204 No Content）
- **3xx**：重定向（301 永久，302/307/308 临时，304 Not Modified）
- **4xx**：客户端问题（400 坏请求，401 未认证，403 无权限，404 未找到，429 限流）
- **5xx**：服务端问题（500 内部错误，502 坏网关，503 不可用，504 网关超时）

### 8.3 How：几个高频辨析

- **401 vs 403**：401 是"你是谁？"（未登录/Token 无效）；403 是"我知道你是谁，但不让你进"。
- **502 vs 503 vs 504**：502 网关从上游拿到**无效响应**；503 服务**主动**说现在不可用（可配合 `Retry-After`）；504 网关**等上游超时**。
- **301 vs 302**：301 搜索引擎与客户端会更多**固化**到新 URL；302 历史上实现混乱，API 里要小心缓存。

### 8.4 常见误区

- **误区**：返回 200 但 body 里 `{"error": "..."}` 算成功。对 HTTP 层算成功，对业务层是**反模式**（排查、监控、缓存全乱），除非团队明确约定且工具链支持。

### 8.5 Checkpoint

## 9. HTTP 头部：信封上的备注

### 9.1 Why：元数据与协商

头部携带**与表示无关的控制信息**：缓存、认证、内容类型、编码、CORS、Cookie……body 是货物，头部是快递单上的备注。

### 9.2 What：开发常见头

- **`Content-Type`**：body 类型，如 `application/json; charset=utf-8`。
- **`Content-Length`**：body 字节数（与 chunked 二选一逻辑）。
- **`Authorization`**：认证信息，常见 `Bearer <token>`、`Basic ...`。
- **`User-Agent`**：客户端身份；有些 API 会限 UA。
- **`Accept`**：客户端能接受的表示类型（内容协商）。
- **`Cache-Control` / `ETag` / `Last-Modified`**：缓存与条件请求。
- **`Cookie` / `Set-Cookie`**：状态（见第 12 节）。

### 9.3 How：在 Python 里看头

```python
import requests

r = requests.get("https://httpbin.org/get", timeout=10)
print(r.headers.get("Content-Type"))
print(r.request.headers)  # 实际发出的请求头
```

### 9.4 常见误区

- **误区**：头部大小写敏感。HTTP/1.x 头**名**不区分大小写；HTTP/2 传输时会进一步规范化，但语义相同。
- **误区**：自定义头可以随便起。建议 `X-` 前缀或反向域名风格，避免与未来标准冲突。

## 10. HTTP 版本演进：1.0 → 1.1 → 2 → 3

### 10.1 Why：性能与安全推动换代

核心矛盾：**减少延迟**（连接数、RTT）、**提高带宽利用率**（多路复用）、**加密成为默认**。

### 10.2 What：各版要点

- **HTTP/1.0**：一请求一连接，用完就关，慢。
- **HTTP/1.1**：**持久连接**（默认 keep-alive）、**Host**、**chunked** 传输、管线化（实际少用）。
- **HTTP/2**：基于 TLS 的实践中主流；**二进制分帧**、**多路复用**（一条连接并行多个请求）、**HPACK** 头压缩、服务器推送（用得少）。
- **HTTP/3**：基于 **QUIC（UDP）**，减少握手 RTT，改善丢包时的队头阻塞；连接迁移对移动网络更友好。

### 10.3 How：你怎么感知版本

浏览器 DevTools → Network → Protocol 列；curl：

```bash
curl -I --http2 https://www.cloudflare.com
```

### 10.4 常见误区

- **误区**：HTTP/2 一定比 1.1 快。小文件、低延迟场景收益有限；还要 CDN/服务端支持。

## 11. HTTPS 与 TLS：加密信件与公章

### 11.1 Why：防窃听、防篡改、验身份

裸 HTTP 经过的每一个路由器都能看到内容。HTTPS = HTTP over **TLS**：**加密**（保密性）、**MAC/AEAD**（完整性）、**证书链**（服务端身份认证）。可选 **mTLS**（双向证书）做高安全内网。

### 11.2 What：证书与握手直觉

- **服务器证书**含公钥与域名（SAN/CN）等信息，由 **CA** 签名；客户端用内置信任根验证链是否可信。
- **TLS 握手**（1.2/1.3 细节不同）直觉版：协商算法、验证证书、用**密钥交换**得到会话密钥，后续对称加密传 HTTP。

自签名证书：没有公共 CA 背书，浏览器会警告——适合内网调试，不适合对公网用户。

### 11.3 How：命令行看一眼

```bash
openssl s_client -connect www.google.com:443 -servername www.google.com </dev/null 2>/dev/null | openssl x509 -noout -dates -subject
```

Python `requests` 默认校验证书；测试环境可关校验（**极度危险，仅调试**）：

```python
requests.get("https://...", verify=False)  # 不要提交到生产代码
```

### 11.4 常见误区

- **误区**：HTTPS 只加密 body。错，**路径与查询串在 TLS 里也是加密的**（但 SNI 在部分场景可能泄露域名给中间设备，另有 ESNI/ECH 等演进）。
- **误区**：用了 HTTPS 就防 XSS。错，XSS 是浏览器执行恶意脚本，和传输加密不同层面。

## 12. Cookie / Session / Token / JWT：状态保持方案的四次进化

### 12.1 Why：HTTP 默认"无状态"

HTTP 本身不记住"你是谁"。登录后要识别同一用户，需要**状态机制**：从 Cookie、Session 到 Token、JWT，本质是**用凭证换识别**。

### 12.2 What：四种东西各是什么

- **Cookie**：浏览器存的一小段字符串，每次同站请求可自动带上。服务端用 `Set-Cookie` 下发。
- **Session**：服务端内存或 Redis 里存的会话数据，通常用 **session id** 当钥匙；session id 常放在 Cookie 里。
- **Token**：广义上任何"拿着就能证明身份"的字符串；常指随机不透明 token，服务端查库或缓存校验。
- **JWT**（JSON Web Token）：**自包含**的三段 Base64（header.payload.signature），payload 里放声明（如 `sub`、`exp`），用密钥签名；服务端用签名验证真伪，**不必每次查 session 存储**（但要注意撤销与刷新）。

### 12.3 How：安全属性（Cookie 必会）

- **`HttpOnly`**：JS 读不到，降 XSS 偷 Cookie 风险。
- **`Secure`**：只走 HTTPS。
- **`SameSite`**：`Strict` / `Lax` / `None`（`None` 须配合 `Secure`），缓解 CSRF。

动手：`demos/06_session_cookie.py`（Cookie/Session 演示）、`demos/07_jwt_demo.py`（签发与校验思路）。

### 12.4 常见误区

- **误区**：JWT 不能伪造。错，没有密钥任何人都能造 payload，但**签不过**；密钥泄露则全完。
- **误区**：JWT 里可以放密码。绝对不要。payload 只是 Base64，不是加密。
- **误区**：有了 JWT 就不要 HTTPS。错，Bearer Token 被偷一样完蛋。

- [ ] `SameSite=Lax` 大致防什么攻击？

## 13. CORS：浏览器画的那条红线

### 13.1 Why：同源策略

浏览器允许网页发请求，但**默认禁止 JS 读取跨域响应**（同源：协议+主机+端口一致）。这是安全模型：**恶意网站不能用你的登录态去读银行站点的响应**。

### 13.2 What：简单请求与预检

- **简单请求**（GET/HEAD/POST 且头与 `Content-Type` 在允许集合内等）：直接发，响应里带 `Access-Control-Allow-Origin` 等则 JS 可读。
- **预检（preflight）**：对"非简单"请求，浏览器先发 `OPTIONS`，问服务器是否允许；通过后再发真实请求。

服务端典型头：`Access-Control-Allow-Origin`、`Access-Control-Allow-Methods`、`Access-Control-Allow-Headers`、`Access-Control-Allow-Credentials`（与 `*`、Cookie 组合有细则）。

### 13.3 How：为什么 Postman 通、浏览器挂

Postman/curl **不执行**同源策略；只有浏览器里的前端 JS 才受 CORS 限制。所以"接口 200 但前端报错"多半是 CORS。

动手：`demos/09_cors_demo.py`。

### 13.4 常见误区

- **误区**：CORS 是防黑客。错，主要防**用户浏览器里的恶意页面**滥用用户身份；不替代鉴权。
- **误区**：`Allow-Origin: *` 配 `Allow-Credentials: true`。浏览器会拒绝这种组合。

## 14. RESTful API 设计

### 14.1 Why：可预测、可缓存、可协作

REST 不是框架，是一种**风格**：用 **URL 表示资源**，用 **HTTP 方法表示动作**，用 **状态码表达结果**，尽量**无状态**（鉴权信息放在头里每请求带上）。

### 14.2 What：常见约定

- 资源用名词复数：`/users`、`/users/{id}/orders`。
- 用方法区分操作：GET 查、POST 建、PUT/PATCH 改、DELETE 删。
- **分页**：`?page=&limit=` 或 `cursor`；**过滤**：查询参数。
- **版本**：路径 `/v1/...` 或头 `Accept: application/vnd.xxx+json`（团队选一种坚持）。

### 14.3 How：对照 demo

`demos/08_restful_server.py`：`demos/05_json_api.py` 常与 JSON 组合练习。

### 14.4 常见误区

- **误区**：REST 等于 CRUD。REST 还可表达动作型资源（如 `POST /jobs/{id}/cancel`），关键是**一致性与超媒体/HATEOAS**（实际项目很少做到纯 HATEOAS）。
- **误区**：所有东西塞进 GET 查询参数方便调试。一旦涉及敏感或副作用，应用 POST 并正确设计。

## 15. GraphQL 与 gRPC：HTTP 之外的选择

### 15.1 Why：REST 不是银弹

- **GraphQL**：前端**一次查询拿多资源**，减少往返；但缓存、限流、复杂度控制要在网关/服务端额外做。
- **gRPC**：基于 **HTTP/2** 的二进制 RPC，**强 schema（protobuf）**、对流式调用支持好；跨语言微服务常用，浏览器侧要 **gRPC-Web** 或网关转换。

### 15.2 What：怎么选型（直觉）

| 场景 | 更常选 |
|------|--------|
| 公网开放 API、第三方集成 | REST + OpenAPI |
| 移动端/复杂页面数据聚合 | GraphQL（有治理） |
| 内网微服务东西向流量 | gRPC |
| 实时推送 | WebSocket / SSE（见下节） |

## 16. Python requests：最常用的 HTTP 客户端

### 16.1 Why：人机工程好

`requests` 把 `urllib3` 包了一层，API 直白，是脚本与中小服务的默认选择。

### 16.2 What：必会 API

- `requests.get/post/...`：`params`、`data`、`json`、`headers`、`timeout`。
- **`timeout`**：务必设置（连接超时、读超时可写元组），否则挂死。
- **`Session()`**：连接池、默认头、Cookie 罐，**爬虫与多请求 API 客户端首选**。
- **流式**：`stream=True` + 迭代 `resp.iter_lines()`，适合大文件与 **SSE**。

### 16.3 How：重试与 LLM 调用注意

`requests` 本身不内置重试；用 `urllib3.util.Retry` 配 `HTTPAdapter` 挂到 `Session` 上，或对 429/5xx 做指数退避。**读流**时重试要更谨慎（可能重复消费）。

动手：`demos/03_http_requests.py`、`demos/05_json_api.py`。

### 16.4 常见误区

- **误区**：`resp.json()` 一定成功。只有 `Content-Type` 与 body 真是 JSON 才行；否则抛异常，要先看 `status_code` 与 `text`。
- **误区**：不信任响应就 `raise_for_status()`。对 4xx/5xx 直接抛，便于统一错误处理。

### 16.5 Checkpoint

- [ ] 为什么生产代码必须设 `timeout`？
- [ ] 何时用 `Session` 而不是次次 `requests.get`？

## 17. httpx：同步 + 异步 + HTTP/2

### 17.1 Why：现代替代

`httpx` API 接近 `requests`，同时支持 **`async/await`**，可选 **HTTP/2**，适合 **FastAPI/Starlette** 生态里异步下游调用。

### 17.2 What：用法对照

```python
import httpx

# 同步
with httpx.Client(http2=True) as client:
    r = client.get("https://example.com", timeout=10.0)

# 异步
async with httpx.AsyncClient() as client:
    r = await client.get("https://example.com", timeout=10.0)
```

### 17.3 常见误区

- **误区**：异步一定更快。错，CPU 密集或 DNS/SSL 仍可能阻塞事件循环；要配合连接池、合理 `gather`。

## 18. WebSocket 与 SSE 速览

### 18.1 Why：HTTP 请求-响应不够用

- **WebSocket**：握手仍用 HTTP，之后**全双工**帧传输，适合聊天、协同、游戏。
- **SSE**（Server-Sent Events）：基于 HTTP，**服务端单向**推事件流，`text/event-stream`，浏览器 `EventSource`；适合通知、**部分 LLM 流式输出**。

### 18.2 What：与轮询对比

轮询浪费；长轮询凑合；WebSocket/SSE 按场景选。**代理与超时**：中间层可能断长连接，要调 nginx `proxy_read_timeout` 等。

## 20. 和 AI Agent 后端的关联

### 20.1 Why：Agent = 编排 + 大量 HTTP

调用嵌入、向量库、工具 HTTP、LLM API，全是**超时、重试、流式、计费与限流**问题。

### 20.2 What：实战清单

- **鉴权**：`Authorization: Bearer ...`、部分云厂商签名头；密钥只放服务端与环境变量。
- **流式**：OpenAI 兼容接口常见 **SSE**；客户端用 `stream=True` 或 httpx 流式读，**注意异常时半包**。
- **超时**：连接超时、首字节超时、**读空闲超时**（流式中间停顿）。
- **重试**：仅对**幂等**或带 **idempotency key** 的请求；429 读 `Retry-After`。
- **观测**：记录 `request_id`、状态码、延迟、token 用量；便于对账与排障。
