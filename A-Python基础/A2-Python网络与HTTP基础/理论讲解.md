## 0. 写在前面：为什么要花时间学网络

### 0.1 一个你每天都在做但从没想过的动作

你打开浏览器，输入 `https://www.google.com`，按下回车。0.5 秒之后 Google 的首页出现了。

这 0.5 秒内发生了什么？粗略数一下：

1. 你的电脑问"Google 这个名字对应哪个 IP 地址？" —— **DNS 解析**
2. 电脑和 Google 的服务器建立了一条"线路" —— **TCP 三次握手**
3. 双方协商用什么密码说话，交换了密钥 —— **TLS 握手**
4. 你的电脑说了一句"请把首页给我" —— **HTTP 请求**
5. Google 的服务器说了一句"给你，这是 HTML" —— **HTTP 响应**
6. 你的浏览器把 HTML 画出来

这六步里，任何一步卡住、错了、被拦截，你都会看到熟悉的"网页打不开"。而作为后端工程师，你的工作就是**写那个回答请求的服务器**——它必须在 0.5 秒内完成所有这些动作。

### 0.2 你可能已经踩过的坑

下面这些问题，如果有任何一个你回答不上来，都说明网络基础需要补：

- 为什么我本地跑 `localhost:8000` 可以，别人用我的 IP 就打不开？
- 前端说"CORS 错误"，为什么后端明明返回了 200，浏览器还是报错？
- 为什么 POST 请求偶尔会出现两次，GET 却不会？
- 为什么 `https` 比 `http` 安全？自签名证书为什么浏览器会警告？
- 接口 504 Gateway Timeout，到底是谁超时了？我的代码还是 nginx？
- 调 OpenAI API 偶尔会断流，为什么需要配 `timeout` 和 `retry`？

这些问题的答案都在本章。

## 1. 网络分层模型：快递系统的 OSI 与 TCP/IP

### 1.1 Why：为什么要分层

想象你要寄一个包裹给南京的朋友。整个流程大致是：

1. 你写一封信，装进信封，写上"南京市鼓楼区 XX 路 123 号"
2. 你把信封拿到顺丰网点
3. 网点把你的信和其他很多信一起装进一个大包裹，贴上"南京分拨中心"的标签
4. 这个大包裹被装上卡车，卡车走高速路到南京
5. 南京分拨中心拆开大包裹，把你的信单独拿出来
6. 快递员骑电动车送到朋友家

这里有三个很关键的观察：

**观察一：每一层只关心自己的任务**
- 你只关心"地址写对没有"，不关心卡车走哪条高速
- 卡车司机只关心"到南京分拨中心"，不关心包裹里是信还是文件
- 快递员只关心"这封信该送到哪一户人家"，不关心信的内容

**观察二：每一层只和相邻层打交道**
- 你只和网点打交道，不会直接跟卡车司机说话
- 卡车司机只对接分拨中心，不直接对接收件人

**观察三:换掉其中一层不影响别的层**
- 卡车换成飞机、高铁，你写信的方式不变
- 你把信封换成快递盒，卡车照常跑

网络也是一样的。数据从你的浏览器到 Google 的服务器，中间要经过网卡、路由器、光纤、海底电缆……如果不分层，每换一个网络设备都要改所有代码，根本没法维护。所以**前人把网络通信切成若干层，每层只做一件事，每层之间接口固定**。这就是"网络分层模型"。

### 1.2 What：OSI 七层模型

OSI（Open Systems Interconnection）是国际标准化组织在 1984 年提出的参考模型，它把网络通信分成 7 层。从上到下是：

```
┌───────────────────────────────────┐
│ 7. 应用层 Application             │  你看得到的"业务": HTTP, FTP, SMTP, DNS
├───────────────────────────────────┤
│ 6. 表示层 Presentation            │  数据格式转换、加密压缩 (TLS 逻辑上在这)
├───────────────────────────────────┤
│ 5. 会话层 Session                 │  会话管理、断点续传
├───────────────────────────────────┤
│ 4. 传输层 Transport               │  端到端可靠传输: TCP, UDP
├───────────────────────────────────┤
│ 3. 网络层 Network                 │  路由和寻址: IP, ICMP
├───────────────────────────────────┤
│ 2. 数据链路层 Data Link           │  相邻节点间传输: Ethernet, Wi-Fi, ARP
├───────────────────────────────────┤
│ 1. 物理层 Physical                │  电信号、光信号、电磁波
└───────────────────────────────────┘
```

**类比映射回快递系统：**

| OSI 层 | 快递类比 | 做什么 |
|--------|----------|--------|
| 应用层 | 你写信的内容 | 业务数据本身 |
| 表示层 | 信封的格式、把中文翻译成英文 | 数据编码/加密 |
| 会话层 | "这是给同一个收件人的系列邮件" | 会话管理 |
| 传输层 | 顺丰 vs 平邮，要不要签收 | 可靠性保证 |
| 网络层 | 省内 / 跨省的路由规划 | 选择路径 |
| 数据链路层 | 同一个分拨中心内部的传送带 | 点对点传输 |
| 物理层 | 电动车、飞机、卡车 | 物理媒介 |

### 1.3 What：TCP/IP 四层模型（工程中真正使用的）

OSI 是教科书上的模型，好看但过于学术。工程上实际用的是 **TCP/IP 四层模型**（也叫五层混合模型）：

```
┌─────────────────────────────────┐
│ 应用层 Application              │  HTTP, WebSocket, gRPC, DNS, SSH
│ (OSI 5+6+7)                     │
├─────────────────────────────────┤
│ 传输层 Transport                │  TCP, UDP, QUIC
├─────────────────────────────────┤
│ 网络层 Network / Internet       │  IPv4, IPv6, ICMP
├─────────────────────────────────┤
│ 链路层 Link                     │  Ethernet, Wi-Fi, PPP
└─────────────────────────────────┘
```

TCP/IP 把 OSI 的 5、6、7 层合并成了"应用层"，因为实践中根本分不清。以 HTTPS 为例：TLS 算哪一层？从功能看像"表示层"，但它其实是运行在应用和传输之间的一个库，不纯粹属于任何一层。硬分没意义。

**本文后续讨论都用 TCP/IP 四层**，但你面试时要知道 OSI 是 7 层。

### 1.4 How：数据封装与解封装

当你发送一个 HTTP 请求时，数据从上往下一路"加头"，到网卡上变成一堆二进制：

```
你的 HTTP 请求报文
   "GET / HTTP/1.1\r\nHost: google.com\r\n\r\n"
           │
           ▼
   [应用层: HTTP]  这就是原始数据 (payload)
           │
           ▼
   [传输层: TCP]   [TCP 头 | HTTP 数据]   加 20 字节 TCP 头(源端口/目的端口/序列号...)
           │
           ▼
   [网络层: IP]    [IP 头 | TCP 头 | HTTP 数据]   加 20 字节 IP 头(源IP/目的IP)
           │
           ▼
   [链路层: 以太网] [以太网头 | IP 头 | TCP 头 | HTTP 数据 | 以太网尾]
           │
           ▼
   物理层 ~~~~~~~ 比特流 0101010101010... ~~~~~~~>
```

对方收到后，从下往上"剥头"：以太网层剥掉以太网头 → 交给 IP 层剥 IP 头 → 交给 TCP 层剥 TCP 头 → 最后交给应用层处理原始 HTTP 数据。

**每一层只看自己的头**。TCP 不关心 HTTP 内容是什么，IP 不关心 TCP 的内容是什么。这就是"每层只关心本层协议"的工程含义。

### 1.5 常见误区

**误区一：以为 HTTP 和 TCP 是并列的两个东西**
错。HTTP 运行在 TCP 之上，HTTP 的每个字节最终都会被 TCP 封装。没有 TCP（或 QUIC）就没有 HTTP。

**误区二：以为每层都有"协议头"加起来开销很大**
确实有，但开销只在发送开始和接收最后。中间的路由器只看 IP 头就够了，不会拆 TCP 头。

**误区三：以为"应用层 = 写代码的那一层，我不用懂下面"**
错。你写的 HTTP 客户端偶尔出现的"Connection reset"、"Timeout"、"SSL handshake failed"都是下面几层的错误冒上来。不懂下面，排错时就只能瞎猜。

### 1.7 深度补充：封装顺序——写一段「伪代码」就永远不会忘

把「发一条 HTTP」想象成调用四个函数（真实内核更复杂，但**心智模型**够用）：

```text
send_http(payload):
    frame = eth_encap( ip_encap( tcp_encap( payload ) ) )
    nic_transmit(frame)
```

- **最内层** `payload`：你的 HTTP 字节（应用层）。  
- **`tcp_encap`**：加上源/目的端口、序列号等，交给 IP。  
- **`ip_encap`**：加上源/目的 IP，交给以太网驱动。  
- **`eth_encap`**：加上源/目的 MAC，交给网卡。  

接收端是**逆序剥洋葱**：网卡中断 → 以太网驱动看类型字段 `0x0800` → 交 IP → 看协议号 `6`（TCP）→ 交 TCP 按端口排队 → 交监听 443 的进程 → 若是 TLS 则再解密后才是 HTTP。

### 1.9 本节小结（三张卡片）

- **一句话**：分层 = **职责分离 + 替换独立**；数据下行**加头**、上行**剥头**。

## 2. IP、端口、MAC — 互联网的三层地址

### 2.1 Why：为什么需要三种地址

继续用快递类比。要把一封信送到朋友手里，你需要：

1. **小区地址**（"XX 小区 3 栋"）：把信送到对的小区 —— 对应 **IP 地址**（哪台机器）
2. **门牌号**（"301 室"）：小区里有很多住户，要送到对的那户 —— 对应 **端口**（机器上哪个程序）
3. **身份证号**（"这封信是给张三本人的"）：门牌号认门不认人，身份证号确认收件人 —— 对应 **MAC 地址**（网卡的物理标识）

现实中你写快递只写前两个（小区+门牌），MAC 地址是快递员（路由器）内部用的，你感知不到。下面分别讲。

### 2.2 IP 地址：网络层的门牌号

**IPv4** 是 32 位整数，写成 `a.b.c.d` 的四段点分形式，每段 0-255：

```
192.168.1.100
   │   │  │  │
 8位 8位 8位 8位   一共 32 位
```

32 位意味着总共有 `2^32 ≈ 43 亿` 个地址。听起来很多，但全球网民超过 50 亿，设备比人多，早就不够用了。所以有了 **IPv6**（128 位），地址看起来像这样：

```
2001:0db8:85a3:0000:0000:8a2e:0370:7334
```

目前国内大部分家宽默认是 IPv4 + NAT（一会讲）。

**特殊 IP：**

| IP 段 | 含义 |
|-------|------|
| `127.0.0.1` | 本机回环地址（localhost），只能自己访问自己 |
| `10.0.0.0/8` | 私有网络（A 类） |
| `172.16.0.0/12` | 私有网络（B 类） |
| `192.168.0.0/16` | 私有网络（C 类），家用路由器常用 |
| `0.0.0.0` | "所有 IP"，服务端 `bind("0.0.0.0")` 表示监听所有网卡 |
| `255.255.255.255` | 广播地址 |

**子网掩码与 CIDR：**

`192.168.1.0/24` 中的 `/24` 叫 CIDR，表示前 24 位是网络号，后 8 位是主机号。所以这个子网里有 `2^8 = 256` 个地址（去掉网络号和广播号还剩 254 个可用）。

```python
# Python 计算 CIDR
import ipaddress

net = ipaddress.ip_network("192.168.1.0/24")
print(net.num_addresses)   # 256
print(net.network_address) # 192.168.1.0
print(net.broadcast_address) # 192.168.1.255
print(list(net.hosts())[:3])  # 前 3 个可用主机
# [IPv4Address('192.168.1.1'), IPv4Address('192.168.1.2'), IPv4Address('192.168.1.3')]

ip = ipaddress.ip_address("192.168.1.100")
print(ip in net)  # True
```

上面这段代码里 `ipaddress` 是 Python 标准库（不用装），专门处理 IP 的各种运算。`ip_network("192.168.1.0/24")` 会解析出这个子网的所有属性，`ip in net` 判断一个 IP 是否属于这个子网——实际工作中判断"是否是内网 IP"就是这么写。

### 2.3 NAT：为什么你家里所有设备共享一个外网 IP

你手机、笔记本、电视都连家里的 Wi-Fi，它们的 IP 都是 `192.168.x.x` 这种私有地址。但家里只有一个外网 IP（运营商给的），它们怎么同时上网？

答案是 **NAT（Network Address Translation）**。家用路由器会在出站包上做一次翻译：

```
你的笔记本 (192.168.1.5:51234) ─┐
你的手机  (192.168.1.6:50000) ─┼─路由器─> 外网 (公网IP:6xxxx)
你的电视  (192.168.1.7:49999) ─┘
```

路由器维护一张表："记住这个外网端口 6xxxx 对应的是笔记本的 51234"，回包时再翻译回去。所以从外面看，你全家只有一个 IP。

**NAT 的副作用**：
- 从外网无法主动连接到你家内网的设备（"内网穿透"就是为了解决这个）
- P2P 连接变困难（所以有 STUN、TURN 这些技术）
- 让 IPv4 的 43 亿地址勉强够用了 20 多年

### 2.4 端口：同一台机器上的多个程序

一台服务器可能同时跑着 Web 服务、数据库、SSH、Redis……它们都在 IP = `192.168.1.100` 这一个地址上。外面的请求到了，操作系统怎么知道该把包交给哪个程序？

靠**端口号**。端口是一个 16 位整数（0-65535），每个程序监听一个或几个端口：

```
192.168.1.100
     │
     ├─ :22     ← SSH
     ├─ :80     ← HTTP
     ├─ :443    ← HTTPS
     ├─ :3306   ← MySQL
     ├─ :5432   ← PostgreSQL
     ├─ :6379   ← Redis
     └─ :8000   ← 你的 FastAPI
```

**端口分类：**
- `0-1023`：**知名端口**（well-known），需要管理员权限才能绑定。HTTP 80、HTTPS 443、SSH 22 都在这里
- `1024-49151`：**注册端口**，常用软件会约定俗成占用，比如 MySQL 3306、Redis 6379
- `49152-65535`：**动态端口**，操作系统给客户端临时分配的

**一个 TCP 连接是"四元组"**：源 IP、源端口、目的 IP、目的端口。哪怕同一个客户端同时开两个浏览器标签访问同一个服务器，也会用两个不同的源端口，所以是两个独立的连接。

```python
# 查看本机 TCP 连接的四元组（Python)
import psutil  # pip install psutil

for conn in psutil.net_connections(kind="tcp"):
    if conn.status == "ESTABLISHED":
        laddr = f"{conn.laddr.ip}:{conn.laddr.port}"
        raddr = f"{conn.raddr.ip}:{conn.raddr.port}"
        print(f"{laddr} <-> {raddr}  pid={conn.pid}")
```

上面的脚本会打印出本机所有处于 "已建立" 状态的 TCP 连接——你能看到浏览器连着各种服务器的情况。`laddr` 是本地地址（源 IP+端口），`raddr` 是远端地址（目的 IP+端口）。

### 2.5 MAC 地址：网卡的身份证

MAC（Media Access Control）地址是硬件层面的地址，每张网卡出厂时都被烧入一个全球唯一的 48 位地址，写成 `AA:BB:CC:DD:EE:FF`。

IP 地址可以变（你换个 Wi-Fi 就换了），MAC 不会变（除非你刷机或换网卡）。

**它存在的意义**：在同一个局域网（比如同一个 Wi-Fi）里，数据包最终是通过 MAC 地址送到具体设备的。路由器要问"谁的 IP 是 192.168.1.100？"，这个过程叫 **ARP（Address Resolution Protocol）**，答案是"我的 MAC 是 AA:BB:CC..."。

作为后端开发，你几乎永远不用直接处理 MAC 地址。知道有这么个东西、能分清它和 IP 的区别就够了。

### 2.6 一图串起来：数据包如何到达

```
你的浏览器                                     Google 服务器
[应用]  HTTP 请求
[传输]  源端口 51234, 目的端口 443 (HTTPS)
[网络]  源 IP 192.168.1.5, 目的 IP 142.250.xxx.xxx
[链路]  源 MAC 你的网卡, 目的 MAC 家里路由器的网卡
          │
          ▼
      家里路由器（NAT 翻译）
          │ 源 IP 改成公网 IP、源端口改成 6xxxx，记一笔
          ▼
      ISP 骨干网(一路 IP 路由)
          │
          ▼
      Google 数据中心入口
          │
          ▼
      Google 服务器（目的端口 443 → 交给 nginx）
```

### 2.7 常见误区

**误区一：IP 地址是固定给每台机器的**
错。IP 是网络层的逻辑地址，你换个 Wi-Fi、重启路由器都可能变。真正"固定的"是 MAC。

**误区二：端口和 IP 是分开的东西**
TCP 的连接是四元组 `(srcIP, srcPort, dstIP, dstPort)`。单独的端口号没有意义——必须和 IP 配对。

**误区三：127.0.0.1 和 0.0.0.0 一样**
差别很大。`127.0.0.1` 只能本机访问（loopback），`0.0.0.0` 表示"监听所有网卡"（包括内网、公网）。把 FastAPI 用 `uvicorn --host 127.0.0.1` 启动，同事的机器用你的 IP 访问就会连不上——他需要你监听 `0.0.0.0`。

**误区四：内网 IP 是"不安全"的**
内网 IP 本身没有安全/不安全的属性。它的特点是从公网无法直接访问，但不意味着自动安全——同一个 Wi-Fi 下的其他设备一样能访问你的内网服务。

### 2.9 深度补充：IPv6 与「双栈」你在终端里怎么确认

**IPv6** 不是「IPv4 多几位」那么简单：地址空间极大，**端到端**恢复可能（不再处处 NAT），邻居发现用 **NDP** 替代 ARP 的很大一部分角色。你本机可能同时有 IPv4 与 IPv6 地址——叫 **双栈**。

### 2.10 深度补充：`ipaddress` 判断私有 IP（逐行）

```python
import ipaddress

def is_private(ip: str) -> bool:
    a = ipaddress.ip_address(ip)          # 解析为 IPv4Address / IPv6Address
    return a.is_private                     # RFC 1918 / 唯一本地地址等规则封装在标准库

print(is_private("10.8.1.5"))             # True
print(is_private("8.8.8.8"))              # False
```

- **`ip_address`**：把字符串变成「可运算」的地址对象；非法格式会 `ValueError`。  
- **`is_private`**：把 RFC 细节藏在库里，避免自己手写 `10.`、`192.168.` 分支漏判。

## 3. TCP：可靠传输的秘密

### 3.1 Why：业务要的是"按顺序、不丢、不重复"

UDP 像寄明信片：便宜、快，但可能丢、可能乱序。网页、API、数据库连接要的是**字节流**：先发的一定先到达（在同一条连接内）、丢了会重传、重复了会识别。TCP（Transmission Control Protocol）就是干这个的。

### 3.2 What：连接、序号、确认、窗口

- **面向连接**：通信前先"握手"，双方确认对方在线、初始序号协商好。
- **可靠传输**：每个字节有序号；收方用 **ACK** 告诉发方"我收到到哪了"；超时未 ACK 就重传。
- **流量控制**：接收方告诉对方"我还能收多少"（接收窗口），防止把慢机器冲垮。
- **拥塞控制**：网络堵时主动减速（经典算法如 Reno、CUBIC），这是 TCP "公平"上网的原因。
- **字节流**：TCP **不保留消息边界**。你 `send` 两次 100 字节，对端可能一次 `recv` 到 200 字节。应用层要自己定"分包协议"（例如 HTTP 用 `Content-Length` 或 chunked）。

**滑动窗口（直觉）**：发送方不必等每个字节都 ACK 才发下一批，可以在「对端允许的窗口内」**流水线发送**；窗口大小随 RTT、拥塞动态调整。你调大 socket buffer 不等于网速变快，但能减少「窗口满了发不动」的概率。

**半关闭（half-close）**：一端 `shutdown(SHUT_WR)` 后不能再写，但仍可读对端数据；四次挥手里的状态与此相关。Python `socket` 里 `close` 会释放 fd；细粒度控制要用 `shutdown`。

### 3.3 How：三次握手与四次挥手（理解即可）

**三次握手（建立连接）**：

1. 客户端 → 服务端：`SYN`（我带初始序号 J，想连你）
2. 服务端 → 客户端：`SYN-ACK`（我收到了，我的初始序号是 K）
3. 客户端 → 服务端：`ACK`（我收到了你的 K）

为什么三次而不是两次？防止**历史重复连接**误伤新连接，并确保双方收发能力都就绪。面试背不住细节没关系，能说清"同步双方初始序号 + 防止旧包"即可。

**四次挥手（关闭连接）**：

一方说"我发完了"（`FIN`），对方 `ACK`；对方也可能还有数据要发；最后双方都 `FIN`/`ACK` 完毕。你会在抓包里看到 `TIME_WAIT` 状态——本端要等足够久，确保最后的 ACK 对方收到，避免端口立刻复用时混包。`SO_REUSEADDR` 在服务端快速重启时常用（见 demo）。

### 3.4 How：Python 里 TCP 长什么样

标准库 `socket`：`AF_INET` + `SOCK_STREAM` 就是 IPv4 TCP。完整收发流程见：

- `demos/01_tcp_server_client.py`（`bind` / `listen` / `accept` / `recv` / `send` / `close`）

要点对照：

| 调用 | 含义 |
|------|------|
| `bind((host, port))` | 服务端占用端口；`127.0.0.1` 只本机，`0.0.0.0` 所有网卡 |
| `listen(backlog)` | 排队等待 accept 的连接数上限 |
| `accept()` | 阻塞直到有客户端连上，返回**新** socket 专门跟这个客户端聊 |
| `recv(n)` | 最多读 n 字节，可能少于 n；返回 `b""` 表示对端关了写方向 |
| `send(data)` | 可能只发送一部分，生产代码常要循环 send |

### 3.5 常见误区

- **误区**：TCP "保证消息一条是一条"。错，保证的是**字节顺序与可靠**，不保证与应用层"一条消息"对齐。
- **误区**：`accept` 返回的 socket 和监听 socket 是同一个。错，监听 socket 只负责接客，对话用新 socket。
- **误区**：客户端 `connect` 失败一定是对方没开服务。还可能是防火墙、安全组、Docker 端口没映射、监听在 127.0.0.1。

### 3.7 深度补充：Nagle 与延迟 ACK（调优时才会碰到）

**Nagle 算法**：小报文不立刻发，凑一凑或等 ACK，减少 tinygram 对带宽的浪费。交互式游戏或低延迟 RPC 有时会 `TCP_NODELAY` 关掉 Nagle。

**延迟 ACK**：接收方不必对每个包立刻 ACK，可合并。与 Nagle 组合时，可能出现**暂时性延迟**（经典「延迟 40ms」量级，视 OS）。排障时若怀疑：抓包看 ACK 是否「成批」到达。

这两点在写 **自定义 TCP 协议** 或压测数据库时比写普通 HTTP 客户端更常遇到；HTTP 库已帮你挡掉大部分。

### 3.8 深度补充：`send` 与 `recv` 的「短读短写」（与 demo 逐行对齐）

```python
# 发送端常见正确姿势（简化）
data = b"x" * 100000
sent_total = 0
while sent_total < len(data):
    n = sock.send(data[sent_total:])   # n 可能 < 剩余长度
    if n == 0:
        raise RuntimeError("对端关连接")
    sent_total += n
```

- **`send` 返回值 `n`**：本调用**实际写入内核发送缓冲**的字节数，可能小于你想发的那一段。  
- **循环**：直到全部推入内核；是否到达对端由 TCP 保证（除非断连）。  

```python
# 接收端：一次 recv 不保证读完对端一次 send
buf = bytearray()
while True:
    chunk = sock.recv(4096)
    if not chunk:
        break
    buf.extend(chunk)
```

- **`recv` 返回空 `b""`**：对端关闭了写方向（半关闭或全关）。  
- **HTTP 层**用 `Content-Length` / chunked 告诉你还要读多少，本质就是在**字节流上重建消息边界**。

### 3.9 本节小结（三张卡片）

- **一句话**：TCP = **有序、可靠、字节流**；连接用四元组标识；关连与半关闭有状态机。  
- **典型场景**：大 body 上传要循环 `send`；读 HTTP 响应要按头决定读多少。  
- **和 A1 类比**：`recv` 循环像 **手动消费迭代器** —— 一次 `next()` 不一定给你「逻辑上的一条」，要自己组协议。

## 4. UDP：快但不可靠

### 4.1 Why：实时、广播、自己实现可靠性

视频会议、游戏同步、DNS 查询、QUIC（HTTP/3 底层）等场景：**延迟敏感**或**一次一问一答**，宁愿丢包重传由应用控制，也不要 TCP 的队头阻塞。UDP（User Datagram Protocol）无连接、无重传、无顺序保证，头部极小，发送路径简单。

### 4.2 What：数据报模型

每个 UDP 包是独立的 **datagram**（尽力投递，不保证到）。可能丢、可能重复、可能乱序。需要可靠时，自己在应用层做序号和 ACK（例如 QUIC 就是在 UDP 之上做的）。

**和 TCP 对比时脑子里要有的一张表：**

| 维度 | TCP | UDP |
|------|-----|-----|
| 连接 | 要先握手 | 无连接，发就发 |
| 语义 | 有序字节流 | 保留报文边界（一次 `recvfrom` 通常对应一次 `sendto`） |
| 可靠性 | 内核重传、去重、按序递交 | 全靠自己 |
| 开销 | 头至少 20B，状态多 | 头 8B，状态少 |
| 适用 | HTTP/1.x、数据库协议、SSH | DNS、NTP、视频、游戏、QUIC 底层 |

**MTU 与分片（和 UDP 关系最大）**：以太网常见 **MTU 1500**，扣掉 IP 头，UDP 载荷一般建议 **≤ 1472 字节** 一段（具体还看是否带 IP 可选头）。UDP 包太大时，可能在 **IP 层被分片**；任一片丢失，**整包重组失败**，等于「丢包放大」。所以大包要么改 TCP，要么应用层自己分片并编号，要么用 Path MTU 发现（复杂）。面试爱问：**UDP 为什么不无限大**——根因是 IP 分片脆弱。

**UDP 校验和**：可选在 IPv4、强制在 IPv6；能发现传输中的比特错误，但**不保证送达、不防重放**，别和「安全」混为一谈。

### 4.3 How：Python UDP

`socket.SOCK_DGRAM`，用 `sendto` / `recvfrom`，无需 `listen`/`accept`。见 `demos/02_udp_server_client.py`。

**写 UDP 服务端时多留一手：**`recvfrom` 返回的缓冲区要够大；若业务需要「像连接一样」区分客户端，要自己维护 `(ip, port) -> 状态` 表。广播/组播（`SO_BROADCAST`、Multicast）也是 UDP 常见玩法，本课不展开，知道「实时发现协议很多走 UDP」即可。

### 4.4 常见误区

- **误区**：UDP 一定比 TCP 快。不一定：丢包严重时应用层重传可能更糟；要实测。
- **误区**：DNS 只有 UDP。大响应可能走 TCP（例如区域传输、大 TXT）。

### 4.6 深度补充：`recvfrom` 多出来的那一元组是什么

```python

# addr 形如 ("192.168.1.5", 5353)  —— 对端 IP + 对端 UDP 端口
```

无连接的 UDP 没有 `accept()`，服务端必须**从每个包里读出「谁发来的」**，才能按源地址回复。若你写游戏服务器，还要防 **IP 欺骗**（伪造源地址发大包）——那是安全专题，这里只建立「addr 必须信但要校验」的意识。

### 4.7 本节小结（三张卡片）

- **一句话**：UDP = **无连接、不可靠、有报文边界**；大包要小心 IP 分片。  
- **典型场景**：DNS、视频、游戏、QUIC；或「局域网发现」类广播。  
- **和 A1 类比**：UDP 像一条 **只尽力投递、不保序、不自动重传** 的消息队列 —— 你 `sendto` 了，内核尽量发，不替你排队重传。

## 5. DNS：把"人话"翻译成 IP

### 5.1 Why：人类记域名，路由认 IP

浏览器访问 `api.openai.com`，第一步是问 DNS：**这个名字对应的 IP（和可选的其它记录）是什么**。没有 DNS，你就得背 IP。

### 5.2 What：解析链路与记录类型

- **递归解析器**：通常是你路由器或运营商或 `8.8.8.8` 替你层层问，直到拿到结果。
- **权威 DNS**：域名注册商或 CDN 配置的最终答案来源。
- 常见记录：**A**（IPv4）、**AAAA**（IPv6）、**CNAME**（别名）、**TXT**（验证域名所有权等）。

**TTL**：缓存多久。TTL 短，切流量快；TTL 长，省查询、但故障切换慢。

### 5.2.1 递归解析在干什么（口述版）

1. 应用调用 `getaddrinfo` / 浏览器要访问 `api.example.com`。
2. **stub resolver**（本机 `systemd-resolved`、macOS 解析器或你手动设的 DNS）把问题转给 **递归解析器**（路由器、运营商 `202.x`、公共 `8.8.8.8` / `223.5.5.5`）。
3. 递归解析器若缓存未命中，从**根** `.` → **顶级域** `com` → **权威** `example.com` 逐级问下去（迭代查询），直到拿到 **A/AAAA**，必要时跟 **CNAME 链**。
4. 结果带 TTL 写进各级缓存；你 `dig` 看到的秒数就是「这条答案还能信多久」。

**负面缓存**：域名不存在（NXDOMAIN）也会被缓存一段时间，刚注册/刚改错时会出现「我这通了别人还不通」的玄学，本质是 TTL。

**安全与隐私（了解即可）**：DNS 明文可被劫持/窥探，所以有 **DNS over TLS / HTTPS (DoT/DoH)**；浏览器可能走 DoH 而系统 `dig` 仍走传统 53 端口，**排障时两边结果不一致**不要惊讶。

### 5.3 How：本机怎么查

```bash
# 看解析结果与 TTL
dig +noall +answer api.openai.com
```

Python 标准库：

```python
import socket

# 同时支持 IPv4/IPv6 的解析（底层 getaddrinfo）
infos = socket.getaddrinfo("www.example.com", "https", type=socket.SOCK_STREAM)
for fam, typ, proto, canon, sockaddr in infos:
    print(fam, sockaddr)
```

`requests`/`httpx` 发请求前内部会做类似解析；**DNS 失败**时你会看到 `NameResolutionError` 一类错误，与 HTTP 状态码无关。

### 5.4 常见误区

- **误区**：改完 DNS 立刻全球生效。错，受 TTL 与各级缓存影响，可能要等很久。
- **误区**：`/etc/hosts` 优先级高于 DNS（通常），本地劫持调试时很有用。

### 5.6 深度补充：常见记录类型一张表背熟（面试够用）

| 类型 | 作用（一句话） | 备注 |
|------|----------------|------|
| **A** | 域名 → IPv4 | 最常用 |
| **AAAA** | 域名 → IPv6 | 双栈时常与 A 并存 |
| **CNAME** | 别名 → 另一个域名 | 解析要再跟一跳；**根域名 `@` 上 CNAME 有历史限制**（云厂商常提供 ALIAS/ANAME 变通） |
| **MX** | 邮件交换器 | 有优先级数值 |
| **TXT** | 任意文本 | SPF、DKIM、域名验证 |
| **NS** | 哪台是权威 DNS | 委派 |
| **PTR** | IP → 域名 | 反查，邮件反垃圾常用 |

### 5.7 深度补充：`dig` 输出逐行读（A1 式慢读）

```bash
dig +noall +answer api.openai.com
```

典型一行形如：

```text
api.openai.com.  300  IN  A  162.159.140.245
```

从左到右：**所有者名**（末尾点表示 FQDN）、**TTL 秒**、**class 几乎总是 IN**、**类型 A**、**Rdata（IPv4）**。TTL 从 300 变 60 说明权威或上游刚调过缓存策略。

### 5.8 本节小结（三张卡片）

- **一句话**：DNS 是**分布式数据库 + 缓存**；失败表现常为「Name or service not known」。  
- **典型场景**：切 CDN、切机房后「有的地方已生效有的地方没有」= TTL + 多级缓存。  
- **和 A1 类比**：缓存链像 **多层装饰器** —— 每一层都可能短路返回，你要清楚「谁先被问到」。

## 6. HTTP 协议详解：请求与响应的一生

### 6.1 Why：应用层事实标准

HTTP（HyperText Transfer Protocol）是 Web 与绝大多数 API 的**应用层协议**。它约定：客户端发**请求报文**，服务端回**响应报文**；通常底下承载在 **TCP** 上（HTTPS 则是 TLS 再承载 HTTP）。

### 6.2 What：报文结构

**请求**：

```
方法 路径 HTTP版本\r\n
Header1: 值\r\n
Header2: 值\r\n
\r\n
[可选 body]
```

**响应**：

```
HTTP版本 状态码 原因短语\r\n
Header1: 值\r\n
\r\n
[可选 body]
```

行结束符在协议里是 **CRLF**（`\r\n`），不是单独 `\n`。`Host` 在 HTTP/1.1 里**必须有**，否则很多服务器直接 400。

**请求目标（request-target）的两种常见形式：**

- **origin-form**（最常用）：`GET /path?query HTTP/1.1`，再靠 `Host` 头指明虚拟主机。反向代理后面挂多站点全靠这个组合。
- **absolute-form**：`GET http://example.com/path HTTP/1.1`，主要用于**客户端经 HTTP 代理**转发时告诉代理完整 URL；直连源站时少见。

**Content-Length vs 分块（chunked）**：有 body 时，要么给出精确字节数 `Content-Length`，要么用 `Transfer-Encoding: chunked` 流式一块块传。二者冲突时以 RFC 规定为准；写库时别混用。**HEAD 响应**可以没有 body，但 `Content-Length` 可能表示「若发 GET 会有多长」，读规范时注意语义。

### 6.3 How：用最小 TCP 服务器看裸 HTTP

`demos/04_http_server.py` 用标准库搭了一个极简 HTTP 服务：读一行请求行、可选读头、回固定响应。适合理解"HTTP 就是一段按格式的文本 + 可选二进制 body"。

用 curl 对照：

```bash
curl -v http://127.0.0.1:8000/
```

`-v` 会打印请求行、响应行和头，是后端必备技能。

### 6.4 常见误区

- **误区**：HTTP 报文全是 UTF-8 文本。头基本是 ASCII；body 编码由 `Content-Type` 决定，可能是二进制。
- **误区**：URL 里看见的中文就是服务器收到的样子。实际上会 **percent-encode**（UTF-8 字节转 `%XX`）。

### 6.6 深度补充：一份「最小但合法」的 HTTP/1.1 请求（逐行注释）

下面字节流中每行结尾都是 **`\r\n`**（回车换行），**肉眼不可见**，但协议要求它；只用 `\n` 在严格服务器上会 400。

```http
GET /hello?name=world HTTP/1.1
Host: 127.0.0.1:8000
User-Agent: curl/8.0
Accept: */*
Connection: close

```

逐行读：

1. **`GET /hello?name=world HTTP/1.1`** —— **方法**、**请求目标（路径+查询）**、**协议版本**。查询串 `?name=world` 仍是这一行的一部分。  
2. **`Host: ...`** —— HTTP/1.1 **强制**；同一 IP 上虚拟主机靠它区分站点。  
3. **`User-Agent` / `Accept`** —— 客户端自报家门与可接受表示；可省略但不专业。  
4. **`Connection: close`** —— 告诉对端响应后关连接；调试时好用，高并发站点常 keep-alive。  
5. **空行** —— 表示**头结束**；本例无 body，空行后立刻可以开始等响应。  

**响应镜像读法**（极简）：

```http
HTTP/1.1 200 OK
Content-Type: text/plain; charset=utf-8
Content-Length: 5

hello
```

- 响应行：`200` 是状态码，`OK` 是**原因短语**（给人看的，机器别依赖它判逻辑）。  
- `Content-Length: 5` 声明 body 正好 5 字节 → 紧接着的 `hello` 五个字母读完即一条完整响应（在 keep-alive 下读边界全靠它）。

### 6.7 深度补充：URL 拆给 HTTP 的哪几部分

`https://user:pass@api.example.com:8443/v1/chat?stream=1#frag`

- **`https`**：决定默认端口 443，并走 TLS。  
- **`user:pass`**：Userinfo（**不推荐明文密码在 URL**，日志与 Referer 会泄露）。  
- **`api.example.com`**：给 TLS **SNI** 与 HTTP **Host**（注意 Host **不含** userinfo，**不含** fragment）。  
- **`:8443`**：显式端口；省略时用默认端口。  
- **`/v1/chat`**：出现在请求行的 path；`?stream=1` 在查询串。  
- **`#frag`**：**不发给服务器**；只在浏览器端处理。

### 6.8 本节小结（三张卡片）

- **一句话**：HTTP/1.1 是**文本头 + 可选 body**；**CRLF 与空行**是解析骨架；`Host` 与长度信息决定读多少字节。  
- **典型场景**：自己写 `recv` 循环读 HTTP 时，必须先读到空行再按 `Content-Length`/chunked 读 body。  
- **和 A1 类比**：报文像 **多行字符串协议** —— `split("\r\n\r\n", 1)` 思想对头，但生产要用严谨解析器（httplib、hyper、h11）。

## 7. HTTP 方法：GET、POST 和它的朋友们

### 7.1 Why：用动词表达意图（在 REST 风格下）

方法（Method）告诉服务器**想对资源做什么**。REST 里常把方法语义和资源 URL 组合使用。

### 7.2 What：常见方法与语义

| 方法 | 典型语义 | 是否幂等* | 是否有 body |
|------|----------|-----------|-------------|
| GET | 获取 | 是 | 通常无（不规范） |
| POST | 创建 / 触发动作 | 否 | 常有 |
| PUT | 全量替换 | 是 | 常有 |
| PATCH | 部分更新 | 视实现 | 常有 |
| DELETE | 删除 | 是 | 可有可无 |

\***幂等**：同样请求执行多次，**资源状态**与执行一次一致（数学意义在 HTTP 里是约定，不是编译器保证）。

**安全**：GET/HEAD/OPTIONS 不应产生副作用（仍可能被日志、统计记录，但不应改业务数据）。

**PATCH 与幂等（补一句）**：RFC 里 PATCH **不保证**幂等——同一段 JSON Patch 执行两次可能第二次是 no-op，也可能你写的业务「再减一次库存」就错了。设计接口时要么 PATCH 语义定义成幂等（如绝对量替换某字段），要么用 **ETag + If-Match** 做乐观锁，避免重试搞出幽灵更新。

### 7.3 How：`requests` 演示

`demos/03_http_requests.py` 对 httpbin 演示了 GET/POST/PUT/DELETE 与 `params`、`json=`、`headers`。注意：`data=` 与 `json=` 触发的 `Content-Type` 不同。

### 7.4 常见误区

- **误区**：GET 绝对不能带 body。RFC 不禁止，但中间代理、缓存、客户端实现可能不靠谱，**工程上别用 GET 传 body**。
- **误区**：POST 一定比 GET 安全。错，安全靠 **HTTPS**，不靠方法；GET 只是更容易被缓存、进浏览器历史。

### 7.5 Checkpoint

### 7.6 深度补充：HEAD、OPTIONS、CONNECT 是干什么的

| 方法 | 典型用途 |
|------|----------|
| **HEAD** | 与 GET 相同资源，但**不要 body**；只要头（查缓存 `ETag`、测存活）。 |
| **OPTIONS** | 问「这个 URL 允许哪些方法」；**CORS 预检**就是 OPTIONS（见 §13）。 |
| **CONNECT** | 建隧道，常用于 **HTTPS 代理**（客户端与代理说「请帮我 TCP 接到 example.com:443」）。 |
| **TRACE** | 回显收到的请求链路（**生产应禁用**，有安全风险）。 |

### 7.7 深度补充：`data=` vs `json=` vs `files=`（requests 必踩三角）

```python
import requests

# 1) form-urlencoded：默认 application/x-www-form-urlencoded
requests.post(url, data={"a": 1})

# 2) JSON：默认 application/json；body 是 JSON 序列化结果
requests.post(url, json={"a": 1})

# 3) multipart：上传文件或混合格式
requests.post(url, files={"f": open("x.bin", "rb")})
```

- **`data=dict`**：像 HTML `<form method=POST>` 提交；Flask 里常进 `request.form`。  
- **`json=dict`**：现代 API；FastAPI 里常进 Pydantic 模型。  
- **混用**：可读性差，网关与签名容易踩坑；团队规范里通常**禁止「又 data 又 json」除非文档写明」**。

### 7.8 本节小结（三张卡片）

- **一句话**：方法 = **动词**；幂等/安全是**协作契约**，不是语言关键字保证。  
- **典型场景**：写 SDK 时把「可重试」与幂等写进文档；支付类 POST 必须 idempotency key。  
- **和 A1 类比**：方法语义像 **魔术方法约定** —— `__eq__` 该怎么表现，团队要遵守协议，否则「对象相等」全乱。

## 8. HTTP 状态码：1xx–5xx 全家桶

### 8.1 Why：机器可读的第一句结论

状态码让客户端**在不解析 body** 的情况下决定重试、跳转、换 Token、还是直接报错。排障时先看状态码大类，再看具体码。

### 8.2 What：分类

- **1xx**：信息性（如 101 Switching Protocols，WebSocket 升级）
- **2xx**：成功（200 OK，201 Created，204 No Content）
- **3xx**：重定向（301 永久，302/307/308 临时，304 Not Modified）
- **4xx**：客户端问题（400 坏请求，401 未认证，403 无权限，404 未找到，429 限流）
- **5xx**：服务端问题（500 内部错误，502 坏网关，503 不可用，504 网关超时）

**再记几个日常会碰到的码：**

| 码 | 含义（简） | 典型场景 |
|----|------------|----------|
| **202 Accepted** | 已接受处理，结果稍后给 | 异步任务、队列投递 |
| **206 Partial Content** | 分段内容 | 断点续传、Range 下载 |
| **304 Not Modified** | 缓存仍有效 | 带 `If-None-Match` / `If-Modified-Since` 的条件 GET |
| **409 Conflict** | 与当前资源状态冲突 | 版本号乐观锁失败、重复创建 |
| **412 Precondition Failed** | 前置条件不满足 | If-Match ETag 不匹配 |
| **413 Payload Too Large** | body 太大 | 上传超 nginx `client_max_body_size` |
| **414 URI Too Long** | URL 太长 | GET 塞了巨量 query |
| **418 I'm a teapot** | RFC 玩笑 | 少数框架彩蛋，真排障很少见 |

**原则**：状态码表达 **HTTP 传输层结论**；业务成功/失败用 **2xx + 明确 body schema** 或 **4xx 可机器处理的 problem+json**，避免「全 200 + error 字段」除非全链路工具都认。

### 8.3 How：几个高频辨析

- **401 vs 403**：401 是"你是谁？"（未登录/Token 无效）；403 是"我知道你是谁，但不让你进"。
- **502 vs 503 vs 504**：502 网关从上游拿到**无效响应**；503 服务**主动**说现在不可用（可配合 `Retry-After`）；504 网关**等上游超时**。
- **301 vs 302**：301 搜索引擎与客户端会更多**固化**到新 URL；302 历史上实现混乱，API 里要小心缓存。

### 8.4 常见误区

- **误区**：返回 200 但 body 里 `{"error": "..."}` 算成功。对 HTTP 层算成功，对业务层是**反模式**（排查、监控、缓存全乱），除非团队明确约定且工具链支持。

### 8.5 Checkpoint

- [ ] 504 一般是谁超时：客户端、网关、还是上游应用？
- [ ] 429 出现时，响应头里常配合什么字段告诉客户端多久后重试？

### 8.6 深度补充：2xx / 3xx 速查扩充（背这些比背 418 有用）

**2xx（成功族，细语义）**

| 码 | 短语 | 何时用 |
|----|------|--------|
| **200** | OK | 通用成功；GET 有 body、PUT/PATCH 成功常返回表示 |
| **201** | Created | 已创建资源；常配合 `Location` 指向新 URI |
| **202** | Accepted | 已接受异步处理 |
| **204** | No Content | 成功但**无 body**（DELETE 成功、触发型 POST 无返回体） |

**3xx（重定向族，缓存与 SEO 敏感）**

| 码 | 要点 |
|----|------|
| **301** | 永久迁移；浏览器与爬虫会更新书签/权重到新 URL |
| **302** | 历史上实现混乱，常被当成临时；新代码优先 **307/308** 表达临时重定向语义 |
| **303** | POST 后见其它资源（PRG 模式防刷新重复提交） |
| **304** | 条件 GET 未改；body 空，用缓存副本 |

### 8.7 深度补充：4xx 里「鉴权三角」401 / 403 / 419（扩展）

- **401 Unauthorized**：未认证或凭证无效 —— 「请先登录 / 换 Token」。  
- **403 Forbidden**：已识别身份但无权限 —— 「这资源你不能看」。  
- **419**（非标准但 Laravel 等使用）：Session 过期 —— 前端常配合跳转登录页。

**412 / 428**：乐观锁与前置条件（与 `If-Match`、`If-Unmodified-Since` 搭配）；写协作编辑、库存扣减 API 时会碰到。

### 8.8 本节小结（三张卡片）

- **一句话**：状态码是**跨语言、跨客户端**的契约；业务错误不要藏进 200。  
- **典型场景**：网关 502/504 与业务 500 分流；429 指数退避 + `Retry-After`。  
- **和 A1 类比**：状态码像 **异常类型体系** —— 先分大类（4xx/5xx）再 catch 具体子类。

## 9. HTTP 头部：信封上的备注

### 9.1 Why：元数据与协商

头部携带**与表示无关的控制信息**：缓存、认证、内容类型、编码、CORS、Cookie……body 是货物，头部是快递单上的备注。

### 9.2 What：开发常见头

- **`Content-Type`**：body 类型，如 `application/json; charset=utf-8`。
- **`Content-Length`**：body 字节数（与 chunked 二选一逻辑）。
- **`Authorization`**：认证信息，常见 `Bearer <token>`、`Basic ...`。
- **`User-Agent`**：客户端身份；有些 API 会限 UA。
- **`Accept`**：客户端能接受的表示类型（内容协商）。
- **`Cache-Control` / `ETag` / `Last-Modified`**：缓存与条件请求。
- **`Cookie` / `Set-Cookie`**：状态（见第 12 节）。

### 9.3 How：在 Python 里看头

```python
import requests

r = requests.get("https://httpbin.org/get", timeout=10)
print(r.headers.get("Content-Type"))
print(r.request.headers)  # 实际发出的请求头
```

### 9.4 常见误区

- **误区**：头部大小写敏感。HTTP/1.x 头**名**不区分大小写；HTTP/2 传输时会进一步规范化，但语义相同。
- **误区**：自定义头可以随便起。建议 `X-` 前缀或反向域名风格，避免与未来标准冲突。

### 9.5 Checkpoint

- [ ] JSON API 通常用什么 `Content-Type`？
- [ ] 请求头和响应头各举一个与缓存相关的名字。

### 9.6 深度补充：「内容协商」与压缩——Accept 家族

客户端声明自己能吃什么格式、什么压缩：

```http
Accept: application/json
Accept-Language: zh-CN,en;q=0.8
Accept-Encoding: gzip, br
```

服务端用 **`Content-Type`**、**`Content-Language`**、**`Content-Encoding: gzip`** 回应。若服务端**无法**提供任何一种 Accept 里的类型，可返回 **406 Not Acceptable**（实际 API 很少严格实现，但标准如此）。

**`Vary` 头**：告诉缓存「响应随哪些请求头而变」。CDN 常见 `Vary: Accept-Encoding`，否则可能把 gzip 版误缓存给不支持 gzip 的客户端。

### 9.7 深度补充：Range / 206 —— 断点续传与大文件

```http
GET /video.mp4 HTTP/1.1
Range: bytes=0-1048575
```

- 服务端若支持，返回 **206 Partial Content**，`Content-Range` 标明这一段在整个资源中的位置。  
- 下载器多线程拉同一文件、浏览器暂停续传，都依赖这套。

### 9.8 深度补充：请求追踪头（排障必备）

| 头 | 用途 |
|----|------|
| **`X-Request-Id` / `X-Correlation-Id`** | 全链路日志串联（命名团队统一即可） |
| **`Traceparent`（W3C Trace Context）** | 分布式追踪标准，OpenTelemetry 生态 |

**不要**把密码、Token 放进自定义 `X-*` 头当「隐蔽」——TLS 外仍有日志与代理可见风险；鉴权用 **`Authorization`** 并配合 HTTPS。

### 9.9 本节小结（三张卡片）

- **一句话**：头 = **元数据 + 控制面**；缓存、认证、内容表示、追踪都靠它。  
- **典型场景**：CDN 不更新 → 查 `Cache-Control` / `ETag` / `Vary`；gzip 乱码 → `Content-Encoding` 与解压不一致。  
- **和 A1 类比**：头字典像 **函数的 kwargs** —— 核心参数走位置（请求行/path），其余全进 kwargs。

## 10. HTTP 版本演进：1.0 → 1.1 → 2 → 3

### 10.1 Why：性能与安全推动换代

核心矛盾：**减少延迟**（连接数、RTT）、**提高带宽利用率**（多路复用）、**加密成为默认**。

### 10.2 What：各版要点

- **HTTP/1.0**：一请求一连接，用完就关，慢。
- **HTTP/1.1**：**持久连接**（默认 keep-alive）、**Host**、**chunked** 传输、管线化（实际少用）。
- **HTTP/2**：基于 TLS 的实践中主流；**二进制分帧**、**多路复用**（一条连接并行多个请求）、**HPACK** 头压缩、服务器推送（用得少）。
- **HTTP/3**：基于 **QUIC（UDP）**，减少握手 RTT，改善丢包时的队头阻塞；连接迁移对移动网络更友好。

### 10.2.1 为什么 1.1 会「慢」：队头阻塞（HOL）

HTTP/1.1 虽然默认 **keep-alive**（同一条 TCP 上串行多个请求），但**响应必须按顺序完全返回**，前面的响应慢会堵住后面的——这叫 **HTTP 层的队头阻塞**。浏览器 workaround 是**对同一域名开 6 个左右并行 TCP**（域名分片、雪碧图、打包 JS，都是历史债）。

HTTP/2 在**一条 TCP** 上用 **stream ID** 多路复用，多个请求/响应帧可交错，**从 HTTP 角度**不再互相堵。但若底层 **TCP 丢一个包**，整个 TCP 窗口要等重传，所有 stream 一起等——这是 **TCP 层的队头阻塞**。HTTP/3 把传输换成 **QUIC**，丢包只影响对应 stream，其它 stream 可继续传。

**TLS 与 HTTP/2**：浏览器实践里 h2 几乎总是 **TLS 上跑**（`h2` ALPN）。本地明文 h2 少见，调试时注意工具是否支持。

### 10.3 How：你怎么感知版本

浏览器 DevTools → Network → Protocol 列；curl：

```bash
curl -I --http2 https://www.cloudflare.com
```

### 10.4 常见误区

- **误区**：HTTP/2 一定比 1.1 快。小文件、低延迟场景收益有限；还要 CDN/服务端支持。

### 10.5 Checkpoint

- [ ] HTTP/2 解决 HTTP/1.1 的什么痛点？
- [ ] HTTP/3 传输层主要依赖 TCP 还是 UDP？

### 10.6 深度补充：HTTP/2 的「帧」脑子里长什么样（不必会手撕二进制）

HTTP/2 把通信切成 **帧（frame）**，公共头里有 **stream ID**。同一 TCP 连接上，stream 5 的请求帧可以与 stream 7 的响应帧**交错发送**——接收端按 stream ID 重组。

**HPACK**：压缩头字段，尤其重复 `cookie:`、`:authority` 等。副作用：**HPACK  bomb** 曾是实现漏洞来源——写库的人要考虑，用库的人知道「头也有攻击面」即可。

**Server Push**：服务端主动推资源；实践中因缓存一致性难搞，**用得少**；HTTP/3 也在弱化相关叙事。

### 10.7 深度补充：从 ALPN 看「同端口上怎么选 h2」

TLS **ClientHello** 里带 **ALPN 扩展**，列出客户端支持的 `http/1.1`、`h2`、`h3`。服务器选中一个回在 **ServerHello**。所以：**443 端口上可以同时协商出 h2 或 HTTP/1.1**，不靠魔法，靠 TLS 扩展。

### 10.8 本节小结（三张卡片）

- **一句话**：1.1 解决「连接复用与 Host」；2 解决「同连接多请求并行」；3 用 QUIC 解决「TCP 层队头阻塞 + 握手 RTT」。  
- **典型场景**：静态资源多、TLS 往返高 → CDN 开 HTTP/2/3；调试老设备 → 仍要会读 HTTP/1.1 文本。  
- **和 A1 类比**：版本演进像 **Python 2→3** —— 表面兼容、底层模型变，迁移要测全链路。

## 11. HTTPS 与 TLS：加密信件与公章

### 11.1 Why：防窃听、防篡改、验身份

裸 HTTP 经过的每一个路由器都能看到内容。HTTPS = HTTP over **TLS**：**加密**（保密性）、**MAC/AEAD**（完整性）、**证书链**（服务端身份认证）。可选 **mTLS**（双向证书）做高安全内网。

### 11.2 What：证书与握手直觉

- **服务器证书**含公钥与域名（SAN/CN）等信息，由 **CA** 签名；客户端用内置信任根验证链是否可信。
- **TLS 握手**（1.2/1.3 细节不同）直觉版：协商算法、验证证书、用**密钥交换**得到会话密钥，后续对称加密传 HTTP。

自签名证书：没有公共 CA 背书，浏览器会警告——适合内网调试，不适合对公网用户。

### 11.2.1 TLS 1.2 与 1.3（直觉差异）

- **TLS 1.2**：常见 **2-RTT** 完整握手（还可与 TCP 三次握手叠加），密码套件多、历史包袱重；配置错误易出现弱算法。
- **TLS 1.3**：砍掉大量不安全算法，**1-RTT**（甚至 **0-RTT** 重连，有重放风险需慎用）；握手更快、实现更简单。

无论版本，**握手完成后**对称密钥用于 **AEAD**（加密+认证），篡改密文会在解密时失败。你调 API 看到的 `SSLError`、`certificate verify failed`，多在握手或校验阶段抛出。

### 11.3 How：命令行看一眼

```bash
openssl s_client -connect www.google.com:443 -servername www.google.com </dev/null 2>/dev/null | openssl x509 -noout -dates -subject
```

Python `requests` 默认校验证书；测试环境可关校验（**极度危险，仅调试**）：

```python
requests.get("https://...", verify=False)  # 不要提交到生产代码
```

### 11.4 常见误区

- **误区**：HTTPS 只加密 body。错，**路径与查询串在 TLS 里也是加密的**（但 SNI 在部分场景可能泄露域名给中间设备，另有 ESNI/ECH 等演进）。
- **误区**：用了 HTTPS 就防 XSS。错，XSS 是浏览器执行恶意脚本，和传输加密不同层面。

### 11.5 Checkpoint

- [ ] TLS 主要解决哪三类威胁？
- [ ] 为什么自签名证书浏览器会报警？

### 11.6 深度补充：证书链、信任锚、CN/SAN 到底校验什么

浏览器内置一批 **根 CA 证书**（信任锚）。服务器发来的是 **叶子证书**（你的网站）+ 可选 **中间证书**；验证链：**叶子签名的域名必须匹配访问的 host（SAN 为主，CN 为辅）**，且每一级签名直到某个根为止。

**常见失败**：证书只签了 `www.example.com`，你访问 `example.com` → 不匹配；**证书过期**；**链不全**（少中间证书，部分客户端能猜、部分不能）。

### 11.7 深度补充：mTLS、OCSP Stapling（知道名字即可排障）

- **mTLS（双向 TLS）**：客户端也出示证书，常用于金融/内网服务网格。排障时客户端要配 `cert=`/`key=`。  
- **OCSP Stapling**：服务器在 TLS 握手时附带「证书未吊销」证明，减少客户端再去 CA 问一遍的延迟。

### 11.8 深度补充：Python 里三类 TLS 相关异常（读 traceback 用）

| 典型异常/信息 | 可能原因 |
|----------------|----------|
| `CERTIFICATE_VERIFY_FAILED` | 系统 CA 不全、自签证书、错误 `server_hostname`、抓包 MITM |
| `SSL: WRONG_VERSION_NUMBER` | 对 `https` 端口发了明文 HTTP 或反了 |
| `SSLEOFError` | 对端突然关 TLS，或中间设备注入 RST |

### 11.9 本节小结（三张卡片）

- **一句话**：TLS = **身份（证书）+ 密钥协商 + 对称加密传 HTTP**；HTTPS 不是可选安全补丁。  
- **典型场景**：公司抓包解密要装根证书；K8s Ingress 证书续期失败 → 浏览器突然红锁。  
- **和 A1 类比**：证书链验证像 **import 系统查模块路径** —— 少一层中间模块，解析就失败。

## 12. Cookie / Session / Token / JWT：状态保持方案的四次进化

### 12.1 Why：HTTP 默认"无状态"

HTTP 本身不记住"你是谁"。登录后要识别同一用户，需要**状态机制**：从 Cookie、Session 到 Token、JWT，本质是**用凭证换识别**。

### 12.2 What：四种东西各是什么

- **Cookie**：浏览器存的一小段字符串，每次同站请求可自动带上。服务端用 `Set-Cookie` 下发。
- **Session**：服务端内存或 Redis 里存的会话数据，通常用 **session id** 当钥匙；session id 常放在 Cookie 里。
- **Token**：广义上任何"拿着就能证明身份"的字符串；常指随机不透明 token，服务端查库或缓存校验。
- **JWT**（JSON Web Token）：**自包含**的三段 Base64（header.payload.signature），payload 里放声明（如 `sub`、`exp`），用密钥签名；服务端用签名验证真伪，**不必每次查 session 存储**（但要注意撤销与刷新）。

### 12.3 How：安全属性（Cookie 必会）

- **`HttpOnly`**：JS 读不到，降 XSS 偷 Cookie 风险。
- **`Secure`**：只走 HTTPS。
- **`SameSite`**：`Strict` / `Lax` / `None`（`None` 须配合 `Secure`），缓解 CSRF。

**Session 与 JWT 的工程取舍（补全）：**

- **Session**：服务端有状态，**水平扩展**要 **Sticky Session** 或 **集中存储**（Redis）。注销、改密、踢人立刻生效（删 session 即可）。
- **JWT**：服务端可**无状态**只验签，适合微服务网关统一验签。但**吊销困难**（需黑名单、短 `exp` + **refresh token**、或改密钥全局作废）。**refresh token** 应存 HttpOnly Cookie 或安全存储，**access token** 短寿命；别把「长期有效的 JWT」塞 localStorage 还当 Session 用。

动手：`demos/06_session_cookie.py`（Cookie/Session 演示）、`demos/07_jwt_demo.py`（签发与校验思路）。

### 12.4 常见误区

- **误区**：JWT 不能伪造。错，没有密钥任何人都能造 payload，但**签不过**；密钥泄露则全完。
- **误区**：JWT 里可以放密码。绝对不要。payload 只是 Base64，不是加密。
- **误区**：有了 JWT 就不要 HTTPS。错，Bearer Token 被偷一样完蛋。

### 12.5 Checkpoint

- [ ] Session 和 JWT 各适合什么部署形态（单机 vs 多实例）？
- [ ] `SameSite=Lax` 大致防什么攻击？

### 12.6 深度补充：CSRF 攻击长什么样（与 CORS 不同层）

**场景**：用户已登录 `bank.com`，Cookie 里带 session。用户又打开了恶意站 `evil.com`，其中有一张图片：

```html
<img src="https://bank.com/transfer?to=attacker&amount=9999" />
```

浏览器会**带着 bank.com 的 Cookie** 去发 GET（若银行接口设计失误用 GET 做转账就完了）。**同源策略不阻止「发请求」**，它主要限制的是**读跨域响应**；CSRF 利用的是「浏览器自动带 Cookie」。

**防御组合拳**：`SameSite` Cookie、`CSRF Token`（表单隐藏域 + 服务端校验）、关键操作用 **POST + 二次确认**、**校验 Referer/Origin**。

### 12.7 深度补充：`SameSite` 三档对照（背表）

| 值 | 行为直觉 |
|----|----------|
| **Strict** | 仅「第一方」导航带 Cookie；从外链点进来首跳不带 |
| **Lax**（Chrome 默认策略常见） | 跨站 **子资源** 请求不带；**顶级 GET 导航**可带 |
| **None** | 跨站也带，但必须 **`Secure`**（HTTPS） |

### 12.8 本节小结（三张卡片）

- **一句话**：Cookie/Session/JWT 解决「**是谁**」；属性与安全头解决「**被偷、被冒用**」。  
- **典型场景**：SPA + API 跨域 → JWT 放 `Authorization` 还是 Cookie？与 §13 一起设计。  
- **和 A1 类比**：Session 像 **全局 dict 用 id 索引用户态**；JWT 像 **把部分状态序列化进 token 本身**（但有签名与过期边界）。

## 13. CORS：浏览器画的那条红线

### 13.1 Why：同源策略

浏览器允许网页发请求，但**默认禁止 JS 读取跨域响应**（同源：协议+主机+端口一致）。这是安全模型：**恶意网站不能用你的登录态去读银行站点的响应**。

### 13.2 What：简单请求与预检

- **简单请求**（GET/HEAD/POST 且头与 `Content-Type` 在允许集合内等）：直接发，响应里带 `Access-Control-Allow-Origin` 等则 JS 可读。
- **预检（preflight）**：对"非简单"请求，浏览器先发 `OPTIONS`，问服务器是否允许；通过后再发真实请求。

服务端典型头：`Access-Control-Allow-Origin`、`Access-Control-Allow-Methods`、`Access-Control-Allow-Headers`、`Access-Control-Allow-Credentials`（与 `*`、Cookie 组合有细则）。

**带 Cookie 的跨域（.credentials）**：前端 `fetch(..., { credentials: 'include' })` 时，浏览器要求：

1. 响应不能是 `Access-Control-Allow-Origin: *`，必须是**具体源**（例如 `https://app.example.com`）。
2. 需要 `Access-Control-Allow-Credentials: true`。
3. 若用 Cookie，往往还要正确的 **`SameSite`** 与 **HTTPS**（`Secure`）。

否则浏览器会拦截响应体，你在 Network 里仍可能看到 200，但 JS `response.json()` 失败。

### 13.3 How：为什么 Postman 通、浏览器挂

Postman/curl **不执行**同源策略；只有浏览器里的前端 JS 才受 CORS 限制。所以"接口 200 但前端报错"多半是 CORS。

动手：`demos/09_cors_demo.py`。

### 13.4 常见误区

- **误区**：CORS 是防黑客。错，主要防**用户浏览器里的恶意页面**滥用用户身份；不替代鉴权。
- **误区**：`Allow-Origin: *` 配 `Allow-Credentials: true`。浏览器会拒绝这种组合。

### 13.5 Checkpoint

- [ ] 预检请求用的 HTTP 方法是什么？
- [ ] 为什么说 CORS 主要是浏览器的约束？

### 13.6 深度补充：预检请求「长什么样」（两段报文对照）

**第一段：浏览器自动发的 OPTIONS**

```http
OPTIONS /api/v1/chat HTTP/1.1
Host: api.example.com
Origin: https://app.example.com
Access-Control-Request-Method: POST
Access-Control-Request-Headers: authorization,content-type
```

服务端若允许，应返回（示例）：

```http
HTTP/1.1 204 No Content
Access-Control-Allow-Origin: https://app.example.com
Access-Control-Allow-Methods: POST, GET, OPTIONS
Access-Control-Allow-Headers: authorization, content-type
Access-Control-Max-Age: 86400
```

浏览器缓存 **Max-Age** 秒内可减少预检次数。缺任一关键头，真实 POST **不会发出**或发出后 JS **读不了体**。

### 13.7 深度补充：简单请求的「白名单」记忆法（不必背 RFC 全文）

**方法**：GET、HEAD、POST（POST 还要配合「安全头」与有限 `Content-Type`）。  

**Content-Type** 仅当为 `application/x-www-form-urlencoded`、`multipart/form-data`、`text/plain` 之一时，POST 才可能算简单（另有自定义头限制）。**带 `Authorization` 几乎必预检**。

### 13.8 本节小结（三张卡片）

- **一句话**：CORS = **浏览器实施的读权限**；不是防火墙、不是鉴权替代品。  
- **典型场景**：本地 `localhost:5173` 调 `localhost:8000` 也算跨源（端口不同）→ 要配 CORS。  
- **和 A1 类比**：同源策略像 **模块可见性** —— 默认模块外读不到，除非 `export`（Allow-Origin）明确允许。

## 14. RESTful API 设计

### 14.1 Why：可预测、可缓存、可协作

REST 不是框架，是一种**风格**：用 **URL 表示资源**，用 **HTTP 方法表示动作**，用 **状态码表达结果**，尽量**无状态**（鉴权信息放在头里每请求带上）。

### 14.2 What：常见约定

- 资源用名词复数：`/users`、`/users/{id}/orders`。
- 用方法区分操作：GET 查、POST 建、PUT/PATCH 改、DELETE 删。
- **分页**：`?page=&limit=` 或 `cursor`；**过滤**：查询参数。
- **版本**：路径 `/v1/...` 或头 `Accept: application/vnd.xxx+json`（团队选一种坚持）。

### 14.3 How：对照 demo

`demos/08_restful_server.py`：`demos/05_json_api.py` 常与 JSON 组合练习。

### 14.4 常见误区

- **误区**：REST 等于 CRUD。REST 还可表达动作型资源（如 `POST /jobs/{id}/cancel`），关键是**一致性与超媒体/HATEOAS**（实际项目很少做到纯 HATEOAS）。
- **误区**：所有东西塞进 GET 查询参数方便调试。一旦涉及敏感或副作用，应用 POST 并正确设计。

### 14.5 Checkpoint

- [ ] 说一个"不幂等"的 REST 操作例子。
- [ ] 为什么在 API 里滥用 POST 处理所有事会有问题？

### 14.6 深度补充：错误体与 OpenAPI——REST 也要「契约」

仅约定 URL 不够。工程上常配合 **OpenAPI（Swagger）** 描述路径、方法、请求/响应 JSON Schema、错误码。**RFC 7807 Problem Details**（`application/problem+json`）把 `type`、`title`、`detail`、`instance` 标准化，方便客户端统一弹窗与日志。

示例（思想，不必死记字段名）：

```json
{
  "type": "https://api.example.com/errors/out-of-credit",
  "title": "余额不足",
  "status": 402,
  "detail": "账户 acct_123 余额不足以调用本接口",
  "instance": "/logs/trace-abc"
}
```

### 14.7 深度补充：分页 `offset` vs `cursor`

| 方式 | URL 例 | 优点 | 缺点 |
|------|--------|------|------|
| **offset/limit** | `?page=2&size=20` | 实现简单、可跳页 | 数据变动大时**漏项/重复**；大 offset DB 慢 |
| **cursor** | `?after=eyJpZCI6MTIzfQ` | 稳定流式翻页 | 难「跳到第 500 页」；要设计稳定排序键 |

公开 API 大列表几乎都在向 **cursor + 稳定排序** 迁移。

### 14.8 深度补充：`Idempotency-Key` 头（支付与 LLM 工具通用）

客户端对每个「业务上只能发生一次」的操作生成 UUID，放在 **`Idempotency-Key`** 头发给服务端；服务端在 Redis 记「这个 key 处理结果」。**网络重试**时即使 POST 重复到达，也不会重复扣款/重复下单。

### 14.9 本节小结（三张卡片）

- **一句话**：REST = **资源名词 + HTTP 动词 + 状态码 + 文档化契约**。  
- **典型场景**：版本 `/v1`；列表分页；错误用 problem+json；写操作带幂等键。  
- **和 A1 类比**：OpenAPI 像 **类型提示 + stub 文件** —— 给人与机器双重可读契约。

## 15. GraphQL 与 gRPC：HTTP 之外的选择

### 15.1 Why：REST 不是银弹

REST 用「很多 URL + 标准 HTTP 方法」建模资源，**简单、可缓存、易调试**，但前端常遇到：

- 一个页面要 **10 个 REST 请求** 才能拼齐（**过度获取**与 **请求瀑布**）。
- 要么拿一整坨字段（浪费带宽），要么再开一个 **字段裁剪** 的 query 参数约定（各团队自行发明轮子）。

**GraphQL** 和 **gRPC** 从不同方向解决「协作与效率」问题，也都带来**新成本**。

### 15.2 What：GraphQL 在解决什么

- **单一端点**（通常 `POST /graphql`），客户端用 **查询语言**描述「要哪些字段、嵌套关系」。
- **一次往返**拉齐用户 + 文章列表 + 评论摘要，减少瀑布请求。
- **强类型 schema**（SDL），前后端可 codegen 出 TS 类型。

**典型代价（面试/架构必说）：**

1. **HTTP 缓存几乎废了**：同一 URL 不同 body，CDN/浏览器难以像 GET 静态资源那样缓存；要靠 **persisted query**、BFF、或 Apollo 等客户端缓存。
2. **复杂度与 DoS**：恶意或手滑的深层嵌套查询可打爆 DB（**N+1**、笛卡尔积）；必须 **query depth/cost 限制**、DataLoader 批处理、超时。
3. **错误模型**：部分成功 + `errors` 数组，和 REST「一状态码一整体」心智不同；监控要适配。
4. **文件上传**：无统一标准，常配合 REST 分传或 multipart 扩展。

适合：**BFF、移动端、数据图复杂且团队有治理**。不适合：想无脑蹭 HTTP 缓存的纯静态 GET 场景。

### 15.3 What：gRPC 在解决什么

- **IDL 即契约**：`.proto` 定义 service/message，**protobuf 二进制**编码，体积小、解析快。
- **HTTP/2 上帧**：流式 **unary / server streaming / client streaming / bidi** 一等公民。
- **跨语言**：生成各语言 stub，微服务内部调用常见。

**典型代价：**

1. **浏览器不原生支持**：要 **gRPC-Web** + Envoy 等代理，或网关转成 JSON REST 给前端。
2. **调试**：二进制不如 JSON 直观；需 `grpcurl`、反射（reflection）或 good docs。
3. **版本兼容**：`.proto` 字段编号、optional/required 演进要有纪律（field 勿复用）。

### 15.4 What：怎么选型（直觉表）

| 场景 | 更常选 |
|------|--------|
| 公网开放 API、第三方集成 | REST + OpenAPI |
| 移动端/复杂页面数据聚合 | GraphQL（有治理） |
| 内网微服务东西向流量 | gRPC |
| 实时推送 | WebSocket / SSE（见下节） |

### 15.5 Checkpoint

- [ ] GraphQL 的典型代价是什么？
- [ ] gRPC 默认常用什么接口定义格式？

### 15.6 深度补充：`.proto` 一眼能读（A1 式「慢读语法」）

```protobuf
syntax = "proto3";
package demo;

service Greeter {
  rpc SayHello (HelloRequest) returns (HelloReply);
}

message HelloRequest { string name = 1; }
message HelloReply { string message = 1; }
```

- **`service`**：RPC 方法集合，生成代码里会变成 `Stub` 类。  
- **`rpc Name (Req) returns (Res)`**：一次调用的输入输出类型都强类型。  
- **`string name = 1`**：字段号 `1` **永久占用**，勿复用（兼容旧客户端靠字段号）。  

GraphQL 的 **SDL** 也有类似「类型即文档」的味道，但运行时查询更灵活。

### 15.7 本节小结（三张卡片）

- **一句话**：GraphQL 换 **查询灵活性**；gRPC 换 **二进制效率与流式 RPC**。  
- **典型场景**：BFF 聚合用 GraphQL；K8s 东西向用 gRPC；对外开放仍 REST。  
- **和 A1 类比**：`.proto` 像 **TypedDict/dataclass 的跨语言版** —— schema 即法律。

## 16. Python requests：最常用的 HTTP 客户端

### 16.1 Why：人机工程好

`requests` 把 `urllib3` 包了一层，API 直白，是脚本与中小服务的默认选择。

### 16.2 What：必会 API

- `requests.get/post/...`：`params`、`data`、`json`、`headers`、`timeout`。
- **`timeout`**：务必设置（连接超时、读超时可写元组），否则挂死。
- **`Session()`**：连接池、默认头、Cookie 罐，**爬虫与多请求 API 客户端首选**。
- **流式**：`stream=True` + 迭代 `resp.iter_lines()`，适合大文件与 **SSE**。

**Session 与连接池（补全）**：`requests.Session()` 通过 urllib3 **复用 TCP/TLS 连接**，对同一 host 高频调 API 能省握手。注意 **Session 非线程安全**，多线程应每线程一个 Session 或加锁。`with requests.Session() as s:` 离开上下文会关闭连接池。复杂场景用 **HTTPAdapter + Retry** 挂载到 `Session.mount("https://", ...)`。

### 16.3 How：重试与 LLM 调用注意

`requests` 本身不内置重试；用 `urllib3.util.Retry` 配 `HTTPAdapter` 挂到 `Session` 上，或对 429/5xx 做指数退避。**读流**时重试要更谨慎（可能重复消费）。

动手：`demos/03_http_requests.py`、`demos/05_json_api.py`。

### 16.4 常见误区

- **误区**：`resp.json()` 一定成功。只有 `Content-Type` 与 body 真是 JSON 才行；否则抛异常，要先看 `status_code` 与 `text`。
- **误区**：不信任响应就 `raise_for_status()`。对 4xx/5xx 直接抛，便于统一错误处理。

### 16.5 Checkpoint

- [ ] 为什么生产代码必须设 `timeout`？
- [ ] 何时用 `Session` 而不是次次 `requests.get`？

### 16.6 深度补充：带指数退避的 `Retry` + `Session`（逐行注释模板）

```python
import requests
from requests.adapters import HTTPAdapter
from urllib3.util import Retry

def make_session() -> requests.Session:
    s = requests.Session()
    # 只对幂等方法默认重试；POST 默认不重试，除非显式 allowed_methods
    retries = Retry(
        total=5,                          # 最多 5 次（含首次）
        backoff_factor=0.5,               # 0.5, 1, 2… 秒级退避（乘子）
        status_forcelist=(429, 500, 502, 503, 504),
        allowed_methods=("GET", "HEAD", "OPTIONS"),
        respect_retry_after_header=True,  # 429 读 Retry-After
    )
    s.mount("https://", HTTPAdapter(max_retries=retries))
    s.mount("http://", HTTPAdapter(max_retries=retries))
    return s

sess = make_session()
r = sess.get("https://httpbin.org/status/500", timeout=(3.05, 27))
r.raise_for_status()
```

- **`HTTPAdapter`**：把连接池与重试策略**挂到前缀**上。  
- **`timeout=(connect, read)`**：连接阶段与读阶段分别卡死，避免「连不上还傻等」。  
- **POST 重试**：默认危险；若必须做，用 **`Idempotency-Key`**（§14.8）或业务去重。

### 16.7 本节小结（三张卡片）

- **一句话**：`requests` = 好用；生产 = **`timeout` + `Session` + 可控重试 + 日志脱敏**。  
- **典型场景**：爬 OpenAI 兼容流式 → `stream=True` + 读超时；别 `verify=False` 上线。  
- **和 A1 类比**：`Session` 像 **复用 `httplib` 连接的对象** —— 减少重复握手，和复用 DB 连接池同一哲学。

## 17. httpx：同步 + 异步 + HTTP/2

### 17.1 Why：现代替代

`httpx` API 接近 `requests`，同时支持 **`async/await`**，可选 **HTTP/2**，适合 **FastAPI/Starlette** 生态里异步下游调用。

与 `requests` 相比的常见动机：**同一套 API 写同步脚本与异步服务内调用**；需要 **HTTP/2** 时不必再绕 urllib3 实验特性；对 **HTTP/1.1 与 HTTP/2 自动协商**（ALPN）更一体化。

### 17.2 What：用法对照

```python
import httpx

# 同步
with httpx.Client(http2=True) as client:
    r = client.get("https://example.com", timeout=10.0)

# 异步
async with httpx.AsyncClient() as client:
    r = await client.get("https://example.com", timeout=10.0)
```

**`timeout` 在 httpx 里可细化**：`timeout=httpx.Timeout(5.0, connect=2.0)` 分别限制连接、读等（具体参数以文档为准）。**连接池**：`Client` 复用 TCP/TLS，对同一主机的 burst 请求明显省 RTT。

**异步里注意**：`asyncio.gather` 并发 100 个下游前，要想好**对方限流**与**本机文件描述符**；必要时用 **semaphore** 限并发。`aiohttp` 老用户迁移时留意 API 差异（cookies、auth 插件生态）。

### 17.3 常见误区

- **误区**：异步一定更快。错，CPU 密集或 DNS/SSL 仍可能阻塞事件循环；要配合连接池、合理 `gather`。
- **误区**：开了 `http2=True` 对方就一定是 h2。实际取决于服务端 ALPN；失败会静默或协商回 h1（以版本为准，排障时看 Wireshark `Client Hello` 里 ALPN 列表）。

### 17.4 Checkpoint

- [ ] 什么时候你会选 httpx 而不是 requests？
- [ ] 异步并发调用下游时，为什么要加 semaphore？

### 17.5 深度补充：`AsyncClient` + `Semaphore` 模板（逐行）

```python
import asyncio
import httpx

async def fetch_one(client: httpx.AsyncClient, url: str) -> str:
    r = await client.get(url, timeout=10.0)
    r.raise_for_status()
    return r.text[:80]

async def main(urls: list[str]) -> None:
    sem = asyncio.Semaphore(8)  # 最多 8 个请求同时在飞，防把下游打爆

    async with httpx.AsyncClient(http2=True) as client:

        async def bound(u: str) -> str:
            async with sem:       # 先抢并发票
                return await fetch_one(client, u)

        results = await asyncio.gather(*(bound(u) for u in urls), return_exceptions=True)
    print(results)

# asyncio.run(main([...]))
```

- **只建一个 `AsyncClient`**：内部复用连接池；`gather` 并发多个 `bound`。  
- **`return_exceptions=True`**：其中一个 URL 挂不会拖死全体，便于批量排障（记得判断 `isinstance(r, Exception)`）。

### 17.6 本节小结（三张卡片）

- **一句话**：httpx = **requests 手感 + async + HTTP/2 选项**。  
- **典型场景**：FastAPI 路由里 `await client.get` 调下游；限并发用 semaphore。  
- **和 A1 类比**：`gather` 像 **批量启动线程但由事件循环调度** —— 协作式多任务。

## 18. WebSocket 与 SSE 速览

### 18.1 Why：HTTP 请求-响应不够用

经典 HTTP：**一问一答**，服务端不能「主动推」（除非客户端再发请求）。实时通知、协作、**流式 AI 输出**需要「连接尽量保持、数据边算边传」。

- **WebSocket**：先走 **`HTTP/1.1` 升级握手**（`Upgrade: websocket`、`Sec-WebSocket-Key` 等），成功后 TCP 连接上交换**二进制帧**（文本/二进制/ping/pong）。**全双工**：客户端和服务端可随时发帧。适合聊天、游戏、强交互。
- **SSE**（Server-Sent Events）：仍是**普通 HTTP 响应不结束**，`Content-Type: text/event-stream`，body 里按行发 **`data: ...\n\n`**。浏览器用 **`EventSource` API** 自动重连（可配 `Last-Event-ID`）。**仅服务端→客户端**，若要上行仍需另开请求或配合 fetch。

### 18.2 What：与轮询、长轮询对比

| 方式 | 优点 | 缺点 |
|------|------|------|
| 短轮询 | 实现无脑 | 延迟高、浪费请求 |
| 长轮询 | 比短轮询省一点 | 连接仍频繁；服务端要挂住请求 |
| SSE | 基于 HTTP、穿透简单、HTTP/2 多路复用友好 | 单向；老代理缓冲 body 可能坏事 |
| WebSocket | 双向、帧头极小 | 代理/负载均衡要支持 **sticky** 或协议识别；状态机复杂 |

**代理与超时**：Nginx 默认 `proxy_read_timeout` 较短，**SSE/流式 LLM** 常在中间就断；需调大超时或关闭 **buffering**（`proxy_buffering off` / `X-Accel-Buffering: no`）。WebSocket 则要 `Upgrade` 与 `Connection` 相关指令配好。

### 18.3 What：帧与事件（实现时会长什么样）

WebSocket 一帧（简化）：`FIN`、opcode（文本 `0x1` / 二进制 `0x2` / 关闭 `0x8` / ping `0x9`）、payload；大 payload 可 **分片**（多帧一个逻辑消息），应用库通常帮你拼。

SSE 事件（简化）：

```
data: {"token": "hello"}

data: {"token": " world"}

```

空行分隔事件；可带 `event:`、`id:`、`retry:` 行。Python 服务端写 SSE 时记得 **`flush`**，否则被 WSGI/代理缓冲看不到流式效果。

### 18.4 Checkpoint

- [ ] SSE 是双向还是单向？
- [ ] WebSocket 第一下用什么协议升级？
- [ ] 为什么生产里 SSE 常要关代理 buffering？

### 18.5 深度补充：WebSocket 心跳与断线重连（实现视角）

长时间空闲连接会被 NAT 或 LB 悄悄丢掉。协议层用 **Ping/Pong 帧**保活；应用层也可发 JSON `{"type":"ping"}`。客户端要处理 **onclose → 指数退避重连**，并考虑 **last seen message id** 做增量补偿。

### 18.6 深度补充：Python 里读 SSE 流（不用浏览器时）

```python
import requests

with requests.get("https://example.com/stream", stream=True, timeout=(5, 300)) as r:
    for line in r.iter_lines(decode_unicode=True):
        if not line or line.startswith(":"):
            continue
        if line.startswith("data:"):
            payload = line[5:].lstrip()
            print(payload)
```

- **`stream=True`**：不要一次性 `content` 读入内存。  
- **`iter_lines`**：按行增量；`:` 开头是**注释/心跳行**（SSE 规范允许）。  
- **超时元组**：连接 5s，读之间总等待可到 300s（按业务调）。

### 18.7 本节小结（三张卡片）

- **一句话**：WebSocket = **双向帧**；SSE = **单向 HTTP 长响应**。  
- **典型场景**：LLM token 流多用 SSE；协同编辑多用 WebSocket。  
- **和 A1 类比**：SSE 像 **惰性迭代器 yield 一行行事件** —— 消费者 for 循环挂在上面。

## 19. 排查 HTTP 问题：curl、DevTools、抓包

### 19.1 Why：先分离"浏览器问题"还是"网络问题"

同样一个 URL：**curl 通、浏览器挂** → 优先怀疑 **TLS 客户端校验、CORS、Cookie、HSTS、混合内容**。**curl 挂、浏览器也挂** → 优先 **DNS、路由、防火墙、证书链、服务未监听**。

**分层定位**：先 `ping`/`dig`（ICMP 通不代表 TCP 443 通）→ `curl -v` 看卡在哪（DNS、TCP、TLS、HTTP）→ 再决定要不要抓包。

### 19.2 How：curl 最小集

```bash
# 看完整握手与头
curl -v https://api.example.com/v1/foo

# 指定解析到某 IP（绕过 DNS 测 CDN/本机）
curl -v --resolve api.example.com:443:127.0.0.1 https://api.example.com/

# 只看状态码
curl -o /dev/null -s -w "%{http_code}\n" https://example.com
```

**再记几条排障利器：**

```bash
# 显示 TLS 握手与证书链简况
curl -vI https://example.com 2>&1 | sed -n '1,40p'

# POST JSON（与 Python requests 对照）
curl -v https://httpbin.org/post -H 'Content-Type: application/json' -d '{"a":1}'

# 走企业代理调试（示例）
curl -v -x http://proxy.corp:8080 https://example.com
```

`-v` 里 **`>`** 表示**发出去**的字节（请求），**`<`** 表示**收到**的（响应）。`--trace-time` 可看各阶段耗时。

### 19.3 DevTools

Network：Status、Type、Initiator、Timing（DNS/SSL/TTFB）、Response/Headers。**Copy as cURL** 给后端复现。

**TTFB（Time To First Byte）** 大致包含：排队、**DNS**、**TCP+TLS 握手**、请求发送、**服务端排队+执行**、首字节回程。若 SSL 很长而 TTFB 短，问题多在握手证书链；若 Waiting（TTFB）很长，多在**应用或网关等上游**。

**Waterfall**：看是否「串行请求过多」或「单资源阻塞」；与 HTTP/2 多路复用、资源合并策略对照。

### 19.4 Wireshark / tcpdump

过滤器示例：`tcp.port == 443`（TLS 内层默认看不到明文，需密钥日志才能解密 HTTP）。

**实用心算**：抓到 **SYN / SYN-ACK / ACK** 三连看 TCP 是否建立；看有没有 **Client Hello** 判断 TLS 是否开始；**RST** 常是中间墙或本机无进程监听。**明文 HTTP** 在 80 上可直接跟，适合对照自己写的 `04_http_server.py`。

```bash
echo | openssl s_client -connect example.com:443 -servername example.com 2>/dev/null \
  | openssl x509 -noout -issuer -subject -dates
```

- **`-servername`**：SNI，必须和浏览器地址栏主机名一致，否则拿到错误站点证书。  
- 输出里 **issuer** 是谁签的、**subject** 是谁、**notAfter** 何时过期 —— 排障「突然全红」三板斧。

### 19.7 深度补充：mitmproxy / Charles（中间人调试）

在**受控环境**装根证书后，可用 **mitmproxy** 明文看 HTTPS 应用层（与 Wireshark+密钥日志不同路）。**禁止**对用户生产流量做未授权 MITM；仅本机调试自己的 App。

### 19.8 本节小结（三张卡片）

- **一句话**：排障 = **二分法** + **分层证据**（DNS/TCP/TLS/HTTP/应用）。  
- **典型场景**：`curl` 复现 + DevTools Copy as cURL + 网关 access log 三方对表。  
- **和 A1 类比**：抓包像 **`logging.DEBUG` 开到网络栈** —— 噪音大，只在怀疑栈本身时开。

## 20. 和 AI Agent 后端的关联

### 20.1 Why：Agent = 编排 + 大量 HTTP

调用嵌入、向量库、工具 HTTP、LLM API，全是**超时、重试、流式、计费与限流**问题。

Agent 运行时往往是**长链路**：规划 → 多次工具调用 → 再调模型；任一 hop **慢或 429**，用户体验是「卡一下」或「整段失败」。后端工程要把 **HTTP 客户端当一等公民** 设计，而不是 `requests.get` 一行糊上。

### 20.2 What：实战清单

- **鉴权**：`Authorization: Bearer ...`、部分云厂商签名头；密钥只放服务端与环境变量。
- **流式**：OpenAI 兼容接口常见 **SSE**；客户端用 `stream=True` 或 httpx 流式读，**注意异常时半包**。
- **超时**：连接超时、首字节超时、**读空闲超时**（流式中间停顿）。
- **重试**：仅对**幂等**或带 **idempotency key** 的请求；429 读 `Retry-After`。
- **观测**：记录 `request_id`、状态码、延迟、token 用量；便于对账与排障。

**补几条落地细节：**

1. **上下文长度与截断**：模型输入有上限；RAG 拼 prompt 前要估算 token（tiktoken 等），否则 400 或静默截断。
2. **工具调用与幂等**：`POST /orders` 类工具被 LLM **重复触发**很常见；要有 **idempotency key** 或服务端去重。
3. **并发与供应商限流**：多 agent 并行调同一 key，易 **429**；用 **队列 + 退避** 或分散 key（合规前提下）。
4. **流式解析**：SSE 按行缓冲，`\n\n` 才是一个事件；半行时不要当 JSON 解析。异常中断时要 cancel generator / close response，避免连接泄漏。
5. **成本**：流式同样按 usage 计费；日志里打 **prompt_tokens / completion_tokens** 便于审计。

### 20.3 Checkpoint

- [ ] 为什么流式 API 的重试比非流式更难？
- [ ] 429 时优先读响应里哪个头？
- [ ] 为什么 Agent 工具 HTTP 要特别考虑幂等？

### 20.4 深度补充：OpenAI 兼容流式响应——HTTP 层长什么样（概念）

常见模式：**POST** `/v1/chat/completions` + `stream: true` → 响应 **`Content-Type: text/event-stream`** 或 vendor 自定义 chunked JSON line。每一行/每一块是一个 **JSON 对象**（如 `{"choices":[{"delta":{"content":"你"}}]}`）。客户端要：

1. **边读边解析**（不要等 `read` 到 EOF 才 `json.loads` 整块 body）。  
2. 处理 **DONE** 哨兵（OpenAI 用 `[DONE]` 行）。  
3. 对 **非 200** 先读错误 JSON（同一 URL 流式与非流式错误体结构可能不同）。

### 20.5 深度补充：结构化输出与工具调用——仍是 HTTP

**JSON Schema / tool_calls** 等特性不改变传输层：仍是 **TLS 上的 HTTP POST**；只是 body 与解析状态机更复杂。排障时先把 **`response.headers` 与首包 body** 打日志，再谈 prompt。

### 20.6 本节小结（三张卡片）

- **一句话**：Agent 后端 = **编排逻辑 + 稳健的 HTTP 客户端 + 可观测性**。  
- **典型场景**：429 退避、流式读空闲超时、工具 HTTP 幂等、密钥只服务端。  
- **和 A1 类比**：编排像 **把多个迭代器与装饰器链在一起** —— 任一环抛异常要有统一 `finally` 关连接。

## 22. 代理、网关与谁超时了

### 22.1 正向代理 vs 反向代理

- **正向代理（Forward Proxy）**：代表**客户端**去访问外网；企业内网「上网代理」、科学工具常见。服务端看到的是**代理的 IP**，不是你的真实出口（除非透传 `X-Forwarded-For`）。
- **反向代理（Reverse Proxy）**：站在**服务端前面**，代表站点接客；**Nginx、Caddy、云 LB、API Gateway** 都是。客户端只认一个公网 IP/域名，后面挂多台应用机。

Python `HTTP_PROXY` / `HTTPS_PROXY` 环境变量，是让 **requests/httpx 作为客户端**走正向代理；**不是你写的 FastAPI 自动变成反向代理**——反向代理要在前面单独配一层。

### 22.2 网关（Gateway）在链路上的位置

「API Gateway」通常做：鉴权、限流、路由到微服务、TLS 终结、日志。请求路径常为：

`客户端 → CDN/WAF → 入口 LB → 网关 → 业务服务 → DB`

**502 Bad Gateway**：本机（网关）从上游拿到了**无效/空响应**（上游崩了、连接被 RST、非 HTTP 垃圾数据）。

**504 Gateway Timeout**：本机在等上游时**超时**——常见是 **nginx `proxy_read_timeout`** 小于 LLM 流式首包时间，或上游计算太久。此时**你的 Python 代码可能还在跑**，但网关已经切断对客户端的连接。

排障口诀：**先确定 504 是谁返回的**（看响应头 `Server`、body 里 HTML 是否 nginx）、再对齐 **哪一层的 timeout**（客户端 `requests`、LB idle、网关、上游 Gunicorn worker）。

### 22.3 `X-Forwarded-*` 与「应用以为自己在哪」

TLS 在 nginx 终结后，到 uWSGI/Gunicorn 的已是 **明文 HTTP**。应用若要生成绝对 URL、判断 `https://`、记真实客户端 IP，必须信任并解析 **`X-Forwarded-For` / `X-Forwarded-Proto` / `Forwarded`**，且**仅信任来自反向代理的 hop**（否则客户端可伪造）。FastAPI/Starlette 有 `ProxyHeadersMiddleware` 一类机制，生产要配**受信代理 IP 列表**。

### 22.4 深度补充：Nginx 与 SSE/LLM 流式（配置片段只作心智模型）

```nginx
location /v1/ {
    proxy_pass http://127.0.0.1:8000;
    proxy_http_version 1.1;
    proxy_set_header Host $host;
    proxy_set_header X-Forwarded-For $proxy_add_x_forwarded_for;
    proxy_set_header X-Forwarded-Proto $scheme;
    proxy_read_timeout 3600s;      # 流式首包/中间停顿可能很长
    proxy_buffering off;           # 别让 nginx 攒一整坨再吐给客户端
    add_header X-Accel-Buffering no;
}
```

实际值按业务调；**3600s 不是越大越好**——占满 worker 连接数同样是事故。

### 22.5 本节小结（三张卡片）

- **一句话**：代理改变「**谁看见谁**」；网关改变「**超时与错误由谁背锅**」。  
- **典型场景**：504 先对齐 nginx `proxy_read_timeout` 与上游 Gunicorn `timeout`。  
- **和 A1 类比**：转发头像 **显式传 contextvars** —— 不经过代理注入，下游就不知道真实请求上下文。
