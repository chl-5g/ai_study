## 1. 为什么要消息队列

### 1.1 从一个"看起来没问题"的需求讲起

假设你正在给一家公司的 AI Agent 平台做后端。产品经理说了一个需求：

> "用户上传 PDF 之后，我们要自动切分、调 embedding 模型、写入向量数据库，最后通知用户'知识库准备好了'。"

作为一个刚转后端的人，你的第一反应可能是直接写一个 FastAPI 接口：

```python
@app.post("/knowledge/upload")
def upload_pdf(file: UploadFile):
    # 第 1 步：保存文件，大概 200ms
    path = save_file(file)

    # 第 2 步：PDF 切分，大概 2 秒
    chunks = split_pdf(path)

    # 第 3 步：调 OpenAI embedding 接口，每个 chunk 一次网络调用
    # 假设 100 个 chunk、每次 500ms、还可能限流重试，整体 60 秒起步
    embeddings = [openai.embed(c) for c in chunks]

    # 第 4 步：写入 pgvector，大概 1 秒
    db.insert_vectors(embeddings)

    # 第 5 步：发邮件/站内信通知用户，大概 500ms
    send_notification(user_id)

    return {"status": "ok"}
```

这个接口**能跑通**，但几乎每一行都是坑。我们一行一行看它在生产环境会变成什么样。

### 1.2 同步调用的第一个问题：响应慢到离谱

用户点了"上传"按钮之后，浏览器那个转圈圈的 spinner 要转 **60 秒以上**。超过 30 秒大部分浏览器/网关会直接超时，接口还没返回就被切断了。

你说"那我把超时调长"？nginx 默认 60s、AWS ALB 默认 60s、CloudFlare 默认 100s。你可以调一点，但是从用户体验角度，**用户等待超过 3 秒心里就开始烦了**。产品经理会指着你鼻子说：别人的上传都是秒回，你这个怎么要等 1 分钟？

**核心矛盾**：有些任务天生就慢（embedding、训练、文件处理），把它们放在 HTTP 请求里同步执行，就是用户体验的灾难。

### 1.3 同步调用的第二个问题：耦合

看第 5 步 `send_notification`。假设通知服务依赖的 SMTP 服务器挂了 30 秒。在这 30 秒里：

- `upload_pdf` 接口的所有请求都会 hang 住等通知返回
- 就算前面第 1-4 步都成功了，第 5 步失败会把整个接口标为 500
- 用户看到"上传失败"，以为自己的 PDF 没存进去，重新上传一次 → 数据重复

这就是**耦合**：一个小功能的故障，能把一个核心功能拖死。

用日常生活类比：你去点外卖，饿了么 App 如果要等到餐品做好、骑手取到、送到家门口才告诉你"下单成功"，那这个 App 一秒钟用户都留不住。现实中的流程是：**你下单 → 系统立刻回"下单成功"→ 后面做饭、送餐都是异步的，你通过推送消息知道进度。**

### 1.4 同步调用的第三个问题：峰值崩溃

再想象一个场景：你的 Agent 平台搞了个限时活动，10 万用户同时上传 PDF。每个请求占用一个 Worker 线程 60 秒。

- 假设你的 FastAPI 用 4 进程 × 10 个 worker = 40 个并发槽位
- 10 万请求除以 40 个槽位 = 2500 批次
- 2500 批 × 60 秒 = 150000 秒 ≈ **41 小时**才能处理完

而且中间任何一个瞬时请求超过 40 个，多出来的那些就直接**报 503**，用户看到的是"服务不可用"。

这就是**峰值压力没被削平**：前端请求的"瞬时高峰"直接砸到了最慢的那个环节（embedding 接口）。

### 1.5 "快递驿站"类比

解决以上三个问题的思路其实和**快递驿站**一模一样。

想象一下你要给朋友寄一本书：
- **没有快递驿站的世界**：你亲自开车把书送到朋友家，朋友不在就等到他回来。你这一天什么都干不了。
- **有快递驿站的世界**：你把书放进驿站、拿到一个单号、立刻走。快递员按他自己的节奏去配送。朋友不在就先放代收点。
  - 你（发送方）**瞬间解放**
  - 朋友家没人也无所谓（**解耦**）
  - 一天来 1 万个包裹，驿站会排队处理（**削峰**）
  - 单号可以查状态（**追踪**）
  - 包裹丢了有保险兜底（**可靠性**）

**消息队列（Message Queue，MQ）就是系统之间的快递驿站**：
- 生产者把"要做的事情"写成一条消息扔进队列，立刻返回
- 消费者按自己的节奏从队列里取消息、处理、确认
- 队列本身负责缓冲、持久化、重试、限流

改造后的代码是这样的：

```python
@app.post("/knowledge/upload")
def upload_pdf(file: UploadFile):
    path = save_file(file)  # 200ms，同步完成（必须知道文件存成功了）
    # 关键改动:只投递一个任务,立刻返回
    process_pdf_task.delay(path, user_id)
    return {"status": "queued", "message": "我们正在处理你的文件"}
```

接口响应时间从 60 秒变成了 **0.2 秒**。后面那些耗时的活儿，由后台的 Worker 进程慢慢做。

> **读到这里你应该理解：** 消息队列不是"一个新潮的技术组件"，而是"同步调用在生产环境里跑不下去了之后不得不引入的工程解法"。遇到任何耗时操作、第三方依赖、流量不稳定的场景，都要本能地想到"能不能异步化"。

### 1.6 初学者常见误区

- **误区 1**："消息队列太重了，我先同步写，以后有需要再改。" → 事实是"以后"这个词在软件工程里基本等于"永远不"，等系统炸了之后再改代价是原始成本的 10 倍。
- **误区 2**："我用多线程不就解决了吗？" → 多线程解决的是"单机 CPU 利用率"，不解决"任务持久化、重试、跨进程调度、削峰"。你进程一重启任务就全没了。
- **误区 3**："异步 = 快" → 错。异步只是**把等待从用户面前移到后台**，总耗时没变，甚至因为引入队列开销还会稍微变慢。异步的价值是**响应快 + 解耦 + 削峰**。
- **误区 4**："FastAPI 不是有 `async def` 吗？为什么还要 MQ？" → `async def` 是**单进程内**的协程调度，不能跨进程、不持久化、不削峰。FastAPI 的异步 ≠ 任务队列的异步。这两件事经常被搞混。

## 2. MQ 的核心概念

在往下讲之前，先把一堆"术语"讲清楚。后面无论是 Redis Stream、Kafka、RabbitMQ 还是 Celery，本质上都在玩这几个角色。

### 2.1 六个核心角色

| 角色 | 英文 | 做什么 | 类比 |
|------|------|--------|------|
| 生产者 | Producer | 把消息写入队列的那一方 | 寄快递的人 |
| 消费者 | Consumer | 从队列里取消息处理的那一方 | 取快递的人 |
| 消息 | Message | 一条被传递的数据（字节流） | 一个快递包裹 |
| 队列 | Queue | 消息的容器，通常 FIFO | 快递驿站的货架 |
| 主题 | Topic | 一类消息的逻辑分组（发布订阅） | 小区的公告栏 |
| Broker | Broker | 消息队列的服务端进程 | 快递公司本身 |

### 2.2 Producer（生产者）

生产者就是"往队列里写东西"的那个程序。在 Agent 场景里，生产者往往是 **HTTP 接口层**：

```python
# 伪代码
def upload_pdf(file):
    save(file)
    mq.send("pdf_tasks", {"file_id": 123, "user_id": "u-001"})  # ← 这里就是生产者
    return {"status": "ok"}
```

生产者的核心关注点有三个：

1. **消息发出去了吗？** 网络有可能丢包，需要 Broker 回应 "我收到了"（ACK）才算数。
2. **消息发重了吗？** 没收到 ACK 时生产者会重试，有可能 Broker 其实收到了只是 ACK 回复丢了，导致消息发了两次。
3. **发消息的速度比消费快怎么办？** 需要考虑 Broker 的限流或者背压（back pressure）。

### 2.3 Consumer（消费者）

消费者是"从队列里取东西干活"的那一方，通常是一个**独立的长驻进程**（叫 Worker）。

```python
# 伪代码
def worker_loop():
    while True:
        msg = mq.receive("pdf_tasks")  # 阻塞式取一条消息
        try:
            process_pdf(msg)           # 干活
            mq.ack(msg)                # 告诉 Broker:我处理完了,可以删了
        except Exception:
            mq.nack(msg)               # 处理失败,要求重新投递
```

消费者的核心关注点是：

1. **ACK 时机**：先 ack 还是后 ack？先 ack 会丢消息，后 ack 会重复消费。（这个后面专门讲。）
2. **并发数**：同一个 topic 可以被几个消费者同时消费？顺序怎么保证？
3. **幂等性**：同一条消息可能被消费多次，重复处理会不会把数据搞乱？

### 2.4 Message（消息）

消息就是一段字节。从 Broker 的角度看它不关心你是 JSON 还是 Protobuf 还是纯文本。但实际工程中，消息通常长这样：

```json
{
  "id": "msg-2026-04-10-abc123",
  "type": "pdf.uploaded",
  "timestamp": 1712700000,
  "payload": {
    "file_id": 123,
    "user_id": "u-001"
  },
  "metadata": {
    "retry_count": 0,
    "trace_id": "trace-xxx"
  }
}
```

- `id`：消息全局唯一 ID，用来做**幂等去重**（后面详细讲）
- `type`：消息类型，帮助消费者路由到不同的处理函数
- `timestamp`：便于排查"这条消息堆积了多久"
- `payload`：真正的业务数据
- `metadata`：重试次数、链路追踪 ID 等框架用的字段

**建议从第一天就用结构化消息**，不要直接扔一个裸字符串进去。等到线上要回溯问题的时候你会感谢过去的自己。

### 2.5 Queue 和 Topic 的区别

这俩是最让初学者懵的概念，我用一个极简的例子讲透。

**Queue（队列）** = 一根水管。水从一头进去，另一头出来，出来一次就没了。
- 多个消费者共享一根水管：**每条消息只被其中一个消费者拿到**（竞争消费）。
- 典型场景：任务分发。"把这 1000 个 PDF 分给 5 个 worker 处理"，每个 PDF 只被处理一次。

**Topic（主题）** = 一个广播电台。电台发出的信号，每个打开收音机的人都能听到。
- 多个消费者订阅同一个 topic：**每条消息会被每个消费者都收到一份**（广播消费）。
- 典型场景：事件通知。"用户下单了"这条事件，订单服务要收到、积分服务要收到、通知服务也要收到。

### 2.6 Broker（代理/中间人）

Broker 就是**队列服务本身**，比如 RabbitMQ 进程、Kafka 集群、Redis 进程。它负责：

- 接收生产者发来的消息
- 把消息持久化（写硬盘，防止进程挂了丢消息）
- 按规则分发给消费者
- 处理 ACK、重试、死信等机制
- 管理消费进度（offset）

**为什么叫 Broker？** "Broker" 在英语里是"中间人/经纪人"的意思，就像股票经纪人站在买卖双方之间撮合交易。Broker 的存在让生产者和消费者**完全不用认识对方**——生产者只需要知道"我往哪个队列塞东西"，消费者只需要知道"我从哪个队列取东西"。

> **读到这里你应该理解：** Producer / Consumer / Broker / Message / Queue / Topic 这六个词是 MQ 的"普通话"。无论后面看 Redis、Kafka、RabbitMQ、Celery 的文档，都是在这六个词上做变形。如果你在一篇文档里看到陌生词，先问自己"它对应上面哪个角色"。

### 2.7 初学者常见误区

- **误区 1**："Broker 就是 Redis/RabbitMQ。" → Broker 是**角色**，Redis 是一个可以**充当**Broker 的产品。同一个 Redis 实例既可以当 Broker 也可以当缓存。
- **误区 2**："Topic 里的消息只能广播。" → 不对。Kafka 的 Topic 可以通过 **Consumer Group** 同时实现"竞争消费"和"广播"，后面讲 Redis Stream 会详细说。
- **误区 3**："Queue 就是 FIFO，100% 按顺序。" → 单消费者单分区确实是 FIFO。多消费者并发消费时，**不能保证全局顺序**，只能保证"同一个分区/key 内的顺序"。

## 3. MQ 解决的四大类问题

前面用快递驿站类比讲了 MQ 为什么存在。现在系统性地把它能解决的问题分成四类。记住这四类，以后设计任何系统都能条件反射地判断"这个场景是不是该用 MQ"。

### 3.1 异步解耦

**场景**：A 系统要告诉 B、C、D 系统某件事情发生了，但 A 不应该依赖这三个系统的可用性。

- 同步写法：A 直接调 B、C、D 的 HTTP 接口。B 挂了 → A 挂了。
- MQ 写法：A 把事件写进 Topic，B/C/D 自己订阅。B 挂了只影响 B 自己，A 正常。

**Agent 场景举例**：
- "对话结束"事件：需要触发 (1) token 计费 (2) 对话摘要生成 (3) 质检打分 (4) 知识图谱更新。
- 如果 HTTP 接口里同步调用这四个服务，接口会变成四者中最慢的那个；其中任何一个挂了都会影响核心对话功能。
- 把"对话结束"写成事件发到 `conversation.finished` topic，四个下游各自消费，完全独立。

### 3.2 流量削峰

**场景**：上游请求是**脉冲式**的，下游处理能力是**恒定**的。

- 同步：上游 1 秒 1 万请求 → 下游只能处理 100 QPS → 剩下 9900 请求全报错。
- MQ：1 万请求全部先进队列 → 下游按 100 QPS 慢慢拉消息处理 → 用户看到的是"稍等片刻"而不是"报错"。

这种模式在电商秒杀、AI 平台的批量任务里特别常见。**MQ 在这里的价值是"蓄水池"**——把"瞬间的洪水"变成"稳定的水流"。

**代价**：用户从"立刻看到结果"变成"稍后告知结果"。这个产品层面要配合。

### 3.3 广播通知

**场景**：一件事情发生，多个互不相识的下游都要知道。

典型的是审计日志、实时监控、多副本缓存失效。MQ 的 Topic + 多 Consumer Group 模式天然适合。

**Agent 场景举例**：
- "某模型调用失败率超过 5%" 是一个事件。
- 运维告警系统要知道 → 发短信
- 监控大屏要知道 → 更新 UI
- 成本优化系统要知道 → 触发降级策略
- 三个系统都订阅 `model.alert` topic，一条消息触发三个动作。

### 3.4 任务调度

**场景**：我有一堆活儿要干，需要分配给多台机器做，机器数量可能动态伸缩。

这是 MQ 最"工业化"的用法。生产者把任务扔进队列，队列就是**工作池**，消费者数量可以随时扩缩容：

- 白天任务多 → 起 20 个 worker
- 凌晨任务少 → 缩到 2 个 worker
- 队列堆积超过 1000 条 → K8s 自动扩容

**核心优势**：生产者、队列、消费者三者完全解耦，消费者可以任意伸缩。这就是 Celery 的核心用法，后面会详细展开。

### 3.5 四类问题总结表

| 问题 | MQ 角色 | 典型场景 | Agent 场景举例 |
|------|---------|---------|---------------|
| 异步解耦 | "缓冲层" | 耗时操作、第三方依赖 | PDF 上传后异步建索引 |
| 流量削峰 | "蓄水池" | 秒杀、批量任务 | 批量导入 10 万个知识库文档 |
| 广播通知 | "广播站" | 事件订阅、多系统同步 | 模型失败告警 |
| 任务调度 | "工作池" | 分布式任务处理 | LLM 批量推理任务分发 |

### 3.6 什么时候**不该**用 MQ

这是很多人忘记讲的反面。引入 MQ 不是免费的：

1. **简单 CRUD 场景**：数据库单表增删改查，同步返回就好，加 MQ 纯粹增加复杂度。
2. **强一致要求**：需要事务回滚的场景。一旦走 MQ 就进入最终一致性世界，要重新设计回滚流程。
3. **需要立即看到结果**：查账户余额不能异步。
4. **团队经验不足**：MQ 运维门槛不低（监控、堆积告警、死信处理），没专人搞会变成定时炸弹。

**判断法则**：这个动作"晚一秒钟做完用户能不能接受"？能接受 → 可以异步；不能 → 必须同步。

## 4. 消息传递语义

这是 MQ 面试题里出镜率最高的概念之一，也是理解一切 MQ 产品的基础。一定要啃透。

### 4.1 三种语义的定义

消息从生产者到消费者，有三种可能的"投递保证"级别：

- **At-most-once（最多一次）**：消息可能丢，但绝不重复。
- **At-least-once（至少一次）**：消息不会丢，但可能重复。
- **Exactly-once（精确一次）**：每条消息严格只被处理一次。

这三个词你必须能用 1 分钟讲清楚——面试高频题。

### 4.2 用日常生活类比

想象你和朋友打电话约见面时间：

- **At-most-once**：你说"星期六下午 3 点见"，朋友没听清就挂了，你也不问一句"听清了吗"。结果朋友没来。→ 消息丢了但绝对不会重复。
- **At-least-once**：你说完一句问一句"你听清了吗？"如果朋友没回答，你就再说一遍。有可能朋友听清了但他的"嗯"没传回来，于是你又说了一遍。→ 消息绝对不丢，但可能重复。
- **Exactly-once**：你和朋友用一个共享的笔记本，"星期六下午 3 点见"写进去之前要先确认上面有没有这句话。这需要额外的协调机制。→ 最完美但实现代价最高。

### 4.3 各自的实现方式

#### At-most-once

生产者发完就不管了，不等 ACK；消费者拿到消息先 ACK 再处理，处理失败也不重试。

```python
# 伪代码
mq.send(msg)            # 发完不管
msg = mq.receive()
mq.ack(msg)             # 先 ack
process(msg)            # 失败就失败了,不重试
```

**代价**：简单、快，但会丢消息。只适合"日志、监控"这种丢一两条无所谓的场景。

#### At-least-once

生产者等 ACK，没收到就重试；消费者先处理再 ACK，处理失败不 ACK，Broker 就会重新投递。

```python
# 伪代码
while True:
    try:
        mq.send(msg)
        if wait_ack(): break    # 收到 ACK 才跳出
    except TimeoutError: continue

msg = mq.receive()
try:
    process(msg)                # 先处理
    mq.ack(msg)                 # 再 ack
except:
    pass  # 不 ack → broker 稍后重投
```

**代价**：不丢消息，但会重复。**这是 99% 的生产环境默认选择**，因为"重复"可以通过幂等性解决，"丢失"一般无法弥补。

#### Exactly-once

做到精确一次需要**生产者端去重 + 消费者端幂等 + 事务性提交**三重保障。Kafka 从 0.11 版本开始支持 Transactional Producer + Idempotent Producer，业务侧配合可以近似做到 exactly-once。

但请记住一个工程常识：**"真正的"exactly-once 几乎不存在**。即便 Kafka 的事务也只是在 Kafka → Kafka 之间保证 exactly-once；一旦出了 Kafka（比如写数据库），就还是 at-least-once + 业务幂等。

### 4.4 工程取舍

**结论**：绝大多数场景选 **at-least-once + 消费者幂等**。这个组合在实现复杂度和正确性之间达到最佳平衡。

记住这个公式：

```
at-least-once (MQ 提供) + 幂等消费 (业务实现) ≈ exactly-once
```

这是整个 MQ 世界的"圣经"。后面讲 Celery 重试、Redis Stream 消费、Kafka 消费 offset 时都会反复用到。

### 4.5 初学者常见误区

- **误区 1**："我选 exactly-once 就对了，反正是最好的。" → exactly-once 的性能开销和实现复杂度都远高于 at-least-once，而且很多时候是伪装的。真实工程里几乎都是 at-least-once。
- **误区 2**："at-least-once 就等于会乱数据。" → 只有在**消费者不幂等**时才会乱数据。幂等消费是工程师的责任，不是 MQ 的责任。
- **误区 3**："我只要开启 broker 持久化就不会丢消息了。" → 持久化只解决"broker 崩溃时不丢盘上消息"。生产者发的时候没写到盘上就崩了、消费者 ack 之后还没处理完就崩了、这些都会丢。

> **读到这里你应该理解：** 消息语义是 MQ 的"交通规则"。你设计系统的第一件事就是问自己"我要什么语义？代价能不能接受？"如果你的回答是"那个什么我不知道"，就是 at-least-once + 幂等。

## 5. 点对点 vs 发布订阅

MQ 里有两大"物种"：**点对点（Point-to-Point，P2P）** 和 **发布订阅（Publish-Subscribe，Pub/Sub）**。第 2 章提过 Queue 和 Topic 的区别，这里展开讲。

### 5.1 点对点（Queue 模式）

一条消息**只会被一个消费者拿走**。多个消费者同时监听一个队列 → **竞争关系**。

```
            ┌──────────┐
Producer → │  Queue   │ ─→ Consumer A  ← 只有 A 拿到 msg1
            └──────────┘ ─→ Consumer B
                          ─→ Consumer C
```

**典型场景**：任务分发（负载均衡）。工人池的每一条任务都只应该被处理一次。

**代表产品**：RabbitMQ 的 Queue、Redis List、Celery 的任务队列本质。

### 5.2 发布订阅（Topic 模式）

一条消息**会被所有订阅者都收到一份**。消费者之间是**独立关系**。

```
                    → Subscriber A  ← A 拿到 msg1 的副本
Producer → Topic  → Subscriber B  ← B 也拿到 msg1 的副本
                    → Subscriber C  ← C 也拿到 msg1 的副本
```

**典型场景**：事件广播、实时通知。

**代表产品**：Redis Pub/Sub、Kafka（配合 Consumer Group 使用，见下）。

### 5.3 Kafka/Redis Stream 的"鸭子模式"

Kafka 和 Redis Stream 同时支持两种模式，秘诀是 **Consumer Group**（消费者组）。规则如下：

- 同一个 Consumer Group 内的消费者之间 **点对点**：一条消息只被组内的一个消费者拿走。
- 不同 Consumer Group 之间 **发布订阅**：每个 Group 都独立消费所有消息。

用一张图说明：

```
                 ┌── Group-A ──┐
                 │  Consumer1 │ ← 拿到 msg1
Topic msgs  ─→  │  Consumer2 │ ← 拿到 msg2
                 └────────────┘

                 ┌── Group-B ──┐
                 │  Consumer3 │ ← 同时拿到 msg1
                 │  Consumer4 │ ← 同时拿到 msg2
                 └────────────┘
```

**解读**：
- Group-A 和 Group-B 独立消费，都能看到完整的消息流 → 发布订阅
- Group-A 内部 Consumer1 和 Consumer2 分摊消息 → 点对点负载均衡
- 这就是为什么现代 MQ（Kafka、Redis Stream、Pulsar）的 Consumer Group 设计是"一鸭两吃"

**Agent 场景举例**：你有一个 `llm.request` 事件流，三个消费方：
- `billing-group`：负责计费，只需要一个进程处理
- `analytics-group`：负责分析，多个进程并行处理
- `audit-group`：负责审计日志

每个 Group 内的进程点对点分摊任务，Group 之间广播 → 一份消息流同时满足三种需求。

### 5.4 初学者常见误区

- **误区 1**："Topic 就是广播，Queue 就是点对点。" → 不准确。现代 MQ 里 Topic + Consumer Group 同时实现两种模式。要看产品和 Group 配置。
- **误区 2**："消费者组就是一堆消费者的名字。" → Group 名字是持久化的，Broker 会为每个 Group 记录"这个 Group 消费到哪了"（offset）。换个 Group 名相当于从头开始消费。这个在 Kafka 里是面试常考点。

## 6. Redis 当消息队列的三种方式

Redis 不是专业的 MQ，但它被当作 MQ 用得太普遍了，因为：
- 轻量、几乎所有后端项目都已经装了 Redis
- 足够快（10 万 QPS）
- 中小规模场景够用

Redis 提供了三种"当队列用"的方法，从原始到专业依次是：**List / Pub-Sub / Stream**。

### 6.1 方式一：List + LPUSH/BRPOP

最原始的方式。Redis 的 List 就是双向链表，直接拿来当队列。

```bash
# 生产者（左边塞）
LPUSH tasks "{'file_id':123}"

# 消费者（右边拿，阻塞等待最多 30 秒）
BRPOP tasks 30
```

- `LPUSH` = Left Push，从左边塞
- `BRPOP` = Blocking Right Pop，阻塞式从右边取
- 组合起来就是 FIFO 队列（先进先出）

**优点**：
- 极简，几行代码就是一个队列
- 速度快

**缺点**（非常重要）：
- **没有 ACK 机制**：消费者 BRPOP 之后消息就被删了，处理到一半崩溃 → 消息丢了。
- **没有 Consumer Group**：多消费者只能竞争消费，没法广播。
- **没有历史回放**：消息被消费就没了，不能重新读。

**适用**：简单的后台任务、丢一条没事的场景。**不适合**：关键业务。

### 6.2 方式二：Pub/Sub

Redis 原生支持发布订阅模式。

```bash
# 订阅者（先开着等待）
SUBSCRIBE notifications

# 发布者
PUBLISH notifications "new task arrived"
```

- 订阅了频道的客户端会收到消息
- 没订阅的客户端发完就扔了

**优点**：
- 天然广播语义
- API 简单

**致命缺点**：
- **消息完全不持久化**。订阅者离线期间的消息全部丢失。
- **没有 ACK、没有重试、没有历史**。
- **Redis 重启 → 消息归零**。

**适用**：实时通知、聊天室（消息丢了也无所谓）。**不适合**：任何业务逻辑。

在 Agent 场景里，Pub/Sub 可以用来实现"服务端推 WebSocket 消息给前端"这种瞬时场景，但绝对不能用来做核心任务流。

### 6.3 方式三：Stream（Redis 5.0+，强烈推荐）

Stream 是 Redis 5.0 引入的专门为消息队列设计的数据结构。它基本上是把 Kafka 的核心思想搬到了 Redis 里：

- **持久化**（写 AOF）
- **Consumer Group**
- **ACK 机制**
- **消息 ID 严格递增**，可以按 ID 范围回放历史

```bash
# 生产者：XADD
XADD pdf_tasks * file_id 123 user_id u-001
# * 表示让 redis 自动生成 ID (毫秒时间戳 + 序列号)

# 创建消费者组
XGROUP CREATE pdf_tasks workers 0

# 消费者：XREADGROUP 带 ACK
XREADGROUP GROUP workers worker-1 COUNT 1 STREAMS pdf_tasks >

# 处理完后 ACK
XACK pdf_tasks workers 1712700000000-0
```

**优点**：
- 消息持久化
- Consumer Group 支持点对点+广播
- ACK 机制，崩溃后可以回到 PEL（Pending Entries List）找回未确认消息
- 支持历史回放

**缺点**：
- API 比较复杂，刚上手有点懵
- 仍然不如专业 MQ（Kafka/RabbitMQ）在极端高并发下稳定

**适用**：中小规模的任务队列、事件流。Agent 平台的大部分异步任务场景都够用。

### 6.4 三种方式对比

| 维度 | List | Pub/Sub | Stream |
|------|------|---------|--------|
| 持久化 | 有（内存+AOF） | **无** | 有（内存+AOF） |
| 消费模式 | 竞争 | 广播 | 两种都支持 |
| ACK 机制 | 无 | 无 | **有** |
| Consumer Group | 无 | 无 | **有** |
| 历史回放 | 无 | 无 | **有** |
| 消息 ID | 无 | 无 | **自动生成，严格递增** |
| 实现复杂度 | 极简 | 极简 | 中等 |
| 推荐度 | ★☆ | ☆（只能做瞬时通知） | **★★★★** |

**结论**：Redis 5 之后做消息队列**直接用 Stream**，除非你的场景真的只是"后台日志式的丢几条没事的任务"才考虑 List。

> **读到这里你应该理解：** Redis 的三种 MQ 方式分别对应"能跑 / 不能用 / 推荐用"。下一节我们把 Stream 拆细讲，因为它既是 Redis 的现代 MQ 推荐方案，也是理解 Kafka 的最佳入门模型。

## 7. Redis Stream 深度

这一节要把 Redis Stream 讲透。理解了 Stream，你就理解了 Kafka 的核心理念 80%。两者的概念模型几乎一一对应。

### 7.1 Stream 的本质：一个永久增长的日志

把 Stream 想成一个**永远只能在末尾追加、每条记录有全局唯一 ID 的日志文件**。

```
pdf_tasks Stream:
  1712700001000-0 → {file_id: 123, user: u-001}
  1712700002000-0 → {file_id: 124, user: u-002}
  1712700003000-0 → {file_id: 125, user: u-001}
  1712700003000-1 → {file_id: 126, user: u-003}   ← 同一毫秒第二条
```

**消息 ID 格式**：`<毫秒时间戳>-<序列号>`，保证严格递增。这个 ID 就是 Kafka 的 offset，是你可以反向"寻位"的坐标。

### 7.2 XADD：追加一条消息

```bash
XADD pdf_tasks * file_id 123 user u-001
# 返回: "1712700001000-0"
```

参数解释：
- `pdf_tasks`：Stream 名字
- `*`：让 Redis 自动生成 ID；也可以手动指定 `<ms>-<seq>`，但一般别手动
- 后面是键值对形式的 body，可以传任意多个字段

**关键点**：一个 Stream 的消息 body 不是 JSON，而是**字段键值对**的 map 结构。你可以把 JSON 序列化后放到某个字段里，也可以直接用多个字段。

### 7.3 XREAD：单消费者读消息

```bash
XREAD COUNT 10 BLOCK 5000 STREAMS pdf_tasks 0
```

参数解释：
- `COUNT 10`：最多读 10 条
- `BLOCK 5000`：没消息时阻塞最多 5 秒
- `STREAMS pdf_tasks 0`：从 `pdf_tasks` 的 ID 0 开始读（`0` 意思是"从头读"）

**注意**：`XREAD` 是"单消费者"模式，**没有 ACK，没有进度保持**。你需要自己记住"上一次读到哪个 ID 了"，下次从那个 ID + 1 开始读。非常原始，基本不建议直接用。

生产环境真正用的是 `XREADGROUP`。

### 7.4 Consumer Group：点对点 + 进度追踪

Consumer Group 是 Stream 的核心特性。它解决两个问题：
1. 多消费者**协作**处理同一个 Stream（一条消息只被一个消费者拿走）
2. 消费进度由 Broker 管理，消费者崩溃重启后能**接着上次**继续

#### 创建 Group

```bash
XGROUP CREATE pdf_tasks workers 0
```

- `workers`：Group 名字
- `0`：从最早的消息开始消费；也可以用 `$` 表示"只消费将来的新消息"

#### XREADGROUP：组内消费

```bash
XREADGROUP GROUP workers worker-1 COUNT 1 BLOCK 5000 STREAMS pdf_tasks >
```

参数解释：
- `GROUP workers worker-1`：所属 Group 叫 `workers`，当前消费者 ID 叫 `worker-1`
- `>`：特殊符号，意思是"给我这个 Group 还没分给任何消费者的新消息"

多个进程都用同一个 Group 名字、不同的 consumer ID 去 `XREADGROUP`，Redis 会自动给它们分配不同的消息。**这就是点对点负载均衡**。

#### XACK：确认消费完成

```bash
XACK pdf_tasks workers 1712700001000-0
```

消息被 `XREADGROUP` 读到之后，会进入**待处理列表 PEL（Pending Entries List）**。只有在你 `XACK` 之后，Redis 才认为"这条消息处理完了"。

**如果消费者崩了怎么办？**
- 没被 XACK 的消息还在 PEL 里
- 重启后 `XREADGROUP GROUP workers worker-1 ... 0`（注意这里用 `0` 而不是 `>`）
- 这会返回"这个消费者自己的 PEL 里的未确认消息"
- 或者用 `XCLAIM` 把消息转给其他消费者（比如检测到 worker-1 挂了，让 worker-2 接管）

### 7.5 PEL 机制深挖

PEL 是 Stream 区别于 Redis List 最重要的地方。我再强调一遍流程：

```
消息进入 Stream → XREADGROUP 读出 → 进入 PEL (标记 "已分发给 worker-1 但未 ACK")
  → 消费者处理完 → XACK → 从 PEL 删除
  → 消费者崩溃 → 消息还在 PEL → 其他消费者可以 XCLAIM 接管
```

PEL 里的每条消息都记录：**哪个消费者拿的、拿了多久了、被投递过几次**。这些信息让"崩溃恢复 + 死信检测"变得可行。

#### XPENDING：查看谁在拖后腿

```bash
XPENDING pdf_tasks workers
# 返回: 总 pending 数、最早 ID、最晚 ID、每个消费者的 pending 数
```

生产环境可以监控这个指标：如果某个消费者 pending 数一直涨，说明它卡住了。

#### XCLAIM：转移消息所有权

```bash
# 把 pdf_tasks 里超过 60 秒没 ACK 的消息转给 worker-2
XCLAIM pdf_tasks workers worker-2 60000 1712700001000-0
```

这是做"死信队列"或者"故障接管"的关键命令。

### 7.6 Stream 和 Kafka 的类比

写给学完这一节想去看 Kafka 的同学：

| Redis Stream | Kafka | 含义 |
|--------------|-------|------|
| Stream | Topic | 消息的逻辑通道 |
| 消息 ID | Offset | 唯一递增坐标 |
| XADD | Producer.send | 写入消息 |
| XREADGROUP | Consumer.poll | 读取消息 |
| Consumer Group | Consumer Group | 进度管理单元 |
| PEL | Offset commits + rebalance | 未确认消息跟踪 |
| XACK | commitSync | 确认消息已处理 |
| XCLAIM | Rebalance 分区转移 | 故障接管 |

**看到这个表的含义**：如果你搞懂了 Redis Stream，去看 Kafka 文档只是**换了些名词而已**，核心模型完全一样。这也是为什么不少团队面试喜欢先聊 Redis Stream 再聊 Kafka——能把 Stream 讲透的人，迁移到 Kafka 术语体系也相对顺利。

### 7.7 Stream 的限制

作为一个嵌入在 Redis 里的"迷你 Kafka"，Stream 也有它的边界：

1. **单机内存**：所有数据在 Redis 进程的内存里，超过内存限制要配 `MAXLEN` 裁剪。Kafka 是分区磁盘存储，理论上无限。
2. **分区有限**：Stream 本身没有"分区"的概念，水平扩展比 Kafka 难。可以通过 `stream:A / stream:B` 这种 key 分片自己做，但是复杂。
3. **吞吐上限**：单机 Redis 的 Stream 能做到几十万 QPS，但 Kafka 集群可以做到百万 QPS 级别。
4. **生态**：Kafka 有 Connect、Streams、KSQL 这些庞大生态，Stream 就是 Stream。

**结论**：百万级以下的日均消息量用 Stream 完全够用。上到千万/亿级再考虑 Kafka。

### 7.8 初学者常见误区

- **误区 1**："XREAD 和 XREADGROUP 差不多。" → 差别巨大。XREAD 没有进度管理，纯粹是"当前时间点的快照"。生产环境只用 XREADGROUP。
- **误区 2**："ACK 是给我自己看的。" → 错。ACK 是给 Redis 看的，决定消息从 PEL 里删不删。忘了 ACK 消息会一直堆积在 PEL。
- **误区 3**："Group 名字可以随便换。" → 换 Group 名等于新建一个 Group，从头重新消费。生产环境这种操作要慎之又慎。
- **误区 4**："Stream 消息自动清理。" → 不会。Stream 只会无限增长，需要你用 `XADD ... MAXLEN ~ 100000` 或者定时 `XTRIM` 主动清理。

## 8. 主流 MQ 全家福对比

市面上的 MQ 产品很多，面试经常问"你知道 RabbitMQ 和 Kafka 的区别吗？"这一节我们把六个主流产品过一遍，建立一个"看到场景能选对产品"的直觉。

### 8.1 六大产品一句话定位

| 产品 | 一句话定位 | 核心协议/模型 |
|------|-----------|--------------|
| **RabbitMQ** | 老牌、灵活路由、功能全 | AMQP |
| **Kafka** | 日志/大数据场景之王 | 分区 Log |
| **RocketMQ** | 阿里出品、电商订单首选 | 类 Kafka + 事务消息 |
| **NATS** | 极简、低延迟、云原生 | 自定义轻协议 |
| **Redis Stream** | 已有 Redis 就够了 | Redis 内置 |
| **Pulsar** | 云原生下一代（可选了解） | 存储计算分离 |

### 8.2 RabbitMQ

**定位**：AMQP（Advanced Message Queuing Protocol）协议的标准实现。**灵活路由是它的王牌**。

**核心概念**：Exchange（交换机） + Queue（队列） + Binding（绑定）。生产者把消息发给 Exchange，Exchange 按规则路由到 Queue，消费者从 Queue 拉取。

**四种 Exchange 类型**：
- `direct`：按 routing key 精确匹配
- `topic`：按通配符匹配（`user.*.login`）
- `fanout`：广播到所有绑定的队列
- `headers`：按消息头部字段匹配

**优点**：
- 路由灵活，复杂业务场景很舒服
- 管理界面好用
- 支持优先级队列、延迟队列、死信队列这些高级特性

**缺点**：
- 吞吐不如 Kafka（单机 1-5 万 QPS 级别）
- Erlang 写的，对国内团队运维不太友好
- 不适合日志/流处理场景

**适用**：传统企业应用、复杂路由、**Celery 默认 Broker 之一**。

### 8.3 Kafka

**定位**：**分布式提交日志**。不是传统意义的 MQ，更像一个"可以被多人同时读写的日志文件"。

**核心概念**：Topic → 分成多个 Partition → 每个 Partition 是一个顺序追加的日志文件。Consumer Group 里每个 Consumer 分到几个 Partition 独占读。

**关键特性**：
- **分区**是水平扩展的基础单元，消息按 key 哈希到分区
- **Offset** 是每条消息在分区里的位置，Consumer 记录自己消费到哪
- 消息持久化在硬盘，吞吐依然高（顺序 IO 快）
- 百万 QPS，万级延迟（毫秒）

**优点**：
- 吞吐量之王
- 消息持久化 + 可回放
- 大数据生态最完善（Kafka Connect、Kafka Streams、Flink 集成）

**缺点**：
- 运维门槛高（ZooKeeper/KRaft 集群管理）
- 不支持复杂路由（没有 AMQP 那种灵活 Exchange）
- 延迟不如 RabbitMQ、NATS 低

**适用**：日志聚合、用户行为分析、事件溯源、大数据管道、Agent 平台的模型调用日志流。

### 8.4 RocketMQ

**定位**：阿里开源的 Java MQ，类似 Kafka 但更贴近业务场景。**事务消息** 是它的招牌。

**核心特性**：
- 顺序消息（按 key 有序）
- 事务消息（两阶段提交，保证"本地事务 + 发消息"的原子性）
- 延迟消息（原生支持定时投递）

**适用**：电商、金融、订单这类对"消息一致性"要求高的场景。国内互联网大厂用得很多。

### 8.5 NATS

**定位**：**极简、低延迟**。单个二进制文件，启动就能用。云原生时代的选择。

**特性**：
- 延迟极低（亚毫秒级）
- 轻协议，易部署
- JetStream（NATS 的持久化扩展）提供类似 Kafka 的流语义

**适用**：微服务之间的事件总线、物联网、实时通信。

### 8.6 Redis Stream

已经讲过。**已经有 Redis** 的项目用它做 MQ 最省事。上限是"单机 Redis 能承受的消息量"。

### 8.7 Pulsar

**定位**：Kafka 的下一代挑战者。**存储和计算分离**（Broker 只做路由，数据存在 BookKeeper）。多租户、地理复制。

**特点**：比 Kafka 更现代的架构，但成熟度和生态稍弱。有兴趣的同学可以了解，面试不会强求。

### 8.8 怎么选？

一个快速决策树：

```
需求是什么?
├── 我只有 Redis,日消息量 < 百万
│   → Redis Stream
├── 企业应用,路由复杂,要优先级/延迟队列
│   → RabbitMQ
├── 日志/事件流/大数据,高吞吐
│   → Kafka
├── 电商/金融,需要事务消息+顺序
│   → RocketMQ
├── 微服务事件总线,极低延迟
│   → NATS
└── 做 AI 后端任务队列,一般选择
    → Celery + Redis/RabbitMQ (99% 场景够用)
```

**面试话术**：如果面试官问"你怎么选 MQ？"别给一个教条答案，而是**反问需求**：日消息量多少、延迟要求、是否需要回放、团队运维能力、现有技术栈。这才是架构师的思维。

### 8.9 初学者常见误区

- **误区 1**："Kafka 一定比 RabbitMQ 强。" → 错。Kafka 在日志/流场景强，RabbitMQ 在业务路由场景强。工具没有好坏，只有合适不合适。
- **误区 2**："用了 MQ 就能高可用。" → MQ 本身也会挂。RabbitMQ 要镜像队列，Kafka 要副本因子，Redis 要主从/哨兵/Cluster。没做这些配置的 MQ 就是单点。

## 9. 为什么还需要 Celery

前面讲完了 Redis Stream 和 Kafka 这些 MQ 产品，你可能会问：

> "既然 Redis Stream 这么强，我直接用它做任务队列不就行了？为什么还要 Celery？"

这个问题问得非常好。答案是：**Redis Stream 是 Broker，Celery 是框架**。两者不是同一个层次。

### 9.1 "裸 MQ"有什么不够用的地方

假设你决定不用 Celery，直接用 Redis Stream 做任务队列。你需要自己实现以下一堆东西：

1. **任务序列化**：Python 函数调用怎么序列化成字节流？参数怎么 dumps/loads？
2. **任务路由**：多种类型的任务怎么分发到不同的处理函数？
3. **Worker 进程管理**：起几个 worker？每个 worker 并发多少协程/线程？怎么优雅关停？
4. **重试机制**：失败了怎么重试？怎么做指数退避？最多重试几次？
5. **结果存储**：任务处理完的返回值存哪？客户端怎么查？
6. **超时处理**：任务执行超过 5 分钟自动 kill？
7. **任务取消**：怎么取消一个在排队的任务？
8. **定时任务**：怎么做 cron 风格的定时触发？
9. **任务链式编排**：A 完成后自动触发 B、C、D 怎么做？
10. **任务状态追踪**：PENDING/STARTED/SUCCESS/FAILURE 怎么记录？
11. **监控面板**：怎么看队列堆积、worker 健康度？

这 11 件事情，每一件你都要自己写代码。写到后面你会发现：**你在重新发明一个 Celery**。

### 9.2 Celery 的价值

Celery 是 Python 生态里最成熟的**分布式任务队列框架**。它把上面 11 件事情全都给你封装好了：

- 你只管写业务函数，加一个 `@app.task` 装饰器
- 调用用 `.delay(args)` 就能把函数调用变成"投递到队列"
- Worker 进程启动命令 `celery worker`，自动帮你做进程池、协程调度、信号处理
- 重试、超时、定时、链式，全部一行注解或一行代码搞定

**一句话总结**：Celery = MQ（底层 Broker）+ 一套围绕"分布式任务"的完整工程封装。

### 9.3 Celery 和 MQ 的关系

```
    你的代码 (FastAPI 接口)
         ↓ (task.delay())
    Celery 客户端
         ↓ (序列化为消息)
    Broker (RabbitMQ / Redis / Redis Stream)
         ↓ (消息投递)
    Celery Worker 进程
         ↓ (反序列化 + 执行业务函数)
    业务函数执行
         ↓ (结果)
    Result Backend (Redis / DB)
```

**Broker 是 Celery 的"左脚"，Result Backend 是它的"右脚"**。两者都可以选：
- Broker: RabbitMQ（官方推荐）/ Redis / Redis Sentinel / SQS
- Result Backend: Redis / PostgreSQL / MongoDB / RPC

通常的组合：
- 轻量项目：Broker=Redis, Backend=Redis
- 企业级：Broker=RabbitMQ, Backend=Redis/DB

### 9.4 初学者常见误区

- **误区 1**："我用 Redis Stream 就行，不需要 Celery。" → 小项目可以，但一旦需要重试/定时/链式，自己写的成本会爆炸。
- **误区 2**："Celery 就是 Redis。" → Celery 的 Broker 可以是 RabbitMQ、Redis、SQS 等多种。Redis 只是最常见的一种。
- **误区 3**："Celery Worker 就是 Python 的多线程。" → Celery Worker 支持多种并发池：prefork（默认，多进程）、gevent（协程）、eventlet、solo、threads。默认 prefork 意味着你的任务会在**独立的子进程**里执行，不共享内存。

## 10. Celery 的架构

### 10.1 四大组件

Celery 运行起来有四个核心组件：

```
  ┌────────────┐      ┌───────────┐      ┌─────────────┐
  │   Client   │ ───→ │  Broker   │ ───→ │   Worker    │
  │(FastAPI等) │      │(Redis等)  │      │ (执行任务)  │
  └────────────┘      └───────────┘      └─────────────┘
         │                                       │
         │ 查询结果                              │ 写入结果
         ↓                                       ↓
  ┌──────────────────────────────────────────────────┐
  │          Result Backend (Redis / DB)             │
  └──────────────────────────────────────────────────┘
```

#### Client（客户端）

Client 就是你的应用代码。它只做两件事：
1. **投递任务**：`my_task.delay(arg1, arg2)`
2. **查询结果**（可选）：`result = my_task.AsyncResult(task_id); result.get()`

Client 本身**不执行**任务，它只是把任务序列化成消息扔给 Broker。

#### Broker（消息中间件）

Broker 是任务消息的"中转站"。它接收 Client 投递的任务，保存起来，等 Worker 来拉。

**常见选择**：
- **Redis**：最常用，简单快。缺点是重启可能丢消息（如果没开 AOF）。
- **RabbitMQ**：Celery 官方推荐，支持更复杂的路由和优先级。
- **Amazon SQS**：云上首选。

#### Worker（工作进程）

Worker 是**长驻进程**，启动命令是 `celery -A myapp worker`。它在一个循环里：

```python
while True:
    msg = broker.receive()        # 从 Broker 拉取一条任务
    task = deserialize(msg)       # 反序列化成 Python 调用
    try:
        result = task.run()       # 执行业务函数
        backend.store(task.id, result)  # 存结果
        broker.ack(msg)           # 确认
    except Exception as e:
        handle_retry_or_fail(e)   # 重试或进死信
```

一个 Worker 进程内部可以有多个"执行单元"（子进程、协程、线程），由 `--concurrency` 参数控制。

#### Result Backend（结果存储）

很多任务不需要返回值（比如"发一封邮件"），这时可以配置 `task_ignore_result=True` 省掉这一层。但如果你需要知道"任务做完了没、结果是什么"，就需要 Result Backend。

**常见选择**：
- **Redis**：默认、快、适合短期结果
- **PostgreSQL/MySQL**：持久化、方便审计
- **RPC**：直接回传给 Client

### 10.2 一个任务的完整生命周期

我用一个伪场景把整个流程串一遍。

**场景**：FastAPI 接口接到 `/knowledge/upload` 请求，投递一个 PDF 索引任务。

```
时间线:
T0  用户调 POST /knowledge/upload
T0+ FastAPI 调 process_pdf.delay(file_path, user_id)
     ├─ Celery Client 把 ("process_pdf", (file_path, user_id), {}) 序列化成 JSON
     ├─ 生成 task_id (UUID)
     └─ LPUSH (或 XADD) 到 Redis 的 celery 队列
T0+ FastAPI 立刻返回 {"task_id": "xxx", "status": "queued"}

(几毫秒后)
T1  Celery Worker 从队列拉到消息
     ├─ 反序列化找到 process_pdf 函数对象
     ├─ 写入 Result Backend: state=STARTED
     └─ 开始执行 process_pdf(file_path, user_id)

T1+10s 执行完毕
     ├─ 写入 Result Backend: state=SUCCESS, result={"indexed": 100}
     └─ ACK 给 Broker

(用户可以随时查)
T2  用户调 GET /tasks/{task_id}
     ├─ FastAPI 调 AsyncResult(task_id).state
     └─ 从 Result Backend 查到 SUCCESS,返回给用户
```

仔细看这个流程，你会发现**从"用户点按钮"到"FastAPI 返回"只有 T0 到 T0+ 这段毫秒级的时间**，所有耗时操作都在 Worker 里异步完成。这就是第 1 节那个 PDF 上传案例的正确实现方式。

### 10.3 并发模型（这一点初学者必须知道）

`celery -A myapp worker -c 4` 里的 `-c 4` 是什么意思？取决于你用的"池"类型。

| 池类型 | 含义 | 适用 |
|--------|------|------|
| **prefork**（默认） | 4 个子进程，每个跑一个任务 | CPU 密集型任务 |
| **gevent** | 1 个进程 + 4 个协程 | I/O 密集型（大部分 Agent 场景） |
| **eventlet** | 同 gevent，另一种协程库 | 同上 |
| **threads** | 4 个线程 | 轻量 I/O |
| **solo** | 单进程单线程（调试用） | 本地开发 |

**Agent 场景选哪个？** Agent 后端的任务 90% 是 I/O 密集（调 LLM 接口、读写数据库、调工具），强烈推荐 **gevent**。

```bash
pip install gevent
celery -A myapp worker -P gevent -c 100  # 100 个协程
```

这样一个 worker 进程可以同时处理 100 个 LLM 调用，远比 `-P prefork -c 4` 效率高。

### 10.4 初学者常见误区

- **误区 1**："Celery worker 启动会自动加载我的任务代码。" → 必须用 `-A myapp` 告诉它去哪找 `app`，否则它根本不知道你定义了什么任务。
- **误区 2**："Worker 挂了任务就丢了。" → 取决于 `acks_late` 配置。默认 `acks_late=False`，任务一开始执行就 ACK 了，worker 崩溃确实会丢。生产环境推荐 `acks_late=True`，执行完才 ACK。
- **误区 3**："并发 100 = 同时处理 100 个任务。" → 对 gevent 是对的，对 prefork 意味着 100 个子进程占用 100 份内存。别混了。

## 11. Celery 任务定义

### 11.1 最小例子

创建一个文件 `tasks.py`：

```python
from celery import Celery

# 创建 Celery 应用实例
# 第一个参数 'myapp' 是应用名,用来标识任务归属
# broker 指定用 Redis 作消息队列
# backend 指定用 Redis 存结果
app = Celery(
    'myapp',
    broker='redis://localhost:6379/0',
    backend='redis://localhost:6379/1',  # 用不同 db 号避免和 broker 混用
)

@app.task
def add(x, y):
    """一个最简单的任务:把两个数相加。"""
    return x + y
```

然后分两个终端：

```bash
# 终端 1：启动 Worker
celery -A tasks worker --loglevel=info

# 终端 2：Python 解释器里投递任务
>>> from tasks import add
>>> result = add.delay(3, 4)   # 注意是 .delay(),不是直接调用
>>> result.get()
7
```

**核心区别**：
- `add(3, 4)` → **同步执行**，立刻返回 7（相当于没用 Celery）
- `add.delay(3, 4)` → **异步投递**，返回一个 `AsyncResult` 对象，`.get()` 会阻塞等结果

### 11.2 .delay() 和 .apply_async() 的区别

`.delay()` 是 `.apply_async()` 的快捷方式。区别在于**能传的参数**：

```python
# .delay() 只能传业务参数
add.delay(3, 4)

# .apply_async() 可以传所有控制参数
add.apply_async(
    args=(3, 4),                  # 位置参数
    kwargs={},                    # 关键字参数
    queue='high_priority',        # 路由到哪个队列
    countdown=60,                 # 60 秒后执行
    expires=300,                  # 5 分钟内必须消费,否则丢弃
    retry=True,                   # 启用自动重试
    task_id='custom-id-123',      # 自定义 task ID
)
```

**记忆法则**：`.delay()` 是"我不关心细节"，`.apply_async()` 是"我要精确控制"。

### 11.3 任务的返回值和 AsyncResult

`task.delay()` 返回的 `AsyncResult` 对象有几个关键方法：

```python
result = add.delay(3, 4)

result.id           # task ID (UUID)
result.state        # 当前状态: PENDING/STARTED/SUCCESS/FAILURE/RETRY
result.ready()      # 是否执行完了 (bool)
result.successful() # 是否成功了 (bool)
result.failed()     # 是否失败了 (bool)
result.get(timeout=10)  # 阻塞等结果,最多等 10 秒
result.get(propagate=False)  # 如果任务报错,不在这里抛出,返回 Exception 对象
```

**千万别在 FastAPI 接口里 `.get()`！** 这会把异步任务变回同步的，用户等的时间和不用 Celery 一样。正确做法是：

```python
# FastAPI 接口: 只投递,立刻返回 task_id
@app.post("/tasks")
def create_task():
    result = add.delay(3, 4)
    return {"task_id": result.id}

# 另一个查询接口: 前端轮询
@app.get("/tasks/{task_id}")
def get_task(task_id: str):
    result = AsyncResult(task_id)
    return {"state": result.state, "result": result.result if result.ready() else None}
```

### 11.4 任务的参数序列化

Celery 把任务的参数序列化成 JSON（默认），放进 Broker 消息里。这意味着：

**可以传的**：
- 基本类型：int、float、str、bool、None
- 容器类型：list、dict、tuple（会被转成 list）
- 可 JSON 序列化的对象

**不能传的**：
- 数据库 ORM 对象（Session、Model 实例）
- 文件句柄、网络连接
- 任意 Python 对象（除非用 pickle 序列化器，但有安全风险）

**Agent 场景的最佳实践**：任务参数永远只传 **ID 和原始值**，在任务内部再去数据库查。比如：

```python
# 错误：传一整个对象
@app.task
def process_user(user):  # ← user 是 SQLAlchemy 模型,不能序列化!
    do_something(user)

# 正确：只传 ID,任务内查库
@app.task
def process_user(user_id: int):
    user = db.query(User).get(user_id)
    do_something(user)
```

### 11.5 任务的签名（Signature）

Celery 支持一个叫 **Signature** 的概念，可以把"任务 + 参数"打包成一个对象传递。这是后面 chain/group/chord 编排的基础。

```python
from celery import signature

# 创建一个 signature
sig = add.s(3, 4)  # .s() 是 .signature() 的快捷方式

# signature 可以单独执行
sig.delay()

# 也可以作为参数传给别的任务,或者组合起来
```

现在只需要知道 `add.s(3, 4)` 的意思是"把 add(3, 4) 这个调用打包起来，但还不执行"。第 14 节讲编排时会大量用到。

## 12. Celery 任务状态机

一个任务从创建到完成经历几个状态。搞清楚状态流转是排查生产问题的前提。

### 12.1 六个核心状态

| 状态 | 含义 | 转入条件 |
|------|------|---------|
| **PENDING** | 等待中（未知状态） | 默认状态 / 任务不存在 |
| **RECEIVED** | Worker 已收到但未开始 | Worker 从队列拉到消息 |
| **STARTED** | 正在执行 | Worker 开始调用业务函数（需开启 `task_track_started=True`） |
| **SUCCESS** | 成功完成 | 业务函数正常返回 |
| **FAILURE** | 执行失败 | 业务函数抛未处理异常 |
| **RETRY** | 正在重试 | 任务抛出异常且配置了重试 |

### 12.2 PENDING 的坑

这是 Celery 最容易踩坑的地方。看这个命题：

> `AsyncResult('fake-id-不存在的').state` 返回什么？

答案是 **PENDING**。不是 `NOT_FOUND`，而是 PENDING。

**原因**：Celery 把 "任务不存在" 和 "任务排队中" 合并为同一个状态，都叫 PENDING。这是为了简化。但对初学者来说非常迷惑——你写错 task_id 了以为是在排队。

**排查技巧**：如果一个任务一直 PENDING，先检查三件事：
1. 任务 ID 对不对
2. Worker 进程是不是起来了
3. Worker 是不是订阅了对应的队列

### 12.3 状态流转图

```
       .delay()
         ↓
      PENDING ──────────────┐
         ↓  (worker 拉到)    │
      RECEIVED               │
         ↓                   │
      STARTED                │
         ↓                   │
    ┌────┴─────┐             │
    ↓          ↓             │
  SUCCESS   FAILURE          │
              ↓              │
          (retry?)           │
              ↓              │
            RETRY ───────────┘
              ↓
         (重回 PENDING, 等待再次被拉取)
```

### 12.4 自定义状态

除了上面六个，你可以自定义状态来反馈进度：

```python
@app.task(bind=True)
def long_task(self, total):
    # bind=True 让 self 指向 task 实例
    for i in range(total):
        do_work(i)
        # 更新状态: 自定义叫 PROGRESS
        self.update_state(
            state='PROGRESS',
            meta={'current': i, 'total': total}
        )
    return 'done'
```

客户端可以这样查进度：

```python
result = AsyncResult(task_id)
if result.state == 'PROGRESS':
    print(f"{result.info['current']} / {result.info['total']}")
```

**Agent 场景用途**：
- RAG 索引构建显示"已处理 30/100 页"
- LLM 批量推理显示"已完成 5/10 样本"
- 长时运行任务给用户进度条

### 12.5 错误时能拿到什么

```python
result = long_task.delay(total=10)
# ... 等一会儿 ...
if result.state == 'FAILURE':
    print(result.result)       # 异常对象
    print(result.traceback)    # 完整的 traceback 字符串
```

**重要**：`result.get(propagate=False)` 不会抛异常，`result.get()` 默认会把 worker 那边的异常在 client 这边重新抛出来。调试时用前者更友好。

## 13. Celery Beat 定时任务

任务不一定都是"用户点了按钮才触发"。有的任务是"每小时跑一次"、"每天凌晨 3 点跑一次"、"每 5 分钟检查一次健康"。这种场景用 **Celery Beat**。

### 13.1 Beat 是什么

Celery Beat 是一个**独立进程**，它的职责很单纯：**按配置的时间表，往 Broker 里投递任务**。

```
Celery Beat (定时调度) ───投递──→ Broker ───→ Celery Worker (执行)
```

注意 Beat **不执行**任务，它只是个"闹钟"。真正执行还是靠 Worker。所以生产环境你会起两个进程：

```bash
celery -A myapp beat          # 定时器
celery -A myapp worker        # 执行器
```

### 13.2 配置定时任务

```python
from celery import Celery
from celery.schedules import crontab

app = Celery('myapp', broker='redis://localhost:6379/0')

app.conf.beat_schedule = {
    # 每 30 秒跑一次
    'ping-every-30-sec': {
        'task': 'tasks.ping',
        'schedule': 30.0,                  # 秒数
    },
    # 每天凌晨 3 点跑
    'backup-daily-3am': {
        'task': 'tasks.backup_db',
        'schedule': crontab(hour=3, minute=0),
    },
    # 工作日每小时整点
    'hourly-report': {
        'task': 'tasks.send_report',
        'schedule': crontab(minute=0, hour='*', day_of_week='mon-fri'),
    },
    # 带参数的定时任务
    'clean-old-logs': {
        'task': 'tasks.clean_logs',
        'schedule': crontab(hour=2, minute=30),
        'args': (7,),   # 清理 7 天前的日志
    },
}
```

### 13.3 crontab 表达式

Celery 的 `crontab` 几乎和 Linux 的 crontab 一样，参数：

- `minute`：分钟 (0-59)
- `hour`：小时 (0-23)
- `day_of_week`：星期几 (0-6 或 `mon/tue/...`)
- `day_of_month`：几号 (1-31)
- `month_of_year`：几月 (1-12)

常用例子：

```python
crontab()                                  # 每分钟
crontab(minute=0)                          # 每小时整点
crontab(minute=0, hour=0)                  # 每天午夜
crontab(minute='*/15')                     # 每 15 分钟
crontab(minute=0, hour='*/2')              # 每 2 小时
crontab(0, 0, day_of_month='1')            # 每月 1 号 0 点
crontab(0, 0, day_of_week='sun')           # 每周日 0 点
```

### 13.4 踩坑提醒

- **Beat 必须单独启动**：别以为起了 Worker 就够了。Beat 是独立进程。
- **Beat 不要起多个实例**：同一个配置起两个 beat 会触发两次任务，生产环境要么单点部署要么用 `redbeat`（分布式 Beat）。
- **Beat 的持久化**：默认 Beat 的"上次触发时间"存在本地 `celerybeat-schedule` 文件里。容器化环境要挂 volume 或者用 `redbeat` 把状态存到 Redis。

### 13.5 Agent 场景用例

- 每 10 分钟检查一次所有 Agent 的健康状态
- 每天凌晨 2 点重建一次向量索引
- 每小时同步一次第三方知识库
- 每 5 秒扫描一次队列堆积状况并报警
- 每月 1 号生成上月使用量账单

## 14. Celery 高级编排

实际业务里，任务不是孤立的，常常是"一组任务按某种顺序/结构执行"。Celery 提供了三个核心原语：**chain、group、chord**。

### 14.1 链式任务（chain）：串行

A → B → C，一个接一个，前一个的输出是后一个的输入。

```python
from celery import chain

# 方式 1: 用 chain() 函数
workflow = chain(
    fetch_data.s(url='http://...'),    # 第一步
    parse_data.s(),                     # 第二步,接收上一步的返回值
    save_to_db.s(),                     # 第三步
)
result = workflow.apply_async()

# 方式 2: 用管道运算符
workflow = fetch_data.s('http://...') | parse_data.s() | save_to_db.s()
result = workflow.apply_async()
```

**要点**：每个任务的 `s()` 里**不**传上一步的参数，上一步的返回值会**自动作为最后一个位置参数**传入下一步。

**Agent 场景**：RAG 索引构建 → 下载 PDF → 切分 → embedding → 写入 pgvector，天然的 chain。

### 14.2 并行任务（group）：并行

多个独立任务同时跑，结果聚合。

```python
from celery import group

# 同时跑 5 个 embedding 任务
job = group(
    embed_chunk.s(chunk) for chunk in chunks
)
result = job.apply_async()

# 等全部完成
results = result.get()  # 返回一个 list
```

**要点**：group 里的任务**互相独立**，Worker 会尽量并行。`.get()` 阻塞等全部完成。

**Agent 场景**：一篇文档切成 100 个 chunk，并发调 100 次 embedding 接口。

### 14.3 chord：并行 + 汇总

group 跑完后触发一个"收尾任务"。这是 chord 的核心：

```python
from celery import chord

# 结构:
# group (100 个 embed 任务) → 全部完成后 → merge_and_index 任务
result = chord(
    (embed_chunk.s(chunk) for chunk in chunks),  # header: 并行的 group
    merge_and_index.s(doc_id=doc_id)             # body: 收尾任务
)()
```

merge_and_index 会收到 header 里所有任务的结果列表作为第一个参数。

**Agent 场景**：并行 embed 100 个 chunk → 全部完成后，把 embedding 批量写入 pgvector 并触发"索引完成"通知。

### 14.4 组合使用

chain、group、chord 可以互相嵌套。实际的 RAG 索引 pipeline 可能长这样：

```python
pipeline = chain(
    download_pdf.s(url),                         # 下载
    split_pdf.s(),                               # 切分
    chord(                                       # 并行 embedding
        (embed_chunk.s(i) for i in range(100)),
        merge_embeddings.s()
    ),
    save_to_pgvector.s(user_id=uid),             # 入库
    send_ready_notification.s(user_id=uid),      # 通知
)
pipeline.apply_async()
```

这一串相当于一个"工作流"，Celery 会自动帮你管理任务之间的触发关系。

### 14.5 优先级队列和路由

不同优先级的任务应该用不同的队列，避免高优任务被低优任务堵住。

```python
# 任务定义时指定默认队列
@app.task(queue='low_priority')
def batch_index():
    pass

@app.task(queue='high_priority')
def realtime_query():
    pass

# 配置 worker 分别消费不同队列
# 终端 1: 专门处理高优
# celery -A myapp worker -Q high_priority -c 20 --hostname=high@%h

# 终端 2: 专门处理低优
# celery -A myapp worker -Q low_priority -c 5 --hostname=low@%h
```

**Agent 场景**：
- `high_priority` 队列：用户实时对话触发的任务（比如工具调用）
- `default` 队列：后台 RAG 检索
- `low_priority` 队列：夜间批量索引构建

### 14.6 初学者常见误区

- **误区 1**："chain 和 group 一样。" → chain 是串行（顺序依赖），group 是并行（相互独立）。
- **误区 2**："chord 的 header 必须是 group。" → 从概念上 header 就是一组并行任务，实现里就是 group。
- **误区 3**："一个 worker 能消费所有队列。" → 默认只消费 `celery` 这一个队列。要消费多个队列必须用 `-Q` 指定。

## 15. 事件驱动架构 EDA

**事件驱动架构（Event-Driven Architecture，EDA）** 是比"消息队列"更上层的一种架构范式。前面讲的 MQ、Celery 都是实现 EDA 的技术手段，但 EDA 本身是一种设计思想。

### 15.1 从"命令"到"事件"的思维转变

传统的同步调用是**命令式**的：

> "用户服务，请给我创建一个用户。"

命令式调用的特点：
- 调用方知道被调用方是谁
- 调用方等待结果
- 逻辑在"指挥者"手里

EDA 是**事件式**的：

> "用户服务宣布：有一个新用户创建了。" —— 其他服务爱听不听。

事件式的特点：
- 发布方不知道谁在监听（解耦）
- 发布方不等结果（异步）
- 逻辑分布在各个"订阅者"手里

**一句话对比**：命令式是"我叫你干活"，事件式是"我告诉你发生了啥，你自己决定干啥"。

### 15.2 EDA 的好处

1. **极致解耦**：发布方和订阅方完全不认识，可以独立演化
2. **易于扩展**：想加一个新功能？订阅事件就行，不用改原有代码
3. **天然审计**：所有"发生过的事"都变成事件流，天然是审计日志

### 15.3 EDA 的代价

1. **最终一致性**：事件传播有延迟，数据短期内不一致
2. **调试困难**：事件满天飞，不知道一个 bug 是哪个订阅者干的
3. **消息顺序和重复**：分布式系统的老问题全都有

### 15.4 事件溯源（Event Sourcing）

这是 EDA 的一个高阶模式：**不保存"当前状态"，只保存"发生过的事件"**，当前状态是事件流"回放"出来的。

类比：银行账户系统不保存"余额 = 100"，而是保存：

```
2024-01-01: deposit 50
2024-01-02: deposit 80
2024-01-03: withdraw 30
```

查余额 = 把所有事件 reduce 一遍 = 100。

**优点**：
- 完整审计（什么时候发生了什么一清二楚）
- 可以"时光回溯"（算 2024-01-02 时的余额）
- 可以衍生出多个"视图"（同一份事件流可以支持不同的查询模型）

**代价**：
- 实现复杂度高
- 查询需要 reduce，性能不如直接存状态

**适合场景**：金融交易、订单状态、工作流引擎。

### 15.5 CQRS（命令查询职责分离）

CQRS = Command Query Responsibility Segregation。核心思想：**"写模型"和"读模型"分开**。

- 写操作 → 直接写到 Event Store
- 读操作 → 查专门为查询优化的"读模型"（可以是 DB/ES/缓存）

写模型和读模型通过**事件同步**。常和 Event Sourcing 搭配使用。

**Agent 场景**：Agent 的对话历史（写模型用事件流）+ 对话搜索（读模型用 Elasticsearch 索引）。

### 15.6 初学者须知

EDA、Event Sourcing、CQRS 这些概念在面试里会出现，但**实际项目里请谨慎使用**。它们的实现复杂度很高，很多团队用 EDA 的结果是"调试地狱 + 最终一致性 bug 满天飞"。

建议先从**简单的"发事件 + 几个订阅者"**做起，不要一上来就 Event Sourcing + CQRS 全家桶。

## 16. 幂等性设计

前面第 4 节讲消息语义时说过："at-least-once + 幂等消费 ≈ exactly-once"。这一节把幂等性讲透。它是 MQ 面试的另一个高频考点。

### 16.1 什么是幂等性

数学定义：函数 f 满足 `f(f(x)) = f(x)` 就叫幂等。

工程定义：**同一个操作执行 N 次和执行 1 次的效果相同**。

例子：
- `SET user.name = 'Allen'` → 幂等（执行 100 次结果都一样）
- `UPDATE accounts SET balance = balance + 100 WHERE id = 1` → **不幂等**（执行 2 次就多加了 100）
- `INSERT INTO orders ...` → **不幂等**（可能插两条）

### 16.2 为什么消息队列场景必须幂等

我再回顾一遍"为什么消息会被重复消费"：

1. 消费者处理完业务了，正要 ACK → 进程崩了 → Broker 认为没处理，重新投递
2. 网络抖动，ACK 包丢了 → Broker 重新投递
3. 生产者发消息时没收到 ACK，重试发送 → 同一条消息在队列里有两份
4. 人为操作：Worker 重启、消费组 reset offset、维护期间回放消息

这些场景你**无法完全避免**，只能接受"消息会被重复消费"的事实，然后**在消费逻辑里做幂等保护**。

### 16.3 幂等的五种实现方式

#### 方式一：数据库唯一约束（推荐）

给业务表加一个"去重键"的唯一索引：

```sql
CREATE TABLE orders (
    id UUID PRIMARY KEY,
    idempotency_key TEXT UNIQUE NOT NULL,  -- 去重键
    user_id INT,
    amount NUMERIC(10,2),
    created_at TIMESTAMPTZ DEFAULT NOW()
);
```

消费者：

```python
def consume_order_create(msg):
    try:
        db.execute(
            "INSERT INTO orders (id, idempotency_key, user_id, amount) VALUES (%s, %s, %s, %s)",
            (uuid.uuid4(), msg['idempotency_key'], msg['user_id'], msg['amount'])
        )
    except IntegrityError:
        # 唯一约束冲突 = 已经处理过了 = 幂等成功
        logger.info(f"duplicate message {msg['idempotency_key']}, skipped")
```

**优点**：简单、可靠、依赖数据库的 ACID。
**缺点**：每条消息都要访问数据库。

**去重键怎么选？** 业务唯一 ID（订单号、支付单号）最理想；没有的话用 `task_id`（MQ 消息 ID）或 UUID 生成一个带进 payload。

#### 方式二：去重表（Redis Set）

如果你不想污染业务表，可以维护一个专门的去重表：

```python
def consume(msg):
    msg_id = msg['id']
    # setnx = set if not exists
    if not redis.setnx(f"processed:{msg_id}", "1"):
        return  # 已处理,跳过
    redis.expire(f"processed:{msg_id}", 86400 * 7)  # 7 天过期

    do_business(msg)
```

**优点**：集中式、对业务无侵入。
**缺点**：
- 需要处理 Redis 故障时的降级
- 有"先 setnx 成功但业务失败"的一致性问题（需要配合事务或补偿）

**改进**："先业务再标记" vs "先标记再业务"需要根据业务容忍度权衡。推荐：

```python
def consume(msg):
    msg_id = msg['id']
    if redis.get(f"processed:{msg_id}"):
        return
    do_business(msg)
    redis.setex(f"processed:{msg_id}", 86400 * 7, "1")
```

但这种模式有一个漏洞：`do_business` 完成但 `setex` 之前崩溃，下次还会重复。100% 严格需要把 setex 和 do_business 放在同一个数据库事务里。

#### 方式三：状态机

业务数据本身有状态，消费时先判断状态：

```python
def pay_order(order_id):
    order = db.query("SELECT * FROM orders WHERE id = %s", order_id)
    if order.status != 'pending':
        return  # 已经处理过了
    order.status = 'paid'
    db.commit()
```

**优点**：天然幂等，不需要额外表。
**缺点**：要求业务有明确的状态流转，不是所有场景都有。

#### 方式四：分布式锁

```python
def consume(msg):
    lock_key = f"lock:{msg['id']}"
    if not redis.set(lock_key, "1", nx=True, ex=60):
        return  # 有人在处理了,跳过
    try:
        do_business(msg)
    finally:
        redis.delete(lock_key)
```

**优点**：防止并发重复。
**缺点**：锁本身不是幂等，是防并发。需要配合其他方式做真正的幂等。

#### 方式五：乐观锁（版本号）

```sql
UPDATE accounts SET balance = balance + 100, version = version + 1
WHERE id = 1 AND version = 5;
```

如果更新行数 = 0，说明版本号已变，别人改过了。

### 16.4 选型建议

| 场景 | 推荐方式 |
|------|---------|
| 有自然业务主键（订单号等） | 数据库唯一约束 |
| 业务状态机明确 | 状态机 + 事务 |
| 高并发、业务复杂 | Redis 去重表 + 事务 |
| 资金类、高可靠 | 数据库唯一约束 + 事务 |

**Agent 场景实战**：LLM 调用计费。每次调用都会产生一条计费记录。用 `request_id` 作为唯一约束，就算消费重复也只会扣一次费。

### 16.5 初学者常见误区

- **误区 1**："我让消息不重复就行了。" → 分布式系统里消息重复无法彻底避免，只能在消费端做保护。这是心智上的第一道坎。
- **误区 2**："我加个 if 判断就行了。" → 没加锁的情况下，if 判断和后续写入之间会有竞态条件（TOCTOU）。必须用数据库约束或者事务。
- **误区 3**："Celery 有 @task 就自动幂等了。" → 完全没有。Celery 只管投递和重试，幂等永远是业务代码的责任。

## 17. 死信队列与重试策略

### 17.1 什么是死信队列

**死信（Dead Letter）** 是指"怎么也消费不成功"的消息。常见死因：

1. 消息格式错误（反序列化失败）
2. 业务异常无法恢复（参数越界、找不到对应数据）
3. 超过最大重试次数仍然失败
4. 超过 TTL 还没被消费

如果你什么都不处理，这些消息有两种去处：
- 被无限重试（堵塞队列）
- 被直接丢弃（丢数据）

两种都不好。**死信队列（Dead Letter Queue，DLQ）** 就是专门收容这些消息的"垃圾暂存处"，让它们**暂时离开主队列**，等人来分析。

### 17.2 死信队列的价值

```
主队列 ──(消费成功)──→ 删除
  │
  └──(消费失败 N 次)──→ 死信队列 ──→ 人工排查 or 工具修复 or 重新投递
```

- **主队列畅通**：不会被毒消息堵死
- **可观察**：DLQ 堆积数量是核心监控指标
- **可恢复**：分析后可以修复数据再重投

### 17.3 重试策略

消费失败后到底怎么重试？有几种常见策略：

#### 1. 固定间隔

失败了等 10 秒再试，再失败等 10 秒，直到 N 次。简单但不够聪明。

#### 2. 指数退避（Exponential Backoff）

每次重试间隔翻倍：`10s → 20s → 40s → 80s → ...`

**为什么要指数退避？** 因为"失败"通常不是单条消息的问题，而是下游依赖（比如数据库、第三方 API）暂时挂了。立刻重试只会继续轰炸它。指数退避给下游"喘息"的时间，大家都好受。

#### 3. 指数退避 + 抖动（Jitter）

在指数退避的基础上加一个随机数，避免所有重试在同一时刻打过去。

```python
import random

def backoff(retry_count):
    base = 2 ** retry_count  # 2, 4, 8, 16, ...
    jitter = random.uniform(0, base * 0.3)  # 最多 30% 抖动
    return base + jitter
```

这是 AWS 官方推荐的重试方式。

### 17.4 Celery 的重试机制

Celery 内置了重试机制：

```python
@app.task(bind=True, max_retries=3, default_retry_delay=60)
def call_llm(self, prompt):
    try:
        return openai.chat(prompt)
    except APIError as exc:
        # 抛出 retry 异常,Celery 会按配置重试
        raise self.retry(exc=exc, countdown=2 ** self.request.retries)
```

参数解释：
- `bind=True`：让 self 可用，能调用 self.retry()
- `max_retries=3`：最多重试 3 次
- `default_retry_delay=60`：默认 60 秒后重试
- `self.request.retries`：当前是第几次重试（从 0 开始）
- `countdown=2 ** self.request.retries`：指数退避，1s → 2s → 4s

**自动重试的更优雅写法**：

```python
@app.task(
    bind=True,
    autoretry_for=(APIError, TimeoutError),  # 遇到这些异常自动重试
    retry_backoff=True,                       # 启用指数退避
    retry_backoff_max=600,                    # 最长 10 分钟
    retry_jitter=True,                        # 抖动
    max_retries=5,
)
def call_llm(self, prompt):
    return openai.chat(prompt)
```

这种写法比手动 raise retry 清爽得多。

### 17.5 Celery 没有原生 DLQ，怎么办？

Celery 本身不提供死信队列概念，但你可以用几种方式实现：

#### 方式一：失败时写到特殊队列

```python
@app.task(bind=True, max_retries=3)
def process(self, msg):
    try:
        do_work(msg)
    except Exception as exc:
        if self.request.retries >= self.max_retries:
            # 达到重试上限,投递到 DLQ
            dead_letter_handler.delay(msg, str(exc))
            return
        raise self.retry(exc=exc)

@app.task(queue='dead_letter')
def dead_letter_handler(msg, error):
    db.insert_dead_letter(msg, error)
    alert.send(f"DLQ new entry: {error}")
```

#### 方式二：用 RabbitMQ 的原生 DLX

RabbitMQ 有原生的 Dead Letter Exchange 机制，可以在队列配置里指定"消息被 nack 或超时后自动转发到哪"。Celery 可以配合：

```python
app.conf.task_queues = (
    Queue('main', Exchange('main'), routing_key='main',
          queue_arguments={
              'x-dead-letter-exchange': 'dlx',
              'x-dead-letter-routing-key': 'dead',
          }),
    Queue('dead_letter', Exchange('dlx'), routing_key='dead'),
)
```

#### 方式三：Redis Stream 手动 XCLAIM

用 Redis Stream 时，定时检查 PEL 中"待处理过久"的消息，用 XCLAIM 把它们搬到死信 stream。

### 17.6 初学者常见误区

- **误区 1**："重试次数越多越好。" → 重试是把双刃剑。业务错误（比如参数非法）重试 100 次也是失败，只会浪费资源、拖慢队列。区分"可重试错误"和"不可重试错误"。
- **误区 2**："重试 = 幂等的替代品。" → 两者是组合关系。重试没有幂等保护，数据会被越跑越乱。
- **误区 3**："DLQ 有了就万事大吉。" → DLQ 不是垃圾桶，必须有"监控 + 告警 + 人工处理流程"。没人看的 DLQ 等于没有。

## 18. 生产环境常见坑

这一节把真实生产环境的几个典型事故场景列一下。记住这些比背理论更值钱。

### 18.1 消息堆积

**现象**：队列越积越长，消费者跟不上。

**可能原因**：
1. 消费者处理变慢（某个环节性能退化）
2. 消费者数量不够（流量涨了但没扩容）
3. 消费者卡死（某条消息触发死循环）
4. 下游依赖变慢（数据库压力大）

**应对**：
1. 有**实时监控**队列长度（Prometheus + grafana 的看板是标配）
2. 堆积阈值告警（比如超过 1 万条拉 Oncall）
3. **消费者水平扩容**（K8s HPA 按队列长度自动扩容）
4. **降级策略**：实在跟不上，降级部分非核心任务

### 18.2 消费速度慢

**现象**：明明 Worker 在跑，但消费速度就是上不去。

**排查思路**：
1. `celery inspect active` 看有没有任务卡住
2. 加 metrics 看每个任务的执行耗时分布
3. Worker 是 prefork 还是 gevent？I/O 密集用 gevent，不然会被 CPU 核心数卡住
4. 是不是单个任务做了太多事？拆成更小的任务

### 18.3 消息丢失

**现象**：明明投递了，Worker 没收到。

**可能原因**：
1. Broker 没持久化（Redis 没开 AOF、RabbitMQ 没设 durable）
2. `acks_late=False`（默认），Worker 拉到就 ACK，崩溃时消息丢了
3. 生产者发的时候没等 ACK（`fire-and-forget`）

**应对**：
1. Redis 开 AOF（`appendfsync everysec`）
2. RabbitMQ 用 durable queue + persistent message
3. Celery 设 `task_acks_late=True, task_reject_on_worker_lost=True`
4. 关键业务在写 MQ 之前先写一条"待处理"记录到数据库（"本地消息表"模式）

### 18.4 重复消费

这个前面讲幂等性时讲过了。**原则：不靠 MQ 不重复，靠消费端幂等。**

### 18.5 顺序性问题

**现象**：明明先发的消息 A，结果先被处理的是 B。

**原因**：
- 多消费者并发 → 乱序
- 单分区消息路由到不同分区 → 乱序

**应对**：
1. **同一业务 key 路由到同一分区**：Kafka 的 `key=order_id` 哈希路由
2. **单分区 + 单消费者**：牺牲并发保顺序
3. **业务层不依赖顺序**：设计上尽量用幂等而不是顺序依赖

### 18.6 雷鸣群 effect（Thundering Herd）

**现象**：定时任务一起触发，瞬间打爆下游。

比如"每天 0 点刷新所有用户的额度"，100 万用户同时被触发，DB 直接 OOM。

**应对**：
- **分片 + 打散**：把任务拆成"每小时处理 4 万用户"
- **加随机延迟**：每个任务入队时加 `countdown=random(0, 3600)`
- **限流**：消费端加令牌桶限流

### 18.7 序列化失败

**现象**：Worker 日志里大量 `UnpicklingError`、`TypeError: Object not serializable`。

**原因**：把不能序列化的对象塞进了任务参数（比如 SQLAlchemy 的 Session、文件句柄、lambda）。

**应对**：
- 参数只传 ID 和原始类型
- 用 JSON 序列化器（默认），别用 pickle

### 18.8 Worker 僵死

**现象**：Worker 进程还在，但不处理任务。

**原因**：
- 某个任务阻塞住了（调了个 `time.sleep(10000)` 或死锁）
- 内存泄漏到 OOM 前

**应对**：
- `task_time_limit`：硬超时，超过就 kill worker
- `task_soft_time_limit`：软超时，任务可以捕获并优雅退出
- `worker_max_tasks_per_child`：worker 跑 N 个任务后自动重启（缓解内存泄漏）

### 18.9 任务洗牌问题

**现象**：重启 Worker 后，PEL 里的消息被"归还"到队列头，导致历史任务和新任务混杂，看起来顺序错乱。

**应对**：理解这是正常现象 + 业务不依赖严格顺序。

## 19. AI Agent 场景落地

前面讲了一堆理论，这一节把所有知识点揉在一起，看一个 AI Agent 平台怎么用 MQ/Celery。

### 19.1 Agent 平台的典型异步任务

一个生产级的 Agent 平台通常需要处理这些异步任务：

| 任务类型 | 为什么异步 | 推荐方案 |
|---------|-----------|---------|
| **知识库文档索引** | 切分+embedding 耗时几十秒到几分钟 | Celery + chain(split → group(embed) → save) |
| **长时工具调用** | 外部 API 可能慢且不稳定 | Celery + 指数退避重试 |
| **批量推理** | 一次几百上千条输入 | Celery + group + gevent 池 |
| **定时同步外部知识** | 每小时/每天同步 | Celery Beat |
| **对话摘要生成** | 对话结束触发，和用户无直接交互 | Celery task + Redis Stream 事件 |
| **审计/计费** | 每次调用都要记 | Redis Stream event (广播到 billing + audit) |
| **告警通知** | 指标超阈值触发 | Redis Stream + Pub/Sub |
| **WebSocket 推送** | 后端推消息给前端 | Redis Pub/Sub |

### 19.2 一个完整的 RAG 索引任务链

这是 OpenAgentic 一类 **Agent 平台**里最常见的异步任务。完整流程：

```python
# tasks.py
from celery import Celery, chain, chord, group

app = Celery('agent', broker='redis://localhost:6379/0',
             backend='redis://localhost:6379/1')

@app.task
def download_file(url: str) -> str:
    """下载文件到本地,返回本地路径。"""
    return local_path

@app.task
def split_document(local_path: str) -> list[str]:
    """切分文档,返回 chunks 列表。"""
    return chunks

@app.task(
    autoretry_for=(ConnectionError, TimeoutError),
    retry_backoff=True,
    max_retries=5,
)
def embed_chunk(chunk: str) -> list[float]:
    """调 embedding 接口,失败自动指数退避重试。"""
    return openai.embed(chunk)

@app.task
def save_embeddings(embeddings: list[list[float]], doc_id: int):
    """批量写入 pgvector。"""
    db.insert_vectors(doc_id, embeddings)

@app.task
def notify_user(result, user_id: int):
    """通知用户索引完成(通过 Redis Pub/Sub 推给 WebSocket)。"""
    redis.publish(f'user:{user_id}', '{"type":"index_done"}')

def build_rag_pipeline(url: str, doc_id: int, user_id: int):
    return chain(
        download_file.s(url),
        split_document.s(),
        # 关键: chord 做并行 embedding + 汇总写入
        # 这里需要一点技巧,因为 chord 的 header 要是 group
        # 实际代码里需要用 .map / 动态构造
    )
```

实际生产里还需要：
- 进度上报（每个 chunk 完成后 update_state）
- 失败时把 `doc_id` 的状态改为 `FAILED`
- 整体超时（task_time_limit）
- 监控面板展示当前有多少文档在索引

### 19.3 工具调用事件流

Agent 调工具时，每次调用都是一个事件。把这些事件丢进 Redis Stream，可以同时做多件事：

```python
# Agent 主流程: 调完工具后发事件
def after_tool_call(agent_id, tool_name, args, result, latency_ms):
    redis.xadd('tool_events', {
        'agent_id': agent_id,
        'tool_name': tool_name,
        'args': json.dumps(args),
        'latency_ms': latency_ms,
        'success': 'true' if result else 'false',
    })

# 三个消费者组各自处理:
# 1. billing_group: 计费
# 2. analytics_group: 统计热门工具
# 3. audit_group: 审计日志
```

一个事件流，三种消费方式，互不干扰。这就是典型的 EDA。

### 19.4 多 Agent 协作的事件总线

MultiAgent 场景里，Agent 之间需要通信：Agent A 产出一个中间结果，Agent B 基于它继续工作。

**反模式**：Agent A 直接 HTTP 调 Agent B。这会造成耦合爆炸。

**正确姿势**：用事件总线（Redis Stream 或 Kafka）作为通信媒介。

```python
# Agent A 完成后发事件
redis.xadd('agent_events', {
    'type': 'research_done',
    'task_id': task_id,
    'result': json.dumps(research_result),
})

# Agent B 订阅 agent_events,匹配 type=research_done 的事件
for msg in xreadgroup('agent_events', 'agent_b'):
    if msg['type'] == 'research_done':
        continue_work(msg['task_id'], json.loads(msg['result']))
```

优点：
- Agent A 不知道 Agent B 是谁
- Agent C/D/E 可以随时加入（订阅同样的事件）
- 所有事件都可以回放、审计

## 20. 面试考点

下面是这一章的面试高频题。在看答案前，先自己默答一遍。

### 20.1 为什么要用消息队列？（基础题）

**参考答案**：解决三个核心问题。(1) **异步**：耗时操作不阻塞主流程，响应更快；(2) **解耦**：生产者和消费者互不相识，一个挂了不影响另一个；(3) **削峰**：把突发流量变成稳定的处理节奏。典型例子是电商下单后的通知、积分、物流这些副流程都应该走 MQ。

**追问**：什么情况下不该用 MQ？
**答**：强一致场景（必须立刻看到结果）、简单 CRUD、团队没有 MQ 运维经验的项目。

### 20.2 At-least-once 和 exactly-once 的区别和取舍？

**参考答案**：
- at-least-once 保证不丢但可能重复；
- exactly-once 保证严格只处理一次；
- 工程上几乎没有真正的 exactly-once，常见做法是 **at-least-once + 消费端幂等 = 近似 exactly-once**；
- 选择理由：at-least-once 实现简单、开销小；exactly-once 的事务机制（比如 Kafka 的 Transactional Producer）性能代价高，且只保证 Kafka 到 Kafka，跨系统还是要业务幂等。

### 20.3 Redis Stream 和 Kafka 有什么区别？

**参考答案**：设计哲学几乎一样（都是追加日志 + Consumer Group）；区别主要在工程实现：
- Stream 数据在 Redis 单机内存里，容量受限；Kafka 分区磁盘存储，近乎无限；
- Stream 吞吐上限几十万 QPS，Kafka 能做到百万；
- Kafka 有完整的分区、副本、大数据生态；Stream 是 Redis 的一个数据结构；
- Stream 适合百万级以下的业务队列，Kafka 适合日志/大数据/事件流。

### 20.4 Celery 的 Worker 并发池有哪些？怎么选？

**参考答案**：prefork（默认，多进程，CPU 密集用）/ gevent（协程，I/O 密集用）/ eventlet / threads / solo。Agent 场景大部分任务都是 I/O 密集（调模型、调工具、读写数据库），**强烈推荐 gevent**，一个 worker 进程可以并发几百个任务。

### 20.5 怎么保证消息不丢失？

**参考答案**：三层保证。
- **生产者端**：发消息等 ACK，失败重试（配合本地消息表做兜底）；
- **Broker 端**：持久化（Redis AOF、RabbitMQ durable、Kafka 副本）；
- **消费者端**：`acks_late=True`，处理完业务再 ACK；配合幂等防止重复。

### 20.6 怎么实现幂等消费？

**参考答案**：四种主流方式。
- 数据库唯一约束（推荐）：用业务 ID 或 `task_id` 建 unique 索引；
- Redis 去重表 + setnx；
- 业务状态机 + 乐观锁；
- 分布式锁防并发 + 以上任一种防重复。
实际工程里优先选"数据库唯一约束 + 事务"，最可靠最简单。

### 20.7 Celery 的任务重试怎么配？

**参考答案**：推荐 declarative 配置：

```python
@app.task(
    autoretry_for=(TimeoutError,),
    retry_backoff=True,
    retry_backoff_max=600,
    retry_jitter=True,
    max_retries=5,
)
def my_task(): ...
```

关键点是：**只对可恢复错误重试**（网络、超时），**不对业务错误重试**（参数非法）；**指数退避 + 抖动**避免打爆下游；**达到上限后进 DLQ**让人工介入。

### 20.8 描述一个你设计过的异步任务系统

**参考答案模板**（结合你自己的项目调整）：

> 我在做 OpenAgentic 的知识库模块时，用 Celery + Redis 实现了 RAG 索引 pipeline。用户上传 PDF 后，FastAPI 只做文件落地和投递任务，立刻返回 task_id。后台有两类 worker：一类 prefork 处理文件切分（CPU 密集），一类 gevent 处理 embedding 调用（I/O 密集）。索引链路用 chain + group + chord 编排：下载 → 切分 → 并行 embedding → 汇总写入 pgvector → 发事件到 Redis Stream 通知用户。关键设计：(1) 每次 embedding 调用用指数退避重试处理限流；(2) 用 doc_id + chunk_index 做数据库唯一约束保证幂等；(3) 超过 3 次重试失败的任务投递到 dead_letter 队列；(4) Celery Beat 每小时扫一次队列长度，堆积超过阈值触发告警。

这个答案会让面试官明显感觉到你"真的做过"。如果你真的跑过 demos/ 里的代码，就能讲得出来。

### 20.9 你怎么监控 Celery？

**参考答案**：
- **Flower**：官方 Web 面板，看 worker 活跃度、任务历史；
- **Prometheus Exporter**：把队列长度、任务耗时暴露成 metrics；
- **Grafana**：展示 dashboard，关键指标是队列长度、任务成功率、P95 延迟、DLQ 堆积；
- **业务埋点**：每个关键任务自己上报 metrics。
- 告警：堆积超过阈值、失败率超过 5%、worker 心跳丢失。

### 20.10 设计一个分布式事件总线

**参考答案**：这是架构题，要从需求问起。关键点：
- 用什么做 Broker？Kafka（大吞吐）/ NATS（低延迟）/ Redis Stream（简单够用）
- 事件结构怎么定？`type + payload + metadata(trace_id, timestamp, version)`
- 消费者怎么解耦？每个订阅方独立 Consumer Group
- 怎么保证不丢/不重？at-least-once + 消费端幂等
- 怎么回放？基于 offset/ID 从历史位置重新消费
- 怎么做版本演进？事件 schema 加 version 字段，消费者兼容多版本

## 结语

到这里你应该对消息队列和任务队列有了一个相对完整的理解。再往下就是**跑 demos、动手踩坑**了。这一章的知识点总结成一张表：

| 知识点 | 核心理解 | Agent 场景用处 |
|--------|---------|---------------|
| 为什么要 MQ | 异步解耦削峰 | 几乎所有耗时操作都应该异步化 |
| 消息语义 | at-least-once + 幂等 ≈ exactly-once | 所有任务消费都要幂等 |
| Redis Stream | Kafka 的简化版 | 中小规模事件流首选 |
| Celery | Python 任务队列标准 | RAG 索引、工具调用、批量推理 |
| 任务编排 | chain/group/chord | 复杂 pipeline 的表达 |
| 定时任务 | Celery Beat | 定期同步/清理/报表 |
| 幂等 | 业务 ID + 唯一约束 | 防止重复计费、重复投递 |
| DLQ + 重试 | 指数退避 + 死信兜底 | 处理外部 API 不稳定 |
| EDA | 事件驱动，"宣告而非命令" | 多 Agent 协作、审计流 |

> 1. 把 demos/ 目录下所有代码跑一遍，修改参数观察行为

> 3. 在 OpenAgentic 项目里找一个同步调用，用 Celery 改造成异步，体会"用户响应时间变短"的真实效果
