## 1. 怎么学模式才不算白学

### 1.1 Why

面试和博客都爱提「单例 / 观察者 / 策略」，但**真正值钱的是**：你能说清**当前代码痛在哪**，以及**引入某个结构之后谁依赖谁变了**。

### 1.2 What：推荐顺序

1. 用一句话写清**症状**（改一处崩一片、复制粘贴、分支爆炸……）。
2. 在 **§2 按痛归类**里找方向，再用 **一览表**锁定模式名。
3. 画**依赖箭头**：谁调用谁、谁实现接口、谁发布事件。
4. 再写代码；先写测试或调用方更稳。

### 1.3 常见误区

- **误区**：项目里类名带 `Factory` 就是工厂模式。可能只是命名；要看是否**隔离了构造细节**。
- **误区**：模式用得越多越好。模式是**代价**：多一层抽象就多一层阅读成本。

## 2. GoF 23：整合速览

**GoF**（《Design Patterns》, 1994）处理的是**单进程内**面向对象设计的复用与变化：创建型 5、结构型 7、行为型 11。下面不重复长故事，只做两件事：**按常见痛点把模式归堆**，再给**二十三式一句话**，便于定位和查漏。

### 2.1 按痛点归类（同一痛可能有多种解法，需权衡）

| 你在痛什么 | 常提到的模式（括号内为 GoF 编号） |
|------------|-----------------------------------|
| 规则/算法经常换、不想到处改 if | **策略**(21)、**状态**(20)；流程固定只变某几步用 **模板方法**(22) |
| 树形结构、要统一对待「单个 vs 组合」 | **组合**(8)；在稳定结构上叠加多种遍历/报表操作 → **访问者**(23) |
| 第三方/老接口跟我不一致 | **适配器**(6)；子系统太杂想一口对外 → **外观**(10) |
| 两个独立维度组合爆炸 | **桥接**(7) |
| 运行时给对象「加料」又不改原类 | **装饰器**(9)（OO 结构意义）；要控制访问、延迟加载、打日志替身 → **代理**(12) |
| 一个变、好多对象都要跟着动 | **观察者**(19)；对象互相指来指去 → **中介者**(17) |
| 多个处理者依次试、不想写长 if-else | **责任链**(13) |
| 操作要排队、撤销、日志、宏 | **命令**(14) |
| 构造复杂、参数多、步骤多 | **建造者**(4)；具体类型由子类决定 → **工厂方法**(2)；整套产品族一起换风格 → **抽象工厂**(3)；复制比重新拉数据便宜 → **原型**(5)；全局唯一入口（慎用）→ **单例**(1) |
| 遍历方式要统一、又不想暴露内部 | **迭代器**(16) |
| 要快照/回滚对象状态 | **备忘录**(18) |
| 海量小对象内存吃不消 | **享元**(11) |
| 小语言/规则引擎 | **解释器**(15)（日常业务里相对少见） |

**整合的意思**：先对上表找**痛点桶**，再到下表对**名字**，而不是从 1 背到 23。

### 2.2 二十三式一句话（速查）

| # | 模式 | 一句话 |
|---|------|--------|
| 1 | 单例 | 全局只要一个实例时的入口（注意测试与并发）。 |
| 2 | 工厂方法 | 父类定接口，子类决定实例化哪个具体类。 |
| 3 | 抽象工厂 | 一次创建一整套配套产品，避免风格混用。 |
| 4 | 建造者 | 分步构造复杂对象，治构造参数爆炸。 |
| 5 | 原型 | 用克隆代替昂贵构造。 |
| 6 | 适配器 | 包一层，让不兼容接口能对接。 |
| 7 | 桥接 | 抽象与实现解耦，治多维度类爆炸。 |
| 8 | 组合 | 树结构里叶子和容器同一套操作。 |
| 9 | 装饰器 | 动态叠职责，少子类（与 Python `@` 语法同名不同义）。 |
| 10 | 外观 | 给复杂子系统一个简单门面。 |
| 11 | 享元 | 共享不变部分，省内存。 |
| 12 | 代理 | 替身控制访问（懒加载、保护、日志等）。 |
| 13 | 责任链 | 请求沿链传递，直到有人处理。 |
| 14 | 命令 | 把请求变对象，支持撤销、队列、日志。 |
| 15 | 解释器 | 为小型语法建解释执行结构。 |
| 16 | 迭代器 | 统一遍历，不暴露内部表示。 |
| 17 | 中介者 | 多对象不直连，改通过中介，降耦合。 |
| 18 | 备忘录 | 外部存快照，支持恢复。 |
| 19 | 观察者 | 一主题多订阅，状态变则通知。 |
| 20 | 状态 | 状态一变行为就变，整理状态机。 |
| 21 | 策略 | 封装算法族，可互换。 |
| 22 | 模板方法 | 骨架固定，子类填步骤。 |
| 23 | 访问者 | 数据结构稳定，往上面挂新操作。 |

### 2.3 学模式到底在对抗什么

1. **可维护性**：牵一发动全身 → 靠解耦。  
2. **可扩展性**：加功能要翻遍旧类 → 开闭原则。  
3. **复用性**：逻辑粘死在某一调用栈 → 高内聚低耦合。

**可运行示例**：策略、责任链、观察者见 `demos/`。

### 2.4 与现代后端的关系

GoF 在**策略/适配器/装饰（OO 义）**等场景仍然顺手，但大量痛苦已外移到 **HTTP、队列、事务、重试**（见 §3–§7 与 B8）。

## 3. 企业应用与 DDD 常见模式

| 模式 | 解决的痛 | 落地提示 |
|------|----------|----------|
| **Repository（仓储）** | 领域逻辑不想写 SQL / ORM 细节 | 接口在领域层，实现在基础设施层；注意别做成「万能 DAO」 |
| **Aggregate（聚合）** | 一致性边界不清，到处改同一张表 | 一个事务内只改一个聚合根；跨聚合用最终一致 |
| **Domain Event（领域事件）** | 「做完 A 还要通知 B/C」写满 if | 领域内发事件，由 handler 订阅；与消息中间件结合要小心顺序与重复 |
| **Anti-Corruption Layer（防腐层）** | 外部模型又丑又稳变，污染核心模型 | 边界处翻译 DTO ↔ 领域对象；厚一点没关系 |
| **Value Object / Entity** | 把「标识」和「可替换的值」混写 | 不变式集中在实体；值对象不可变 |
| **Application Service** | 用例编排无处可放 | 薄层：事务边界、鉴权、调用领域；避免胖 service 堆业务规则 |

这些名字来自 **DDD / 企业应用架构** 传统，**不在 GoF 23 里**，但比「再写一个 Utils」更能救命。

## 4. 集成与消息：系统跟系统怎么说话

| 模式 | 解决的痛 | 备注 |
|------|----------|------|
| **Transactional Outbox（发件箱）** | DB 提交成功、MQ 没发出去，或反过来 | 业务表 + outbox 表同事务；异步进程扫表投递 |
| **Saga（编排 / 编排+补偿）** | 跨服务没有廉价 XA | 每步本地事务 + 补偿；与 B8 理论稿呼应 |
| **Idempotency Key** | 重试导致重复扣款 / 重复建单 | HTTP 头或消息体带幂等键；服务端存结果一段时间 |
| **Event Carried State Transfer** | 消费者总要回调查询才能干活 | 事件里带足够状态（注意体积与隐私） |
| **CQRS（命令查询分离）** | 读模型和写模型争用同一 schema | 读写分离存储；小项目慎用，复杂度高 |
| **Event Sourcing** | 要完整审计与回放 | 存事件流而非当前状态；存储与迁移成本极高，**非默认选项** |

## 5. 可靠性模式：在会挂的世界里活下去

（与 **B8** `demos` 互补：那边有可运行熔断 / 重试 / 令牌桶示例。）

| 模式 | 解决的痛 |
|------|----------|
| **Circuit Breaker（熔断）** | 下游死了你还打满连接，拖死自己 |
| **Retry + Exponential Backoff + Jitter** | 偶发抖动；忌对非幂等写无脑重试 |
| **Timeout / Deadline** | 慢请求占满线程或协程 |
| **Bulkhead（舱壁）** | 一个下游把全池连接吃光；按租户或按依赖分池 |
| **Rate Limiting / Throttling** | 被刷或被自己重试风暴打爆 |
| **Health Check + Readiness** | K8s 要知道能不能切流量 |
| **Graceful Degradation（降级）** | 非核心失败时返回缓存或简化结果 |

## 6. 并发与吞吐：后台服务常见结构

| 模式 | 典型场景 |
|------|----------|
| **Producer–Consumer** | 请求线程只入队，worker 池处理慢任务（见 B6） |
| **Worker Pool / 线程池 / 协程池** | 限制并发、复用资源 |
| **Fan-out / Fan-in** | 并行调多个依赖再聚合（Agent 多工具并行） |
| **Pipeline** | 阶段化处理：解析 → 校验 → 持久化，每阶段可伸缩 |
| **Backpressure** | 下游跟不上时上游减速或拒绝（队列满、503、drop） |

## 7. API 与前端协作：网关与 BFF

| 模式 | 解决的痛 |
|------|----------|
| **API Gateway** | 统一 TLS、鉴权、限流、路由；客户端不面对 N 个微服务 |
| **BFF（Backend for Frontend）** | Web 与 App 需要不同聚合粒度；避免一个「万能大 API」 |
| **Strangler Fig（绞杀榕）** | 老系统切新系统，边沿路由逐步替换（见 §8） |

## 8. 演进式架构：绞杀榕与抽象分叉

- **绞杀榕**：新功能走新服务，老 URL 在网关层转发；旧代码自然萎缩。
- **抽象分叉**：在边界上新接口继承旧契约，或并行两套 adapter，等调用方迁完再删。

与「大爆炸重写」相比，更适合真实业务连续运行。

## 9. Python / FastAPI 语境下的「模式」

- **依赖注入（框架级）**：FastAPI `Depends` 本质是 **注入解析**，不是 GoF 的「依赖注入」类图那一套，但目的一样：**可测、可替换**。
- **装饰器（语言特性）**：Python `@` 常用于 **横切**（日志、重试、鉴权）；与 GoF **Decorator**（对象结构）名字撞车，别混谈。
- **Protocol / Typing**：鸭子类型 + 结构化子类型，便于 **Strategy / Repository** 不写臃肿继承树。

---

## 10. 反模式与过度设计

- **God Object / 万能 Service**：几千行的 `XxxManager`。
- **Premature Generalization**：只有两个实现就先上「抽象工厂 + 插件体系」。
- **Service Locator 滥用**：全局偷偷 `getBean()`，依赖看不清。
- **分布式单体**：很多微服务却共享数据库、同步链式调用、一起发布。

---

## 11. 与 AI Agent 代码的关系

- **策略 / 责任链**：不同意图路由、不同工具选择，可用策略对象或链式 handler 替代巨型 `if`（见 `demos/`）。
- **Fan-out**：多检索源、多 MCP 工具并行调用 → 注意 **超时与部分失败** 聚合。
- **幂等与 Outbox**：长任务写库 + 发通知，与 B6/B8 同一套思路。
