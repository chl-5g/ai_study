## 1. PostgreSQL 核心

### 1.1 数据类型

PostgreSQL 的类型系统是所有关系型数据库中最丰富的。

| 类别 | 常用类型 | 说明 |
|------|---------|------|
| 整数 | `SMALLINT`, `INTEGER`, `BIGINT` | 2/4/8 字节 |
| 精确小数 | `NUMERIC(p,s)`, `DECIMAL` | 金融场景必用，不丢精度 |
| 浮点 | `REAL`, `DOUBLE PRECISION` | 有精度损失，科学计算用 |
| 字符串 | `VARCHAR(n)`, `TEXT` | PG 中 `TEXT` 和 `VARCHAR` 性能一样，推荐直接用 `TEXT` |
| 布尔 | `BOOLEAN` | `TRUE` / `FALSE` / `NULL` |
| 日期时间 | `TIMESTAMP`, `TIMESTAMPTZ`, `DATE`, `INTERVAL` | 带时区的 `TIMESTAMPTZ` 是最佳实践 |
| JSON | `JSON`, `JSONB` | **JSONB 是二进制存储，支持索引，强烈推荐用 JSONB** |
| 数组 | `INTEGER[]`, `TEXT[]` | PG 原生支持数组列 |
| UUID | `UUID` | 分布式系统主键首选 |
| 网络 | `INET`, `CIDR`, `MACADDR` | 网络地址专用类型 |
| 全文搜索 | `TSVECTOR`, `TSQUERY` | 全文搜索的核心类型 |

**实际应用场景：** 设计 AI Agent 平台时，对话消息用 `TIMESTAMPTZ` 记录时间，工具调用参数用 `JSONB` 存储（结构不固定），用户 ID 用 `UUID`。

### 1.2 索引

索引是数据库性能优化的第一道防线。PG 支持多种索引类型，不同场景选不同索引。

#### B-tree（默认）

最常用的索引，适合**等值查询**和**范围查询**。

```sql
-- 创建 B-tree 索引（默认就是 B-tree）
CREATE INDEX idx_users_email ON users(email);

-- 复合索引：左前缀原则，查询条件要匹配索引列的顺序
CREATE INDEX idx_orders_user_date ON orders(user_id, created_at);

-- 唯一索引
CREATE UNIQUE INDEX idx_users_email_unique ON users(email);
```

**核心原理：** B-tree 是一棵平衡多路搜索树，每个节点包含多个 key 和指针。查找时间复杂度 O(log N)。数据按顺序存储，所以范围查询（`BETWEEN`, `<`, `>`）也很高效。

#### GIN（Generalized Inverted Index）

倒排索引，适合**包含查询**：全文搜索、JSONB 字段查询、数组包含。

```sql
-- JSONB 字段的 GIN 索引
CREATE INDEX idx_config_gin ON agents USING GIN (config);

-- 全文搜索的 GIN 索引
CREATE INDEX idx_search_gin ON articles USING GIN (to_tsvector('english', content));

-- 数组包含查询
CREATE INDEX idx_tags_gin ON posts USING GIN (tags);
-- 查询：SELECT * FROM posts WHERE tags @> ARRAY['python'];
```

**核心原理：** GIN 把字段值拆成多个元素（如 JSONB 的每个 key、文本的每个词），对每个元素建立到行的映射。就像一本书的索引——"Python" -> 第3,15,42页。

#### GiST（Generalized Search Tree）

广义搜索树，适合**空间数据**、**范围类型**、**近似搜索**。

```sql
-- 地理位置索引（配合 PostGIS）
CREATE INDEX idx_location ON stores USING GiST (location);

-- 范围类型索引
CREATE INDEX idx_booking_period ON bookings USING GiST (during);

-- pgvector 的向量索引也可以用 GiST（但推荐 HNSW）
```

**核心原理：** GiST 把数据空间递归地划分为区域，每个节点覆盖一个边界区域。查询时自顶向下，只访问与查询范围重叠的子树。

#### 索引选择速查

| 查询模式 | 推荐索引 |
|---------|---------|
| `WHERE x = 1` / `WHERE x > 10` | B-tree |
| `WHERE jsonb_col @> '{"key": "val"}'` | GIN |
| `WHERE tags @> ARRAY['a']` | GIN |
| 全文搜索 `@@` | GIN |
| 地理范围 / 最近邻 | GiST |
| 向量相似度搜索 | HNSW / IVFFlat（pgvector） |

### 1.3 EXPLAIN 查询分析

`EXPLAIN` 是优化 SQL 的核心工具。它告诉你数据库打算怎么执行一条查询。

```sql
-- 查看执行计划（不实际执行）
EXPLAIN SELECT * FROM users WHERE email = 'test@example.com';

-- 实际执行并显示真实耗时（推荐）
EXPLAIN (ANALYZE, BUFFERS, FORMAT TEXT)
SELECT * FROM users WHERE email = 'test@example.com';
```

**输出解读：**

```
Seq Scan on users  (cost=0.00..25.00 rows=1 width=100) (actual time=0.015..0.230 loops=1)
  Filter: (email = 'test@example.com'::text)
  Rows Removed by Filter: 999
  Buffers: shared hit=10
Planning Time: 0.050 ms
Execution Time: 0.250 ms
```

关键指标：

| 指标 | 含义 |
|------|------|
| `Seq Scan` | 顺序扫描（全表扫描），通常是慢查询的信号 |
| `Index Scan` | 索引扫描，高效 |
| `Index Only Scan` | 仅索引扫描（覆盖索引），最高效 |
| `cost=0.00..25.00` | 启动成本..总成本（相对单位） |
| `rows=1` | 估算返回行数 |
| `actual time` | 实际执行耗时（毫秒） |
| `Buffers: shared hit=10` | 从缓存读了 10 个页面 |

**优化思路：**
1. 看到 `Seq Scan` + 大表 → 加索引
2. `rows` 估算值和实际差很多 → 执行 `ANALYZE` 更新统计信息
3. `Nested Loop` 在大表上 → 考虑 `Hash Join`（可能需要调整 `work_mem`）

### 1.4 事务与隔离级别

事务是数据库可靠性的基石。ACID 四大特性：

- **Atomicity（原子性）：** 事务内操作要么全成功，要么全回滚
- **Consistency（一致性）：** 事务完成后数据满足所有约束
- **Isolation（隔离性）：** 并发事务互不干扰
- **Durability（持久性）：** 提交后数据永久保存

#### 四种隔离级别

| 隔离级别 | 脏读 | 不可重复读 | 幻读 | PG 实现 |
|---------|------|-----------|------|---------|
| Read Uncommitted | 可能 | 可能 | 可能 | PG 实际等同 Read Committed |
| **Read Committed**（PG默认） | 不可能 | 可能 | 可能 | 每条语句看到最新已提交数据 |
| Repeatable Read | 不可能 | 不可能 | 可能* | 快照隔离，事务开始时冻结视图 |
| Serializable | 不可能 | 不可能 | 不可能 | 完全串行化，最安全但最慢 |

> *PG 的 Repeatable Read 实际基于 MVCC（多版本并发控制），比标准定义更严格，幻读也被避免。

```sql
-- 设置事务隔离级别
BEGIN;
SET TRANSACTION ISOLATION LEVEL REPEATABLE READ;
SELECT balance FROM accounts WHERE id = 1;  -- 看到快照
-- ... 其他操作 ...
COMMIT;

-- 常见模式：转账
BEGIN;
UPDATE accounts SET balance = balance - 100 WHERE id = 1;
UPDATE accounts SET balance = balance + 100 WHERE id = 2;
COMMIT;  -- 原子性：两个 UPDATE 要么都成功，要么都回滚
```

**实际应用：** 在 AI Agent 平台中，扣减用户 token 额度必须用事务，否则并发调用可能超额。关键业务用 `Serializable`，普通查询用默认的 `Read Committed`。

### 1.5 JSONB 字段

JSONB 是 PostgreSQL 的杀手级特性，让关系型数据库也能处理半结构化数据。

```sql
-- 创建带 JSONB 的表
CREATE TABLE agents (
    id UUID PRIMARY KEY DEFAULT gen_random_uuid(),
    name TEXT NOT NULL,
    config JSONB NOT NULL DEFAULT '{}',
    created_at TIMESTAMPTZ DEFAULT NOW()
);

-- 插入 JSONB 数据
INSERT INTO agents (name, config) VALUES
('翻译助手', '{"model": "gpt-4", "temperature": 0.3, "tools": ["translate", "summarize"]}');

-- 查询 JSONB 字段
-- -> 返回 JSON 对象，->> 返回文本
SELECT name, config->>'model' AS model FROM agents;

-- 嵌套查询
SELECT * FROM agents WHERE config->'tools' ? 'translate';

-- 包含查询（使用 GIN 索引加速）
SELECT * FROM agents WHERE config @> '{"model": "gpt-4"}';

-- 修改 JSONB 字段（不用全量替换）
UPDATE agents
SET config = config || '{"max_tokens": 4096}'
WHERE name = '翻译助手';

-- 删除 JSONB 中的 key
UPDATE agents
SET config = config - 'temperature';

-- JSONB 路径查询（PG 12+）
SELECT * FROM agents
WHERE jsonb_path_exists(config, '$.tools[*] ? (@ == "translate")');
```

**JSONB vs 独立列的选择：**
- 结构固定、需要强约束 → 独立列
- 结构不固定、不同记录差异大 → JSONB
- AI Agent 的配置参数（每个 Agent 不同）→ 适合 JSONB

### 1.6 全文搜索

PostgreSQL 内置全文搜索，中小规模场景不需要 Elasticsearch。

```sql
-- 核心概念：
-- TSVECTOR：文档的词向量表示（分词 + 词频 + 位置）
-- TSQUERY：查询表达式

-- 基本使用
SELECT to_tsvector('english', 'The quick brown fox jumps over the lazy dog');
-- 结果：'brown':3 'dog':9 'fox':4 'jump':5 'lazi':8 'quick':2

SELECT to_tsquery('english', 'quick & fox');

-- 全文搜索查询
SELECT title, content
FROM articles
WHERE to_tsvector('english', content) @@ to_tsquery('english', 'database & optimization');

-- 为全文搜索创建索引（必须，否则每次查询都重新分词）
-- 方式一：表达式索引
CREATE INDEX idx_articles_fts ON articles
USING GIN (to_tsvector('english', content));

-- 方式二：存储计算列（PG 12+）
ALTER TABLE articles ADD COLUMN search_vector TSVECTOR
GENERATED ALWAYS AS (to_tsvector('english', coalesce(title,'') || ' ' || coalesce(content,''))) STORED;
CREATE INDEX idx_articles_search ON articles USING GIN (search_vector);

-- 搜索排名
SELECT title,
       ts_rank(to_tsvector('english', content), query) AS rank
FROM articles, to_tsquery('english', 'python & web') AS query
WHERE to_tsvector('english', content) @@ query
ORDER BY rank DESC;

-- 搜索高亮
SELECT ts_headline('english', content, to_tsquery('english', 'database'),
       'StartSel=<b>, StopSel=</b>, MaxFragments=3')
FROM articles
WHERE to_tsvector('english', content) @@ to_tsquery('english', 'database');
```

**中文全文搜索：** PG 默认的分词器不支持中文。需要安装 `zhparser` 或 `pg_jieba` 扩展：
```sql
-- 安装 zhparser 后
CREATE TEXT SEARCH CONFIGURATION chinese (PARSER = zhparser);
ALTER TEXT SEARCH CONFIGURATION chinese ADD MAPPING FOR n,v,a,i,e,l WITH simple;

SELECT to_tsvector('chinese', '人工智能正在改变世界');
```

**实际应用：** AI Agent 平台的知识库搜索、对话历史搜索。中小规模（百万级文档以内）用 PG 全文搜索足够，超大规模再上 Elasticsearch。

## 2. pgvector 扩展 -- RAG 的基础

pgvector 让 PostgreSQL 变成向量数据库，是构建 RAG（Retrieval-Augmented Generation）系统的关键基础设施。

### 2.1 向量存储

```sql
-- 安装扩展
CREATE EXTENSION vector;

-- 创建向量列（1536 维，对应 OpenAI text-embedding-ada-002）
CREATE TABLE documents (
    id SERIAL PRIMARY KEY,
    content TEXT NOT NULL,
    embedding vector(1536),  -- 1536 维向量
    metadata JSONB DEFAULT '{}'
);

-- 插入向量
INSERT INTO documents (content, embedding)
VALUES ('PostgreSQL 是最先进的开源数据库', '[0.1, 0.2, ..., 0.05]'::vector);
```

### 2.2 相似度搜索

三种距离度量：

| 运算符 | 距离类型 | 公式 | 适用场景 |
|--------|---------|------|---------|
| `<->` | L2（欧式距离） | `sqrt(sum((a_i - b_i)^2))` | 通用 |
| `<=>` | 余弦距离 | `1 - cosine_similarity` | **文本嵌入首选**（归一化向量） |
| `<#>` | 内积（负数） | `-sum(a_i * b_i)` | 已归一化时等价余弦 |

```sql
-- 余弦相似度搜索：找最相似的 5 个文档
SELECT content, 1 - (embedding <=> query_embedding) AS similarity
FROM documents
ORDER BY embedding <=> query_embedding
LIMIT 5;

-- 带过滤条件的相似度搜索
SELECT content, 1 - (embedding <=> query_embedding) AS similarity
FROM documents
WHERE metadata->>'category' = 'tech'
ORDER BY embedding <=> query_embedding
LIMIT 5;
```

**余弦相似度 vs 欧式距离：**
- 余弦相似度衡量方向是否一致（-1 到 1），不关心向量长度
- 欧式距离衡量空间中的绝对距离
- 文本嵌入模型输出的向量通常已归一化，此时两者等价。**推荐用余弦距离。**

### 2.3 HNSW 索引

HNSW（Hierarchical Navigable Small World）是目前最推荐的向量索引。

```sql
-- 创建 HNSW 索引（余弦距离）
CREATE INDEX idx_embedding_hnsw ON documents
USING hnsw (embedding vector_cosine_ops)
WITH (m = 16, ef_construction = 64);

-- 查询时设置搜索范围（越大越精确，越慢）
SET hnsw.ef_search = 100;
```

**核心原理：** HNSW 构建一个多层图。上层是稀疏的"高速公路"，下层是密集的"地方公路"。搜索时从上层的随机入口开始，逐层下降，每层都贪心地跳到离目标最近的节点。时间复杂度 O(log N)。

**参数调优：**
- `m`：每个节点的连接数（默认 16），越大索引越准但占用更多内存
- `ef_construction`：构建时的搜索范围（默认 64），越大构建越慢但索引质量越高
- `ef_search`：查询时的搜索范围（默认 40），越大查询越准但越慢

### 2.4 IVFFlat 索引

IVFFlat（Inverted File with Flat compression）是另一种向量索引，构建速度快但精度不如 HNSW。

```sql
-- 先确保有足够数据（至少是 lists 数量的 10-100 倍）
-- 创建 IVFFlat 索引
CREATE INDEX idx_embedding_ivf ON documents
USING ivfflat (embedding vector_cosine_ops)
WITH (lists = 100);  -- 聚类数量

-- 查询时设置探查的聚类数量
SET ivfflat.probes = 10;
```

**核心原理：** 先用 k-means 将所有向量聚成 `lists` 个簇。查询时，先找到最近的 `probes` 个簇，再在这些簇内做精确搜索。牺牲一定精度换取速度。

**HNSW vs IVFFlat：**

| 特性 | HNSW | IVFFlat |
|------|------|---------|
| 查询精度 | 更高 | 较低 |
| 查询速度 | 更快 | 较慢 |
| 构建速度 | 较慢 | **更快** |
| 内存占用 | 较大 | **较小** |
| 增量插入 | 支持 | 需要重建 |
| 推荐场景 | **生产环境首选** | 数据量大且对精度要求不高 |

### 2.5 RAG 中的向量检索流程

```
用户提问 → Embedding 模型 → 查询向量
                                ↓
                    pgvector 相似度搜索
                                ↓
                    Top-K 相关文档片段
                                ↓
                  拼接到 Prompt 中 → LLM 生成回答
```

```python
# 伪代码：RAG 检索流程
query = "如何优化数据库性能？"
query_embedding = embedding_model.encode(query)  # [0.1, 0.2, ..., 0.05]

# pgvector 搜索
results = db.execute("""
    SELECT content, 1 - (embedding <=> %s::vector) AS similarity
    FROM documents
    WHERE similarity > 0.7
    ORDER BY embedding <=> %s::vector
    LIMIT 5
""", [query_embedding, query_embedding])

# 拼接 Prompt
context = "\n".join([r.content for r in results])
prompt = f"根据以下参考资料回答问题：\n{context}\n\n问题：{query}"
answer = llm.generate(prompt)
```

## 3. Redis

Redis（Remote Dictionary Server）是内存数据库，核心价值：**快**。单线程模型，QPS 可达 10万+。

### 3.1 数据结构

#### String

最基本的类型，可以存字符串、整数、浮点数、二进制数据。

```bash
SET user:1:name "Allen"
GET user:1:name           # "Allen"
INCR page:views           # 原子递增，适合计数器
SETEX session:abc 3600 "user_data"  # 设置带过期时间（秒）
MSET k1 v1 k2 v2         # 批量设置
```

**应用场景：** 缓存、计数器、分布式锁、Session 存储。

#### Hash

类似 Python 的 dict，适合存储对象。

```bash
HSET user:1 name "Allen" age 30 city "Hangzhou"
HGET user:1 name          # "Allen"
HGETALL user:1            # 获取所有字段
HINCRBY user:1 age 1      # 字段原子递增
```

**应用场景：** 用户信息、商品详情等结构化对象。比 String 存 JSON 更节省内存（当字段少时用 ziplist 编码）。

#### List

双向链表，支持头尾操作。

```bash
LPUSH queue:tasks "task1" "task2"  # 左侧插入
RPOP queue:tasks                    # 右侧弹出 → "task1"
LRANGE queue:tasks 0 -1            # 获取全部
BRPOP queue:tasks 30               # 阻塞弹出，最多等 30 秒
```

**应用场景：** 消息队列（简单场景）、最新 N 条记录、时间线。

#### Set

无序集合，自动去重。

```bash
SADD user:1:tags "python" "ai" "database"
SMEMBERS user:1:tags      # 所有成员
SISMEMBER user:1:tags "python"  # 是否存在
SINTER user:1:tags user:2:tags  # 交集（共同标签）
SUNION user:1:tags user:2:tags  # 并集
```

**应用场景：** 标签系统、共同好友、去重（如防止重复投票）。

#### Sorted Set（ZSet）

有序集合，每个元素关联一个 score，按 score 排序。

```bash
ZADD leaderboard 100 "player1" 200 "player2" 150 "player3"
ZREVRANGE leaderboard 0 9 WITHSCORES  # Top 10（降序）
ZRANK leaderboard "player1"            # 排名
ZINCRBY leaderboard 50 "player1"       # 加分
ZRANGEBYSCORE leaderboard 100 200      # 分数范围查询
```

**应用场景：** 排行榜、延迟队列（score 为时间戳）、限流（滑动窗口）。

#### Stream（Redis 5.0+）

专为消息队列设计的数据结构，类似 Kafka。

```bash
XADD mystream * name "Allen" action "login"  # 添加消息
XREAD COUNT 10 STREAMS mystream 0            # 读取消息
XRANGE mystream - +                           # 范围读取

# 消费者组
XGROUP CREATE mystream group1 0
XREADGROUP GROUP group1 consumer1 COUNT 1 STREAMS mystream >
XACK mystream group1 <message-id>             # 确认消费
```

**应用场景：** 事件流、消息队列（需要消费者组、消息确认的场景）。比 List 做队列更专业。

### 3.2 缓存策略

#### TTL（Time To Live）

```bash
SET cache:user:1 '{"name":"Allen"}' EX 3600   # 1 小时过期
TTL cache:user:1                                # 查看剩余时间
PERSIST cache:user:1                            # 移除过期时间
```

#### LRU 淘汰策略

当 Redis 内存满时，需要淘汰策略。在 `redis.conf` 中配置：

```
maxmemory 2gb
maxmemory-policy allkeys-lru    # 所有 key 中淘汰最久未使用的
```

| 策略 | 说明 |
|------|------|
| `noeviction` | 不淘汰，内存满时返回错误（默认） |
| `allkeys-lru` | **推荐**：所有 key 中 LRU 淘汰 |
| `volatile-lru` | 仅淘汰设了 TTL 的 key |
| `allkeys-random` | 随机淘汰 |
| `volatile-ttl` | 淘汰 TTL 最短的 key |
| `allkeys-lfu` | 所有 key 中淘汰最少使用的（Redis 4.0+） |

### 3.3 发布订阅（Pub/Sub）

```bash
# 终端1：订阅频道
SUBSCRIBE notifications

# 终端2：发布消息
PUBLISH notifications "新任务已分配"

# 模式订阅
PSUBSCRIBE events.*    # 订阅 events.login, events.logout 等
```

**注意：** Pub/Sub 消息不持久化，订阅者离线时消息会丢失。需要可靠消息队列用 Stream。

### 3.4 常见使用场景

| 场景 | Redis 方案 | 数据结构 |
|------|-----------|---------|
| 缓存热点数据 | Cache-Aside 模式 | String / Hash |
| Session 存储 | `SET session:xxx data EX 3600` | String |
| 分布式锁 | `SET lock:resource NX EX 30` | String |
| 计数器/限流 | `INCR` / 滑动窗口 | String / Sorted Set |
| 排行榜 | Sorted Set | Sorted Set |
| 消息队列 | Stream（简单场景用 List） | Stream / List |
| 实时通知 | Pub/Sub | - |
| 去重 | `SADD` + `SISMEMBER` | Set |
| 地理位置 | `GEOADD` / `GEORADIUS` | Geo |

## 4. SQLAlchemy ORM

SQLAlchemy 是 Python 最强大的 ORM，分为 Core（SQL 表达式）和 ORM（对象关系映射）两层。

### 4.1 Engine — 连接管理

```python
from sqlalchemy import create_engine

# 创建引擎（连接池）
engine = create_engine(
    "postgresql://user:pass@localhost:5432/mydb",
    pool_size=5,           # 连接池大小
    max_overflow=10,       # 超出 pool_size 的额外连接
    pool_timeout=30,       # 等待连接的超时时间
    echo=True,             # 打印 SQL（调试用）
)

# SQLite（开发/测试用）
engine = create_engine("sqlite:///./test.db")
```

**核心原理：** Engine 不直接管理连接，而是持有一个连接池（Connection Pool）。当需要数据库连接时，从池中获取；用完归还。避免频繁创建/销毁连接的开销。

### 4.2 Session — 工作单元

Session 是 ORM 的核心，实现了 Unit of Work 模式。

```python
from sqlalchemy.orm import sessionmaker, Session

SessionLocal = sessionmaker(bind=engine)

# 使用方式一：手动管理
session = SessionLocal()
try:
    user = User(name="Allen")
    session.add(user)
    session.commit()
except:
    session.rollback()
    raise
finally:
    session.close()

# 使用方式二：上下文管理器（推荐）
with SessionLocal() as session:
    with session.begin():  # 自动 commit/rollback
        user = User(name="Allen")
        session.add(user)
```

### 4.3 Model 定义

```python
from sqlalchemy.orm import DeclarativeBase, Mapped, mapped_column, relationship
from sqlalchemy import String, ForeignKey, DateTime
from datetime import datetime
import uuid

class Base(DeclarativeBase):
    pass

class User(Base):
    __tablename__ = "users"

    id: Mapped[uuid.UUID] = mapped_column(primary_key=True, default=uuid.uuid4)
    name: Mapped[str] = mapped_column(String(100))
    email: Mapped[str] = mapped_column(String(255), unique=True)
    created_at: Mapped[datetime] = mapped_column(DateTime, default=datetime.utcnow)

    # 关系
    posts: Mapped[list["Post"]] = relationship(back_populates="author")

class Post(Base):
    __tablename__ = "posts"

    id: Mapped[int] = mapped_column(primary_key=True, autoincrement=True)
    title: Mapped[str] = mapped_column(String(200))
    content: Mapped[str] = mapped_column(String)
    author_id: Mapped[uuid.UUID] = mapped_column(ForeignKey("users.id"))

    author: Mapped["User"] = relationship(back_populates="posts")
```

> 上面是 SQLAlchemy 2.0 的新式写法（`Mapped` + `mapped_column`），类型注解更清晰。旧式写法用 `Column(Integer, ...)`，两者都可用。

### 4.4 关系映射

```python
# 一对多（User -> Posts）
class User(Base):
    posts: Mapped[list["Post"]] = relationship(back_populates="author")

class Post(Base):
    author_id: Mapped[int] = mapped_column(ForeignKey("users.id"))
    author: Mapped["User"] = relationship(back_populates="posts")

# 多对多（Post <-> Tag）
post_tags = Table(
    "post_tags", Base.metadata,
    Column("post_id", ForeignKey("posts.id")),
    Column("tag_id", ForeignKey("tags.id")),
)

class Tag(Base):
    __tablename__ = "tags"
    id: Mapped[int] = mapped_column(primary_key=True)
    name: Mapped[str] = mapped_column(String(50), unique=True)
    posts: Mapped[list["Post"]] = relationship(secondary=post_tags, back_populates="tags")

# 一对一
class UserProfile(Base):
    user_id: Mapped[int] = mapped_column(ForeignKey("users.id"), unique=True)
    user: Mapped["User"] = relationship(back_populates="profile")
```

### 4.5 查询构建

```python
from sqlalchemy import select, func, and_, or_

# 基本查询
stmt = select(User).where(User.name == "Allen")
user = session.execute(stmt).scalar_one_or_none()

# 条件组合
stmt = select(Post).where(
    and_(
        Post.created_at >= datetime(2024, 1, 1),
        or_(Post.title.contains("Python"), Post.title.contains("AI"))
    )
).order_by(Post.created_at.desc()).limit(10)

# 聚合
stmt = select(
    User.name,
    func.count(Post.id).label("post_count")
).join(Post).group_by(User.name).having(func.count(Post.id) > 5)

# 预加载关系（避免 N+1 问题）
from sqlalchemy.orm import joinedload, selectinload
stmt = select(User).options(selectinload(User.posts))
users = session.execute(stmt).scalars().all()

# 子查询
subq = select(func.count(Post.id)).where(Post.author_id == User.id).scalar_subquery()
stmt = select(User, subq.label("post_count"))
```

### 4.6 Alembic 迁移

Alembic 是 SQLAlchemy 的数据库迁移工具，类似 Django 的 `makemigrations` / `migrate`。

```bash
# 初始化
pip install alembic
alembic init alembic

# 配置 alembic.ini 中的数据库 URL
# sqlalchemy.url = postgresql://user:pass@localhost/mydb

alembic revision --autogenerate -m "add users table"

# 执行迁移

# 回滚
alembic downgrade -1

# 查看历史
alembic history
```

迁移脚本示例（`alembic/versions/xxx_add_users_table.py`）：

```python
def upgrade():
    op.create_table(
        'users',
        sa.Column('id', sa.Integer(), primary_key=True),
        sa.Column('name', sa.String(100), nullable=False),
        sa.Column('email', sa.String(255), unique=True),
    )

def downgrade():
    op.drop_table('users')
```

**最佳实践：**
- 每个功能一个迁移文件，不要积攒
- 迁移文件提交到 Git
- 生产环境迁移前先在 staging 测试
- 大表加列用 `ALTER TABLE ... ADD COLUMN ... DEFAULT NULL`（不锁表）

## 5. MySQL vs PostgreSQL 对比

| 维度 | MySQL | PostgreSQL |
|------|-------|------------|
| **定位** | Web 应用首选，简单高效 | 功能最完整的开源数据库 |
| **ACID** | InnoDB 引擎支持 | 完整支持 |
| **JSON 支持** | `JSON` 类型（5.7+） | **`JSONB`，支持索引，远强于 MySQL** |
| **全文搜索** | InnoDB FTS（基础） | **内置强大的 FTS，支持多语言** |
| **扩展性** | 有限 | **极强：pgvector、PostGIS、TimescaleDB...** |
| **并发控制** | 行级锁（InnoDB） | **MVCC，读写不互锁** |
| **复制** | 主从复制、Group Replication | 流复制、逻辑复制 |
| **存储过程** | 支持（SQL） | **支持 PL/pgSQL、Python、JavaScript...** |
| **类型系统** | 较基础 | **极丰富：数组、范围、UUID、网络地址...** |
| **标准兼容** | 较多自定义语法 | **最接近 SQL 标准** |
| **性能** | 简单查询略快 | 复杂查询更优 |
| **运维** | 生态成熟，DBA 多 | 运维工具不如 MySQL 丰富 |
| **适用场景** | 传统 Web 应用、已有 MySQL 栈 | **AI 应用、GIS、复杂查询、新项目首选** |

**结论：** 新项目、尤其是 AI 相关应用，**强烈推荐 PostgreSQL**。它的 JSONB、pgvector、全文搜索、扩展生态是 MySQL 无法比拟的。

## 6. 数据库设计原则

### 6.1 范式化（Normalization）

范式化的目标：消除数据冗余，避免更新异常。

#### 第一范式（1NF）：列的原子性

每个字段只存一个值，不允许嵌套表或重复组。

```sql
-- 违反 1NF
CREATE TABLE orders (
    id INT,
    products TEXT  -- "手机,电脑,耳机" ← 存了多个值
);

-- 符合 1NF
CREATE TABLE order_items (
    order_id INT,
    product_id INT,
    quantity INT
);
```

#### 第二范式（2NF）：消除部分依赖

非主键列必须完全依赖于整个主键（不能只依赖主键的一部分）。

```sql
-- 违反 2NF（复合主键 order_id + product_id）
CREATE TABLE order_details (
    order_id INT,
    product_id INT,
    product_name TEXT,   -- 只依赖 product_id，不依赖 order_id
    quantity INT,
    PRIMARY KEY (order_id, product_id)
);

-- 符合 2NF：拆分
CREATE TABLE products (id INT PRIMARY KEY, name TEXT);
CREATE TABLE order_details (order_id INT, product_id INT REFERENCES products(id), quantity INT);
```

#### 第三范式（3NF）：消除传递依赖

非主键列不能依赖于其他非主键列。

```sql
-- 违反 3NF
CREATE TABLE employees (
    id INT PRIMARY KEY,
    department_id INT,
    department_name TEXT  -- 依赖于 department_id，不直接依赖主键
);

-- 符合 3NF
CREATE TABLE departments (id INT PRIMARY KEY, name TEXT);
CREATE TABLE employees (id INT PRIMARY KEY, department_id INT REFERENCES departments(id));
```

### 6.2 反范式化（Denormalization）

范式化减少冗余，但**查询可能变慢**（需要 JOIN）。在读多写少的场景，有意地冗余数据换取查询性能。

```sql
-- 范式化：查询需要 JOIN
SELECT o.id, u.name, p.name, oi.quantity
FROM orders o
JOIN users u ON o.user_id = u.id
JOIN order_items oi ON oi.order_id = o.id
JOIN products p ON oi.product_id = p.id;

-- 反范式化：冗余用户名和产品名
CREATE TABLE order_denormalized (
    id INT PRIMARY KEY,
    user_id INT,
    user_name TEXT,       -- 冗余
    product_name TEXT,    -- 冗余
    quantity INT
);
-- 单表查询，零 JOIN，速度快
```

**何时反范式化：**
- 读远多于写
- JOIN 成为性能瓶颈
- 报表/分析场景
- 注意：需要维护冗余数据一致性（触发器或应用层同步）

### 6.3 索引设计策略

#### 原则

1. **选择性高的列优先索引**
   - 选择性 = 不同值数量 / 总行数
   - `email`（选择性接近 1）→ 适合索引
   - `gender`（只有 2-3 个值）→ 不适合单独索引

2. **复合索引的列顺序**
   - 最左前缀原则：`INDEX (a, b, c)` 可加速 `WHERE a=?`、`WHERE a=? AND b=?`、`WHERE a=? AND b=? AND c=?`
   - 但不能加速 `WHERE b=?` 或 `WHERE c=?`
   - 把等值查询的列放前面，范围查询的列放后面

3. **覆盖索引**
   ```sql
   -- 如果查询只需要 name 和 email
   CREATE INDEX idx_covering ON users(email) INCLUDE (name);
   -- Index Only Scan，不需要回表
   ```

4. **部分索引**
   ```sql
   -- 只索引活跃用户（减少索引大小）
   CREATE INDEX idx_active_users ON users(email) WHERE is_active = true;
   ```

5. **索引不是越多越好**
   - 每个索引都有写入开销（INSERT/UPDATE/DELETE 变慢）
   - 每个索引占用磁盘空间
   - 经验法则：一张表的索引数量 <= 列数量的一半

#### 常见反模式

```sql
-- 反模式1：对低选择性列建索引
CREATE INDEX idx_gender ON users(gender);  -- 几乎没用

-- 反模式2：冗余索引
CREATE INDEX idx_a ON t(a);
CREATE INDEX idx_a_b ON t(a, b);  -- idx_a 是冗余的，因为 idx_a_b 覆盖了

-- 反模式3：在函数结果上查询却不建表达式索引
SELECT * FROM users WHERE LOWER(email) = 'test@example.com';
-- 没用到 idx_users_email！需要：
CREATE INDEX idx_lower_email ON users(LOWER(email));
```

### 6.4 实战设计示例：AI Agent 平台

```sql
-- 用户表
CREATE TABLE users (
    id UUID PRIMARY KEY DEFAULT gen_random_uuid(),
    email TEXT UNIQUE NOT NULL,
    name TEXT NOT NULL,
    quota_tokens BIGINT DEFAULT 100000,  -- token 额度
    created_at TIMESTAMPTZ DEFAULT NOW()
);

-- Agent 定义表
CREATE TABLE agents (
    id UUID PRIMARY KEY DEFAULT gen_random_uuid(),
    owner_id UUID REFERENCES users(id),
    name TEXT NOT NULL,
    system_prompt TEXT,
    config JSONB DEFAULT '{}',  -- 模型参数、工具配置等
    is_public BOOLEAN DEFAULT FALSE,
    created_at TIMESTAMPTZ DEFAULT NOW()
);

-- 对话表
CREATE TABLE conversations (
    id UUID PRIMARY KEY DEFAULT gen_random_uuid(),
    user_id UUID REFERENCES users(id),
    agent_id UUID REFERENCES agents(id),
    title TEXT,
    created_at TIMESTAMPTZ DEFAULT NOW(),
    updated_at TIMESTAMPTZ DEFAULT NOW()
);

-- 消息表（最大的表，需要精心设计索引）
CREATE TABLE messages (
    id UUID PRIMARY KEY DEFAULT gen_random_uuid(),
    conversation_id UUID REFERENCES conversations(id) ON DELETE CASCADE,
    role TEXT NOT NULL CHECK (role IN ('user', 'assistant', 'system', 'tool')),
    content TEXT,
    tool_calls JSONB,          -- 工具调用记录
    token_count INTEGER,
    created_at TIMESTAMPTZ DEFAULT NOW()
);

-- 关键索引
CREATE INDEX idx_messages_conv_time ON messages(conversation_id, created_at);
CREATE INDEX idx_conversations_user ON conversations(user_id, updated_at DESC);
CREATE INDEX idx_agents_owner ON agents(owner_id);
CREATE INDEX idx_messages_tool_calls ON messages USING GIN (tool_calls)
    WHERE tool_calls IS NOT NULL;  -- 部分索引

-- 知识库（RAG 用）
CREATE TABLE knowledge_chunks (
    id UUID PRIMARY KEY DEFAULT gen_random_uuid(),
    agent_id UUID REFERENCES agents(id),
    content TEXT NOT NULL,
    embedding vector(1536),    -- pgvector
    metadata JSONB DEFAULT '{}',
    created_at TIMESTAMPTZ DEFAULT NOW()
);

CREATE INDEX idx_chunks_embedding ON knowledge_chunks
USING hnsw (embedding vector_cosine_ops);
CREATE INDEX idx_chunks_agent ON knowledge_chunks(agent_id);
```
