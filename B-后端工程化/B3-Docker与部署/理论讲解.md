## 1. 为什么要容器化

### 1.1 一个你多半经历过的痛苦场景

假设你花了一周写好了一个 FastAPI 应用，本地跑得挺好。现在要部署到服务器上让同事测试。你登上服务器开始操作：

```bash
# 服务器上 Python 是 3.8，你本地是 3.11
sudo apt install python3.11       # 装不上，Ubuntu 20.04 默认源没有

# 换源
sudo add-apt-repository ppa:deadsnakes/ppa
sudo apt update && sudo apt install python3.11

# pip 装依赖
pip3 install -r requirements.txt  # 报错，系统 pip 不能装
python3.11 -m venv venv           # 创建虚拟环境
source venv/bin/activate
pip install -r requirements.txt   # 某个库编译失败，缺 libpq-dev
sudo apt install libpq-dev        # 装好，重试
pip install -r requirements.txt   # 又一个库要 gcc
sudo apt install build-essential  # 装好编译工具链后再试 pip

# 终于能跑起来了，但 PG 数据库要怎么装？Redis 呢？
# 服务器 PG 版本是 12，你本地是 16，某个 SQL 语法在 12 不支持
# ...
```

一个下午过去了。你还没开始测试。同事过来问进展，你只能说"在装环境"。

再假设下个月你升级了一台新服务器，要把应用搬过去。上面这整套噩梦，你要**重新来一遍**。

再假设你要把应用交付给客户，客户的服务器是 CentOS 7，你的命令不能直接用了……

这就是传统部署的根本问题：**开发环境和运行环境不一致，依赖混乱，无法复现。**

### 1.2 容器化要解决的三件事

容器化的核心承诺可以浓缩成三句话：

1. **一次构建，到处运行（Build Once, Run Anywhere）**：你在自己电脑上构建一个镜像，它在任何支持 Docker 的机器上都能跑出一样的结果。
2. **环境隔离**：每个应用带着自己的依赖（Python 版本、系统库、配置文件），互不干扰。同一台服务器上可以同时跑 Python 3.8 和 3.11，没问题。
3. **可复现**：你的 Dockerfile 就是部署文档，任何人拿到它都能构建出完全一样的运行环境。

### 1.3 历史上容器化之前的几种方案（以及为什么都不够好）

为了真正理解容器的价值，我们需要看看它的竞争对手：

**方案一：物理机直接部署**
- 你有一台物理服务器，上面装操作系统，然后 `apt install` 所有依赖，把代码 clone 上去，跑起来。
- 痛点：机器资源浪费（一台 32 核服务器只跑一个应用）、依赖冲突（两个应用需要不同版本的 Python）、无法快速迁移（硬盘挂了全完）。

**方案二：虚拟机（VM）**
- 在物理机上用 VMware / VirtualBox / KVM 跑多个虚拟操作系统，每个虚拟机跑一个应用。
- 优点：隔离彻底（各自独立的内核），迁移方便（一个镜像文件复制就行）。
- 痛点：**太重了**。每个 VM 要跑一个完整的操作系统，内存动辄几 GB，启动要几十秒。一台物理机跑不了几个 VM。

**方案三：Ansible / Puppet / Chef 自动化脚本**
- 写一堆脚本描述"如何从空服务器装成可用状态"，用工具自动执行。
- 优点：比手工部署可复现多了。
- 痛点：还是要在目标机器上真的装东西，速度慢、受目标机器环境影响大。一个 apt 源挂了整个部署失败。

**容器的位置**：容器既有 VM 的"隔离"和"可迁移"，又比 VM 轻得多（共享宿主机内核，启动 1 秒）。它是虚拟化技术从"虚拟整个机器"进化到"只虚拟应用运行环境"的产物。

### 1.4 类比：集装箱运输

Docker 的 logo 是一头鲸鱼背上顶着一堆集装箱，这不是随便选的。容器化的整个思想就来自**集装箱运输革命**：

- **前集装箱时代**：不同形状的货物（酒桶、麻袋、家具……）被散装进船舱，码头工人一件件装卸，一艘船装货要几天，装卸过程还经常损坏货物。
- **集装箱时代**：所有货物先装进标准化的集装箱，船、火车、卡车都支持同一规格，装卸用吊车，一艘船几个小时装完。船员不关心集装箱里是什么，码头不关心货物要去哪个国家。

容器技术对软件的意义完全对应：
- 集装箱 = 容器镜像（标准化的封装单位）
- 船/火车/卡车 = 云服务商 / 物理机 / 开发机（任何支持 Docker 的环境）
- 吊车 = Docker runtime（标准化的装卸工具）

你写一个 Dockerfile 就像设计了一种集装箱的装箱方式：装进去的东西到哪儿都一个样。

## 2. Docker 核心概念

### 2.1 三个核心对象：镜像、容器、仓库

刚接触 Docker 最容易混的三个词：**镜像（Image）、容器（Container）、仓库（Registry）**。我们用程序员最熟悉的"类和对象"来类比：

| Docker 术语 | 类比 | 作用 |
|---|---|---|
| **Image（镜像）** | 类（Class） | 静态的、只读的模板，描述"我能变成什么样的运行环境" |
| **Container（容器）** | 对象/实例（Instance） | 镜像运行起来的进程，可以启动、停止、删除 |
| **Registry（仓库）** | PyPI / Maven Central | 存储和分发镜像的中心，Docker Hub 是最出名的公共仓库 |

具体讲：

- **镜像**：你可以把它想象成一个"装了应用和所有依赖的压缩包"。它是**不可变**的——一旦构建完成就不能改。`nginx:1.25` 是一个镜像，`python:3.11-slim` 是另一个镜像。
- **容器**：从镜像启动出来的一个正在运行的"实例"。同一个镜像可以启动多个容器（就像同一个类可以创建多个对象）。容器之间相互隔离，一个挂了不影响另一个。
- **仓库**：存镜像的地方。公共的有 Docker Hub、GHCR（GitHub Container Registry）、阿里云容器镜像服务。私有的你可以自己搭一个 Harbor。

生活中的对应：
- **Image = 蛋糕配方 + 所有原料的打包**
- **Container = 按配方烤出来的一块蛋糕**
- **Registry = 配方大全网站**

### 2.2 一个最简单的 Docker 工作流

看一遍完整流程，你会立刻理解三者的关系：

```bash
# 1. 从 Docker Hub 拉取（pull）一个镜像到本地
docker pull python:3.11-slim
# 本地现在有这个镜像了，可以 docker images 查看

# 2. 用这个镜像启动一个容器
docker run -it --rm python:3.11-slim python -c "print('hello from container')"
# -it     : 交互模式
# --rm    : 容器退出后自动删除（干净）
# python  : 镜像名
# 后面的命令会在容器里执行

# 3. 我自己写一个 Dockerfile，定制一个新镜像
echo "FROM python:3.11-slim
RUN pip install requests
CMD [\"python\", \"-c\", \"import requests; print(requests.__version__)\"]" > Dockerfile
docker build -t my-python-app .
# -t : tag，给新镜像取名
# .  : 构建上下文（把当前目录交给 Docker）

# 4. 运行我构建的新镜像
docker run --rm my-python-app
# 输出：2.31.0（容器里装了 requests）

# 5. 推送到仓库（别人就可以拉了）
docker tag my-python-app myusername/my-python-app:v1
docker push myusername/my-python-app:v1
```

读这段代码时，注意对应关系：
- `docker pull` / `docker build`：搞到一个镜像
- `docker run`：从镜像启动容器
- `docker push`：把镜像发到仓库

### 2.3 镜像的"标签"（tag）机制

你经常看到这样的镜像名：`python:3.11-slim`、`nginx:1.25-alpine`、`postgres:16`。冒号前是**镜像名**，冒号后是**标签（tag）**，类似 Git 里的 tag。

- `python:3.11` — Python 3.11 的最新次版本
- `python:3.11-slim` — slim 变体（更小的基础镜像）
- `python:3.11-alpine` — alpine 变体（最小但有坑，下面讲）
- `python:latest` — 最新稳定版（**生产环境绝不要用 latest**，因为它会变）
- `python:3.11.7-slim-bookworm` — 完全固定的版本

**生产环境黄金守则**：永远用明确的 tag，永远不用 `latest`。`latest` 意味着下次构建可能拉到不同的镜像，导致 "昨天还好好的怎么今天挂了" 的神秘问题。

### 2.4 初学者常见误区

1. **"容器就是虚拟机"**：不对。容器共享宿主机内核，虚拟机有自己的内核。这是最根本的区别。
2. **"镜像越小越好，所以全用 alpine"**：不一定。alpine 用的是 musl libc 不是 glibc，很多 Python 轮子不兼容，反而导致漫长的本地编译。生产 Python 应用推荐 slim 而不是 alpine。
3. **"容器里可以随便存数据"**：容器被删除时里面的数据全丢。要持久化必须用 volume。
4. **"每个容器跑多个服务"**：反模式。Docker 哲学是"一个容器一个主进程"，需要多服务用 docker-compose 编排。

## 3. Linux 容器底层原理

### 3.1 容器不是魔法：它是 Linux 内核的三个功能的组合

很多人觉得容器很神秘——为什么一个镜像能在任何 Linux 机器上跑出一样的效果？答案是：容器**根本就是一个普通的 Linux 进程**，只是这个进程被三个内核功能"骗"了：

1. **namespace（命名空间）**：让这个进程"看不见"其他进程和资源，以为自己是系统里唯一的存在
2. **cgroups（控制组）**：限制这个进程能用多少 CPU、内存、磁盘 IO
3. **UnionFS（联合文件系统）**：让这个进程看到的文件系统是分层堆叠出来的，而不是一个整体

理解这三个概念之后，"容器" 就从魔法变成了清晰的工程构造。

### 3.2 namespace：隔离的幻觉

**namespace** 是 Linux 内核提供的隔离机制。Linux 有 6 种主要的 namespace：

| Namespace | 隔离的是什么 | 类比 |
|---|---|---|
| PID | 进程 ID | 容器里的 PID 1 其实是宿主机的某个高位 PID |
| NET | 网络接口、路由表、端口 | 每个容器有自己的 "网卡" |
| MNT | 文件系统挂载点 | 每个容器看到不同的 / 根目录 |
| UTS | 主机名、域名 | 容器可以有自己的 hostname |
| IPC | 进程间通信（信号量、共享内存） | 不同容器的 IPC 不会互相影响 |
| USER | 用户 / 组 ID 映射 | 容器里的 root 可以不是宿主机的 root |

一个具体例子帮你"看见"namespace 的效果：

```bash
# 宿主机上
ps -ef | wc -l                     # 几百个进程
docker run -it --rm alpine sh
# 进了容器后：
ps -ef                              # 可能只有 2 行：sh 和 ps 本身
# 容器看不到宿主机的进程，这就是 PID namespace 的隔离

ip addr                             # 只看到 lo 和 eth0 两个接口
# 容器看不到宿主机的 wifi、docker0 这些，这是 NET namespace
```

### 3.3 cgroups：资源的硬限制

namespace 解决了"谁能看见谁"，**cgroups（control groups）** 解决"谁能用多少"。没有 cgroups，一个失控的容器可以耗光整台机器的内存或 CPU。

```bash
# 启动容器时限制资源
docker run -d \
    --cpus="1.5" \                  # 最多用 1.5 核
    --memory="512m" \               # 最多用 512MB 内存
    --memory-swap="512m" \          # 禁止用 swap（防止内存超时被 swap 拖慢）
    nginx:1.25
```

这些限制是**内核层面强制执行**的。容器进程尝试超过 memory 会被 OOM killer 杀掉，CPU 会被调度器按比例压制。从容器里看，它仍然觉得自己有整台机器——但真正要用资源时就会撞墙。

### 3.4 UnionFS：一层叠一层的文件系统

这是镜像"可分发 + 可复用"的关键。普通文件系统是"一个整体"，UnionFS 允许多个目录**叠起来看起来像一个目录**，每一层只读，最上面加一层可写。

直观理解：

```
镜像层（只读）             容器层（可写）
┌───────────────┐
│ 你的应用代码   │ ← 第 5 层
├───────────────┤
│ pip install 的 │ ← 第 4 层
│ 依赖           │
├───────────────┤
│ Python 3.11   │ ← 第 3 层
├───────────────┤
│ apt install   │ ← 第 2 层
│ 的系统库       │
├───────────────┤
│ Debian 基础层  │ ← 第 1 层
└───────────────┘
        +
┌───────────────┐
│ 容器运行时产生 │ ← 容器层（可写）
│ 的新文件       │
└───────────────┘
```

这种分层带来三大好处：

1. **复用**：多个容器共享同一镜像层，不用每个容器都完整复制一份。100 个 nginx 容器只占一份镜像的空间。
2. **可缓存**：Dockerfile 每条指令产生一层。改了最后一层不影响前面的层，下次构建直接用缓存。
3. **可增量分发**：`docker pull` 时只下载本地没有的层。

具体实现上，Docker 用的是 **overlay2** 驱动（Linux 默认），你可以 `docker info | grep Storage` 查看。

## 4. Docker 架构

### 4.1 一条 `docker run` 命令背后的调用链

当你敲下 `docker run nginx` 之后，发生了什么？这比你想的要复杂：

```
┌──────────────────────────────────────────────────────┐
│  你敲：docker run nginx                               │
└──────────────────┬───────────────────────────────────┘
                   │
                   ▼
┌──────────────────────────────────────────────────────┐
│ docker CLI（就是一个 HTTP 客户端）                   │
│ 通过 /var/run/docker.sock 向 daemon 发 HTTP 请求     │
└──────────────────┬───────────────────────────────────┘
                   │
                   ▼
┌──────────────────────────────────────────────────────┐
│ dockerd（Docker daemon，后台常驻进程）                │
│ - 处理 API 请求                                       │
│ - 管理镜像、网络、卷                                 │
│ - 调用 containerd 创建容器                           │
└──────────────────┬───────────────────────────────────┘
                   │
                   ▼
┌──────────────────────────────────────────────────────┐
│ containerd（容器运行时守护进程）                      │
│ - 负责容器的生命周期管理                             │
│ - 调用 runc 真正创建容器进程                         │
└──────────────────┬───────────────────────────────────┘
                   │
                   ▼
┌──────────────────────────────────────────────────────┐
│ runc（真正干活的）                                   │
│ - 调用 Linux 内核的 clone() / unshare() 系统调用     │
│ - 建 namespace / 挂载 rootfs / 设 cgroups            │
│ - 然后 exec 容器主进程                               │
└──────────────────────────────────────────────────────┘
```

### 4.2 为什么这么多层？

历史演进：早期 Docker 是个单体程序，后来为了规范化（OCI 容器运行时标准），被拆成了 daemon + containerd + runc 三段。好处是你可以用 runc 以外的运行时（比如 Kata Containers 用 VM 来跑容器，crun 更轻量）。

对你日常使用来说，**只需要知道 docker 命令不等于容器本身**。docker daemon 挂了，正在运行的容器其实不会停（因为 runc 已经把进程拉起来了）。

### 4.3 Docker Desktop vs 纯 Linux 上的 Docker

- **Linux 上**：docker daemon 直接跑在主机上，容器是主机的原生进程。
- **Mac/Windows 上**：Docker Desktop 会起一个 Linux 虚拟机（HyperKit/WSL2），所有容器实际在这个 VM 里跑。所以 Mac 上的 "container 比 Linux 慢"——多了一层 VM。

## 5. Dockerfile 详解

Dockerfile 是一个**文本文件**，用一系列指令描述"如何从一个基础镜像出发，构建出你想要的镜像"。Docker 按顺序执行指令，每条指令产生一个新层。

### 5.1 最常用的 10 条指令

以一个真实的 FastAPI 应用 Dockerfile 为例：

```dockerfile
# FROM：基础镜像。每个 Dockerfile 必须以 FROM 开头
# 选 slim 变体：比完整版小很多，比 alpine 兼容性好
FROM python:3.11-slim

# LABEL：镜像元信息（可选但推荐）
LABEL maintainer="allen@example.com"
LABEL description="AI Agent FastAPI backend"

# ENV：设置环境变量，整个构建和运行时都生效
# PYTHONDONTWRITEBYTECODE: 不生成 .pyc 文件
# PYTHONUNBUFFERED: 标准输出不缓冲（日志实时可见，生产必设）
ENV PYTHONDONTWRITEBYTECODE=1 \
    PYTHONUNBUFFERED=1 \
    PIP_NO_CACHE_DIR=1 \
    PIP_DISABLE_PIP_VERSION_CHECK=1

# WORKDIR：切换工作目录（不存在会自动创建）
# 类似 cd，但后续所有指令都基于这个目录
WORKDIR /app

# RUN：在镜像构建过程中执行命令
# 安装系统依赖。--no-install-recommends 减少附带依赖
# rm -rf /var/lib/apt/lists/* 清理 apt 缓存减小镜像
RUN apt-get update && apt-get install -y --no-install-recommends \
    gcc \
    libpq-dev \
 && rm -rf /var/lib/apt/lists/*

# COPY：把构建上下文里的文件复制到镜像
# 先只复制 requirements.txt 是为了利用缓存（下面详述）
COPY requirements.txt ./
RUN pip install -r requirements.txt

# 然后才复制整个应用代码
COPY . .

# EXPOSE：声明容器暴露的端口（仅文档作用，不真的开端口）
EXPOSE 8000

# USER：切换到非 root 用户（安全最佳实践）
RUN useradd --create-home --uid 1000 appuser
USER appuser

# HEALTHCHECK：健康检查，容器运行时定期执行
HEALTHCHECK --interval=30s --timeout=3s --start-period=5s \
  CMD python -c "import urllib.request; urllib.request.urlopen('http://localhost:8000/health')" || exit 1

# CMD：容器启动时运行的命令
# 推荐 exec 形式（数组），而不是 shell 形式（字符串）
# exec 形式直接启动进程，shell 形式会套一层 /bin/sh，信号传递有问题
CMD ["uvicorn", "app.main:app", "--host", "0.0.0.0", "--port", "8000"]
```

逐条再讲一次重点：

- **FROM**：起点。选镜像时考虑（a）体积（b）安全更新频率（c）兼容性。Python 选 `python:3.11-slim`，Node 选 `node:20-alpine` 或 `node:20-slim`。
- **ENV**：运行时和构建时都生效。设 `PYTHONUNBUFFERED=1` 是 Python 容器**铁律**，不设的话日志会卡在 stdout 缓冲区直到程序退出才输出，线上排查极其痛苦。
- **WORKDIR**：比 `RUN cd /app && ...` 更干净，每条 RUN 都会回到起始目录。
- **COPY vs ADD**：COPY 就是复制，ADD 会多两个"魔法"功能（自动解压 tar、下载 URL）。推荐**只用 COPY**，要下载就显式 `RUN curl`。
- **CMD vs ENTRYPOINT**：两者都定义容器启动命令。差别在"是否可被覆盖"：`docker run 镜像 命令` 会覆盖 CMD，但追加到 ENTRYPOINT 后面。一般 web 应用用 CMD 就够了。

### 5.2 ENTRYPOINT + CMD 的组合技巧

一个常见模式：用 ENTRYPOINT 指定"总是要跑的程序"，用 CMD 指定"默认参数"：

```dockerfile
ENTRYPOINT ["python", "manage.py"]
CMD ["runserver", "0.0.0.0:8000"]
```

这样：
- `docker run myapp` → 执行 `python manage.py runserver 0.0.0.0:8000`
- `docker run myapp migrate` → 执行 `python manage.py migrate`

ENTRYPOINT 保证了 "manage.py 永远跑"，CMD 提供默认命令但可以被覆盖。

### 5.3 初学者常见误区

1. **把所有 RUN 拆开**：每个 RUN 是一层，层越多镜像越大。应该用 `&&` 合并相关命令。
2. **忘了清理**：`apt install` 后不 `rm /var/lib/apt/lists/*`，镜像里留一堆用不着的包索引。
3. **COPY . . 在最前面**：代码一改整个缓存失效，下次构建要重装所有依赖。正确做法：先 COPY 依赖文件单独装，最后再 COPY 代码。
4. **用 root 运行**：镜像里的 root 虽然在 namespace 里隔离，但如果有内核漏洞被利用，风险大增。**生产镜像必须用非 root 用户**。
5. **CMD 用 shell 形式**：`CMD uvicorn ...` 会变成 `CMD ["/bin/sh", "-c", "uvicorn ..."]`，导致信号（SIGTERM）被 sh 吞掉，容器停不下来只能强杀。必须用 exec 形式 `CMD ["uvicorn", ...]`。

## 6. 镜像分层与缓存机制

### 6.1 每条指令 = 一层

Dockerfile 里每条 `FROM / RUN / COPY / ADD / ENV / CMD` 都会生成一个新的镜像层。用 `docker history 镜像名` 可以看到所有层：

```bash
$ docker history my-python-app
IMAGE          CREATED     CREATED BY                          SIZE
abc123         2 min ago   CMD ["uvicorn" "app.main:app"]      0B
def456         2 min ago   COPY . .                            15.2MB
ghi789         3 min ago   RUN pip install -r requirements.txt 180MB
jkl012         3 min ago   COPY requirements.txt ./            0.5KB
mno345         5 days ago  FROM python:3.11-slim               130MB
```

### 6.2 缓存规则：上游变了下游全废

构建时 Docker 会按顺序检查每条指令：
- 如果指令和上次一样，且输入文件内容没变，**直接用缓存**，这一层不重新执行
- 一旦某一层需要重建，**后面所有层都要重建**（因为底层变了，上面不能再用旧缓存）

这意味着 Dockerfile 里指令的**顺序非常重要**。越不容易变的东西越要写在前面。

### 6.3 实战：Dockerfile 优化前后

**❌ 糟糕版本**：

```dockerfile
FROM python:3.11-slim
WORKDIR /app
COPY . .                                # 代码一改这一层就失效
RUN pip install -r requirements.txt     # 后果：依赖每次都要重装
CMD ["uvicorn", "app.main:app"]
```

问题：你只改了一行代码，下次 `docker build` 时 pip install 会全部重跑，几分钟起步。

**✅ 优化版本**：

```dockerfile
FROM python:3.11-slim
WORKDIR /app
COPY requirements.txt ./                # 依赖文件变化才重建这一层
RUN pip install -r requirements.txt
COPY . .                                # 代码变化只影响这一层和之后
CMD ["uvicorn", "app.main:app"]
```

现在只改代码时，pip install 那一层走缓存，构建时间从几分钟降到几秒。

### 6.4 .dockerignore：避免无关文件进入构建上下文

`COPY . .` 会把**当前目录所有东西**复制到镜像。你不希望 `.git/`、`__pycache__/`、`.venv/`、`node_modules/` 等进镜像。在项目根目录放一个 `.dockerignore`：

```
.git
.gitignore
.venv
venv/
__pycache__
*.pyc
.pytest_cache
.mypy_cache
node_modules
*.log
.env
.env.*
tests/
docs/
README.md
```

作用有二：
1. 减小镜像体积
2. 更重要：**防止敏感文件（.env）意外进镜像**

### 6.5 BuildKit：新一代构建器

Docker 18.09+ 内置了 BuildKit，新版本默认启用。它比老的构建器快得多，支持：
- 并行构建多个阶段
- 更智能的缓存（基于内容而不是时间戳）
- 构建密钥注入（`--secret`）
- 更漂亮的进度输出

老版本可以用环境变量开启：`DOCKER_BUILDKIT=1 docker build ...`

## 7. 多阶段构建

### 7.1 为什么需要多阶段

假设你要构建一个 Go 应用。编译 Go 需要 Go 编译器（几百 MB），但运行只需要编译出来的二进制（几 MB）。你不想把编译器带到生产镜像里。

Python 也有类似场景：编译某些库（如 `cryptography`、`psycopg2`）需要 gcc、libpq-dev 等系统包，运行时根本不需要。

**多阶段构建**允许你在同一个 Dockerfile 里定义多个 `FROM`，前面的阶段用来编译/准备，最后一个阶段只复制需要的产物。

### 7.2 Python 多阶段示例

```dockerfile
# ========== Stage 1: builder ==========
# 带编译器的完整镜像
FROM python:3.11-slim AS builder

WORKDIR /build

# 安装构建依赖（只在这个阶段需要）
RUN apt-get update && apt-get install -y --no-install-recommends \
    gcc \
    libpq-dev \
 && rm -rf /var/lib/apt/lists/*

COPY requirements.txt .

# 把所有依赖装到一个固定目录
# --prefix 把依赖安装到 /install 下，方便后面 COPY 出来
RUN pip install --no-cache-dir --prefix=/install -r requirements.txt

# ========== Stage 2: runtime ==========
# 干净的运行时镜像，没有编译器
FROM python:3.11-slim AS runtime

ENV PYTHONDONTWRITEBYTECODE=1 \
    PYTHONUNBUFFERED=1 \
    PATH="/install/bin:$PATH"

WORKDIR /app

# 只装运行时必须的系统库（不装 gcc、不装 dev 包）
RUN apt-get update && apt-get install -y --no-install-recommends \
    libpq5 \
 && rm -rf /var/lib/apt/lists/*

# 从 builder 阶段把装好的依赖复制过来
COPY --from=builder /install /install
# 再复制应用代码
COPY . .

# 非 root 用户
RUN useradd --create-home --uid 1000 appuser
USER appuser

EXPOSE 8000
CMD ["uvicorn", "app.main:app", "--host", "0.0.0.0", "--port", "8000"]
```

构建出的镜像只包含 runtime 阶段的东西，体积可能从 800MB 降到 200MB。

### 7.3 更激进：distroless

Google 的 distroless 镜像连 shell 都没有（只有 glibc + 你要的运行时），安全性和体积都是极致：

```dockerfile
FROM python:3.11-slim AS builder
# ... 装依赖 ...

FROM gcr.io/distroless/python3-debian12
COPY --from=builder /install /install
COPY . /app
WORKDIR /app
CMD ["app/main.py"]
```

代价：出了问题你没法 `docker exec -it` 进去排查（没有 shell），调试手段受限。权衡取舍。

## 8. 镜像体积优化

### 8.1 同一个应用的五种镜像体积对比

一个简单 FastAPI 应用用不同基础镜像构建后的体积（大致数字，实际略有浮动）：

| 基础镜像 | 最终体积 | 优点 | 缺点 |
|---|---|---|---|
| `python:3.11` | ~1.0 GB | 兼容性最好 | 太大 |
| `python:3.11-slim` | ~180 MB | **生产推荐** | 某些 C 扩展要自己装编译器 |
| `python:3.11-alpine` | ~80 MB | 最小 | musl libc 不兼容很多 wheel，编译慢 |
| 多阶段 + slim | ~150 MB | 兼顾体积和兼容性 | Dockerfile 稍复杂 |
| distroless | ~120 MB | 最安全，无 shell | 调试困难 |

**一般建议**：
- 快速原型：`python:3.11-slim` 单阶段
- 生产环境：多阶段 + slim
- 极致安全（金融/合规）：distroless

### 8.2 为什么不推荐 alpine for Python

Alpine Linux 用 **musl libc**，而大多数 Python wheel 编译时是针对 **glibc** 的。这意味着在 alpine 上 `pip install` 很多流行库（numpy、pandas、lxml、pillow……）会**从源码编译**，慢到绝望，而且经常失败。

除非你对体积极其敏感且只用纯 Python 库，否则老实用 slim。

### 8.3 体积优化清单

按效果从大到小排：

1. **用多阶段构建剥离编译依赖**（可能节省 50%+）
2. **用 slim 而不是完整镜像**（节省 80%+ 体积）
3. **合并 RUN 指令并清理缓存**
4. **写好 .dockerignore**
5. **`pip install --no-cache-dir`**
6. **删除 __pycache__ 和 .pyc**（Python 的 `PYTHONDONTWRITEBYTECODE`）

## 9. 容器运行

`docker run` 是最常用、也最复杂的命令，光参数就有几十个。我们只讲你真正会用到的。

### 9.1 最常用参数速查

```bash
docker run \
    -d \                          # 后台运行（daemon 模式）
    --name myapi \                # 给容器命名，方便后续操作
    -p 8000:8000 \                # 端口映射 主机:容器
    -e DATABASE_URL=... \         # 设置环境变量
    --env-file .env \             # 从文件批量设置环境变量
    -v $(pwd)/data:/app/data \    # 挂载数据卷
    --network mynet \             # 加入指定网络
    --restart unless-stopped \    # 重启策略
    --cpus="1.5" \                # CPU 限制
    --memory="512m" \             # 内存限制
    myusername/myapi:v1.2         # 镜像
```

详细讲几个重点：

**`-d` vs `-it`**：
- `-d`：后台运行，适合服务类容器
- `-it`：交互式 + 分配 TTY，适合调试 (`docker run -it ubuntu bash`)
- 不要组合 `-dit`，这没意义

**`-p` 端口映射**：
- `-p 8000:8000`：把容器 8000 暴露到主机 8000
- `-p 127.0.0.1:8000:8000`：只绑定到 localhost（不对外）
- `-P`：随机端口，不推荐生产用

**`--restart` 重启策略**：
- `no`（默认）：挂了不重启
- `on-failure`：非 0 退出码才重启
- `always`：永远重启，包括手动 stop 后开机自启
- `unless-stopped`：**推荐**，除非你手动 stop 否则会重启

### 9.2 容器操作命令

```bash
docker ps                     # 列出运行中的容器
docker ps -a                  # 包括已停止的
docker stop myapi             # 优雅停止（SIGTERM + 10 秒宽限）
docker kill myapi             # 强制停止（SIGKILL）
docker rm myapi               # 删除已停止的容器
docker rm -f myapi            # 强制删除（包括运行中的）
docker logs myapi             # 看日志
docker logs -f myapi          # 跟随日志（类似 tail -f）
docker logs --tail 100 myapi  # 只看最后 100 行
docker exec -it myapi bash    # 进入运行中的容器调试
docker stats                  # 实时监控所有容器资源占用
docker inspect myapi          # 看容器详细信息（JSON）
```

### 9.3 信号和优雅关闭

`docker stop` 做两件事：
1. 发 SIGTERM 给容器主进程
2. 等 10 秒（可以用 `-t` 改）
3. 还没停？发 SIGKILL 强杀

这意味着你的应用**必须正确处理 SIGTERM**，否则每次 stop 都要等 10 秒才能杀掉。FastAPI + Uvicorn 默认处理得很好，自己写的 Python 守护进程要注意。

## 10. 数据卷 Volumes

### 10.1 容器删除 = 数据丢失

容器的"可写层"（上面讲 UnionFS 时提到的最顶层）在容器被删除时**一起删掉**。如果你的数据库数据写在容器里，`docker rm` 之后数据就没了。

三种数据持久化方案：

| 类型 | 命令 | 位置 | 适用场景 |
|---|---|---|---|
| Bind mount | `-v /host/path:/container/path` | 主机任意目录 | 开发：挂代码实时同步 |
| Named volume | `-v myvolume:/container/path` | Docker 管理的目录 | 生产：数据库、缓存 |
| tmpfs | `--tmpfs /container/path` | 内存 | 临时敏感数据 |

### 10.2 Bind mount vs Named volume 怎么选

**Bind mount**（主机路径挂载）：
- 优点：直接操作主机文件，开发时改代码立即生效
- 缺点：路径绑定到主机，迁移机器要改
- 典型：`-v $(pwd):/app` 开发时用

**Named volume**（命名卷）：
- 优点：Docker 管理位置，迁移可导出
- 缺点：要进容器才能直接看数据
- 典型：`-v pg_data:/var/lib/postgresql/data` 数据库持久化

生产环境**数据库数据必须用 Named volume 或外部存储**，绝不要用 bind mount（路径权限、迁移问题）。

### 10.3 volume 的生命周期

```bash
docker volume create myvol        # 手动创建
docker volume ls                  # 列出所有 volume
docker volume inspect myvol       # 看详情（包括实际存储路径）
docker volume rm myvol            # 删除（有容器在用会失败）
docker volume prune               # 删除所有没被容器引用的 volume
```

**重要陷阱**：`docker rm 容器` 不会自动删 volume。长期运行的服务器上你会看到一堆"孤儿 volume"。`docker volume prune` 定期清理。

## 11. Docker 网络

### 11.1 容器间怎么通信

默认情况下每个容器有自己的 network namespace，有独立的网卡、IP 地址。Docker 给你几种"网络模式"：

| 模式 | 特点 | 使用场景 |
|---|---|---|
| `bridge`（默认）| 容器连到 docker0 虚拟网桥，有独立 IP | 一般场景 |
| `host` | 容器直接用主机网络栈，共享主机 IP 和端口 | 需要极致性能或特殊端口 |
| `none` | 容器没有网络 | 纯计算任务 |
| `container:xxx` | 和另一个容器共享网络 | sidecar 模式 |
| 用户自定义 bridge | 推荐，支持 DNS 解析容器名 | **生产推荐** |

### 11.2 用户自定义网络的核心优势：容器名即 DNS

默认的 bridge 网络里，容器之间只能用 IP 通信（IP 还会变）。创建一个自定义 bridge 网络后，容器名字就是 DNS 名字：

```bash
# 创建网络
docker network create mynet

# 启动 PG
docker run -d --name pg --network mynet -e POSTGRES_PASSWORD=secret postgres:16

# 启动应用，可以直接用 "pg" 作为主机名
docker run -d --name api --network mynet \
    -e DATABASE_URL=postgresql://postgres:secret@pg:5432/postgres \
    myapi
```

注意 `DATABASE_URL` 里写的是 `pg`（容器名）而不是 IP！这是自定义网络的神奇之处。

### 11.3 端口映射的本质

`-p 8000:8000` 并不是 "容器监听主机的 8000"，而是 **"iptables 规则：进到主机 8000 的流量 NAT 到容器的 8000"**。

影响：
- 主机上 `netstat` 看不到容器监听的端口（是 docker-proxy 或 iptables 在处理）
- 容器内部监听必须是 `0.0.0.0`，如果监听 `127.0.0.1` 从主机访问不到（因为容器内外的 loopback 不是同一个）

## 12. docker-compose 编排

### 12.1 为什么单用 docker run 不够

一个真实应用通常有多个容器：FastAPI + PostgreSQL + Redis + Nginx。用 `docker run` 启动需要敲一堆命令，顺序还很重要，容易出错。

`docker-compose` 是一个用 YAML 描述"多容器应用拓扑"的工具，一条 `docker compose up` 就能把所有容器按依赖关系正确启动。

### 12.2 完整示例：AI Agent 后端栈

```yaml
# docker-compose.yml
services:
  # PostgreSQL + pgvector（用于 RAG 向量存储）
  postgres:
    image: pgvector/pgvector:pg16
    container_name: agent_pg
    restart: unless-stopped
    environment:
      POSTGRES_USER: agent
      POSTGRES_PASSWORD: ${PG_PASSWORD}    # 从 .env 读取
      POSTGRES_DB: agent
    volumes:
      - pg_data:/var/lib/postgresql/data
    # healthcheck 让依赖服务知道 PG 是否就绪
    healthcheck:
      test: ["CMD-SHELL", "pg_isready -U agent"]
      interval: 5s
      timeout: 3s
      retries: 10

  # Redis 用作缓存和任务队列
  redis:
    image: redis:7-alpine
    container_name: agent_redis
    restart: unless-stopped
    command: ["redis-server", "--appendonly", "yes"]
    volumes:
      - redis_data:/data
    healthcheck:
      test: ["CMD", "redis-cli", "ping"]
      interval: 5s
      retries: 10

  # FastAPI 后端
  api:
    build:
      context: ./api
      dockerfile: Dockerfile
    container_name: agent_api
    restart: unless-stopped
    environment:
      DATABASE_URL: postgresql://agent:${PG_PASSWORD}@postgres:5432/agent
      REDIS_URL: redis://redis:6379/0
      OPENAI_API_KEY: ${OPENAI_API_KEY}
    # depends_on + condition 等待 PG/Redis 真正就绪
    depends_on:
      postgres:
        condition: service_healthy
      redis:
        condition: service_healthy
    ports:
      - "127.0.0.1:8000:8000"  # 只对 localhost 暴露，靠 Nginx 对外

  # Nginx 反向代理（对外入口）
  nginx:
    image: nginx:1.25-alpine
    container_name: agent_nginx
    restart: unless-stopped
    ports:
      - "80:80"
      - "443:443"
    volumes:
      - ./nginx/nginx.conf:/etc/nginx/nginx.conf:ro
      - ./nginx/certs:/etc/nginx/certs:ro
    depends_on:
      - api

volumes:
  pg_data:
  redis_data:
```

### 12.3 compose 常用命令

```bash
docker compose up -d              # 后台启动所有服务
docker compose up -d api          # 只启动 api 服务
docker compose down               # 停止并删除所有容器（保留 volume）
docker compose down -v            # 连 volume 一起删（数据会丢）
docker compose ps                 # 状态
docker compose logs -f api        # 跟随某个服务的日志
docker compose exec api bash      # 进入运行中的服务
docker compose restart api        # 只重启 api 服务
docker compose build              # 重新构建镜像（改了 Dockerfile 要手动 build）
docker compose pull               # 拉最新的镜像
```

### 12.4 .env 文件

compose 会自动加载项目根目录的 `.env`：

```
# .env
PG_PASSWORD=super_secret_password
OPENAI_API_KEY=sk-...
```

**注意**：`.env` 必须加入 `.gitignore`，否则密码会提交到 git 仓库。

### 12.5 profiles：选择性启动

一个 compose 文件可以描述多个环境（开发 / 生产 / 调试），用 profiles 区分：

```yaml
services:
  api:
    # 不带 profile 默认启动
    ...
  mailhog:
    image: mailhog/mailhog
    profiles: ["dev"]  # 只在 dev profile 下启动
```

启动：`docker compose --profile dev up -d`

## 13. 生产级 Dockerfile 的八个细节

快速清单，每一条都要做到：

1. **基础镜像固定 tag**，不用 `latest`：`python:3.11.7-slim-bookworm`
2. **非 root 用户**运行：`RUN useradd appuser && USER appuser`
3. **HEALTHCHECK** 必配：K8s / compose 靠这个判断服务是否真的可用
4. **处理 PID 1 信号**：Python 有 signal 模块正确处理，或用 [`tini`](https://github.com/krallin/tini) 包一层
5. **`PYTHONUNBUFFERED=1`**：保证日志实时输出
6. **日志到 stdout/stderr**：不要写文件，让容器编排工具（docker logs / K8s）统一收集
7. **.dockerignore 配置完善**，避免敏感文件和无关文件进镜像
8. **多阶段构建剥离编译依赖**

### 13.1 时区问题

默认容器是 UTC 时区。Python 代码里 `datetime.now()` 会得到 UTC 时间，打日志时很迷惑。解决方案：

```dockerfile
# 方法一：设环境变量（推荐，轻量）
ENV TZ=Asia/Shanghai

# 方法二：装 tzdata 包（更彻底，但增加体积）
RUN apt-get update && apt-get install -y tzdata \
 && ln -sf /usr/share/zoneinfo/Asia/Shanghai /etc/localtime \
 && echo "Asia/Shanghai" > /etc/timezone
```

但**更好的实践是 Python 代码始终用 UTC，展示层再转时区**。容器就保持 UTC。

## 14. Kubernetes 入门概念

本章不深入 K8s（那是一本书的内容），只建立概念对应关系，方便你看懂相关文章和面试答题。

### 14.1 K8s 是什么

Docker 解决了"单机跑多个容器"。K8s 解决"多台机器上跑一堆容器，自动调度、自动扩缩容、自动恢复"。

一个 K8s 集群包含：
- **Master 节点**（Control Plane）：大脑，调度所有决策
- **Worker 节点**：真正跑容器的机器
- **Pod**：K8s 调度的最小单位，**一个 Pod 里可以有多个容器**（通常是 1 个主容器 + sidecar）
- **Deployment**：描述"我想让这个应用始终有 N 个副本"
- **Service**：给一组 Pod 提供稳定的访问入口（IP 和 DNS 名）
- **Ingress**：集群外部访问 Service 的网关（类似反向代理）
- **ConfigMap / Secret**：配置文件和密钥管理
- **Namespace**：逻辑隔离多个项目或团队

### 14.2 最小 Deployment 示例

```yaml
apiVersion: apps/v1
kind: Deployment
metadata:
  name: agent-api
spec:
  replicas: 3                      # 我要 3 个副本
  selector:
    matchLabels:
      app: agent-api
  template:
    metadata:
      labels:
        app: agent-api
    spec:
      containers:
        - name: api
          image: myregistry/agent-api:v1.2
          ports:
            - containerPort: 8000
          env:
            - name: DATABASE_URL
              valueFrom:
                secretKeyRef:
                  name: agent-secrets
                  key: database-url
          # 存活和就绪探针（K8s 关键概念）
          livenessProbe:
            httpGet:
              path: /health
              port: 8000
            periodSeconds: 30
          readinessProbe:
            httpGet:
              path: /ready
              port: 8000
            periodSeconds: 5
          resources:
            requests:
              memory: "256Mi"
              cpu: "200m"
            limits:
              memory: "512Mi"
              cpu: "500m"
---
apiVersion: v1
kind: Service
metadata:
  name: agent-api-svc
spec:
  selector:
    app: agent-api                 # 匹配上面 Deployment 的标签
  ports:
    - port: 80
      targetPort: 8000
  type: ClusterIP                  # 集群内部 IP
```

### 14.3 docker-compose 和 K8s 的对应关系

| docker-compose | K8s |
|---|---|
| service | Deployment + Service |
| volumes | PersistentVolume + PersistentVolumeClaim |
| networks | 自动（同 namespace 内互通） |
| depends_on | initContainers 或 readiness 探针 |
| environment | env / ConfigMap / Secret |
| ports | Service + Ingress |

**实用建议**：创业和中小项目用 docker-compose 足矣。只有当你真的要跑在多台机器上、需要自动扩缩容、自动恢复时才上 K8s。K8s 学习曲线陡峭，上生产要至少一个专职 DevOps。

## 15. Nginx 反向代理与负载均衡

### 15.1 为什么需要 Nginx 放在 FastAPI 前面

Uvicorn 本身是一个 ASGI 服务器，可以直接对外，但生产环境几乎都会在它前面加一层 Nginx：

1. **TLS 终止**：Nginx 处理 HTTPS，FastAPI 只处理 HTTP（减负）
2. **静态文件**：CSS/JS/图片让 Nginx 直接吐出来，不过 Python
3. **负载均衡**：多个 FastAPI 实例时 Nginx 按策略分发
4. **限流 / 访问控制 / WAF**：Nginx 有成熟的模块
5. **缓冲慢客户端**：慢连接会占用 Uvicorn worker，Nginx 能缓冲请求后一次性转发

### 15.2 最小 Nginx 配置

```nginx
# /etc/nginx/nginx.conf
worker_processes auto;

events {
    worker_connections 1024;
}

http {
    # 上游 FastAPI（多实例时列多行）
    upstream fastapi_backend {
        server api:8000;
        # server api2:8000;  # 水平扩展时加
        keepalive 32;      # 复用后端连接
    }

    # HTTP → HTTPS 重定向
    server {
        listen 80;
        server_name api.example.com;
        return 301 https://$host$request_uri;
    }

    # HTTPS 主站
    server {
        listen 443 ssl http2;
        server_name api.example.com;

        ssl_certificate     /etc/nginx/certs/fullchain.pem;
        ssl_certificate_key /etc/nginx/certs/privkey.pem;

        # SSE 流式输出必须关闭 buffering
        location /stream {
            proxy_pass http://fastapi_backend;
            proxy_http_version 1.1;
            proxy_buffering off;
            proxy_cache off;
            proxy_set_header Connection '';
            chunked_transfer_encoding on;
            proxy_read_timeout 3600s;  # 长时间流不超时
        }

        # WebSocket
        location /ws {
            proxy_pass http://fastapi_backend;
            proxy_http_version 1.1;
            proxy_set_header Upgrade $http_upgrade;
            proxy_set_header Connection "upgrade";
            proxy_read_timeout 3600s;
        }

        # 普通 API
        location / {
            proxy_pass http://fastapi_backend;
            proxy_set_header Host $host;
            proxy_set_header X-Real-IP $remote_addr;
            proxy_set_header X-Forwarded-For $proxy_add_x_forwarded_for;
            proxy_set_header X-Forwarded-Proto $scheme;
        }
    }
}
```

逐段讲解：

- **`upstream`**：定义后端服务池，可以列多个 server 实现负载均衡。`keepalive 32` 让 Nginx 到后端保持长连接减少开销。
- **`proxy_pass`**：转发目的地。用 `http://fastapi_backend` 会走 upstream 的负载均衡。
- **流式和 WebSocket 的特殊设置**：`proxy_buffering off` 是关键。默认 Nginx 会缓冲后端响应再一次性返回给客户端，流式输出会变成"一次性吐"。
- **X-Forwarded-For / X-Real-IP**：把真实客户端 IP 传给后端（FastAPI 用 `request.headers` 读取）。

## 17. CI/CD 与镜像发布

### 17.1 GitHub Actions 构建并推送镜像

一个完整的 workflow：

```yaml
# .github/workflows/build.yml
name: Build and Push Docker Image

on:
  push:
    branches: [main]
    tags: ['v*']
  pull_request:
    branches: [main]

jobs:
  build:
    runs-on: ubuntu-latest
    permissions:
      contents: read
      packages: write   # 推送到 GHCR 需要

    steps:
      - name: Checkout
        uses: actions/checkout@v4

      # 启用 BuildKit
      - name: Set up Docker Buildx
        uses: docker/setup-buildx-action@v3

      # 登录 GHCR
      - name: Log in to GHCR
        if: github.event_name != 'pull_request'
        uses: docker/login-action@v3
        with:
          registry: ghcr.io
          username: ${{ github.actor }}
          password: ${{ secrets.GITHUB_TOKEN }}

      # 生成镜像 tag
      - name: Extract metadata
        id: meta
        uses: docker/metadata-action@v5
        with:
          images: ghcr.io/${{ github.repository }}
          tags: |
            type=ref,event=branch
            type=semver,pattern={{version}}
            type=sha,prefix=sha-

      # 构建 + 推送
      - name: Build and push
        uses: docker/build-push-action@v5
        with:
          context: ./api
          push: ${{ github.event_name != 'pull_request' }}
          tags: ${{ steps.meta.outputs.tags }}
          labels: ${{ steps.meta.outputs.labels }}
          cache-from: type=gha
          cache-to: type=gha,mode=max
```

关键点：
- `cache-from / cache-to: type=gha`：用 GitHub Actions 缓存加速构建
- `docker/metadata-action`：自动根据分支/tag 生成合理的镜像 tag
- push 到 GHCR 使用自动注入的 `GITHUB_TOKEN`，不用管理秘钥

### 17.2 部署触发

有几种模式：

1. **手动部署**：CI 只负责构建镜像，服务器上手动 `docker compose pull && up -d`
2. **SSH 触发**：workflow 完成后 SSH 到服务器执行命令
3. **Watchtower**：服务器上跑一个容器监控镜像更新，自动拉取新版本
4. **K8s + ArgoCD**：完整的 GitOps，K8s 集群订阅 git 变化自动部署

小团队推荐 1 或 3。

## 18. 常见坑总结

按我见过的频率排序：

1. **忘了 `PYTHONUNBUFFERED=1`**：日志不实时，以为服务挂了
2. **用了 `latest` tag**：某天拉到新镜像行为变了，原因不明
3. **Dockerfile 顺序不对**：每次改代码都要重装依赖
4. **没用 .dockerignore**：`.env` 或 `.git` 进了镜像，镜像膨胀 + 泄密
5. **root 用户运行**：有安全风险，一些云服务商直接拒绝
6. **Alpine + Python**：装 numpy 等库编译半小时还失败
7. **容器内监听 127.0.0.1**：主机访问不到
8. **没配 HEALTHCHECK**：K8s / compose depends_on 等不到真的就绪
9. **把日志写文件**：volume 没挂文件就随容器死掉
10. **忽略信号处理**：`docker stop` 每次都要等 10 秒超时
11. **`docker-compose down -v` 失手删 volume**：数据全没
12. **CI 里 `docker build` 没启用 cache**：每次全量构建，几分钟起步
13. **Nginx 不关 `proxy_buffering`**：SSE/WebSocket 变"一次性吐"
14. **时区全部是 UTC**：日志时间难读，排错头疼
15. **pip install 不指定版本**：环境漂移，复现问题困难

## 20. 面试考点

面试时围绕 Docker 的高频问题（配简要答题思路）：

### Q1：Docker 和虚拟机的区别？

**答题框架**：
- 隔离层次：VM 在硬件虚拟化层，容器在 OS 进程层
- 内核：VM 各自独立内核，容器共享宿主机内核
- 启动速度：VM 秒到分钟，容器秒以内
- 资源开销：VM 每个几 GB 内存，容器几十 MB
- 隔离强度：VM 强（KVM 级别），容器弱（namespace + cgroups，依赖内核）
- 适用场景：VM 跑不同 OS 或强隔离需求；容器跑同一内核下的应用

### Q2：Dockerfile 优化镜像体积有哪些手段？

答：
1. 用 slim / distroless 基础镜像
2. 多阶段构建剥离编译依赖
3. 合并 RUN 指令 + 清理缓存
4. `.dockerignore` 排除无关文件
5. `pip install --no-cache-dir`
6. 删除不必要的文档和 locale 文件

### Q3：`CMD` 和 `ENTRYPOINT` 的区别？

答：
- ENTRYPOINT 定义"总是要跑的程序"，不可被 `docker run 镜像 xxx` 覆盖
- CMD 定义"默认参数或命令"，可以被覆盖
- 常见组合：ENTRYPOINT 指定可执行文件，CMD 给默认参数
- exec 形式 (`CMD ["cmd"]`) vs shell 形式 (`CMD cmd`)：exec 形式直接启动，shell 形式会套 /bin/sh，影响信号传递

### Q4：容器数据怎么持久化？bind mount 和 named volume 怎么选？

答：
- bind mount：挂主机任意路径，适合开发（代码热加载）
- named volume：Docker 管理，适合生产（数据库等）
- tmpfs：内存，适合临时敏感数据
- 生产数据库必须 named volume 或外部存储，绝不 bind mount 到某个主机路径

### Q5：多个容器之间怎么通信？

答：
- 同一自定义 bridge 网络，用容器名作为 hostname 直接访问
- 默认 bridge 网络只能用 IP（不推荐）
- host 模式共享主机网络栈（性能好但失去隔离）
- Docker Compose 自动把同一 compose 项目的服务放同一网络

### Q6：K8s 的 Pod / Deployment / Service 分别是什么？

答：
- Pod：K8s 调度最小单位，可含多个容器（通常 1 主 + 若干 sidecar）
- Deployment：管理 Pod 的副本集，描述期望状态（比如 "始终有 3 个副本"），支持滚动升级
- Service：给一组 Pod 提供稳定访问入口（IP 和 DNS 名）
- Ingress：集群外部访问 Service 的 7 层网关

### Q7：容器化一个 Python 应用时最常踩的坑是什么？

答：（挑 3 个讲）
- `PYTHONUNBUFFERED` 没设，日志不实时
- COPY 顺序不对，缓存全废
- 用了 alpine 导致 pip 编译地狱
- 以 root 用户运行，安全风险
- CMD shell 形式，停止要等 10 秒

### Q8：你们生产环境的 CI/CD 流程是什么样的？

**STAR 答题示例**（结合 OpenAgentic 一类真实项目场景）：
> Situation：我们的 Agent 后端需要快速迭代，每天多次发布
