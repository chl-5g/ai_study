理解设计模式的关键，不在于记住名字，而在于明白 **它到底解决了什么痛苦**。

下面我把 23 种 GoF 设计模式逐一列出，用一句话说清楚它的 **核心意图** 和 **要解决的问题**。

### 一、创建型模式：解决“怎么优雅地造对象”

这 5 种模式专注于将对象的**创建过程**与**使用过程**分离。

| 模式 | 要解决的痛点 |
| :--- | :--- |
| **1. 单例** | **全局唯一性。** 确保一个类只有一个实例，并提供全局访问点。解决“大家都要用同一个配置/连接池”的问题。 |
| **2. 工厂方法** | **推迟具体实例化。** 父类定义创建接口，子类决定具体 `new` 谁。解决“基类不想绑定具体子类”的问题。 |
| **3. 抽象工厂** | **产品族配套。** 创建一系列**相关**或**相互依赖**的对象。解决“Windows 风格按钮配 Linux 风格菜单”这种风格不统一的问题。 |
| **4. 建造者** | **参数爆炸与可选参数。** 分步骤构建一个**复杂对象**。解决构造函数有 20 个参数，且很多参数是可选的痛苦。 |
| **5. 原型** | **复制代价高。** 通过**克隆**已有实例来创建新对象。解决“创建对象需要复杂数据库查询或 IO，不想重复执行”的问题。 |

---

### 二、结构型模式：解决“怎么把类/对象搭积木搭得漂亮”

这 7 种模式关注如何组织类或对象的**结构**，让系统更灵活、更省资源。

| 模式 | 要解决的痛点 |
| :--- | :--- |
| **6. 适配器** | **接口不兼容。** 把“圆头插头”转成“扁头插座”。解决“想用一个老库/第三方库，但它接口跟我现在代码对不上”的问题。 |
| **7. 桥接** | **多维度变化爆炸。** 将**抽象部分**与**实现部分**分离。解决“`红钢笔`、`蓝钢笔`、`红毛笔`、`蓝毛笔`”这种 `M x N` 类膨胀的问题。 |
| **8. 组合** | **部分-整体层级关系。** 让单个对象和组合对象拥有**统一的操作接口**。解决“处理文件夹和文件时，总要写 `if 是文件夹` 这种丑陋判断”。 |
| **9. 装饰器** | **动态添加职责，且不修改原类。** 解决“我想给某些按钮加个边框，给某些输入框加个音效，但我不能去改源码，也不想搞出无数子类”。 |
| **10. 外观** | **子系统太复杂。** 提供一个简单的高层**统一入口**。解决“看个电影要手动开投影、关灯、开音响、选输入源...我只想按一个‘播放’键”。 |
| **11. 享元** | **海量相似对象的内存占用。** 共享**不可变**的公共状态。解决“写一个文字编辑器，100万个 'A' 字符不能真的 `new` 100万个 A 对象吧？” |
| **12. 代理** | **控制访问，而非增加功能。** 用替身挡在真实对象前面。解决“大图片太耗内存先占位显示（虚拟代理）”、“要记录访问日志但不想侵入业务逻辑”。 |

---

### 三、行为型模式：解决“对象之间怎么互相喊话、协作、分配责任”

这 11 种模式关注**算法**和**对象间的责任分配**。

| 模式 | 要解决的痛点 |
| :--- | :--- |
| **13. 责任链** | **请求发送者与接收者解耦。** 让多个对象都有机会处理请求，形成一条链。解决“请假流程 1 天经理批、3 天总监批...我不想在业务代码里写一长串 `if-else`”。 |
| **14. 命令** | **请求的封装与回滚。** 将“动作”封装为对象。解决“编辑器里的 `Ctrl+Z` 撤销功能，必须把执行了什么操作给**记下来**”。 |
| **15. 解释器** | **自定义语法解析。** 定义一种语言文法的表示。解决“要做个简单的 SQL 解析器或正则表达式引擎”（*注：实际开发中用得少*）。 |
| **16. 迭代器** | **遍历数据但不想暴露内部结构。** 提供统一方法顺序访问聚合对象元素。解决“有的集合是数组，有的是树，有的是哈希表，我不想记三种遍历写法”。 |
| **17. 中介者** | **网状依赖变星状。** 用一个中介对象封装一系列对象交互。解决“A 改了要通知 B、C、D，B 改了通知 A、E...类之间互相耦合得像一团意大利面”。 |
| **18. 备忘录** | **状态快照与恢复。** 捕获对象内部状态并在外部保存。解决“游戏存档：既要保存角色血量位置，又**不能让外部代码随便修改存档数据**”。 |
| **19. 观察者** | **一对多的状态同步。** 一个对象改变，所有依赖它的对象自动更新。解决“股票行情变了，所有打开的 K 线图窗口都得跟着变”。 |
| **20. 状态** | **复杂的状态机 `if-else`。** 允许对象内部状态改变时改变行为。解决“订单有 10 种状态，每种状态下对‘取消’‘发货’的响应不同，代码里全是 `switch-case`”。 |
| **21. 策略** | **算法可以随时替换。** 定义算法族并封装，让它们可以互相替换。解决“双 11 用 A 打折算法，618 用 B 打折算法，日常用 C 算法。我不想到处改核心计算逻辑”。 |
| **22. 模板方法** | **流程骨架固定，细节可变。** 父类定义流程，子类实现具体步骤。解决“泡茶流程：烧水、冲泡、倒入杯子、加料。红茶加柠檬，绿茶加蜂蜜，流程一样，就是最后一步不同”。 |
| **23. 访问者** | **数据结构稳定但算法多变。** 在不修改类的前提下定义新操作。解决“写好了 `Car` 类包含 `Engine`、`Wheel`，现在想加个‘打印零件清单’功能，又想加个‘检查环保标准’功能，不想去改 `Engine` 类代码”。 |

### 总结：为什么要学这些套路？

所有的设计模式，最终都是为了对抗 **软件设计中的三大根本问题**：

1. **可维护性差** —— 牵一发而动全身（**解决：解耦**）。
2. **可扩展性差** —— 加个功能要把以前写的类翻个底朝天（**解决：开闭原则**）。
3. **复用性差** —— 这段代码抠不出来，只能在当前项目里用（**解决：高内聚低耦合**）。
