## 1. 从串行到并发：为什么需要并发

### 1.1 一个让你痛苦的场景

想象你写了一段代码：Agent 收到用户的一句话"帮我查一下北京、上海、深圳今天的天气"，于是你的后端要去调 3 次天气 API，把结果汇总后喂给 LLM 总结。

最朴素的写法是这样：

```python
import requests, time

def get_weather(city):
    # 假设每次请求需要 1 秒（网络往返 + 对方服务器处理）
    resp = requests.get(f"https://api.example.com/weather?city={city}")
    return resp.json()

start = time.time()
beijing  = get_weather("beijing")   # 等 1 秒
shanghai = get_weather("shanghai")  # 再等 1 秒
shenzhen = get_weather("shenzhen")  # 再等 1 秒
print("总耗时：", time.time() - start)  # 约 3 秒
```

这就叫**串行执行（serial execution）**：一件事做完再做下一件。3 个请求加起来 3 秒。用户在屏幕前干等 3 秒，体验非常糟糕。

但仔细想想：这 3 秒里你的 CPU 其实几乎什么都没做。它发出一个 HTTP 请求后就一直在等对方服务器回包，占用 CPU 不到 1%。这就好比你去三家餐厅点外卖，你先去第一家下单，然后**站在第一家门口什么也不干地等它做好**，拿到后再去第二家下单……一共等 3 顿饭的时间。

显然更聪明的做法是：**三家店同时下单，然后边等边干别的事**。三家店的等待时间可以重叠，总时间只需要最慢的那一家 + 少量下单时间，大约 1 秒多。

这就是**并发（concurrency）**的核心直觉——**把"等待"重叠起来**。

### 1.2 什么是"等待"：CPU 密集 vs IO 密集

要理解并发能帮到哪类任务，先要区分任务的两种性质：

**IO 密集（I/O-bound）任务**：大部分时间在等外部资源。典型例子：
- 发 HTTP 请求，等对方响应
- 读数据库，等磁盘 / 网络返回结果
- 读写文件，等磁盘机械臂移动或 SSD 响应
- 等用户输入

这类任务的特点是：**CPU 几乎没事干，都在等**。并发可以显著加速。

**CPU 密集（CPU-bound）任务**：大部分时间 CPU 在疯狂计算。典型例子：
- 加密 / 解密
- 图像处理、视频编码
- 矩阵运算、深度学习推理
- 压缩 / 解压

这类任务的特点是：**CPU 已经 100% 忙着了，再怎么"并发"单核也快不起来，除非真正用多个核一起算**。

这两种任务的区别非常重要，因为后面你会看到：**Python 的多线程对 IO 密集任务有效，对 CPU 密集任务几乎无效**（因为 GIL）。选错工具会让你白忙一场。

### 1.3 一句话总结这一节

- 串行执行的浪费点在于"CPU 明明闲着却在空等"
- 并发的本质是把"等待"重叠起来
- 能从并发里获益最多的是 IO 密集任务，比如 AI Agent 里常见的 HTTP 调用 / 数据库查询 / 文件读写
- CPU 密集任务需要的是"真并行"（多核一起算），这是另一个故事

## 2. 并发、并行、异步：三个容易混淆的概念

### 2.1 三个词的字面区别

这三个词中文翻译很近，很多教程也混着用，但它们指的是不同层次的东西。我们用"餐厅服务员"的类比一次讲清楚。

**并行（Parallelism）**：**同一时刻**，多个任务在**物理上**同时进行。
- 类比：一家餐厅有 4 个服务员，每个桌子一个，4 桌客人同时被服务。
- 对应硬件：多核 CPU，每个核在同一时刻真的在跑不同代码。
- 前提：你得有多个"执行者"（多核或多机器）。

**并发（Concurrency）**：**一段时间内**，多个任务**在宏观上都在进行**，但微观上可能轮流占用同一个执行者。
- 类比：一家餐厅只有 1 个服务员，但他同时照看 4 桌——这桌点个菜，那桌送杯水，另一桌结个账，每桌都感觉"服务员一直在忙我这桌"。
- 核心：**任务之间的"等待"被交错重叠了**。服务员在等第一桌客人决定点什么菜的时候，就跑去第二桌送水。
- 前提：任务之间有"等待时间"可以利用。

**异步（Asynchronous）**：一种**编程模型 / 编程风格**。它提供一种写法，让你可以把"等待"显式地交还给调度器，不要阻塞后续代码。
- 异步是"怎么写"的问题，不是"怎么跑"的问题。
- 单线程可以写异步代码（asyncio 就是这样）；多线程也可以写异步代码。
- 异步通常用于**实现并发**，但它不等于并发，更不等于并行。

一句精炼的对比：

> **并发是一个目标（多件事一起进行），并行是实现并发的一种方式（靠多个核），异步是实现并发的另一种方式（靠协作式调度）。**

### 2.2 用具体代码感受三种模型

假设任务是"下载 3 个文件"。

**串行**：

```python
download(file1)   # 等 5 秒
download(file2)   # 等 5 秒
download(file3)   # 等 5 秒
# 总计 15 秒
```

**并行（多进程或多线程）**：

```python
# 创建 3 个线程/进程同时下载
t1 = Thread(target=download, args=(file1,))
t2 = Thread(target=download, args=(file2,))
t3 = Thread(target=download, args=(file3,))
t1.start(); t2.start(); t3.start()
t1.join();  t2.join();  t3.join()
# 总计约 5 秒
```

**异步（单线程 asyncio）**：

```python
async def main():
    await asyncio.gather(
        download_async(file1),
        download_async(file2),
        download_async(file3),
    )
asyncio.run(main())
# 总计约 5 秒
```

第二种和第三种都实现了"并发"——3 个文件**在宏观上一起下载**。但实现机制完全不同：
- 多线程靠**操作系统的线程调度**：OS 认为这 3 个线程都在"等网络"，于是把 CPU 让给其他线程。
- asyncio 单线程：**整个程序只有一个线程**。靠一种叫"事件循环"的机制，在某个协程"等网络"的时候，主动切到另一个协程去跑。

后者听起来有点神奇，没关系，后面第 8 节我们会用大量篇幅讲清楚事件循环到底在干什么。

### 2.3 什么时候选哪种？

这是个高频面试题，也是写代码时最容易犯错的决策点。先给一个**速查表**，后面会反复印证：

| 任务性质 | 推荐工具 | 理由 |
|---|---|---|
| IO 密集，任务数少（< 几十） | `threading` 或 `concurrent.futures.ThreadPoolExecutor` | 写法简单，无需重构代码 |
| IO 密集，任务数多（几百到几万） | `asyncio` | 单线程维持几万个连接远比几万线程省资源 |
| IO 密集，整个项目本来就是异步框架（FastAPI 等） | `asyncio` | 风格统一，不要混用 |
| CPU 密集 | `multiprocessing` 或 `ProcessPoolExecutor` | 绕开 GIL，真正用满多核 |
| CPU 密集且计算量极大 | C 扩展 / Numpy / Cython / Rust | Python 本身不适合干脏活 |

### 初学者常见误区

- **误区 1**：以为异步 = 多线程。错。asyncio 默认是**单线程**的，只不过这个线程里塞了一个事件循环在调度很多个协程。
- **误区 2**：以为多线程 = 并行。错（在 CPython 里）。因为 GIL 的存在，CPython 的多线程在任何时刻只有一个线程真正在执行 Python 字节码，多线程更像是"并发"而不是"并行"。
- **误区 3**：以为 asyncio 能加速 CPU 密集任务。错。asyncio 是单线程的，CPU 密集任务在一个协程里跑完之前，其他协程都卡着。用 asyncio 跑大矩阵乘法比串行还慢（因为还多一层事件循环开销）。
- **误区 4**：以为既然异步好，就所有代码都该写成异步。错。异步代码有复杂度成本，且只对 IO 密集有价值。单机脚本、离线数据处理用同步代码更简单。

## 3. Python GIL 深度解析

GIL（Global Interpreter Lock，全局解释器锁）是 Python 并发领域最有名的"坑"。你一定在哪听过"Python 多线程没用因为有 GIL"，这种说法对了一半，另一半是错的。要写对并发代码，你必须真正理解 GIL 是什么、存在的原因、以及在什么场景下它是瓶颈，什么场景下它根本不影响你。

### 3.1 GIL 是什么

一句话定义：

> **GIL 是 CPython 解释器内部的一把互斥锁。在任意时刻，一个 Python 进程内只有持有 GIL 的那一个线程能执行 Python 字节码。**

注意关键词：
- **CPython 解释器内部**：GIL 是 CPython 这个具体实现的特性，不是 Python 语言本身的规定。Jython（JVM 上的 Python）、IronPython（.NET 上的 Python）都没有 GIL。但 99.99% 的人用的就是 CPython。
- **一个进程内**：GIL 是进程级的锁。开两个进程，就有两把独立的 GIL，所以多进程可以真正用满多核。
- **Python 字节码**：只有解释器执行字节码的时候需要持有 GIL。调 C 扩展（比如 numpy 的底层运算）时 C 代码可以主动释放 GIL，让其他线程跑。
- **只有一个线程能执行**：这是多线程不能加速 CPU 密集的根本原因。

### 3.2 为什么要有 GIL：历史原因

这不是某个设计师脑子一热加的，而是历史选择。

CPython 用**引用计数（reference counting）** 管理内存。每个对象有个 `ob_refcnt` 字段，被引用一次就 +1，解引用就 -1，归零就销毁。这个机制简单高效。

但引用计数的增减**不是原子操作**：`ob_refcnt += 1` 实际上是"读值-加一-写回"三步。两个线程同时给一个对象加引用，可能会导致计数错误，要么提前释放内存（导致段错误），要么永远不释放（内存泄漏）。

要修复这个问题有两条路：

1. **给每个对象加一把细粒度锁**——但 Python 里一切皆对象，这意味着随便一个 `a = b` 都要加锁解锁，性能暴跌。
2. **用一把全局大锁，直接规定任何时刻只有一个线程能跑 Python 代码**——性能损失集中在"想用多线程加速 CPU 密集"这一个场景，其他场景几乎没影响。

CPython 选了第二条——这就是 GIL。在 1992 年 Guido 设计 Python 的时代，多核 CPU 还不普及，这个选择非常合理。等到多核时代来了，再想去掉 GIL 已经困难重重（大量 C 扩展依赖 GIL 的存在才能线程安全地被调用）。

### 3.3 GIL 什么时候会被释放

GIL 不是一辈子抱着不放，有几个释放时机：

1. **每 sys.getswitchinterval() 秒（默认 5ms）** 主动释放一次，让其他线程有机会抢。
2. **执行到 IO 操作时主动释放**。典型的：`time.sleep()`、`socket.recv()`、`open().read()`、`requests.get()`。这些操作进入 C 层的系统调用，C 代码会显式 `Py_BEGIN_ALLOW_THREADS` 释放 GIL，然后进入阻塞等待，等 OS 把数据交回来再 `Py_END_ALLOW_THREADS` 重新抢 GIL。
3. **C 扩展代码可以主动释放**。比如 numpy 做一个大矩阵乘法时会释放 GIL，这时其他 Python 线程可以趁机跑。所以"用 numpy 写 CPU 密集任务 + 多线程"是有效的！这是一个很多人不知道的技巧。

### 3.4 GIL 对 CPU 密集的影响：实测

来看一个对比。先是串行：

```python
def count(n):
    while n > 0:
        n -= 1

start = time.time()
count(10_000_000)
count(10_000_000)
print(time.time() - start)  # 假设 1.0 秒
```

再用两个线程并行跑：

```python
t1 = Thread(target=count, args=(10_000_000,))
t2 = Thread(target=count, args=(10_000_000,))
t1.start(); t2.start(); t1.join(); t2.join()
print(time.time() - start)  # 约 1.0 秒，甚至 1.1 秒（GIL 切换有开销）
```

**多线程没有加速，甚至更慢**。因为两个线程在抢同一把 GIL，等效还是一个线程在跑，外加锁切换开销。这就是"Python 多线程对 CPU 密集无效"的来源。

用多进程呢：

```python
from multiprocessing import Process
p1 = Process(target=count, args=(10_000_000,))
p2 = Process(target=count, args=(10_000_000,))
p1.start(); p2.start(); p1.join(); p2.join()
# 约 0.5 秒，真正用上了 2 个核
```

**多进程真的加速了**，因为每个进程有独立的 GIL。

详细的实测代码在 `demos/01_gil_bench.py`，强烈建议你亲自跑一遍。

### 3.5 GIL 对 IO 密集的影响：几乎没有

IO 密集时情况就大反转：

```python
def io_task():
    time.sleep(1)  # 模拟等网络

# 串行
t = time.time()
io_task(); io_task(); io_task()
print(time.time() - t)  # 约 3 秒

# 多线程
ts = [Thread(target=io_task) for _ in range(3)]
for t in ts: t.start()
for t in ts: t.join()
# 约 1 秒！真的加速了
```

为什么？因为 `time.sleep(1)` 在底层实现里会**主动释放 GIL** 后再阻塞。所以线程 1 刚 sleep 就把 GIL 放了，线程 2 马上拿到 GIL、也进入 sleep、也放 GIL，线程 3 同理。三个线程在 1 秒后几乎同时醒来。

所以一句话记住：**GIL 坑的是 CPU 密集，不坑 IO 密集**。AI Agent 后端里绝大部分任务是 IO 密集（调 LLM API、调数据库、调工具），所以我们后面主力用 asyncio（或多线程），很少用多进程。

### 3.6 为什么不去掉 GIL

社区一直在讨论。2023 年 PEP 703 被正式接受，Python 3.13 开始提供了**可选的 free-threaded 模式**（编译时开关 `--disable-gil`），允许真正的多线程并行。但：
- 目前还在实验阶段，大量 C 扩展还没适配
- 单线程性能会下降一点（因为引用计数要加细粒度锁）
- 生产环境可能要等到 3.14 / 3.15 之后才真正稳定

在你真正需要写 Python 并发代码的今天（2026 年），还是要默认按"有 GIL"来思考。

### 初学者常见误区

- **误区 1**：GIL 是 Python 的锁。错。是 **CPython** 这个具体实现的锁。
- **误区 2**：Python 多线程没用。错。对 IO 密集非常有用。
- **误区 3**：numpy / pytorch 这类库也会被 GIL 卡住。不一定。这些库的核心运算在 C 层会释放 GIL，所以"多线程 + numpy"可以用上多核。
- **误区 4**：GIL 会让多线程程序出错。错。GIL 反而是在"防止"出错（至少引用计数不会错）。但 GIL 不保护业务数据，业务级的竞态条件你自己得管（后面 Lock 一节讲）。

## 4. 多线程 threading

### 4.1 线程是什么

先补一点操作系统知识，很多同学卡在这里。

**进程（Process）**：一个独立运行的程序实例。有自己的内存地址空间、打开的文件、资源。进程之间完全隔离——A 进程不能直接读 B 进程的变量。

**线程（Thread）**：进程内的一条执行流。同一个进程内的多个线程**共享内存空间**，所以线程之间可以直接读写同一个变量。线程比进程轻——创建和切换成本更低。

一个比喻：
- 进程 = 一栋独立的办公楼
- 线程 = 同一栋楼里的多个员工，共享办公桌、打印机、资料柜

Python 里线程就是操作系统原生线程的薄封装（在 CPython 里叫 `PyThreadState`）。你 `Thread(target=func).start()` 的时候，真的会创建一个 OS 级别的线程。

### 4.2 Thread 基本用法

```python
import threading
import time

def worker(name, seconds):
    print(f"{name} 开始工作")
    time.sleep(seconds)          # 模拟耗时 IO
    print(f"{name} 完成")

# 创建两个线程
t1 = threading.Thread(target=worker, args=("A", 2))
t2 = threading.Thread(target=worker, args=("B", 3))

t1.start()   # 启动线程 A
t2.start()   # 启动线程 B
# 此时主线程继续往下跑，不会等 t1/t2

t1.join()    # 主线程在这里等 A 结束
t2.join()    # 再等 B 结束
print("全部完成")
```

几个关键 API：
- `Thread(target=f, args=(...), kwargs={...}, daemon=False)` 创建线程对象。
- `.start()` 启动线程。注意：一个 Thread 对象只能 start 一次。
- `.join(timeout=None)` 阻塞当前线程，直到目标线程结束（或超时）。
- `daemon=True` 设置为守护线程。**守护线程会在主线程退出时被强制杀掉**；非守护线程则会拖着主进程不退出。规则记住：**长期后台任务设为 daemon，关键业务不要设为 daemon（因为被杀的时候数据可能没保存）**。

### 4.3 竞态条件（Race Condition）

多线程最头疼的问题：**多个线程同时读写同一个变量时，结果不可预测**。来看个经典例子：

```python
counter = 0

def inc():
    global counter
    for _ in range(100000):
        counter += 1   # 看起来是一步，其实是三步

ts = [threading.Thread(target=inc) for _ in range(10)]
for t in ts: t.start()
for t in ts: t.join()

print(counter)  # 预期 1_000_000，实际可能是 900000 多，每次还不一样
```

为什么会错？因为 `counter += 1` 在字节码层面是 3 步：
1. 读出 `counter` 当前值到寄存器
2. 寄存器 +1
3. 写回 `counter`

假设两个线程都读到 counter=100，各自 +1 后写回 101。结果本该是 102，却变成了 101。**一次自增丢失了**。GIL 保护了"每条字节码是原子的"，但不保护"一段代码整体是原子的"。

这就叫**竞态条件**——结果依赖于线程调度顺序，调度一变结果就变，bug 难以复现。

### 4.4 Lock：互斥锁

解决办法：用锁把"读-改-写"这段代码**强制串行化**。

```python
lock = threading.Lock()
counter = 0

def inc():
    global counter
    for _ in range(100000):
        with lock:            # 获取锁，其他线程等
            counter += 1
        # 出了 with 块自动释放锁

ts = [threading.Thread(target=inc) for _ in range(10)]
for t in ts: t.start()
for t in ts: t.join()
print(counter)   # 稳定输出 1_000_000
```

`Lock` 的三个要点：
1. 同一时刻只有一个线程能拿到锁。
2. 拿不到锁的线程会**阻塞**等待。
3. 锁一定要配合 `try/finally` 或 `with` 用，否则异常时锁不释放，其他线程会死等。

另外还有两个值得知道的锁：

**RLock（Reentrant Lock，可重入锁）**：同一个线程可以多次 `acquire`，其他线程还是进不来。适合递归函数或一个持锁方法调用另一个持锁方法的场景。用普通 `Lock` 这种情况会自己把自己锁死。

```python
rlock = threading.RLock()

def outer():
    with rlock:
        inner()   # 这里如果是普通 Lock 会死锁

def inner():
    with rlock:
        print("hi")
```

**Condition（条件变量）**：更高级的同步原语。核心场景是"一个线程要等某个条件满足后才继续"，比如生产者消费者模型里消费者要等队列不空。

```python
cv = threading.Condition()
items = []

def consumer():
    with cv:
        while not items:
            cv.wait()        # 释放锁并等待通知
        item = items.pop()

def producer():
    with cv:
        items.append("x")
        cv.notify()          # 唤醒一个等待者
```

`cv.wait()` 做了三件事：释放锁、阻塞等待通知、被唤醒后重新抢锁。手写这个模式容易错，实际开发更常用 `queue.Queue`（下面会讲）。

### 4.5 Event：一次性开关

`Event` 最简单：就是一个 bool flag，加上"等它变成 True"的能力。

```python
stop_event = threading.Event()

def worker():
    while not stop_event.is_set():
        do_something()
        stop_event.wait(timeout=1)    # 每秒检查一次，也支持立即被唤醒

# 主线程想停的时候
stop_event.set()
```

常用于"优雅停止后台线程"的场景。

### 4.6 queue.Queue：线程安全队列

这是最常用、也最该优先使用的东西。标准库 `queue.Queue` 是**线程安全**的，内部已经帮你处理好了锁和 Condition，你只管往里 `put` 和 `get`。

```python
from queue import Queue

q = Queue(maxsize=100)

def producer():
    for i in range(1000):
        q.put(i)           # 队列满了会阻塞
    q.put(None)            # 哨兵：告知消费者结束

def consumer():
    while True:
        item = q.get()
        if item is None:
            q.put(None)    # 传给下一个消费者
            break
        process(item)
        q.task_done()       # 标记一个任务完成
```

想实现生产者消费者，优先用 `Queue`，不要自己 Lock+list+Condition 手搓。

### 4.7 多线程在 AI Agent 后端里的真实场景

什么时候会手写 `Thread` 或 `Lock`？在现代 Python 后端里，直接用 `threading` 的场景其实不多，主要是：

1. **FastAPI 同步视图函数里**：如果你用 `def` 而不是 `async def`，FastAPI 会把这个视图放到一个线程池里跑。你自己很少直接创建 Thread，但理解多线程能帮你避免全局变量的坑。
2. **定时任务 / 监控**：比如后台起一个线程定期刷新某个缓存。
3. **包装某些只提供同步 API 的 SDK**：下面第 15 节会讲，asyncio 里想跑阻塞 SDK 就得 `run_in_executor` 到线程池。
4. **Streamlit / 桌面应用**：为了不卡 UI 得把长任务放线程里。

### 初学者常见误区

- **误区 1**：所有共享变量都要加锁。只有**会被多个线程同时写**（或一写多读）的才需要。只读的不用。
- **误区 2**：锁越大越安全。错。锁范围越大，并发度越低。应该尽量缩小锁住的代码。
- **误区 3**：用 `print` 调试多线程代码。`print` 本身也不是完全线程安全的（`write` 和 `flush` 是两步），输出会交织。用 `logging` 更安全。
- **误区 4**：`time.sleep` 不会释放锁。确实不会释放 Lock，但会释放 GIL。容易混淆。

## 5. 多进程 multiprocessing

### 5.1 为什么需要多进程

前面说过：多线程绕不开 GIL，CPU 密集任务用多线程白费。如果你要做：
- 视频转码
- 大批量图片处理
- 离线模型推理（不在 C 层释放 GIL 的场景）
- 纯 Python 写的计算密集任务

那就必须用**多进程**。每个子进程有独立的 Python 解释器和独立的 GIL，互不干扰，能真正用满多核。

代价是：进程之间**不共享内存**，要传数据必须"序列化 -> 进程间通信 -> 反序列化"。对于大对象（比如几 GB 的 DataFrame），这个开销不容忽视。

### 5.2 Process 基本用法

API 几乎和 threading 一模一样：

```python
from multiprocessing import Process
import os

def worker(name):
    print(f"{name} 在进程 {os.getpid()} 中运行")

if __name__ == "__main__":     # 重要！必须有 main guard
    p = Process(target=worker, args=("A",))
    p.start()
    p.join()
```

**重点**：`multiprocessing` 在 Windows / macOS 下默认用 `spawn` 启动方式，子进程会**重新 import 你的脚本**。如果没有 `if __name__ == "__main__":` 保护，子进程会再次执行创建子进程的代码，导致无限递归。Linux 下默认 `fork` 启动没这个问题，但跨平台写法必须加 main guard。

### 5.3 序列化开销：pickle

子进程和父进程要交换数据，中间必须**序列化**成字节流发过去，然后反序列化回来。Python 用的是 `pickle`。

不是所有对象都能 pickle：
- lambda 和嵌套函数不行
- 打开的文件句柄不行
- 数据库连接、socket 不行
- 线程锁不行

所以你传给 Process 的 `target` 必须是**模块顶层定义的函数**，`args` 里的对象必须都能 pickle。

还要记住：**进程间传大数据非常贵**。传一个 1GB 的 numpy array 到子进程，光是序列化就几秒。对于这种情况要用 `shared_memory` 或 `multiprocessing.Array`（下面讲）。

### 5.4 Pool：进程池

手动创建 Process 麻烦，实际更常用 `Pool`：

```python
from multiprocessing import Pool

def square(x):
    return x * x

if __name__ == "__main__":
    with Pool(processes=4) as pool:
        # map：阻塞，返回结果列表
        result = pool.map(square, range(10))
        print(result)   # [0, 1, 4, 9, ..., 81]

        # imap：返回迭代器，边算边拿
        for r in pool.imap(square, range(10)):
            print(r)

        # apply_async：异步提交单个任务
        future = pool.apply_async(square, (5,))
        print(future.get())  # 25
```

Pool 默认进程数 = CPU 核数，够用不浪费。

### 5.5 Queue 和 Pipe：进程间通信

多个进程要通信，不能直接访问对方变量，必须用专门的通道。

`multiprocessing.Queue`（注意和 `queue.Queue` 不是一回事！）：

```python
from multiprocessing import Process, Queue

def producer(q):
    for i in range(5):
        q.put(i)

def consumer(q):
    while True:
        item = q.get()
        if item is None:
            break
        print(item)

if __name__ == "__main__":
    q = Queue()
    p1 = Process(target=producer, args=(q,))
    p2 = Process(target=consumer, args=(q,))
    p1.start(); p2.start()
    p1.join()
    q.put(None)
    p2.join()
```

`multiprocessing.Queue` 底层其实就是一个管道 + 锁，数据要 pickle。比 `queue.Queue` 慢得多，但跨进程必须用它。

`Pipe` 更原始，提供一对 `Connection`，两端可以互相 send/recv。一对一通信场景用 Pipe，多对多用 Queue。

### 5.6 共享内存：传大数据的正确姿势

如果你要传一个几百 MB 的数组，每次 pickle 都让你崩溃。`multiprocessing.shared_memory`（Python 3.8+）可以让多个进程访问同一块物理内存：

```python
from multiprocessing import shared_memory
import numpy as np

# 父进程创建
shm = shared_memory.SharedMemory(create=True, size=1024*1024*100)  # 100MB
arr = np.ndarray((100*1024*1024//8,), dtype=np.float64, buffer=shm.buf)
arr[:] = np.random.random(arr.shape)

# 子进程通过名字连接同一块
def worker(name):
    shm2 = shared_memory.SharedMemory(name=name)
    arr2 = np.ndarray((100*1024*1024//8,), dtype=np.float64, buffer=shm2.buf)
    print(arr2.mean())
    shm2.close()

p = Process(target=worker, args=(shm.name,))
p.start(); p.join()

shm.close()
shm.unlink()   # 释放物理内存
```

这种方式零拷贝，适合传大数组。代价是你要自己管理内存生命周期。

### 5.7 多进程在 AI Agent 里的使用场景

坦白说：AI Agent 后端服务里，**直接用 multiprocessing 的场景很少**。因为后端的工作几乎都是 IO 密集（调 LLM、查数据库、调工具）。多进程主要出现在：

1. **后端服务的 Worker 模型**：比如 `gunicorn -w 4` 起 4 个 worker 进程跑 FastAPI，这其实就是多进程。但你不需要自己写 `multiprocessing`，gunicorn 帮你管。
2. **离线数据处理脚本**：比如把 1000 万条对话日志做情感分析预处理。
3. **本地 embedding 批量计算**：如果你用 sentence-transformers 本地跑 embedding，而 GIL 没被释放，就需要多进程加速。

整体来说，本章 multiprocessing 的部分"知道有这个东西、会基本写法"即可，不必过深。

### 初学者常见误区

- **误区 1**：忘记 `if __name__ == "__main__":`。Windows/macOS 下会无限递归创建子进程，瞬间把机器撑爆。
- **误区 2**：以为多进程一定比多线程快。启动子进程的开销、进程间通信的序列化开销都很大，小任务用多进程反而更慢。
- **误区 3**：把一个打开的数据库连接传给子进程。连接对象不能 pickle，就算能 pickle，跨进程用一个 TCP socket 也会出错。正确做法：每个子进程自己新建连接。
- **误区 4**：用 `multiprocessing.Queue` 替代 `queue.Queue`。前者慢几十倍，线程间通信不要用它。

## 6. 线程池与进程池：concurrent.futures

### 6.1 为什么需要"池"

回看前面的写法：`Thread(target=f).start()`，每来一个任务就创建一个线程，用完就销毁。问题：
- 线程创建本身有开销（OS 级的系统调用）
- 线程数失控：来 10 万个任务就起 10 万个线程，OS 会崩
- 没有结果收集机制，要自己写共享变量

解法：**池化（pooling）**。预先创建一组 worker，任务排队等空闲 worker 来处理。

### 6.2 concurrent.futures：统一的 API

`concurrent.futures` 是 Python 3.2 引入的标准库，提供了**线程池和进程池的统一接口**。两者 API 几乎一模一样，你只要切换类名就能从线程池换成进程池。

```python
from concurrent.futures import ThreadPoolExecutor, ProcessPoolExecutor, as_completed

def fetch(url):
    import requests
    return requests.get(url).status_code

urls = ["https://example.com"] * 10

# 线程池版本（适合 IO 密集）
with ThreadPoolExecutor(max_workers=5) as pool:
    results = list(pool.map(fetch, urls))   # 保持输入顺序

# 完全相同的代码切换成进程池（适合 CPU 密集）
with ProcessPoolExecutor(max_workers=4) as pool:
    results = list(pool.map(fetch, urls))
```

### 6.3 submit vs map

两种提交任务的方式：

**`pool.map(func, iterable)`**：和内建 `map` 一样，但每个元素并发跑。
- 返回结果**按输入顺序**
- 遇到异常会在取结果时抛
- 简单场景用这个

**`pool.submit(func, *args)`**：提交单个任务，返回一个 **Future 对象**。
- Future 是"一个尚未完成的结果"的占位符
- 可以 `.result()` 阻塞等结果，`.done()` 查询是否完成，`.cancel()` 尝试取消
- 灵活度更高

```python
with ThreadPoolExecutor(max_workers=5) as pool:
    futures = [pool.submit(fetch, url) for url in urls]

    # 方式一：按提交顺序收结果
    for f in futures:
        print(f.result())

    # 方式二：谁先完成先处理谁（更常用）
    for f in as_completed(futures):
        print(f.result())
```

`as_completed` 是非常常用的函数——它返回一个**迭代器**，谁先完成就谁先 yield 出来。当任务耗时差异很大时，用这个比 map 体验好很多：快的先出结果不用等慢的。

### 6.4 异常处理

`future.result()` 会把子任务里的异常**重新抛到调用方**：

```python
def bad():
    raise ValueError("oops")

with ThreadPoolExecutor() as pool:
    f = pool.submit(bad)
    try:
        f.result()
    except ValueError as e:
        print("捕获到：", e)
```

如果你忘了 `.result()`，异常就被吞了——future 里静静地放着一个错误，你啥都看不到。**一定要显式取结果或至少调用 `.exception()` 检查**。

### 6.5 线程池 vs 进程池：怎么选

前面已经讲过规则，这里再强调一次：

| 任务 | 选哪个 |
|---|---|
| `requests` 调 HTTP | ThreadPoolExecutor |
| `psycopg2` 查 PG | ThreadPoolExecutor |
| `open().read()` 读文件 | ThreadPoolExecutor |
| `numpy.matmul` 大矩阵 | ThreadPoolExecutor（numpy 底层释放 GIL） |
| 纯 Python 写的哈希 / 加密 | ProcessPoolExecutor |
| PIL 图像处理 | ProcessPoolExecutor |
| 编译 / 压缩 | ProcessPoolExecutor |

默认选线程池。只有在"明确证实线程池没加速"并且"任务是 CPU 密集"时才切换到进程池。

### 6.6 max_workers 怎么定

- **线程池**：IO 密集可以开到几十甚至几百，因为线程大部分时间在等。实际值要测（常见范围 10-100）。
- **进程池**：建议 = CPU 核数 或稍大。开再多也没意义，反而增加切换开销。

## 7. 协程的演进史：从 generator 到 async/await

接下来进入本章的真正主角——asyncio。在讲 asyncio 之前，我们必须先理解"协程"这个词到底指什么，以及 Python 里协程是怎么一步步演变到今天 `async/await` 这个样子的。知道历史不是考古，是帮你看懂各种现存代码和文档，避免困惑。

### 7.1 什么是协程（coroutine）

**协程**是一种可以**暂停和恢复**的函数。它长得像普通函数，但跑到一半可以说"我先歇一下，把控制权还给调用者"，稍后又可以从暂停的位置继续。

对比普通函数：
- 普通函数：一旦 call，就要执行到 return 才把控制权还回来。
- 协程：可以在中间多次"交出控制权"，还能多次"被恢复"。

这种能力让单线程也能实现"多任务交错"——协程 A 跑到一半去等 IO，暂停自己，把控制权交给调度器；调度器看 B 有活可干，跑 B；B 也等 IO 时再切回 A。**不需要多线程就能模拟并发**。

### 7.2 第一代：基于 generator 的协程（Python 2.5+）

Python 2.5 引入了 generator（生成器），用 `yield` 暂停。最早大家发现：generator 不就是能暂停的函数吗？用它来模拟协程！

```python
def simple_coroutine():
    print("开始")
    x = yield              # 暂停，等 send
    print("收到:", x)
    y = yield
    print("收到:", y)

c = simple_coroutine()
next(c)         # 启动到第一个 yield
c.send("hello") # 恢复，x = "hello"
c.send("world") # 恢复，y = "world"
```

这可以跑。但写起来很别扭，`yield` 原本是为生成器设计的，借用它做协程需要一堆约定。

### 7.3 第二代：yield from（Python 3.3）

Python 3.3 引入 `yield from`，允许一个 generator 把控制权委托给另一个 generator：

```python
def inner():
    yield 1
    yield 2
    return "done"

def outer():
    result = yield from inner()
    print(result)   # "done"
```

这让"协程调用协程"变得可能，于是有人基于此写出了 `asyncio` 的早期版本（Python 3.4）。但写法依然很绕：

```python
@asyncio.coroutine
def fetch():
    data = yield from read_from_network()   # 等待另一个协程
    return data
```

这个写法既要装饰器又要 `yield from`，对初学者非常不友好，而且容易和普通生成器混。

### 7.4 第三代：async / await（Python 3.5+）

Python 3.5（2015 年）引入了专门的关键字 `async def` 定义协程，`await` 暂停并等待。这是**我们今天写的所有 asyncio 代码的标准语法**。

```python
async def fetch():
    data = await read_from_network()    # 暂停自己，等 read_from_network 完成
    return data
```

看着像同步代码，但行为完全不同：
- `async def` 定义的函数**不是一个普通函数**。调用它不会执行函数体，而是返回一个 **coroutine 对象**。
- 只有把 coroutine 对象交给 **事件循环**（asyncio.run / await / create_task），它才会真的跑。
- `await expr` 会"暂停当前协程，等 expr 完成，然后用 expr 的结果继续"。

写一个对比你就懂：

```python
def sync_hello():
    print("hi")
    return 1

async def async_hello():
    print("hi")
    return 1

sync_hello()    # 直接执行，打印 hi，返回 1
async_hello()   # 什么都不发生！返回一个 <coroutine object> 对象
```

很多新手踩的第一坑就是：调用了 async 函数却没 await 它，于是什么都没发生，还以为是 bug。

### 7.5 为什么这个演进史对你有用

因为现实中你会看到三种代码混在一起：
- 古董代码用 `yield from` + `@asyncio.coroutine`
- 稍新的代码用 `async def` + `await`
- 还有一些库内部依然用 generator 实现协程

当你看到 `@asyncio.coroutine` 或 `yield from` 时，它们**等价于** `async def` 和 `await`。知道这点你就不会困惑，且能维护老代码。

## 8. asyncio 事件循环原理

这一节是本章的理解核心，请耐心读。

### 8.1 事件循环：一个永不打烊的服务员

再用餐厅的类比，这次仔细一点。

想象一个只有 1 个服务员的餐厅（= 1 个线程），要同时服务 100 桌客人（= 100 个协程）。怎么做？

服务员有一个小本子，上面记着：
- 第 1 桌：菜上齐了，现在在吃，**大概 10 分钟后来收盘子**
- 第 2 桌：刚点完菜，**厨房还在做，菜好了通知我**
- 第 3 桌：要结账，**需要立刻处理**
- 第 4 桌：让我推荐菜，**需要立刻处理**
...

服务员的工作流程：
1. 看小本子，找到"立刻就能做的事"（第 3 桌结账、第 4 桌推荐），依次处理
2. 处理完后再看本子，继续找"立刻就能做的事"
3. 如果一个能做的都没有（所有桌都在吃 / 等菜），服务员就**站着等**，同时盯着"谁的菜好了""谁吃完了"
4. 一有新事件（厨房铃响、客人招手），服务员立刻反应，加到本子上，循环 1-3

这个"服务员 + 小本子 + 不停循环"就是 **事件循环（event loop）**。

对应到 Python：
- 服务员 = 单个线程
- 小本子 = 就绪队列 + 等待 IO 的协程集合
- 立刻能做的事 = 协程里"不需要 await 外部资源"的那段纯 Python 代码
- "菜好了通知我" = OS 层面的 IO 多路复用（epoll / kqueue / IOCP）

### 8.2 事件循环究竟在循环什么

伪代码：

```python
while True:
    # 1. 处理所有"现在就能跑"的回调
    while ready_callbacks:
        cb = ready_callbacks.popleft()
        cb()    # 跑一段协程代码直到下一个 await

    # 2. 算出最近的定时任务还有多久到期
    timeout = compute_nearest_timer()

    # 3. 调用 OS 的 IO 多路复用（epoll / kqueue / select）
    #    这一步会"睡"一下，但有任何 IO 事件就醒
    events = selector.select(timeout)

    # 4. 把收到的 IO 事件对应的协程重新放进就绪队列
    for event in events:
        ready_callbacks.append(event.callback)

    # 5. 也检查哪些定时任务到期了，也放进就绪队列
    for timer in expired_timers():
        ready_callbacks.append(timer.callback)
```

就这么简单。`asyncio` 的核心代码本质就是上面这个循环 + 一堆辅助数据结构。

### 8.3 一段协程代码跑起来的过程

来细化一下"跑一段协程代码直到下一个 await"这句话。假设有这段代码：

```python
async def fetch_user(uid):
    print("start", uid)
    data = await http_get(f"/users/{uid}")
    print("got", uid)
    return data
```

调用 `fetch_user(1)` 返回一个协程对象。事件循环把它包装成 Task，放进就绪队列。循环从就绪队列取出它，开始跑：

1. 执行 `print("start", uid)`
2. 遇到 `await http_get(...)`，`http_get` 内部建了一个 socket，发出请求，**把这个 socket 注册给 selector**，然后 `yield` 把控制权交回事件循环
3. 事件循环从就绪队列里取下一个协程继续跑，**我们这个协程被挂起**
4. 过了一会儿，OS 通知 selector："fetch_user 的那个 socket 有数据可读了！"
5. 事件循环把 fetch_user 重新放进就绪队列
6. 循环再次处理它：继续执行 `print("got", uid)` 和 `return data`

这就是"一个线程里看起来同时跑 100 个协程"的真相：**所有协程都是在同一个线程里串行跑的，只是每个协程跑到 `await` 就主动歇一下，让事件循环去跑别的协程**。

### 8.4 "await 点"是唯一的切换机会

这句话要背下来：**在 asyncio 里，协程之间只能在 `await` 的地方切换。**

如果一段协程代码里没有 `await`，它就会**一口气跑完**，不会被打断。这和多线程完全不同——多线程可能在任意字节码处被操作系统切走。

后果：**如果你在协程里调用了一个阻塞的同步函数（比如 `time.sleep(5)`），整个事件循环会卡住 5 秒，所有其他协程都不能跑**。这是 asyncio 最大的坑，后面第 16 节会详细讲。

### 8.5 run / create_task / gather 的关系

这几个 API 初学者最容易混：

- **`asyncio.run(coro)`**：创建一个事件循环，跑 coro 直到它结束，然后关闭循环。**一个程序通常只在最顶层调用一次**。
- **`asyncio.create_task(coro)`**：在"当前事件循环"里调度一个 Task，**立即返回**。你可以稍后 await 这个 Task 收结果。用来真正实现"多个协程并发"。
- **`await some_coro()`**：在当前协程里暂停，等 some_coro 完成。**这是串行！** 如果你想并发跑两个协程，不能 `await a(); await b()`，那样是先跑完 a 再跑 b。
- **`asyncio.gather(c1, c2)`**：把多个协程/Task 打包并发运行，全部完成后一起返回结果。最常用。

看个对比：

```python
async def slow(name):
    await asyncio.sleep(1)
    return name

async def main():
    # 写法一：串行，2 秒
    a = await slow("A")
    b = await slow("B")

    # 写法二：并发，1 秒
    results = await asyncio.gather(slow("A"), slow("B"))

    # 写法三：create_task，1 秒，更灵活
    ta = asyncio.create_task(slow("A"))
    tb = asyncio.create_task(slow("B"))
    a = await ta
    b = await tb
```

三种写法的区别务必看懂。**"await" 不等于 "并发"**，这是新手最普遍的误区。

### 8.6 底层是 epoll / kqueue / select

最后补一点 OS 知识，帮你破除神秘感。

asyncio 之所以能单线程管几万个连接，是因为操作系统早就有"IO 多路复用"机制：
- Linux：`epoll`
- macOS / BSD：`kqueue`
- Windows：`IOCP` + `select`

你只需要一个系统调用，就能同时监视几万个 socket："这些里面谁有数据可读了？" OS 会阻塞等着，有任何一个准备好就立刻醒来并告诉你。没有多路复用的话，你得为每个 socket 开一个线程轮询，那才是几万线程的由来。

Python 的 `selectors` 模块封装了这些底层 API，asyncio 再在 `selectors` 之上搭了事件循环。整条栈是：

```
你的协程
   ↓
asyncio 事件循环
   ↓
selectors（封装 epoll/kqueue）
   ↓
操作系统内核
```

### 初学者常见误区

- **误区 1**：asyncio 是多线程。错。默认单线程。
- **误区 2**：asyncio 加速了计算。错。它只是把 IO 等待重叠起来。CPU 跑代码的总时间一点都没少。
- **误区 3**：`await a(); await b()` 是并发。错。是串行。
- **误区 4**：用 asyncio 就可以不用理解 event loop。错。你会一直踩坑。

## 9. async / await 语法详解

### 9.1 定义协程：async def

```python
async def hello():
    return 42
```

要点：
- 这是一个**协程函数（coroutine function）**。
- 调用它会返回一个**协程对象（coroutine object）**，**不会执行函数体**。
- 协程对象必须被事件循环调度才会运行（通过 `asyncio.run` / `await` / `create_task`）。

可以这样验证：

```python
c = hello()
print(type(c))     # <class 'coroutine'>
# 如果到这里不做 await 或 run，程序会发出警告："coroutine was never awaited"
```

### 9.2 await：等待一个"可等待对象"

`await` 后面必须是 **awaitable**（可等待对象）。awaitable 有三种：
1. **协程对象**（coroutine）：由 `async def` 函数调用产生
2. **Task**：`asyncio.create_task` 创建的
3. **Future**：低级的结果占位符（大多数情况下你不会直接用）

`await` 本质上做了两件事：
1. 如果目标还没完成，**把自己挂起**，控制权还给事件循环
2. 目标完成后，**恢复执行**，表达式的值就是目标的 return 值

```python
async def child():
    await asyncio.sleep(1)
    return "done"

async def parent():
    result = await child()   # 暂停 parent，等 child 跑完，再把结果赋给 result
    print(result)
```

### 9.3 能 await 什么、不能 await 什么

能 await：`asyncio.sleep(1)`、`httpx.AsyncClient().get(...)`、`asyncpg.fetch(...)`、另一个 `async def` 函数的调用结果……任何**awaitable**。

不能 await：
- `time.sleep(1)` — 这是同步函数，返回 None，await None 会报错
- `requests.get(...)` — 同步，返回 Response，不是 awaitable
- `1 + 1` — 不是 awaitable

**规则**：`await` 只能 "等一个能让出控制权的东西"。同步函数调用完就完了，没什么可等的。

### 9.4 协程只能在协程里调用

```python
async def a():
    return 1

def b():
    result = await a()   # SyntaxError: 'await' outside async function
```

`await` 必须写在 `async def` 函数里。那最顶层想调一个协程怎么办？用 `asyncio.run`：

```python
async def a():
    return 1

result = asyncio.run(a())   # 顶层唯一的入口
```

`asyncio.run` 内部做的事：创建事件循环 → 跑 a() → 关闭循环。整个 Python 程序通常只该调用它一次。

### 9.5 await 的返回值

`await` 表达式的值 = 被等待对象完成后的"结果值"。

```python
async def double(x):
    await asyncio.sleep(0)   # 模拟切一下
    return x * 2

async def main():
    r = await double(5)      # r == 10
```

如果被等待的协程里 raise 了异常，`await` 会把异常**重新抛到当前协程**。就像普通函数调用一样。

```python
async def bad():
    raise ValueError("oops")

async def main():
    try:
        await bad()
    except ValueError as e:
        print("caught:", e)
```

### 初学者常见误区

- **误区 1**：`async def` 就是更快的函数。错。不 await 任何东西的 `async def` 只是平白加了一层包装，比普通函数还慢。
- **误区 2**：所有函数都加 `async` 就变成并发了。错。协程之间必须通过 `gather` / `create_task` 等才能真正并发。
- **误区 3**：`await` 后面可以放同步函数。错。必须是 awaitable。
- **误区 4**：漏写 `await`。调用了 `fetch()` 但没 `await`，就只是拿到一个协程对象，什么都没跑。Python 会在退出时发警告 "coroutine was never awaited"，看到这个警告就是漏 await 了。

## 10. Task 与 Future

### 10.1 Task：在后台跑的协程

前面说过，单纯 `await another_coro()` 是串行。要并发就要让一个协程"后台跑"，这就是 **Task** 的作用。

```python
async def work(n):
    await asyncio.sleep(1)
    return n * 10

async def main():
    # 把 work(1) 提交给事件循环后台跑，立即返回一个 Task
    task = asyncio.create_task(work(1))
    # 此时 work(1) 已经在事件循环里调度了！
    # 我们还可以并发做其他事
    other = await work(2)
    # 最后收回 task 结果
    result = await task
    print(result, other)  # 10 20

asyncio.run(main())
```

`asyncio.create_task(coro)` 做了三件事：
1. 把 coro 包成一个 Task 对象
2. 把 Task 提交到当前事件循环的就绪队列
3. 立即返回 Task，**不等执行**

之后你可以：
- `await task` 等它结束并拿结果
- `task.done()` 查询是否完成（不阻塞）
- `task.result()` 拿结果（必须已 done，否则抛异常）
- `task.cancel()` 请求取消它

### 10.2 Future：更底层的结果占位符

Future 是比 Task 更底层的东西。它表示"一个将来会有结果的容器"。Task 其实是 Future 的一个子类（"跑一个协程得到结果"的 Future）。

日常写 asyncio 你几乎不需要手动创建 Future，除非你在**桥接回调式 API 和协程**时会用到。了解即可：

```python
loop = asyncio.get_event_loop()
fut = loop.create_future()

def callback_from_c_library(result):
    fut.set_result(result)   # 从外部设置结果，协程端的 await 会被唤醒

# 在协程里
result = await fut
```

场景：某个 C 扩展只提供"给我一个回调，数据来了我调你"的 API，没法直接 await。你就 create_future，让回调去 set_result，协程就能 await 这个 future 了。

### 10.3 Task 的典型陷阱：孤儿 Task

```python
async def main():
    asyncio.create_task(work(1))   # 没留引用！
    await asyncio.sleep(0.1)
```

这段代码有个问题：`create_task` 返回的 Task 对象没有被任何人引用。**事件循环对 Task 只是弱引用**，如果你不保存它，**可能被 GC 回收，任务莫名其妙消失**。这是 Python 3.11 之前的大坑。

正确做法：

```python
tasks = set()

async def main():
    t = asyncio.create_task(work(1))
    tasks.add(t)
    t.add_done_callback(tasks.discard)   # 完成后从集合里移除
```

或者更常见的用 `gather`（下一节讲），它内部帮你保管引用。

### 10.4 取消任务：CancelledError

`task.cancel()` 会往 task 里注入一个 `asyncio.CancelledError` 异常——下次这个 task 尝试 `await` 的时候，会收到这个异常。

```python
async def longrun():
    try:
        await asyncio.sleep(100)
    except asyncio.CancelledError:
        print("被取消了，做清理")
        raise   # 重要：要重新抛，否则取消没生效

async def main():
    t = asyncio.create_task(longrun())
    await asyncio.sleep(1)
    t.cancel()
    try:
        await t
    except asyncio.CancelledError:
        print("主流程收到取消确认")
```

要点：
- `CancelledError` 在 Python 3.8+ 不再继承自 `Exception`，而是继承自 `BaseException`。**不要用 `except Exception` 捕获后吞掉它**，否则取消机制会失效。
- 标准模式是：捕获 CancelledError → 做清理（关闭连接、回滚事务） → **重新 raise**。
- 被取消的 Task 状态变为 cancelled，`await task` 会抛 CancelledError。

## 11. asyncio 并发原语

本节介绍 5 个最常用的"把多个协程组合起来"的函数。

### 11.1 asyncio.gather — 全部并发，全部等完

最常用。

```python
async def main():
    r1, r2, r3 = await asyncio.gather(
        fetch("a"),
        fetch("b"),
        fetch("c"),
    )
```

特点：
- **并发启动**所有协程
- **全部完成后**一起返回，结果**按输入顺序**
- 任意一个抛异常，默认会把异常传出来，**其他协程继续跑**（Python 3.8+）
- 可以 `return_exceptions=True` 让异常作为结果返回而不是抛出

```python
results = await asyncio.gather(
    fetch("a"),
    fetch("b"),
    return_exceptions=True,
)
# results 可能是 [data_a, Exception("...")]
```

### 11.2 asyncio.as_completed — 谁先完成先处理谁

```python
async def main():
    tasks = [fetch(url) for url in urls]
    for coro in asyncio.as_completed(tasks):
        result = await coro
        print("got:", result)
```

特点：
- **流式返回**，快的先出
- 适合"处理单个结果比较耗时，想让快的先跑起来"

### 11.3 asyncio.wait — 更灵活的等待

```python
done, pending = await asyncio.wait(
    tasks,
    timeout=5,
    return_when=asyncio.FIRST_COMPLETED,   # 或 ALL_COMPLETED / FIRST_EXCEPTION
)
```

特点：
- 可以设定"等所有 / 等第一个完成 / 等第一个出错"
- 返回两个集合：`done`（已完成）和 `pending`（未完成），需要你自己处理 pending（通常是取消）
- 比 gather 底层，适合特殊场景

### 11.4 asyncio.timeout / wait_for — 超时控制

Python 3.11+ 推荐用 `asyncio.timeout`：

```python
async def main():
    try:
        async with asyncio.timeout(5):
            await long_task()
    except TimeoutError:
        print("超时")
```

老版本用 `wait_for`：

```python
try:
    result = await asyncio.wait_for(long_task(), timeout=5)
except asyncio.TimeoutError:
    print("超时")
```

两者行为相同：超过时间就取消内部的 task 并抛 `TimeoutError`。

### 11.5 asyncio.shield — 保护不被取消

```python
async def main():
    critical = asyncio.create_task(save_to_db())
    try:
        await asyncio.wait_for(asyncio.shield(critical), timeout=1)
    except asyncio.TimeoutError:
        # critical 并没被取消，继续在后台跑
        await critical
```

用途：你想给某个调用加超时，但又不想超时发生时真的取消它（比如写数据库操作一旦开始就要完成）。`shield` 让外层取消不会穿透到里面。实际用得不多，知道有这个东西即可。

## 12. 异步上下文管理器与异步迭代器

### 12.1 async with：异步上下文管理器

普通上下文管理器用 `with`，要求对象实现 `__enter__` 和 `__exit__`。异步版本用 `async with`，要求实现 `__aenter__` 和 `__aexit__`（两个都是 coroutine）。

为什么需要？因为"进入/退出上下文"的过程本身可能要做 IO。比如从连接池借一个 HTTP client，借用过程可能要等；关闭一个数据库连接可能要 flush；这些都是异步操作。

```python
async with httpx.AsyncClient() as client:
    resp = await client.get("https://example.com")
    print(resp.text)
```

`async with httpx.AsyncClient()` 展开等价于：

```python
client = httpx.AsyncClient()
await client.__aenter__()
try:
    resp = await client.get(...)
finally:
    await client.__aexit__(exc_type, exc, tb)
```

自己写一个：

```python
class MyPool:
    async def __aenter__(self):
        self.conn = await acquire()
        return self.conn

    async def __aexit__(self, exc_type, exc, tb):
        await release(self.conn)

async def main():
    async with MyPool() as conn:
        await conn.query("...")
```

### 12.2 async for：异步迭代器

普通 for 循环要求对象实现 `__iter__` 和 `__next__`。异步版本 `async for` 要求实现 `__aiter__` 和 `__anext__`（`__anext__` 是协程）。

用途：**流式处理**。不想等所有数据来齐再开始处理，而是来一条处理一条。

```python
async def main():
    async with httpx.AsyncClient() as client:
        async with client.stream("GET", url) as resp:
            async for chunk in resp.aiter_bytes():
                process(chunk)
```

LLM 的 SSE 流式响应就是典型场景：你不想等整段话生成完再显示，而是 token 流一个来就显示一个。这种场景几乎必然要 `async for`。

自己写一个异步生成器（`async def` + `yield`）：

```python
async def token_stream():
    for i in range(5):
        await asyncio.sleep(0.2)
        yield f"token-{i}"

async def main():
    async for t in token_stream():
        print(t)
```

注意：一个函数里同时出现 `async` 和 `yield` 就是 **async generator（异步生成器）**，它是 awaitable iterator，不是 coroutine，不能 await 它，只能 `async for`。

## 13. 异步生态库

Python 标准库内置的异步能力主要是 asyncio 本身 + 少量原语。真正实用的工具在第三方库里。下面这几个是 AI Agent 后端必用：

### 13.1 httpx — 异步 HTTP 客户端

`requests` 的 async 替代品。API 几乎一模一样：

```python
import httpx

async def fetch(url):
    async with httpx.AsyncClient(timeout=10) as client:
        resp = await client.get(url)
        resp.raise_for_status()
        return resp.json()

async def main():
    urls = [f"https://example.com/{i}" for i in range(10)]
    results = await asyncio.gather(*(fetch(u) for u in urls))
```

要点：
- **重用 client**：`AsyncClient` 内部有连接池，一次性创建、多次请求，比每次都新建快得多
- **用 `async with` 管理生命周期**
- `timeout` 强烈建议设置，防止某个请求卡死
- 支持 `stream` 模式处理 LLM SSE

### 13.2 aiofiles — 异步文件 IO

```python
import aiofiles

async def save(data):
    async with aiofiles.open("out.txt", "w") as f:
        await f.write(data)
```

为什么需要？因为 `open().write()` 是阻塞的系统调用，会卡住事件循环。aiofiles 通过把文件操作丢到后台线程池来模拟异步。严格说它不是"真异步"（Linux 下直到有 io_uring 才有真异步文件 IO），但对事件循环来说效果一样。

### 13.3 asyncpg — PostgreSQL 驱动

```python
import asyncpg

async def main():
    conn = await asyncpg.connect("postgresql://user:pass@host/db")
    rows = await conn.fetch("SELECT * FROM users WHERE age > $1", 18)
    for row in rows:
        print(row["name"])
    await conn.close()
```

生产环境用**连接池**：

```python
pool = await asyncpg.create_pool(dsn, min_size=5, max_size=20)

async def query():
    async with pool.acquire() as conn:
        return await conn.fetch("...")
```

asyncpg 是纯 async 实现的，性能非常好（比 psycopg2 + 线程池快几倍）。

### 13.4 redis.asyncio — 异步 Redis

```python
import redis.asyncio as redis

async def main():
    r = redis.from_url("redis://localhost:6379")
    await r.set("key", "value")
    value = await r.get("key")
    await r.close()
```

API 和同步的 `redis` 库基本一致，只是所有调用前加 await。

### 13.5 SQLAlchemy 2.x async

SQLAlchemy 2.0 原生支持 async：

```python
from sqlalchemy.ext.asyncio import create_async_engine, AsyncSession

engine = create_async_engine("postgresql+asyncpg://...")

async with AsyncSession(engine) as session:
    result = await session.execute(select(User).where(User.id == 1))
    user = result.scalar_one()
```

底层用 asyncpg，上层是熟悉的 ORM 接口。生产推荐。

### 13.6 选型建议

| 需求 | 推荐 |
|---|---|
| HTTP 客户端 | httpx |
| PostgreSQL | asyncpg 或 SQLAlchemy async + asyncpg |
| Redis | redis.asyncio |
| 文件 | aiofiles（偶尔用用，别滥用） |
| Kafka | aiokafka |
| RabbitMQ | aio-pika |
| gRPC | grpc.aio |

**全链路都用 async 库**很重要。一旦链路中混进一个同步库（比如某个 SDK 没有异步版本），那一段就会阻塞事件循环，整个性能就掉下来。混用的应对办法见第 15 节。

## 14. 并发控制：Semaphore / Queue / Lock

光会"并发发起 N 个任务"还不够，你经常需要**控制并发度**——比如 LLM API 有 QPS 限制，你必须限制最多同时发 10 个请求。这就要 Semaphore。

### 14.1 asyncio.Semaphore — 限流利器

Semaphore（信号量）的概念：有一个初始容量为 N 的计数器，acquire 时 -1，release 时 +1，降到 0 时 acquire 会等。

```python
sem = asyncio.Semaphore(10)    # 最多 10 个并发

async def fetch_limited(url):
    async with sem:            # acquire 并在退出时 release
        async with httpx.AsyncClient() as c:
            return await c.get(url)

async def main():
    urls = [...]  # 1000 个 URL
    await asyncio.gather(*(fetch_limited(u) for u in urls))
```

这样你扔进去 1000 个任务，但任意时刻只有 10 个真正在跑。**这是 AI Agent 里限流的标准模式**。

常见的变体：**限 QPS（每秒请求数）**。Semaphore 限的是并发度，QPS 限制要用 `aiolimiter` 这类库，或者自己写一个令牌桶。

### 14.2 asyncio.Queue — 异步队列

和 `queue.Queue` 类似，但所有操作都是 awaitable：

```python
q = asyncio.Queue(maxsize=100)

async def producer():
    for i in range(1000):
        await q.put(i)

async def consumer():
    while True:
        item = await q.get()
        if item is None:
            break
        await process(item)
        q.task_done()

async def main():
    producers = [asyncio.create_task(producer()) for _ in range(2)]
    consumers = [asyncio.create_task(consumer()) for _ in range(5)]
    await asyncio.gather(*producers)
    await q.join()              # 等所有已入队的都被 task_done
    for _ in consumers:
        await q.put(None)       # 哨兵通知退出
    await asyncio.gather(*consumers)
```

这是 asyncio 版的生产者消费者。对比多线程版本，代码结构几乎一样，但是没有锁，全靠协程切换保证同步。

### 14.3 asyncio.Lock

asyncio 也有 Lock，防止两个协程同时进入某段关键代码：

```python
lock = asyncio.Lock()

async def critical():
    async with lock:
        # 同一时刻只有一个协程能进这里
        await update_shared_state()
```

注意：**asyncio 的 Lock 不保护跨 await 的数据**，它是协程级的锁。由于协程只在 await 处切换，如果你某段代码没有 await，它天然就是"原子"的，根本不需要锁。所以 asyncio 里用 Lock 的场景远少于多线程。主要用于"读-改-写"中间夹着 await 的情况。

### 14.4 asyncio.Event / Condition

和 threading 版本语义一致，只是 await 替代阻塞。

```python
event = asyncio.Event()

async def waiter():
    await event.wait()   # 挂起直到 event.set()
    print("go!")

async def setter():
    await asyncio.sleep(1)
    event.set()
```

## 15. 混合编程

现实里你经常要混用同步和异步代码——因为不是所有库都有 async 版本。

### 15.1 从同步调异步：asyncio.run

你在一个普通 Python 脚本 / Django 同步视图里，想调一个异步函数怎么办？

```python
import asyncio

def sync_view(request):
    result = asyncio.run(async_fetch(url))
    return result
```

`asyncio.run` 会创建事件循环、跑目标协程、关闭循环。**但是有一个大坑**：

**不能在已经有事件循环运行的线程里调用 `asyncio.run`**！比如你在 FastAPI 的 async 视图里调 `asyncio.run(...)`，会报错 "asyncio.run() cannot be called from a running event loop"。FastAPI 视图本身就已经在事件循环里了，你应该直接 `await` 而不是 `asyncio.run`。

规则：
- **顶层脚本** / **同步环境**：用 `asyncio.run`
- **已经在协程里**：直接 `await`

### 15.2 从异步调同步阻塞函数：run_in_executor

假设你在 async 视图里，需要调一个只提供同步 API 的 SDK（比如某个冷门的 ML 库）：

```python
async def handler():
    # ❌ 错误：这会阻塞整个事件循环
    result = sdk.do_something()

    # ✅ 正确：丢到线程池
    loop = asyncio.get_running_loop()
    result = await loop.run_in_executor(None, sdk.do_something)
    # None 表示用默认线程池
```

Python 3.9+ 有更简洁的语法：

```python
result = await asyncio.to_thread(sdk.do_something, arg1, arg2)
```

`asyncio.to_thread` 内部就是把调用丢到默认 ThreadPoolExecutor。注意线程池默认大小有限（`min(32, os.cpu_count() + 4)`），如果你大量丢阻塞调用进去会排队。需要的话自己创建一个更大的 Executor：

```python
executor = ThreadPoolExecutor(max_workers=100)
result = await loop.run_in_executor(executor, sdk.do_something)
```

这个模式是**异步框架里集成同步库的标准做法**。FastAPI 的同步视图 (`def` 而不是 `async def`) 内部就是自动 `run_in_executor`。

### 15.3 CPU 密集用 run_in_executor(ProcessPool)

如果阻塞调用是 CPU 密集，线程池没用（GIL），要用进程池：

```python
from concurrent.futures import ProcessPoolExecutor

executor = ProcessPoolExecutor()

async def handler():
    result = await loop.run_in_executor(executor, heavy_cpu_work, data)
```

这样做 CPU 工作就真的到别的进程去了，事件循环线程继续跑其他协程。

### 15.4 在子线程里跑 asyncio：asyncio.run_coroutine_threadsafe

偶尔你有一个事件循环跑在主线程，子线程里想提交一个协程进去：

```python
import asyncio, threading

loop = asyncio.new_event_loop()

def run_loop():
    asyncio.set_event_loop(loop)
    loop.run_forever()

threading.Thread(target=run_loop, daemon=True).start()

# 从另一个线程
future = asyncio.run_coroutine_threadsafe(some_coro(), loop)
result = future.result()   # 阻塞等结果
```

这种模式用得少，但偶尔能救命。

## 16. asyncio 常见坑与调试

### 16.1 坑 1：忘记 await

```python
async def main():
    fetch(url)    # 漏了 await！
```

后果：`fetch` 协程对象被创建但没跑，到程序退出时 Python 警告 "coroutine was never awaited"。数据没写、请求没发。

**防范**：看到 async def 函数调用一定 await。IDE 里把"Coroutine was never awaited"的 warning 当 error 看。

### 16.2 坑 2：阻塞调用污染事件循环

```python
async def handler():
    time.sleep(5)   # 灾难
    data = requests.get(url).json()   # 灾难
```

`time.sleep` 和 `requests.get` 都是阻塞的同步调用。在协程里执行它们会**卡住整个事件循环 5 秒**，期间所有其他协程都不能跑，用户看到所有请求都在 hang。

**修复**：
- `time.sleep(5)` → `await asyncio.sleep(5)`
- `requests.get(...)` → `await httpx.AsyncClient().get(...)` 或 `await asyncio.to_thread(requests.get, url)`

**检测**：Python 3.7+ 设置环境变量 `PYTHONASYNCIODEBUG=1` 或代码里 `asyncio.get_event_loop().set_debug(True)`，事件循环会检测一次调度超过 100ms 没完成的协程并警告。这是定位阻塞调用的利器。

### 16.3 坑 3：CancelledError 被吞

```python
async def worker():
    try:
        await long_task()
    except Exception:       # ❌ 会吞掉 CancelledError（3.8+ 继承自 BaseException 才幸免）
        pass
```

Python 3.8 之前 `CancelledError` 继承自 Exception，这样写会吞掉取消信号。Python 3.8+ 改成了继承 BaseException 就没这个问题，但还是建议**显式捕获并重新抛**：

```python
async def worker():
    try:
        await long_task()
    except asyncio.CancelledError:
        await cleanup()
        raise
    except Exception as e:
        log.error(e)
```

### 16.4 坑 4：孤儿 Task 被 GC

前面讲过。解决：把 Task 存到一个集合，完成时 discard。

### 16.5 坑 5：混用多个事件循环

```python
asyncio.run(main1())
asyncio.run(main2())
```

每次 run 都创建新循环，两个 run 之间共享的 Task / Queue 就跨循环了，报错 "attached to a different loop"。规则：**一个 Python 程序只跑一个事件循环**。

### 16.6 坑 6：异步里建同步连接

```python
async def handler():
    conn = psycopg2.connect(...)   # 同步建连接 + 后续所有操作都阻塞
```

老老实实用 asyncpg。

### 16.7 调试手段

1. **debug mode**：`asyncio.run(main(), debug=True)`，开启后会警告慢协程、异常未处理等
2. **logging**：asyncio 内部用 `asyncio` logger，设 DEBUG 级别能看调度详情
3. **`task.get_name()` / `Task.current_task()`**：日志里打任务名，定位问题
4. **py-spy / scalene**：采样式 profiler，可以看事件循环在哪阻塞
5. **print callstack**：协程 hang 住时，`import traceback; traceback.print_stack(frame)` 配合 Task 调试

## 17. 在 AI Agent 后端里的应用场景

前面讲了一堆原理，现在回到主线：**这些东西在 AI Agent 后端服务里到底是怎么用的**。这是你面 BetterYeah 时要讲清楚的东西。

### 17.1 场景一：并发调用多个 LLM / 多个工具

Agent 一次对话经常要：
- 同时调多个工具（search、calculator、code_exec）
- 或并行跑几个 LLM（不同 prompt 做投票）
- 或同时请求多个 embedding

典型代码：

```python
async def agent_turn(user_msg):
    # 同时发起 3 个 LLM 调用
    results = await asyncio.gather(
        llm.call(prompt_a, user_msg),
        llm.call(prompt_b, user_msg),
        llm.call(prompt_c, user_msg),
        return_exceptions=True,
    )
    # 投票选最优
    return majority_vote(results)
```

用 `gather` + `return_exceptions=True` 保证个别失败不影响整体。

### 17.2 场景二：流式响应（SSE / WebSocket）

用户要求"实时看到 LLM 一个词一个词吐出来"。LLM SDK 提供 async iterator，FastAPI StreamingResponse 承接：

```python
async def stream_chat(req):
    async def token_gen():
        async for chunk in llm.stream(req.messages):
            yield f"data: {chunk}\n\n"
    return StreamingResponse(token_gen(), media_type="text/event-stream")
```

这里 `async for` 是必需的——同步的 for 会等整段生成完才能 yield 第一个词。

### 17.3 场景三：LLM API 限流

OpenAI / Anthropic 都有 rate limit，超了会 429。用 Semaphore + 重试：

```python
LLM_SEMAPHORE = asyncio.Semaphore(20)

@retry(attempts=3, backoff="exponential")
async def llm_call(msgs):
    async with LLM_SEMAPHORE:
        return await client.chat.completions.create(model=..., messages=msgs)
```

### 17.4 场景四：后台长任务与任务队列

有些 Agent 任务要跑几十分钟（爬大量数据、多轮推理）。不能卡住 HTTP 请求。做法：
- HTTP 视图收到请求 → 往 Redis Stream / Celery 投一个任务 → 立即返回 task_id
- 后台 worker 消费任务（`asyncio.create_task` 或 worker 进程）
- 客户端轮询 task_id 或用 WebSocket 接收进度

这里就需要 `asyncio.Queue`、`create_task`、以及正确处理取消（用户中途撤销）。

### 17.5 场景五：连接池管理

- HTTP 到 LLM：一个全局的 `httpx.AsyncClient`，整个服务生命周期共用
- 数据库：asyncpg 的 `create_pool`，通常 min=5 max=20
- Redis：全局一个 `redis.asyncio.Redis` 实例，内部有连接池

**反模式**：每个请求新建 client。你会很快把服务器的文件描述符耗光。

### 17.6 场景六：超时与取消

用户关闭浏览器时 FastAPI 会收到取消信号。你的 handler 要能正确传递：

```python
async def handler():
    try:
        async with asyncio.timeout(30):
            return await long_llm_call()
    except asyncio.TimeoutError:
        return {"error": "timeout"}
    except asyncio.CancelledError:
        await rollback()
        raise
```

### 17.7 场景七：RAG 检索的并发

RAG 检索过程经常是：
- 同时用多个 retriever（vector、BM25、knowledge graph）
- 各自返回 top-k 后做 rerank

```python
async def retrieve(query):
    vec, bm25, kg = await asyncio.gather(
        vector_search(query),
        bm25_search(query),
        kg_search(query),
    )
    candidates = vec + bm25 + kg
    return await rerank(query, candidates)
```

三路并行发起，总耗时 = 最慢那一路。

### 17.8 小结：设计原则

写 AI Agent 后端时的几条硬规则：
1. **全链路 async**：httpx + asyncpg + redis.asyncio，不混同步
2. **连接池全局共享**：HTTP client、DB pool、Redis client 都是进程级单例
3. **每个 LLM/工具调用都加超时和限流**：Semaphore + timeout + retry
4. **长任务异步化**：HTTP 视图只负责派发，实际工作在 worker
5. **正确处理取消**：顶层 handler 捕获 CancelledError 做清理然后 re-raise

### 题 1：解释 Python 的 GIL，它对多线程的影响是什么？

**参考答案骨架**：

1. **定义**：GIL 是 CPython 解释器里的一把互斥锁，保证同一时刻只有一个线程在执行 Python 字节码。
2. **历史原因**：CPython 用引用计数管理内存，引用计数的增减不是原子的，加 GIL 是最便宜的线程安全方案。
3. **对多线程影响分场景**：
   - CPU 密集：多线程无加速，因为任意时刻只能有一个线程在跑字节码。
   - IO 密集：多线程有效。`time.sleep` / socket IO 这些阻塞系统调用会主动释放 GIL，让别的线程跑。
4. **怎么绕过**：
   - 用多进程（每进程一把独立 GIL）
   - 用 asyncio 单线程 + 事件循环
   - 把 CPU 密集部分下沉到 C 扩展（numpy / pytorch），C 层会释放 GIL
5. **未来**：Python 3.13+ 有 PEP 703 的 free-threaded build，实验中。

### 题 2：并发、并行、异步的区别？

先讲三句话定义（见 §2.1），再用餐厅类比，最后用代码说：
- 多进程 = 并行（物理上同时）
- 单进程多线程 = 并发（OS 调度让 CPU 看起来同时跑）
- asyncio 单线程 = 并发（协作式调度，靠 await 让权）

强调：**异步是编程风格，不是运行方式**。

### 题 3：asyncio 的事件循环是怎么工作的？

按 §8 讲：
1. 维护就绪队列 + 注册到 OS IO 多路复用的 fd 集合
2. 循环：跑就绪队列里所有回调 → 算出最近的 timer → selector.select 阻塞等待 → 把 ready 的事件塞回就绪队列
3. 协程跑到 await 就挂起，把对应的 fd 注册给 selector，控制权回到循环
4. 所有协程在同一个线程，串行执行，只是在 await 处切换

### 题 4：我的 FastAPI 服务 CPU 不高但 QPS 上不去，怎么排查？

参考答案：
1. 先看视图函数签名，是 `def` 还是 `async def`
2. 如果是 `async def`，看内部有没有调同步阻塞库：requests、psycopg2、time.sleep、大的同步计算
3. 用 `asyncio.get_event_loop().set_debug(True)` 或 `PYTHONASYNCIODEBUG=1` 检测慢回调
4. 用 py-spy dump 看有没有协程卡在某个同步调用
5. 确认数据库连接池大小合理（min/max、timeout）
6. 确认 httpx Client 是全局重用而非每次新建
7. 如果真是 CPU 瓶颈，考虑把计算丢 ProcessPoolExecutor

### 题 5：1000 个 URL 要爬，每个 API QPS 上限 20，写出伪代码。

考察 Semaphore + gather：

```python
sem = asyncio.Semaphore(20)

async def fetch(url):
    async with sem:
        async with httpx.AsyncClient(timeout=10) as c:
            return (await c.get(url)).text

async def main():
    urls = load_urls()
    results = await asyncio.gather(*(fetch(u) for u in urls), return_exceptions=True)
```

追问：怎么做成 QPS 而不是并发度？答：用 aiolimiter 或手写令牌桶。

### 题 6：协程里 time.sleep(5) 会怎样？

卡住整个事件循环 5 秒。所有其他协程都不能运行，所有等待中的请求都 hang。修复：改成 `await asyncio.sleep(5)` 或 `await asyncio.to_thread(time.sleep, 5)`（后者更 hack，正常应用不需要）。

### 题 7：如何正确取消一个 asyncio 任务？

1. 调 `task.cancel()`
2. 被取消的协程会在下一次 await 时收到 `CancelledError`
3. 协程内部应该捕获它 → 做清理 → **重新 raise**
4. 外层代码 `await task` 也会抛 `CancelledError`，据此知道确实取消了
5. 陷阱：不要用 `except Exception` 捕获后吞掉

### 题 8：多线程 + Lock 和 asyncio + Lock 的区别？

- 多线程 Lock：保护"跨字节码"的原子性。线程随时会被 OS 切走，任意时刻都可能有竞态。
- asyncio Lock：只保护"跨 await"的原子性。协程之间只在 await 处切换，没有 await 的代码天然原子，不需要锁。
- 结论：asyncio 里 Lock 用得少，多线程里 Lock 是日常。

---

## 总结

| 技术 | 场景 | 核心理解 |
|---|---|---|
| threading | IO 密集，任务数中等 | 多线程共享内存，要用 Lock 管理竞态 |
| multiprocessing | CPU 密集 | 绕 GIL，代价是序列化开销 |
| concurrent.futures | 两者的统一 API | 日常用 ThreadPoolExecutor |
| asyncio | IO 密集，任务数大，框架友好 | 单线程事件循环 + 协程协作切换 |
| httpx/asyncpg/redis.asyncio | 异步生态 | 全链路 async，切勿混同步 |
| Semaphore/Queue/Lock | 并发控制 | 限流、生产者消费者、临界区 |
| run_in_executor | 混合编程 | 异步里调同步阻塞的救急手段 |

最后一段话送你：**并发不是银弹。先写同步代码跑通，需要并发时再加。过早引入 asyncio 会让代码复杂度翻倍，而收益可能是零。但在 AI Agent 后端这个 IO 密集的场景里，不用 asyncio 几乎就是放弃性能。**
